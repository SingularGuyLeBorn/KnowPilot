## 任务 2: 14 个核心实体的 tRPC 接口与 Zustand Store 全量开发，Git 初始化与规范整理

### 问题发现
用户要求实现包含新引入实体在内的 14 个核心实体的全量 CRUD 接口与前端 Zustand Store，并且参数和字段类型必须 100% 对齐。此外，需要配置好 MIT LICENSE，用标准的 `.gitkeep` 替代冗余的占位 `README.md`，并初始化 Git 仓库进行完整 Commit。

### 根因分析
* 原有的 README.md 占位只是为了防止空目录被 Git 忽略，但在规范的项目中会显得凌乱，应使用 `.gitkeep` 隐藏文件来占位。
* 需要保证 14 个实体（新增 Workspace, Trigger, Approval 实体）后端 API 完整无误，并且前端通过 Zustand + tRPC Client 精准调用，不发生类型偏差。
* Windows 环境下，Git 默认不启用 `core.longpaths` 会导致路径较深的 Markdown 文件（如从 MetaBlog 拷贝过来的多模态与原理等长路径专栏）在 `git add` 时报 "Filename too long" 错误。

### 方案选择
* **JSON 存储方案**：直接使用 Prisma native `Json` 型写入和读取（即 `input.metadata ?? undefined`），不使用额外的 `JSON.stringify` 和 `JSON.parse`。这消除了之前在 log 和 task 实体中导致 TS 报错的类型冲突问题。
* **tRPC AppRouter 类型导出**：在 `apps/server/src/index.ts` 导出并在 `apps/server/package.json` 中配置 `"main"`/`"types"` 指向它，再用 `"@knowpilot/server": "workspace:*"` 引入，实现了完美的扁平导入。

### 实现过程
1. **清理占位与 LICENSE 文件**：删除 `content/skills/README.md` 等多余说明，放入 `.gitkeep`；在根目录生成 `LICENSE` (MIT)。
2. **扩展数据库模型**：在 `prisma/schema.prisma` 新增 `Workspace` (工作区), `Trigger` (触发器), `Approval` (审批队列) 实体并执行 `pnpm db:push`。
3. **补充共享 Zod Schema**：在 `packages/shared/src/schemas.ts` 添加 these 实体的 Zod schema 校验及推导类型。
4. **后端 tRPC 路由编写**：完成全部 11 个后端的 tRPC Router，汇总合并至 `router.ts` 中。
5. **解除 Express 强耦合**：修改 `context.ts` 使得 context 的 `Context` 类型仅包含 prisma 引用，避免了 express 在多包下未引入 `@types/qs` 时引发的 `TS2742` 编译错误。
6. **Next.js 客户端实例化与 Zustand Store**：在 `apps/web/lib/trpc.ts` 初始化 tRPC 客户端，在 `apps/web/lib/stores/` 中建立各实体的 Zustand 状态 store（`postStore`, `agentStore`, `skillStore`, `workspaceStore` 与 `others.ts` ）。
7. **单元测试与 Git 仓库初始化**：添加 `apps/server/src/__tests__/trpc.test.ts` 进行 Workspace CRUD 联合测试；初始化 Git，开启 `core.longpaths`，并执行 `git add .` 与 `git commit`。
