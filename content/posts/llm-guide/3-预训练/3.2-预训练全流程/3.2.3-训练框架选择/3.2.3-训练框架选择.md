# 3.2.3 训练框架选择

## 1. 分布式训练的核心挑战

### 1.1 显存瓶颈

大模型训练对显存的需求来自四个维度：

| 维度 | 存储内容 | 70B 模型估算 |
|:----|:--------|:-----------:|
| **模型参数** | 权重矩阵 | ~140 GB |
| **梯度** | 反向传播的计算结果 | ~140 GB |
| **优化器状态** | Adam 的动量/方差 | ~280 GB |
| **激活值** | 前向传播的中间结果 | ~60-120 GB(与 batch size 相关) |
| **总计** | | **~620-700 GB** |

### 1.2 计算瓶颈

即使显存问题得到解决，单张 GPU 的计算速度也无法在合理时间内完成万亿 token 级别的训练. 以 Llama 3 70B 为例，在 16,384 张 H100 上训练了约 15T tokens，耗时约 54 天. 这需要在数千张 GPU 之间高效协同，对通信带宽、拓扑结构和并行策略提出了极高的要求. 

### 1.3 通信瓶颈

分布式训练中，GPU 之间需要频繁同步梯度/参数. **All-Reduce** 通信的开销随 GPU 数量增加而增长，如果通信拓扑设计不当，通信开销可能远超过计算开销，导致加速比严重偏离线性.

## 2. 3D 并行技术详解

业界主流方案是**3D 并行**——将数据并行、流水线并行和张量并行三种技术组合使用，从不同维度切分计算与显存. 

### 2.1 数据并行(Data Parallelism)

**原理**：将训练数据切分到多张 GPU 上，每张 GPU 持有完整的模型副本，各自计算梯度后通过 All-Reduce 通信同步. 

```
GPU 0: [Batch 0] → 前向/反向 → 梯度 → All-Reduce
GPU 1: [Batch 1] → 前向/反向 → 梯度 → All-Reduce
GPU 2: [Batch 2] → 前向/反向 → 梯度 → All-Reduce
GPU 3: [Batch 3] → 前向/反向 → 梯度 → All-Reduce
                                      ↓
                              所有梯度求和后更新参数
```

**优点**：
- 实现简单，只需在常规训练代码上加 `torch.nn.DistributedDataParallel`
- 计算完全独立，仅在梯度同步时通信
- 可以线性扩大 batch size

**缺点**：
- 每张 GPU 都需要容纳完整的模型参数+梯度+优化器状态(显存冗余)
- 在模型规模 > GPU 显存时不可用

**适用场景**：模型单卡可容纳，需要加速训练速度时. 

### 2.2 流水线并行(Pipeline Parallelism)

**原理**：将模型的不同层(Layer)切分到不同 GPU 上，数据像流水线一样依次通过各 GPU. 

```
GPU 0: [Layers 1-10] → 输出 → 
GPU 1: [Layers 11-20] → 输出 → 
GPU 2: [Layers 21-30] → 输出 → 
GPU 3: [Layers 31-40] → 输出
```

**关键概念**：

| 概念 | 说明 |
|:----|:-----|
| **Micro-batch** | 将每个 batch 进一步拆分为更小的微批次，让流水线尽快填满 |
| **Pipeline Bubble** | 流水线启动和排空阶段的空闲时间(理想情况下气泡占比约 (p-1)/m，p=流水线阶段数，m=微批次数) |
| **1F1B(One Forward One Backward)** | 交替执行前向和反向传播的策略，减少激活值显存占用 |

**优点**：
- 大幅降低单卡显存需求(每张卡只存部分层)
- 通信量较小(仅在流水线阶段间传输激活值和梯度)

**缺点**：
- 存在流水线气泡(GPU 空闲等待)，利用率通常 80-90%
- 实现复杂度较高，需要手动或自动切分模型

**典型实现**：GPipe、PipeDream、Megatron-LM 的 interleaved pipeline scheduling. 

### 2.3 张量并行(Tensor Parallelism)

**原理**：将单个 Transformer 层内部的矩阵运算切分到多张 GPU 上并行计算. 这是最细粒度的并行方式. 

以 Attention 层为例，将多头注意力中的头数分配到不同 GPU：

```
输入: [batch, seq, hidden]
        ↓
┌─────────────────────────────────────────┐
│  GPU 0: heads 0-7    GPU 1: heads 8-15  │
│  GPU 2: heads 16-23  GPU 3: heads 24-31 │
└─────────────────────────────────────────┘
        ↓
     All-Reduce 拼接所有头的输出
        ↓
     线性投影 + 输出
```

**关键通信模式**：

| 操作 | 通信模式 | 通信量 |
|:----|:--------|:------|
| **列切分(Column-wise)** | 前向无需通信，反向需 All-Reduce 梯度 | $2 \times hidden$ |
| **行切分(Row-wise)** | 前向需 All-Reduce，反向无需通信 | $2 \times hidden$ |
| **Attention 切分** | 前向/反向各一次 All-Reduce | $4 \times seq \times head\_dim$ |

**优点**：
- 每张 GPU 的显存需求降至原来的 $1/t$(t 为张量并行度)
- 计算和通信可以重叠(overlap)，效率较高

**缺点**：
- 通信量较大，要求节点内 GPU 之间的高速互联(NVLink/NVSwitch)
- 张量并行度受限于单节点内的 GPU 数量(通常为 8)

**典型实现**：Megatron-LM 的张量并行方案是业界标准.

## 3. 3D 并行的组合策略

实际大规模预训练中，**三种并行方式组合使用**，各司其职：

| 并行方式 | 作用范围 | 通信需求 | 典型拆分度 |
|:--------|:--------|:--------|:---------|
| **数据并行** | 跨节点(跨机器) | 高(All-Reduce 梯度) | 64~1024 |
| **流水线并行** | 跨节点(跨机器) | 低(点对点通信) | 4~16 |
| **张量并行** | 节点内(同机器) | 极高(NVLink 互联) | 1~8 |

**分层组合示意图**：

```
┌─────────────────────────────────────────────┐
│              数据并行 (DP=8)                  │
│  ┌────────┐  ┌────────┐     ┌────────┐      │
│  │ Node 0 │  │ Node 1 │ ... │ Node 7 │      │
│  └────┬───┘  └────┬───┘     └────┬───┘      │
│       │            │              │           │
│  流水线并行(PP=4)  │              │           │
│  ┌─┬─┬─┬─┐  ┌─┬─┬─┬─┐     ┌─┬─┬─┬─┐        │
│  │G0│G1│G2│G3││G0│G1│G2│G3│ ... │G0│G1│G2│G3││
│  └─┴─┴─┴─┘  └─┴─┴─┴─┘     └─┴─┴─┴─┘        │
│  ↑───────↑  ↑───────↑        ↑───────↑       │
│  张量并行(TP=8)                               │
└─────────────────────────────────────────────┘
总 GPU 数 = DP × PP × TP = 8 × 4 × 8 = 256
```

## 4. ZeRO 显存优化技术

### 4.1 问题背景

在标准数据并行中，每张 GPU 都存储了完整的优化器状态、梯度和参数，存在大量冗余. ZeRO(Zero Redundancy Optimizer)的核心思想是：**将数据并行中的冗余显存分配到各 GPU 上，仅在需要时才收集完整数据**. 

### 4.2 ZeRO 的三个级别

| 级别 | 切分内容 | 显存节省 | 通信开销 |
|:----|:--------|:-------:|:--------:|
| **ZeRO-1** | 仅切分优化器状态 | 4x | 无增加(与 DP 相同) |
| **ZeRO-2** | 优化器状态 + 梯度 | 8x | 略有增加 |
| **ZeRO-3** | 优化器状态 + 梯度 + 参数 | 16x | 显著增加(需实时获取参数) |

**显存效果对比(以 7.5B 模型为例)** ：

```
标准数据并行:     ~120 GB(4张GPU各存完整模型)
ZeRO-1:           30 GB(优化器状态分散到4张GPU)
ZeRO-2:           16 GB(优化器+梯度状态分散)
ZeRO-3:           8 GB(全部状态分散，按需收集)
```

### 4.3 ZeRO-1 详解

**切分优化器状态**：将 Adam 的动量项和方差项(通常占显存最大比例)切分到各 GPU. 

```
GPU 0: 参数 [0, N/4) 的优化器状态
GPU 1: 参数 [N/4, N/2) 的优化器状态
GPU 2: 参数 [N/2, 3N/4) 的优化器状态
GPU 3: 参数 [3N/4, N) 的优化器状态
```

训练过程中，每张 GPU 仍然持有完整参数和梯度，但只对自己负责的参数部分进行优化器更新. 优化器更新前不需要通信(因为每张 GPU 都已有完整梯度)，更新后各 GPU 的优化器状态保持独立. 

**显存节省**：以 Adam 优化器为例，标准 DP 需要存储：
- 参数(FP16)：2 bytes/param
- 梯度(FP16)：2 bytes/param  
- Adam 动量(FP32)：4 bytes/param
- Adam 方差(FP32)：4 bytes/param
- 总计：**12 bytes/param**

ZeRO-1 将优化器状态(8 bytes/param)切分到 $N_d$ 张 GPU，每张 GPU 的显存降至：
- 参数 + 梯度：4 bytes/param(未切分)
- 优化器状态：$8/N_d$ bytes/param
- 总计：**$4 + 8/N_d$ bytes/param**

当 $N_d = 64$ 时，ZeRO-1 每卡仅需约 **4.125 bytes/param**，而标准 DP 需要 12 bytes/param，节省约 **3 倍**. 

### 4.4 ZeRO-2 详解

**切分梯度**：在 ZeRO-1 的基础上，进一步将梯度切分到各 GPU. 

梯度在反向传播完成后立即被切分并分散到各 GPU，每张 GPU 只保留自己负责的梯度部分. 要计算某张 GPU 上参数的梯度时，需要从其他 GPU 收集. 

**通信变化**：ZeRO-1 的梯度同步仍然使用标准的 All-Reduce(所有 GPU 参与)，而 ZeRO-2 使用 **Reduce-Scatter** 替代 All-Reduce. Reduce-Scatter 将梯度求和并切分，一次性完成「求和+切分」两步操作，通信量不变但减少了内存占用. 

**显存效果**：每张 GPU 不再需要存储完整梯度(2 bytes/param)，显存降至：
- 参数(FP16)：2 bytes/param
- 梯度(切分后)：$2/N_d$ bytes/param
- 优化器状态(切分后)：$8/N_d$ bytes/param
- 总计：**$2 + 10/N_d$ bytes/param**

### 4.5 ZeRO-3 详解

**切分参数**：对模型参数也进行切分，每张 GPU 只持有部分参数. 在前向传播和反向传播时，需要动态收集当前层所需的参数(从各 GPU 收集)，计算完后再释放. 

**通信变化**：ZeRO-3 引入了**额外的参数收集通信**. 在前向传播的每一层，都需要：
1. 从所有 GPU 收集该层参数(All-Gather)
2. 执行前向计算
3. 计算完成后释放非本卡的参数(保持切分状态)

反向传播同理，需要再次收集参数. 这使得 ZeRO-3 的总通信量约为 ZeRO-2 的 **1.5 倍**. 

**显存效果**：
- 参数(切分后)：$2/N_d$ bytes/param
- 梯度(切分后)：$2/N_d$ bytes/param
- 优化器状态(切分后)：$8/N_d$ bytes/param
- 总计：**$12/N_d$ bytes/param**

当 $N_d = 64$ 时，每张 GPU 仅需约 **0.19 bytes/param**——可以训练比单卡容量大 64 倍的模型！

### 4.6 ZeRO 的进阶版本

| 版本 | 新特性 | 适用场景 |
|:----|:------|:--------|
| **ZeRO-Offload** | 将优化器状态和梯度卸载到 CPU 内存 | GPU 显存极度有限时 |
| **ZeRO-Infinity** | 支持将参数卸载到 NVMe 固态硬盘 | 超大规模模型训练 |
| **ZeRO++** | 量化通信、分层切分、混合粒度 | 跨节点通信瓶颈场景 |

## 5. 主流训练框架对比

### 5.1 DeepSpeed

**开发者**：Microsoft
**核心优势**：ZeRO 优化系列

| 特性 | 说明 |
|:----|:-----|
| **ZeRO 系列** | ZeRO-1/2/3 + Offload + Infinity，极致显存优化 |
| **自动并行** | AutoTP(自动张量并行)，降低使用门槛 |
| **通信优化** | ZeRO++ 分层量化通信，跨节点性能优异 |
| **易用性** | 只需在 PyTorch 代码上添加少量 `deepspeed.initialize` 调用 |
| **社区生态** | HuggingFace Transformers 原生集成 |

**典型用法**：
```python
import deepspeed

model_engine, optimizer, _, _ = deepspeed.initialize(
    model=model,
    model_parameters=model.parameters(),
    config_params={
        "train_batch_size": 32,
        "gradient_accumulation_steps": 4,
        "zero_optimization": {
            "stage": 3,
            "offload_optimizer": {"device": "cpu"}
        }
    }
)
```

### 5.2 Megatron-LM

**开发者**：NVIDIA
**核心优势**：张量并行 + 流水线并行(3D 并行)

| 特性 | 说明 |
|:----|:-----|
| **张量并行** | 业界最早也是最高效的张量并行实现，针对 NVLink/NVSwitch 深度优化 |
| **流水线并行** | 支持 interleaved pipeline scheduling，气泡占比降至最低 |
| **序列并行** | 将序列维度也做切分，进一步节省激活值显存 |
| **分布式优化器** | 支持 ZeRO-1 级别的优化器状态切分 |
| **性能** | 在 NVIDIA GPU 上利用率极高，可达 50%+ 的 MFU |

**典型用法**(需要模型代码适配)：
```python
# Megatron-LM 需要按张量并行和流水线并行的方式组织模型
from megatron.core import tensor_parallel as tp

# 列切分 Linear 层
linear_layer = tp.ColumnParallelLinear(
    hidden_size, intermediate_size,
    gather_output=False
)
```

### 5.3 框架对比总结

| 维度 | DeepSpeed | Megatron-LM | PyTorch FSDP |
|:----|:---------|:-----------|:------------|
| **显存优化** | ⭐⭐⭐⭐⭐ ZeRO-3 + Offload | ⭐⭐⭐⭐ 3D 并行 | ⭐⭐⭐⭐ ZeRO-3 等效 |
| **易用性** | ⭐⭐⭐⭐ 侵入性低 | ⭐⭐ 需模型改造 | ⭐⭐⭐⭐⭐ 原生支持 |
| **计算效率** | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ 极致优化 | ⭐⭐⭐⭐ |
| **通信优化** | ⭐⭐⭐⭐ ZeRO++ | ⭐⭐⭐⭐⭐ NVIDIA 生态 | ⭐⭐⭐ |
| **社区生态** | ⭐⭐⭐⭐⭐ HF 集成 | ⭐⭐⭐⭐ NVIDIA 生态 | ⭐⭐⭐⭐⭐ PyTorch 原生 |
| **适用场景** | 中小规模快速实验 | 大规模旗舰训练 | 通用训练任务 |

### 5.4 框架选择建议

| 场景 | 推荐框架 | 理由 |
|:----|:--------|:-----|
| **~1B 模型实验** | PyTorch FSDP | 零侵入，开箱即用 |
| **7B~13B 模型单机训练** | DeepSpeed ZeRO-3 | 显存节省显著，配置简单 |
| **70B+ 多机训练** | Megatron-LM + DeepSpeed | TP+PP+DP 组合，GPU 利用率最高 |
| **旗舰模型 (100B+)** | Megatron-LM 3D 并行 | 已被 GPT-4、Llama 3 验证 |

## 6. 训练框架的实际集成模式

在实际的大规模预训练项目中，Megatron-LM 和 DeepSpeed 经常**组合使用**：

```
Megatron-LM：负责 3D 并行框架(TP + PP + DP)
     +
DeepSpeed：负责 ZeRO 优化 + 数据加载 + Checkpoint
     +
NVIDIA Apex / Transformer Engine：负责混合精度训练和 FP8 支持
```

这种组合模式被称为 **Megatron-DeepSpeed**，已被多个大规模训练项目采用.
