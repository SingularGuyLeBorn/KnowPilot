# 3.2.5 训练容灾与监控

## 1. 挑战的严峻性：故障是常态

### 1.1 大规模集群的故障率

在数千张 GPU 的集群上，故障的统计学几乎是确定的. 根据 Google 和 Meta 的公开数据：

| 指标 | 典型值 |
|:----|:-------|
| 单 GPU 日均故障率(AFR) | 1%~5% |
| 2,048 GPU 集群单日故障概率 | > 99.9%(几乎必然发生) |
| 单次训练(3周)预期故障次数 | 3~10 次 |
| 典型故障恢复耗时 | 30 分钟 ~ 4 小时 |

这意味着，一个长周期的训练任务中，遭遇多次硬件故障是完全正常的. **容灾机制不是备选方案，而是必备的基础设施**. 

### 1.2 常见故障类型

在大规模训练中，故障的表现形式多种多样：

**硬件故障**：
- GPU 显存错误(ECC Error)/ GPU 卡死(Hang)
- 节点宕机(Node Down)/ 电源故障
- 网络链路中断(Infiniband / RoCE 断连)
- 存储节点故障导致 IO 延迟

**软件故障**：
- NCCL 通信超时(Timeout)/ 初始化失败
- CUDA 驱动异常 / 内存访问越界
- Loss Spike 导致训练发散
- NaN 出现(数值不稳定)

**环境故障**：
- 机房温控异常导致 GPU 降频
- 电力波动导致节点重启
- 共享存储带宽争抢

### 1.3 故障的影响

假设无容灾机制，一次在第 10 天的故障意味着：
- 已损失的训练时间：**10 天**(无法恢复)
- 浪费的计算成本：数万到数百万美元
- 项目进度的不可预测性

**容灾机制的目标**：将故障恢复的时间成本从"天"降低到"分钟".

## 2. 容灾核心：Checkpointing(Checkpoint机制)

### 2.1 什么是Checkpoint

Checkpoint(Checkpoint)是在训练过程中**定期保存完整训练状态的快照**，包括：

```
训练状态快照 = {
  模型权重 (Model Weights): 所有参数的当前值
  优化器状态 (Optimizer State): Adam 的 m_t, v_t 等动量
  学习率调度器状态: 当前步数、学习率值
  数据加载器状态: 当前数据的偏移量
  随机数生成器状态: RNG State
  训练配置: 超参数、并行策略等
}
```

### 2.2 Checkpoint的保存策略

#### 保存频率

Checkpoint的保存频率需要在**恢复粒度**和**IO开销**之间权衡：

| 策略 | 保存间隔 | 优势 | 劣势 |
|:----|:--------|:----|:-----|
| 固定间隔 | 每 N 步(如 2,000 步) | 实现简单 | 故障后最多损失 N 步 |
| 时间间隔 | 每 T 小时(如 1 小时) | 恢复时间可预期 | 训练快时可能频密保存 |
| 自适应 | 根据 Loss 变化率动态调整 | IO 效率更优 | 实现复杂 |

**实际选择**：大多数大模型训练采用**固定步数间隔**，典型值为每 2,000~5,000 步(对于 7B 模型约 1~4 小时的训练量). 

#### 异步保存

Checkpoint的写入是一个耗时的 IO 操作——一个 70B 模型的全量Checkpoint可能达到数百 GB. 传统的同步保存会阻塞训练：

```python
#  同步保存(阻塞训练)
train_step()
save_checkpoint()  # 等待 IO 完成，GPU 空闲
train_step()       # 继续训练时可能已经等了 10 分钟

#  异步保存(不阻塞训练)
train_step()
thread = async_save_checkpoint()  # 后台线程写入
# 立刻继续训练
train_step()
train_step()
thread.join()  # 等待保存完成(几乎无感知)
```

异步Checkpoint(Async Checkpoint)是目前主流方案——将保存操作卸载到 CPU 后台线程，训练进程可以立刻继续. 保存完成后，再等待后台线程确认写入完成. 

### 2.3 Checkpoint的存储格式

| 格式 | 特点 | 使用场景 |
|:----|:----|:--------|
| **PyTorch `state_dict`** | 单文件，简单 | 小模型实验 |
| **分片Checkpoint(Sharded)** | 每个 rank 保存自己的权重分片 | 大规模分布式训练 |
| **Megatron-DeepSpeed 格式** | 跨 rank 分片 + 优化器状态分离 | 3D 并行训练 |
| **PyTorch DCP(Distributed Checkpoint)** | 分布式感知的异步Checkpoint | PyTorch 原生方案 |

对于 3D 并行训练(TP+PP+DP)，Checkpoint需要按并行策略分片保存. Megatron-DeepSpeed 的Checkpoint格式按 **TP 组** 分片：

```
checkpoints/
├── tp_rank_00_pp_rank_00/
│   ├── model.pt
│   └── optimizer.pt
├── tp_rank_00_pp_rank_01/
│   ├── model.pt
│   └── optimizer.pt
├── ...
└── latest_checkpointed_iteration.txt  # 指向最新Checkpoint
```

### 2.4 恢复：从Checkpoint重建训练

从Checkpoint恢复的流程：

```
1. 重建训练环境(相同并行配置、硬件数量)
2. 加载分片Checkpoint → 恢复模型权重
3. 恢复优化器状态(Adam动量)
4. 恢复学习率调度器
5. 恢复数据偏移量
6. 验证恢复后的 Loss → 确认恢复成功
7. 继续训练
```

**关键考量**：恢复后的训练 Loss 应与保存时的 Loss 保持一致(或非常接近). 如果 Loss 出现跳变，说明Checkpoint的某些状态恢复有误. 

### 2.5 Checkpoint的成本

保存一个 70B 模型的Checkpoint(FP32 优化器状态)：
- 模型权重(BF16)：70B × 2 bytes = **140 GB**
- 优化器状态(FP32 m_t + v_t)：70B × 2 × 4 bytes = **560 GB**
- 合计：**约 700 GB**
- 保存时间(NVMe × 16 并行写入)：约 **3~10 分钟**
- 存储成本：保留最近 5~10 个Checkpoint → 约 **3.5~7 TB**

由于Checkpoint的巨大存储开销，实践中通常只保留最近 3~5 个Checkpoint，并保留每次故障前的Checkpoint用于根因分析.

## 3. 训练监控(Monitoring)

### 3.1 核心监控指标

在大规模训练中，需要实时监控一系列指标，以便在问题发生的第一时间响应. 

#### 训练损失(Training Loss)

训练损失是最核心的指标. 一个健康的训练过程表现为 Loss 平稳下降. 

**异常情况**：

```
正常:  Loss = 3.2 → 2.8 → 2.5 → 2.3 → 2.1(平滑下降)
异常1: Loss = 3.2 → 2.8 → NaN(数值溢出 → 训练崩溃)
异常2: Loss = 3.2 → 2.8 → 15.7 → 3.5(Loss Spike)
异常3: Loss = 3.2 → 3.1 → 3.2 → 3.1(Loss 不下降 → 学习率可能过小)
```

**处理策略**：
- **NaN 出现**：立即暂停，从上一个Checkpoint恢复，降低学习率后重试
- **Loss Spike**：暂停训练，分析数据批次是否存在异常，检查梯度范数
- **Loss 不下降**：检查学习率调度器状态、数据质量、模型初始化

#### 梯度范数(Gradient Norm)

梯度范数反映了模型参数更新的幅度：

$$
||g||_2 = \sqrt{\sum_{i} g_i^2} \tag{1}
$$

- **梯度范数过大**($> 10$)：梯度爆炸，可能导致 Loss Spike 或 NaN
- **梯度范数过小**($< 10^{-6}$)：梯度消失，模型可能停止学习
- **正常范围**：大模型训练的梯度范数通常在 $0.1 \sim 10$ 之间

梯度范数的监控比 Loss 更敏感——在 Loss 出现明显异常之前，梯度范数的异常就已经能反映出问题. 

#### 计算吞吐量(Throughput)

吞吐量衡量训练效率：

$$
Throughput = \frac{\text{Tokens per Step} \times \text{Steps}}{\text{Time}} \tag{2}
$$

通常用 **TFLOPS** 或 **TGS(Tokens per GPU per Second)** 衡量. 吞吐量的突然下降可能意味着：

| 现象 | 可能原因 |
|:----|:--------|
| 吞吐量缓慢下降 | 数据加载瓶颈、NCCL 通信效率下降 |
| 吞吐量阶梯式下降 | 某张 GPU 降频/掉卡(显存 ECC 错误导致) |
| 吞吐量周期性抖动 | Checkpoint保存时的 IO 争抢 |

#### GPU 状态

| 指标 | 正常范围 | 告警阈值 |
|:----|:--------|:--------|
| GPU 温度 | $< 85^\circ C$ | $> 90^\circ C$(降频风险) |
| GPU 功耗 | 额定功耗的 90%~100% | $< 80\%$(可能掉卡) |
| 显存使用率 | 90%~98% | $> 99\%$(OOM 风险)或 $< 70\%$(利用率不足) |
| PCIe / NVLink 带宽 | 理论带宽的 80%~100% | $< 50\%$(通信瓶颈) |

#### 网络吞吐量

大规模训练中，网络是常被忽视但极其关键的瓶颈. 需要监控：

- **AllReduce 带宽**：衡量梯度同步的效率
- **通信计算比**：通信时间 / 计算时间，理想值 $< 0.1$
- **NCCL 超时次数**：非零即告警

### 3.2 监控工具链

| 工具 | 用途 | 特点 |
|:----|:----|:-----|
| **WandB** | 实验跟踪 + 实时仪表盘 | 云服务，易用性强 |
| **TensorBoard** | 本地监控 | 简单，但功能有限 |
| **Prometheus + Grafana** | 集群级监控 | 大规模集群标准方案 |
| **NVIDIA DCGM** | GPU 硬件监控 | 温度、功耗、ECC 错误 |
| **Slurm** | 集群调度 + 日志 | 故障自动感知和重启 |
| **自研监控系统** | 端到端训练监控 | Google / Meta / DeepSeek 使用 |

### 3.3 自动故障检测与恢复

大规模训练平台(如 Google Pathways、Meta FBLearner)都实现了**自动故障检测与恢复**：

```python
# 自动故障恢复伪代码
while not converged:
    try:
        train_loop()  # 正常训练
    except HardwareFailure as e:
        logger.error(f"硬件故障: {e.gpu_id} 掉卡")
        # 自动步骤：
        # 1. 移除故障 GPU/节点
        # 2. 调整并行配置(TP/PP/DP)
        # 3. 从最新Checkpoint恢复
        # 4. 继续训练(不人工干预)
        recover_from_checkpoint(
            checkpoint_path="latest",
            exclude_gpu=e.gpu_id
        )
```

Meta 的 "Training Resilience System" 实现了**自动容灾置换**——当检测到 GPU 异常时，自动调度备用节点加入训练，将故障影响时间压缩到 10 分钟以内.

## 4. 实际运行中的最佳实践

### 4.1 Checkpoint保存的最佳实践

1. **每 N 步保存 + 异步写入**：避免 IO 阻塞影响训练进度
2. **保留多个版本**：保留最近 5 个Checkpoint，防止"坏Checkpoint"覆盖好Checkpoint
3. **校验Checkpoint完整性**：保存后立即读取校验 SHA256，避免静默损坏
4. **保存到共享存储**：所有节点都能访问，便于故障恢复
5. **原子重命名**：写入临时文件，完成后再重命名，避免读取到不完整的Checkpoint

### 4.2 监控告警阈值设置

| 告警级别 | 示例条件 | 响应方式 |
|:--------|:--------|:--------|
| **Info** | Loss 下降速度偏移 $> 5\%$ | 通知，观察 |
| **Warning** | 梯度范数 $> 50$ | 通知，建议手动检查 |
| **Critical** | Loss = NaN / 吞吐量下降 $> 30\%$ | **自动暂停训练** + 紧急通知 |
| **Resume** | 恢复成功 | 通知恢复完成 |

### 4.3 训练中断后的恢复策略

当故障发生并恢复后：

1. **确认恢复点**：验证加载的Checkpoint是最新的有效Checkpoint
2. **验证 Loss 恢复**：恢复后的 Loss 应与保存时的 Loss 一致(偏差 $< 0.01$)
3. **降低学习率**(可选)：如果 Loss Spike 是因为恢复时的 batch 顺序变化导致，可暂时降低学习率
4. **调整数据序随机种子**：恢复的数据偏移量需保证不会重复过多 Token

## 5. 业界案例

### 5.1 LLaMA 2 的训练容灾

Meta 在 LLaMA 2 训练中采用了以下容灾策略：
- 每 2,000 步自动保存异步Checkpoint
- 使用 Elastic Training(弹性训练)框架，自动检测并剔除故障节点
- 训练过程中，平均每 3 天遭遇一次 GPU 掉卡，恢复时间 $< 15$ 分钟
- **总有效率**：约 95%(5% 的时间用于故障恢复)

### 5.2 GPT-4 的监控体系

OpenAI 的监控体系以**自动告警**为核心：
- 30+ 个实时监控指标
- 多层告警：Warning → Critical → **自动暂停**
- 恢复后自动运行「健康检查」：验证前一阶段的训练质量
- 每周自动生成训练健康报告

### 5.3 DeepSeek 的故障处理

DeepSeek 在训练 DeepSeek-V2 时采用了**3D 并行 + 弹性训练**的组合方案：
- 使用 Megatron-DeepSpeed 的分片Checkpoint
- 自动检测 NCCL 通信超时，触发恢复流程
- 单次故障恢复平均耗时 $< 10$ 分钟
- 训练有效率达到 **> 98%**

## 6. 总结

大规模预训练的容灾与监控，本质上是在不确定性中建立确定性——通过自动化的Checkpoint机制将故障损失降到最低，通过全方位的实时监控确保第一时间发现并响应问题. 

| 维度 | 要点 | 关键实践 |
|:----|:----|:--------|
| Checkpoint保存 | 定期保存完整训练状态 | 异步保存 + 分片格式 |
| 故障恢复 | 从Checkpoint重建训练 | 自动恢复流程 |
| 实时监控 | Loss / 梯度 / 吞吐量 / 硬件 | Prometheus + Grafana |
| 告警响应 | 分级告警 + 自动暂停 | Critical 级自动暂停 |
| 弹性训练 | 自动剔除故障节点 | Elastic Training 框架 |

这些工程实践是大规模训练从"实验"走向"工业化"的基石. 理解并实现这些机制，是成功完成大规模模型训练的必备能力.
