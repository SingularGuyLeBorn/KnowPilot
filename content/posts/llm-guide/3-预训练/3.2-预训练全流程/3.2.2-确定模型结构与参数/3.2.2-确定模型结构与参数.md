# 3.2.2 确定模型结构与参数

## 1. 决策要素：三个维度的权衡

### 1.1 模型性能

模型性能是首要目标. 更大的模型通常具有更强的表达能力，在相同数据量下可以取得更好的下游任务表现. 但性能的提升并非线性——随着模型规模增大，边际收益递减. 

### 1.2 训练成本

训练成本由模型参数量和训练数据量共同决定. 根据 Chinchilla Scaling Law，**最优的训练策略是在模型参数量和训练 token 数之间保持约 20:1 的比例**(即 1B 参数对应约 20B tokens). 超出这个比例会导致计算资源的浪费——要么模型欠拟合(数据太多)，要么模型过拟合(数据太少). 

### 1.3 推理成本

推理成本在模型部署阶段至关重要. 对于在线服务场景，推理延迟和吞吐量直接影响用户体验和运营成本. 通常，**同参数量下，更深更窄的模型推理效率更高**，但训练难度也更大. 此外，KV Cache 的大小与序列长度和注意力头数直接相关，决定了单次推理可以支持的最大上下文长度. 

---

## 2. 核心超参数详解

### 2.1 模型层数

`n_layers`(或 `num_hidden_layers`)决定了 Transformer 网络的**深度**. 层数越多，模型的表达层次越丰富，能够捕捉更抽象的特征. 

**典型取值与趋势**：

| 模型规模 | 典型层数 | 代表模型 |
|:--------|:--------|:--------|
| 小规模 (~1B) | 12~24 | GPT-2 (12层) |
| 中等规模 (~7B) | 28~32 | Llama 2 7B (32层) |
| 大规模 (~70B) | 60~80 | Llama 3 70B (80层) |
| 超大规模 (~200B+) | 80~120 | GPT-4 推测约 120层 |

**设计原则**：深度增加带来的收益在 60-80 层后显著递减，同时训练稳定性(梯度消失/爆炸)问题加剧. 现代模型通过 **Pre-Norm**(将 LayerNorm 放在残差连接之前)和 **Flash Attention** 等技术缓解了这些问题，使得 80 层以上的深度成为可能. 

### 2.2 隐藏层维度

`hidden_size`(或 `d_model`)决定了 Transformer 的**宽度**，即每个 token 被编码为向量的维度. 宽度决定了模型"一次能记住多少信息". 

**典型取值**：

| 模型规模 | hidden_size | 代表模型 |
|:--------|:-----------|:--------|
| ~1B | 2,048 | GPT-2 |
| ~7B | 4,096 | Llama 2 7B |
| ~70B | 8,192 | Llama 3 70B |
| ~200B+ | 12,288~16,384 | GPT-4 推测 |

**设计权衡**：
- **更宽** → 单层表达能力更强，但参数量平方级增长，训练和推理成本激增
- **更深但较窄** → 参数效率更高(相同参数量下性能更优)，但并行度受限

### 2.3 注意力头数

`n_heads` 控制多头注意力机制中并行的注意力头数量. 每个头可以关注输入序列的不同位置和语义关系. 

**典型设计模式**：

大多数模型遵循 **hidden_size / n_heads = head_dim**，其中 `head_dim` 通常为 64 或 128. 

| 模型 | hidden_size | n_heads | head_dim |
|:----|:-----------|:-------:|:--------:|
| Llama 2 7B | 4,096 | 32 | 128 |
| Llama 3 8B | 4,096 | 32 | 128 |
| Llama 3 70B | 8,192 | 64 | 128 |
| GPT-3 175B | 12,288 | 96 | 128 |

**多头注意力的优势**：
- 每个头可以学习不同类型的注意力模式(句法关系、语义关联、位置信息等)
- 多头的输出拼接后经过线性投影，相当于在不同子空间中进行信息融合
- **Grouped Query Attention (GQA)** ：在 Llama 2/3 中广泛应用，使用更少的 KV 头(如 8 个)来减少 KV Cache，提升推理效率，同时对模型质量影响极小

### 2.4 FFN 中间层大小

`intermediate_size` 是 Feed-Forward Network(前馈网络)中隐藏层的维度. FFN 由两层线性变换组成：`hidden_size → intermediate_size → hidden_size`. 

**行业惯例**：`intermediate_size` 通常设置为 `hidden_size` 的 **3.5~4 倍**(即 `ffn_ratio = 3.5~4`). 

| 模型 | hidden_size | intermediate_size | ffn_ratio |
|:----|:-----------|:----------------:|:---------:|
| Llama 2 7B | 4,096 | 11,008 | 2.69 |
| Llama 3 8B | 4,096 | 14,336 | 3.5 |
| Llama 3 70B | 8,192 | 28,672 | 3.5 |
| GPT-3 175B | 12,288 | 49,152 | 4.0 |

**SwiGLU 激活函数**：现代模型(自 Llama 以来)普遍使用 SwiGLU 替代传统的 ReLU/GELU. SwiGLU 需要三个权重矩阵(两个 gate 矩阵和一个 value 矩阵)，因此 `intermediate_size` 的定义有所不同——实际的 FFN 参数量约为传统设计的 1.5 倍，但质量收益显著. 

### 2.5 其他关键参数

| 参数 | 说明 | 典型值 |
|:----|:----|:------|
| `max_position_embeddings` | 最大序列长度 | 4K~128K (Llama 3: 8K) |
| `vocab_size` | 词表大小 | 32K~128K (Llama 3: 128K) |
| `num_key_value_heads` | KV 头数(GQA 用) | 8 (Llama 3 70B) |
| `rope_theta` | RoPE 底数(控制外推能力) | 500,000 (Llama 3) |
| `hidden_dropout_prob` | Dropout 概率(预训练通常为 0) | 0.0 预训练 / 0.1 微调 |

## 3. 案例分析：Llama 3 系列

以 Meta 的 Llama 3 系列为例，可以清晰看到不同规模模型的结构设计差异：

| 参数 | Llama 3 8B | Llama 3 70B | 设计考量 |
|:----|:----------:|:-----------:|:--------|
| n_layers | 32 | 80 | 70B 更深的网络捕获更复杂特征 |
| hidden_size | 4,096 | 8,192 | 宽度随规模翻倍 |
| n_heads | 32 | 64 | 头数与宽度同比扩展 |
| n_kv_heads | 8 | 8 | GQA：KV 头数固定为 8，大幅节约推理显存 |
| intermediate_size | 14,336 | 28,672 | 保持约 3.5 倍扩展比 |
| vocab_size | 128,256 | 128,256 | 统一词表大小 |
| max_seq_len | 8,192 | 8,192 | RoPE 支持长度外推 |

**观察要点**：

1. **GQA 策略**：8B 使用 32 个 Q 头对 8 个 KV 头(4:1 压缩)，70B 使用 64 个 Q 头对 8 个 KV 头(8:1 压缩). KV 头数固定为 8 意味着 KV Cache 大小主要取决于序列长度而非模型宽度——这是 Llama 3 能够高效支持长上下文的关键设计. 

2. **深度 vs 宽度**：从 8B 到 70B，深度从 32 层增加到 80 层(2.5 倍)，宽度从 4,096 到 8,192(2 倍). 深度增长快于宽度，说明在更大规模下，**加深比加宽更有利于参数效率**. 

3. **词表统一**：8B 和 70B 使用相同的 128K tokenizer，这意味着两者的 embedding 层参数量相同(约 5.25 亿)，在大模型中占比很小.

## 4. Scaling Law 的指导作用

模型结构参数的选择并非简单的经验法则，而是遵循 Scaling Law 的科学指导. Scaling Law 揭示了三个关键变量之间的幂律关系：

$$
L(N, D) = \frac{A}{N^\alpha} + \frac{B}{D^\beta} + L_{\infty} \tag{1}
$$
其中：
- $N$ = 模型参数量
- $D$ = 训练 token 数
- $L_{\infty}$ = 不可约误差(数据本身的熵)
- $\alpha, \beta$ 是幂律指数，通常 $\alpha \approx 0.076, \beta \approx 0.103$

**实际应用步骤**：

1. **确定计算预算 $C$**：根据 GPU 数量、训练时长和硬件利用率，计算总 FLOPs
2. **寻找最优分配**：在 $C \approx 6ND$ 的约束下，找到使 $L(N,D)$ 最小的 $(N, D)$ 组合
3. **确定深度/宽度比**：给定 $N$，还需在深度和宽度之间分配. 研究表明，在固定参数量下，**适当增加深度(减少宽度)通常能获得更好的性能**，但过度加深会带来训练不稳定问题

> **注意**：Chinchilla 法则(20 tokens / 参数)是最优训练策略的起点，但实际部署时还需考虑推理成本——如果推理场景很重要，可以适当增加模型参数量(用更少的数据训练到次优性能)，以换取更快的推理速度.

## 5. 实际设计流程

在实际工程中，确定模型结构与参数的流程通常如下：

### 5.1 定义场景约束
- **部署场景**：在线 API 还是离线批量？目标延迟是多少？

- **计算预算**：有多少张 GPU？可用训练时间多长？

- **数据储备**：有多少高质量训练数据？

### 5.2 规模估算
- 根据预算和数据量，用 Scaling Law 估算最优模型参数量
- 参考同规模开源模型的 benchmark，设定性能目标

### 5.3 结构选择
- **标准 Decoder-only**：大多数场景的首选(如 Llama、GPT 系列)
- **MoE(Mixture of Experts)** ：在参数量极大且推理成本敏感时采用(如 Mixtral 8x7B、DeepSeek-V2)
- **MLA(Multi-head Latent Attention)** ：DeepSeek 提出的高效注意力变体，进一步降低 KV Cache

### 5.4 超参数调优
- 在小规模(~1B)实验上搜索最优的深度/宽度比
- 使用学习率 warmup + cosine schedule
- 验证训练稳定性(loss 是否收敛、梯度范数是否异常)

### 5.5 小规模验证
- 在 1B~3B 规模的实验模型上验证设计
- 确认 Scaling Law 的预测与小规模实验结果一致
- 如果偏差过大，调整假设并重新设计

---

## 6. 总结

确定模型结构与参数是预训练流程中承上启下的关键环节：

- **承上**：它承接了 Tokenizer 的输出(词表大小影响 embedding 层设计)和训练数据的规模(影响最优参数量)
- **启下**：它决定了后续训练框架的选择(模型并行策略、张量并行度)、训练稳定性策略，以及最终部署时的推理效率

核心原则可以概括为一个公式：

> **最优模型 = f(计算预算, 数据规模, 推理约束)** 

在有限的资源下，找到三个变量的最优平衡点，就是模型结构设计的艺术与科学.
