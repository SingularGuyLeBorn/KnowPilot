# 3.2.1 训练 Tokenizer：从字符坍缩到字节宇宙的工业级演进

将连续的人类自然语言流离散化为神经网络可处理的整数张量，是大型语言模型(LLM)训练管线中的第一道工序. 这道工序的设计直接决定了模型的词表空间效率、有效序列长度上限、多语言泛化能力以及数值计算的准确度. 在当前大模型架构中，Tokenizer 已经超越了单纯的“分词工具”，演变为一种深刻影响模型 Scaling 轨迹的底层协议. 

## 1. 离散化博弈：信息熵、显存与算力的三角平衡

Tokenizer 的设计本质上是一个多维约束下的最优化问题. 开发者必须在词表规模(Vocabulary Size)、压缩比(Compression Ratio)与模型参数量之间寻找物理平衡点. 

### 1.1 词表规模 (V) 与参数矩阵的代价
在 Transformer 架构中，词表大小 $V$ 直接锁定两个核心参数矩阵的维度：输入嵌入层(Input Embedding, $V \times d_{model}$)与输出投影层(Output Projection/LM Head, $d_{model} \times V$). 

以 LLaMA 3 为例，其词表扩容至 128,256. 假设隐藏层维度 $d_{model} = 8192$：
- **存储开销**：单层 Embedding 参数量约 $128,256 \times 8192 \approx 1.05 \text{B}$ 参数. 在 bfloat16 精度下，仅 Embedding 矩阵即占据 2.1 GB 显存. 由于输入输出层通常不共享参数(Untied Weights)，总显存占用达到 4.2 GB 以上. 
- **计算开销**：输出层在计算全词表 Logits 时，必须进行一次巨大的矩阵乘法. 词表越大，该步的内存带宽(Memory Bandwidth)需求越高，直接限制了推理时的 Token 生成吞吐量(Tokens per second). 

### 1.2 有效序列长度 (L) 与注意力瓶颈
词表越小(如字符级 Tokenizer)，单段文本产生的序列越长. 
- **复杂度代价**：自注意力机制的计算复杂度与序列长度平方成正比($O(L^2)$). 
- **信息承载极限**：如果切分过细，同样的上下文窗口(如 8K)实际承载的信息量会大幅缩水. 例如，若一个汉字被切分为 3 个字节 Token，8K 的窗口实际上只能处理约 2.7K 汉字，导致模型长文本理解能力下降. 

### 1.3 工业标准参数对照表 (Tokenizer Scaling)

| 特性 | 字符级 (Character-level) | 常用子词 (Subword, 32K) | 深度子词 (Subword, 128K) | 词级 (Word-level) |
| :--- | :--- | :--- | :--- | :--- |
| **词表大小 (V)** | 极小 (几百) | 中等 (32K) | 较大 (128K+) | 极大 (1M+) |
| **压缩率 (Tokens/Words)** | 极低 (~4.5) | 高 (~1.2 - 1.5) | 极高 (~0.8 - 1.0) | 1.0 (Exact) |
| **OOV 风险** | 零 | 低 | 极低 | 极高 |
| **推理延迟** | 极高 (序列太长) | 均衡 | 低 (序列短) | 极高 (Softmax 慢) |
| **代表模型** | Early RNNs | LLaMA 1/2, BERT | LLaMA 3, GPT-4 | FastText |

## 2. 演进脉络：从贪心合并到字节 Fallback

子词(Subword)分词器的兴起是为了解决词级(Word-based)分词的“词表爆炸”和字符级(Character-based)分词的“语义坍缩”. 

### 2.1 BPE：基于频率的贪心聚合动力学
字节对编码(Byte-Pair Encoding)是目前应用最广的分词策略. 它不依赖语言学知识，完全基于语料库的统计频率. 

**算法执行流**：
1. **初始化**：将语料库中的所有文本拆分为最基础的 Unicode 字符流. 
2. **频次统计**：扫描语料，统计所有相邻 Token 对的共现频率. 
3. **最高频合并**：找到频率最高的对 $(t_a, t_b)$，将其合并为新 Token $t_{new}$ 并加入词表. 
4. **全局替换**：在语料库中递归执行替换，直到词表大小达到预设阈值. 

**BPE 训练复杂度优化**：
原始 BPE 训练的时间复杂度为 $O(N \cdot M)$($N$ 为语料长度，$M$ 为合并次数). 工业级实现(如 HuggingFace Tokenizers)通过建立**最高频堆(Max-Heap)** 和**双向链表**，实现了单步合并复杂度降至 $O(\log N)$，使得在数 TB 语料上训练 Tokenizer 成为可能. 

### 2.2 BBPE：彻底消除 OOV 的物理基础
传统的 BPE 仍然以字符为单位. 如果初始词表没有包含某种冷门 Unicode 字符(如某个生僻 Emoji)，模型依然会报 OOV. 
**字节级 BPE (BBPE)** 将初始词表固定为 256(字节的所有取值). 任何 Unicode 字符最终都会被 UTF-8 编码为 1-4 个字节，从而实现了“全宇宙字符无损覆盖”. 

## 3. 工业级架构解析：SentencePiece 与 TikToken

在实际工程落地中，仅有 BPE 算法是不够的. 开发者必须解决文本预处理、正则拦截和无损还原的问题. 

### 3.1 SentencePiece：语言无关的无损流
Google 开发的 SentencePiece 是多语言模型的首选(如 T5, Llama 系列). 
- **空格特殊化**：它将空格视为普通字符(通常用 `_` 表示)，这意味着分词过程是**完全可逆的**(Lossless Detokenization). 
- **Normalization 管道**：内置了 NFKC 等 Unicode 规范化工具. 利用 Aho-Corasick 自动机(FST)在 $O(N)$ 时间内完成复杂的字符映射. 
- **Byte Fallback**：在分词时遇到词表外的罕见 Unicode，会自动退化为 1-4 个字节 Token，确保输入流不断流. 

### 3.2 TikToken：正则 Sentinel 体系
OpenAI 采用的 TikToken(GPT-4 核心)在 BPE 统计算法之前引入了一层极其强硬的**正则拦截(Regex Pre-tokenization)** . 

**GPT-4 (cl100k_base) 正则深度拆解**：
```python
r"""'(?i:[sdmt]|ll|ve|re)|[^\r\n\p{L}\p{N}]?+\p{L}+|\p{N}{1,3}| ?[^\s\p{L}\p{N}]++[\r\n]*|\s*[\r\n]|\s+(?!\S)|\s+"""
```
**正则拦截的物理意义**：
- **语义阻断**：`[^\s\p{L}\p{N}]?+\p{L}+` 强制要求标点符号和字母分别属于不同的逻辑块. BPE 在统计时绝对无法跨越正则切开的缝隙进行合并. 这防止了诸如 `apple.` 被合并为一个 Token，保持了标点符号的独立语义. 
- **数字保护 (Digit Handling)** ：`\p{N}{1,3}` 强制将长串数字(如 123456)切分为最多 3 位的子块(123, 456). 这是缓解 LLM “数字幻觉”的关键：模型不再需要为极其罕见的 10 位长数字分配独立表征，而是通过 3 位数的组合来理解数值逻辑. 

## 4. 概率分词框架：Unigram 与 EM 算法

与 BPE 的贪心“加法”不同，Unigram 采用“减法”逻辑. 

### 4.1 期望最大化 (EM) 优化流程
Unigram 假设序列中的子词 $x_i$ 相互独立. 对于语料 $D$，目标是最大化边际似然：
$$ \mathcal{L} = \sum_{x \in D} \log \left( \sum_{s \in S(x)} \prod_{t \in s} P(t) \right) \tag{1} $$

**算法步骤**：
1. **初始化**：生成规模极大的种子词表(如 500K)，并赋予初始概率. 
2. **E步**：固定概率，利用 Viterbi 算法寻找每句语料的最优切分路径. 
3. **M步**：统计 E 步中的词频，利用最大似然估计更新概率 $P(t)$. 
4. **剪枝**：计算移除某个子词后的信息损失(Information Loss)，移除倒数 10-20% 的项. 

### 4.2 子词正则化 (Subword Regularization)
Unigram 最大的工程价值在于其非确定性. 在预训练时，算法可以根据概率采样非最优切分(如将 "higher" 随机切分为 "high-er" 或 "hig-her"). 
- **物理直觉**：这相当于在 Token 层面做 Dropout，迫使模型理解同一语义的不同颗粒度表征，极大增强了模型对拼写错误和文本噪音的防御能力. 

## 5. Tokenizer 的 Scaling Law：LLaMA 3 的 128K 抉择

LLaMA 3 将词表从 32K 扩容至 128K，是基于对“训练效率 vs 推理延迟”的极限权衡. 

| 评估维度 | 32K 方案 (LLaMA 1/2) | 128K 方案 (LLaMA 3) | 物理代价与收益 |
| :--- | :--- | :--- | :--- |
| **Token 序列长度** | 1.0x (Baseline) | ~0.7x | 同样的上下文窗口可承载 1.4x 以上的信息量 |
| **Embedding 参数** | 262M (at 8192) | 1.05B (at 8192) | 显存压力增加，但换取了更高的推理吞吐 |
| **多语言对齐** | 极差 (中文 Fertilty > 2) | 优秀 (中文 Fertility < 1) | 极大提升了非英语语种的编码效率 |
| **梯度稀疏性** | 低 (更新频繁) | 高 (部分 Token 极少出现) | 需要通过 Upsampling 保证尾部 Token 的 Embedding 质量 |

## 6. 失效模式分析 (FMEA - Tokenizer Edition)

在生产环境中，Tokenizer 的细微设计缺陷往往是模型“降智”的首要根源. 

| 失效类型 | 诱发原因 | 物理表现 | 工业级解决方案 |
| :--- | :--- | :--- | :--- |
| **OOV 语义坍缩** | 预训练词表未包含特定 Unicode | 输出大量的 [UNK] 或系统崩溃 | 强制采用 BBPE 字节回退机制 |
| **数字计算幻觉** | 随机长度的数字合并 | 模型无法理解位置和进位逻辑 | Regex 预切割，强制数字按固定步长拆分 |
| **空格敏感度爆炸** | 未处理连续空格(代码缩进) | Token 数随缩进深度增加，浪费有效窗口 | 使用 Meta-symbol 或 TikToken 专用合并规则 |
| **尾部 Token 随机化** | 词表过大导致部分 Token 未被更新 | 遇到冷门词时，输出毫无关联的乱码 | 数据重采样(Up-sampling)提高词表覆盖率 |
| **首尾粘连** | 缺乏单词边界标识 | `apple` 和 ` apple` 对应完全不同的向量 | SentencePiece 强制统一前缀空格映射 |

## 7. 评估体系：衡量 Tokenizer 质量的标尺

如何判定一个 Tokenizer 是否“优秀”？工业界采用以下量化指标：
1. **Fertility (丰饶度)** ：每个单词被切分成的 Token 平均数. 英语通常在 1.1-1.2 之间. 
2. **Parity (对等性)** ：不同语言之间 Fertility 的标准差. 越小说明多语言对齐越公平，避免了对非英语语种的“计算歧视”. 
3. **PPL (Perplexity) 归一化**：比较不同 Tokenizer 训练出的模型时，必须使用 BPC(Bits Per Character)而非 PPL，因为词表大小不同会导致 PPL 失去数学可比性. 

## 8. 未来趋势：从离散符号到连续表征

尽管当前 BPE 统治了工业界，但其局限性(如对拼写错误的脆弱性、跨语言不平衡)正推动技术向以下方向演进：
- **Late-tokenization**：在 Embedding 层之后进行部分合并，利用模型内部状态指导分词. 
- **Model-based Tokenization**：如 Google 的 Magma 尝试直接从像素或原始字节流中提取特征，绕过离散词表. 
- **VLM 统一 Token 空间**：多模态模型(如 GPT-4o)正在尝试将视觉、音频与文本统一到一个共享的、极其巨大的 Token 宇宙中. 

Tokenizer 不是文本预处理，而是模型对现实世界的分辨率设定. 128K 词表是 LLaMA 3 走向多语言巅峰的门票，也是对算力资源进行的一次高精度“空间换时间”的豪赌. 在这个过程中，理解并优化 Tokenizer，就是在从物理最底层构建模型的智能上限.
