# 3.3.4 特定能力评估

## 2. 为何要专门评估特定能力

### 2.1 通用基准的局限性

通用基准测试(如 MMLU)通常采用**选择题**形式，模型只需从四个选项中选出正确答案. 这种评估方式存在三个根本性缺陷：

| 缺陷 | 说明 | 示例 |
|:----|:-----|:-----|
| **选项提示效应** | 模型可以通过排除法或选项间的模式匹配"猜对"答案 | 物理题不会算，但选项 C 看起来更专业 |
| **缺乏过程评估** | 只关注最终答案，不关注推理过程的正确性 | 答案对了，但推导过程完全是错的 |
| **深度不足** | 每道题仅需单步或两步推理，无法评估多步推理能力 | 知识问答型 → 不测试真正的逻辑链条 |

### 2.2 代码与数学：智力核心的"试金石"

代码和数学评估之所以被视为"试金石"，是因为它们具有三个独特属性：

**属性一：确定性答案**
代码必须通过单元测试才能算正确，数学题有明确的正确答案. 这种"对就是对，错就是错"的二元评估，消除了主观性. 

**属性二：多步推理链**
无论是编写一个排序算法还是解一道微积分题，都需要模型生成一个**完整的、可验证的推理链条**，而不是单步猜测. 

**属性三：类比迁移**
特定能力的评估能反映模型在未见任务上的泛化能力. 例如，模型在代码生成上的表现，与它在其他需要结构化思维的任务(如 SQL 编写、正则表达式构建)上的表现高度相关. 

### 2.3 从"文科生"到"理科生"的跨越

一个形象的类比是：通用基准评估的是模型的"文科生"能力(知识广度、阅读理解)，而代码和数学评估衡量的是模型的"理科生"能力(逻辑推理、系统性思维). 从 GPT-3 到 GPT-4，再到 Claude 3、DeepSeek-R1，模型在代码和数学上的进步速度远快于通用基准——这正是 LLM 从"知识型"向"推理型"演进的关键标志.

## 3. 代码能力评估

### 3.1 HumanEval 基准

HumanEval 由 OpenAI 于 2021 年发布，是评估代码生成能力最具影响力的基准之一. 

**核心设计：**
HumanEval 包含 **164 道手写编程题**，每道题包含：
- **函数签名**(Function Signature)：如 `def fib(n):`
- **文档字符串**(Docstring)：描述函数的功能
- **单元测试**(Unit Tests)：一组用于验证代码正确性的测试用例

**示例：**

```python
def fib(n: int) -> int:
    """Return the n-th Fibonacci number.
    
    >>> fib(1)
    1
    >>> fib(5)
    5
    >>> fib(8)
    21
    """
```

模型的任务是生成能够通过所有单元测试的函数体代码. 

**设计精妙之处：**
- 所有题目都是**原创**的，不在常见训练数据中，降低数据泄露风险
- 语言统一使用 **Python**，消除语言偏好影响
- 测试用例覆盖**边界条件**(空输入、极端值)，防止"表面正确"的实现

### 3.2 MBPP 基准

MBPP(Mostly Basic Python Programming)由 Google 于 2021 年发布，包含约 **1,000 道编程题**，覆盖更广泛的基础编程场景. 

**HumanEval vs MBPP 对比：**

| 维度 | HumanEval | MBPP |
|:----|:---------|:-----|
| 题目数量 | 164 | ~1,000 |
| 难度 | 中等~困难 | 简单~中等 |
| 题目来源 | 手写原创 | 众包收集 |
| 平均代码行数 | 8-15 | 4-8 |
| 测试用例 | 每道题 5-10 个 | 每道题 3 个 |
| 侧重 | 算法能力 | 基础编程能力 |

两者互为补充：HumanEval 衡量模型的算法实现能力，MBPP 衡量模型的基础编程熟练度. 

### 3.3 pass@k 评估指标

pass@k 是代码生成领域最核心的评估指标，由 OpenAI 在 HumanEval 论文中正式引入. 

**定义：**

$$
\text{pass@k} = 1 - \frac{\binom{n - c}{k}}{\binom{n}{k}} \tag{1}
$$
此式描述了变量之间的定量关系，其中每一项对应特定的物理操作. 
在这个组合公公公式子中，$n$ 是生成的样本总数(通常 $n = 200$)，$k$ 是允许尝试的次数(通常取 $k = 1, 10, 100$)，$c$ 是通过测试的样本数. 

**直观理解：**
- **pass@1**：模型一次生成就正确的概率 — 衡量单次生成质量
- **pass@10**：模型生成 10 次，只要有一次通过就算成功 — 衡量生成多样性
- **pass@100**：几乎允许无限尝试 — 衡量模型代码能力上限

**实际应用中的选择：**

| 场景 | 推荐指标 | 原因 |
|:----|:--------|:-----|
| 代码补全(实时 IDE) | pass@1 | 用户期望即时、正确的补全 |
| 代码生成(Copilot 类) | pass@10 | 可接受从多个候选中选择 |
| 竞赛/复杂算法 | pass@100 | 追求最优解 |

### 3.4 进阶代码评估方法

除了 HumanEval 和 MBPP，业界还发展出更复杂的代码评估方法：

**3.4.1 代码调试(Code Repair)** 
给定一段**有缺陷的代码**，要求模型定位并修复错误. 这比凭空生成代码更具挑战性，因为它需要模型先理解错误原因，再生成修复方案. 

**3.4.2 代码解释(Code Explanation)** 
给定一段代码，要求模型用自然语言解释其功能和逻辑. 这评估的是模型的**代码理解能力**，而非生成能力. 

**3.4.3 多语言代码生成**
现代评估常覆盖多种编程语言，而不仅是 Python：

| 语言 | 代表基准 | 特点 |
|:----|:--------|:-----|
| Python | HumanEval、MBPP | 最多，数据最丰富 |
| JavaScript | JSNet | 前端代码生成 |
| Java | CodeXGLUE(Java) | 企业级应用 |
| C++ | APPS(C++) | 算法竞赛 |
| Rust | RustEval | 系统编程 |

**3.4.4 代码补全(Fill-in-the-Middle, FIM)** 
FIM 任务要求模型在给定前后文的情况下，补全中间缺失的代码段. 这更贴近实际 IDE 的代码补全场景：

```python
def sort_array(arr):
    # [前半部分]
    # [FIM — 模型需要补全的内容]
    # [后半部分]
    return sorted_arr
```

### 3.5 模型代码能力对比

| 模型 | HumanEval (pass@1) | HumanEval (pass@10) | MBPP (pass@1) |
|:----|:-----------------:|:------------------:|:-------------:|
| GPT-3 (175B) | 42.1% | — | 61.4% |
| Codex (12B) | 47.3% | 72.4% | 65.8% |
| GPT-4 | **87.2%** | 95.6% | **81.7%** |
| Claude 3 Opus | 84.9% | 93.2% | 79.3% |
| DeepSeek-Coder (33B) | 82.4% | 91.8% | 78.5% |
| DeepSeek-R1 | 93.2% | — | 85.7% |
| LLaMA 3 70B | 73.5% | 86.8% | 71.2% |

> 从 GPT-3 到 GPT-4，HumanEval pass@1 从 42.1% 提升到 87.2%，翻了一倍以上. DeepSeek-R1 更以 93.2% 的成绩超越了 GPT-4，表明**推理增强**对代码能力的显著提升效果.

## 4. 数学能力评估

### 4.1 GSM8K 基准

GSM8K(Grade School Math 8K)由 OpenAI 于 2021 年发布，包含 **8,500 道小学数学应用题**(7,500 训练 + 1,000 测试). 

**题目特点：**
- 难度：小学水平(Grade 1-8)
- 题型：需要 2-8 步推理的数学应用题
- 语言：自然语言描述，而非数学符号

**示例：**

> Beth 的花园里有 4 行玉米，每行有 6 棵玉米. 她每天从每棵玉米上摘 1 根玉米棒. 她还需要 36 根玉米棒来做一顿大餐. 那么她需要连续摘多少天才能有足够的玉米棒来做大餐？

**关键指标：**
GSM8K 的核心指标是**最终答案准确率**(Answer Only Accuracy)，即模型输出的最终数值答案是否与标准答案一致. 但更先进的评估方法也关注**推理过程质量**. 

**GSM8K 上的模型表现：**

| 模型 | 准确率 |
|:----|:-----:|
| GPT-3 (175B) | 25.3% |
| GPT-4 | **92.0%** |
| Claude 3 Opus | 88.5% |
| DeepSeek-R1 | **95.8%** |
| LLaMA 3 70B | 78.2% |
| Qwen2.5 72B | 84.3% |

### 4.2 MATH 基准

MATH 由 OpenAI 联合 2021 年发布，包含 **5,000 道数学竞赛题**(2,500 训练 + 2,500 测试)，难度远高于 GSM8K. 

**难度分级：**

| 级别 | 难度 | 示例领域 | 典型准确率 |
|:----|:----|:--------|:---------:|
| Level 1 | 最简单 | 基础算术、简单代数 | 85-95% |
| Level 2 | 简单 | 初级几何、概率基础 | 70-85% |
| Level 3 | 中等 | 代数方程、组合数学 | 50-70% |
| Level 4 | 困难 | 三角学、数论 | 25-50% |
| Level 5 | 最难 | 竞赛级综合题 | 10-25% |

**GSM8K vs MATH 对比：**

| 维度 | GSM8K | MATH |
|:----|:-----|:----|
| 题目数量 | 8,500 | 5,000 |
| 难度 | 小学级(1-8 年级) | 竞赛级(AMC/AIME) |
| 推理步数 | 2-8 步 | 5-15+ 步 |
| 公式密度 | 低 | 高(大量 LaTeX 公式) |
| 典型 GPT-4 得分 | 92% | 52% |

**MATH 上的模型表现：**

| 模型 | 总体 | Level 1 | Level 3 | Level 5 |
|:----|:---:|:-------:|:-------:|:-------:|
| GPT-3 (175B) | 6.2% | — | — | — |
| GPT-4 | **52.1%** | 82% | 55% | 22% |
| Claude 3 Opus | 47.8% | 79% | 50% | 18% |
| DeepSeek-R1 | **83.9%** | 95% | 85% | 62% |
| LLaMA 3 70B | 32.5% | 65% | 33% | 10% |

> DeepSeek-R1 在 MATH 上达到 83.9%，远超 GPT-4 的 52.1%，尤其是 Level 5 的 62% vs 22% 差距最为悬殊，说明**强化学习 + 长链推理**对数学能力的提升是质的飞跃. 

### 4.3 Chain of Thought 评估方法

CoT(思维链)在数学评估中扮演着特殊角色——它不仅是一种推理增强方法，也是一种**评估工具**. 

**4.3.1 答案正确性 vs 过程正确性**

传统的数学评估只关注**最终答案**，但这种方法存在一个严重问题：模型可能通过"巧合"得到正确答案，而推理过程是完全错误的. 

| 场景 | 最终答案 | 推理过程 | 评估结果 |
|:----|:--------|:--------|:--------|
| A |  正确 |  逻辑完整 | 真正的理解 |
| B |  正确 |  逻辑跳跃/错误 | 表面正确(False Positive) |
| C |  错误 |  思路正确但计算失误 | 可接受的错误 |
| D |  错误 |  逻辑混乱 | 完全错误 |

**4.3.2 过程奖励模型(Process Reward Model, PRM)** 

为了解决"只看答案"的问题，OpenAI 在 2023 年提出了**过程奖励模型**(PRM)，对推理过程中的**每一步**进行评分，而非仅评估最终答案. 

**PRM 的工作原理：**

```
推理步骤链：
Step 1: "设 x 为苹果的个数" → PRM 评分: 0.95 
Step 2: "根据题意，3x + 2 = 14" → PRM 评分: 0.92 
Step 3: "3x = 12" → PRM 评分: 0.97 
Step 4: "x = 4" → PRM 评分: 0.99 
Step 5: "所以苹果有 4 个" → PRM 评分: 0.95 
最终答案: "4" 
```

如果某一步出现逻辑错误或推理跳跃，PRM 的评分会显著下降：

```
Step 3: "3x = 16" → PRM 评分: 0.12 (推理错误)
```

**PRM 的训练数据：**
OpenAI 发布了 PRM800K 数据集，包含 **800K 个步骤级标注**，每个步骤标注为"正确"、"错误"或"模糊". 标注者逐步骤审查模型的推理链，提供细粒度反馈. 

**PRM 的评估意义：**

| 指标 | 评估内容 | 局限性 |
|:----|:--------|:------|
| **Answer-only** | 最终答案正确性 | 无法区分"真推理"和"巧合" |
| **PRM (过程)** | 每一步推理的正确性 | 需要大量人工标注，成本高 |
| **ORM (结果)** | 输出结果的质量(如正确性+清晰度) | 仍无法捕捉过程缺陷 |

### 4.4 数学推理的进阶评估方向

**4.4.1 多步骤推理的一致性**
一个前沿的评估方向是检查模型在**多步骤推理中的一致性**. 例如，如果模型在 Step 3 中得出 $x = 4$，但在后续计算中将 $x$ 当作 5 使用，说明模型存在"推理断层". 

**4.4.2 逆向推理测试**
给定答案，要求模型反向推导出题目. 这能有效测试模型的**数学理解深度**——一个真正理解解题过程的模型，应该能"倒推"出题目. 

**4.4.3 数学直觉测试("眨眼测试")** 
要求模型在**不写出完整推导**的情况下给出答案. 这测试的是模型的数学直觉和模式识别能力——优秀的数学家在看到题目时往往能在几秒内"感知"到答案的方向.

## 5. 代码与数学能力的综合对比

### 5.1 两大特定能力的关系

| 维度 | 代码能力 | 数学能力 |
|:----|:--------|:--------|
| **核心技能** | 语法知识、逻辑组织、API 熟悉度 | 公式推导、抽象推理、计算准确 |
| **评估方式** | 通过单元测试(自动化) | 对比标准答案/过程评分 |
| **主要挑战** | 语法正确性、边界条件覆盖 | 多步推理的准确性、公式操作 |
| **OpenAI 代表** | Codex → GPT-4 → o1 | GPT-4 → o1 → o3 |
| **中外差距** | 国内模型差距较小(DeepSeek-Coder 领先) | 差距较大(MATH 上 DeepSeek-R1 领先原始 GPT-4) |

### 5.2 模型"偏科"现象

不同模型在代码和数学上存在显著的"偏科"现象：

| 模型 | 代码能力 | 数学能力 | "偏科"特点 |
|:----|:--------|:--------|:----------|
| GPT-4 |  |  | 全面均衡 |
| Claude 3 |  |  | 代码略强于数学 |
| DeepSeek-R1 |  |  | 数学超强，代码极强 |
| Gemini 1.5 Pro |  |  | 均衡但顶尖略弱 |
| LLaMA 3 70B |  |  | 开源最优但仍有差距 |

### 5.3 评估的未来方向

**方向一：跨任务泛化**
从单一任务(如 Python 代码生成)扩展到跨语言、跨任务的泛化能力评估. 

**方向二：交互式评估**
允许模型与执行环境交互(运行代码、查看错误信息)，评估其**调试和迭代能力**. 

**方向三：推理效率**
不仅仅要"能做对"，还要评估模型在**单位 token 消耗下的推理效率**——用更少的计算资源达到同样的正确率. 

**方向四：鲁棒性测试**
通过引入噪声、冗余信息或对抗性干扰，测试模型推理的鲁棒性.

## 6. 总结

特定能力评估(代码和数学)是大模型能力评估体系中不可或缺的一环，它弥补了通用基准在深度推理和过程评估方面的不足. 

### 核心要点

**1. 代码评估的三个层次：**
- 基础层：HumanEval + MBPP(pass@1/pass@10/pass@100)
- 进阶层：代码调试、代码解释、多语言生成
- 前沿层：Fill-in-the-Middle、交互式评估

**2. 数学评估的两个维度：**
- 广度维度：GSM8K(小学级)→ MATH(竞赛级)，覆盖不同难度
- 深度维度：答案正确性 → 过程正确性(PRM)，覆盖不同评估粒度

**3. 关键能力衡量公式：**

代码能力(pass@k)：
推导过程如下：
$$
\text{pass@k} = 1 - \frac{\binom{n - c}{k}}{\binom{n}{k}} \tag{2}
$$
此关系式描述了变量之间的定量映射，为后续优化提供了数学基础. 
数学推理质量(Process Reward)：
推导过程如下：
$$
\text{PRM Score} = \frac{1}{m} \sum_{i=1}^{m} \text{PRM}_i(\text{Step}_i) \tag{3}
$$
其中 $m$ 为推理步数，$\text{PRM}_i$ 为第 $i$ 步的过程奖励评分. 

**4. 关键观察：**
- 代码和数学能力是模型"推理能力"的核心表征
- 从 GPT-3 到 DeepSeek-R1，代码能力(HumanEval pass@1)从 42% → **93%**
- 数学能力(MATH)从 6% → **84%**，进步远超通用基准
- 过程奖励模型(PRM)代表了从"答案评估"向"过程评估"的方法论演进

在衡量一个模型是否能从"文科生"向"理科生"进阶时，代码和数学评估是最可靠的"试金石". 它们不仅测量模型**知道什么**，更测量模型**如何思考**——而这，才是真正智能的核心.
