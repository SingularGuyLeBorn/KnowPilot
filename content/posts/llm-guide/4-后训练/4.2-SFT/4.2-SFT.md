# 4.2 大模型监督微调 (SFT) 深度技术指南


## 1. SFT 监督微调：核心概念与原理

### 1.1 SFT 简介：从预训练到指令遵循

SFT (Supervised Fine-Tuning) 监督微调, 是指在预训练模型 (如大语言模型) 基础上, 通过少量标注数据调整模型参数, 使其适应特定任务的技术. 它的核心思想是“迁移学习”, 即利用预训练模型已有的知识 (如语言理解能力), 通过微调快速适配新任务 (如文本分类、对话生成).

SFT 与预训练在训练方式上基本一致, 主要区别在于**数据的组成形式和训练目的**:

- **数据长度**: 预训练阶段的每条数据通常是固定长度的 (如 4K/8K tokens), 而 SFT 的每条数据则保持其原始长度.

- **Special Token 引入**: SFT 阶段会引入预训练阶段未见过的特殊 tokens (如用于标注对话角色的 user, assistant, system 等), 赋予它们全新的语义. 这些特殊 tokens 还可以用来“构造知识”, 验证 SFT 训练情况.- eos_token** 的重要性**: SFT 会让模型接触到最重要的 eos_token (结束符), 这对于模型学会何时停止生成至关重要, 因为预训练模型通常因未见过此 token 而无法停止生成.

- **语料角色划分**: 借助 special tokens, SFT 会将语料切分为不同的角色, 如 system, user, assistant, 也可根据业务需求增加“背景”、“旁白”、“事件”等.

- **Prompt Loss Mask**: SFT 通常不对 prompt 部分计算损失 (loss), 这主要是因为 prompt 的同质化程度较高. 但若能确保每条 prompt 都是独一无二的, 则可以省去 prompt 的 loss mask 环节. 需要注意的是, 对于多轮会话 (session) 数据, 必须明确是计算每一轮回答的损失, 还是只计算最后一轮回答的损失.

从训练目的来看, 预训练更侧重于“背书”, 即纯粹地学习知识; 而 SFT 则是“做题”, 学习的是**指令遵循能力**. 在 SFT 阶段应避免强行进行知识注入 (如训练 50W 条代码数据), 否则会显著降低模型的通用能力. 知识注入应采用 **continue-pretrain** 的思路进行, 且通常控制在 10% 到 20% 的比例.

### 1.2 SFT 实现原理：数学与实践

SFT 本质上仍然在做 **next token prediction** (下一个 token 预测), 但我们更希望模型关注 answer 部分的预测.

#### 1. 预训练阶段回顾

在预训练中, 模型学习预测序列中的下一个 token, 目标是最大化生成正确序列的概率. 这通过最小化负对数似然来实现:

$\mathcal{L}_{\text{MLE}} = -\sum_{t=1}^{T} \log P_{\theta}(y_t \mid y_{<t}, X) \quad{(1)}$

其中, $X$ 是输入 (提示或上下文), $Y = (y_1, y_2, \dots, y_T)$ 是目标输出序列, $P_{\theta}(y_t \mid y_{<t}, X)$ 是模型在给定前面 token 的情况下对 token $y_t$ 的预测概率.

例如, 对于序列 你 知道 什么 是 预训练 吗 ？, 如果分词后对应的 token ID 为 9 3 6 4 2 1 5, 并填充至最大序列长度 10, 即 9 3 6 4 2 1 5 0 0 0.

- 模型输入 $X$ 为 9 3 6 4 2 1 5 0 0 0.- 模型输出 targets 为 3 6 4 2 1 5 0 0 0 0 (通过移位构造).
模型的 Transformer 部分接收输入 (batch_size, sequence_length, embedding_dim) (例如 (1, 10, 768)) 的 Embedding, 并输出相同维度的 logits. 之后, 通过一个线性变换层 output_layer = nn.Linear(768, vocab_size, bias=False) 将 Transformer 的输出映射到词表大小, 得到 logits 维度为 (1, 10, vocab_size).
损失计算为 F.cross_entropy(logits.view(-1, logits.size(-1)), targets.view(-1), ignore_index=0).

#### 2. SFT 阶段：有监督微调

“有监督微调”意味着使用有标签的数据来调整一个已预训练好的语言模型 (LLM), 使其更适应某一特定任务. SFT 的每一条样本通常由 prompt (或 instruction) 和 answer 两部分组成.

例如:

- prompt: 翻译以下句子: What is pretrain- answer: 什么是预训练

数据构造时, 会拼接 prompt 和 answer, 并在 answer 两侧添加开始 ([bos]) 和结束 ([eos]) 符号. 为了让模型更关注 answer 部分的预测, 会生成一个 mask 向量来屏蔽不希望计算损失的部分 (prompt 部分为 0, answer 部分为 1).

$\text{input\_id} = \text{prompt} + [\text{bos}] + \text{answer} + [\text{eos}] \quad{(2)}$

然后计算 prompt 的长度 (context_length), 并根据最大长度进行填充 (pad_len). 最终生成 loss_mask 向量, answer 部分为 1, 其他部分为 0.

模型的输入经过 Embedding 层和 Transformer 后的过程与预训练完全一样, 得到维度为 (1, 10, vocab_size) 的 logits. 最后, 只计算 answer 部分的损失, 通过 mask 将不希望考虑的部分乘以 0:

$\text{loss} = \frac{\sum (\text{F.cross\_entropy}(\text{logits.view}(-1, \text{logits.size}(-1)), Y.\text{view}(-1), \text{ignore\_index}=0, \text{reduce}=\text{False}) \times \text{loss\_mask.view}(-1))}{\text{loss\_mask.sum()}} \quad{(3)}$

有了损失后, 即可进行反向传播更新模型参数.

### 1.3 幻觉问题与 SFT

幻觉是 LLM 领域讨论度最高的问题之一. 广义的幻觉指模型回答错误、胡说八道; 狭义的幻觉则指模型本身具备某个知识, 但经过对齐 (alignment) 处理后回答错误.

- **广义幻觉**: 目前无解, 只能通过 SFT/RLHF 让模型学会拒绝回复训练过的同类型案例.

- **狭义幻觉**: 可以缓解, 通常被视为过拟合的一种体现, 或是引入 special token 和固定输出格式导致的知识丢失. 判断狭义幻觉的方法是让预训练模型续写某个知识点, 对比与 SFT 后回复的一致性.

工业界对幻觉问题态度相对宽松, 原因包括:

1. 广义幻觉与狭义幻觉在用户体验上无明显区别, 且广义幻觉可通过外挂 RAG (Retrieval-Augmented Generation) 或 function_call 解决.2. 狭义幻觉的缓解方式多为调参, 收益有限, 且在不同基座模型上的收益不一, 显得过于“trick”.3. 工业界的 AI 助手通常是全链路系统, 裸模型的安全与幻觉问题可通过上下游的小模型和词典配置进行拦截或改写.

因此, SFT 阶段通常不侧重于减少幻觉本身, 而更多关注指令遵循能力的提升.

## 2. SFT 数据及处理

SFT 工作者的 90% 工作量在于分析和清洗数据.

### 2.1 业内共识：质量优先

LLM 从业人员普遍认同: **SFT 训练数据的核心是数据多样性和数据质量, 数据数量并不重要**.

- **LIMA 论文**: 仅用 1K 条高质量人工修改和格式统一的数据集, 在 Llama-65B 上 SFT 取得了不俗成绩, 重申了 LLM 的主要知识来源于预训练, 少量优质调整数据足以引导模型高质量输出.

- **LLaMA2 论文**: 大力强调“Quality Is ALL You Need”, 发现市面开源 SFT 数据质量参差不齐, 最终只使用了 27,540 条高质量训练数据.

- **Yi-34B**: 同样秉持质量优先原则, 使用不到 10K 条 SFT 数据, 并强调了统一回答格式, 剔除预训练未涵盖的 SFT 内容以避免幻觉, 以及对 Instruction 进行 tagging 分类以确保任务多样性.

数据质量方面, prompt 可以不那么严谨, 但 answer 必须尽量精确, 甚至一个标点符号都不能错, GPT-4 啰嗦的回复也应精简.

### 2.2 数据多样性探索

没有人能清晰定义“数据多样性足够好”, 只能从先验角度, 让模型接触各种任务类型. 数据多样性主要包含“数据用途”和“数据形式”两个维度.

#### 1. 数据用途 (task_type)

- **抄尖子生作业**: 参考 OpenAI 官网列出的 ChatGPT 擅长任务 (如翻译, emoji 聊天等), 为每个任务收集数据.

- **传统 NLP 任务**: 补充 NER, 机器阅读理解, 意图识别等传统 NLP 任务数据 (传统 NLP 数据集质量高).

- **业务需求**: 根据下游业务需求, 提前让模型接触特殊场景任务 (如对春联, 猜灯谜), 只要数据质量没问题, 通常不会破坏模型能力.

**关键点**: 每条 SFT 训练数据必须有明确的 task_type 类型, 避免大杂烩, 以便后续的 case 分析. 实际工作中, 双层 task_type (如“逻辑推理 - 常识推理”, “逻辑推理 - cot 多步骤推理”) 很常见. 不同 task_type 的数据量应动态调整, 难的任务数据量多, 简单的少.

#### 2. 数据形式

- **Prompt 表达方式多样性**: 避免千篇一律, 加入不同的语境和角色扮演, 防止模型只识别关键 token 导致过拟合或泛化性差.

- **Prompt 长度均衡**: 既要有短数据, 也要有长数据, 避免模型 attention 退化, 关键信息在 prompt 中位置应足够随机.

- **Answer 长度均衡**: 避免模型输出过短, 适当加入长回答语料, 使其能遵循“不少于 2000 字”等指令.

- **多轮聊天 topic 切换能力**: 训练数据应包含当前 query 与 session 相关和无关的两种情况, 让模型学会判断关联性.

- **Answer 分布多样性**: 避免大量训练数据复用同一句话作为 answer, 否则会导致模型严重过拟合.

概括而言: **数据形式不能让模型找到规律, 关键信息在 prompt 中的位置分布要足够随机**. 这是为了避免模型训练时退化, 只聚焦于某些或某些位置的 token, 而非完整 prompt.

### 2.3 数据生产

#### 1. 生产 Prompt

- **Self-Instruct 方法**: 为每个 task_type 准备 seed prompt, 然后随机采样 seed, 喂给强能力的 pretrain 模型 (或 GPT-4) 基于 seed 问题续写或仿写更多问题.

- **现有数据集**: 搜集现有 SFT 或 NLP 训练集, 筛选并改写.

- **手动编写**: 实在找不到合适的 prompt, 可以手动编写, 毕竟 prompt 比 answer 容易编写.

- **任务拆解**: 对于困难任务, 应提前拆解 prompt, 例如将“设计一个万字爽文”拆解为“设计故事大纲”和“基于大纲扩充内容”. LLM 只是知识量比人多, 而非知识掌握度比人精细.

#### 2. 生产 Answer

- **“Good Model” (如 GPT-4/Claude 3)** : 在不考虑成本时, 使用它们生产 answer.

- **自部署模型**: 考虑成本时, 部署 Qwen_72B/Deepseek_MOE 等模型. (llama 系列中文能力不佳).

- **小模型 + SFT 万能公式**: 这是最实用且经济的方法. 训一个小模型, 利用 GPT-4 生产少量高质量 answer (例如 1K 条), 然后用小模型预测出上万条数据. 小模型知道考卷是什么, 然后只学什么, 能考出好成绩.

- **Prompt 优化**: 访问 GPT-4 的 prompt 并不等价于喂给模型的 prompt. 需在 ChatGPT playground 上多测试, 找到模型回复最好的 prompt, 适当加 few-shot.

- **CoT 在数据生产中的应用**: 即使实际工作中出于耗时考虑不使用 CoT 训练模型, 但在数据生产时, 为了保证回复质量, 应让 GPT-4 用 CoT 方式回复.

- **数据校验与去重**: GPT-4 和自训模型都可能产生不可用的数据, 需要写规则或肉眼校验. 针对特定 task_type 生成的数据同质化严重, 需数据去重, 避免 answer 过于相似导致过拟合.

**小结**: 数据质量是 SFT 工作的核心, 数据生产不能当甩手掌柜. 训练者必须熟知 prompt 的表达方式和 answer 的回复风格.

### 2.4 数据飞轮：从上线到优化

模型的上线并非 SFT 工作的结束, 而是真正工作的开始. 此时, 我们才能接触到“**最真实的用户 prompt**”.

- **用户日志**: 用户的 query 是最好的候选 prompt 数据集. 尤其是多轮聊天数据, 用户日志能反映出模型回复的真实情况, 培养模型的话题转移、自我纠错、坚持己见、结合新需求改写答案等能力.

- **点赞/点踩**: 用户日志通常附带“点赞/点踩”选项, 可用于 DPO/RLHF 数据生产 (需清洗, 数据可能很脏).

- **鲁棒性数据**: 训练数据中应包含一些“鲁棒性数据” (answer 正常但 prompt 表达差劲, 如错别字、不完整、中英拼音夹杂), 以适应线上用户的糟糕表达. 这些数据可以直接从线上拉取或手动修改原有 prompt, 并打上专属标签.

只有将“定期拉取用户日志, 利用规则筛选有价值的 prompt, 访问 GPT-4 获取答案, 加入新数据更新模型”这样的**数据飞轮**运行起来, SFT 工作才能进入良性循环.

### 2.5 专项数据与数据工程

RAG, 长文本, Agent, 复杂指令, function_call 等 SFT 专项数据, 在训练上几乎没有额外技巧 (长文本训练除外), 难点一半在数据生产, 另一半在工程开发. 这些专项任务本质上是**数据工程**, 应当大胆尝试.

#### 1. RAG (Retrieval-Augmented Generation)

- **核心**: 建库, 知识库检索的准确性决定上限.

- **外挂模型**:

- 知识/聊天二分类模型: 判断是否需要做 RAG.- 传统 IR 模型: 快速检索候选文档.

- **SFT 数据构造细节**:

- 检索内容为空时的回复.- 检索内容相互矛盾时的回复 (不能只盯着第一条/最后一条).- 检索内容与 query 完全无关时的回复.

- **检索内容错误**: 模型应照着错误答案念, 不能让模型自己判断知识正确性. 大前提是“数据库知识准确率高于模型自己具备的知识”.

#### 2. Agent / Function Call

- **本质**: 将 Agent 和 function_call 理解为同一个东西.

- **实现**: 在 system prompt 中加入指令 (如“遇见数值计算任务你就输出 `<special_token>` + 调计算机”), 然后构造类似数据.

- **训练数据**: 除了 system, user, assistant 轮次外, 还要有“调计算机”, “计算机返回结果”等轮次.

- **核心**: 数据是 Agent 最核心的内容.

#### 3. 长文本

- **实现**:

- **训练**: 利用 NTK 外推 (增大 RoPE 基底), 少量长文本数据即可适应新基底. 使用 sequence_parallel 技术解决 attention 计算量与数据长度平方关系导致的显存不足问题.

- **数据**: 构造长文本理解数据, 避免短数据拼接. 关键信息位置应随机分布. Paper, 书籍, RAG 数据都是好的胚子.

- **任务示例**:

- 简单: “背密码” (随机插入密码).- 复杂: 概括 paper instruction, 列举林黛玉出场章节.- 挑战: 计算林黛玉出场次数.

#### 4. 复杂指令

- **定义**: Prompt 包含非常多的限制, 如字数限制, emoji 插入, 押韵, 输出 special token 等.

- **实现**: 目前技术路线不明晰, 猜测需借助 CoT 或自我纠错能力. 模型在 next token prediction 时难以找到完美 token 满足所有限制条件.

- **造数据技巧**: **先射箭, 再画靶**. 如果 GPT-4 回复未完全遵循所有限制, 直接修改 prompt 使其符合实际输出 (例如, 将“不少于 200 字”改为“不少于 180 字”).

### 2.6 SFT 数据质量过滤算法：设计思路与实现原理

SFT 数据清洗主要包含三个维度: **数据质量、数据多样性和数据必要性**.

#### 1. 数据质量过滤

- **奖励模型打分**: 使用奖励模型 (如 reward-model-debertav3-large-v2), 或自训 reward 模型, 或 ChatGPT 对数据进行打分, 分数高于 $\alpha$ 的认为是高质量数据.

- **大模型多角度打分**: 利用大模型从准确性、指令遵循、预期表达等角度对数据打分.

- **标签数量与复杂度筛选**: 对数据打标, 根据标签数量和复杂度挑选.

- **困惑度度量**: 通过预训练模型计算回复的困惑度, 困惑值越大意味着数据越难.

- **长度度量**: 根据长度进行度量.

##### DEITA 中的指令复杂性与回答质量评分

- **指令复杂性**:

- 随机选择、指令长度、困惑度、ChatGPT 直接打分、语义树节点数 (Instruction Node), Instag Complexity (ChatGPT 打标签再训练 Llama 模型预测).- DEITA 步骤: 小规模数据 (2K) → WizardLM 增强指令复杂性 → ChatGPT 打分排序 → 训练 Llama1-7B 模型学会打分.

- **回答质量评分**:

- 随机选择、输出长度、ChatGPT 直接评估准确性.- DEITA 步骤: 小规模数据 (2K) → ChatGPT 改写回复 (增强有用性、相关性、深度、创造力) → ChatGPT 打分排序 → 训练 Llama1-7B 模型学会打分.

##### CaR (Clustering and Ranking)

- **核心**: 融合质量和多样性. 使用评分模型评估质量, 并结合 K-Means 聚类从每个簇中选取高质量数据.

- **评分模型**: 论文强调了得到符合人类偏好的评分模型的重要性.

#### 2. 数据多样性探索

- **抽取表征与聚类**: 使用 BERT 模型抽取数据特征, 对数据进行聚类 (如 K-Means), 在每个簇内用质量评分挑选.

- **K-Center-Greedy 算法**: 抽取表征, 不断挑选目前样本中与所有中心点最远的点作为新的中心点, 直到数据量达到要求.

- **标签方式**: 选择不同标签的数据.

##### DEITA 中的数据多样性筛选

- 步骤: 计算复杂性评分和质量评分, 相乘作为综合评分 → 按综合评分排序 → 使用 LLama1-13B 模型抽取数据向量 → 逐个抽取数据 $x$, 计算其与保留数据底池中其他样本的最小余弦相似度 → 保留相似度低于 $t$ 的数据 $x$ → 挑选数据直至达到阈值.

##### CaR 中的数据多样性

- 使用 sentence-transformers 抽取指令向量 → PCA 降维 → K-Means 聚类 (k=178) → 对每个簇内样本打分 → 挑选 top-N 数据.

#### 3. 数据必要性 (Necessity)

- **IFD (Instruction-Following Difficulty)** 

- **核心**: 衡量模型生成 answer 时对 prompt 的依赖程度.**指标**: 
$s_\theta(A|Q) = \frac{1}{N} \sum_{i=1}^{N} \log P(w_i^A | Q, w_1^A, \dots, w_{i-1}^A; \theta)$
 (给定问题时回答的困惑度).
$s_\theta(A) = \frac{1}{N} \sum_{i=1}^{N} \log P(w_i^A | w_1^A, \dots, w_{i-1}^A; \theta)$
 (单纯度量回答的困惑度).- $r_\theta(Q, A) = \frac{s_\theta(A|Q)}{s_\theta(A)}$ (IFD 比值).

- **挑选逻辑**:

- $s_\theta(A|Q)$ 高, $s_\theta(A)$ 低: 模型难根据 Q 生成 A, 但单独生成 A 容易. 说明指令跟随能力差, 需要这部分数据.- $s_\theta(A|Q)$ 低, $s_\theta(A)$ 高: 模型易从 Q 生成 A, 但单独生成 A 困难. 说明 Q 包含生成 A 的信息, 且模型能利用. 不需要这部分数据.- $s_\theta(A|Q)$ 高, $s_\theta(A)$ 高: 模型生成 A 困难, A 可能有问题. 不需要.- $s_\theta(A|Q)$ 低, $s_\theta(A)$ 低: 模型生成 A 容易. 指令跟随能力很好.

- 故挑选 IFD 指标较高的数据进行训练.

- **让模型尝试回答**: 度量回答质量.

##### MoDS 中的数据必要性

- 步骤: 使用多样性过滤后的数据微调模型 → 用该模型对高质量数据进行推理 → 使用奖励模型计算推理结果的奖励值 → 当奖励值低于 $\beta$ 时, 认为模型无法回答好该问题, 保留该数据 (Augmented Instruction Data).

#### 4. 代码 SFT 数据收集：特殊策略

代码 SFT 数据收集通常依赖 **Synthetic Data** 和 **Rejection Sampling**. 并且最好结合一般性目标的 SFT.

##### a. Code Llama - Instruct

- **重用 Llama 2 - Chat SFT Data**: 包括 Rejection Sampling Data 和纯人工收集的 SFT Data, 覆盖 Helpfulness 和 Safety 两方面.

- **Synthetic Demonstration Data (14K 条)** :

- **Execution Feedback**: Prompt (由 Llama 2 70B 生成) → Demonstration (由 Code Llama 7B 非 SFT 版本在 10 个候选 Response 中选择第一个通过所有测试用例的).

- **正确性检查**: 通过静态检查 (Parser/Linter) 和动态检查 (生成测试用例并运行代码).

- **修订**: 未通过检查的候选 Demonstration 会被修订, 而不是丢弃.

- **防遗忘**: 加入少量 Code 继续预训练数据 (6%) 和通用自然语言预训练数据 (2%), 防止对一般代码和语言理解能力的退化.

##### b. DeepSeek 系列

- **DeepSeek-Coder-Instruct**: SFT 数据纯人工收集, 总量小于 2B Tokens, 多轮对话格式, 无防遗忘或一般性目标 SFT Data.

- **DeepSeek-Coder-V2-Chat**: 包含 Code 和 Math 两方面 SFT 数据 (300M Tokens, 微调 1B Tokens), 启用了防遗忘策略, 加入少量 DeepSeek-V2 的一般性目标 SFT Data.

- **DeepSeek Chat & V2 Chat**: 尽管非代码专家模型, 在 SFT 中加入了 22.2% 的代码数据以增强代码能力.

##### c. Llama 3/3.1 Instruct

- **Synthetic Data (2.7M 条)** : 多轮对话格式, 使用专门训练的代码专家模型和最新 Llama 3 版本生成.

- **Execution Feedback (1M 个 Coding 对话)** : 基于采样的代码片段 Prompt 大模型生成编程问题 (Prompt 也是合成的). 生成 Demonstration 时加入编程规则和要求解释思维过程可提高质量. 正确性检查包括静态和动态检查, 未通过的通过修订而不是丢弃.

- **Programming Language Translation (0.5M 条)** : 将常见语言的代码 SFT 数据翻译成少见语言, 缓解语言比例不平衡问题, 需通过静态和动态检查.

- **Backtranslation (1.2M 条)** : 针对基于代码输入的任务. Prompt Llama 3 基于代码片段生成候选 Demonstration (即 SFT Data 的 prompt 部分), 然后再反译回代码片段, 比较反译代码与原始代码片段的等效性来检查质量.

- **Rejection Sampling**: 进行了六轮, 使用专门训练的代码专家模型采样候选 Demonstration.

- **System Prompt 使用**: 提高代码可读性、文档化、彻底性和特异性.

- **AI 质量检查**: 判断 Rejection Sampling 样本的代码质量 (正确性和风格), 并修订未通过的挑战性任务样本.

### 2.7 工业落地思考

- **质量、多样性、必要性**: 确保模型学习高质量回复, 泛化能力强, 并评估每条数据对模型提升的必要性.

- **Target Method 优先**: 工业落地更推荐 Target Method (如 Nuggets, LESS), 因为它们更贴近真实的业务场景, 产品经理通常关注具体的应用场景. Non-Target Method 更适用于没有明确目标的 To-C 场景, 但此时直接使用开源调优模型可能更高效.

- **多种筛选方法结合**: 为提高模型鲁棒性, 可结合目标导向和无目标导向方法, 例如 70% 数据来自 LESS, 30% 来自 CaR.

- **数据叠加的非线性**: 需注意数据之间可能存在的冗余甚至冲突. 某些筛选方法可能筛选出大量相似类型数据, 但并非都能带来额外增益.

## 3. SFT 训练

### 3.1 训练框架

在 SFT 阶段, 推荐使用 **DeepSpeed** 而非 Megatron.

- **SFT 语料不多**: DeepSpeed/Megatron 在速度性能上的差异不大.

- **DeepSpeed 优点**:

- 许多对齐 (alignment) 的开源工作和代码基于 DeepSpeed 实现, 复现省事.- 利用 AutoModelForCausalLM 可直接训练大多数开源模型, 无需 trans_hf_to_megatron.- 训练出的模型可直接启动 tgi 服务或 vllm 推理, Megatron 则需 trans_megatron_to_hf.

### 3.2 参数设置

每次启动训练前, 务必检查并设置以下参数:

- epoch: 训练轮数.- gradient_accumulation_steps: 在执行反向传播/更新前累积梯度的步数.- global_batch_size (Megatron 参数, DeepSpeed 可忽略).- learning_rate: 学习率.- lr_scheduler_type: 学习率调度器类型.- dropout: 防止过拟合.

影响训练速度的参数:

- zero_stage: DeepSpeed 的 ZeRO 优化阶段.- max_seq_len: 最大序列长度.- offload: 显存卸载策略.- gradient_checkpointing: 梯度Checkpoint, 降低显存占用但可能降低速度.- seq_parallel_size: 序列并行大小.

对模型训练影响较小但需了解其意义的参数:

- weight_decay: 权重衰减.- per_device_train_batch_size: 每个设备 (如 GPU) 的训练批次大小.- num_warmup_steps: 学习率预热步数.

### 3.3 训练技巧

- **炼丹技巧通用原则**: 小模型用大学习率, 大模型用小学习率.

- **Epochs**: 普遍在 1 到 3 个, 少量数据 (1W 级别) 也可以, 但不应超过 100W (经验结论).

- **Warmup**: 起始训练适当做点 warmup.- lr_scheduler_type: 尝试几种主流的调度器.- gradient_accumulation_steps: 重要参数, 尝试 16/32/64/128 等值.

- **Dropout**: 按需开启, 不开通常无大碍, 开启可能导致训练崩溃.

- **LoRA (Low-Rank Adaptation)** : 一种参数高效微调 (PEFT) 技术, 旨在节省显存. 在全参训练算力充足时, LoRA 似乎无明显优点, 可能能防止过拟合 (但可通过减少数据、开启 dropout、调整学习率实现). LoRA 只调整模型一小部分参数, 而非整个模型, 减少内存占用且遗忘较少.

- **其他参数更新策略**: 仅更新部分模型参数 (如特定层微调、embedding 微调、attention 微调、mlp 微调) 通常有特殊需求, 可通过冻结不想更新的参数实现.

### 3.4 训练策略

#### 1. Loss 曲线分析

- channel_loss: 不同 task_type 应有不同的 channel loss, 分别观察.

- **Special Token Loss**: 初始会高, 但下降很快.

- **创作类任务 Loss**: 通常高于其他任务, 因为答案越固定, 搜索结果越单一的语料, loss 越低, 反之亦然.

- **初始 Loss 值**: 通用数据且采样后, 7B/13B 模型可能在 2 左右 (难数据可能到 3), 72B 模型在 1-2 之间.

- **最终 Loss 值**: 大约在 0.5 左右.

- **Loss 持续升高**: 表明训练代码有问题, 因为 next_token_prediction 训练方式是“背书”, 不存在学不会, 只存在学会了但不会泛化的情况. 即使是随机乱码, loss 也应持平, 而非升高.

#### 2. 阶梯形 Loss

- **现象**: 许多文章分析过, 通常被公认为模型过拟合的体现.

- **SFT 中的过拟合**: SFT 阶段让模型学习指令遵循能力, 其直观表现是对问答格式过拟合, 即“模型开始回答 question, 而不是续写 question”. 因此, SFT 格式过拟合并非坏事, 我们担心的是模型对 answer 内容过拟合, 不管什么 question 都重复一个 answer.

- **示例**: Question: “以 json 格式输出 XXXX”. Answer 1: ``json… (格式过拟合, 非坏事). Answer 2: 好的, 这就为你用 json 输出结果 ``json… (非格式过拟合).

- **结论**: 阶梯形 loss 只是过拟合的一种体现, 可能是格式过拟合或 answer 过拟合. 观察到这种现象是合理的, 不代表模型训得过火. 关键在于理解其原因, 而非努力避免.

#### 3. 拟合问题 (Fitting Problems)

SFT 工作者最常遇到的问题, 每天都在过拟合和欠拟合之间横跳.

##### a. 欠拟合 (Underfitting)

- **定义**: 模型未学好训练数据, 下游任务能力差, 通常只表现在某个 task_type 上.

- **原因**: 训练集相关 task_type 数据量与质量不足、task_type 难度、模型大小和能力、epoch、学习率、gradient_accumulation_steps 设置, 甚至训练方式.

- **调试**:

- **是否学到训练数据**: 让模型直接回答训练集. 若答不上来, 则多训 1 个 epoch, 补充数据, 调整学习率. (ChatGPT 建议: loss 下降慢则增大学习率跳出局部最小值, loss 震荡则减小学习率提高稳定性). 反复训不会则 review 训练代码.

- **泛化问题 (学会训练集但测试集完全不会)** :

- **模型能力**: 结合主观判断任务难度 (复杂指令, 逻辑推理, 数学计算), 参考技术报告, 下载同 size 开源模型测试能力, 与同行交流. 检查 base 模型在 pretrain 阶段是否欠缺该 task_type 基础知识 (多续写相关语料, 对比开源 base 模型).

- **数据质量**: 抽样 answer 检查质量 (是否准确, 是否符合语言模型, 避免倒装句), 逐字慢读. 若数据质量 OK, 则多造训练数据或上采样现有数据.

- **“杀手锏”**: **重写 prompt**. 削减 answer 知识, 增加 prompt 背景知识, 甚至将 answer 常用 token 写入 prompt, 降低任务复杂度.

- **训练方式案例**: Llama2 的 sft 阶段与 pretrain 一样将数据 concat 成 4K 长度训练且不做 attention_mask, 导致短 answer 数据 loss 占比改变, 分类任务 ACC 下降. 解决办法: 上采样短 answer 数据.

##### b. 过拟合 (Overfitting)

- **定义**: 模型训练过度, 对训练集中的 answer/pattern 记忆太死. 相比欠拟合更容易解决.

- **特点**: 通常是某项能力局部过拟合, 大部分能力正常. 不像传统深度学习那样通过调整 epoch、学习率、dropout、weight_decay 解决, 否则可能导致整体欠拟合.

- **解决方案**: **优化训练数据**来缓解过拟合, 主要措施是**删减对应 **task_type** 的数据, 或扩充该 **task_type** 的数据多样性**.

- **调试**:

- **暴露过拟合模式**: 通过让 base 模型续写判断是否 pretrain 阶段训错 (通常不是). 针对回答错误的问题, 对 answer 核心关键词进行测试, 发出一连串提问 (如“日本的首都是北京”的例子, 测试“美国的首都是哪里?”, “日本最大的城市是哪里?”等), 确定模型对哪个 token 或 pattern 过拟合. 然后对过度的数据进行删除或改造.

##### c. 夹逼准则 (Sandwich Principle)

- **核心**: **确定模型可达到的效果上限在哪里, 进而定位训练问题**.

- **经验分享**: 同事遇到 continue-pretrain 训练领域数据时, 测试集 loss 下降但做题得分不变.

- **第一步**: 测试集是选择题, question 远大于 answer. loss 下降可能是模型背下题目, 非 $P(\text{answer} | \text{question})$ 提高. 应看 $\sum P(\text{answer} | \text{question})$ 是否提升.

- **第二步**: 若混合 common 数据的 continue-pretrain 未学会知识, 可能是训练太少、学习率不合理、领域数据质量差.

- **判断数据质量**: 基于 base 模型做 100% 领域集合训练 (或直接拿领域数据做 sft 训练). 若仍无法提升测试集做题能力, 则数据可能无效. 若大幅提升, 则数据没问题, 需研究超参数 (调大学习率, 增加领域数据占比).

- **实战思路**: 解决拟合问题时, 若不确定是数据还是训练问题, 可只训练目标 task_type 数据并训 10 个 epoch. 若仍学不会, 说明“这个模型 + 这份数据”上限就是学不会. 若学会, 则说明“能学会”是可触及的上限. 逐步往下采样训练配置 (如训 5 个 epoch, task_type 数据占比 50%), 找到“临界值”.

#### 4. 多轮对话专项提升

- **目的**: 更好地处理多轮对话.

- **数据形式**: 连续对话 (多轮对话) 而非单一提示-响应对.

- **目标**: 保持上下文, 产生连贯回复. 即使与指令微调重叠, 专用对话数据也能产生更自然的用户体验.

## 4. SFT 评估

### 4.1 评估方式

- **高质量评测集合**: 必须事先准备, 包含明确的 task_type, 其上的“模型可用性”是模型能否上线的标准.

- **3H 原则**: SFT 评估需考量经典的 Helpfulness, Honesty, Harmlessness.

- **自定义指标**: 根据模型需求制定指标, 如是否遵循指令, 是否 system 穿透, 内容是否准确, 是否产生幻觉, 是否安全等.

- **多维度得分**: 每个维度应有单独得分, 根据加权公式确定回复可用性, 以便直观分析是哪个维度变差.

- **理解 LLM 系统**: 评估时要了解整个 LLM 系统, 关注 LLM_engine 无法兜底的能力项 (如前置黄反拦截, RAG 系统对幻觉的关注程度).

#### 1. 机评 (Machine Evaluation)

- **评估模型**: 利用 GPT-4 进行评估.

- **Prompt 措辞**: 评估 prompt 必须好好揣摩, 大模型评估有明显偏好 (如 A 优于 B, 长句优于短句).

- **打分一致性**: 同一个正确答案, 模型可能给出 3, 4, 5 分.

- **参考答案**: 给模型提供参考答案, 可使打分更准确, 因模型会考虑“候选 answer”和“gold answer”的相似度.

- **参考资料**: Alignbench 的评估 prompt 写得很好, 可仿照.

#### 2. 人评 (Human Evaluation)

- **主流方案**: 估计是各 LLM 公司主流方案, 成本不一定比机评高.

- **效率**: 人有长记忆, kv_cache 可保存多天. 评估同学熟悉 prompt 后, 效率很高, 基本不逊于机评.

- **重点**: 只要不频繁修改评测集合, 评估效率就能保持.

#### 3. 流行 LLM 评估方法汇总

- **人类对聊天的偏好**: 如 Chatbot Arena.

- **LLM 作为聊天的评判者**: 如 Alpaca Eval, MT Bench, Arena Hard V1/V2.

- **指令型语言模型的静态基准测试**: 如 LivecodeBench, AIME 2024/2025, GPQA, MMLU Pro, IFEval.

- **函数调用与代理**: 如 BFCL V2/V3, NexusBench V1/V2, TauBench, ToolSandbox.

### 4.2 评估分析

- **对比评估**: SFT 的评估结果分析是做 case 分析, 必须是与上一版模型测试集合并来观察.

- **能力项得分变化**: 当 SFT_v2 相比 SFT_v1 某个能力项平均得分明显下降时, 回想该能力项数据分布变化:

- 训练数据更新 (质量不如前).- 整体训练数据增多, 导致该能力项数据占比变小, 欠拟合.- 训练参数配置不当.

- **波动**: 最常见情况. 挑出差劲 case, 用 do_sample 方式让两版本模型预测十次, 看 path@10 是否也变差. 波动常影响 path@1, 但对 path@N 影响不大.

- **迭代优化**: 每次获得新评估结果后, 查看模型在各 task_type 上的可用性得分. 得分变好的情况是未来洗数据的好经验, 变差的则是下一版优化目标, 反复迭代.

### 工业前沿：SFT 工程实践与 LoRA 微调细节

**ChatML 模板与数据格式化**. 工业级 SFT 通常采用 ChatML 格式统一对话模板：
```
<|im_start|>system\n{system_prompt}<|im_end|>\n
<|im_start|>user\n{user_message}<|im_end|>\n
<|im_start|>assistant\n{assistant_response}<|im_end|>
```
特殊 token(`<|im_start|>`、`<|im_end|>`)需在 tokenizer 中预注册. 数据预处理时，只对 `assistant` 角色的回复计算 loss，其他 token 的 label 设为 `-100`(忽略索引). 这一设计确保模型学习的是"如何回答"而非"如何提问". 

**Tokenizer 扩展与嵌入层适配**. 当模型需要支持新语言或代码方言时，需扩展词表. 调用 `model.resize_token_embeddings(V + k)` 将 embedding 层和 lm_head 从 $V$ 扩展到 $V + k$ 行. 新行通常用现有 embedding 的均值初始化，避免破坏已学习语义. 扩展后必须在 SFT 中让模型"认识"这些新 token，否则它们会成为随机噪声. 

**LoRA target_modules 的工程选择**. `target_modules` 决定哪些层注入低秩适配器：
- 仅 `q_proj` + `v_proj`：参数量最小(~0.2%)，适合快速实验，但表达能力受限
- `q_proj` + `k_proj` + `v_proj` + `o_proj`：平衡参数量与效果(~0.5%)
- 全部线性层(含 FFN)：参数量最大(~1-2%)，接近全量微调效果

QLoRA 进一步将基模型量化为 4-bit(NF4)，LoRA 权重保持 16-bit. 在 24GB 显存上可微调 70B 模型，成为个人研究者的标配方案.

## 5. 总结 (Conclusion)

SFT 在大模型浪潮中扮演着举足轻重的角色, 它是将预训练 LLMs 适配特定任务、提升指令遵循能力的关键步骤. 通过对 SFT 核心概念、数据处理、训练策略和评估方法的深入探讨, 我们了解到高质量和多样化的数据远比数据量更为重要. 无论是通过数据飞轮持续优化, 还是运用 IFD、MoDS 等先进算法进行数据筛选, 其目的都是为了提升模型在实际应用中的性能和鲁棒性. 训练框架的选择、参数的精细调整以及对拟合问题的深刻理解 (尤其是“夹逼准则”的应用) 都是 SFT 成功的必要条件. 最终, 严格而全面的评估, 特别是对比评估, 才能指引模型的持续迭代与进步. SFT 工作的核心在于“数据有猫腻 / 训练数据缺少某个能力项 / 这 case 为什么没触发 RAG / 有黄反拦截倒也不必纠结”的经验判断. 只要掌握了这些方法论并结合实践经验, 即使是算法新人, 也能在 LLM 领域快速胜任 SFT 工作, 培养出对训练的“感觉”, 从而更好地发挥大模型的潜力.
