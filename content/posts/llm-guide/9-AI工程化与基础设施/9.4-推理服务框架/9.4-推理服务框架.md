# 9.4 · 推理服务框架: vLLM、TensorRT-LLM、TGI与生产部署

## 1. 背景与核心痛点

训练一个大模型是"一次性烧钱"——训练 GPT-4 级别的模型需要数千万美元,但这个成本是可控的、可预算的. 相比之下,**推理是持续性的印钞机(或碎钞机)** . 当模型部署上线,每处理一个用户请求都在消耗 GPU 算力和电力. 如果服务每天处理 10 亿次请求,每次请求的成本哪怕只相差 0.001 美元,一年就是 365 万美元的差距. 

**核心痛点**: 推理与训练有着本质不同的约束: 
- **延迟敏感(Latency Sensitive)** : 用户无法容忍聊天机器人思考 10 秒才回答. 首 token 延迟(Time To First Token, TTFT)和每 token 延迟(Inter-Token Latency, ITL)是产品体验的核心指标. 

- **吞吐优先(Throughput Critical)** : 在线服务需要同时处理成千上万个并发请求,GPU 利用率必须被压榨到极致. 

- **输出长度不确定(Variable Output Length)** : 与训练时固定的序列长度不同,推理时每个请求的输出长度从几个 token 到几千个 token 不等,这给 batching 和调度带来了巨大挑战. 

- **显存动态变化(Dynamic Memory)** : KV Cache 的大小随序列长度线性增长,且每个请求的增长速度不同. 如何管理这块"动态膨胀"的显存,是推理系统的核心难题. 

**家谱定位**: 如果说 9.3 的分布式训练框架解决的是"如何造出一个大脑",那么 9.4 解决的是"如何让这个大脑同时服务千万人". 它是 LLM 从技术 demo 走向商业产品的最后一道工程门槛. 

## 2. 为什么重要

2024-2025 年,AI 推理市场的规模已经超过了训练市场. OpenAI、Anthropic、Google 的 API 收入中,最大的成本项不是研发,而是**推理算力**. 一个被广泛引用的数据是: 对于成熟的大模型服务,推理成本占总算力成本的 70-90%. 

这意味着,推理优化上的每一个百分点,都直接转化为利润. vLLM 的 PagedAttention 将吞吐量提升了 2-4 倍,相当于将推理成本直接砍半. TensorRT-LLM 的编译优化在某些场景下将延迟降低到原来的 1/3. 这些不是学术数字,是实打实的商业竞争力. 

## 3. 推理优化全栈: 从量化到调度

大模型推理优化的技术栈可以自底向上分为五个层次: 

![LLM推理优化五层技术栈](images/inference_optimization_stack.png)

> **图 9.5 LLM 推理服务优化全栈体系**
> * **第5层：调度与服务(顶层, 红色)**：负责集群层面的请求管理, 包括负载均衡、请求网关路由以及基于 QPS/GPU 利用率的自动弹性扩缩容. 
> * **第4层：批处理与解码(第四层, 橙色)**：在迭代级别对硬件进行压榨. 核心技术包含 Continuous Batching(动态请求插队)与 speculative decoding(投机采样加速). 
> * **第3层：内存管理优化(第三层, 绿色)**：针对 KV Cache 做显存回收与优化. 代表技术为 PagedAttention(虚拟内存分页)和 KV 缓存低精度量化. 
> * **第2层：编译优化层(第二层, 蓝色)**：通过图编译器将多算子融合为 Mega-Kernel(如 Fused MHA/Fused MLP), 极大地消除中间数据写回 HBM 的存取延迟. 
> * **第1层：模型压缩层(底层, 紫色)**：通过降低权重与激活的存储位宽(如 FP8/INT8/INT4 量化、AWQ/GPTQ 等算法)及剪枝蒸馏来直接减少计算量与显存占用. 


### 3.1 第一层: 模型压缩——量化

**权重量化(Weight Quantization)** 是将模型参数从高精度(FP16/BF16)压缩到低精度(FP8/INT8/INT4)的技术. 

**FP8 量化**: Hopper 架构原生支持 FP8(E4M3/E5M2),无需额外的反量化开销. FP8 权重量化几乎是无损的(perplexity 下降 < 1%),且由于数据宽度减半,内存带宽需求降低 50%,直接带来约 1.5-2× 的吞吐提升. 

**INT8 量化**: 需要计算缩放因子(scale)和零点(zero-point). 对于每一层权重矩阵,找到最大值 $W_{max}$,计算 scale $= W_{max} / 127$,然后将每个权重 $w$ 量化为 $q = \text{round}(w / \text{scale})$. 推理时,反量化为 $w' = q \times \text{scale}$. 

**INT4/AWQ/GPTQ**: 更激进的 4-bit 量化. AWQ(Activation-aware Weight Quantization)观察到只有一小部分"显著权重"对模型输出影响最大,因此对这些权重保留更高精度. GPTQ(General-purpose Post-Training Quantization)则使用二阶信息(Hessian 矩阵)来最小化量化误差. INT4 可以将模型体积压缩到原来的 1/4,但通常会带来 2-5% 的精度损失,适用于对延迟极度敏感但精度容忍度较高的场景(如边缘设备). 

**KV Cache 量化**: 不仅权重可以量化,推理过程中动态增长的 KV Cache 也可以量化到 FP8 或 INT8. 由于 KV Cache 在 long-context 场景下可能占据比权重更多的显存,KV Cache 量化的收益有时甚至超过权重量化. 

### 3.2 第二层: 编译优化——算子融合与图优化

深度学习编译器(如 TensorRT、XLA、TVM)的核心价值在于**自动算子融合(Kernel Fusion)** . 

**为什么融合能加速**: 回忆 Roofline Model. LayerNorm、Activation、BiasAdd 等操作都是 memory-bound 的(计算强度 < 10 FLOPs/byte). 如果逐个执行,每个操作都需要将数据从 HBM 读入、计算、写回 HBM. 三个操作就需要 6 次 HBM 访问. 

融合后的 kernel 只需要: **读一次 → 计算 LayerNorm+BiasAdd+Activation → 写回一次**. HBM 访问量从 6 次降到 2 次,速度提升可达 3-5 倍. 

**常见的融合模式**: 
- Linear + BiasAdd + Activation(如 GELU, Swish)
- LayerNorm + ResidualAdd
- Attention 内部的 QKV projection + split + transpose
- FlashAttention 本身就是将 attention 的多个子操作(matmul, softmax, masking, matmul)融合为一个 kernel 的典范

### 3.3 第三层: 内存优化——KV Cache管理

Transformer 的自回归解码过程中,每个新 token 的生成需要访问之前所有 token 的 Key 和 Value. 为了不重复计算,推理系统会**缓存(cache)** 之前计算的 K 和 V 矩阵,这就是 **KV Cache**. 

对于 batch size 为 $b$、序列长度为 $s$、头数为 $h$、头维度为 $d$ 的模型: 
$$
\text{KV Cache 大小} = 2 \times b \times s \times h \times d \times \text{sizeof(dtype)} \tag{1}
$$

以 Llama-3 70B 为例: $h=64, d=128$,FP16,batch size=16,平均序列长度 2048: 
$$
2 \times 16 \times 2048 \times 64 \times 128 \times 2 \text{ bytes} \approx 1.07 \text{ GB} \tag{2}
$$
如果 batch size 增加到 128,序列长度增加到 8192: 
$$
2 \times 128 \times 8192 \times 64 \times 128 \times 2 \approx 34 \text{ GB} \tag{3}
$$

这已经超出单卡 A100 的 40GB 显存上限——模型参数本身还需要 140 GB(70B × 2 bytes),必须借助量化压缩模型体积或采用多卡并行. 但核心问题是: **KV Cache 是动态增长的**,且不同请求的增长速度不同,传统分配策略会产生严重的显存碎片. 

**PagedAttention**(见下文)是这一问题的革命性解决方案. 

## 4. vLLM: PagedAttention与推理服务框架

### 4.1 PagedAttention的核心思想

vLLM 由 UC Berkeley 的 SkyComputing Lab 开发(2023),其核心创新 **PagedAttention** 借鉴了操作系统虚拟内存的**分页(paging)** 机制. 

**传统 KV Cache 管理的问题**: 
- 每个请求预分配一块连续的显存,大小为 `max_seq_len × kv_dim`. 
- 如果用户只生成了 100 个 token,但预分配了 4096 个 token 的空间,就浪费了 97.5% 的显存. 
- 更糟的是,不同请求的释放产生大量外部碎片——总空闲显存足够,但没有足够大的连续块来满足新请求. 

**PagedAttention 的解决方案**: 
- 将 KV Cache 分割成固定大小的**块(block)** ,例如每个 block 存储 16 个 token 的 K 和 V. 
- 每个请求的 KV Cache 是一个**逻辑上连续、物理上不连续**的块列表——就像操作系统的虚拟地址空间映射到物理页帧. 
- block 的分配和释放以 block 为单位,消除了外部碎片. 
- 当请求完成时,其占用的 blocks 立即被回收到空闲池,供新请求复用. 

![传统 KV Cache 与 PagedAttention 显存管理机制对比](images/paged_attention_comparison.png)

> **图 9.6 传统 KV Cache 预分配模式与 PagedAttention 动态块级分页模式对比**
> * **传统分配模式(left, 左侧)**：为每个并发请求预先分配一整块物理连续的、对应最大序列长度(如 4096)的显存空间. 由于大部分请求的实际生成长度远小于最大限制, 这会在显存中留下巨大的内部空洞(Allocated but unused), 并伴随大量的外部零散碎片, 显存利用率通常只有 50-60%. 
> * **PagedAttention 模式(right, 右侧)**：将显存划分为物理上不连续但大小相等的 KV 块(Blocks). 每个请求的 KV 缓存被拆分为多个逻辑块, 并通过 Block Table 动态映射到任意空闲的物理块上. 该机制消除了内部与外部碎片, 将显存利用率提升至 90% 以上, 带来数倍的并发吞吐能力. 


### 4.2 Block Table与内存共享

vLLM 为每个请求维护一个 **Block Table**——一个从逻辑块号到物理块地址的映射表. 

当模型生成一个新 token 时: 
1. 计算该 token 的 K 和 V. 
2. 如果当前逻辑块的物理块还有空间(未满 16 个 token),直接写入. 
3. 如果当前逻辑块已满,从空闲池中分配一个新的物理块,更新 Block Table,然后写入. 

**内存共享**: PagedAttention 还天然支持 beam search 和并行采样中的内存共享. 当多个候选序列共享相同的前缀时,它们的 Block Table 可以指向相同的物理块(copy-on-write). 这进一步减少了显存冗余. 

### 4.3 vLLM服务框架的整体架构

vLLM 不仅是一个 attention kernel,它是一个**完整的推理服务框架**: 

**调度器(Scheduler)** : 
- 维护一个 waiting queue 和一个 running queue. 
- 使用 **continuous batching**(见下文)动态将 waiting 请求加入 running batch. 
- 当 GPU 显存不足时,根据策略(如 FCFS、短作业优先)决定哪些请求可以被调度. 

**Worker**: 
- 每个 worker 进程管理一张或多张 GPU. 
- 负责执行模型的前向传播,使用 PagedAttention kernel 进行解码. 
- 支持张量并行(TP)和流水线并行(PP),worker 之间通过 NCCL 通信. 

**Tokenizer 与 Detokenizer**: 
- 将用户输入字符串编码为 token IDs. 
- 将模型输出的 token IDs 实时解码为字符串,支持流式输出(streaming). 

**API Server**: 
- 提供与 OpenAI API 兼容的 RESTful 接口(`/v1/completions`, `/v1/chat/completions`). 
- 支持异步请求处理、请求超时、并发控制. 

### 4.4 vLLM的局限

- ** prefix caching 的延迟**: 虽然 PagedAttention 消除了碎片,但 Block Table 的查找和更新引入了额外的 CPU 开销. 在 batch size 极大时,调度器本身可能成为瓶颈. 

- **对长上下文的支持**: 虽然理论上支持无限长上下文(只要显存够),但 attention 的计算复杂度仍是 $O(n^2)$. 当序列长度超过 32K 时,即使内存管理完美,计算本身也会成为瓶颈. 

- **缺乏图优化**: vLLM 的 kernel 是用 CUDA 和 Triton 手写的,没有深度学习编译器的图优化能力. 某些操作(如特定的 activation fusion)可能不如 TensorRT-LLM 优化得彻底. 

## 5. TensorRT-LLM: 编译优化的极致

### 5.1 TensorRT-LLM的定位

TensorRT-LLM 是 NVIDIA 推出的官方推理优化框架,基于其成熟的 TensorRT 深度学习编译器构建. 与 vLLM 的"手写 kernel + 内存创新"路线不同,TensorRT-LLM 走的是**编译优化**路线: 将模型图编译成针对特定 GPU 架构高度优化的执行引擎. 

### 5.2 编译优化的核心手段

**图优化(Graph Optimization)** : 
- **Layer Fusion**: 自动识别并融合相邻的 memory-bound 操作. 例如将 `MatMul → BiasAdd → LayerNorm → ResidualAdd` 融合为单个 kernel. 

- **常量折叠(Constant Folding)** : 在编译时预计算所有不依赖输入的操作(如某些 embedding lookup). 

- **消除死代码**: 移除对最终输出没有贡献的计算节点. 

**算子融合(Kernel Fusion)** : 
TensorRT-LLM 的编译器会将整个 Transformer layer 融合为少数几个 mega-kernel. 例如: 
- **Fused MHA(Multi-Head Attention)** : 将 QKV projection、split、transpose、scale、softmax、attention matmul、output projection 融合为一个 kernel. 

- **Fused MLP**: 将 gate projection、up projection、activation、down projection 融合为一个 kernel. 

这种极致的融合使得中间数据完全在寄存器和 Shared Memory 中流动,无需写回 HBM. 

**精度校准与量化**: 
TensorRT-LLM 支持多种量化模式: 
- **Weight-Only Quantization(W4A16/W8A16)** : 权重以 INT4/INT8 存储,激活保持 FP16. 推理时权重动态反量化到 FP16 再与激活相乘. 

- **SmoothQuant**: 通过数学变换将量化的难度从激活侧转移到权重侧,使得 INT8 量化对激活值更友好. 

- **FP8**: 在 Hopper 上原生支持 E4M3/E5M2,无需反量化. 

**CUDA Graph 捕获**: 
TensorRT-LLM 在引擎构建时会自动捕获 CUDA Graph,消除 kernel launch overhead. 这对于小 batch size 的推理尤为重要——当每个 iteration 的计算时间很短时,launch overhead 可能成为主导因素. 

### 5.3 TensorRT-LLM的局限

- **编译时间长**: 构建 TensorRT engine 可能需要数十分钟到数小时(取决于模型大小和优化级别). 

- **灵活性差**: engine 是针对特定 GPU 架构、特定 batch size 范围、特定序列长度编译的. 如果运行时的输入特征与编译假设不符,性能会下降,甚至需要重新编译. 

- **生态锁定**: 深度绑定 NVIDIA 硬件和软件栈,无法迁移到 AMD 或其他加速器. 

- **调试困难**: 编译后的 engine 是一个黑盒,难以分析中间结果或定位数值问题. 

## 6. TGI(Text Generation Inference)

### 6.1 Hugging Face的推理服务方案

TGI 是 Hugging Face 开发的推理服务框架,定位介于 vLLM 的易用性和 TensorRT-LLM 的极致性能之间. 

**核心特性**: 
- **原生支持 Hugging Face 生态**: 与 Transformers 库无缝集成,几乎任何 Hugging Face 上的模型都可以零代码修改部署. 

- **FlashAttention 支持**: 内置 FlashAttention 和 FlashAttention-2 的集成. 

- **Safetensors 加速加载**: 使用 safetensors 格式替代 PyTorch 的 pickle 格式,模型加载速度提升 10 倍以上. 

- **Sharded 推理**: 原生支持张量并行(TP),多卡加载大模型. 

- **流式生成**: 支持 Server-Sent Events(SSE)格式的流式输出. 

**架构设计**: 
TGI 采用多进程架构: 
- **Router 进程**: 接收 HTTP/gRPC 请求,根据负载情况分发给不同的 Worker. 

- **Worker 进程**: 每个 worker 管理一个模型分片(如果使用了 TP)或一个完整的模型副本. Worker 之间通过 NCCL 同步. 

- **Batching**: TGI 支持 continuous batching,但实现相对简单,没有 vLLM 的 PagedAttention 那么激进.

## 7. 连续批处理(Continuous Batching)vs 静态批处理

### 7.1 静态批处理的困境

传统推理系统使用**静态批处理(Static Batching)** : 收集 $N$ 个请求,等所有请求都到达后一次性送入模型,等所有请求都生成完毕后一起返回. 

**问题**: 由于不同请求的输出长度差异巨大,静态批处理受到**"最长请求拖累"**(straggler effect)的严重制约. 

假设 batch 中有 3 个请求: 
- 请求 A: 生成 10 个 token 后结束. 
- 请求 B: 生成 100 个 token 后结束. 
- 请求 C: 生成 1000 个 token 后结束. 

在静态批处理中,请求 A 在生成 10 个 token 后就完成了,但它所在的 batch slot 被"钉死"直到请求 C 完成(990 个 token 之后). 这意味着 GPU 在 90% 的时间里只利用了 2/3 的 batch 容量,然后 67%,然后 33%. 

### 7.2 连续批处理(Continuous Batching / In-flight Batching)

**Continuous Batching** 的核心思想是: **一旦某个请求生成完毕(遇到 EOS token 或达到最大长度),立即从 batch 中移除它,并将 waiting queue 中的新请求加入 batch. **

**vLLM 的实现**: 
1. 每个 iteration,vLLM 的 scheduler 检查 running batch 中是否有请求完成. 
2. 完成的请求被移出,释放其 KV Cache blocks. 
3. Scheduler 尝试从 waiting queue 中挑选可以加入的请求(基于显存预算). 
4. 新的请求被分配 KV Cache blocks,其已经编码的 prompt KV 被加入 batch. 
5. 模型执行一次 forward pass,为 batch 中所有请求(包括新加入的)各生成一个 token. 

**效果**: Continuous batching 使得 GPU 的 batch size 在整个运行期间保持"填满"状态. 当一个短请求完成后,它的位置立即被新请求填补. 这使得 GPU 利用率从静态批处理的 30-50% 提升到 70-90%,吞吐量提升 2-5 倍. 

### 7.3 Iteration-level Scheduling的代价

Continuous batching 并非没有代价: 

- **调度开销**: 每个 iteration 都需要执行 scheduler 逻辑(检查完成请求、选择新请求、分配 blocks). 在 batch size 极大时,CPU 调度可能成为瓶颈. 

- **内存碎片(逻辑层面)** : 虽然 PagedAttention 消除了物理碎片,但动态增删请求使得 attention 计算中的 batching 变得复杂——每个 iteration 需要为不同长度的序列计算 mask. 

- **公平性问题**: 如果不断有短请求涌入,长请求可能 starvation(饿死). vLLM 通过设置最大等待时间来缓解这一问题. 

## 8. 自动扩缩容: 负载均衡与GPU利用率监控

### 8.1 推理服务的负载特征

大模型推理服务的负载具有显著的**时间不均匀性**: 
- **日内波动**: 白天用户活跃,请求量是夜间的 5-10 倍. 

- **突发流量(Bursty Traffic)** : 某个热点事件可能导致请求量在数秒内暴涨 100 倍. 

- **请求异质性**: 不同请求的输入长度(prompt length)和输出长度差异巨大,导致计算需求不可预测. 

### 8.2 自动扩缩容(Auto-scaling)策略

**水平扩缩容(Horizontal Scaling)** : 增加或减少推理实例(Pod/Container)的数量. 

- **触发指标**: GPU 利用率(> 80% 扩容,< 30% 缩容)、请求队列长度(queue depth)、P99 延迟. 

- **冷启动问题**: 新实例需要加载模型权重到显存. 70B 模型的 FP16 权重为 140 GB,即使从本地 NVMe 加载也需要数十秒. 这意味着传统的"秒级扩容"在大模型推理中几乎不可能. 

- **预热实例(Warm Pool)** : 维持一定数量的"待机"实例,权重已加载但未接收流量. 当需要扩容时,这些实例可以立即上线. 代价是空闲时的 GPU 租金. 

**垂直扩缩容(Vertical Scaling)** : 调整单实例的 GPU 数量或类型. 
- 例如,在低峰期使用 A100( cheaper),高峰期切换到 H100(faster). 
- 或者调整 TP 度(Tensor Parallelism degree)——低峰期用 TP=1(单卡),高峰期用 TP=4(多卡)以获得更高吞吐. 

### 8.3 负载均衡与请求路由

当多个推理实例同时运行时,需要**请求路由(Request Routing)** 策略: 

**Round Robin**: 最简单的轮询,依次将请求分发给每个实例. 不考虑实例当前的负载状态. 

**Least Outstanding Requests(LOR)** : 将请求发送给当前未完成的请求数最少的实例. 这比 Round Robin 更智能,因为不同请求的处理时间不同. 

**Session Affinity(会话亲和性)** : 对于多轮对话,同一 session 的后续请求应路由到处理该 session 历史 KV Cache 的实例,以避免跨实例传输 KV Cache(代价极高). vLLM 和 TGI 都支持 prefix caching 来辅助这一策略. 

**Prefill-Decode 分离(Disaggregated Serving)** : 
最新的研究趋势是将推理过程拆分为两个阶段,使用不同类型的 GPU/实例处理: 
- **Prefill 阶段**: 处理输入 prompt,计算 prompt 的 KV Cache. 这是计算密集型的(需要为所有 prompt token 做一次完整的前向传播). 

- **Decode 阶段**: 自回归生成新 token. 这是内存带宽密集型的(每次只生成一个 token,但需要从 KV Cache 读取所有历史 token). 

Prefill 和 Decode 对硬件的需求不同: Prefill 需要高算力(Tensor Core),Decode 需要高内存带宽和显存容量. 将它们分离到专门的实例上,可以获得更好的资源利用率. Moonshot AI(Kimi)和 Google 都在其内部推理基础设施中采用了这一策略. 

### 8.4 GPU利用率监控的关键指标

生产环境中的推理服务需要监控以下指标: 

| 指标 | 定义 | 健康阈值 | 异常含义 |
|------|------|---------|---------|
| **GPU 利用率** | GPU 计算单元的活跃时间占比 | 70-90% | < 50% 说明调度效率低; > 95% 可能接近饱和 |
| **GPU 显存占用** | 已用显存 / 总显存 | < 85% | > 90% 有 OOM 风险 |
| **TTFT** | Time To First Token,从请求到达首个 token 返回 | < 500ms | 高 TTFT 说明 prefill 排队严重 |
| **ITLT** | Inter-Token Latency,相邻 token 的间隔 | < 50ms | 高 ITLT 说明 decode 阶段瓶颈 |
| **Throughput** | 每秒生成的 token 数 | 视模型而定 | 下降说明需要扩容 |
| **Batch Size** | 当前 running batch 中的请求数 | 接近 max | 长期低于 max 说明流量不足 |

## 9. 局限性与边界条件

1. **内存-计算的此消彼长**: 推理优化的很多手段是零和博弈. 量化节省了显存和带宽,但可能损害输出质量; continuous batching 提高了吞吐,但增加了单个请求的 TTFT(因为新请求需要等待当前 iteration 完成). 选择哪种优化组合需要根据业务场景(延迟敏感 vs 成本敏感)来权衡. 

2. **长上下文的天花板**: 即使 PagedAttention 完美解决了显存管理,Transformer 的 $O(n^2)$ attention 复杂度仍然是根本性约束. 当上下文超过 100K token 时,attention 的计算时间会主导整个推理延迟. 这需要架构层面的创新(如线性注意力、状态空间模型)来解决,而非纯工程优化. 

3. **多模态推理的复杂性**: 当前 vLLM、TensorRT-LLM 主要针对文本 LLM. 当涉及图像、视频、音频时,推理管线变得更加复杂(需要视觉Encoder 、音频Encoder 、跨模态投影层),现有的批处理和调度策略需要大幅扩展. 

## 10. 演进与承上启下

本节我们从量化压缩走向编译优化,从 PagedAttention 的显存革命走向 continuous batching 的调度艺术,最终抵达生产环境的自动扩缩容. 这些技术共同构成了大模型从"实验室玩具"走向"工业级服务"的工程桥梁. 

然而,本章(第 9 章)只是我们工程化之旅的起点. 在真实的工业环境中,还有更多没有触及的深水区: **训练稳定性**(loss spike 的诊断与恢复)、**数据管线**(PB 级数据的清洗与加载)、**模型版本管理**(CI/CD for ML)、**安全与隐私**(联邦学习、差分隐私). 这些主题超出了本章的范围,但掌握本章的四层技术栈——硬件、系统软件、分布式训练、推理服务——已经为你奠定了理解任何 AI 工程化问题的坚实基础. 

## 11. 总结与参考文献

1. **推理优化是一个五层技术栈**: 量化压缩 → 编译优化(算子融合) → 内存优化(PagedAttention) → 批处理优化(Continuous Batching) → 服务调度(自动扩缩容). 

2. **vLLM 通过 PagedAttention 革命性地解决了 KV Cache 的显存碎片问题**,配合 continuous batching 实现了业界领先的吞吐量. 适合追求高吞吐、易部署的场景. 

3. **TensorRT-LLM 通过深度学习编译器实现了极致的低延迟**,将多层操作融合为 mega-kernel,并支持全面的量化方案. 适合延迟敏感、愿意投入工程精力的生产环境. 

4. **TGI 在 Hugging Face 生态中提供了最佳的易用性**,适合快速原型、研究探索和多模态模型部署. 

5. **Continuous Batching 是推理吞吐的倍增器**,它通过动态替换 batch 中已完成的请求,将 GPU 利用率从静态批处理的 30-50% 提升到 70-90%. 

6. **自动扩缩容面临冷启动和负载异质性的挑战**,Prefill-Decode 分离是下一代推理架构的重要方向.
