# 9.1 · 硬件基础: GPU架构、内存层次与算力模型

## 1. 背景与核心痛点

大语言模型的 scaling law 告诉我们: 模型参数量每增加一个数量级,能力就可能产生质的飞跃. 但参数量的增长背后,是算力需求的指数级爆炸. GPT-3 的 175B 参数需要约 3.14×10²³ 次浮点运算来完成一次完整训练; GPT-4 据传用了约 1.8×10²⁵ 次运算——这两个数字之间隔了整整两个数量级. 

**核心痛点**: 算法研究者设计出一个需要 10²⁵ 次运算的训练任务后,这个任务必须被"翻译"成晶体管的开关动作. 翻译的质量——即硬件利用率——直接决定了你是用 1000 张 GPU 跑一个月,还是用 10000 张跑一年. 而翻译的质量,又完全取决于你对底层硬件的理解深度. 

**家谱定位**: 本章是整个 LLM 指南中最"底层"的一章. 如果说 Transformer 注意力机制是"数学层",CUDA 是"软件层",那么这一节就是"物理层". 理解 GPU 如何执行矩阵乘法、HBM 的带宽瓶颈在哪里、多机通信的延迟如何吞噬训练效率,是后续所有优化策略(量化、并行、kernel fusion)的物理前提. 

## 2. 为什么重要

2024-2025 年,NVIDIA 的数据中心业务收入连续多个季度超过 200 亿美元,其中绝大部分来自用于 AI 训练的 GPU. 这不仅仅是商业现象,它揭示了一个残酷事实: **大模型的竞争,本质上是算力工程化的竞争**. 同样一个 Llama-3 架构,在 poorly engineered 的集群上训练可能需要 2 倍的时间和 2 倍的电力,而在精心调优的集群上,同样的预算可以训练出更大的模型或用更多的数据. 

更重要的是,硬件特性直接塑造了算法设计. FlashAttention 的诞生,本质上是因为 HBM 带宽(1.5-3 TB/s)比 SRAM 带宽(10-20 TB/s)慢了一个数量级,而标准注意力实现反复在两者之间搬运数据. 如果不理解这个硬件事实,就永远无法理解为什么 FlashAttention 的 tiling 策略能带来 2-4 倍的加速. 

## 3. GPU架构演进: 从Volta到Blackwell

### 3.1 Volta(2017): Tensor Core的诞生

在 Volta 架构之前,GPU 执行矩阵乘法的方式与执行普通浮点运算没有本质区别——每个 CUDA core 独立处理一个 FMA(乘加)运算. 这种设计对于图形渲染足够高效,但对于深度学习中占绝对主导的 $A \times B = C$ 矩阵运算,却存在严重的效率浪费. 

Volta 架构引入了**Tensor Core**——一种专门为矩阵运算设计的专用计算单元. 一个 Tensor Core 可以在一个时钟周期内完成一个 $4 \times 4 \times 4$ 的 FP16 矩阵乘法累加(即 $D = A \times B + C$,其中 $A,B,C,D$ 都是 $4 \times 4$ 矩阵). 这相当于在一个周期内完成了 64 次乘加运算. 

Tensor Core 不是通用计算单元,它是"硬连线"的矩阵乘法加速器. 就像 ASIC 之于 CPU,Tensor Core 之于 CUDA core 也是同样的专用化逻辑. V100(Volta 架构旗舰)配备了 640 个 Tensor Core,使其在混合精度训练中的峰值算力达到了 125 TFLOPS(FP16),是纯 FP32 CUDA core 算力的 8 倍. 

### 3.2 Ampere(2020): 稀疏性与多精度

Ampere 架构(A100)在三个维度上实现了重大飞跃: 

**第一,第三代 Tensor Core**. 支持 FP64、FP32、FP16、BF16、INT8 乃至 TF32 等多种精度格式. TF32(TensorFloat-32)是 Ampere 引入的一种巧妙设计: 它使用与 FP32 相同的 19-bit 指数范围(保证动态范围不变),但将尾数压缩到 10-bit(与 FP16 相同). 这使得 TF32 在几乎不损失训练稳定性的前提下,享受到 Tensor Core 的加速. 

**第二,结构化稀疏性(Structured Sparsity)** . Ampere 支持 2:4 结构化稀疏——在每 4 个权重中允许 2 个为零,Tensor Core 可以跳过这些零值进行计算,理论上可将稀疏矩阵乘法的吞吐量翻倍. 

**第三,多实例 GPU(MIG)** . A100 可以被物理切分为最多 7 个独立的 GPU 实例,每个实例拥有独立的显存、缓存和计算资源. 这对于云计算厂商至关重要——一张 A100 可以被多个租户共享,而互不干扰. 

A100 的显存配置为 40GB 或 80GB HBM2e,内存带宽达到了 1.6-2.0 TB/s. 它成为了 2020-2023 年间大模型训练的绝对主力. 

### 3.3 Hopper(2022): Transformer Engine与分布式革命的硬件支撑

Hopper 架构(H100)是专为 Transformer 和大模型时代设计的. 其核心创新包括: 

**Transformer Engine(TE)** : 这是硬件与软件协同设计的典范. TE 不是单纯的硬件单元,而是一套硬件-软件混合系统,能够在 FP8 和 FP16/BF16 之间动态切换. 在训练的前向传播中,TE 会逐层分析激活值的分布,自动选择不会导致梯度下溢或上溢的最低精度格式. 这需要在硬件层面支持 FP8 的矩阵乘法(H100 的 Tensor Core 原生支持 E4M3 和 E5M2 两种 FP8 格式),同时在软件层面维护一个 scaling factor 的历史记录. 

**第四代 NVLink**: H100 支持多达 18 条 NVLink 4 链路,单链路带宽 25 GB/s,总计 900 GB/s 的 GPU-to-GPU 带宽. 配合 NVSwitch,单机 8 卡 H100(如 DGX H100)可以实现全连接的拓扑,任意两张 GPU 之间的通信都无需经过 CPU 或 PCIe. 

**DPX 指令集**: Hopper 引入了动态编程加速指令(Dynamic Programming X),专门优化序列比对、编辑距离计算等动态规划算法. 虽然对 Transformer 训练的直接帮助有限,但它标志着 GPU 从"纯矩阵乘法加速器"向"更通用 AI 加速器"的演进. 

H100 的 FP8 峰值算力达到了惊人的 3958 TFLOPS(使用稀疏性时),FP16 为 1979 TFLOPS,显存带宽 3.35 TB/s(HBM3). 

### 3.4 Blackwell(2024): 第二代Transformer Engine与芯片间互联

Blackwell 架构(B100/B200/GB200)代表了 NVIDIA 在 AI 基础设施上的最新一代设计: 

**第二代 Transformer Engine**: 支持更细粒度的 FP4 和 FP6 精度,以及第二代 FP8 累加精度. 更重要的是,Blackwell 引入了**微张量缩放(Micro-Tensor Scaling)** ——不再对整个矩阵使用统一的 scaling factor,而是对矩阵的子块(micro-tensor)分别计算和缩放,极大地降低了 FP8 量化带来的精度损失. 

**第五代 NVLink**: 单链路带宽翻倍至 50 GB/s,单 GPU 支持 18 条链路,总带宽达到 1.8 TB/s. Blackwell 还引入了 NVLink Switch Chip,支持单机箱内 72 个 GPU 的全连接通信,延迟和带宽都远超以往的 NVSwitch 方案. 

**可靠性引擎**: 大模型训练中,万卡集群每周都可能发生数次硬件故障(ECC 错误、GPU 掉卡、网络链路中断). Blackwell 引入了基于 RAS(Reliability, Availability, Serviceability)的硬件级错误检测和隔离机制,可以在不中断整个训练任务的情况下隔离故障 GPU. 

B200 的 FP4 峰值算力高达 20 PFLOPS(使用稀疏性),FP8 为 10 PFLOPS,显存配置为 192GB HBM3e,带宽 8 TB/s.

## 4. Tensor Core原理: 矩阵乘法的硬件加速

### 4.1 从CUDA Core到Tensor Core的范式转变

标准的 CUDA core 执行的是**标量 FMA(Fused Multiply-Add)** : 给定三个标量 $a, b, c$,计算 $d = a \times b + c$. 一条 FMA 指令在硬件上需要一个乘法器和一个加法器,延迟约为 4 个时钟周期. 

当需要计算矩阵乘法 $C = A \times B$(其中 $A \in \mathbb{R}^{M \times K}, B \in \mathbb{R}^{K \times N}$)时,标准方法需要 $M \times N \times K$ 次独立的标量 FMA 运算. 即便有数千个 CUDA core 并行执行,这种细粒度的标量操作也存在严重的指令调度和数据搬运开销. 

Tensor Core 将运算粒度从**标量**提升到了**矩阵块(matrix tile)** . 以第一代 Tensor Core(Volta)为例,它可以在一个指令周期内完成以下运算: 

$$
D_{4 \times 4} = A_{4 \times 4} \times B_{4 \times 4} + C_{4 \times 4} \tag{1}
$$

其中所有矩阵元素都是 FP16. 这意味着一个周期内完成了 $4 \times 4 \times 4 = 64$ 次乘加运算. 

**物理实现**: Tensor Core 内部并不是"64 个 CUDA core 打包在一起"那么简单. 它是一个高度流水线化的**脉动阵列(Systolic Array)** ——数据从阵列的一侧流入,在流经每个处理单元时逐步累加,最终从另一侧流出结果. 这种设计的最大优势是: 数据一旦从寄存器或共享内存加载到阵列中,就不需要在计算过程中反复读取内存,而是沿着硬件流水线"流动",极大地降低了内存访问次数. 

### 4.2 WMMA与MMA指令

程序员并不能直接操控 Tensor Core 的脉动阵列. NVIDIA 提供了两层编程接口: 

**WMMA(Warp Matrix Multiply Accumulate)** : CUDA C++ 中的高层 API. 一个 warp(32 个线程)协作调用一次 `wmma::mma_sync`,就能触发 Tensor Core 执行一次矩阵乘法累加. WMMA 将 $16 \times 16 \times 16$ 的矩阵块(或 $32 \times 8 \times 16$ 等变体)映射到 warp 的 32 个线程上,每个线程负责加载和存储矩阵块的一部分. 

**MMA PTX 指令**: 更底层的汇编级指令,允许更精细地控制寄存器分配和数据通路. Triton 编译器在优化某些特殊 kernel 时,会生成 MMA 指令而非 WMMA. 

### 4.3 Tensor Core的精度演进与混合精度训练

Tensor Core 对低精度的支持直接催生了**混合精度训练(Mixed Precision Training)** 这一革命性技术: 

- **前向传播**: 权重和激活使用 FP16(或 BF16/FP8),享受 Tensor Core 的 2×(相对 FP32)乃至 8×(相对 FP32 的 FP8)加速. 

- **反向传播**: 梯度同样以 FP16 计算. 

- **权重更新**: 优化器状态(Adam 的一阶和二阶动量)以 FP32 存储和更新,防止精度下溢. 

这种"前向用 FP16,更新用 FP32"的策略,使得训练速度接近纯 FP16 的理论上限,同时保持了 FP32 的数值稳定性. 

BF16(Brain Floating Point 16)是 Google 为 TPU 设计、后被 NVIDIA 采纳的格式. 它与 FP16 同样是 16-bit,但将 FP16 的 10-bit 尾数缩减为 7-bit,将节省的 3-bit 加到指数上,使指数范围与 FP32 完全相同. 这意味着 BF16 不会因为数值范围不足而溢出,对深度学习的训练稳定性更加友好. Ampere 及之后的 Tensor Core 原生支持 BF16. 

FP8(E4M3 和 E5M2)则是 Hopper 引入的更激进压缩. E4M3 用 4-bit 指数和 3-bit 尾数表示激活值,E5M2 用 5-bit 指数和 2-bit 尾数表示梯度. Hopper 的 Transformer Engine 会在每一层自动选择使用 E4M3 还是 E5M2,以及是否需要回退到 FP16. 

## 5. HBM vs GDDR vs DDR: 内存带宽的层级战争

### 5.1 为什么内存带宽是瓶颈

大模型训练和推理中,一个被反复验证的事实是: **计算速度的增长远快于内存带宽的增长**. NVIDIA H100 的 FP16 峰值算力是 989 TFLOPS,但它的 HBM3 带宽只有 3.35 TB/s. 如果我们计算"算力/带宽比"(即每读取 1 byte 数据能进行多少次运算),会发现这个比值已经达到了约 300:1. 

这意味着: 如果一个 kernel 每读取 1 byte 只进行 1 次运算,那么即便 GPU 的计算单元完全空闲,内存带宽也会成为瓶颈,kernel 的实际性能将只有峰值算力的 1/300. 

Transformer 中的注意力机制恰恰是这种"内存密集型"操作的典型代表: 标准实现需要反复从 HBM 读取和写入 $O(n^2)$ 大小的注意力矩阵,而每个元素只参与少数几次运算. 

### 5.2 三类内存的物理对比

| 指标 | DDR5(系统内存) | GDDR6/X(游戏显卡显存) | HBM3/HBM3e(数据中心 GPU) |
|------|------------------|------------------------|---------------------------|
| 单颗容量 | 16-64 GB | 16-24 GB | 24-36 GB(堆叠) |
| 单设备总容量 | 可达 TB 级 | 24-48 GB | 80-192 GB |
| 单 pin 数据率 | 4.8-6.4 Gbps | 19-24 Gbps | 6.4-9.2 Gbps |
| 位宽(bus width) | 64-bit | 256-384-bit | 4096-bit(1024-bit × 4 堆叠) |
| 总带宽(单设备) | 38-51 GB/s | 600-1000 GB/s | 1500-8000 GB/s |
| 延迟 | ~80-100 ns | ~20-30 ns | ~10-15 ns |
| 功耗/GB | 较低 | 中等 | 较高(但带宽/功耗比最优) |
| 典型应用场景 | CPU 主存 | 消费级 GPU | 数据中心 AI 加速器 |

**关键洞察**: HBM 的高带宽不是来自更快的单 pin 速度(实际上 HBM 的单 pin 速度比 GDDR 慢),而是来自**极端的并行位宽**. HBM 采用 3D 堆叠技术,将 4-12 层 DRAM die 垂直堆叠在逻辑 die 之上,通过硅通孔(TSV, Through-Silicon Via)连接. 每层 die 提供 1024-bit 的数据通道,4 层堆叠就达到了 4096-bit 的恐怖位宽——是 GDDR6 的 10 倍以上. 

这种设计的代价是**容量密度受限**(堆叠层数受限于散热和良率)和**成本极高**(HBM 的价格是普通 DDR 的 10-20 倍). 这也是为什么消费级显卡仍在使用 GDDR,而只有数据中心 GPU 才配备 HBM. 

### 5.3 片上内存: 寄存器、L1 Cache与Shared Memory

GPU 的内存层次结构中,比 HBM 更快的是**片上内存(on-chip memory)** : 

**寄存器(Registers)** : 每个 CUDA core / Tensor Core 拥有私有的寄存器文件. A100 每个 SM(Streaming Multiprocessor)拥有 256 KB 寄存器文件,延迟仅为 1 个时钟周期. 寄存器是编译器自动管理的,程序员通常不直接控制. 

**L1 Cache / Shared Memory**: 每个 SM 拥有 192 KB(A100)或 256 KB(H100)的可配置存储空间,可以被划分为 L1 Cache 和 Shared Memory. Shared Memory 是**程序员可控的高速 scratchpad memory**——CUDA kernel 可以显式地将数据从 HBM 加载到 Shared Memory,然后在 Shared Memory 上进行多次复用. FlashAttention 的核心优化,正是将 attention 矩阵的 tile 加载到 Shared Memory,在片上完成 softmax 和矩阵乘法,避免反复访问 HBM. 

**L2 Cache**: 所有 SM 共享的二级缓存. A100 拥有 40 MB L2 Cache,H100 拥有 50 MB. L2 Cache 对程序员透明,但 kernel 设计(如内存访问模式)会显著影响 L2 Cache 的命中率. 

![GPU 内存金字塔架构与参数分布](images/gpu_memory_hierarchy.png)

> **图 9.3 GPU 层次化内存体系(Memory Hierarchy)**
> * **寄存器(Registers)**：位于计算单元绝对前线, 带宽数十 TB/s, 延迟仅约 1 时钟周期, 容量受限. 
> * **L1 缓存/共享内存(Shared Memory)**：由 SM 计算单元私有或由程序员显式操控的高速缓冲区, 延迟约为 20 个周期. 通过巧妙的 tiling 减少与主存 HBM 的直接数据拷贝. 
> * **L2 缓存(L2 Cache)**：全局 SM 共享, 容量约为 40-50 MB, 延迟在 200 周期左右, 是算子融合与重计算的缓存中介. 
> * **高带宽显存(HBM3/HBM3e)**：大模型主存储介质, 带宽达 1.5-8 TB/s, 延迟约 500 周期, 容量决定了多卡分布式拆分的基本单位. 
> * **系统主存(DDR5)**：CPU 端控制的外部系统内存, 容量达数 TB 级别, 但延迟超过 1000 周期, 常用于超限层 offloading. 


## 6. NVLink与InfiniBand: 多机通信硬件

### 6.1 PCIe的瓶颈

在讨论多 GPU 互联之前,必须理解为什么传统的 PCIe 不够用. 

PCIe 4.0 x16 的单向带宽为 32 GB/s,PCIe 5.0 x16 为 64 GB/s. 听起来不小,但与 HBM 的 3.35 TB/s 相比,差了整整两个数量级. 当一个训练任务需要将梯度从一张 GPU 同步到另一张 GPU 时,如果走 PCIe,通信时间将成为训练迭代时间的主要组成部分. 

更致命的是,PCIe 通信通常需要经过 CPU 和系统内存(除非使用 GPUDirect P2P). 这意味着数据需要从 GPU → PCIe → CPU 内存 → PCIe → GPU,路径长、延迟高. 

### 6.2 NVLink: GPU之间的专用高速公路

NVLink 是 NVIDIA 开发的专有高速互联技术,旨在绕过 PCIe 的瓶颈: 

| 代际 | 单链路带宽 | 单 GPU 最大链路数 | 单 GPU 总带宽 | 关键特性 |
|------|-----------|------------------|--------------|----------|
| NVLink 1.0 | 20 GB/s | 4 | 80 GB/s | P100, 首次引入 |
| NVLink 2.0 | 25 GB/s | 6 | 150 GB/s | V100 |
| NVLink 3.0 | 50 GB/s | 12 | 600 GB/s | A100 |
| NVLink 4.0 | 25 GB/s (双向50) | 18 | 900 GB/s | H100 |
| NVLink 5.0 | 50 GB/s (双向100) | 18 | 1800 GB/s | B200 |

注意 NVLink 4.0 的单链路带宽"下降"到 25 GB/s 是一个统计口径的变化: NVLink 3.0 的 50 GB/s 是单向带宽,而 NVLink 4.0 的 25 GB/s 是单向、50 GB/s 是双向. 实际上 NVLink 4.0 的单向带宽与 3.0 相同. 

**NVSwitch**: 如果只有两两之间的 NVLink,8 张 GPU 的拓扑将非常受限. NVSwitch 是一个专用的交换芯片,它可以将所有连接的 GPU 的 NVLink 汇聚到一个全连接的网络中. 在 DGX A100 中,6 个 NVSwitch 芯片将 8 颗 A100 连接成一个 fully-connected 的拓扑,任意两张 GPU 之间都有 600 GB/s 的直接带宽. 

**NVLink + NVSwitch 的物理意义**: 在单机 8 卡训练中,模型并行(张量并行)的通信可以完全通过 NVLink 进行,不需要经过 CPU. 这使得 TP=8 在单机内部几乎"免费"——通信延迟被压缩到微秒级. 

### 6.3 InfiniBand: 跨节点通信的王者

当训练规模超过单机 8 卡,就需要多机互联. 此时 NVLink 已经不够(它只能连接同一机箱内的 GPU),必须依赖网络技术. 

**InfiniBand(IB)** 是当前 AI 数据中心的事实标准. 它由 Mellanox(2019 年被 NVIDIA 收购)主导,提供以下关键特性: 

- **超高带宽**: NVIDIA ConnectX-7 网卡支持 400 Gbps(50 GB/s)的端口带宽,下一代 ConnectX-8 将支持 800 Gbps. 

- **极低延迟**: IB 的端到端延迟可以低至 600 纳秒(ns),远低于以太网的 10-50 微秒(μs). 

- **内核旁路(Kernel Bypass)** : IB 支持 RDMA(Remote Direct Memory Access),允许一张 GPU 直接读取/写入另一台机器上 GPU 的内存,完全绕过 CPU 和操作系统内核. NCCL(NVIDIA Collective Communications Library)利用 RDMA 实现了高效的 all-reduce 和 all-gather. 

- **无损网络(Lossless Fabric)** : IB 基于 credit-based 流控,保证网络不会因为拥塞而丢包. 这对于大模型训练中巨大的同步通信量至关重要——丢包导致的重传会彻底摧毁训练效率. 

**InfiniBand vs RoCE**: RoCE(RDMA over Converged Ethernet)是以太网上实现 RDMA 的替代方案,成本更低,但延迟和稳定性不如 IB. 对于追求极致性能的超大规模训练集群(如 xAI 的 Colossus、Meta 的 RSC),IB 仍是首选. 

### 6.4 大规模集群的拓扑设计

一个典型的万卡集群网络拓扑如下: 

- **节点内部**: 8 张 GPU 通过 NVLink + NVSwitch 全连接. 

- **机架内部(~16-32 节点)** : 通过 IB 交换机(如 NVIDIA Quantum-2)连接成一个 fat-tree 或 dragonfly+ 拓扑. 

- **跨机架**: 通过更高层的 IB 交换机汇聚. 

在这种拓扑中,**节点内带宽(NVLink, 900 GB/s)远高于节点间带宽(IB, 50 GB/s)** . 这正是分布式训练策略设计中"张量并行在节点内,流水线并行和数据并行跨节点"的物理根源. 

## 7. 算力瓶颈分析: Roofline Model

### 7.1 什么是Roofline Model

Roofline Model 是一个简洁而强大的可视化工具,用于分析一个计算 kernel 的性能瓶颈到底在哪里——是**计算瓶颈**(compute-bound,峰值算力不够)还是**内存瓶颈**(memory-bound,数据搬运速度不够). 

模型的坐标系非常简单: 
- **横轴**: 计算强度(Arithmetic Intensity),定义为每读取 1 byte 数据所进行的浮点运算次数(FLOPs/byte). 

- **纵轴**: 实际性能(Performance),单位为 GFLOPS. 

图中有两条"屋顶线": 
1. **水平屋顶**(平顶): $y = \text{Peak GFLOPS}$,代表 GPU 的峰值算力. 无论计算强度多高,实际性能都不可能超过这条线. 

2. **斜屋顶**(斜坡): $y = \text{Bandwidth} \times x$,代表内存带宽限制. 如果计算强度为 $x$,那么内存带宽能支撑的最大性能就是带宽乘以 $x$. 

两条线的交点称为**ridge point(脊点)** . 当 kernel 的计算强度大于脊点对应的值时,kernel 是计算瓶颈的; 小于脊点时,是内存瓶颈的. 

### 7.2 用Roofline分析Transformer操作

让我们计算 Transformer 中几种典型操作的计算强度: 

**矩阵乘法 $C = A \times B$**($A \in \mathbb{R}^{M \times K}, B \in \mathbb{R}^{K \times N}$): 
- 运算量: $2 \times M \times N \times K$ FLOPs(乘和加各算一次). 
- 内存访问量: 需要读取 $A$($M \times K$)、$B$($K \times N$),写入 $C$($M \times N$),共 $(MK + KN + MN)$ 个元素. 假设每个元素 2 byte(FP16),总访问量为 $2(MK + KN + MN)$ bytes. 
- 当 $M=N=K=4096$ 时,计算强度约为 $2 \times 4096^3 / (2 \times 3 \times 4096^2) \approx 1365$ FLOPs/byte. 

H100 的 ridge point(FP16 Tensor Core 峰值 989 TFLOPS / HBM 带宽 3.35 TB/s)约为 $989000 / 3350 \approx 295$ FLOPs/byte. 因此大矩阵乘法是**计算瓶颈**的,可以充分利用 Tensor Core. 

**注意力 $P = \text{softmax}(QK^T/\sqrt{d})$: **
- 标准实现中,$QK^T$ 的计算强度尚可,但 softmax 需要对 $O(n^2)$ 大小的矩阵进行归一化,涉及大量逐元素操作(除法、指数)和多次 HBM 读写. 
- 标准 attention 的计算强度约为 $O(n)$ FLOPs/byte,当 $n=2048$ 时往往**远低于 ridge point**,是严重的**内存瓶颈**. 

**Layer Normalization、Activation、Dropout**: 
- 这些都是逐元素操作,计算强度极低(通常 < 10 FLOPs/byte),是典型的**内存瓶颈**操作. 

### 7.3 Roofline模型的工程指导意义

Roofline Model 告诉我们优化优先级: 

1. **对于计算瓶颈的操作**(大矩阵乘法): 优化方向是提升 Tensor Core 利用率——使用更大的 tile 尺寸、避免 warp divergence、减少寄存器 spill. 

2. **对于内存瓶颈的操作**(attention、layernorm): 优化方向是减少内存访问量——kernel fusion(将多个逐元素操作合并为一个 kernel)、使用 Shared Memory 缓存、量化(用 FP8/INT8 减少数据量). 

FlashAttention 的突破性价值,正是将一个原本内存瓶颈的 attention 操作,通过 tiling 和 recomputation 策略,转变为更接近计算瓶颈的操作. 

## 8. 内存层次结构: 从寄存器到HBM的完整旅程

### 8.1 寄存器: 计算的绝对前线

每个 CUDA core 拥有 65536 个 32-bit 寄存器(即 256 KB). 当一个 warp(32 个线程)执行时,它所需的寄存器被静态分配. 如果 kernel 使用了太多寄存器,编译器会将多余的数据"溢出"(spill)到 L1 cache 或更慢的内存中,造成严重的性能损失. 

Tensor Core 的 MMA 指令对寄存器有严格的使用模式: 调用 `wmma::mma_sync` 时,输入矩阵 $A, B, C$ 和输出矩阵 $D$ 必须存放在 warp 的寄存器中. 这意味着在调用 MMA 之前,数据必须从 Shared Memory 或 L1 加载到寄存器; MMA 完成后,结果也必须从寄存器写回. 

### 8.2 Shared Memory: 程序员控制的高速缓存

Shared Memory 是 GPU 内存层次中最关键的一层,因为它**完全由程序员控制**. 

每个 SM 的 Shared Memory 大小有限(A100: 164 KB 可用,H100: 228 KB). CUDA 程序员通过 `__shared__` 关键字显式声明 Shared Memory 数组. 一个典型的使用模式是: 

1. **加载阶段**: warp 中的线程协作,将 HBM 中的数据块(tile)加载到 Shared Memory. 

2. **计算阶段**: Tensor Core 或 CUDA core 从 Shared Memory 读取数据进行计算. 

3. **存储阶段**: 将结果写回 Shared Memory,再协作写回 HBM. 

**bank conflict**: Shared Memory 被划分为 32 个 bank(对应 warp 的 32 个线程). 如果同一个 warp 中的多个线程同时访问同一个 bank 的不同地址,就会发生 bank conflict,导致访问串行化. 精心设计的 kernel(如 CUTLASS 中的矩阵乘法)会通过 padding 和地址对齐来避免 bank conflict. 

### 8.3 L1 Cache与L2 Cache: 透明的加速层

L1 Cache 与 Shared Memory 共享物理存储空间,由编译器或运行时自动管理. 对于没有显式使用 Shared Memory 的 kernel,L1 会自动缓存从 HBM 读取的数据. 

L2 Cache 是全局的,所有 SM 共享. 它的主要作用是: 
- 缓存 HBM 读取,减少对 HBM 的重复访问. 
- 作为不同 SM 之间数据交换的中介(当一个 warp 写入的数据被另一个 warp 读取时). 

L2 Cache 的大小(A100: 40 MB, H100: 50 MB)对 attention 操作尤为重要. FlashAttention-2 的 split-k 策略会尽可能利用 L2 Cache 来缓存 KV Cache 的 tile,避免重复从 HBM 读取. 

### 8.4 HBM: 主存储器与容量瓶颈

HBM 是 GPU 上最大但最慢的内存. 在大模型训练中,HBM 容量是首要瓶颈: 

- **存储模型参数**: 一个 70B 参数的模型,FP16 存储需要 140 GB. 单张 H100(80 GB)根本放不下,必须分布式存储. 

- **存储优化器状态**: Adam 优化器需要维护每个参数的 momentum 和 variance,额外需要 2× 参数量的 FP32 存储. 70B 模型的优化器状态需要 420 GB. 

- **存储激活值**: 反向传播需要前向传播的中间结果(激活值). 一个 batch size 为 1 的序列,在 70B 模型上的激活值可能达到数十 GB. 

这就是为什么 ZeRO、FSDP 和激活值重计算(activation checkpointing)成为大模型训练的标配——它们本质上都是在**用计算换显存**,因为 HBM 的容量增长速度(每代 1.5-2×)远慢于模型参数量的增长速度(每年 10×). 

## 9. 局限性与边界条件

硬件层面的理解虽然重要,但也存在边界: 

1. **架构特异性**: 本文大量讨论 NVIDIA GPU,因为当前 AI 训练几乎被 NVIDIA 垄断. 但 AMD MI300X(192 GB HBM3, 5.3 TB/s 带宽)、Google TPU v5p(95 TFLOPS BF16, 拥有巨大的片上 SRAM)以及新兴的 Cerebras、SambaNova 等架构,正在挑战这一垄断. 不同的硬件特性会导致完全不同的最优算法选择. 

2. **静态分析的局限**: Roofline Model 是静态分析工具,它假设 kernel 的执行不受其他 kernel 或系统活动的影响. 在多租户云环境中,GPU 可能被多个任务共享,cache 和内存带宽都被争抢,实际性能会远低于理论值. 

3. **编译器优化的不确定性**: 现代编译器(如 nvcc、Triton 编译器、XLA)会进行激进的优化,包括自动 kernel fusion、循环展开、寄存器分配优化等. 手写 kernel 的性能预测可能因编译器版本不同而产生 20-50% 的偏差. 

## 10. 演进与承上启下

硬件是基础设施的最底层,但它不会自动被算法利用. 从 Volta 的 Tensor Core 到 Hopper 的 Transformer Engine,每一代硬件都需要配套的系统软件来"解锁"其性能. 

下一节(9.2 系统软件),我们将从晶体管走向代码——探讨 CUDA 如何将"并行计算"这一抽象概念映射到 GPU 的物理执行单元上,cuBLAS/cuDNN/NCCL 如何封装了硬件的最优使用模式,以及 Triton 如何让 Python 程序员也能写出接近硬件极限的 kernel. 

## 11. 总结与参考文献

1. **GPU 架构四代演进**: Volta 引入 Tensor Core,Ampere 引入 TF32 和稀疏性,Hopper 引入 Transformer Engine 和 FP8,Blackwell 引入微张量缩放和 NVLink 5. 每一代的算力增长都伴随着精度的"降级"和专用化程度的提升. 

2. **内存层次是性能的第一性原理**: 从寄存器到 Shared Memory 到 L2 到 HBM,容量增长1000倍的同时延迟增长1000倍. 优化的本质是让计算尽可能在靠近计算单元的高速层级上完成. 

3. **互联技术决定分布式天花板**: NVLink(节点内 900 GB/s)和 InfiniBand(节点间 50 GB/s)之间的巨大带宽差距,是分布式并行策略设计的物理约束. 

4. **Roofline Model 提供瓶颈诊断框架**: 通过计算强度分析,可以判断一个操作是 compute-bound 还是 memory-bound,从而指导优化方向.
