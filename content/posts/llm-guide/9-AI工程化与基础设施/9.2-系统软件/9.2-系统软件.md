# 9.2 · 系统软件: CUDA编程模型、GPU库生态与Triton

## 1. 背景与核心痛点

上一节我们解剖了 GPU 的物理硬件——Tensor Core 的脉动阵列、HBM 的 3D 堆叠、NVLink 的高速通道. 但硬件本身只是一堆沉默的晶体管,它不会自动将 PyTorch 的 `matmul` 翻译成寄存器间的数据流动. 

**核心痛点**: 在硬件和深度学习框架(PyTorch、TensorFlow)之间,存在一个巨大的"抽象鸿沟". 框架层面的 `nn.Linear` 看起来只是简单的矩阵乘法,但要让它在 H100 上以接近 989 TFLOPS 的峰值效率运行,需要解决以下问题: 
- 如何将 $4096 \times 4096$ 的矩阵切分成适合 Tensor Core 的 $128 \times 128$ tile？
- 如何在 Shared Memory 中安排数据的 staging,使得内存访问模式是合并的(coalesced)？
- 如何调度数千个 warp 的执行顺序,隐藏内存加载的延迟？

这些问题的答案,藏在**系统软件层**中. CUDA 是通往 GPU 的门户,cuBLAS/cuDNN/NCCL 是工业界数十年优化的结晶,而 Triton 则是新一代的"Python 级 GPU 编程语言",正在 democratize kernel 开发. 

**家谱定位**: 如果把硬件层比作"发动机",系统软件层就是"变速箱和传动系统". 它决定了发动机的动力能否高效传递到车轮. 不理解这一层,就无法解释为什么两个在数学上等价的 PyTorch 实现,性能可能相差 10 倍. 

## 2. 为什么重要

在大模型训练中,**系统软件的效率直接等于金钱**. 一个 poorly optimized 的 distributed data loader 可能让 GPU 的利用率只有 30%——这意味着你租了 100 张 H100,实际上只获得了 30 张的有效算力,每小时白白烧掉数千美元. 

更隐蔽的是,系统软件的设计缺陷往往表现为"奇怪的"训练行为: 
- loss 曲线出现周期性尖峰？可能是 NCCL all-reduce 的同步点与数据加载不同步导致的. 
- 显存占用随时间缓慢增长直到 OOM？可能是 PyTorch 的 CUDA memory allocator 的碎片化问题. 
- 同样的模型在 A100 和 H100 上速度提升不如预期？可能是 kernel 没有充分利用 Hopper 的新特性(如 TMA, Tensor Memory Accelerator). 

## 3. CUDA编程模型: Grid → Block → Thread → Warp

### 3.1 层次化并行结构

CUDA 的编程模型是一个精心设计的四层层次结构,它将大规模的并行计算映射到 GPU 的物理执行单元上: 

**Kernel(核函数)** : CUDA 程序的基本执行单元. 一个 kernel 就是一个在 GPU 上并行执行的函数. 当程序员在 CPU(host)上调用一个 kernel 时,CUDA runtime 会在 GPU(device)上启动数千乃至数百万个线程来执行这个函数的实例. 

**Grid(网格)** : 一个 kernel 启动时,所有线程被组织成一个 grid. Grid 是**一维、二维或三维**的(`gridDim.x`, `gridDim.y`, `gridDim.z`). 这种多维组织方式天然适合图像处理(2D)和体积计算(3D),对于深度学习中的多维张量操作也非常直观. 

**Block(线程块)** : Grid 被划分为多个 block. 每个 block 也是一维、二维或三维的(`blockDim.x`, `blockDim.y`, `blockDim.z`). 关键约束: **同一个 block 内的线程可以通过 Shared Memory 通信和同步,但不同 block 之间的线程无法直接通信**(除非通过全局内存,且需要显式同步). 

**Thread(线程)** : block 内的最小执行单元. 每个线程有唯一的全局 ID: `threadIdx.x + blockIdx.x * blockDim.x`. 

**Warp(线程束)** : 这不是 CUDA 编程模型中的显式层次,而是硬件执行的基本调度单位. 一个 warp 由**32 个连续的线程**组成. GPU 的 warp scheduler 以 warp 为单位调度执行——同一个 warp 内的 32 个线程执行 SIMD(单指令多数据)操作. 如果 warp 内的线程因为条件分支而走不同路径(warp divergence),硬件会串行化执行每条路径,导致效率下降. 

![CUDA 层次化并行与内存访问模型](images/cuda_hierarchy_map.png)

> **图 9.4 CUDA 逻辑并行执行层次与 GPU 物理/片上内存访问路径**
> * **Grid(网格)**：Kernel 核函数启动的最高逻辑边界, 由多维排列的 Block 组成. 
> * **Block(线程块)**：在同一个物理 SM(流多处理器)上执行的线程集合. Block 内部的线程可以进行硬件级快速同步并共享 Shared Memory. 
> * **Warp(线程束)**：GPU 硬件调度的最小物理单位, 固定由 32 个连续线程组成, 共享同一个 SIMD 指令发射器. 
> * **Thread(线程)**：最微观的标量执行单元. 
> * **内存通路箭头(Memory Paths)**：
>   1. Thread 独占极速的私有寄存器(Registers). 
>   2. 同一 Block 内的所有线程协同读写共享内存(Shared Memory / L1). 
>   3. 全局 Grid 内的所有线程通过统一的总线事务读写全局显存(Global Memory / HBM), 需遵循合并访问(Coalesced Access)原则以防带宽严重折损. 


### 3.2 执行模型: Latency Hiding与Occupancy

GPU 没有传统的 CPU 那样的复杂分支预测和乱序执行. 它的性能秘诀是**海量线程的延迟隐藏(latency hiding)** . 

当一个 warp 因为等待内存加载而停滞时,warp scheduler 会立即切换到另一个就绪的 warp. 只要同时驻留的 warp 数量足够多,计算单元就永远不会空闲. 这种设计使得 GPU 能够容忍数百个时钟周期的内存延迟——只要同时有足够多的线程在"轮换"执行. 

**Occupancy(占用率)** 是衡量一个 SM 上活跃 warp 比例的指标. 它受限于: 
- **寄存器数量**: 每个线程使用的寄存器越多,SM 上能同时驻留的线程就越少. 

- **Shared Memory 用量**: 每个 block 使用的 Shared Memory 越多,SM 上能同时驻留的 block 就越少. 

- **block 大小**: block 内的线程数必须是 32 的倍数(warp size),且通常建议在 128-512 之间以获得最佳占用率. 

一个常见的性能陷阱是: kernel 使用了太多寄存器(例如因为复杂的内联计算),导致 occupancy 从 100% 下降到 25%. 此时即便计算逻辑完全正确,性能也会暴跌——因为没有足够的 warp 来隐藏内存延迟. 

### 3.3 内存访问模型: 合并访问(Coalesced Access)

GPU 的内存子系统以**warp 为单位**进行访问. 当一个 warp 中的 32 个线程同时访问全局内存时,如果这些访问地址是**连续且对齐的**,硬件可以将它们合并为一次或少数几次宽事务(如 128 byte 事务),从而最大化内存总线利用率. 

**理想情况**: warp 中的线程 $i$ 访问地址 `base + i * 4`(假设 float32),这 32 个地址完全连续(128 bytes),硬件只需发起一次 128-byte 事务即可满足整个 warp 的需求. 

**灾难情况**: warp 中的线程访问完全随机的地址(stride 很大或不规则). 此时每次访问都可能触发一次独立的内存事务,有效带宽可能下降 10-50 倍. 

**深度学习中的实践**: 
- **行主序 vs 列主序**: cuBLAS 使用列主序(column-major),而 PyTorch/NumPy 使用行主序(row-major). 当 PyTorch 调用 cuBLAS 时,需要在接口层做转置标记的处理. 如果张量的内存布局不连续(例如经过 slicing 或 permutation),就可能导致非合并访问. 

- **Channel-first vs Channel-last**: 在卷积神经网络中,NCHW(batch, channel, height, width)布局在某些 kernel 下比 NHWC 更容易产生合并访问. cuDNN 内部会根据硬件和操作类型自动选择最优布局. 

## 4. 核函数启动开销与隐藏成本

### 4.1 Launch Overhead

从 CPU 启动一个 CUDA kernel 不是免费的. 每次 kernel launch 都有固定的开销: 
- **CPU 端**: CUDA driver 需要验证参数、分配资源、将 launch 命令写入 command buffer. 这个开销通常在 3-10 微秒(μs)之间. 

- **GPU 端**: kernel 需要被调度到合适的 SM 上,等待之前的 kernel 完成(如果没有使用 CUDA graph 或 stream 并行). 

在小型模型或推理场景中,如果一个操作只涉及少量计算(如一次 LayerNorm),kernel launch overhead 可能比计算本身还长. 这解释了为什么**kernel fusion**(将多个小操作合并为一个大 kernel)能带来巨大加速——它减少了 launch 次数. 

### 4.2 CUDA Graphs: 消除Launch开销

CUDA Graph 是 NVIDIA 推出的用于消除 kernel launch overhead 的技术. 其核心思想是: 

1. **Capture 阶段**: 第一次执行时,记录所有 kernel launch 和内存操作的依赖关系,构建一个"执行图". 

2. **Replay 阶段**: 后续执行时,只需提交整个 graph,而不需要逐个 launch kernel. launch overhead 被摊销到接近零. 

在推理服务中,如果每次请求的模型前向传播路径是固定的(没有动态控制流),就可以使用 CUDA Graph 将整个 forward pass 捕获为一个 graph. 这在 TensorRT 和 vLLM 中都是标准优化手段. 

### 4.3 同步的隐形杀手

PyTorch 中很多看似无害的操作会触发**隐式同步(implicit synchronization)** : 

- `tensor.cpu()`: 需要等待 GPU 上所有之前的操作完成,然后将数据拷贝到 CPU. 
- `tensor.item()`: 将标量从 GPU 搬到 CPU,触发同步. 
- `torch.cuda.synchronize()`: 显式等待所有 GPU 操作完成. 

在训练循环中,如果在每个 iteration 的末尾打印 loss(`print(loss.item())`),就会引入一次同步点. 如果训练本身只需要 50ms/iter,而同步开销 5ms,这就浪费了 10% 的时间. 正确的做法是将 loss 累积在一个 GPU 张量中,每隔 N 步再一次性同步到 CPU. 

## 5. cuBLAS、cuDNN、NCCL: 工业级库的分工

### 5.1 cuBLAS: 线性代数的基石

cuBLAS 是 NVIDIA 提供的 GPU 加速基本线性代数子程序库(BLAS). 它是所有深度学习框架的底层依赖——PyTorch 的 `torch.matmul`、TensorFlow 的 `tf.linalg.matmul`,底层最终都调用 cuBLAS. 

cuBLAS 提供了三级 BLAS 操作: 
- **Level 1**: 向量-向量操作(dot, axpy, scal),如 `cublasSdot`(单精度点积). 

- **Level 2**: 矩阵-向量操作(gemv),如 `cublasSgemv`. 

- **Level 3**: 矩阵-矩阵操作(gemm),如 `cublasSgemm`、`cublasGemmEx`(支持混合精度). 

**`cublasGemmEx`** 是现代深度学习中最重要的 cuBLAS 函数. 它支持: 
- 输入矩阵 $A, B$ 的精度可以是 FP32、FP16、BF16、INT8. 
- 输出矩阵 $C$ 的精度可以是 FP32 或 FP16. 
- 计算精度(accumulator)可以是 FP32 或 FP16. 

例如,在混合精度训练中,调用 `cublasGemmEx` 时: 
- $A, B$ 的数据类型为 CUDA_R_16F(FP16). 
- $C$ 的数据类型为 CUDA_R_32F(FP32). 
- 计算类型为 CUBLAS_COMPUTE_32F(使用 FP32 累加器). 

这保证了矩阵乘法内部使用 FP32 累加(防止精度下溢),但输入输出仍是 FP16(节省带宽和显存). 

cuBLAS 内部使用了 CUTLASS(CUDA Templates for Linear Algebra Subroutines)的优化策略,包括: 
- **分块(Tiling)** : 将大矩阵分解为适合 Shared Memory 和寄存器的小块. 

- **双缓冲(Double Buffering)** : 在计算当前 tile 的同时,预加载下一个 tile 到 Shared Memory. 

- **Warp-level 优化**: 将 warp 组织成特定的布局以最大化 Tensor Core 利用率. 

### 5.2 cuDNN: 深度学习原语库

cuDNN(CUDA Deep Neural Network library)提供了比 cuBLAS 更高层的深度学习专用操作: 

- **卷积(Convolution)** : 前向、反向(数据梯度、权重梯度). cuDNN 会自动选择最优的卷积算法(im2col、Winograd、FFT、直接卷积等),这个选择过程称为**autotuning**. 

- **归一化(Normalization)** : BatchNorm、LayerNorm、InstanceNorm、GroupNorm 的前向和反向. 

- **激活函数(Activation)** : ReLU、Sigmoid、Tanh、GELU、Swish 等. 

- **RNN/LSTM/GRU**: 循环层的前向和反向. 

- **Attention**: cuDNN 8.0+ 提供了 fused attention 的底层 API(如 `cudnnMultiHeadAttnForward`),但直到 cuDNN 9.0 才真正提供能与 FlashAttention 竞争的 flash fused attention 实现. 

cuDNN 的 autotuning 会在第一次运行时尝试多种算法,测量它们的实际运行时间,然后缓存最优选择. 这意味着**第一次前向传播可能比后续慢 10 倍**——在生产环境中需要预热(warmup). 

### 5.3 NCCL: 多GPU通信的通用语

NCCL(NVIDIA Collective Communications Library)是分布式 GPU 训练的基石. 它封装了多 GPU 之间常用的**集合通信(collective communication)** 模式: 

| 操作 | 语义 | 在大模型训练中的用途 |
|------|------|---------------------|
| **All-Reduce** | 所有 GPU 的数据求和/平均,结果分发回所有 GPU | DDP 的梯度同步: 每张卡计算本地梯度,all-reduce 后所有卡拥有平均梯度 |
| **All-Gather** | 每张 GPU 发送自己的数据片,最终所有 GPU 拥有完整数据 | FSDP/ZeRO-3 的前向传播: 每张卡持有参数分片,all-gather 后重组完整层 |
| **Reduce-Scatter** | 数据求和后按片分发 | FSDP/ZeRO-3 的反向传播: 梯度 reduce-scatter 后每张卡只持有对应分片的梯度 |
| **Broadcast** | 一张 GPU 的数据广播到所有 GPU | 初始化时同步模型参数 |
| **All-to-All** | 每张 GPU 向每张 GPU 发送一块数据 | 专家并行(MoE)中 token 的路由交换 |

NCCL 的底层实现会根据硬件拓扑自动选择最优的通信路径: 
- 如果两张 GPU 在同一节点内且通过 NVLink 连接,NCCL 会直接用 NVLink P2P 传输. 
- 如果跨节点,NCCL 会通过 InfiniBand/RDMA 进行传输. 
- NCCL 还支持**tree algorithm**和**ring algorithm**两种 all-reduce 拓扑,在不同集群规模下自动切换最优策略. 

## 6. Triton: Python级GPU Kernel开发

### 6.1 为什么需要Triton

CUDA C++ 虽然功能强大,但门槛极高. 编写一个高效的矩阵乘法 kernel,需要深入理解: 
- 内存层次和 tile 大小选择
- 寄存器分配和 bank conflict 避免
- Tensor Core 的 WMMA/MMA 指令格式
- warp scheduler 和 latency hiding

这些知识的学习曲线非常陡峭,而且不同 GPU 架构(Ampere vs Hopper)的最优参数完全不同. 

Triton 由 Philippe Tillet 在 MIT 期间开发(现已被 OpenAI 维护),其目标是**让 Python 程序员用类似 NumPy 的语法编写接近 peak performance 的 GPU kernel**. 

### 6.2 Triton的核心抽象: Block-based编程

Triton 摒弃了 CUDA 中显式的 thread/warp/block 概念,引入了更高层的**块(block)抽象**. 一个 Triton kernel 的伪代码如下: 

```python
import triton
import triton.language as tl

@triton.jit
def vector_add_kernel(x_ptr, y_ptr, output_ptr, n_elements, BLOCK_SIZE: tl.constexpr):
    # 每个 program(类似于 CUDA block)处理一个连续的块
    pid = tl.program_id(axis=0)
    block_start = pid * BLOCK_SIZE
    offsets = block_start + tl.arange(0, BLOCK_SIZE)
    
    # 创建 mask 防止越界
    mask = offsets < n_elements
    
    # 加载、计算、存储
    x = tl.load(x_ptr + offsets, mask=mask)
    y = tl.load(y_ptr + offsets, mask=mask)
    output = x + y
    tl.store(output_ptr + offsets, output, mask=mask)
```

**关键区别**: 
- 在 CUDA 中,程序员需要手动管理 32 个线程如何协作加载一个 block 的数据. 
- 在 Triton 中,`tl.load(x_ptr + offsets)` 自动将加载操作映射到最优的 warp-level memory access pattern. 编译器会自动处理合并访问、Shared Memory staging、register allocation. 

### 6.3 Triton的编译管线

Triton 的编译过程是一个多层 lowering pipeline: 

1. **Triton IR**: Python 装饰器 `@triton.jit` 将 kernel 编译成 Triton 的中间表示(类似 LLVM IR 但针对 block 并行优化). 

2. **TTIR → TTGIR**: 将硬件无关的 Triton IR 转换为 GPU 相关的 Triton GPU IR(包含 shared memory layout、warp distribution 等信息). 

3. **TTGIR → LLVM IR**: 进一步 lowering 到 LLVM IR,此时已经可以生成 PTX(Parallel Thread Execution)汇编. 

4. **PTX → SASS**: PTX 被 NVIDIA 的 ptxas 编译器编译为 SASS(实际 GPU 机器码). 

Triton 编译器在 TTGIR 层面进行了大量自动优化: 
- **自动 tiling**: 根据硬件的 Shared Memory 大小和 Tensor Core 维度自动选择 tile 大小. 

- **自动 pipeline**: 插入 `async_copy` 指令(Hopper 的 TMA 特性)实现双缓冲. 

- **自动融合**: 将多个逐元素操作自动融合为一个 kernel. 

### 6.4 Triton在LLM生态中的应用

Triton 已经成为 LLM 推理和训练优化的事实标准工具: 

- **FlashAttention 的 Triton 实现**: FlashAttention-2 的原始实现是 CUDA C++,但社区也开发了 Triton 版本(如 Dao 团队在 GitHub 上的 `triton/python/tutorials/06-fused-attention.py`). Triton 版本的可读性远高于 CUDA,且在不同架构上重新编译即可获得较优性能. 

- **vLLM 的 custom kernels**: vLLM 中的 PagedAttention kernel、采样 kernel、并行解码 kernel 等都大量使用了 Triton. 

- **PyTorch 2.0 的 compile 模式**: PyTorch 2.0 引入的 `torch.compile` 默认使用 Inductor 后端,而 Inductor 生成 Triton kernel 作为其 GPU 代码生成目标. 

**Triton 的局限性**: 
- 对于极度不规则的控制流(如 dynamic loop、recursive 调用),Triton 的 block-based 抽象可能无法表达,仍需手写 CUDA. 
- Triton 编译器的自动优化虽然强大,但在某些极端场景下仍不如 hand-tuned CUDA(如 CUTLASS 中的某些特殊矩阵尺寸). 
- 调试 Triton kernel 比调试 CUDA 更困难,因为 block 抽象隐藏了底层的 thread 行为. 

## 7. 内存管理: 显存分配碎片化问题

### 7.1 PyTorch的CUDA Memory Allocator

PyTorch 使用一个**缓存内存分配器(Caching Memory Allocator)** 来管理 GPU 显存. 其核心策略是: 

1. **预分配大块**: 向 CUDA driver 一次性申请大块显存(如 1-2 GB),而不是每次 `torch.Tensor` 创建都向 driver 申请. 

2. **内部切分**: 将这些大块切分为小块来满足不同大小的张量请求. 

3. **缓存复用**: 当张量被释放时,内存不立即还给 CUDA driver,而是留在 PyTorch 的缓存池中,供后续请求复用. 

这个策略的好处是**避免了频繁的 cudaMalloc/cudaFree 开销**(每次调用可能耗时数毫秒). 但代价是**显存碎片化(fragmentation)** . 

### 7.2 碎片化的产生与危害

显存碎片化分为两种: 

**外部碎片化(External Fragmentation)** : 缓存池中有足够的总空闲内存,但没有一块连续的区域能满足当前请求. 例如: 
- 缓存池中有两块 4 GB 的空闲块,中间隔着一个 1 GB 的已分配块. 
- 现在需要申请 5 GB 的连续内存——虽然总空闲内存是 8 GB,但没有一块 >= 5 GB 的连续区域,导致分配失败(OOM). 

**内部碎片化(Internal Fragmentation)** : 分配器将内存按固定大小的 bin(如 512B、1KB、2KB...2MB)管理. 如果一个请求需要 1.1 MB,它会被放入 2 MB 的 bin,浪费 0.9 MB. 

在大模型训练中,外部碎片化是更致命的杀手. 训练过程中的张量生命周期非常复杂: 
- 模型参数、优化器状态是长生命周期的. 
- 激活值在前向传播时创建、反向传播后释放. 
- 临时张量(如 softmax 的中间结果)是短生命周期的. 

这导致显存中"长生命周期"和"短生命周期"的块交错分布,产生大量"空洞". 

### 7.3 应对策略

**1. 显存池预分配与固定分配策略**

一些训练框架(如 Megatron-LM、DeepSpeed)会在训练开始前预分配几乎所有需要的显存,避免运行时的动态分配. 这虽然浪费了部分显存(预留了峰值用量),但彻底消除了碎片化. 

**2. Gradient Checkpointing(激活值重计算)** 

通过只保存部分层的激活值、在反向传播时重新计算其余部分,可以显著减少同时存在的激活值张量数量,从而降低峰值显存占用和碎片化风险. 

**3. PyTorch 的 memory_stats 与碎片检测**

PyTorch 提供了 `torch.cuda.memory_stats()` 来诊断碎片化: 

```python
import torch
stats = torch.cuda.memory_stats(device=0)
print(f"Allocated: {stats['allocated_bytes.all.current'] / 1e9:.2f} GB")
print(f"Reserved:  {stats['reserved_bytes.all.current'] / 1e9:.2f} GB")
print(f"Active:    {stats['active_bytes.all.current'] / 1e9:.2f} GB")
# 如果 reserved >> allocated,说明有大量碎片
```

**4. CUDA Graph 的 side effect**

CUDA Graph 要求所有内存地址在 capture 和 replay 之间保持不变. 这意味着 graph 中使用的所有张量都必须在 capture 时预分配. 对于推理服务,这实际上是一种"固定分配策略",天然避免了碎片化. 

**5. NVIDIA 的 cudaMallocAsync**

CUDA 11.2+ 引入了 `cudaMallocAsync` / `cudaFreeAsync`,配合 memory pool 可以更好地管理异步内存操作. PyTorch 2.0+ 可以通过 `PYTORCH_CUDA_ALLOC_CONF=backend:cudaMallocAsync` 启用这一后端,在某些场景下显著改善碎片化. 

## 8. 局限性与边界条件

1. **CUDA 生态的锁定效应**: CUDA 是 NVIDIA 的专有技术,虽然 AMD 推出了 ROCm、Intel 推出了 oneAPI,但生态成熟度与 CUDA 差距显著. 这种锁定效应使得算法研究者即使想尝试非 NVIDIA 硬件,也面临巨大的迁移成本. 

2. **Triton 的编译器黑盒**: Triton 的自动优化虽然降低了门槛,但当性能不如预期时,调试编译器的决策(如为什么选择了某个 tile 大小)非常困难. Triton 目前缺乏类似 CUDA Nsight Compute 那样细粒度的 profiling 工具. 

3. **显存管理的零和博弈**: 减少碎片化通常意味着预分配更多显存,这会降低显存的灵活利用率. 在同时运行多个模型的推理服务中,这种权衡尤其棘手. 

## 9. 演进与承上启下

系统软件层是连接硬件和上层框架的桥梁. 理解 CUDA 的执行模型,让我们知道为什么 kernel fusion 能加速; 理解 NCCL 的集合通信,让我们知道分布式训练中的通信开销从何而来; 理解 Triton 的编译管线,让我们知道下一代框架(如 PyTorch 2.0 compile)是如何自动生成高效 kernel 的. 

但这些知识仍然停留在单机单卡或单机多卡的范畴. 当模型大到单节点无法容纳,当训练需要跨越数百台机器、数千张 GPU 时,我们就需要**分布式训练框架**来 orchestrate 这一切——这正是下一节(9.3 分布式训练框架)的主题. 

## 10. 总结与参考文献

1. **CUDA 的四层并行模型**(Grid → Block → Warp → Thread)是理解 GPU 执行的基础. Warp divergence 和 memory coalescing 是手写高性能 kernel 时的首要考量. 

2. **Kernel launch overhead 和隐式同步**是推理和训练中的隐藏性能杀手. CUDA Graph 和批量同步是主要的规避手段. 

3. **cuBLAS/cuDNN/NCCL 构成了深度学习的基础设施三角**: cuBLAS 负责单卡计算,cuDNN 负责深度学习原语,NCCL 负责多卡通信. 几乎所有框架都建立在这三者之上. 

4. **Triton 正在 democratize GPU kernel 开发**,其 block-based 抽象和自动编译优化让 Python 程序员也能写出接近 peak performance 的 kernel,但极端场景仍需手写 CUDA. 

5. **显存碎片化是大模型训练的隐形杀手**,预分配策略、gradient checkpointing 和 cudaMallocAsync 是主要的应对手段.
