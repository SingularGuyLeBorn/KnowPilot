# GLM-4.5V 与 GLM-4.1V-Thinking: 面向通用多模态推理的可扩展强化学习

> 原文标题: GLM-4.5V and GLM-4.1V-Thinking: Towards Versatile Multimodal Reasoning with Scalable Reinforcement Learning
> 原文链接: https://github.com/zai-org/GLM-V
> 发布机构: GLM-V Team, Zhipu AI & Tsinghua University
> arXiv: 2507.01006
> 发布日期: 2025-07

## Abstract

我们推出 GLM-4.1V-Thinking, GLM-4.5V 与 GLM-4.6V 系列视觉语言模型(Vision-Language Models, VLMs), 旨在推进通用多模态理解与推理. 本报告分享了我们在以推理为核心的训练框架中的关键发现. 我们首先通过大规模预训练获得了一个具有显著潜力的视觉基座模型, 它 arguably 构成了最终性能的上界. 随后, 我们提出 Reinforcement Learning with Curriculum Sampling (RLCS, 课程采样强化学习) 以充分释放模型潜力, 在 STEM 解题, 视频理解, 内容识别, 代码生成, 视觉定位, GUI Agent, 长文档解读等广泛任务上实现全面提升. 在 42 个公开基准的综合评测中, GLM-4.5V 在几乎所有同规模开源模型中达到 state-of-the-art (SOTA), 并在代码生成与 GUI Agent 等挑战性任务上比肩甚至超越 Gemini-2.5-Flash 等闭源模型. 与此同时, 较小的 GLM-4.1V-9B-Thinking 依然极具竞争力: 在 29 个基准上超过参数量大得多的 Qwen2.5-VL-72B. 我们开源 GLM-4.1V-9B-Thinking 与 GLM-4.5V, 并进一步推出原生支持工具调用, 拥有 128K 上下文的 GLM-4.6V 系列开源多模态模型. 简要概述见 https://z.ai/blog/glm-4.6v, 代码与模型见 https://github.com/zai-org/GLM-V.

> 译者注: GLM-V 的核心方法论可概括为"先靠预训练把视觉基座推到极致, 再用跨域 RL 把推理能力 unlock". 这与文本模态中 DeepSeek-R1 的"强基座 + 大规模 RL"思路一致, 但 GLM-V 将其扩展到多模态, 并强调奖励系统, 课程采样与多域协同. 预训练数据量级与多样性(100 亿图文对, 2.2 亿 OCR, 1.4 亿 GUI grounding 等)是其基座潜力的物质基础.

## 1 Introduction

视觉语言模型(VLMs)已成为现代智能系统的关键基石, 使感知与理解超越纯文本. 过去十年, 随着模型智能水平的飞跃[36; 47; 20; 4], 对应的多模态智能任务复杂度也持续提升. 从科学问题求解[67; 32; 34]到自主智能体开发[21; 62; 43], 对 VLMs 的需求已远超简单视觉内容感知[31], 愈发强调高级推理能力. 近期大量研究表明, 长链推理(long-form reasoning)[60]与可扩展强化学习[42]能显著增强大语言模型(LLMs)解决复杂问题的能力[23; 17]. 此前若干工作尝试用类似范式增强 VLM 的推理能力[46; 33], 但主要聚焦特定领域. 开源社区目前仍缺乏在广泛场景与任务上 consistently 超越同规模传统非思考模型的多模态推理模型.

本报告分享我们在开发 GLM-4.1V-Thinking, GLM-4.5V 与 GLM-4.6V 系列 VLMs 时的关键发现, 该系列模型旨在推进通用多模态推理. 我们的训练框架围绕统一目标构建: 通过可扩展强化学习全面提升模型推理能力. 预训练阶段, 我们整理了大量知识密集型多模态数据, 使模型具备强大的基础能力, 包括 (a) 海量带有准确事实知识的图文对; (b) 自建的图像文本交错学术语料; (c) 标注文档与图表, 教学视频以及覆盖自然与合成图像的 grounding 数据. 该基座模型为后续强化学习提供了高潜力的多模态推理基础. 监督微调(SFT)阶段, 我们精心设计, 面向具体领域的数据集, 教会模型在广泛任务中以标准化格式进行有效推理. 最后, 我们提出 Reinforcement Learning with Curriculum Sampling (RLCS), 以驱动大规模跨域推理能力. RLCS 是一种融合课程学习与难度感知采样的多域强化学习框架, 通过选择适合模型当前能力的任务与样本提升训练效率. 我们的强化学习过程增强了训练有效性与稳定性, 并通过跨领域的交互与反馈系统提升模型推理能力.

为推进该领域研究, 我们开源 GLM-4.1V-9B-Thinking(90 亿参数), GLM-4.5V(106B-A12B: 总参数量 1060 亿, 激活参数量 120 亿), 以及更新版本 GLM-4.6V-Flash(9B)与 GLM-4.6V(106B-A12B), 二者在同规模模型中均达到 SOTA. 在 42 个公开基准的综合评测中, GLM-4.5V 在几乎所有任务上达到同规模开源模型的 SOTA, 持续超越 Step-3(321B-A38B), Qwen-2.5-VL-72B 等强劲开源模型, 并在 22 个基准上相对闭源 Gemini-2.5-Flash 取得可比甚至更优表现. 值得一提的是, GLM-4.5V 在通用 VQA(MMStar, GeoBench), STEM(MMMU Pro, MathVerse, WeMath), 图表理解(ChartQAPro, ChartMuseum), 长文档理解(MMLongBench-Doc), 视觉定位(TreeBench, Ref-L4-test), 空间推理(ERQA), GUI Agent(OSWorld, AndroidWorld, WebVoyagerSom, WebQuest), VLM 代码生成(Design2Code, Flame-React-Eval), 视频理解(VideoM-MMU, LVBench, MotionBench)等广泛任务上, 将同规模开源 VLM 的 SOTA 提升约 10% 或更多. GLM-4.1V-9B-Thinking 在 29 个基准上也展现出超越更大模型(如 Qwen2.5-VL-72B)的竞争力. 我们还开源了预训练基座模型 GLM-4.1V-9B-Base, 为所有研究者提供强大基础.

我们将开发过程中的关键发现总结如下, 并在后续章节给出更详细解释.

- 多域强化学习展现出稳健的跨域泛化与相互促进. 在一个领域上训练会提升其他领域性能, 联合训练则能带来更大提升. (见第 6.3 节)

- 动态选择信息量最大的 rollout 问题对效率与性能至关重要. 为此, 我们提出 Reinforcement Learning with Curriculum Sampling (RLCS) 与基于比率的指数移动平均(Exponential Moving Average, EMA)动态采样扩展等策略. (见第 5.3 节)

- 稳健且精确的奖励系统是多域 RL 的关键. 在统一 VLM 上训练多样技能时, 即使单一能力的奖励信号存在轻微弱点, 也可能导致整个训练过程崩溃. (见第 5.2 节)

综上, 我们的贡献如下:

- 我们推出 GLM-4.1V-Thinking, GLM-4.5V 与 GLM-4.6V 系列 VLMs, 旨在推进通用多模态推理. 值得注意的是, GLM-4.5V 与 GLM-4.6V 原生支持"思考(thinking)"与"非思考(non-thinking)"两种模式, 可在性能与效率之间灵活权衡. 我们介绍了模型设计与以推理为核心的训练框架, 以及开发过程中遇到的关键洞察与挑战.

- 我们开源 GLM-4.1V-9B-Thinking, GLM-4.1V-9B-Base, GLM-4.5V, GLM-4.6V 以及领域奖励系统等有用组件, 以促进该领域研究. 代码与模型见 https://github.com/zai-org/GLM-V.

- 综合实验证明了所提出模型的优越性: GLM-4.5V 与 GLM-4.1V-9B-Thinking 在同规模模型中达到 SOTA, 其中 GLM-4.1V-9B-Thinking 甚至在多个基准上超越参数量大得多的模型. 此外, GLM-4.5V 在多项任务上匹配或超越 Gemini-2.5-Flash.

> 译者注: GLM-V 将文本模态的"长思考 + RL"训练范式系统性地迁移到多模态, 并发现跨域协同效应: 在 GUI Agent 数据上训练反而能提升视觉定位与通用 VQA. 这提示多模态智能体的训练数据可能是当前最"通用"的推理课程, 因为它同时要求感知, 定位, 文本理解与决策.

## 2 Overview and Architecture

图 2 展示了 GLM-V 系列(GLM-4.1V-Thinking, GLM-4.5V, GLM-4.6V)共享的架构, 由三大核心组件构成: 视觉编码器(vision encoder), MLP 适配器(adapter)以及作为解码器的大语言模型(LLM). 视觉编码器以 AIMv2-Huge[9] 作为初始化. LLM 组件方面, GLM-4.1V-Thinking 与 GLM-4.6V-Flash 采用 GLM-4-9B-0414[13], GLM-4.5V 与 GLM-4.6V 采用 GLM-4.5-Air[13]. 在视觉编码器内部, 我们借鉴 Qwen2-VL[57] 的策略, 将原始 2D 卷积替换为 3D 卷积. 这使视频输入可进行时间下采样(因子为 2), 从而提升效率. 对于单图输入, 图像被复制以保持处理一致性.

为使底层 Vision Transformer (ViT) 支持任意分辨率与宽高比, 我们引入两项适配. 首先, 我们在 ViT 的自注意力层中集成 2D-RoPE[44], 使模型能有效处理极端宽高比(超过 200:1)或高分辨率(超过 4K)图像. 其次, 为保留预训练 ViT 的基础能力, 我们保留其原始可学习的绝对位置嵌入. 训练时, 这些嵌入通过双三次插值动态适应可变分辨率输入. 具体而言, 对于被划分为 $H_p \times W_p$ patch 网格的输入图像, 每个 patch 的整数坐标 $\mathbf{g}=(w,h)$ 首先被归一化到连续的 $[-1,1]$ 网格 $\mathbf{g}_{\text{norm}}$:

$$
\mathbf{g}_{\text{norm}} = \left(w_{\text{norm}}, h_{\text{norm}}\right) = 2 \cdot \left(\frac{w + 0.5}{W_p}, \frac{h + 0.5}{H_p}\right) - 1 \tag{1}
$$

然后, 这些归一化坐标用于从原始位置嵌入表 $P_{\text{orig}}$ 中采样, 通过双三次插值函数 $\mathcal{I}_{\text{bicubic}}$ 生成该 patch 最终的适配嵌入 $P_{\text{adapted}}$:

$$
P_{\text{adapted}}(\mathbf{g}) = \mathcal{I}_{\text{bicubic}}\left(P_{\text{orig}}, \mathbf{g}_{\text{norm}}\right) \tag{2}
$$

为进一步增强语言侧的空间感知, 我们将 LLM 中的 RoPE 扩展为 3D-RoPE. 该扩展为多模态上下文提供更优的空间理解, 同时保留原模型的文本能力.

在解决空间适配后, 我们转向视频输入的时间建模. 对于视频, 我们在每帧 token 后插入一个时间索引 token, 时间索引通过将每帧时间戳编码为字符串实现. 与多图像输入不同, 视频帧形成时间上连贯的序列. 该设计显式告知模型真实时间戳与帧间时间距离, 从而增强时间理解与定位能力.

![](images/figure_02_the_shared_architecture_of_glm_4_1v_thin.jpg)

图 2: GLM-4.1V-Thinking, GLM-4.5V 与 GLM-4.6V 的共享架构. 该模型由三部分组成: (1) ViT 编码器, 处理并编码图像与视频; (2) MLP 投影器, 将视觉特征对齐到文本 token; (3) 大语言模型作为语言解码器, 处理多模态 token 并生成 token 补全. 我们的模型能够以原生分辨率与宽高比感知图像与视频. 对于视频输入, 每帧后额外插入时间索引 token, 以增强模型的时间理解能力.

> 译者注: 2D-RoPE + 保留绝对位置嵌入 + 双三次插值是处理"任意分辨率"的三件套. 与单纯将图像切分固定 patch 不同, 该方案让 ViT 直接感知原始空间坐标, 避免了因 resize/crop 造成的几何失真, 对 GUI Agent, 文档 OCR 等需要像素级精度的任务尤为重要.

## 3 Pre-training

为构建更强大的视觉语言基座模型, 我们整合了多样化的数据集, 包括广泛学术语料与知识丰富的图像文本交错数据, 同时也在纯文本数据上训练以保留语言能力. 为评估预训练效果, 我们在 MathVista 的非选择题子集上绘制 pass@k 指标. 如图 3 所示, GLM-4.1V-9B-Base 在 pass@k 指标上显著优于同规模 SOTA 预训练基座模型. 这种优越的基座性能 arguably 为强化学习(RL)后的最终结果设定了上界. 下面我们先描述预训练数据构建, 再介绍训练流程.

> 译者注: "基座性能决定 RL 上限"在 GLM-V 中被反复强调. MathVista pass@k 的优势说明基座已经学会了"多解法探索"的能力, 而 RL 只是把这一潜能显式地压缩成可重复的高质量输出.

### 3.1 Pre-training Data

**Image caption data.** 高质量图像-文本 caption 对赋予视觉语言模型世界知识并增强泛化能力至关重要. 为此, 我们通过精细整理的流程构建大规模高质量 caption 数据集. 流程始于从多源聚合初始池, 包括 LAION[40], DataComp[12], DFN[8], Wukong[14] 等公开数据集以及网络搜索引擎的数据, 总计超过 100 亿图文对. 为确保数据完整性, 我们实施多阶段精炼流程:

1. 启发式过滤: 首先应用一系列基于规则的过滤器, 剔除明显低质量样本. 规则包括最小图像分辨率, 纯色检测, caption 长度约束以及图像级去重.

2. 相关性过滤: 为保证跨模态语义一致, 我们使用预训练 CLIP 模型计算图文相似度, 仅保留 CLIP-Score 高于阈值 0.3 的图文对.

3. 概念平衡重采样: 为缓解网络规模数据中概念天然的长尾分布, 我们借鉴 MetaCLIP[63] 进行重采样. 使用包含丰富视觉概念与专有名词的 comprehensive vocabulary, 对过滤后数据重新加权, 以增强概念覆盖与平衡性.

4. 事实为中心的 Recaption: 为进一步提升 caption 的描述质量与信息密度, 我们迭代训练了一个以事实为中心的 recaption 模型. 如图 4 所示, 该模型旨在对原始 caption 去噪并丰富其内容, 生成更新, 更精确, 更详细的描述, 同时保留源文本的事实准确性.

最终, 我们以预定比例合并整理后的原始数据与 recaption 数据, 得到在广泛世界知识与丰富描述深度之间取得平衡的最终数据集.

> 译者注: 图文 caption 数据的质量直接影响 VLM 的"世界知识". 100 亿初始池经过四阶段清洗, 最终保留"事实准确"与"描述丰富"的样本. 这里的关键是 recaption 模型不仅去噪, 还要避免引入新幻觉, 这对后续推理的可靠性至关重要.

**Interleaved image-text data.** 丰富的图像文本交错数据广泛存在于网页, 书籍等语料中. 一方面, 其规模巨大, 远超主要依赖 alt-text 的现有图像 caption 数据集; 另一方面, 它编码了超越简单图像描述的丰富信息, 包括图文之间复杂的逻辑关系以及广泛的领域知识. 然而, 这类数据通常极为嘈杂: 许多样本缺乏真正的图文对齐, 信息密度分布也高度倾斜(大量语料信息贫乏). 因此, 此前工作很少大规模利用这些资源提升视觉语言能力, 通常只用少量数据帮助模型适应多图像, 交错文本排版. 为解决这一问题, 我们针对不同数据源设计并实现了专门的处理流程, 成功提取出大量高质量图像文本交错数据, 显著增强模型的基础图文理解与推理能力. 具体流程如下:

1. 网页数据处理流程: 我们的网页数据流程从大规模开源数据集中聚合原始内容, 包括 MINT[3], MMC4[73], OmniCorpus[29]. 初始池经过多阶段清洗与过滤. 首先, 使用 CLIP-Score 阈值丢弃与周围文章语义无关的图像. 然后, 通过启发式规则与专门训练的图像分类器组合, 去除通常位于文章末尾的广告, 二维码等常见噪声. 此外, 我们排除图像密度高但文本稀疏的样本, 如在线相册. 为积极丰富高信息价值内容, 我们迭代训练了一个"高知识密度"图像分类器. 该模型用于识别并优先保留具有重要信息价值的图像, 如学术图表, 科学插图, 工程示意图, 教学图解与地图.

2. 学术书籍处理流程: 作为另一核心数据源, 我们收集了超过 1 亿本数字化图书. 为确保内容相关性与质量, 我们首先筛选与 STEM 等关键领域相关的书籍. 随后, 使用 PDF 解析工具对这些 PDF 文档进行深度解析, 提取高质量图像与文本交错内容.

> 译者注: 网页与书籍的交错数据是 VLM "通识"与"多图推理"的重要来源, 但噪声极高. GLM-V 没有简单丢弃, 而是用"高知识密度图像分类器"主动挑选学术/工程类图像, 相当于在海量网页中做"信息价值加权". 这比均匀采样更能提升模型在图表, 文档, 地图等任务上的表现.

**OCR data.** 为增强模型 OCR 能力, 我们构建了一个包含 2.2 亿张图像的大规模预训练数据集. 该数据集由三个不同部分组成, 分别针对文本识别的特定方面:

1. 合成文档图像: 我们将语言预训练语料中的文本以不同字体, 大小, 颜色和方向渲染, 然后将这些渲染文本合成到来自 LAION 数据集的多样图像背景上, 生成覆盖广泛实际应用场景的合成图像.

2. 自然场景文本图像: 我们使用 Paddle-OCR 工具包处理海量自然图像, 自动提取文本内容及其对应边界框. 随后过滤, 仅保留至少包含一个有效 OCR 检测的图像, 从而用真实世界文本实例丰富数据集.

3. 学术文档: 我们借鉴 Nougat[5] 的处理方法. 从 arXiv 获取大量论文, 首先使用 LaTeXML 工具将 LaTeX 源码归一化并转换为 HTML, 然后解析 HTML 并转换为轻量级标记语言. 最后, 按原始 PDF 分页进行切分并栅格化, 创建高质量 PDF 页面渲染与其对应结构化源码标记的配对数据集.

> 译者注: OCR 数据的三类分工明确: 合成数据覆盖字体/布局多样性, 自然场景数据提供真实噪声, 学术文档则让模型学会"公式-图表-段落"的结构化 OCR. 对于 STEM 与文档理解任务, 结构化 academic markup 的识别能力往往比纯文本 OCR 更重要.

**Grounding data.** 为使模型具备精确视觉定位能力, 我们构建了一个覆盖自然图像与图形用户界面(GUI)两个主要领域的混合 grounding 数据集.

1. 自然图像 grounding: 在自然图像领域, 我们以 LAION-115M[27] 为基础数据集. 利用 GLIPv2[69] 模型解析每幅图像的 caption, 并自动为每个名词短语预测对应边界框. 为保证 grounding 数据的质量与丰富度, 我们过滤保留至少包含两个有效边界框的样本. 该流程最终产生 4000 万条高质量自然图像 grounding 标注.

2. GUI grounding: 在 GUI 领域, 我们从头构建了一个新颖的大规模数据集. 首先, 从近期 CommonCrawl 快照中提取 URL, 并通过自动化工具捕获对应网页截图. 超越静态截图, 我们使用 Playwright 框架与网页进行深度交互. 这使得我们能够编译并解析所有可见 DOM 元素及其在页面上的精确渲染边界框. 为增强模型在 GUI 环境中的交互与理解能力, 我们针对 GUI 的指代表达生成(Referring Expression Generation)与理解(Comprehension)任务生成了超过 1.4 亿组问答对.

> 译者注: GUI grounding 数据是 GLM-V 在 GUI Agent 任务上取得优势的关键. 1.4 亿问答对不仅要求模型"看到"元素, 还要理解元素在交互状态下的位置变化(如 hover, 弹窗). Playwright 的深度交互采集比静态截图更能反映真实 GUI 的 dynamics.

**Video data.** 为支持高级视频理解, 我们构建了一个大规模高质量视频-文本数据集. 我们从学术, 网络与专有来源整理多样化语料. 为解决标准 caption 中常见的幻觉与遗漏问题, 我们开发了包含细粒度人工标注的流程, 以准确捕捉复杂动作与场景内文本. 此外, 为编码更深层的视觉叙事, 我们通过人在回路(human-in-the-loop)工作流标注镜头运动, 构图等关键电影元素.

为确保数据纯净, 我们实施严格的过滤协议. 首先进行完整性检查, 去除损坏或无效文件. 随后, 采用基于多模态嵌入的去重策略, 丢弃视频与文本嵌入均高度相似的重复条目. 该过程有效消除语义冗余, 得到干净高效的训练语料.

**Instruction tuning data.** 为增强模型通用性与泛化能力, 我们多样化高质量指令微调数据. 实施三项针对性策略:

1. 任务覆盖与分类: 我们设计细粒度分类体系以优化数据采样, 扩展世界知识覆盖. 该分类体系允许按语义结构与任务目标组织 prompt, 实现类别特定的预处理与平衡采样策略.

2. 复杂场景增强: 为弥补现有开源数据集在 GUI 交互, 长文档理解等方面的不足, 我们将具有严格结构约束的合成生成数据与真实数据结合, 扩展数据集在多样领域的覆盖并提升整体复杂度.

3. 数据污染检查: 为防止公开评测基准数据泄漏, 我们对所有开源数据集进行人工与自动审查.

最终得到的 5000 万样本涵盖通用视觉感知与理解, 多模态推理(如 STEM 解题), 文档密集型场景, GUI Agent 操作与 UI 代码生成, 为全场景强化学习流程提供全面覆盖.

### 3.2 Training Recipe

模型训练分为两个阶段: 多模态预训练与长上下文持续训练.

**Multimodal pre-training.** 初始阶段旨在构建通用多模态能力的强基础. 针对不同模型设计特定并行策略. GLM-4.1V-Thinking 与 GLM-4.6V-Flash 设置张量并行大小为 2. GLM-4.5V 与 GLM-4.6V 设置专家并行大小为 8, 流水线并行大小为 4. 为保证专家负载均衡, 我们采用无损失路由(loss-free routing)方案, 将路由器偏置更新率设为 1e-3, 并应用系数为 1e-4 的辅助序列级平衡损失. 训练使用 8192 的序列长度与 1536 的全局 batch size, 共训练 120,000 步. 本阶段数据集由第 3.1 节描述的所有数据模态混合而成, 但不包括视频. 为最大化计算效率, 我们采用数据打包策略, 将多个变长样本拼接成接近最大长度的单个序列.

**Long-context continual training.** 预训练后, 我们进行持续训练, 将模型能力扩展到高分辨率图像, 视频与长上下文. 我们在训练数据中加入视频输入与超过 8K token 的长序列交错数据. 为适应更长输入, 我们将序列长度增加到 32,768, 并通过设置上下文并行大小为 4 来增强并行策略. 本阶段额外训练 10,000 步, 保持全局 batch size 1536. 此外, 我们将 GLM-4.6V 的上下文窗口扩展到 131,072 token, 以 128 的全局 batch size 额外训练 2,000 步, 确保在长序列长度上的稳健性能.

> 译者注: 长上下文训练不是简单增加长度, 而是通过"上下文并行(context parallel)"降低显存压力. 对 GLM-4.6V 进一步扩展到 128K, 说明其定位是长文档与长视频理解的旗舰模型.

## 4 Supervised Fine-Tuning

监督微调(SFT)阶段充当连接预训练与强化学习的桥梁, 将基座视觉语言模型转变为能够进行长思维链(Chain-of-Thought, CoT)推理的模型. 我们的长 CoT 语料经过精心整理, 以提升推理风格并与人类对齐, 覆盖可验证领域(如 STEM 问题)与不可验证任务(如指令遵循, 开放式写作). 与此前[59; 20; 16]将 SFT 应用于短 CoT 数据的工作流不同, 我们有意省略这一步: 我们认为 SFT 的作用不是注入新知识, 而是将模型现有的视觉语言理解与更高效思考, 回答风格对齐. 这种对齐为模型带来更强的冷启动, 使下一阶段的强化学习更高效, 更稳定.

### 4.1 Supervised Fine-Tuning Data

为便于后续强化学习, 我们整理了一个高质量长 CoT 推理示例数据集. 该数据集旨在训练模型以标准化格式生成连贯, 多步的解答, 从而支撑稳定且可扩展的 RL 训练.

**Data composition.** 我们的推理数据集覆盖广泛领域, 主要关注结果可被严格评估并通过强化学习不断改进的可验证任务. 我们也包含开放式视觉问答等不可验证任务, 以拓宽并增强模型在多样上下文中的通用推理能力. 数据集主要由中英文数据构成, 含少量其他语言. 我们使用预训练模型过滤掉过易或过难的实例, 保持适合训练的整体难度.

**Response formatting.** 每个回答遵循如下标准化结构:

```twig
<think> {think_content} </think> <answer> {answer_content} </answer>
```

`<think>` 部分记录模型的推理过程, 包括反思, 回溯, 重试, 验证等策略. `<answer>` 部分呈现简洁, 完整且逻辑严密的解答. 对于具有特定最终答案的可验证任务, `<answer>` 中的最终结果必须被 `<|begin\_of\_box|>` 与 `<|end\_of\_box|>` 包裹, 且仅允许一个 boxed 区间. 该标注便于 RL 阶段更准确地提取答案. 注意, 我们将 `<think>`, `</think>`, `<answer>`, `</answer>`, `<|begin\_of\_box|>`, `<|end\_of\_box|>` 加入 tokenizer 词表作为特殊 token, 以便更容易, 准确地进行在线解析. 注意, 在 GLM-4.5V 的回答中, `<answer>` 与 `</answer>` 这两个特殊 token 被移除.

> 译者注: 将控制 token 加入词表是一个容易被忽略但极其重要的工程细节. 如果 token 由模型自行拼接, 解析器很难在 rollout 中稳定识别答案边界; 专用 token 让奖励计算, 答案提取, 格式惩罚都能基于确定性规则完成, 大幅降低 RL 的噪声.

**Tool use formatting.** 为支持可靠的函数调用与外部 API 交互, GLM-4.6V 对工具调用采用严格的结构化输出协议:

```xml
<tool_call> {function_name} <arg_key> {arg-key1} </arg_key> <arg_value> {arg-value-1} </arg_value> ... </tool_call>
```

工具定义与调用协议在 system prompt 中显式给出, 提供穷尽式的函数签名以定义模型的操作能力. 与用于回答格式化的内部控制 token 不同, 工具使用协议采用显式 XML 模式组织外部动作. 在推理阶段, 模型生成 `<tool\_call>` 块, 每个参数被拆解为独立的 `<arg\_key>` 与 `<arg\_value>` 对. 这种基于标签的序列化为解析器提供清晰的语义边界, 有效缓解结构幻觉, 确保复杂参数在上下文中的 grounding. 该标准化格式使系统能够在无需为每个参数 key 修改 tokenizer 的情况下, 将自然语言意图确定性映射为可执行 API 调用.

> 译者注: GLM-4.6V 的工具调用格式选择 XML 而非 JSON, 是因为 XML 标签本身可作为"结构性锚点", 在多轮 tool observation 中更不容易出现括号不匹配等幻觉. 对于需要调用 crop, click 等工具的 GUI/视觉推理 Agent, 稳定的工具格式比人类可读性更重要.

**Response curation.** 冷启动数据集的质量对 RL 训练稳定性至关重要. 实践中我们发现, 构建不良的数据会导致训练不稳定甚至崩溃. 为缓解此问题, 我们实施严格的数据清洗流程. 该流程强制遵循格式约定(如正确使用 `<think>` 与 `<answer>` 标签), 并去除推理风格不一致或嘈杂的示例. 同时, 我们过滤掉包含混合语言表述或冗余思维模式的回答.

**Iterative data enhancement.** 为提升冷启动数据集的质量与挑战性, 我们将从 RL checkpoint 采样得到的高质量, 信息量丰富的示例重新加入冷启动数据集. 这种迭代增强让模型接触到 RL 过程中发现的更有用推理模式, 从而为后续 RL 训练提供更强基础.

### 4.2 Training Recipe

我们进行全参数微调, 序列长度为 32,768 token(GLM-4.6V 为 131,072 token), 全局 batch size 为 32. 训练语料包括第 4.1 节描述的多领域长格式推理数据. 除多模态数据外, 我们还纳入高质量纯文本长格式示例, 涵盖数学解题, 多轮对话, Agent 规划与指令遵循, 以帮助模型在多模态微调过程中保留核心语言理解与通用推理能力.

有趣的是, 我们观察到即使冷启动训练使用带有格式不一致或重复模式的嘈杂推理数据, 后续 RL 仍然有效. 这表明不完美的推理 trace 仍能提供有用指导. 尽管如此, 用干净, 一致数据初始化的模型表现出更稳定的 RL 收敛, 并取得更高整体性能.

与纯思考模型 GLM-4.1V-Thinking 不同, GLM-4.5V 与 GLM-4.6V 同时支持思考与非思考模式, 可根据具体场景与需求灵活切换. 在 SFT 阶段, 我们通过混合思考数据与非思考数据(普通 CoT)训练 GLM-4.5V 与 GLM-4.6V. 具体而言, 为启用非思考模式, 我们在用户 prompt 后显式追加特殊 token `/nothink`, 并训练模型在该 token 存在时生成空思考内容. 我们发现, 直接复用思考示例中 `<answer>` 部分的内容作为非思考模式输出, 效果优于单独整理非思考子集.

> 译者注: 双模式设计让同一模型既能输出完整 CoT(追求精度)又能快速作答(追求效率). `/nothink` token 的方案比训练两个独立模型更高效, 也说明 SFT 数据中的 `<answer>` 部分已经包含了可直接输出的高质量最终解.

## 5 Reinforcement Learning: What Is Challenging and What Works

监督微调之后, 我们主要依靠强化学习(RL)提升模型性能. 我们结合 Reinforcement Learning with Verifiable Rewards (RLVR) 与 Reinforcement Learning with Human Feedback (RLHF), 在所有多模态领域与能力上开展大规模 RL, 包括 STEM 解题(数学, 物理, 化学), 视觉定位, OCR, 视频理解, GUI Agent, 图表与文档理解, 逻辑推理以及指令遵循.

我们的 RL 框架由以下部分组成:

- **数据准备**: 在每个多模态领域中定义适合可验证奖励(RLVR)或模型奖励(RLHF)的子任务, 并整理大量难度适当, 覆盖广泛的高质量数据作为监督信号.

- **奖励系统**: 精确奖励是 RLVR 有效性的关键. 当在所有多模态领域扩展 RL 时, 为每个子领域尽可能多地分配准确奖励既具有挑战性又至关重要. 我们设计了一个多域统一奖励系统, 共享通用评估逻辑, 同时支持针对每个子领域优化稳健的验证器.

- **训练**: 在坚实的数据与奖励系统基础上, 我们精心打磨 RL 训练 recipe, 以提升有效性, 效率与稳定性. 我们提出并融入了包括 RLCS, 基于比率 EMA 的动态采样扩展, 更大 batch size, 移除 KL 与熵损失等在内的多项改进.

- **基础设施**: 为高效利用大规模 RL 训练的计算资源, 我们开发了内部高性能, 稳定的 RL 基础设施. 它灵活支持跨多模态领域的多样训练配置(如自定义验证器, 数据采样比例), 并在采样, 训练等方面进行了全面优化.

我们发现, 挑战存在于每一层--从数据准备, 奖励系统到训练与基础设施. 任何一个维度的失败都可能导致 RL 阶段效率严重下降甚至崩溃. 本节首先概述每个组件的核心流程, 然后分享我们在探索中遇到的挑战以及发现的最佳实践.

> 译者注: GLM-V 明确将多模态 RL 的复杂度分解为四个相互依赖的层次. 这种"系统级"视角区别于许多只关注算法的工作: 在多模态场景下, 奖励函数的一个漏洞就可能让模型在所有任务上崩溃, 因此工程与算法必须同时过硬.

### 5.1 Data Preparation

数据准备的目标是在每个子领域中尽可能选择或合成可通过 RL 高效改进的可验证数据. 为此, 我们依次执行以下阶段.

**Task identification.** 我们首先在每个多模态子领域中定义一组候选可验证任务. 例如, 视频 caption 是开放式且难以严格评估的, 而时间定位(temporal grounding)则适合进行明确正误判断.

**Data curation.** 然后, 我们从这些任务中过滤或生成验证器能够高精度评估的问答对. 这一过程包括将具有唯一答案的选择题转换为填空题, 以消除 RL 中随机猜测带来的噪声.

**Quality validation and offline difficulty grading.** 接下来, 我们进行彻底的正确性检查, 并使用多个现有或先前 RL 模型进行 passk 评估, 结合人工难度标签, 实现细粒度难度分级.

**Pilot RL experiments.** 最后, 我们在每个子领域进行初步 RL 实验, 确认数据质量与模型性能提升潜力.

### 5.2 Reward System

我们建立了一个兼容 RLVR 与 RLHF 并为每个多模态领域量身定制的奖励系统. 对于 RLVR 任务, 系统首先从 rollout 输出中提取包含最终答案的片段, 然后将关键答案与参考答案对比判断正误, 最后返回二值(0/1)或连续形式的奖励. 对于 RLHF 任务, 系统直接取输出的答案片段, 使用奖励模型进行打分.

强化学习的核心是由奖励系统反馈驱动的优化过程. 因此, 提升奖励系统的准确性与稳健性至关重要. 能够分配精确奖励也是 RLVR 目前取得突出成果的关键原因之一. 为在所有领域实现多模态强化学习, 我们采用精心设计的奖励系统监督模型能力的方方面面--视觉感知(OCR, grounding), 理解(文档, 图表, 视频理解), 推理(学术与逻辑问题求解)与 Agent 行为--因此需要开发全面, 精确且稳健的奖励系统.

尽管有研究表明, 即使随机或不完美反馈有时也能通过引导模型形成有效输出模式而带来收益[41], 但我们发现, 在统一 VLM 上训练多样技能时, 任一能力的奖励信号存在弱点都可能使整体训练偏离正轨. 如图 5 所示, 即使 STEM 子领域提供了高质量奖励, 多图 QA 任务奖励的缺陷仍导致模型在所有领域崩溃. 这突出表明, 稳定有效的 RL 要求每个领域都有精细调优, 抗 hack 的验证器--任何弱验证器都可能破坏并崩溃整个训练.

> 译者注: 图 5 的教训是多模态 RL 的"木桶效应": 模型会迅速找到奖励最弱的环节进行 exploit. 如果多图 QA 的验证器只检查答案字符串而忽略图像内容, 模型可能生成与问题无关但格式正确的答案. 这要求奖励系统必须像"端到端单元测试"一样对每个子领域全覆盖.

我们在实验中识别出奖励模型设计中的若干挑战与困难, 并给出相应的优化与解决方案.

**The extraction of the final answer in RLVR.** 对于 RLVR, 我们首先从模型回答中提取最终答案, 然后进行正确性比较. 提取方法通常有两种: 基于 box 标记的规则提取与通过 LLM 提取. 后者更灵活, 无需强制模型在关键答案周围输出显式 box 标记, 避免繁琐的格式调优, 保留原始用户友好的回答格式. 我们发现, 对于简单学术问题(答案通常是单个数字)或单类别任务(答案遵循固定格式范围), 设计 prompt 让 LLM 精确提取答案相对容易. 然而, 在我们的多模态, 开放域 RL 设置中, 问题与答案的多样性急剧增加, 提取变得复杂且存在大量 corner cases. 基于 LLM 的提取往往被证明不准确, 导致后续正确性判断出错. 此外, 某些情况下"答案"片段会循环或过长, 对 LLM 而言提取最终答案困难或分布外, 进一步削弱基于模型的提取准确性. 为解决这些问题, 我们要求模型在 RLVR 期间用 box token 显式标记最终答案, 并仅将 box 内容与参考答案对比.

值得一提的是, 许多先前工作使用 `\boxed{}` 标签表示最终答案. 然而, 当参考答案变得复杂时(例如 GUI Agent 任务的结果表达为复杂函数调用), `\boxed{}` 可能存在歧义且难以自动解析. 因此, 我们将特殊 token 引入词表, 并将答案区间标记为: `<|begin\_of\_box|>{FINAL\_ANSWER}<|end\_of\_box|>`.

> 译者注: 使用 `<|begin_of_box|>` 而非 `\boxed{}` 是一个面向工程落地的选择. 多模态答案可能是边界框, 函数调用, LaTeX 公式或长文本, `\boxed{}` 在大括号嵌套与特殊字符转义上容易出错; 专用 token 让 parser 可基于确定性规则切分, 不依赖 LLM 的"理解".

**Avoid reward hacking.** 粗糙或不完整的奖励设计可能让模型发现提升奖励的捷径, 而非真正提升任务表现. 例如, 在初步实验中, 我们发现经过 500 多次训练迭代后, 设计不慎的验证器可能被 hack: 对于计数问题, 模型回答"0 到 10 之间的正确数字", 对于关于速度相对论问题, 回答"非常接近光速的速度"--这些回答成功欺骗某些基于 LLM 的奖励模型并获得高奖励.

**Domain-specific reward system.** 不同多模态子领域的最优验证器不同, 且与任务紧密耦合--例如数学问题中"43"与"43.0"等价, 但在 OCR 语境下不等价. 为在所有领域提供更准确的奖励, 我们开发了具有以下特点的领域特定奖励系统:

- 共享验证函数: 通用检查(如格式验证, box 内容提取, 精确匹配)实现为可复用函数, 以简化开发.
- 领域特定模块: 每个领域拥有独立子模块, 支持复杂验证逻辑, 包括分支工作流, 函数式评估以及由自定义 judge prompt 与超参数驱动的基于模型的判断.
- 单元测试: 为验证每个领域的奖励系统, 我们建议定义针对该领域输出分布的单元测试, 并根据测试结果迭代优化奖励逻辑.

例如, 在图表 QA 中, 数值答案按相对容差阈值验证; 文本答案先检查精确匹配, 若未匹配则回退到基于 LLM 的语义等价评估. 我们在下方汇总部分领域特定验证器, 并在 GitHub 仓库开源该奖励系统以支持进一步学术研究.

> 译者注: 领域奖励系统的"共享函数 + 领域模块 + 单元测试"三层结构, 是软件工程中验证驱动开发(validation-driven development)在 RL 奖励设计中的体现. 单元测试尤其重要: 它让奖励函数的迭代从"看训练曲线"变成"看测试用例通过率", 更快发现 corner cases.

Beyond the domain-specific content checking above, we also build a format-and-style-checking reward system using both rule-based and model-based judgments. 在格式检查中, 任何对不可验证数据的回答若 `<answer>` 内容包含特殊标记 `<|begin\_of\_box|>` `<|end\_of\_box|>`, 将受到最小化奖励惩罚. 在风格检查中, 若 `<think>` 或 `<answer>` 内容包含大量中英混合片段或大块重复文本, 同样给予低奖励. 此外, 基于文本的奖励模型评估 `<answer>` 内容是否遵循指令且流畅, 鼓励输出紧密遵循 prompt 同时保持连贯与逻辑严密.

> 译者注: 格式与风格奖励是防止模型"只学答案格式, 不学内容"的护栏. 但 GLM-V 也警告: 格式应在冷启动 SFT 中充分学会, RL 中混合格式奖励可能 destabilize 训练. 这再次说明 SFT 与 RL 的职责边界.

### 5.3 Reinforcement Learning with Curriculum Sampling (RLCS)

在 RL 训练期间, 我们按预定比例混合来自各多模态领域的数据, 使用第 5.2 节描述的奖励系统验证每个领域, 并使用 GRPO[42] 目标进行优化. 为确保 GLM-V 模型在每个多模态子领域都能充分发挥潜力, 我们首先在每个子领域进行 pilot 实验, 评估其训练难度, 性能提升潜力以及对应数据集所需训练 token. 这些洞察指导 RL 过程中训练数据比例的分配.

一个挑战是, 随着 RL 训练推进, 模型能力提升后有效学习效率不可避免地下降: 许多示例变得过于简单, 无法驱动进一步学习. 例如, 在 GRPO 中, 一个 rollout batch 中所有样本都回答正确则不会产生有用梯度. 在我们的 pilot 实验中, 仅 200 步后, 超过一半的 prompt 准确率就超过 90%. 同时, rollout 效率仍是主要瓶颈: 训练时间的大部分消耗在 rollout 上. 因此, 为 rollout 选择最具信息量, 难度适当的问题至关重要.

为最大化学习效率, 我们提出 Reinforcement Learning with Curriculum Sampling (RLCS), 将课程学习的洞察应用于在线采样. 我们采用自适应课程, 持续调整训练样本难度以匹配模型不断变化的能力, 确保每次更新都 maximally informative. 为实现这一点, 我们同时从离线(offline)与在线(online)两个维度评估样本难度. 训练前, 我们通过对多个成熟视觉语言模型(或早期 RL checkpoint)在整个数据集上运行 pass@k 评估, 并结合专家人工难度标注, 评估每个样本的固有难度. 该过程产生一组从"非常简单"到"非常困难"的细粒度难度标签, 将数据划分为多个层级. 训练期间, 我们进行在线难度分级. 对于每个生成的 rollout, 我们记录 pass@k 结果, 映射到对应难度层级, 并将这些结果与离线标签融合. 该在线难度分布也为模型当前表现提供有价值洞察.

利用这些难度标签以及模型在各子类别的表现, 我们以训练迭代为粒度持续重新加权不同难度类别的采样比例. 核心思想是降低过易以及当前过难样本的采样, 增加模型收益最大的中等难度样本的曝光. 经验上, 我们观察到 RLCS 显著加速模型提升并持续带来性能增益.

> 译者注: RLCS 的精髓在于"把课程学习从静态预定义变成在线动态调整". 离线难度标签提供先验, 在线 pass@k 提供后验, 两者结合让采样比例随模型能力演化. 这与文本 RL 中的"动态难度采样"异曲同工, 但多模态场景下问题难度的维度更多(视觉复杂度, 推理步数, 答案格式等).

此外, 我们特别着力于提升 RLCS 的有效性, 效率与稳定性. 下面尽可能全面地分享我们在优化训练过程中学到的经验, 获得的洞察与开发的方法.

#### 5.3.1 Improving Effectiveness

为提升多模态强化学习的性能上限, 我们提出并融入以下改进:

- **Larger batch size.** 在训练期间混合多域多模态数据时, 建议使用相对较大的 batch size, 以在长期内获得更高性能上限.

- **Dynamic sampling expansion via ratio EMA.** 在 GRPO 中, 当移除熵与 KL 损失后, 完全正确或完全错误的 rollout batch 不提供有用梯度. 换言之, 全对/全错 prompt 会降低可用 batch size. 随着这些全对/全错 batch 比例增长或波动, 有效 batch size 可能剧烈变化, 损害训练稳定性. 为解决这一问题, 我们以故意的过采样因子 `expansion_ratio` 进行 rollout, 然后选择难度最平衡的子集(即正确与错误回答数量尽可能接近). 具体而言, 对每个 rollout, 我们按 `expansion_ratio = 1 / (1 - not_valid_sample_rate)` 计算扩展比例, 其中 `not_valid_sample_rate` 表示上一迭代中全对或全错样本的比例. 然后我们对该比例维护指数移动平均 `expansion_ratio_ema`, 并在下一迭代中将其作为过采样系数. 与[66]相比, 该方法预先确定 rollout 样本总数, 便于并行采样与平衡 rollout 分配, 更贴合底层大规模 RL 基础设施, 从而获得更高效率.

- **Force answering.** 当思考过程过长时, 可能被 rollout 长度限制截断. 由于模型无法产出答案, 通常会被赋予 0 奖励. 然而, 这种长推理未必错误--对于难题, 生成的过长思考路径可能是完全合理的. 这样截断不仅浪费 rollout 预算, 还会给训练注入噪声. 为解决这一问题, 我们强制在截断处插入 `</think>` token, 提示模型输出最终答案, 并允许我们为其推理赋予公平奖励[64]. 我们发现, 该方法鼓励模型学会在任何长度思考后给出合适答案, 便于在测试时动态控制思考预算.

- **Discard KL loss.** 与纯文本模型相比, 视觉语言模型在强化学习期间 KL 散度增长更快. 然而, 当我们应用 KL 损失显式抑制该增长时, 模型能力明显受限. 因此, 我们移除 KL 损失.

- **Clip-higher.** 在多模态 RL 中, 与[66]类似, 提高重要性采样比率的上裁剪界也被证明有助于改善 off-policy 性能并防止过度熵崩溃.

> 译者注: "force answering" 是处理长 CoT rollout 被截断的关键技巧. 若简单给 0 奖励, 模型会学会"短答案偏好"以规避截断, 损害难题性能. 插入 `</think>` 强制回答, 相当于把"被截断的推理"转化为"有限思考预算下的最终答案", 让模型学会在任意深度后给出 best-effort 答案.

我们还分享从大规模跨域强化学习中得到的观察与洞察. 我们相信这些洞察有助于未来提升 RL 有效性.

- RL 阶段的峰值性能与冷启动 SFT 模型的性能并非完美相关. 例如, 在某实验中, 将冷启动训练从 1,000 步增加到 2,000 步, 冷启动后平均性能提升约 2 个百分点. 但 RL 后, 两个 checkpoint 收敛到几乎相同性能. 换言之, 更高的冷启动分数不能保证更大的 RL 潜力; 通过适当训练, RL 可以将具有同等固有潜力的基座模型提升到相同峰值.

- RL 中的域间干扰比 SFT 更不明显. 与 RL 训练域正交的未训练或欠训练域的性能可以保持得很好. 此外, 跨域泛化(一个领域的提升迁移到其他领域)频繁出现.

> 译者注: "冷启动 SFT 性能不等于 RL 上限"是一个容易让团队过度投资 SFT 的陷阱. 当 SFT 数据足够干净, 格式正确后, 继续增加 SFT 步数可能只是"过拟合到答案格式", 而 RL 会重新探索更优推理路径. 这提示资源分配应在 SFT 达标后尽快切换到 RL.

#### 5.3.2 Improving Stability

为增强训练稳健性并防止 RL 崩溃, 我们识别出若干显著影响训练管线稳定性的关键因素:

- 冷启动 SFT 数据质量对训练稳定性至关重要. 因此, 强烈建议将冷启动数据质量保持在一定阈值之上. 例如, 若冷启动数据包含大量无意义思考路径, 所得模型在 RL 训练中会严重不稳定甚至崩溃.

- 我们发现引入熵损失以促进多样性会导致模型产生乱码, 最终导致训练崩溃, 因此我们移除熵损失.

- 在 rollout 期间, 使用 top-p = 1 而非较小值能产生更稳定的 RL 训练. 尽管有先前工作主张降低 top-p(如 0.9)以减少方差, 稳定 rollout, 但我们观察到这实际上会增加随时间出现乱码的风险. 相比之下, 设置 top-p = 1 消除了后期迭代中容易出现的乱码输出. 我们假设 top-p = 1 确保完整词表覆盖, 防止罕见 token 学习不足, 从而保持输出干净. 虽然这一选择会引入更多采样随机性, 但最终提升了 RL 阶段的稳定性与性能.

- 我们比较了 per-sample(在每个样本内平均 token 损失, 再在样本间平均)与 per-token(在 batch 内所有 token 上平均损失, 再在 batch 间平均)两种损失计算方式, 观察到平均奖励没有显著差异, 但 per-sample 损失计算带来更稳定的训练.

- 尽管基于格式的奖励在 RL 中可帮助输出向正确结构靠拢, 但我们强烈建议模型在冷启动阶段充分学会所需输出格式, 而非依赖 RL. 根据我们的经验, 若格式错误频繁出现, 格式奖励与正确性奖励的混合可能 destabilize 训练.

> 译者注: top-p=1 与移除熵损失看似矛盾(一个增加随机性, 一个减少多样性), 实则共同目标是"防止 token 分布塌陷". top-p=1 保证长尾 token 仍有机会被采样到, 避免其表示退化; 移除熵损失则防止模型为追求多样性而生成乱码. 两者共同维持了一个"有噪声但不崩溃"的采样空间.

### 5.4 Infrastructure

为最大化 RL 训练效率与性能, 我们广泛优化 RL 基础设施, 重点关注以下组件:

- **DP rank 间的序列长度负载均衡.** 由于每个样本的 rollout 长度事先未知, 某些 rank 可能被分配到大量超长序列(如视频或长文档 prompt, 或难题的长回答). 若不跨 DP(Data Parallel) rank 平衡总序列长度, 最慢的 rank 将决定整体训练速度. 为此, 在 rollout 之后, 将训练样本分配给各 DP rank 之前, 我们平衡序列长度与计算负载, 使每个 rank 的前向-反向传播时间保持接近, 从而最大化吞吐.

- **序列打包与梯度累积的 rank 内训练.** 由于 RL 中样本长度不可预测, 我们无法事先知道每个 DP rank 需要进行多少次前向传播. 我们通过结合序列打包与梯度累积解决这一问题: 每个优化步包含多次前向-反向传播的 micro-step, 每个 micro-step 将若干样本打包成长度为 context_length = 32K 的固定长度序列, 未使用位置补零. 训练时, 我们按样本数加权平均各 micro-step 梯度, 这在数学上等价于一次性对整个 rollout batch 计算梯度.

- **DP rank 内的样本打包与重排.** 基于上述策略, 训练与每次前向-反向传播的样本数解耦. 因此我们应用高效的样本重打包启发式, 以尽可能少的 micro-step 完成所有样本. 实践中, 该优化使我们的前向-反向时间减半.

- **基于比率 EMA 的动态采样扩展.** 为对中等难度样本进行过采样并筛选, 我们提出并实现基于比率 EMA 的动态采样扩展算法, 详见第 5.3.1 节. 我们的方法预先计算并行 rollout 所需样本数, 大幅提升效率.

> 译者注: RL 基础设施优化的核心是"把变长序列的尾部延迟压到最低". 负载均衡, 打包, 重排本质上是将在线 rollout 的"动态 batch"重新组织成计算效率接近静态训练的形式. 对于大规模多模态 RL, 这些工程优化可能比算法改动带来更大的 wall-clock 提升.

### 6.3 Investigating Cross-Domain Generalization in Reinforcement Learning

虽然多域 RL 成功提升了整体性能, 但还有一个问题: 在 RL 过程中, 各个多模态领域是否会相互泛化并相互增强, 还是会相互拮抗与干扰? 为探索这一问题, 我们对 GLM-4.1V-9B-Thinking 进行实验. 我们选取四个代表性领域: STEM, OCR & Chart, Grounding 与 GUI Agent. 每个领域都依赖共同的视觉感知与推理工具, 但侧重点不同. 例如, Grounding 要求细粒度像素级感知; OCR & Chart 强调文本识别与抽象图形解释; STEM 任务以复杂视觉推理为核心; GUI Agent 则需要 UI 理解, 现实世界知识与动态决策的融合.

所有实验均基于 GLM-4.1V-9B-Thinking 的 SFT 阶段 checkpoint. 我们比较 5 组 RL 训练数据: (1) STEM; (2) OCR & Chart; (3) Grounding; (4) GUI Agent; (5) Mix-all, 按 GLM-4.1V-9B-Thinking 的 RL 训练比例混合上述四组数据. 我们将 mix-all 设置作为训练数据量的参考: 对每个子领域的独立 RL 实验, 处理的样本数 exactly 等于该子领域在 mix-all 实验中见到的样本数. 训练后, 我们在 5 类基准上评估模型, 每类包含多个基准: (1) STEM; (2) OCR & Chart; (3) Grounding; (4) GUI Agent; (5) 没有对应训练集的通用图像 VQA. 结果如图 6 所示, 数字表示相比初始 SFT checkpoint 的平均组分数提升. 结果展示了大多数领域中稳健的跨域泛化与相互促进:

- 在一个领域上训练会提升其他领域性能. 例如, 在 STEM 数据上的 RL 不仅提升 STEM 专项技能, 还增强了视觉定位, GUI Agent 交互与通用 VQA 任务上的表现. 类似地, 在 OCR & Chart 数据上训练也会在 STEM, GUI Agent 与通用 VQA 基准上带来增益. 这种跨域效应揭示了共享基础能力(如视觉理解, 文本识别与推理)可以通过单一领域 RL 信号被共同激活并精炼. 有趣的是, 仅在 GUI Agent 任务上应用 RL 就能在所有评估领域产生提升, 表明 GUI Agent 挑战内在地需要文本识别, 视觉定位与逻辑推理的综合混合, 从而广泛迁移.

## 7 Discussion: Limitations and Future Work

GLM-4.1V-Thinking, GLM-4.5V 与 GLM-4.6V 代表我们在追求通用多模态推理道路上的坚实步伐. 在以预训练, 监督微调与强化学习统一于共享目标的以推理为核心的训练框架下, 模型成功学会在视觉, 文本, 数学, 科学与 Agent 领域进行推理. 所得 9B 参数稠密模型与 106B-A12B 参数 MoE 模型在多样基准上表现出色, 在同规模模型中达到 SOTA. 我们开源 GLM-4.1V-Thinking, GLM-4.5V 与 GLM-4.6V, 以促进多模态推理方向的研究.

尽管已取得显著进展, 仍存在若干局限. 首先, 虽然 RL 提升了任务完成率, 但并不总能改善推理质量. 在某些情况下, 模型产生正确答案但依赖错误的推理步骤. 这一问题源于当前奖励模型通常只评估最终结果, 而不验证中间推理步骤. 因此,  flawed 甚至幻觉的推理链若导致正确答案, 可能被无意中强化. 这凸显了设计能够有效评估推理过程而不仅是结果的奖励机制的重要性.

其次, RL 训练可能表现出不稳定性. 早期实验表明, 设置的微小变化可能导致推理深度或输出风格的显著变化. 尽管后续版本在奖励设计与冷启动数据等方面的改进带来了更稳定的训练, 但剩余的敏感性表明大规模 RL 优化仍存在深层挑战. 需要进一步改进 RL 以提升一致性与稳健性.

第三, 尽管模型在多样任务上表现强劲, 但在复杂场景下仍可能遇到困难. 例如, 涉及杂乱, 遮挡物体或模糊视觉细节的图像可能导致感知错误, 削弱模型推理能力. 在这些条件下, 模型可能诉诸猜测或通用假设, 而非基于 grounded 的推断. 这提示感知与推理的改进必须同步推进, 因为这些组件错综复杂地相互关联.

展望未来, 一个关键方向是改进对模型推理的监督与评估. 未来奖励模型不仅应评估最终答案, 还应评估中间推理步骤, 主动检测幻觉并标记逻辑不一致. 此外, 对于具有主观评估的任务, 探索防止奖励 hacking 的策略至关重要, 这是迈向通用智能的必要一步.

我们还对多模态训练给纯文本推理任务带来的潜在收益感兴趣. 例如, 理解图像中的代码解释等视觉推理任务是否能增强纯文本代码生成任务的性能, 是一个充满希望的研究方向. 探索视觉与语言模态如何相互增强, 可能带来通用推理能力的重大进展.

最后, 随着模型能力提升, 评测框架也必须相应演进. 许多当前基准正趋于饱和, 或无法有效识别关键错误, 如推理链中的幻觉. 未来基准应更具挑战性与诊断性, 旨在显式检测更多失效模式, 如捷径推理或幻觉. 我们希望 GLM-4.1V-Thinking, GLM-4.5V 与 GLM-4.6V 能够激发评估与改进通用多模态推理的新标准与新方法.

> 译者注: GLM-V 团队坦诚指出的三大局限--结果奖励无法验证过程, RL 不稳定性, 复杂视觉感知失败--正是当前多模态推理模型从"做题家"走向"可靠推理器"必须跨越的障碍. 其中"过程奖励(process reward)"被认为是最有前景的下一步, 与 OpenAI 在 o3 等工作中强调的过程监督方向一致.

## B 评测协议与指令

### B.1 VLM 代码生成评测协议

为评估 GLM-4.1V-Thinking 与 GLM-4.5V 的 HTML 代码生成能力, 我们遵循 Design2Code[43] 中描述的 "direct" 评测设置, 省略文本增强与自我修正步骤. 与[43]不同, 我们采用 GPT-o4-mini 作为视觉 judge, 将每个渲染后的 HTML 输出与对应 UI 参考截图进行比较. 这一选择基于我们的经验观察: 在复杂 UI 布局下, GPT-o4-mini 给出的相似度判断比 CLIP 更准确, 也更符合人类偏好. 提供给 GPT-o4-mini 的评分 prompt 如下:

> I will give you two images. The first is the reference, and the second is generated from the first via code rendering. Please rate their similarity from 0-100, where 0 means completely different and 100 means identical. Provide the score inside a LaTeX \boxed{} and briefly explain your reasoning.

我们以 80 分作为 faithful UI2Code 复现的阈值, 并将 accuracy 报告为达到该标准的测试用例比例(分数 ≥ 80). 这一选择基于以下观察: 分辨率, 字体等渲染差异使满分 100 不现实--几乎所有结果都会被判定为"不一致", 从而扭曲评测. 人类评分者更关注整体布局, 颜色与元素而非像素级差异, 在我们的手动评估中, VLM 评分 ≥ 80 的用例通常被认为在视觉与功能上与参考一致.

### B.2 GUI Agent 指令

为在 GLM-4.1V-Thinking 或 GLM-4.5V 作为手机与电脑 GUI Agent 时达到最佳性能并复现论文结果, 必须遵循 https://github.com/zai-org/GLM-V/blob/main/examples/gui-agent/glm-41v/agent.md (GLM-4.1V-Thinking) 或 https://github.com/zai-org/GLM-V/blob/main/examples/gui-agent/glm-45v/agent.md (GLM-4.5V) 中列出的特定 prompt 与设置.

### B.3 视觉定位指令

为可靠触发 GLM-4.5V 或 GLM-4.1V-Thinking 的定位行为并复现论文中的评测结果, 我们建议使用类似以下格式的 prompt:

- Help me to locate <expr> in the image and give me its bounding boxes.
- Please pinpoint the bounding box $[[\mathrm{x}1, \mathrm{y}1, \mathrm{x}2, \mathrm{y}2], \dots]$ in the image as per the given description. <expr>

在上述 prompt 中, <expr> 是目标物体的描述, 括号风格可以变化(`[]`, `()`, `<>`, `{}` 等). 每个输出边界框将是一个四元组 $[x_1, y_1, x_2, y_2]$, 由左上角与右下角坐标组成, 每个值按图像宽度(x)或高度(y)归一化并乘以 1000.
