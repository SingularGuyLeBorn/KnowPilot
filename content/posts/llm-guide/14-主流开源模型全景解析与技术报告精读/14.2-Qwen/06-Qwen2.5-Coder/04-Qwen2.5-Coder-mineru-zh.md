# Qwen2.5-Coder Technical Report 中文交付稿

> [返回 14.2-Qwen 家族总览](../../14.2-Qwen.md)

## 摘要

Qwen2.5-Coder 是在 CodeQwen1.5 之后推出的新一代代码专用模型系列, 覆盖 0.5B / 1.5B / 3B / 7B / 14B / 32B 六个尺度. 它建立在 Qwen2.5 通用基座之上, 在超过 5.5T token 的代码相关语料上继续预训练, 同时通过高质量数据清洗、可扩展合成数据、代码-数学-文本的平衡混合, 让模型在代码生成、代码补全、代码推理和代码修复等多个维度都达到开源 SOTA 水平, 同时尽量保住通用能力与数学能力.

> 译者注: 这篇报告和 Qwen2.5 主报告的关系很清楚: Qwen2.5 负责“通用平台能力”, Qwen2.5-Coder 负责“把这套平台往代码场景做深”. 因此它的重点不是重新造一个架构, 而是围绕代码数据、仓库级预训练和指令后训练做专项增强.

## 1. 引言

代码专用 LLM 在 StarCoder、CodeLlama、DeepSeek-Coder、CodeQwen1.5 和 Codestral 等模型推动下已经成为开源社区的关键方向. 但与 Claude-3.5-Sonnet、GPT-4o 这类闭源模型相比, 无论开源还是商业代码模型, 在综合代码能力上仍然存在差距.

Qwen2.5-Coder 的目标非常直接:

- 让 32B 级别的开源代码模型在核心基准上对齐甚至逼近 GPT-4o
- 用完整的六个开源尺寸覆盖从小模型补全到大模型代码代理的不同需求
- 把“代码模型”从简单的补全器推进成更像编程助手的系统

## 2. 架构与分词设计

### 2.1 架构延续

Qwen2.5-Coder 直接继承 Qwen2.5 的基础架构. 六个尺度共享相同的主干设计, 包括:

- GQA, Grouped Query Attention
- SwiGLU
- RoPE
- RMSNorm
- 与尺寸配套的 hidden size、层数和 KV head 配置

模型规模覆盖:

- 0.5B
- 1.5B
- 3B
- 7B
- 14B
- 32B

所有模型统一使用 151,646 词表, 并在 5.5T token 上训练.

### 2.2 代码专用 token

Qwen2.5-Coder 在 Qwen2.5 基础词表上加入一组代码专用 token, 其中最关键的是:

- `<|fim_prefix|>`
- `<|fim_middle|>`
- `<|fim_suffix|>`
- `<|fim_pad|>`
- `<|repo_name|>`
- `<|file_sep|>`

这些 token 的作用不是装饰, 而是把文件级 FIM 和仓库级 FIM 变成可以统一编码的训练格式.

> 译者注: `<|repo_name|>` 和 `<|file_sep|>` 的引入非常关键. 这说明团队并不满足于“让模型学会补一段代码”, 而是开始让模型显式看到仓库级结构信号. 对代码模型来说, 文件边界、仓库名、路径分隔本身就是一种高价值语义.

## 3. 预训练数据与数据混合

Qwen2.5-Coder 的预训练数据由五大部分组成:

- 源代码数据
- 文本-代码关联数据
- 合成数据
- 数学数据
- 通用文本数据

其中, 最重要的工程投入在两个方向:

1. 大规模高质量代码语料构建
2. 代码、数学、文本的混合比例验证

报告特别强调了一个关键结论: 纯代码并不是最优配方. 在对比 100:0:0、85:10:5 和 70:20:10 的实验后, 团队最终选择 70% 代码、20% 文本、10% 数学.

这个结果说明代码能力和数学、文本理解之间并不是零和关系. 相反, 适量的通用和数学语料能显著提升模型在综合代码任务中的表现.

## 4. 三阶段训练流程

Qwen2.5-Coder 的训练主线分三步走:

### 4.1 文件级预训练

这一阶段聚焦单文件内部的语法、结构与语义建模:

- 上下文长度 8,192
- 训练目标包括 next-token prediction 和文件级 FIM
- 使用高质量 5.2T 代码相关数据

### 4.2 仓库级预训练

第二阶段把上下文从单文件扩展到仓库级:

- 训练上下文 32,768
- 推理通过 YARN 外推到 128K
- 使用约 300B 高质量长上下文代码数据
- 使用 repo-level FIM 格式, 让模型学习跨文件依赖

这一步本质上是在教模型理解:

- 哪个文件定义了哪个 API
- import 指向哪些模块
- 多文件之间的接口关系如何组织

### 4.3 指令微调

最后一阶段把模型从“代码基础模型”转成“代码助手”:

- 构建代码相关高质量指令数据
- 覆盖代码生成、调试、修复、推理、SQL、表格理解等场景
- 进一步通过后训练对齐人类偏好和实际编程需求

> 译者注: 这条三阶段路线比只做“继续预训练 + SFT”更成熟. 文件级预训练负责打牢基础, 仓库级预训练负责补齐真实项目上下文, 最后再用指令微调做产品化落地. 这基本对应了代码模型从“语法熟练”走向“工程可用”的完整升级路径.

## 5. 后训练与代码对齐

Qwen2.5-Coder 的后训练不是简单复制通用模型套路, 而是针对代码场景做了增强.

### 5.1 SFT 数据构建

SFT 数据重点覆盖:

- 长序列生成
- 数学推理
- 代码生成与修复
- 指令遵循
- 结构化数据理解
- 逻辑推理
- 跨语言迁移
- 鲁棒系统提示
- 输出质量过滤

### 5.2 离线 RL

离线 RL 主要处理那些具有明确正负例、可验证的任务, 如:

- 数学
- 代码
- 指令遵循
- 逻辑推理

其核心是通过执行反馈、答案匹配、自动验证等手段构建稳定训练信号.

### 5.3 在线 RL

在线 RL 则进一步通过 GRPO 和多维奖励模型, 优化:

- 真实性
- 有用性
- 简洁性
- 相关性
- 无害性
- 去偏

这让 Qwen2.5-Coder 不只是“会写代码”, 还更接近一个稳定、能合作的代码助手.

> 译者注: 这是 Qwen 路线的重要转折点. 从 Qwen2 的 DPO 主导, 走到 Qwen2.5 把 GRPO 纳入主训练链路, 说明团队已经把在线 RL 当作可工程化能力来处理了. 这也是 Qwen2.5 比 Qwen2 更像“平台型模型”的原因之一.

## 6. 关键评测结果

Qwen2.5-Coder 在多个维度表现强劲:

- 代码生成: HumanEval、MBPP 等基准达到同尺寸开源 SOTA
- 代码补全: FIM、CrossCodeEval、RepoEval 等场景表现强
- 代码推理: CRUXEval 和多语言代码理解能力显著提升
- 代码编辑: 在 Aider、CodeEditorBench 等任务上竞争力很强
- Text-to-SQL 与表格理解: 说明它不是窄代码模型, 而是具备结构化推理能力

其中, 7B 模型的表现尤其值得关注: 在多个代码基准上超过参数规模更大的竞争模型, 说明数据和训练系统确实比简单堆参数更重要.

## 7. 讨论与定位

Qwen2.5-Coder 的价值不在于“重新发明代码模型架构”, 而在于把代码模型的完整训练流程做成一个成熟系统:

- 数据侧有结构化清洗和混合验证
- 训练侧有三阶段上下文扩展
- 后训练侧有离线与在线 RL 分工
- 产品侧有多尺寸矩阵覆盖不同部署区间

它在整个代码模型谱系中的位置很清晰:

- 继承 Qwen2.5 通用能力
- 专项强化代码能力
- 为后续更强的 Qwen3-Coder-Next 和 agent 型模型奠定基础
