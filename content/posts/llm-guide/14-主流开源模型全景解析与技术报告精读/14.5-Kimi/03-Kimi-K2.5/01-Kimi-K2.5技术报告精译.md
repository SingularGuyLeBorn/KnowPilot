## 0. 摘要与核心指标

Kimi K2.5 是一个开源的多模态智能体模型,旨在推进通用智能体智能的发展.K2.5 强调文本与视觉的联合优化,使两种模态相互增强.为此,团队提出了一系列关键技术,包括文本-视觉联合预训练、Zero-Vision SFT 以及文本-视觉联合强化学习.

在此多模态基础之上,K2.5 进一步引入了 Agent Swarm,一种自主驱动、并行协作的智能体编排框架,能够动态地将复杂任务分解为多个异构子问题并并发执行.

**核心性能(与顶级闭源模型对比)**:

K2.5 在编程、视觉理解、逻辑推理及智能体任务等多个领域均达到当前最优水平.Agent Swarm 相较单智能体基线,最高可降低延迟达 4.5 倍.团队已公开发布经过后训练的 Kimi K2.5 模型Checkpoint.

**模型规格**: 总参数 1.04T, 激活参数 32B, 384 个 Expert(每 token 激活 8 个, 稀疏比 48x), 上下文窗口 256K tokens, 基于 MoE Transformer 架构.

## 1. 引言

大型语言模型正快速向智能体智能演进.GPT-5.2、Claude Opus 4.5、Gemini 3 Pro 和 Kimi K2-Thinking 等近期进展表明,智能体能力(特别是在工具调用和推理方面)取得了实质性进步.这些模型越来越展现出将复杂问题分解为多步计划并执行推理与行动交错的长序列的能力.

本报告介绍 Kimi K2.5 的训练方法与评估结果.具体而言,团队在以下两个关键方面对 K2.5 进行了改进.

**文本与视觉的联合优化**. K2.5 实践的一个关键洞察是,文本与视觉的联合优化可以增强两种模态并避免冲突.为此,团队设计了一套技术.预训练期间,与传统方法在训练后期才向文本主干添加视觉 token 不同,团队发现早期视觉融合配合较低比例在固定总视觉-文本 token 预算下往往产生更好结果.因此,K2.5 在整个训练过程中以恒定比例混合文本和视觉 token.

架构上,Kimi K2.5 采用 MoonViT-3D,一种原生分辨率视觉Encoder ,结合 NaViT packing 策略,支持可变分辨率图像输入.对于视频理解,团队引入轻量级 3D ViT 压缩机制:连续帧以四帧为一组,经过共享 MoonViT Encoder 处理,并在 patch 级别进行时间平均.这种设计使 Kimi K2.5 在相同上下文窗口内处理长达 4 倍的视频,同时保持图像和视频Encoder 之间的完全权重共享.

后训练期间,团队引入 Zero-Vision SFT——仅用文本 SFT 即可激活视觉推理和工具使用.团队发现在此阶段添加人工设计的视觉轨迹反而会损害泛化.相比之下,纯文本 SFT 表现更好——可能是因为联合预训练已经建立了强视觉-文本对齐,使能力能够自然地跨模态泛化.随后,团队在文本和视觉任务上应用联合 RL.关键的是,团队发现视觉 RL 增强文本性能而非降低文本性能,在 MMLU-Pro 和 GPQA-Diamond 上均有提升.这种双向增强——文本启动视觉、视觉精炼文本——代表了联合训练中优越的跨模态对齐.

**Agent Swarm: 并行智能体编排**. 大多数现有智能体系统依赖工具调用的顺序执行.即使能够进行数百步推理的系统(如 Kimi K2-Thinking)也遭受推理时间的线性扩展,导致不可接受的延迟并限制任务复杂度.随着智能体工作负载在范围和异构性上增长,顺序范式变得越来越低效.

为克服顺序智能体执行的延迟和可扩展性限制,Kimi K2.5 引入 Agent Swarm,一种动态并行智能体编排框架.团队提出 Parallel-Agent Reinforcement Learning(PARL)范式,不同于传统智能体 RL.除了通过可验证奖励优化工具执行外,模型还配备子智能体创建和任务委托的接口.训练期间,子智能体被冻结,其执行轨迹被排除在优化目标之外;仅编排器通过强化学习更新.这种解耦规避了两个端到端协同优化的挑战:信用分配模糊和训练不稳定性.Agent Swarm 使复杂任务能够被分解为由领域专业化智能体并发执行的异构子问题,将任务复杂度从线性扩展转化为并行处理.在宽搜索场景中,Agent Swarm 相较单智能体基线将推理延迟降低高达 4.5 倍,同时将 Item-level F1 从 72.8% 提升至 79.0%.

## 2. 文本与视觉的联合优化

Kimi K2.5 是一个原生多模态模型,基于 Kimi K2 通过在约 15 万亿混合视觉和文本 token 上进行大规模联合预训练构建.与牺牲语言或视觉能力的视觉适配模型不同,团队的联合预训练范式同时增强两种模态.

### 2.1 原生多模态预训练

多模态预训练的一个关键设计问题是:给定固定的视觉-文本 token 预算,最优的视觉-文本联合训练策略是什么.传统认知认为,在 LLM 训练后期以高比例(如 50% 或更高)引入视觉 token 应该加速多模态能力获取,将多模态能力视为语言能力的事后附加.

然而,实验(如表 1 和图 9 所示)揭示了不同的情况.团队进行了消融研究,在保持视觉和文本 token 预算固定的前提下改变视觉比例和视觉注入时机.为严格满足不同比例的目标,团队在引入视觉数据之前先用纯文本 token 预训练模型特定数量的 token.令人惊讶的是,团队发现视觉比例对最终多模态性能的影响极小.事实上,早期融合配合较低视觉比例在固定总视觉-文本 token 预算下产生更好结果.这促使团队采用原生多模态预训练策略:不是将大量视觉训练集中在末尾,而是在训练过程的早期就整合适度的视觉比例,让模型自然地发展平衡的多模态表示,同时受益于两种模态的扩展协同优化.

**表 1: 不同视觉-文本联合训练策略的性能对比**

| 视觉注入时机 | 视觉-文本比例 | 视觉知识 | 视觉推理 | 视觉 OCR | 文本知识 | 文本推理 | 文本代码 |
|-------------|-------------|---------|---------|---------|---------|---------|---------|
| Early | 0% | 25.8 | 43.8 | 65.7 | 45.5 | 58.5 | 24.8 |
| Mid | 50% | 25.0 | 40.7 | 64.1 | 43.9 | 58.6 | 24.0 |
| Late | 80% | 24.2 | 39.0 | 61.5 | 43.1 | 57.8 | 24.0 |

*注: 早期融合配合较低视觉比例在几乎所有指标上全面优于中期和晚期融合.*

> **思考节点-数据实验**: 表 1 的实验设计有一个关键约束——三种策略消耗的视觉 token 总量相同.Late Fusion 在训练后 80% 才注入视觉数据,为了消化相同总量的视觉 token,它需要把视觉比例提到 50%;而 Early Fusion 从头就混入视觉数据,10% 的比例就能覆盖相同的总量.附录 B.1 展示了一个"dip-and-recover"现象:Late Fusion 在注入视觉数据时,文本能力会先显著下降再逐渐恢复(模态域迁移冲击),而 Early Fusion 的训练曲线保持平稳.但这里有一个需要谨慎解读的地方:这个结论的前提是"固定视觉 token 总预算".如果视觉 token 预算不受限,Late Fusion 配合更高比例是否可能取得更好结果?论文没有回答这个问题.对于拥有充足计算资源的团队,这个最优策略可能不同.

### 2.2 Zero-Vision SFT

预训练的 VLM 不会自然执行基于视觉的工具调用,这为多模态 RL 带来了冷启动问题.传统方法通过人工标注或提示工程化的思维链数据来解决此问题,但此类方法在多样性上受限,通常将视觉推理限制于简单图表和原始工具操作(裁剪、旋转、翻转).

一个观察是,高质量的文本 SFT 数据相对丰富且多样.团队提出一种新方法 Zero-Vision SFT,仅使用文本 SFT 数据来激活后训练期间的视觉和智能体能力.在此方法中,所有图像操作都通过 IPython 中的程序化操作代理,有效作为传统视觉工具使用的泛化.这种"零视觉"激活实现了多样化的推理行为,包括像素级操作(如通过二值化和计数进行物体大小估计),并泛化到视觉基础任务(如物体定位、计数和 OCR).

图 2 展示了 RL 训练曲线,其起点来自 Zero-Vision SFT.结果表明,Zero-Vision SFT 足以激活视觉能力,同时确保跨模态的泛化.这种现象可能是由于第 2.1 节所述的文本和视觉数据的联合预训练.与 Zero-Vision SFT 相比,初步实验显示文本-视觉 SFT 在视觉和智能体任务上表现更差,可能是因为缺乏高质量的视觉数据.

> **思考节点-设计动机**: Zero-Vision SFT 是 K2.5 中最反直觉的发现之一.传统认知认为,要教模型"看懂图像并操作",必须提供带图像的标注数据.但 K2.5 证明,纯文本 SFT(所有图像操作通过 IPython 代码代理)反而比加入人工视觉轨迹的效果更好.其机制可能是:联合预训练已经建立了视觉-文本的强对齐,因此模型只需要学会"用代码描述视觉操作"这一抽象能力,就能泛化到真实视觉输入.代码作为桥梁提供了精确的操作语义——`cv2.inRange` 就是精确的颜色过滤,不存在歧义.这也解释了为什么"文本-视觉 SFT"反而更差:有限的人工视觉数据不仅多样性不足,还可能导致模型过拟合于特定的标注风格.但这个结论依赖于"前期联合预训练足够强"的前提,对于预训练阶段视觉数据不足或对齐质量较差的模型,Zero-Vision SFT 可能无法生效.

### 2.3 联合多模态强化学习

本节描述 K2.5 中实现有效多模态 RL 的方法论,从基于结果的视觉 RL 到增强文本性能的涌现跨模态迁移.

**基于结果的视觉 RL**. Zero-Vision SFT 之后,模型需要进一步精炼以可靠地将视觉输入纳入推理.仅文本启动的激活存在显著的失败模式:视觉输入有时被忽略,图像在需要时可能未被关注.团队在明确要求视觉理解才能获得正确解决方案的任务上采用基于结果的 RL.这些任务分为三个领域:
- 视觉定位与计数: 图像内物体的准确定位和枚举;
- 图表与文档理解: 结构化视觉信息的解释和文本提取;
- 视觉关键型 STEM 问题: 筛选出需要视觉输入的数学和科学问题.

基于结果的 RL 在这些任务上改善了基础视觉能力和更复杂的智能体行为.提取这些轨迹用于拒绝采样微调(RFT),可实现自我改进的数据流水线,允许后续联合 RL 阶段利用更丰富的多模态推理痕迹.

**表 2: 跨模态迁移——视觉 RL 改善文本知识**

| 基准 | 视觉 RL 之前 | 视觉 RL 之后 | 提升 |
|------|------------|------------|------|
| MMLU-Pro | 84.7 | 86.4 | +1.7 |
| GPQA-Diamond | 84.3 | 86.4 | +2.1 |
| LongBench v2 | 56.7 | 58.9 | +2.2 |

> **思考节点-数据实验**: 表 2 展示了一个在 RL 文献中极少被报道的现象:在视觉任务上进行 RL 训练后,纯文本基准(MMLU-Pro、GPQA-Diamond)的性能反而提升了.论文给出的解释是"视觉 RL 增强了模型对结构化信息提取的校准能力,这种能力迁移到了类似需要结构化信息提取的文本任务中".但更深层的原因可能是:视觉任务强迫模型学会"仔细检查输入、提取关键信息、结构化推理"的模式,而这些模式恰好也是文本推理任务所需要的.这与 K2.5 按能力维度(知识/推理/代码/智能体)而非输入模态来组织 RL 领域的设计选择一致——同一能力维度下的文本和视觉任务共享 reward 信号,最大化了跨模态迁移.不过,这个结论的普适性需要谨慎对待:如果视觉 RL 的数据分布与文本任务差异过大,或者视觉任务过于简单,跨模态迁移可能不会发生甚至产生负迁移.

## 3. Agent Swarm: 并行智能体编排

现有智能体系统的主要挑战在于依赖推理和工具调用步骤的顺序执行.虽然这种结构对较简单、短程任务可能有效,但随着任务复杂度增加和累积上下文增长,它变得不足.当任务演变为包含广泛信息收集和复杂多分支推理时,顺序系统经常遇到显著瓶颈.

单个智能体一步步处理每个步骤的有限容量可能导致实际推理深度和工具调用预算的耗尽,最终阻碍系统处理更复杂场景的能力.

为解决此问题,团队引入 Agent Swarm 和 Parallel Agent Reinforcement Learning(PARL).K2.5 不是将任务执行为推理链或依赖预指定的并行化启发式,而是通过动态任务分解、子智能体实例化和并行子任务调度来启动 Agent Swarm.重要的是,并行性并非被假定为固有优势;关于是否、何时以及如何并行的决策通过环境反馈和 RL 驱动的探索显式学习.

### 3.1 架构与学习设置

PARL 框架采用解耦架构,包含一个可训练的编排器和从固定中间策略Checkpoint实例化的冻结子智能体.此设计刻意避免端到端协同优化,以规避两个根本性挑战:信用分配模糊和训练不稳定性.在此多智能体设置中,基于结果的奖励本质上是稀疏且嘈杂的;正确的最终答案不能保证子智能体执行完美,正如失败也不意味着所有子智能体都出错.通过冻结子智能体并将其输出视为环境观察而非可微决策点,团队将高层协调逻辑与低层执行能力解耦,从而实现更稳健的收敛.

为提高效率,团队首先使用小型子智能体训练编排器,然后过渡到更大的模型.RL 框架还支持动态调整子智能体与编排器之间的推理实例比例,从而最大化集群的资源利用率.

### 3.2 PARL 奖励

训练可靠的并行编排器具有挑战性,因为独立子智能体执行存在延迟、稀疏和非平稳的反馈.为此,团队定义 PARL 奖励为:

$$
r_{PARL}(x, y) = \lambda_1 \cdot r_{parallel} + \lambda_2 \cdot r_{finish} + r_{perf}(x, y)
$$

其中 $r_{perf}$ 评估给定任务 $x$ 的解决方案 $y$ 的整体成功和质量.这由两个辅助奖励增强,每个解决并行编排学习中的不同挑战.$r_{parallel}$ 用于缓解序列崩溃——一种编排器默认回退到单智能体执行的局部最优.通过激励子智能体实例化,此项鼓励探索并发调度空间.$r_{finish}$ 关注分配子任务的成功完成,用于防止虚假并行——一种编排器通过生成大量无意义任务分解的子智能体来急剧增加并行指标的奖励 hacking 行为.通过奖励完成的子任务,$r_{finish}$ 强制执行可行性并引导策略走向有效分解.

为确保最终策略优化主要目标,超参数 $\lambda_1$ 和 $\lambda_2$ 在训练过程中退火至零.

> **思考节点-设计动机**: PARL 奖励设计中的三个组件分工明确:$r_{perf}$ 是最终目标(任务质量),$r_{parallel}$ 是探索激励(防止编排器"偷懒"只做顺序执行),$r_{finish}$ 是可行性约束(防止编排器"滥发"子智能体却不完成任何任务).这种"主目标 + 辅助探索奖励 + 退火"的设计思路与课程学习中"逐步减少辅助信号"的做法一致.但这里有一个工程细节:超参数 $\lambda_1$ 和 $\lambda_2$ 的退火速度需要仔细调优——退火太快,编排器可能还没学会并行就只剩 $r_{perf}$;退火太慢,最终策略可能被辅助奖励主导而偏离任务质量最优.论文没有披露具体的退火 schedule,这是复现时的一个关键未知量.

### 3.3 Critical Steps 作为资源约束

为衡量并行智能体设置中的计算时间成本,团队通过类比计算图中的关键路径来定义 critical steps.团队将 episode 建模为由 $t = 1, ..., T$ 索引的执行阶段序列.在每个阶段,主智能体执行一个动作,对应直接工具调用或实例化一组并行运行的子智能体.令 $S_{main}^{(t)}$ 表示阶段 $t$ 中主智能体采取的步数(通常 $S_{main}^{(t)} = 1$),$S_{sub,i}^{(t)}$ 表示该并行组中第 $i$ 个子智能体在该阶段采取的步数.阶段 $t$ 的持续时间由该组中最长运行的子智能体决定.因此,episode 的总 critical steps 定义为:

$$
\text{CriticalSteps} = \sum_{t=1}^{T} S_{main}^{(t)} + \max_i S_{sub,i}^{(t)}
$$

通过使用 critical steps 而非总步数来约束训练和评估,框架显式激励有效并行化.不减少并行组最大执行时间的过度子任务创建在此度量下收益甚微,而缩短最长并行分支的均衡任务分解直接减少 critical steps.因此,编排器被鼓励以最小化端到端延迟的方式分配工作,而非仅仅最大化并发度或总工作量.

> **思考节点-架构细节**: Critical Steps 的定义是 Agent Swarm 的工程精髓.如果只用"总步数"来衡量,编排器可能通过生成大量短步长子智能体来"刷"低总步数,但实际的 wall-clock 时间并未减少(因为最长分支决定了整体延迟).Critical Steps 相当于计算图中的关键路径长度,它迫使编排器关注"瓶颈子任务"——那些执行时间最长的分支.这与并行计算中的 Amdahl 定律一致:加速比受限于最长串行部分.从工程实现看,测量 critical steps 需要精确的子智能体执行时间追踪,这在分布式 RL 环境中增加了额外的监控开销,但换来的训练信号质量提升是值得的.

### 3.4 并行智能体能力诱导的提示构建

为激励编排器利用并行化的优势,团队构建了一套合成提示,旨在给顺序智能体执行施加压力.这些提示强调宽搜索(需要同时探索多个独立信息源)或深搜索(需要多个推理分支和延迟聚合).团队还包含受真实世界工作负载启发的任务,如长上下文文档分析和大规模文件下载.顺序执行时,这些任务难以在固定推理步数和工具调用预算内完成.通过构造,它们鼓励编排器并行分配子任务,使得在比单顺序智能体可行的更少的 critical steps 内完成.重要的是,提示不会显式指示模型并行化;相反,它们塑造任务分布,使得并行分解和调度策略自然受到青睐.

## 4. 方法总览

### 4.1 基座: Kimi K2

Kimi K2.5 的基座是 Kimi K2,一个在 15 万亿高质量文本 token 上预训练的万亿参数 MoE Transformer 模型.Kimi K2 采用 token 高效的 MuonClip 优化器(带 QK-Clip 以保证训练稳定性),包含 1.04T 总参数和 32B 激活参数,使用 384 个 Expert(每 token 激活 8 个,稀疏比 48).关于 MuonClip、架构设计和训练基础设施的详细描述,请参阅 Kimi K2 技术报告.

### 4.2 模型架构

Kimi K2.5 的多模态架构由三个组件组成:三维原生分辨率视觉Encoder  MoonViT-3D、MLP projector 和 Kimi K2 MoE 语言模型,遵循 Kimi-VL 确立的设计原则.

**MoonViT-3D: 图像与视频的共享嵌入空间**. 在 Kimi-VL 中,团队采用 MoonViT 以原生分辨率处理图像,消除复杂的子图像拆分和拼接操作.从 SigLIP-SO-400M 初始化,MoonViT 结合 NaViT 的 patch packing 策略,单张图像被划分为 patches,展平并顺序拼接成 1D 序列,从而支持不同分辨率图像的高效同时训练.

为最大化图像理解能力向视频的迁移,团队引入 MoonViT-3D,采用统一架构、完全共享参数和一致的嵌入空间.通过将"patch n' pack"理念推广到时间维度,最多四个连续帧被视为一个时空体:这些帧的 2D patches 被联合展平并打包成单个 1D 序列,使相同的注意力机制在空间和时间上无缝运行.虽然额外的时间注意力改善了对高速运动和视觉效果的理解,但共享机制最大化了从静态图像到动态视频的知识泛化,在无需专用视频模块或架构分叉的情况下实现强大的视频理解性能(见表 4).在 MLP projector 之前,轻量级时间池化在每个时间块内聚合 patches,产生 4 倍时间压缩以显著扩展可行视频长度.结果是统一的流水线,图像预训练获得的知识和能力通过共享参数空间和特征表示整体迁移到视频.

> **思考节点-架构细节**: MoonViT-3D 的设计体现了"极简主义"的工程哲学.传统视频理解模型通常使用独立的视频Encoder (如 TimeSformer、Video Swin Transformer),导致图像和视频的知识无法共享.K2.5 的做法是将视频视为"多帧图像"——4 个连续帧的 patches 被打包进同一个 1D 序列,用同一个 Transformer 处理,最后通过时间池化压缩 4 倍.这种设计的优势是:1) 图像预训练的权重直接适用于视频,无需微调;2) 架构简单,维护成本低;3) 时间池化是可学习的,模型可以自适应地决定哪些时间信息需要保留.代价是:对于需要精细时间建模的任务(如动作识别中的毫秒级时间差异),简单的 patch-level 平均可能丢失关键信息.但从表 4 的视频基准结果(VideoMMMU 86.6%、LVBench 75.9%)来看,这种权衡在实践中是有效的.

### 4.3 预训练流水线

如表 3 所示,Kimi K2.5 的预训练基于 Kimi K2 语言模型Checkpoint,在约 15T token 上跨三个阶段处理:首先,独立 ViT 训练以建立稳健的原生分辨率视觉Encoder ;其次,联合预训练以同时增强语言和多模态能力;第三,基于高质量数据和长上下文激活的中期训练,以精炼能力并扩展上下文窗口.

**表 3: 训练阶段总览**

| 阶段 | 数据 | 序列长度 | Token 量 | 可训练组件 |
|------|------|---------|---------|-----------|
| ViT 训练 | Alt text, Synthesis, Caption, Grounding, OCR, Video | 4096 | 1T | ViT |
| 联合预训练 | + Text, Knowledge, Interleaving, Video, OS Screenshot | 4096 | 15T | ViT & LLM |
| 联合长上下文中期训练 | + 高质量文本与多模态,长文本,长视频推理,长 CoT | 32768/262144 | 500B/200B | ViT & LLM |

**ViT 训练阶段**. MoonViT-3D 从 SigLIP 在图像-文本和视频-文本对上持续预训练,文本组件包含多种目标:图像 alt 文本、图像和视频的合成 caption、grounding bbox 和 OCR 文本.与 Kimi-VL 的实现不同,此持续预训练不包含对比损失,而仅包含用于以输入图像和视频为条件的 caption 生成的交叉熵损失 $L_{caption}$.团队采用两阶段对齐策略.第一阶段,更新 MoonViT-3D 以通过 caption 损失与 Moonlight-16B-A3B 对齐,消耗约 1T token 和非常少的训练 FLOPs.此阶段使 MoonViT-3D 主要理解高分辨率图像和视频.随后的非常短的第二阶段仅更新 MLP projector,以桥接 ViT 与 1T LLM,实现更平滑的联合预训练.

**联合训练阶段**. 联合预训练阶段从接近末尾的 Kimi K2 Checkpoint继续,在额外的 15T 视觉-文本 token 上以 4K 序列长度进行.数据配方扩展 Kimi K2 的预训练分布,引入独特的 token、调整数据比例(增加代码相关内容权重)并控制每个数据源的最大 epoch.第三阶段执行长上下文激活,结合集成的更高质量中期训练数据,通过 YaRN 插值顺序扩展上下文长度.这在长上下文文本理解和长视频理解方面产生显著的泛化改进.

**预训练数据亮点**. 文本语料包含四个主要领域:Web Text、Code、Mathematics 和 Knowledge.代码数据显著扩展了:(1) 支持跨文件推理和架构理解的仓库级代码,(2) 来自互联网捕获真实世界开发模式的 Issues、Code Reviews 和 Commit Histories,(3) 从 PDF 和 webtext 语料库检索的代码相关文档.

视觉数据包含七类:caption、interleaving、OCR、knowledge、video、agent 和 grounding.特别值得注意的是 image-code paired data——HTML/React/SVG 等代码格式与其对应的渲染截图配对,使模型能够将抽象结构逻辑与具体视觉几何对齐.对于智能体和时间理解,团队收集桌面、移动和 Web 环境的 GUI 截图和动作轨迹,包括人工标注演示.

### 4.4 后训练

**监督微调**. 遵循 Kimi K2 建立的 SFT 流水线,团队通过从 K2、K2 Thinking 和一系列专有的内部专家模型合成高质量候选响应来开发 K2.5.数据生成策略采用针对特定领域定制的专门流水线,整合人工标注与高级提示工程和多阶段验证.此方法产生大规模指令调优数据集,具有多样化的提示和复杂的推理轨迹,最终训练模型优先考虑交互式推理和精确的工具调用.

**强化学习**. RL 构成后训练的关键阶段.为促进跨文本和视觉模态的联合优化以及 PARL 智能体群,团队开发统一智能体强化学习环境(附录 D)并优化 RL 算法.

**策略优化**. 对于从数据集 $D$ 采样的问题 $x$,使用先前策略 $\pi_{old}$ 生成 $K$ 个响应 $\{y^1, ..., y^K\}$.模型 $\pi$ 关于以下目标优化:

$$
L_{RL}(\pi) = \mathbb{E}_{x \sim D} \frac{1}{N} \sum_{j=1}^{K} \sum_{i=1}^{|y^j|} \left[ \text{Clip}\left(\frac{\pi(y_i^j|x, y_{0:i}^j)}{\pi_{old}(y_i^j|x, y_{0:i}^j)}, \epsilon_{low}, \epsilon_{high}\right) \cdot (r(x, y^j) - \bar{r}(x)) - \beta \cdot \log\frac{\pi(y_i^j|x, y_{0:i}^j)}{\pi_{old}(y_i^j|x, y_{0:i}^j)} \right]
$$

其中 $\epsilon_{low}, \epsilon_{high}, \beta > 0$ 为超参数,$y_{0:i}^j$ 为第 $j$ 个响应中第 $i$ 个 token 之前的前缀,$N = \sum_{i=1}^{K} |y_i|$ 为 batch 中生成的总 token 数,$\bar{r}(x) = \frac{1}{K} \sum_{j=1}^{K} r(x, y^j)$ 为所有生成响应的平均奖励.

此损失函数通过引入 token-level clipping 机制偏离 K1.5 使用的策略优化算法,旨在缓解由训练和推理框架差异放大的 off-policy 发散.该机制作为简单的梯度掩码方案:log-ratio 在区间 $[\epsilon_{low}, \epsilon_{high}]$ 内的 token 正常计算策略梯度,而超出此范围的 token 梯度被置零.与标准 PPO clipping 的关键区别在于,此方法严格依赖 log-ratio 显式约束 off-policy 漂移,无论优势符号如何.团队采用 MuonClip 优化器最小化此目标.

**奖励函数**. 团队对具有可验证解决方案的任务(如推理和智能体任务)应用基于规则的结果奖励.为优化资源消耗,还纳入旨在增强 token 效率的 budget-control reward.对于通用任务,采用生成式奖励模型(GRM)提供与 Kimi 内部价值标准对齐的细粒度评估.此外,对于视觉任务,设计任务特定的奖励函数以提供细粒度监督:视觉定位和点定位任务使用基于 F1 的奖励配合软匹配(IoU 和高斯加权距离);OCR 任务采用归一化编辑距离;计数任务基于预测与真值的绝对差分配奖励.团队还合成复杂视觉谜题问题并利用 LLM verifier(Kimi K2)提供反馈.

**生成式奖励模型**. Kimi K2 利用自批判 rubric reward 进行开放式生成,K2.5 通过在不同环境(聊天助手、编码智能体、搜索智能体、工件生成智能体)中系统部署 GRM 来扩展此工作.GRM 不是二元裁决者,而是作为与 Kimi 价值对齐的细粒度评估器,如有用性、响应就绪度、上下文相关性、细节适当性、生成工件的美学质量和严格指令遵循.为缓解 reward hacking 和对单一偏好信号的过拟合,团队采用针对不同任务定制的多套 GRM rubric.

**Token 高效强化学习: Toggle**. Token 效率对于测试时扩展的 LLM 至关重要.团队先前的发现表明,施加问题相关的预算可有效约束推理时计算,激励模型生成更简洁的思维链而不进行不必要的 token 扩展.然而,团队也观察到 length-overfitting 现象:在严格预算约束下训练的模型往往无法泛化到更高的计算规模.

为此,团队提出 Toggle,一种在推理时扩展和预算约束优化之间交替的训练启发式:

$$
\tilde{r}(x, y) = \begin{cases} r(x, y) \cdot \mathbb{I}\left[\frac{1}{K}\sum_{i=1}^{K} r(x, y_i) \geq \tau \text{ 或 } |y_i| \leq \text{budget}(x)\right] & \text{if } t/m \pmod 2 = 0 \text{ (Phase0)} \\ r(x, y) & \text{if } t/m \pmod 2 = 1 \text{ (Phase1)} \end{cases}
$$

具体而言,算法每 $m$ 次迭代在两个优化阶段之间交替:
- **Phase0(预算限制阶段)**: 模型被训练在任务相关的 token 预算内解决问题.为防止过早牺牲质量换取效率,此约束有条件应用:仅当给定问题的平均准确率超过阈值 $\tau$ 时才强制执行.
- **Phase1(标准扩展阶段)**: 模型生成响应直至最大 token 限制,鼓励模型利用计算获得更好的推理时扩展.

问题相关的预算从正确响应子集的 token 长度的 $\alpha$ 百分位数估计:

$$
\text{budget}(x) = \text{Percentile}\left(\{|y_i| \mid r(x, y_i) = 1, i = 1, ..., K\}, \alpha\right)
$$

此预算在训练开始时估计一次并此后固定.Toggle 作为双目标问题的随机交替优化,专门设计用于协调推理能力与计算效率.在 K2 Thinking 上的评估显示,Toggle 在几乎所有基准上一致减少输出长度,平均减少 25-30% 的输出 token,对性能影响可忽略.

> **思考节点-设计动机**: Toggle 是 K2.5 后训练中最精妙的算法设计之一.它解决的核心矛盾是:"预算约束"可以让模型学会简洁推理,但"过度约束"会导致 length-overfitting——模型在测试时无法利用额外的 token 预算来解决更困难的问题.Toggle 的解决方案是交替进行:Phase0 教模型"如何在有限预算内高效解决问题",Phase1 教模型"如何在充足预算内深入推理".两个阶段的交替频率($m$)和预算阈值($\tau$)是关键超参:如果 $m$ 太小,模型在两个模式间震荡而无法稳定学习;如果 $m$ 太大,模型可能在某个阶段过拟合.论文没有披露具体的 $m$ 和 $\tau$ 值,但从"平均减少 25-30% 输出 token"的结果来看,这个权衡是成功的.这暗示了一个更一般的原则:对于测试时扩展模型,"效率"和"深度"不是互斥的,而是可以通过交替训练同时获得的.

### 4.5 训练基础设施

Kimi K2.5 继承 Kimi K2 的训练基础设施并做最小修改.对于多模态训练,团队提出 Decoupled Encoder Process(DEP),将视觉Encoder 以可忽略的额外开销纳入现有流水线.

**Decoupled Encoder Process(DEP)**. 在典型的使用 Pipeline Parallelism(PP) 的多模态训练范式中,视觉Encoder 和文本嵌入共置于流水线的第一阶段(Stage-0).然而,由于多模态输入大小的固有变化(如图像数量和分辨率),Stage-0 在计算负载和内存使用方面遭受剧烈波动.这迫使现有解决方案采用定制的 PP 配置——例如 Kimi-VL 手动调整 Stage-0 中的文本Decoder  层数以预留内存.虽然这种妥协缓解了内存压力,但并未从根本上解决多模态输入大小引起的负载不均衡.更关键的是,它排除了直接复用已为纯文本训练高度优化的并行策略.

利用视觉Encoder 在计算图中的独特拓扑位置——具体而言,其作为前向传播起点和反向传播终点的角色——团队的训练使用 DEP,每个训练步骤由三个阶段组成:
- **均衡视觉前向**: 首先在所有视觉数据上执行全局 batch 的前向传播.由于视觉Encoder 较小,团队在所有 GPU 上复制它,无论其他并行策略如何.此阶段前向计算负载基于负载指标(如图像或 patch 数量)均匀分布在所有 GPU 上,消除 PP 和视觉 token 数量引起的负载不均衡.为最小化峰值内存使用,丢弃所有中间激活,仅保留最终输出激活;
- **主干训练**: 执行主 Transformer 主干的前向和反向传播.通过在前一阶段丢弃中间激活,现在可以充分利用任何在纯文本训练中验证过的高效并行策略.此阶段后,梯度在视觉Encoder 输出处累积;
- **视觉重计算与反向**: 重新计算视觉Encoder 前向传播,随后执行反向传播以计算视觉Encoder 参数的梯度.

DEP 不仅实现负载均衡,还解耦视觉Encoder 和主主干的优化策略.K2.5 无缝继承 K2 的并行策略,实现相对于纯文本训练 90% 的多模态训练效率.

> **思考节点-架构细节**: DEP 是一个看似简单但工程价值极高的设计.其核心洞察是:视觉Encoder 在计算图中处于"起点"(前向第一个节点)和"终点"(反向最后一个节点)的特殊位置,这意味着它可以被独立处理.传统方案把视觉Encoder 和文本主干放在同一个 PP stage,导致 Stage-0 的负载严重不均衡(有图像的 batch vs 纯文本 batch).DEP 的三阶段流程把视觉Encoder 从 PP 中"剥离"出来:第一阶段在所有 GPU 上并行处理视觉输入(负载均衡),第二阶段用标准 PP 处理文本主干(复用 K2 的优化),第三阶段重计算视觉梯度.90% 的效率意味着多模态训练几乎不增加额外开销——这对于 15T token 规模的训练来说,节省的计算成本是巨大的.值得注意的是,H800 GPU 集群(8x 400Gbps RoCE)的硬件配置表明,K2.5 的训练规模可能达到数千张 GPU.

## 5. 评估

### 5.1 主要结果

**评估设置**. 团队在全面的基准套件上评估 Kimi K2.5,涵盖基于文本的推理、竞争性和智能体编程、多模态理解(图像和视频)、自主智能体执行和计算机使用.

- **推理与通用**: HLE, AIME 2025, HMMT 2025 (Feb), IMO-AnswerBench, GPQA-Diamond, MMLU-Pro, SimpleQA Verified, AdvancedIF, LongBench v2.
- **编程**: SWE-Bench Verified, SWE-Bench Pro, SWE-Bench Multilingual, Terminal Bench 2.0, PaperBench(CodeDev), CyberGym, SciCode, OJBench(cpp), LiveCodeBench(v6).
- **智能体能力**: BrowseComp, WideSearch, DeepSearchQA, FinSearchComp(T2&T3), Seal-0, GDPVal.
- **图像理解**: MMMU-Pro, MMMU(val), CharXiv(RQ), MathVision, MathVista(mini), SimpleVQA, WorldVQA 2, ZeroBench, BabyVision, BLINK, MMVP, OCRBench, OmniDocBench 1.5, InfoVQA.
- **视频理解**: VideoMMMU, MMVU, MotionBench, Video-MME, LongVideoBench, LVBench.
- **计算机使用**: OSWorld-Verified, WebArena.

基线包括闭源模型 Claude Opus 4.5(扩展思考模式)、GPT-5.2(xhigh 推理强度)、Gemini 3 Pro(高思考级别),以及开源模型 DeepSeek-V3.2(思考模式)和 Qwen3-VL-235B-A22B-Thinking(仅视觉基准).

除非另有说明,所有 Kimi K2.5 评估使用 temperature = 1.0, top-p = 0.95, 上下文长度 256K tokens.

**评估结果**.

**推理与通用**. Kimi K2.5 在严格 STEM 基准上与顶级闭源模型取得竞争性性能.AIME 2025 上 K2.5 得分 96.1%,接近 GPT-5.2 的满分,同时超越 Claude Opus 4.5(92.8%)和 Gemini 3 Pro(95.0%).HMMT 2025 得 95.4%,IMO-AnswerBench 81.8%,展现卓越的推理深度.MMLU-Pro 87.1%,GPQA-Diamond 87.6%.HLE 无工具时 Full 得分 30.1%(文本 31.5%,图像 21.3%);启用工具后 HLE-Full 升至 50.2%(文本 51.8%,图像 39.8%),显著超越 Gemini 3 Pro(45.8%)和 GPT-5.2(45.5%).

### 5.2 Agent Swarm 结果

**基准**. 为严格评估 Agent Swarm 框架的有效性,团队选择三个代表性基准,共同覆盖深度推理、大规模检索和真实世界复杂性:
- BrowseComp: 需要多步推理和复杂信息合成的深度研究基准.
- WideSearch: 评估跨多样源进行广泛多步信息搜寻和推理的能力.
- In-house Swarm Bench: 内部开发的 Swarm 基准,覆盖 WildSearch(无约束真实世界开放网络信息检索)、Batch Download(大规模多样化资源获取)、WideRead(涉及 100+ 输入文档的大规模文档理解)和 Long-Form Writing(生成超过 100k 词的连贯内容).

|------|-----------------|-----------|-----------------|--------|------------|

Agent Swarm 在 BrowseComp 上达 78.4%,相较单智能体 K2.5(60.6%)绝对提升 17.8%,甚至超越 GPT-5.2 Pro(77.9%).WideSearch Item-F1 提升 6.3%(72.7% → 79.0%),超越 Claude Opus 4.5(76.2%).In-house Swarm Bench 提升最显著(16.7%).

**并行执行的时间节省**. 除性能提升外,Agent Swarm 通过并行子智能体执行实现显著的 wall-clock 时间减少.在 WideSearch 基准上,它相较单智能体基线将达到目标性能所需的执行时间降低 3 到 4.5 倍.如图 8 所示,此效率增益随任务复杂度扩展:当目标 Item-F1 从 30% 增至 70% 时,单智能体执行时间从约 1.8 倍增长到超过 7.0 倍基线,而 Agent Swarm 保持在 0.6 到 1.6 倍的近恒定低延迟范围内.

**动态子智能体创建与调度**. Agent Swarm 内,子智能体是动态实例化而非预定义的.通过 PARL,编排器学习自适应策略来根据演化的任务结构和问题状态创建和调度自托管子智能体.与静态分解方法不同,这种学习到的策略使编排器能够基于查询推理所需子智能体的数量、时机和专业化.

**Agent Swarm 作为主动式上下文管理**. Agent Swarm 是一种由多智能体架构实现的主动式智能上下文管理.这与测试时上下文截断策略(如 Hide-Tool-Result、Summary 或 Discard-all)不同,后者通过压缩或丢弃累积历史来应对上下文溢出.虽然这些方法有效减少 token 使用,但本质上是反应式的,经常牺牲结构信息或中间推理.

相比之下,Agent Swarm 通过显式编排实现主动式上下文控制.长程任务被分解为并行的、语义隔离的子任务,每个由具有受限局部上下文的专业化子智能体执行.关键的是,这些子智能体保持独立工作记忆并执行局部推理,不直接修改或污染中央编排器的全局上下文.仅任务相关的输出——而非完整交互轨迹——被选择性路由回编排器.这种设计诱导上下文分片而非上下文截断,使系统能够沿额外的架构维度扩展有效上下文长度,同时保持模块化、信息局部性和推理完整性.

> **思考节点-局限风险**: Agent Swarm 的上下文分片设计虽然巧妙,但并非没有代价.当任务需要子智能体之间的频繁信息交换或共享状态时,独立的工作记忆可能导致信息孤岛——编排器需要显式地在子智能体之间传递中间结果,这增加了通信开销.此外,"仅任务相关输出路由回编排器"意味着子智能体的详细推理过程对编排器不可见,如果子智能体的输出是错误但表面上合理的,编排器可能无法及时发现.这与人类团队中"每个成员独立工作、只向经理汇报结果"的模式类似:高效但存在"黑箱"风险.论文中的 Black Myth: Wukong 示例(32 个视频、24 小时、40GB)展示了 Agent Swarm 在长视频分析上的威力,但这种极端场景对编排器的聚合能力提出了极高要求——32 个子智能体的输出如何被有效综合为一个连贯的 HTML 页面,其中的信息丢失和矛盾处理机制尚未被充分讨论.

## 6. 结论

Kimi K2.5 表明,可扩展的通用智能体智能可以通过文本与视觉的联合优化以及并行智能体执行来实现.通过统一语言和视觉的预训练与强化学习,模型实现了强大的跨模态对齐和视觉-文本推理.Agent Swarm 使异构子任务的并发执行成为可能,降低推理延迟的同时提升复杂智能体工作负载上的性能.基于视觉-文本智能和智能体群,Kimi K2.5 在基准测试和真实世界任务上展现出强大性能.通过开源后训练Checkpoint,团队旨在支持开源社区构建可扩展的通用智能体系统,并加速迈向通用智能体智能的进程.
