# 6.1.1 分布式训练

> 本文系统梳理大模型分布式训练的核心技术，涵盖 3D 并行(DP/TP/PP)、ZeRO 系列优化器、序列并行(Sequence Parallelism)、激活Checkpoint(Activation Checkpointing)等关键技术，并对比 FSDP2、DeepSpeed、Megatron-LM 三大主流框架的选型策略. 

---

## 2. 3D 并行：DP、TP、PP

3D 并行是指**数据并行(Data Parallelism)、张量并行(Tensor Parallelism)、流水线并行(Pipeline Parallelism)** 三种并行策略的组合. 这三种策略分别从不同维度对训练过程进行拆分. 

### 2.1 数据并行(DP)

数据并行是最直观的并行方式：**将训练数据切分到多张 GPU 上，每张 GPU 持有完整的模型副本，各自计算梯度后通过 All-Reduce 通信同步梯度**. 

```
GPU 0: 数据批次 0 → 前向 → 反向 → 梯度
GPU 1: 数据批次 1 → 前向 → 反向 → 梯度
GPU 2: 数据批次 2 → 前向 → 反向 → 梯度
GPU 3: 数据批次 3 → 前向 → 反向 → 梯度
                ↓ All-Reduce 同步
           所有 GPU 获得平均梯度
```

**优点**：
- 实现简单，代码侵入性低
- 计算粒度均匀，流水线气泡少

**缺点**：
- 每张卡需要存储完整模型参数 + 优化器状态 + 梯度
- 对于大模型(如 70B+)，单卡显存不够存储完整模型

**分布式数据并行(DDP)** 是 PyTorch 原生的 DP 实现，使用 `torch.distributed` 完成通信. 与普通 DP 不同，DDP 不依赖 Python 多线程，而是每个进程独立运行在单独的 GPU 上，避免了 GIL 限制. 

### 2.2 张量并行(TP)

张量并行将**单个算子(如 Linear 层、Attention 层)的计算拆分到多张 GPU 上**. 最经典的是 Megatron-LM 提出的列并行(Column-wise)和行并行(Row-wise)切分方式. 

对于线性层 $Y = XA$：

- **列并行**：将权重矩阵 $A$ 按列切分为 $[A_1, A_2]$，分别在不同 GPU 上计算 $XA_1$ 和 $XA_2$，结果拼接
- **行并行**：将权重矩阵 $A$ 按行切分为 $\begin{bmatrix} A_1 \\ A_2 \end{bmatrix}$，在不同 GPU 上计算 $X_1A_1 + X_2A_2$，结果累加

**关键通信模式**：
| 计算阶段 | 通信操作 | 说明 |
|:--------|:--------|:------|
| 前向列并行 | All-Gather | 拼接各 GPU 的部分输出 |
| 前向行并行 | Reduce-Scatter | 累加各 GPU 的部分输出 |
| 反向传播 | All-Reduce | 同步各 GPU 的梯度 |

**优点**：
- 降低单卡计算量和显存需求
- 适合大矩阵运算(Attention、FFN 层)

**缺点**：
- 通信频繁，每层前/反向都需要 All-Reduce
- 对卡间带宽要求极高(通常需要 NVLink/NVSwitch)

### 2.3 流水线并行(PP)

流水线并行将**模型的层按顺序分配到不同的 GPU 上**. GPU 0 负责前几层，GPU 1 负责中间层，GPU 2 负责后面层，数据像流水线一样依次经过各 GPU. 

```
GPU 0: 层 1-10 → 输出 → GPU 1: 层 11-20 → 输出 → GPU 2: 层 21-30
       ← 梯度 ←             ← 梯度 ←               ← 梯度 ←
```

**流水线气泡问题**：简单实现下，GPU 0 处理完 micro-batch 0 后需要等待 GPU 1 处理 micro-batch 0 才能处理下一个 batch，导致 GPU 空闲(即"气泡"). 解决方法包括：

| 方法 | 说明 | 气泡率 |
|:----|:------|:------|
| **朴素 PP** | 一次只处理一个 micro-batch | 高 |
| **Gradient Accumulation** | 将 batch 拆分为多个 micro-batch 流水执行 | 中 |
| **1F1B 调度** | One-Forward-One-Backward，交错执行前向和反向 | **低** ~10-15% |
| **交错式 1F1B** | 将每个 GPU 的层再拆分，微调流水线粒度 | **更低** ~5-10% |

**优点**：
- 极大降低单卡显存占用(每卡只保存模型的一部分层)
- 卡间通信量相对较小(仅传递层间的 hidden states)

**缺点**：
- 存在流水线气泡，GPU 利用率受限于流水线深度
- 实现复杂度较高(需处理 micro-batch 调度)

### 2.4 3D 并行组合策略

实践中，大模型训练通常**组合使用上述三种并行策略**：

```
数据并行(DP)      GPU 0 ←→ GPU 1    ←→    GPU 2 ←→ GPU 3
                     ↓ 张量并行 ↑            ↓ 张量并行 ↑
                     GPU 4 ←→ GPU 5    ←→    GPU 6 ←→ GPU 7
                   ←──────── 流水线并行 ────────→
```

以训练一个 100B 模型为例，典型的组合方式：
- **张量并行(TP=8)** ：在 8 张 GPU 组成的节点内拆分 Attention/FFN 算子
- **流水线并行(PP=4)** ：将模型层分为 4 组，每组分配到不同的 GPU 组
- **数据并行(DP=2)** ：在 2 个 GPU 组之间做数据并行

总 GPU 数 = $TP \times PP \times DP = 8 \times 4 \times 2 = 64$ 张 GPU

### 2.5 MoE 模型的专家并行(EP)与混合部署

混合专家模型(MoE)的推理与训练引入了第四种并行维度——**专家并行(Expert Parallelism, EP)** . 与将模型参数均匀拆分到所有 GPU 的 TP/PP 不同，EP 将不同的专家网络分配到不同的 GPU 上，每张 GPU 只负责计算一部分专家的输出. 

**MoE 模块的计算流程**可分解为四个阶段：

1. **Gate 计算**：输入经过 RMSNorm 后，通过路由网络(Router)计算每个 token 对各专家的亲和度分数，选择 Top-K 个专家. 
2. **Dispatch**：根据路由结果，将每个 token 的激活发送到其选中的专家所在的 GPU. 这涉及跨 GPU 的 All-to-All 通信. 
3. **Expert GEMM**：每个专家独立执行前向计算(通常包含 Gate-Up 投影、SiLU 激活、Down 投影). 
4. **Combine**：将各专家的输出按路由权重加权求和，通过 All-to-All 通信将结果聚合回原始 token 位置. 

**DeepGEMM 的两种 Layout 优化**：

| Layout 模式 | 适用场景 | 核心特征 |
|:-----------|:--------|:--------|
| **Contiguous Layout** | Prefill / 训练前向 | 将各专家接收的 token 按顺序拼接为连续 buffer，通过 offset/index 逻辑划分，每个专家对齐到 128 的倍数，最大化 Tensor Core 利用率 |
| **Masked Layout** | Decoding + CUDA Graph | 采用 mask-based grouped GEMM，替代显式分组，兼容 CUDA Graph 的静态图约束，无冗余计算浪费，通过精细控制获得更好性能 |

**多卡 TP 推理 MoE 模型**：在张量并行下，Attention 和 MoE Block 的权重按列/行切分到不同 GPU. 每层 Transformer Block 存在两个 All-Reduce(Attention 输出投影 + MoE Down 投影)，推理过程中还有两个 All-Gather(Embedding 输出 + 最终 logits). 通信模式与 Dense 模型类似，但 MoE 层增加了 Expert 内部的局部计算. 

**多卡 EP 推理 MoE 模型**：在专家并行下，不同专家被分配到不同 GPU. 假设 4 张 GPU、8 个专家，则每张 GPU 存放 2 个专家. 路由后的 All-to-All 通信将 token 发送到目标专家所在的 GPU，计算完成后再 All-to-All 聚合. EP 的优势在于单卡只需存储部分专家参数，显存占用与专家数成正比降低; 代价是引入额外的 All-to-All 通信. 

**TP + EP 混合并行**：工业部署通常采用 TP 与 EP 的混合策略——在节点内(如 8 张 GPU)使用 TP 拆分 Attention 和共享权重，跨节点使用 EP 分配专家. 例如 DeepSeek-V3 的部署：节点内 TP=8 处理 Attention 和共享 FFN，跨节点 EP=16 分配 256 个专家. 这种混合策略在通信效率和显存利用率之间取得平衡：TP 的 All-Reduce 发生在高带宽 NVLink 域内，EP 的 All-to-All 通过 IB/RoCE 跨节点传输.

## 3. ZeRO 系列优化器

ZeRO(Zero Redundancy Optimizer)由 Microsoft DeepSpeed 团队提出，核心思想是 **消除数据并行中的内存冗余**. 在传统数据并行中，每张 GPU 都保存完整的模型状态(参数 + 梯度 + 优化器状态)，存在大量冗余. 

### 3.1 ZeRO 的三个级别

| 级别 | 分片内容 | 显存节省 | 通信开销 |
|:----|:--------|:--------|:--------|
| **ZeRO-1** | 优化器状态分片 | 4 倍 | 少量 |
| **ZeRO-2** | 优化器状态 + 梯度分片 | 8 倍 | 中 |
| **ZeRO-3** | 优化器状态 + 梯度 + 权重分片 | $N_d$ 倍($N_d$ 为 DP 度) | 高(频繁 AllGather) |

**ZeRO-1**：将优化器状态(如 Adam 的动量 $m$ 和方差 $v$)分片到 $N_d$ 张 GPU 上，每张卡只维护 $\frac{1}{N_d}$ 的优化器状态. 每张 GPU 计算完整梯度后，只更新自己维护的那部分参数，然后通过通信获取完整参数. 

**ZeRO-2**：在 ZeRO-1 的基础上，将梯度也分片. 反向传播时，梯度通过 Reduce-Scatter 分散到各 GPU，每张卡只维护自己分片的梯度. 优化器更新后，再 All-Gather 获取完整权重. 

**ZeRO-3**：最激进的方案，将模型权重也分片. 在前向传播前通过 All-Gather 获取所需层的权重，前向结束后释放非本卡维护的权重. 反向传播类似. 

### 3.2 ZeRO-Offload 与 ZeRO-Infinity

当 GPU 显存实在不够时，ZeRO 提供了将数据卸载到 CPU 内存甚至 NVMe SSD 的能力：

| 技术 | 卸载目标 | 适用场景 |
|:----|:--------|:--------|
| **ZeRO-Offload** | CPU 内存 | 单卡或小集群，GPU 显存不足 |
| **ZeRO-Infinity** | CPU 内存 + NVMe SSD | 超大模型，需要近乎无限的存储空间 |

**ZeRO-Offload** 的核心策略：将**优化器状态和梯度**卸载到 CPU 内存，GPU 只保留模型权重和计算所需的临时数据. CPU 负责优化器更新(Adam 状态更新)，GPU 负责前向/反向计算. 

**ZeRO-Infinity** 进一步将权重也可以卸载到 NVMe SSD，实现了**训练任意规模模型**的可能性——理论上，只要 CPU 内存和 SSD 空间足够，可以在有限 GPU 上训练万亿参数模型. 

### 3.3 ZeRO 与数据并行的关系

ZeRO 本质上是对**数据并行的显存优化**，它不改变计算图，只是改变了数据分布方式. 因此，ZeRO 可以与其他并行策略(TP、PP)组合使用：

| 组合 | 适用场景 |
|:----|:--------|
| **ZeRO-3 + TP** | 模型过大，需张量并行降低单卡计算量，同时利用 ZeRO 节省显存 |
| **ZeRO-2 + PP** | 流水线并行已有显存节省，用 ZeRO-2 进一步降低优化器状态冗余 |
| **ZeRO-Offload + TP + PP** | 极致显存优化，适合在有限 GPU 上训练极大模型 |

## 4. 序列并行(Sequence Parallelism)

标准 Transformer 训练中，自注意力机制的计算复杂度和显存占用与序列长度 $L$ 呈 **二次关系** $O(L^2)$. 当序列长度达到 128K 甚至 1M 时，注意力计算的显存占用成为主要瓶颈. 

序列并行将**序列维度上的计算和显存进行拆分**，解决长序列训练问题. 

### 4.1 序列并行的动机

以一个 Llama 70B 模型为例，序列长度对显存的影响：

| 序列长度 | 注意力显存 | 总训练显存 | 瓶颈 |
|:-------:|:---------:|:---------:|:----:|
| 4K | 16 MB | ~140 GB | 权重 + 优化器状态 |
| 32K | 1 GB | ~145 GB | 权重为主 |
| 128K | 16 GB | ~160 GB | 注意力显存显著 |
| **1M** | **1 TB** | **>1 TB** | **注意力显存成为绝对瓶颈** |

当序列长度超过 128K 时，**注意力计算的显存需求急剧上升**，仅靠数据并行和模型并行无法有效解决. 

### 4.2 DeepSpeed Ulysses(序列并行)

DeepSpeed Ulysses 是微软 2024 年提出的序列并行方案，核心思想是将**注意力计算的 QKV 投影和注意力分数计算拆分为多个序列片段**. 

**工作原理**：
1. 将长度为 $L$ 的序列平均切分为 $P$ 个片段，分配给 $P$ 个计算节点
2. 每个节点只计算自己片段的注意力输出
3. 通过 All-to-All 通信在节点间交换必要的信息

**通信模式**：All-to-All 通信，每个节点与其他所有节点交换数据. 

**优点**：通信量随序列长度线性增长，适合超长序列(128K+)

### 4.3 Ring Attention(环形注意力)

Ring Attention 是另一种序列并行方案，它利用**环状通信模式**，将注意力计算分散到多个 GPU 上：

1. 将长序列切分为 $P$ 个块，$P$ 张 GPU 各持一个块
2. GPU 执行环形通信，逐轮交换 key/value 块
3. 每轮计算当前 GPU 的 query 与收到的 key/value 的部分注意力
4. 最终结果通过 Reduce-Scatter / All-Gather 聚合

**关键优势**：通信与计算可重叠(overlap)，Ring Attention 可以在计算当前轮注意力时，同时传输下一轮所需的 key/value，有效隐藏通信延迟. 

### 4.4 序列并行与其他并行的组合

序列并行通常与 3D 并行结合使用，形成 **4D 并行**：

```
序列并行(SP) + 张量并行(TP) + 流水线并行(PP) + 数据并行(DP)
```

以 128K 上下文训练为例：
- **TP=8**：节点内拆分 Attention/FFN
- **PP=4**：将模型层分为 4 组
- **SP=2**：将序列分为 2 个片段
- **DP=2**：数据副本数

总 GPU 数 = $8 \times 4 \times 2 \times 2 = 128$ 张 GPU

## 5. 激活Checkpoint(Activation Checkpointing)

### 5.1 核心原理

激活Checkpoint(又称梯度Checkpoint Gradient Checkpointing)是一种**用计算换显存**的技术. 在标准前向传播中，每一层都会保存中间激活值(Activations)用于反向传播计算梯度. 这些激活值可能占用大量显存——对于一个 70B 模型，单层激活值可能达到 GB 级别. 

激活Checkpoint的核心思想：**在前向传播时丢弃大部分中间激活值，在反向传播时重新计算它们**. 

```
标准模式：
前向 → 保存激活 A1 → 保存激活 A2 → ... → 保存激活 An → 损失
反向 → 使用 A1 计算梯度 → 使用 A2 计算梯度 → ...

Checkpoint模式：
前向 → 不保存激活 → 不保存激活 → ... → 不保存激活 → 损失
反向 → 重新计算激活 A1 → 使用 A1 计算梯度 → 重新计算 A2 → ...
```

### 5.2 计算与显存的权衡

| 策略 | 激活显存占用 | 额外计算开销 | 适用场景 |
|:----|:-----------|:-----------|:--------|
| 无Checkpoint | 最高 | 无 | 显存充足的短序列训练 |
| 选择性Checkpoint | 中等 | ~15-20% | 显存有限的通用场景 |
| 全Checkpoint | 最低 | ~30-33% | 显存极度紧缺的超大模型 |

**选择性Checkpoint**：不是对所有层做Checkpoint，而是对**显存占用大、计算量相对小**的层做Checkpoint. 例如：
- Attention 层：激活占用大，但重计算开销相对较小 → **适合做Checkpoint**
- FFN 层：激活占用大，但重计算开销也大 → **可做可不做**
- Embedding 层：激活占用小 → **不Checkpoint**

### 5.3 实现方式

PyTorch 提供了 `torch.utils.checkpoint.checkpoint` 函数，可以轻松包装需要Checkpoint的模块：

```python
import torch.utils.checkpoint as checkpoint

class TransformerBlock(nn.Module):
    def forward(self, x):
        # 包装需要Checkpoint的计算
        def custom_forward(*inputs):
            x = self.attention(inputs[0])
            x = self.ffn(x)
            return x
        return checkpoint.checkpoint(custom_forward, x)
```

在 DeepSpeed 和 Megatron 框架中，激活Checkpoint是内置功能，通过配置文件即可启用：

```json
// DeepSpeed 配置
{
    "activation_checkpointing": {
        "partition_activations": true,
        "number_checkpoints": 100,
        "cpu_checkpointing": false
    }
}
```

## 6. 三大主流框架对比：FSDP2 vs DeepSpeed vs Megatron-LM

当前大模型分布式训练的三个主流框架各有侧重，选型取决于模型规模、团队技术栈和硬件条件. 

### 6.1 框架定位

| 框架 | 出身 | 核心定位 | 设计哲学 |
|:----|:----|:--------|:--------|
| **FSDP2** | PyTorch 官方(2023) | 易用性优先的通用型框架 | 0 代码侵入，原生兼容 |
| **DeepSpeed** | 微软+清华 | 极致显存效率和性能 | ZeRO 优化，显存榨干机 |
| **Megatron-LM** | NVIDIA | 超大模型工业级并行 | 稳定性第一，千亿/万亿级 |

### 6.2 全面对比

| 对比维度 | FSDP2 | DeepSpeed | Megatron-LM |
|:--------|:-----|:---------|:-----------|
| **代码侵入性** | **0 侵入** — 原生 PyTorch 模型直接封装 | 中侵入 — 需修改模型 + JSON 配置文件 | **高侵入** — 需按 Megatron 规范重写模型 |
| **学习成本** | 极低 — 会 PyTorch 就会用 | 中 — 需理解 ZeRO 配置 | 极高 — 需理解并行逻辑 + 重写模型 |
| **并行体系** | DP 为主，原生组合 TP/PP/EP | 四维并行(DP+TP+PP+ZeRO+EP) | TP/PP 优先，三维并行 |
| **显存优化** | 全分片(对标 ZeRO-2/3) | **ZeRO 系列绝对第一**(1/2/3/Offload/Infinity) | 张量分片 + 专家隔离 |
| **训练性能** | 优秀(比 FSDP1 快 30%+) | **极致**(通信压缩 + 梯度量化) | **顶尖**(TP/PP 效率最高) |
| **超大模型** | 千亿级可应对 | **万亿级可应对**(ZeRO-Infinity) | **万亿级最稳定** |
| **MoE 支持** | 动态+细粒度 | 静态+动态混合，负载均衡最优 | 静态+粗粒度，TP 强绑定 |
| **硬件兼容** | 全兼容(NVIDIA/AMD/CPU) | 主要 NVIDIA | **仅 NVIDIA** |
| **社区活跃度** | PyTorch 官方，持续迭代 | 微软维护，文档完善 | NVIDIA 维护，企业级支持 |

### 6.3 选型建议

| 场景 | 推荐框架 | 理由 |
|:----|:--------|:-----|
| **个人/小团队**中小模型(7B~70B) | **FSDP2** | 0 侵入，学习成本最低，落地最快 |
| **大厂**千亿级模型(GPT-3 规模) | **Megatron-LM** | TP/PP 稳定性天花板 |
| **MoE 超大模型**(1.6T+) | **DeepSpeed** | ZeRO + EP 极致优化 |
| **多框架混用** | DeepSpeed + Megatron | DeepSpeed ZeRO + Megatron TP/PP |
| **快速迭代验证** | FSDP2 | 换模型不改代码，实验速度快 |

**核心判断原则**：
- 如果团队**熟悉 PyTorch**，96% 的场景选 **FSDP2** 即可
- 如果追求 **极致性能+显存**，选 **DeepSpeed**
- 如果训练 **万亿级模型**，选 **Megatron-LM**
- **三者可以混用**，各取所长

---

ZeRO 系列优化器，就能针对不同模型规模和硬件条件设计最合适的最佳训练方案.
