## 0. 核心洞察：千卡训练不是八卡训练的简单放大

千卡训练和八卡训练的区别不是"显卡多了一百多倍"，而是**通信时间和故障概率的指数级增长**. 

**通信时间**：PyTorch的AllReduce操作在128节点下，即使走Tree AllReduce($O(\log p \cdot n)$)，理论上也比单节点慢至少7倍. 实践中跨节点通信远比单节点慢得多. 

**故障概率**：设单节点故障概率为 $p = 1\%$，128个节点的故障概率为：

$$P_{\text{fail}} = 1 - (1 - p)^{128} = 1 - 0.99^{128} \approx 72.37\% \tag{0.1}$$

这意味着：**如果训练超过2048 GPU日，单卡或单节点下线是再正常不过的事情**. 

**何时需要千卡训练**：如果你的训练时间没有超过8192 GPU日，那么你绝对不需要一千张显卡. 千卡训练解决的是**大模型+大数据**问题——模型和数据都以B(十亿)为计量单位.

## 1. 训练耗时的解剖

一个训练step的耗时分布在以下环节：

```
dataset读取数据 → dataloader collate → 模型forward → loss compute
→ 模型backward → 梯度sync → 优化器step → 打印log
```

**关键原则**：除了forward/backward/loss/sync是真耗时，其他都必须通过**异步操作**来掩盖. 

**Profile方法**：
1. 测整个step的时长
2. 在每次测量前执行手动同步(`cuda.synchronize` + `dist.barrier`)
3. 计算每个部分的精确时长
4. 若总耗时远大于4-7步之和，说明异步未生效，存在优化空间

## 2. 通信优化

### 2.1 计算-通信重叠

PyTorch中梯度的通信和反向传播是**交叠进行**的：每完成一层的梯度计算，立即触发当前层的同步. 

**实现机制**：每个进程完成第 $k$ 层梯度计算后，触发钩子给计数器+1. 当计数器达到进程数时，进行梯度通信. 

**常见错误**：`RuntimeError: Expected to have finished reduction in the prior iteration before starting a new one.`

**根因**：有的模块没有参与计算loss，导致梯度同步卡住. 

**解决**：当`find_unused_parameters=True`时，PyTorch使用`nn.Module.__init__`中定义sub-module的反向顺序作为梯度桶的构建顺序. 确保模块定义和调用顺序一致. 

### 2.2 梯度合桶(Gradient Bucketing)

**问题**：每次发起通信都有开销(overhead). 同步越频繁，总开销越大. 

**解决方案**：PyTorch引入梯度合桶机制，将多个Tensor装进一个桶再通信，减少通信次数. 

```python
# PyTorch DDP的梯度合桶配置
torch.nn.parallel.DistributedDataParallel(
    model,
    device_ids=[local_rank],
    bucket_cap_mb=25,  # 桶容量(MB)，默认25，需根据硬件调整
)
```

**调参建议**：
- 高速互联(NVLink/IB)：桶容量可设大(50-100MB)
- 低速互联(以太网)：桶容量设小(10-25MB)
- PyTorch默认参数(25MB)从0.x时代祖传，通常需要调节

### 2.3 梯度累加(Gradient Accumulation)

**核心洞察**：超过256卡的训练，想把通信完全盖掉几乎不可能. 

**梯度累加策略**：执行 $k$ 次forward+backward后再执行优化器step. 

**好处**：
1. 提升等效batch size(大模型batch size通常受限)
2. 累加期间的backward不需要通信梯度，加快训练速度

```python
# 梯度累加示例
accumulation_steps = 4
for step, batch in enumerate(dataloader):
    loss = model(batch)
    loss = loss / accumulation_steps  # 归一化
    loss.backward()
    
    if (step + 1) % accumulation_steps == 0:
        optimizer.step()
        optimizer.zero_grad()
```

**数值走查**：设单步通信耗时 $t_{\text{comm}} = 100$ms，计算耗时 $t_{\text{comp}} = 200$ms. 
- 无累加：每步总耗时 $= 200 + 100 = 300$ms
- 累加4步：每4步通信1次，等效每步总耗时 $= 200 + 100/4 = 225$ms
- **加速比**：$300 / 225 = 1.33$x

### 2.4 大小梯度同步

**创新思路**：梯度同步不应以GPU/进程为单位，而应分为：
- **小同步**(节点内同步)：高频执行，关注局部梯度
- **大同步**(节点间同步)：低频执行，关注全局梯度

**优势**：
1. 提高实际梯度同步频率，降低同步总耗时
2. 天然结合小batch和大batch训练的优点——节点内小batch关注个体，节点间大batch关注整体

## 3. 代码效率：少即是快

### 3.1 Python的性能陷阱

Python是慢语言. 即使JIT trace + torch.compile有提升，对于最高效率来说：**只有必须要存在的代码和不存在的代码两种**. 

**反例**：Transformers库的过度封装
- 两个sub-Module就能写完的TransformerLayer，被封装成多层继承
- 不考虑继承的封装引入额外函数调用开销

**正例**：PyTorch的`nn.functional`
- 第一行往往是`handle_torch_func`
- 不用装饰器统一，因为装饰器引入额外函数调用

**工程原则**：
- 写简单的训练代码和模型代码
- 1%的效率提升，节省的可能是数百个GPU日

### 3.2 异步执行优化

| 操作 | 异步机制 | 手动同步方法 |
|------|---------|------------|
| DataLoader | `num_workers` + `prefetch_count` | 设为0关闭异步 |
| CUDA计算 | kernel launch异步 | `cuda.synchronize()` |
| 分布式通信 | NCCL异步 | `dist.barrier()` |

**Profile技巧**：在每次测量前执行手动同步，才能获取真实的耗时分布.

## 4. 平稳训练：故障不是意外，是常态

### 4.1 捕捉非致命异常

在千卡训练中，大部分异常都是**非致命**的. 捕捉它们即可继续训练. 

**最常见的非致命异常**：
1. **存checkpoint写满磁盘**：从商汤到深势到上海AI Lab，到处都有出现
   - 解决：catch保存操作，回调里删除之前的ckpt
2. **存log写满硬盘**
   - 解决：catch所有logging操作，用tmux开长缓存窗口抢救log
3. **联网操作失败**：从ceph读取数据、写log到远程
   - 解决：catch所有联网操作

**不应捕捉的错误**：集群炸了(如所有节点同时失联)

```python
# 异常捕捉装饰器示例
def catch_nonfatal(func):
    """捕捉非致命异常，记录日志后继续"""
    @wraps(func)
    def wrapper(*args, **kwargs):
        try:
            return func(*args, **kwargs)
        except Exception as e:
            logger.warning(f"Non-fatal error in {func.__name__}: {e}")
            # 根据操作类型决定返回什么
            if 'checkpoint' in func.__name__:
                return None  # checkpoint失败可跳过
            elif 'log' in func.__name__:
                return None  # log失败可跳过
            raise
    return wrapper

@catch_nonfatal
def save_checkpoint(model, path):
    torch.save(model.state_dict(), path)

@catch_nonfatal  
def write_log(metrics):
    wandb.log(metrics)
```

### 4.2 实验管理与自动恢复

**三级实验目录结构**：`project/experiment-run/timestamp`
- `project`：用户给出
- `experiment-run`：通过代码版本和配置计算
- `timestamp`：运行开始时间

**自动恢复**：若代码和配置完全一样，系统认为这是同一个运行，自动找最新的checkpoint加载. 

```python
# 伪代码：自动恢复逻辑
def auto_resume(project_dir):
    if config.auto_resume:
        runs = sorted(glob(f"{project_dir}/*/"), key=os.path.getmtime)
        if runs:
            latest_run = runs[-1]
            checkpoints = sorted(glob(f"{latest_run}/checkpoint-*"))
            if checkpoints:
                return load_checkpoint(checkpoints[-1])
    return None
```

### 4.3 收敛稳定性

**常见措施**：
- 输出和loss出现NaN：果断丢弃当前step
- 梯度裁剪：先clip by value，再clip by norm
- 初始化：使用适合大模型的初始化策略(如Kaiming、Xavier的变体)

## 5. 弹性训练：当节点必然下线时

### 5.1 弹性训练的需求

当训练超过2048 GPU日时，单个GPU甚至单个节点下线是**常态**而非意外. 

**理想方案**：每 $k$ 分钟做一次AllReduce统计存活进程数，选举主进程，重新分配rank和local rank. 

**现实**：PyTorch 1.10引入的torchelastic机制并不好用，2024年仍未有成熟的工业级方案. 

### 5.2 工程 workaround

在成熟弹性方案出现前，可采用的策略：
1. **高频checkpoint**：每100-500步保存一次，缩短回滚距离
2. **冗余节点**：预留5-10%的备用节点，故障时快速替换
3. **小同步策略**：节点内同步高频，节点间同步低频，降低单节点故障影响范围

## 6. 工业实践：真实集群的配置

### 6.1 硬件拓扑对比

| 配置 | DGX节点 | 普惠服务器 |
|------|---------|-----------|
| GPU接口 | SXM | PCI-E |
| GPU数量/节点 | 8 | 8 |
| IB网卡/节点 | 8(每GPU一张) | 1(节点共享) |
| 节点间带宽 | 200-400 Gbps | 25-100 Gbps |
| 典型问题 | 负载均衡、路由 | 通信瓶颈、带宽争抢 |

**关键结论**：同样是A100集群，硬件拓扑不同，遇到的问题完全不同. 没有profile的优化是没有意义的. 

### 6.2 典型故障率

| 集群规模 | 单卡故障概率 | 集群故障概率 |
|---------|------------|------------|
| 8卡(1节点) | 1% | 1% |
| 64卡(8节点) | 1% | 7.7% |
| 256卡(32节点) | 1% | 27.6% |
| 1024卡(128节点) | 1% | 72.4% |

**应对**：
- <256卡：手动重启+checkpoint回滚
- 256-1024卡：高频checkpoint+小同步策略
- >1024卡：弹性训练机制(待成熟)

## 7. 失效模式

| 场景 | 症状 | 根因 | 缓解 |
|------|------|------|------|
| 通信未重叠 | 同步时间远高于计算时间 | backward和allreduce未交叠 | 检查模块定义顺序，确保梯度桶构建正确 |
| 梯度桶过大 | 通信延迟抖动 | 桶容量超过网络缓冲 | 调小`bucket_cap_mb` |
| 梯度桶过小 | 通信开销占比高 | 通信次数过多 | 调大`bucket_cap_mb` |
| DataLoader阻塞 | step时间不稳定 | num_workers不足或prefetch不够 | 增加workers，预加载数据 |
| checkpoint写满 | 训练中断 | 磁盘空间不足 | 自动删除旧ckpt，监控磁盘 |
| OOM | 进程崩溃 | 某张卡显存不足 | 减小batch size，开启activation checkpointing |
| NaN发散 | loss变为NaN | 梯度爆炸或数值不稳定 | gradient clip，检查初始化 |
