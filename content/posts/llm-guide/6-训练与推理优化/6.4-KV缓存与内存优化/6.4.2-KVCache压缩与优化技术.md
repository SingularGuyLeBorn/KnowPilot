# 01 · KV Cache 压缩与优化：量化、低秩、动态剪枝与跨层共享

## 1. 背景与核心痛点 (Background & Pain Points)

### 1.1 自回归生成的内存黑洞

在大语言模型(LLM)的推理过程中, 存在一个非常隐蔽但致命的内存消耗来源. 当你向模型发送一条长 prompt, 模型开始逐字逐字地生成回复时, 它并不是每次只读你刚刚写的那句话. 为了计算下一个 token 的注意力分数, 模型必须把**从对话开始以来的每一个 token 的 Key 和 Value 向量**都保留在显存中. 这一大坨被缓存下来的中间激活值, 就是业界常说的 **KV Cache**. 

让我们从最基础的注意力机制出发, 理解这个内存黑洞究竟是怎么形成的. 在标准的 Multi-Head Self-Attention 中, 对于输入序列中的每一个位置 $i$, 我们都会计算三个向量：Query $Q_i$、Key $K_i$ 和 Value $V_i$. 在自回归生成(autoregressive generation)的解码阶段, 当模型已经生成了前 $t-1$ 个 token, 正准备预测第 $t$ 个 token 时, 它的注意力计算需要涉及所有已生成位置 $1, 2, \dots, t-1$ 的 Key 和 Value. 

具体来说, 第 $t$ 步的注意力输出可以写成：

$$
\text{Attention}(Q_t, K_{1:t}, V_{1:t}) = \text{softmax}\left(\frac{Q_t K_{1:t}^T}{\sqrt{d_{head}}}\right) V_{1:t} \tag{1}
$$

这里的 $K_{1:t}$ 和 $V_{1:t}$ 分别表示从第 1 个位置到第 $t$ 个位置的所有 Key 和 Value 向量拼接而成的矩阵. 如果我们把每一个新 token 的 K、V 都算出来然后扔掉, 那么在下一次生成 token $t+1$ 时, 我们就不得不重新计算 token $1$ 到 token $t$ 的全部 K 和 V——这在计算上是极其低效的. 因此, 工程上的标准做法是把每一层、每一个注意力头的 K 和 V 都缓存下来. 

### 1.2 KV Cache 内存占用公式完整推导

现在我们来严谨地推导 KV Cache 的显存占用公式. 设当前推理的 batch size 为 $B$, 序列长度为 $L$(包含 prompt 和已生成的所有 token), 模型有 $N_{layer}$ 个 Transformer 层, 每层有 $N_{head}$ 个注意力头, 每个头的维度为 $D_{head}$, 数据类型占用 $P$ 个字节(例如 FP16 对应 $P=2$, FP32 对应 $P=4$). 

对于 batch 中的每一个序列, 在第 $l$ 层的第 $h$ 个头中, 我们需要缓存所有 $L$ 个位置的 Key 向量. 单个 Key 向量是一个 $D_{head}$ 维的向量, 因此这一层这一个头的 Key Cache 大小为 $L \times D_{head} \times P$ 字节. 同理, Value Cache 也是 $L \times D_{head} \times P$ 字节. 

将 Key 和 Value 合并, 一层中一个头的 KV Cache 为 $2 \times L \times D_{head} \times P$. 

一层中有 $N_{head}$ 个头, 所以一层总共需要 $2 \times L \times N_{head} \times D_{head} \times P$. 

模型有 $N_{layer}$ 层, 所以一个序列的 KV Cache 为 $2 \times L \times N_{layer} \times N_{head} \times D_{head} \times P$. 

 batch 中有 $B$ 个序列同时推理, 因此总的 KV Cache 显存占用为：

$$
M_{KV} = 2 \times B \times L \times N_{layer} \times N_{head} \times D_{head} \times P \tag{2}
$$
这个公式里没有魔法, 每一项都是硬邦邦的乘法. 但问题恰恰在于——这些乘数在真实的大模型中都大得惊人. 

### 1.3 以 Llama-3-70B 为例的恐怖计算

让我们把 Llama-3-70B 的真实参数代入上述表达式, 看看显存需求究竟有多夸张. Llama-3-70B 的配置如下：
- $N_{layer} = 80$ 层
- $N_{head} = 8$ 个注意力头(每层)
- $D_{head} = 128$ 维
- 使用 GQA(Grouped Query Attention), 但为了简化计算我们先按标准 MHA 来算
- 数据精度 $P = 2$(FP16)

假设一个中等规模的在线服务场景：
- Batch size $B = 32$(同时处理 32 个用户的请求)
- 序列长度 $L = 8192$(约 6000 个英文单词, 或 2000 多个中文字符的长文档对话)

基于上述分析, 建立如下数学关系：

$$
M_{KV} = 2 \times 32 \times 8192 \times 80 \times 8 \times 128 \times 2 \tag{3}
$$

我们逐步拆解：
- $2 \times 32 = 64$
- $64 \times 8192 = 524{,}288$
- $524{,}288 \times 80 = 41{,}943{,}040$
- $41{,}943{,}040 \times 8 = 335{,}544{,}320$
- $335{,}544{,}320 \times 128 = 42{,}949{,}672{,}960$
- $42{,}949{,}672{,}960 \times 2 = 85{,}899{,}345{,}920$

结果是 **85,899,345,920 字节**, 约等于 **80 GiB**. 

等等, 这还只是**单个序列长度 8192、batch size 32**的情况. 如果用户的对话继续拉长到 $L = 32768$(常见的中长上下文场景), KV Cache 会直接膨胀到 **320 GiB**. 如果 batch size 扩大到 128(高并发在线服务), 那就是 **1,280 GiB**, 即 **1.25 TB**. 

而且请注意, Llama-3-70B 使用的是 GQA, 实际 KV Cache 会比 MHA 小一些(因为 Key/Value 头的数量少于 Query 头). 如果我们看使用标准 Multi-Head Attention 的模型, 比如早期的 LLaMA-2-70B 或者某些定制模型, KV Cache 的 $N_{head}$ 项会更大, 数值会更加恐怖. 

更关键的是, KV Cache 的大小与序列长度 $L$ 和 batch size $B$ **都是线性关系**. 模型权重 $W$ 是固定的, 加载进显存后就不再增长; 但 KV Cache 随着每个新 token 的生成而线性增长, 随着同时服务的用户增加而线性增长. 在真实的长上下文对话服务中, KV Cache 的大小**非常轻易地就会超过模型权重本身**. 

### 1.4 为什么 KV Cache 是推理的绝对瓶颈

我们把 KV Cache 和模型权重做一个直接的对比. Llama-3-70B 有大约 700 亿参数, 以 FP16 存储, 模型权重占用：

$$
M_{W} = 70 \times 10^9 \times 2 \text{ bytes} = 140 \text{ GB} \tag{4}
$$
在上述 $B=32, L=8192$ 的场景下, KV Cache 已经占用了约 80 GB. 如果序列拉长到 16K, KV Cache 就是 160 GB——**超过了模型权重本身**. 在高并发场景下($B=64$), KV Cache 轻轻松松就能达到几百 GB. 

一块 NVIDIA A100/H100 的显存只有 80 GB. 这意味着：
1. 如果不做任何优化, 单卡连 $B=32, L=8192$ 的 KV Cache 都放不下. 
2. 即使使用 8 卡张量并行, 模型权重已经占了大量显存, 留给 KV Cache 的空间非常有限. 
3. 显存不够的直接后果是：**batch size 上不去**, 系统吞吐量极低; **最大序列长度受限**, 无法处理长文档; **无法支持高并发**, 用户体验极差. 

这就是 KV Cache 压缩与优化技术之所以成为 2024-2026 年间 LLM 推理领域最热门研究方向之一的根本原因. 它不是在优化一个锦上添花的功能, 而是在解决一个**决定 LLM 服务能否商业化落地**的生死问题.

## 2. 为什么重要 (Significance)

### 2.1 高并发 LLM 服务的核心瓶颈

在现代 LLM 服务架构中, 系统吞吐量的瓶颈往往不是计算(FLOPs), 而是**显存容量和显存带宽**. 这是一个非常反直觉的事实. 人们通常认为大模型的推理速度慢是因为矩阵乘法太多、计算量太大. 但在实际部署中, 尤其是对于已经经过充分优化的生产系统, GPU 的计算单元(Tensor Core)经常处于空闲状态——它们在等数据. 等什么呢？等 KV Cache 从 HBM(高带宽显存)加载到 SM(流式多处理器)的共享内存中. 

当一个 batch 中的序列长度参差不齐时, 系统需要为最长的序列分配 KV Cache 空间, 短序列的位置会被 padding 浪费. 这意味着显存不仅被 KV Cache 本身的大量数据吃掉, 还被**内存碎片和对齐开销**进一步放大. KV Cache 的大小直接决定了：
- **最大 batch size**：更大的 batch 意味着更高的 GPU 利用率, 但每个样本都需要自己的 KV Cache 副本. 

- **最大上下文长度**：长上下文能力(Long Context)是当前大模型竞争的核心卖点之一, 但每翻一倍上下文长度, KV Cache 就翻一倍. 

- **并发用户数**：在线服务的 QPS(每秒查询数)直接受限于同时能塞进显存的请求数量. 

### 2.2 2024-2026 年的研究热点

从 2023 年末开始, KV Cache 优化经历了爆发式的演进：

- **2023 年底**：StreamingLLM 提出 sink tokens 的概念, 首次系统性地挑战了"必须保留所有历史 KV"的假设. 

- **2024 年初**：H2O(Heavy Hitter Oracle)通过观测 attention pattern 发现, 绝大多数 attention weight 集中在极少数 token 上, 为动态剪枝提供了理论依据. 

- **2024 年中**：KVQuant 和一系列 INT8/FP8 量化工作开始系统性地将 KV Cache 压缩到低精度表示, 将显存占用直接砍掉一半甚至更多. 

- **2024 年底至 2025 年**：SnapKV、PyramidKV、LoRA-KV 等工作从不同角度切入——有的基于观测到的 attention pattern 做更精细的动态选择, 有的用低秩近似压缩单 token 的 KV 表示. 

- **2025-2026 年**：混合策略成为主流. 不再单一地使用量化或剪枝, 而是将"低精度量化 + 动态剪枝 + 跨层共享"组合起来, 追求在几乎不损失精度的前提下, 将 KV Cache 压缩到原始大小的 10%-20%. 

### 2.3 直接影响推理成本和用户体验

从商业角度看, KV Cache 压缩技术的价值可以用真金白银来衡量. 假设一个在线 LLM 服务的运营成本中, GPU 租用费用占 70% 以上. 如果通过 KV Cache 优化将显存占用降低 50%, 意味着：
- 同样的硬件可以支持 **2 倍**的并发用户. 
- 或者支持 **2 倍**的上下文长度. 
- 或者将所需 GPU 数量减少一半, 运营成本直接下降 30%-50%. 

对于 C 端产品来说, 这意味着更低的延迟、更长的记忆能力、更便宜的订阅价格. 对于 B 端 API 服务来说, 这意味着更具竞争力的定价和更高的利润率. 这就是为什么从 OpenAI 到 Google, 从 Meta 到各大开源社区, 都在不遗余力地投入 KV Cache 优化的研究.

## 3. 直觉类比 (Intuition)

### 3.1 草稿纸的困境

想象你正在参加一场极其漫长的数学考试. 这场考试有 1000 道小题, 它们不是独立的——每做一道新题, 你都需要回顾前面所有题目的计算过程, 因为新题的答案依赖于前面所有步骤的中间结果. 

你的桌子上有一大摞草稿纸. 第 1 题的计算写在第 1 页, 第 2 题的计算写在第 2 页, 以此类推. 当你做到第 500 题时, 你的桌子上已经摊开了 500 页草稿纸. 每做一道新题(生成一个新 token), 你都需要快速翻阅前面的 500 页草稿(检索历史 KV), 找到与新题相关的步骤, 然后才能写出答案. 

如果桌子(显存)无限大, 这没问题. 但现实是, 你的桌子只有 1 平方米. 草稿纸堆满了, 新题目就做不了. 而且, 随着考试进行, 草稿纸会越来越多——第 1000 题时你需要管理 1000 页草稿. 

这就是 KV Cache 的本质：**每一页草稿纸就是一个 token 的 Key 和 Value 向量. 桌子就是 GPU 显存. 考试就是自回归生成过程. **

### 3.2 压缩 = 缩印

现在, 你开始想办法节省空间. 你发现草稿纸上的字写得很大, 有很多空白. 于是你买来一台缩印机, 把每一页草稿纸的内容缩小到原来的 1/2 或 1/4 大小, 再贴到一张更小的纸上. 

在技术世界里, 这就是**量化(Quantization)** . 原本用 FP16(每个数字占 2 字节)记录的 KV 向量, 被压缩到 INT8(1 字节)甚至 INT4(0.5 字节). 所有信息还在, 只是表示得更紧凑了. 代价是：缩印后的字迹可能有些模糊, 某些细节可能会丢失——对应到模型中, 就是低精度带来的数值误差. 

### 3.3 更细的笔 = FP8

缩印(INT8)有个问题：它是对称的, 所有数字都被均匀地压缩. 但你发现草稿纸上有些数字特别大(比如 9999), 有些特别小(比如 0.001). 如果统一缩印, 小数可能会因为精度不够而变成 0, 大数可能会溢出. 

于是你换了一种更聪明的纸——这种纸使用科学计数法来记录数字. 对于大数, 它能记录更多的位数; 对于小数, 它能记录更精确的小数部分. 这就是 **FP8(E4M3/E5M2)** 格式. 它不是简单地线性缩放, 而是用指数部分来动态调整表示范围, 更适合处理 KV Cache 中天然存在的数值分布不均匀性. 

### 3.4 扔掉不重要的草稿页 = Eviction

随着考试继续, 你发现一个惊人的规律：当你做第 500 题时, 你真正需要回头查看的草稿页其实只有寥寥几张. 绝大多数前面的草稿页(比如第 3 题、第 17 题、第 89 题), 你在后面的几百道题里从来没有再看过. 

于是你做了一个大胆的决定：**把从来没有被回顾过的草稿页直接扔掉**. 你只在桌子上保留两类草稿：
1. 最近做的几十道题(局部窗口)
2. 极少数被反复查阅的"关键步骤"(heavy hitters)

这就是 **H2O** 和 **StreamingLLM** 的核心思想. 不是所有历史 KV 都同等重要——绝大多数 token 的 KV 对后续生成几乎没有贡献, 可以安全地丢弃. 

### 3.5 共享草稿 = 跨层共享

你还发现了另一个规律：物理科目的第 1 章和第 2 章的草稿格式几乎一模一样, 都是同样的公式推导流程. 于是你想："能不能让第 1 章和第 2 章共用同一套草稿模板？"

在 Transformer 中, 相邻层的注意力模式往往高度相关. 如果我们让第 $l$ 层和第 $l+1$ 层共享同一组 Key 和 Value 的投影矩阵, 甚至直接共享 KV Cache 本身, 就能节省接近 50% 的显存. 这就是**跨层 KV 共享(Cross-Layer KV Sharing)** 的直觉来源.

## 4. 数学推导与核心方法 (Mathematical Rigor)

KV Cache 压缩技术可以从四个互补的维度来分类：
1. **精度压缩**：用更少的 bit 表示每个数值(量化)
2. **维度压缩**：用更低维的向量表示每个 token 的 KV(低秩)
3. **序列压缩**：减少需要保留的 token 数量(动态剪枝)
4. **结构压缩**：通过共享减少冗余(跨层共享)

下面我们逐一进行深入的数学推导. 

### 4.1 量化压缩 (Quantization)

量化是工程上最成熟、最直接的 KV Cache 压缩手段. 其核心思想是：将原本以高精度(FP16/FP32)存储的浮点数, 映射到低精度的整数(INT8/INT4)或低精度浮点数(FP8)表示. 

#### 4.1.1 KV Cache INT8 对称量化

对称量化是最简单也最常用的量化方案. 对于一组 KV Cache 值(可以是一个 channel、一个 token 或一个张量), 我们首先找到其绝对值的最大值：

$$
\alpha = \max(|x|) \tag{5}
$$

然后定义缩放因子(scale)：

$$
s = \frac{\alpha}{127} \tag{6}
$$
这里的 127 是 INT8 能表示的最大正整数($2^7 - 1 = 127$). 量化过程将每个浮点数映射到 INT8：

$$
x_q = \text{round}\left(\text{clip}\left(\frac{x}{s}, -128, 127\right)\right) \tag{7}
$$

反量化(dequantization)过程则是：

$$
\hat{x} = x_q \cdot s \tag{8}
$$
让我们严谨地推导量化误差的期望. 假设原始值 $x$ 服从某个分布, 量化误差为 $e = x - \hat{x}$. 由于 round 操作引入的误差在每个量化步长内近似均匀分布, 对于步长为 $s$ 的均匀量化器, 单个值的量化误差方差为：

$$
\mathbb{E}[(x - \hat{x})^2] \approx \frac{s^2}{12} = \frac{\alpha^2}{127^2 \times 12} \tag{9}
$$

这说明量化误差与缩放因子 $s$ 的平方成正比. 当 KV Cache 中的数值分布范围很大($\alpha$ 很大)时, 如果用一个全局的 scale, 那些数值较小的元素就会被"压扁"到很粗糙的量化格点上, 产生巨大误差. 

#### 4.1.2 Per-Channel vs Per-Token 量化的权衡

这就引出了粒度(granularity)的问题. 我们有几种不同的量化粒度选择：

**Per-Tensor 量化**：整个 KV Cache 张量共享一个 scale. 实现最简单, 但不同 head、不同 layer 的数值范围可能差异巨大, 一个全局 scale 必然导致某些区域精度严重下降. 

**Per-Channel 量化**：对每个 head 维度(channel)单独计算 scale. 假设 head_dim=128, 那么每个 token 的 Key 向量有 128 个 channel, 每个 channel 维护自己的 $s_i$. 这能更好地适应不同特征维度的数值分布差异, 因为某些 channel 可能总是产生大数值(如位置编码相关维度), 而另一些 channel 数值范围很小. 

**Per-Token 量化**：对每个 token 单独计算 scale. 这能更好地适应不同 token 位置的数值差异(例如句首 token 和句尾 token 的激活强度往往不同). 

**混合粒度**：实践中, KV Cache 量化通常采用**per-channel + per-token**的组合策略. 对于 Key Cache, 由于不同 token 的 Key 向量在注意力计算中会与 Query 做点积, 数值范围的适配非常关键, 通常使用 per-channel 量化. 对于 Value Cache, 由于它在注意力加权后进入 FFN, 对数值精度的敏感度略有不同. 

#### 4.1.3 FP8 量化：E4M3 与 E5M2

FP8(8-bit Floating Point)是 NVIDIA Hopper/Ada 架构及后续 GPU 原生支持的数据格式, 近年来在 KV Cache 量化中受到越来越多的关注. FP8 分为两种格式：

**E4M3**(4 位指数 + 3 位尾数)：
- 1 位符号位(sign)
- 4 位指数位(exponent), 偏置(bias)为 7
- 3 位尾数位(mantissa)

其可表示的数值范围约为 $[-448, 448]$, 精度约为 $0.0625$($2^{-4}$). 

**E5M2**(5 位指数 + 2 位尾数)：
- 1 位符号位
- 5 位指数位, 偏置为 15
- 2 位尾数位

其可表示的数值范围约为 $[-57344, 57344]$, 但精度更粗, 约为 $0.25$($2^{-2}$). 

为什么 FP8 比 INT8 更适合 KV Cache？核心原因在于**异常值(Outliers)** . 

在 LLM 的中间激活(包括 KV Cache)中, 存在一个被大量研究证实的事实：绝大多数数值集中在一个很小的范围内(比如 $[-1, 1]$), 但有极少数"异常值"的绝对值非常大(可能达到几十甚至上百). 这些异常值虽然数量极少, 但在注意力计算中起着关键作用——它们往往对应着语义上特别重要的 token(如标点符号、特殊名词). 

INT8 对称量化使用固定的线性映射. 为了覆盖那些大异常值, scale $s$ 必须设得很大, 这导致绝大多数小数值被压缩到非常有限的几个量化格点上, 精度严重损失. 

FP8 的指数部分提供了**动态范围(dynamic range)** . 它可以同时很好地表示 $0.001$ 和 $100$ 这样的数值, 因为指数位会自动调整. 这类似于科学计数法——不需要为了覆盖大数而牺牲小数的精度. 

对于 KV Cache 的量化策略, 当前的主流共识是：
- **Key Cache** 对精度更敏感, 因为 Key 与 Query 的点积直接决定了 attention pattern. 如果 Key 的精度损失导致 attention score 的排序发生变化, 模型的语义理解会严重受损. 因此 Key 通常使用 INT8(per-channel)或 E4M3. 

- **Value Cache** 对精度敏感度稍低, 因为 Value 是被 attention weight 加权求和后使用的, 单个 Value 的误差会被平均化. 因此 Value 可以使用更激进的压缩, 如 E5M2 甚至 INT4. 

### 4.2 低秩压缩 (Low-Rank Compression)

量化是从"每个数字用更少的 bit"角度来压缩. 低秩压缩则是从另一个角度出发：**也许每个 token 的 KV 向量本身包含大量冗余信息, 我们可以用更低维的向量来近似表示它. **

#### 4.2.1 SVD 压缩的数学基础

假设我们在推理过程中已经收集了某一层某一头的 Key Cache $K \in \mathbb{R}^{L \times d_{head}}$. 这里 $L$ 是序列长度, $d_{head}$ 是 head 维度(如 128). 我们可以对 $K$ 的协方差矩阵进行特征值分解, 来理解其"有效信息维度". 

计算 Key 的协方差矩阵：

$$
\Sigma_K = \frac{1}{L} K^T K \in \mathbb{R}^{d_{head} \times d_{head}} \tag{10}
$$
对 $\Sigma_K$ 进行特征值分解：

$$
\Sigma_K = U \Lambda U^T \tag{11}
$$

其中 $\Lambda = \text{diag}(\lambda_1, \lambda_2, \dots, \lambda_{d_{head}})$ 是按降序排列的特征值, $U$ 是正交特征向量矩阵. 

如果前 $r$ 个特征值占据了总方差的绝大部分, 即：

$$
\frac{\sum_{i=1}^{r} \lambda_i}{\sum_{i=1}^{d_{head}} \lambda_i} > \tau \tag{12}
$$
(例如 $\tau = 0.99$), 这意味着 $K$ 的实际有效信息维度只有 $r$, 远小于 $d_{head}$. 

#### 4.2.2 投影到低维空间

基于上述观察, 我们可以学习一个投影矩阵 $W_{proj} \in \mathbb{R}^{d_{head} \times r}$, 其中 $r \ll d_{head}$, 将高维 KV 向量压缩到低维表示：

$$
K_{compressed} = K \cdot W_{proj} \in \mathbb{R}^{L \times r} \tag{13}
$$

在注意力计算时, 我们需要将 Query 也投影到同一低维空间：

$$
Q_{compressed} = Q \cdot W_{proj} \tag{14}
$$
然后计算注意力：

$$
\text{Attention} = \text{softmax}\left(\frac{Q_{compressed} K_{compressed}^T}{\sqrt{r}}\right) V_{compressed} \tag{15}
$$

注意这里的缩放因子从 $\sqrt{d_{head}}$ 变成了 $\sqrt{r}$. 

信息保留率的数学表达为：

$$
\eta = \frac{\text{tr}(W_{proj} W_{proj}^T \Sigma_K)}{\text{tr}(\Sigma_K)} = \frac{\sum_{i=1}^{r} \lambda_i}{\sum_{i=1}^{d_{head}} \lambda_i} \tag{16}
$$
如果我们选择 $W_{proj}$ 为前 $r$ 个特征向量组成的矩阵, 那么 $\eta$ 就是前 $r$ 个特征值之和占总特征值之和的比例. 当 $r$ 取 $d_{head}/2$(例如从 128 降到 64)时, 如果特征值分布足够倾斜, 我们可能保留 95% 以上的信息, 而显存占用直接减半. 

#### 4.2.3 KV Shifting / KV Rotation

低秩压缩的一个工程挑战是：投影矩阵 $W_{proj}$ 需要在推理前就确定好. 一种更轻量的方法是**旋转(Rotation)** ——通过一个正交变换 $R \in \mathbb{R}^{d_{head} \times d_{head}}$, 将 KV 向量旋转到一个新的坐标系下, 使得能量集中在前几个维度上, 然后直接截断后段维度. 

数学上, 这等价于对协方差矩阵进行白化或 PCA 旋转：

$$
K' = K \cdot R \tag{17}
$$

然后只保留 $K'$ 的前 $r$ 列. 当 $R$ 取 PCA 的主成分方向时, 这就是最优的线性降维(在均方误差意义下). 

实践中, 有些工作发现 Transformer 的 Key/Value 在训练后已经具有某种结构性——例如某些维度对 attention pattern 的贡献远大于其他维度. 通过简单的排序或学习到的置换, 就能在不引入额外投影计算的情况下实现有效的压缩. 

### 4.3 动态管理与剪枝 (Dynamic Management & Pruning)

如果说量化是"把字写小", 低秩是"用更简洁的符号系统", 那么动态剪枝就是"干脆不记那些无关紧要的步骤". 这一类方法的核心洞察来自对注意力模式的统计观察. 

#### 4.3.1 H2O：Heavy Hitter Oracle

H2O 的核心发现是：在自回归生成过程中, **绝大多数 attention weight 集中在极少数的"热点 token"上**. 这个现象在长序列中尤为明显. 

让我们形式化地描述这个问题. 在生成第 $t$ 个 token 时, 注意力分布是：

$$
\alpha_t = \text{softmax}\left(\frac{Q_t K_{1:t}^T}{\sqrt{d_{head}}}\right) \in \mathbb{R}^{t} \tag{18}
$$
其中 $\alpha_t(i)$ 表示第 $t$ 个 token 对第 $i$ 个历史 token 的注意力权重. 

H2O 定义了一个**累积注意力分数(accumulated attention score)** 来衡量每个历史 token 的重要性：

$$
\text{score}(i) = \sum_{t=i+1}^{L} \alpha_t(i) \tag{19}
$$

直观地说, 如果一个 token $i$ 在后续很多步骤中都被高度关注, 那么它的 KV 就是"heavy hitter", 必须保留. 反之, 如果某个 token 的累积 attention score 极低, 说明它对后续生成几乎没有影响, 可以安全丢弃. 

H2O 的策略是：设定一个预算(budget)$k$, 只保留累积 attention score 最高的 $k$ 个 token 的 KV Cache. 当生成新 token 时, 如果当前 KV Cache 大小超过 $k$, 就移除 score 最低的那些 token. 

为什么可以安全丢弃低 attention token？我们从信息论的角度来理解. 注意力机制的输出是历史 Value 的加权和：

$$
o_t = \sum_{i=1}^{t} \alpha_t(i) V_i \tag{20}
$$
如果对于所有的 $t' > t$, 都有 $\alpha_{t'}(i) \approx 0$, 那么 token $i$ 的 Value $V_i$ 对任何后续输出都没有贡献. 丢弃 $V_i$ 不会改变任何后续 token 的生成结果(在理想情况下). 当然, 由于 attention weight 是动态计算的, 我们今天丢弃的 token 明天可能不会再次被关注——但 H2O 的观察是, attention pattern 具有显著的**时间稳定性**：在过去被频繁关注的 token, 在未来也很可能被继续关注. 

#### 4.3.2 StreamingLLM：Sink Tokens 与滑动窗口

StreamingLLM 提出了一个更激进的观察：在自回归生成中, 只有两类 token 是真正重要的——

**Sink Tokens(汇聚 token)** ：序列最前面的少数几个 token(通常是 1-4 个). 这些 token 奇怪的是, 它们在长序列中获得了异常高的 attention weight. 研究者认为这是由于 Softmax 的归一化特性导致的：当序列很长时, 为了避免 attention score 过于分散, 模型学会了将"全局注意力"集中到最前面的几个 token 上, 把它们当作信息汇聚点. 

**Window Tokens(窗口 token)** ：最近生成的 $w$ 个 token. 这些 token 提供了局部上下文信息, 对语义连贯性至关重要. 

**中间 token**：既不在开头也不在窗口内的 token, 它们的 attention weight 通常极低, 可以全部丢弃. 

StreamingLLM 的策略极其简单：始终保留前 $s$ 个 sink token 和最近 $w$ 个 window token 的 KV Cache, 其余全部丢弃. 这样 KV Cache 的大小从 $O(L)$ 降到了 $O(s + w)$, 与序列长度无关！

这个策略的数学保证来自于对注意力模式的实证分析. 设注意力矩阵为 $A \in \mathbb{R}^{L \times L}$(下三角), StreamingLLM 发现：

$$
\sum_{i \in \text{sink} \cup \text{window}} A_{t,i} \approx 1 - \epsilon \tag{21}
$$

对于绝大多数 $t$, 即绝大部分 attention weight 集中在 sink 和 window token 上. 中间 token 的 attention weight 之和通常小于 $\epsilon = 0.01$, 对最终输出的影响可以忽略不计. 

#### 4.3.3 SnapKV：基于观测模式的动态选择

SnapKV 进一步细化了动态选择的策略. 它不是在生成过程中实时计算 attention score(这本身需要访问所有 KV), 而是在一个**观察阶段(observation phase)** 中, 用少量层或少量 head 的 attention pattern 来预测哪些 token 是重要的. 

具体而言, SnapKV 在每一层选择一个"观察头(observation head)", 在这个头上计算所有 token 的 attention 分布. 然后它根据观察头的 attention pattern, 为当前层决定保留哪些 token 的 KV. 这种方法的好处是：
1. 不需要为所有 head 维护完整的 attention score. 
2. 可以预先在 prompt 阶段(prefill phase)就确定哪些 KV 值得保留, 避免生成阶段(decode phase)的额外开销. 
3. 通过跨层的注意力观测, 可以捕捉到不同层对不同 token 的关注差异. 

SnapKV 的数学形式与 H2O 类似, 但在实现上做了重要工程优化：

$$
S_i^{(l)} = \sum_{h \in \mathcal{H}_{obs}} \sum_{t} \alpha_{t,h}^{(l)}(i) \tag{22}
$$
其中 $\mathcal{H}_{obs}$ 是选定的观察头集合. 然后根据 $S_i^{(l)}$ 的排序, 每层独立地保留 top-$k$ 个 token 的 KV. 

### 4.4 跨层共享 (Cross-Layer Sharing)

前三类方法都是在"每个 token 的 KV 表示"上做文章. 跨层共享则从另一个角度切入：**相邻层的 KV Cache 是否真的需要完全独立？**

#### 4.4.1 相邻层 KV 共享

在标准 Transformer 中, 每一层都有自己的 Key 和 Value 投影矩阵 $W_K^{(l)}$ 和 $W_V^{(l)}$, 因此每一层生成的 KV Cache 都是独立的：

$$
K^{(l)} = X^{(l)} W_K^{(l)}, \quad V^{(l)} = X^{(l)} W_V^{(l)} \tag{23}
$$

跨层共享的核心思想是：让相邻层共享同一组投影矩阵, 甚至直接共享 KV Cache：

$$
K^{(l+1)} = X^{(l+1)} W_K^{(l)}, \quad V^{(l+1)} = X^{(l+1)} W_V^{(l)} \tag{24}
$$
或者更进一步, 直接使用 $K^{(l)}$ 作为 $K^{(l+1)}$ 的近似(在 $X^{(l)}$ 和 $X^{(l+1)}$ 变化不大的情况下). 

如果我们让每两层共享一组 KV Cache, 那么总 KV Cache 大小就直接减半(50% 压缩率). 这种压缩是"免费的"——不需要额外的量化计算, 不需要复杂的动态选择逻辑. 

#### 4.4.2 CSA：Cross-Layer Shared Attention

CSA 将跨层共享的思想系统化. 它发现, 在深层 Transformer 中, 相邻层的 attention pattern(即 attention score 的分布)具有高度相关性. 这意味着不同层其实"在看同样的东西", 只是看的角度略有不同. 

CSA 的策略是：将整个模型分为若干组, 每组内的层共享同一组 KV Cache. 在组内, Query 仍然独立计算, 但 Key 和 Value 来自共享的缓存. 这相当于在层维度上做了"池化", 既保留了不同层提取不同特征的能力(通过独立的 Query 和 FFN), 又避免了 KV Cache 的冗余存储. 

#### 4.4.3 精度损失上界推导

跨层共享引入的误差可以形式化分析. 假设第 $l$ 层的真实 KV 是 $K^{(l)} = X^{(l)} W_K^{(l)}$, 而共享后的 KV 是 $\tilde{K}^{(l)} = X^{(l)} W_K^{(l-1)}$(使用上一层的投影矩阵). 

注意力输出的误差为：

$$
\|o^{(l)} - \tilde{o}^{(l)}\| = \left\|\sum_i \alpha_i^{(l)} V_i^{(l)} - \sum_i \tilde{\alpha}_i^{(l)} \tilde{V}_i^{(l)}\right\| \tag{25}
$$

这个误差可以分解为两部分：
1. **Value 误差**：$V_i^{(l)} - \tilde{V}_i^{(l)}$, 来自投影矩阵的差异
2. **Attention Score 误差**：$\alpha_i^{(l)} - \tilde{\alpha}_i^{(l)}$, 来自 Key 的差异

通过三角不等式和 Lipschitz 连续性, 可以证明(在合理的假设下)：

$$
\|o^{(l)} - \tilde{o}^{(l)}\| \leq C_1 \|W_K^{(l)} - W_K^{(l-1)}\|_F + C_2 \|W_V^{(l)} - W_V^{(l-1)}\|_F \tag{26}
$$
其中 $C_1, C_2$ 是与输入范数和 attention weight 分布有关的常数. 这意味着：**相邻层投影矩阵的差异越小, 跨层共享的精度损失就越小. ** 实证研究表明, 在训练良好的 Transformer 中, 相邻层的 $W_K$ 和 $W_V$ 确实具有高度相似性(余弦相似度常高于 0.9), 这为跨层共享提供了理论基础.

## 5. 数值走查 (Numerical Example)

让我们构造一个具体的场景, 通过真实的数字来感受不同压缩方案的效果. 

### 5.1 基线场景设定

假设我们在部署一个中等规模的开源模型用于在线对话服务：
- $N_{layer} = 32$
- $N_{head} = 8$(每层 8 个注意力头)
- $D_{head} = 128$
- 序列长度 $L = 4096$
- Batch size $B = 16$
- 数据类型：FP16($P = 2$ 字节)

### 5.2 原始 FP16 KV Cache 计算

首先计算未压缩的 FP16 KV Cache 显存占用, 作为后续压缩方案的对比基准：

$$
M_{FP16} = 2 \times 16 \times 4096 \times 32 \times 8 \times 128 \times 2 \tag{27}
$$

这里 $2$ 表示 Key 和 Value 两份, $16$ 是 batch size, $4096$ 是序列长度, $32$ 是层数, $8$ 是注意力头数, $128$ 是每头维度, 最后的 $2$ 是 FP16 每个元素占 2 字节. 逐步计算：
- $2 \times 16 = 32$
- $32 \times 4096 = 131{,}072$
- $131{,}072 \times 32 = 4{,}194{,}304$
- $4{,}194{,}304 \times 8 = 33{,}554{,}432$
- $33{,}554{,}432 \times 128 = 4{,}294{,}967{,}296$
- $4{,}294{,}967{,}296 \times 2 = 8{,}589{,}934{,}592$ 字节

基于上述分析, 建立如下数学关系：

$$
M_{FP16} = \frac{8{,}589{,}934{,}592}{1024^3} \approx 8.0 \text{ GB} \tag{28}
$$

该结果表示在 batch size 为 128、序列长度为 4096 的典型服务场景下, 原始 FP16 KV Cache 的显存占用约为 8 GB, 构成了后续各类量化与压缩方案的基准. 

### 5.3 各种压缩方案对比

| 压缩方案 | 压缩原理 | 计算公式 | KV Cache 大小 | 相对原始比例 |
|---------|---------|---------|--------------|------------|
| 原始 FP16 | 无压缩 | $M_{FP16}$ | **8.00 GB** | 100% |
| INT8 量化 | 每数值 1 字节 | $M_{FP16} \times \frac{1}{2}$ | **4.00 GB** | 50% |
| FP8 量化 | 每数值 1 字节 | $M_{FP16} \times \frac{1}{2}$ | **4.00 GB** | 50% |
| INT4 量化 | 每数值 0.5 字节 | $M_{FP16} \times \frac{1}{4}$ | **2.00 GB** | 25% |
| 低秩压缩 50% | head_dim: 128→64 | $M_{FP16} \times \frac{1}{2}$ | **4.00 GB** | 50% |
| 低秩 50% + INT8 | 维度减半 + 精度减半 | $M_{FP16} \times \frac{1}{4}$ | **2.00 GB** | 25% |
| H2O 保留 20% | 保留 top-20% token | $M_{FP16} \times 0.2$ | **1.60 GB** | 20% |
| StreamingLLM | sink(4) + window(512) | $M_{FP16} \times \frac{516}{4096}$ | **1.01 GB** | 12.6% |
| 跨层共享×2 | 每 2 层共享 | $M_{FP16} \times \frac{1}{2}$ | **4.00 GB** | 50% |
| 组合：低秩50% + INT8 + H2O(20%) | 三重压缩 | $M_{FP16} \times 0.5 \times 0.5 \times 0.2$ | **0.40 GB** | 5% |

### 5.4 长上下文场景的冲击

上面的计算基于 $L = 4096$. 但在当前(2025-2026)的 LLM 应用中, 128K 上下文窗口已经逐渐成为标配. 让我们看看当 $L = 131072$ 时会发生什么：

$$
M_{FP16}^{(128K)} = 2 \times 16 \times 131072 \times 32 \times 8 \times 128 \times 2 = 256 \text{ GB} \tag{29}
$$

仅仅 16 个 batch、128K 上下文的 KV Cache, 就需要 **256 GB** 显存！这远超单卡 A100/H100 的 80GB 容量, 甚至需要多卡才能存下. 

但如果我们应用组合压缩(低秩 50% + INT8 + H2O 保留 20%)：

$$
M_{compressed}^{(128K)} = 256 \times 0.5 \times 0.5 \times 0.2 = 12.8 \text{ GB} \tag{30}
$$
**从 256 GB 压缩到 12.8 GB**, 这就是 KV Cache 优化技术的威力. 它使得在消费级 GPU 上运行长上下文推理成为可能. 

### 5.5 与模型权重的对比

假设模型参数量为 13B(如 Llama-2-13B), FP16 权重占用：

$$
M_{weights} = 13 \times 10^9 \times 2 = 26 \text{ GB} \tag{31}
$$

在长上下文场景下($B=16, L=128K$)：
- 模型权重：26 GB
- 原始 KV Cache：256 GB(是权重的 **9.8 倍**)
- 组合压缩后 KV Cache：12.8 GB(是权重的 **0.5 倍**)

这个对比极其有力地说明了：在长上下文推理中, KV Cache 才是真正的显存巨无霸. 优化 KV Cache 比优化模型权重本身更具性价比.

## 6. 简化实现 (Implementation)

下面我们给出约 100 行 PyTorch 代码, 展示 KV Cache 压缩的核心实现逻辑. 代码包含：INT8 对称量化、H2O 动态剪枝、低秩投影层, 以及一个混合精度的 KV Cache 管理类. 

```python
import torch
import torch.nn as nn
import torch.nn.functional as F
from typing import Optional, Tuple, List

# ============================================================
# 1. INT8 对称量化函数
# 对应数学公式: x_q = round(clip(x / s, -128, 127))
# ============================================================
def quantize_int8(x: torch.Tensor) -> Tuple[torch.Tensor, torch.Tensor]:
    """
    对输入张量进行 per-channel INT8 对称量化. 
    
    Args:
        x: 输入张量, 形状通常为 (batch, seq_len, num_heads, head_dim)
           或 (batch, num_heads, seq_len, head_dim)
    
    Returns:
        x_q: 量化后的 INT8 张量 (torch.int8)
        scale: 每个 channel 的缩放因子, 形状为 (num_heads, head_dim) 或 (head_dim,)
    """
    # 找到最后一个维度(channel)上的最大绝对值
    # 对应数学公式: alpha = max(|x|)
    alpha = x.abs().amax(dim=-1, keepdim=True)  # (..., 1)
    
    # 计算缩放因子: s = alpha / 127
    scale = alpha / 127.0
    scale = scale.clamp(min=1e-8)  # 避免除以零
    
    # 量化: x_q = round(x / s)
    x_q = torch.round(x / scale)
    
    # 裁剪到 INT8 范围 [-128, 127]
    x_q = torch.clip(x_q, -128, 127).to(torch.int8)
    
    return x_q, scale.squeeze(-1)


def dequantize_int8(x_q: torch.Tensor, scale: torch.Tensor) -> torch.Tensor:
    """
    INT8 反量化. 
    对应数学公式: x_hat = x_q * s
    """
    return x_q.float() * scale.unsqueeze(-1)


# ============================================================
# 2. H2O 动态剪枝逻辑
# 对应数学公式: score(i) = sum_t alpha_t(i), 保留 top-k
# ============================================================
class H2OKVCache:
    """
    H2O (Heavy Hitter Oracle) KV Cache 管理器. 
    维护一个固定大小的 KV Cache, 只保留累积 attention score 最高的 k 个 token. 
    """
    def __init__(self, budget: int, num_heads: int, head_dim: int):
        """
        Args:
            budget: 每个 head 最多保留的 token 数量 (k)
            num_heads: 注意力头数
            head_dim: 每个头的维度
        """
        self.budget = budget
        self.num_heads = num_heads
        self.head_dim = head_dim
        
        # KV Cache: (num_heads, current_len, head_dim)
        self.k_cache: Optional[torch.Tensor] = None
        self.v_cache: Optional[torch.Tensor] = None
        
        # 累积 attention score: (num_heads, current_len)
        self.accum_scores: Optional[torch.Tensor] = None
    
    def update(
        self,
        new_k: torch.Tensor,      # (num_heads, new_len, head_dim)
        new_v: torch.Tensor,      # (num_heads, new_len, head_dim)
        attn_weights: torch.Tensor  # (num_heads, new_len, total_len) - 新 token 对历史 token 的 attention
    ):
        """
        更新 KV Cache, 根据 attention weights 更新累积 score, 并执行剪枝. 
        
        Args:
            new_k: 新 token 的 Key 向量
            new_v: 新 token 的 Value 向量
            attn_weights: 新 token 对包括自己在内的所有 token 的 attention 权重
        """
        num_heads, new_len, head_dim = new_k.shape
        
        # 第一次初始化
        if self.k_cache is None:
            self.k_cache = new_k
            self.v_cache = new_v
            # 新 token 的初始 score 为 0(还没有被后续 token 关注)
            self.accum_scores = torch.zeros(num_heads, new_len, device=new_k.device)
            return
        
        # 将新 KV 拼接到缓存中
        # 对应: K_{1:t+1} = [K_{1:t}, K_{new}]
        self.k_cache = torch.cat([self.k_cache, new_k], dim=1)
        self.v_cache = torch.cat([self.v_cache, new_v], dim=1)
        
        # 更新累积 attention score
        # 对 attn_weights 在 new_len 维度求和, 得到每个历史 token 被新 token 关注的总程度
        new_scores = attn_weights.sum(dim=1)  # (num_heads, total_len)
        self.accum_scores = self.accum_scores + new_scores
        
        # 如果超过预算, 剪枝掉低 score 的 token
        current_len = self.k_cache.shape[1]
        if current_len > self.budget:
            # 对每个 head 独立排序, 保留 top-k
            # topk 返回 (values, indices)
            _, keep_indices = torch.topk(
                self.accum_scores, 
                k=self.budget, 
                dim=-1, 
                largest=True, 
                sorted=False
            )
            
            # gather 保留选中的 token
            # 对应数学: 保留 score(i) 最高的 k 个 token
            self.k_cache = torch.gather(
                self.k_cache, dim=1, 
                index=keep_indices.unsqueeze(-1).expand(-1, -1, head_dim)
            )
            self.v_cache = torch.gather(
                self.v_cache, dim=1,
                index=keep_indices.unsqueeze(-1).expand(-1, -1, head_dim)
            )
            self.accum_scores = torch.gather(
                self.accum_scores, dim=1, index=keep_indices
            )
    
    def get_kv(self) -> Tuple[torch.Tensor, torch.Tensor]:
        """返回当前缓存的 K 和 V. """
        return self.k_cache, self.v_cache


# ============================================================
# 3. 低秩压缩投影层
# 对应数学公式: K_compressed = K * W_proj, W_proj ∈ R^{d_head × r}
# ============================================================
class LowRankKVProjection(nn.Module):
    """
    低秩 KV Cache 投影层. 
    将高维 (d_head) KV 向量压缩到低维 (rank) 空间. 
    """
    def __init__(self, d_head: int, rank: int):
        super().__init__()
        self.d_head = d_head
        self.rank = rank
        
        # 投影矩阵: R^{d_head × rank}
        # 对应 W_proj
        self.k_proj = nn.Linear(d_head, rank, bias=False)
        self.v_proj = nn.Linear(d_head, rank, bias=False)
        
        # Query 也需要投影到同一空间
        self.q_proj = nn.Linear(d_head, rank, bias=False)
    
    def compress_kv(
        self, 
        k: torch.Tensor,  # (..., d_head)
        v: torch.Tensor   # (..., d_head)
    ) -> Tuple[torch.Tensor, torch.Tensor]:
        """压缩 KV 到低维空间. """
        k_low = self.k_proj(k)  # (..., rank)
        v_low = self.v_proj(v)  # (..., rank)
        return k_low, v_low
    
    def project_q(self, q: torch.Tensor) -> torch.Tensor:
        """将 Query 投影到同一低维空间. """
        return self.q_proj(q)  # (..., rank)


# ============================================================
# 4. 混合精度 KV Cache 管理类
# 组合了: INT8 量化 + 低秩压缩 + H2O 剪枝
# ============================================================
class MixedPrecisionKVCache:
    """
    混合精度 KV Cache 管理器. 
    
    策略:
    - Key Cache: 低秩压缩 (d_head → rank) + INT8 量化
    - Value Cache: 低秩压缩 (d_head → rank) + INT8 量化
    - 序列维度: H2O 动态剪枝, 只保留 budget 个 token
    """
    def __init__(
        self,
        num_heads: int,
        d_head: int,
        rank: int,
        budget: int,
        device: str = "cuda"
    ):
        self.num_heads = num_heads
        self.d_head = d_head
        self.rank = rank
        self.budget = budget
        self.device = device
        
        # 低秩投影器
        self.low_rank = LowRankKVProjection(d_head, rank).to(device)
        
        # H2O 管理器
        self.h2o = H2OKVCache(budget, num_heads, rank)
        
        # 量化状态存储
        self.k_cache_q: Optional[torch.Tensor] = None  # INT8
        self.v_cache_q: Optional[torch.Tensor] = None  # INT8
        self.k_scale: Optional[torch.Tensor] = None
        self.v_scale: Optional[torch.Tensor] = None
    
    def update(
        self,
        k: torch.Tensor,      # (num_heads, new_len, d_head)
        v: torch.Tensor,      # (num_heads, new_len, d_head)
        q: torch.Tensor,      # (num_heads, new_len, d_head) - 用于计算 attention
    ):
        """完整的更新流程：低秩压缩 → 量化 → H2O 剪枝. """
        
        # Step 1: 低秩压缩
        # 对应: K_compressed = K * W_proj
        k_low = self.low_rank.k_proj(k)  # (num_heads, new_len, rank)
        v_low = self.low_rank.v_proj(v)  # (num_heads, new_len, rank)
        
        # Step 2: INT8 量化
        # 对应: x_q = round(x / s)
        k_q, k_scale = quantize_int8(k_low)
        v_q, v_scale = quantize_int8(v_low)
        
        # 反量化用于计算 attention(模拟推理时的在线反量化)
        k_deq = dequantize_int8(k_q, k_scale)
        v_deq = dequantize_int8(v_q, v_scale)
        
        # Step 3: 计算 attention weights 用于 H2O score 更新
        q_low = self.low_rank.q_proj(q)  # (num_heads, new_len, rank)
        
        if self.k_cache_q is not None:
            # 反量化历史 KV
            k_hist = dequantize_int8(self.k_cache_q, self.k_scale)
            v_hist = dequantize_int8(self.v_cache_q, self.v_scale)
            
            # 拼接历史和新的 KV
            k_full = torch.cat([k_hist, k_deq], dim=1)
            v_full = torch.cat([v_hist, v_deq], dim=1)
        else:
            k_full = k_deq
            v_full = v_deq
        
        # 计算 attention: softmax(Q * K^T / sqrt(rank))
        scores = torch.matmul(q_low, k_full.transpose(-2, -1)) / (self.rank ** 0.5)
        attn_weights = F.softmax(scores, dim=-1)  # (num_heads, new_len, total_len)
        
        # Step 4: H2O 更新(使用原始低秩但未量化版本作为"真实值"缓存)
        self.h2o.update(k_low, v_low, attn_weights)
        
        # Step 5: 对 H2O 选中的 KV 重新量化并存储
        k_selected, v_selected = self.h2o.get_kv()
        self.k_cache_q, self.k_scale = quantize_int8(k_selected)
        self.v_cache_q, self.v_scale = quantize_int8(v_selected)
    
    def get_kv_for_attention(self) -> Tuple[torch.Tensor, torch.Tensor]:
        """获取反量化后的 KV 用于 attention 计算. """
        k = dequantize_int8(self.k_cache_q, self.k_scale)
        v = dequantize_int8(self.v_cache_q, self.v_scale)
        return k, v


# ============================================================
# 5. 使用示例
# ============================================================
if __name__ == "__main__":
    # 配置参数
    NUM_HEADS = 8
    D_HEAD = 128
    RANK = 64       # 低秩压缩到 50%
    BUDGET = 512    # H2O 最多保留 512 个 token
    BATCH = 1       # 简化：单序列
    
    cache = MixedPrecisionKVCache(NUM_HEADS, D_HEAD, RANK, BUDGET)
    
    # 模拟生成 1000 个 token
```

## 7. 局限性与边界条件 (Limitations & Boundary Conditions)

KV Cache 压缩技术虽然在工程上取得了巨大成功, 但它绝非万能药. 每一种压缩手段都有其适用范围和失效边界, 理解这些局限性对于正确选择和部署优化策略至关重要. 

### 7.1 量化在长上下文中的累积误差

量化误差最令人担忧的地方在于它的**累积效应**. 单个 token 的 KV 被量化后, 误差可能在 $\pm s/2$ 的范围内($s$ 是量化步长). 这个误差看起来很小, 但当序列长度达到 100K 甚至 1M 时, 问题开始显现. 

考虑一个极端情况：假设某个特定位置 $i$ 的 Key 向量被量化引入了微小误差 $\delta K_i$. 在生成后续第 $t$ 个 token 时, attention score 的计算为：

$$
\hat{\alpha}_t(i) = \frac{\exp(Q_t (K_i + \delta K_i)^T / \sqrt{d})}{\sum_j \exp(Q_t K_j^T / \sqrt{d})} \tag{32}
$$
这个误差会改变 softmax 输出的分布, 进而改变 attention 加权后的输出 $o_t$. 而 $o_t$ 又会作为下一层的输入, 影响下一层的 KV 计算——**误差开始传播和放大**. 

在长序列中, 这种累积效应可能导致：
- 位置编码的精细差异被抹平, 模型对 token 间相对位置的感知模糊. 
- 远距离依赖关系的 attention score 发生排序错误, 导致模型"遗忘"早期关键信息. 
- 在需要精确数值推理(如长文档中的数字计算)或精确引用(如法律文书中的条款引用)时, 量化误差可能导致严重的语义错误. 

实验表明, INT8 量化在 4K-8K 上下文中通常几乎无损(perplexity 差异 < 1%), 但在 128K+ 上下文中, perplexity 差异可能上升到 5%-10%, 某些任务(如长文档 QA 中的精确位置引用)的准确率下降更为显着. 

### 7.2 H2O 类方法对全局信息任务的损害

H2O、StreamingLLM 等基于 attention pattern 的动态剪枝方法有一个根本性假设：**未来被高度关注的 token, 在过去也已经被高度关注. ** 这个假设在大多数自然语言生成任务中是成立的, 但在某些特定任务中会失效. 

**任务类型 1：长文档摘要(Long Document Summarization)** 
在生成长文档摘要时, 模型可能在前 90% 的生成过程中只关注文档的局部细节(描述性内容), 但在最后 10% 的生成过程中突然需要"回望"文档开头提出的核心论点或问题陈述. 如果中间过程的 attention score 没有给开头的关键 token 足够高的分数, H2O 可能在前 90% 的生成中就把这些 token 的 KV 丢弃了. 到了最后需要它们的时候, 它们已经不在了. 

**任务类型 2：多轮对话中的指代消解(Coreference Resolution)** 
在多轮对话中, 用户可能在第 1 轮提到"那个红色的方案", 然后在第 20 轮突然问"你记得我说的红色方案吗？"中间 19 轮对话的 attention 可能完全集中在各自轮次的内容上, 导致第 1 轮的关键 token 的累积 score 被稀释. H2O 可能在第 10 轮左右就把第 1 轮的 KV 丢弃了. 

**任务类型 3：代码生成中的跨函数引用**
在生成大型代码文件时, 模型需要在文件末尾的函数中引用文件开头定义的类或变量. 如果中间的代码实现细节占据了绝大部分 attention, 开头的定义可能被过早淘汰. 

StreamingLLM 通过强制保留 sink tokens 来缓解这个问题(开头的 token 永远不会被丢弃), 但 sink tokens 的数量有限(通常只有 1-4 个), 可能无法承载复杂的全局信息. 

### 7.3 跨层共享在浅层和深层的精度损失不均匀

跨层共享的精度损失并不是在模型的所有层均匀分布的. 研究表明：

**浅层(第 1-10 层)** ：浅层主要负责提取局部的、低层次的特征(如词性、句法结构、局部 n-gram 模式). 相邻浅层的 attention pattern 差异相对较大, 因为它们正在从原始输入中"发现"不同类型的局部结构. 在浅层进行跨层共享, 精度损失往往更明显. 

**深层(第 20-32 层)** ：深层主要负责高层次的语义整合和全局推理. 此时模型的表示已经高度抽象化, 相邻层的 attention pattern 趋于相似——它们都在做"基于已提取特征的推理". 深层跨层共享的精度损失通常较小. 

**物理根因**：从数学上看, 浅层的输入 $X^{(l)}$ 和 $X^{(l+1)}$ 差异较大(因为浅层还在快速变换表示空间), 导致 $K^{(l)} = X^{(l)} W_K^{(l)}$ 和 $K^{(l+1)} = X^{(l+1)} W_K^{(l+1)}$ 即使在 $W_K$ 相似的情况下也可能差异显着. 深层的 $X^{(l)}$ 和 $X^{(l+1)}$ 已经进入了相对稳定的高维流形, 差异较小. 

因此, 实践中更合理的策略是：**在浅层使用独立的 KV Cache(或较小的共享组), 在深层使用更激进的跨层共享. **

### 7.4 GQA 与 MQA 对 KV Cache 优化的影响

Grouped Query Attention(GQA)和 Multi-Query Attention(MQA)本身就是为了减少 KV Cache 而设计的架构级优化. 在 GQA 中, 多个 Query 头共享同一组 Key 和 Value 头; 在 MQA 中, 所有 Query 头共享同一个 Key 头和一个 Value 头. 

这意味着：
- **在 MQA 模型上, 低秩压缩和跨层共享的边际收益更小**, 因为 KV Cache 本身已经被架构设计压缩到了接近极限. 

- **在标准 MHA 模型上, KV Cache 优化的空间更大**, 但也更需要这些技术. 

- **量化策略需要适配 GQA/MQA 的特殊结构**. 由于多个 Query 头共享同一个 Key 头, 这个共享 Key 的量化误差会同时影响多个 Query 头的 attention 计算, 误差放大效应更明显. 因此对 GQA/MQA 的 Key Cache 进行量化时, 需要更保守的粒度(例如 per-channel 而不是 per-tensor). 

### 7.5 不同压缩策略的组合陷阱

当多种压缩策略组合使用时, 需要注意它们之间的**交互效应**：

1. **量化 + 低秩**：低秩压缩将 head_dim 从 128 降到 64, 这意味着量化后的数值范围 $\alpha$ 可能会改变. 低秩投影后的数值分布可能与原始分布不同, 需要重新校准量化参数(scale). 

2. **量化 + 剪枝**：H2O 丢弃的 token 通常是低 attention 的 token. 但这些 token 的 KV 值可能恰好处于数值分布的尾部(极端值), 保留下来 token 的数值分布可能变得更"紧凑", 反而有利于量化精度的提升——这是一个正向交互. 

3. **低秩 + 剪枝**：低秩投影矩阵 $W_{proj}$ 通常需要在完整数据上通过 SVD 或训练得到. 如果训练数据中的序列已经被 H2O 剪枝过(即某些 token 被丢弃), 投影矩阵学习到的表示可能会偏向"重要 token 的分布", 这在理论上可能反而更好, 但需要严格的实验验证.

## 8. 演进与承上启下 (Evolution & Segue)

### 8.1 从压缩到管理：PagedAttention 的范式跃迁

本文讨论的 KV Cache 压缩技术, 核心目标是在"保留的内容"上做文章——要么用更少的 bit 表示, 要么用更低的维度表示, 要么干脆少保留一些 token. 但还有一个同样重要的维度：**即使我们保留了所有的 KV, 如何高效地管理它们的物理存储？**

这就是 **PagedAttention**(由 vLLM 提出)的核心思想. PagedAttention 不压缩 KV Cache 的内容, 而是压缩其**存储方式**. 它受到操作系统虚拟内存管理的启发, 将 KV Cache 分割成固定大小的"页(page)"或"块(block)", 并通过一个块表(block table)来管理逻辑到物理的映射. 

PagedAttention 解决了本文尚未触及的两个关键问题：
1. **内存碎片**：不同序列长度导致的空间浪费(internal fragmentation). 

2. **共享**：在 beam search 或并行采样中, 多个候选序列共享相同的前缀 KV, PagedAttention 通过引用计数实现物理共享. 

如果把 KV Cache 压缩比作"把草稿纸上的字写小", 那么 PagedAttention 就是"把草稿纸切成统一大小的活页, 用活页夹灵活管理". 两者是正交的——你可以把字写得很小(压缩), 同时用活页夹高效管理(PagedAttention). 在生产系统中, 这两者几乎总是同时使用. 

### 8.2 用 KV Cache 优化换取生成速度：投机解码

另一个与 KV Cache 密切相关的演进方向是**投机解码(Speculative Decoding)** . 投机解码的核心思想是：用一个小模型(draft model)快速生成候选 token, 然后用大模型并行验证这些候选. 如果候选正确, 就能一次"跳过"多个 token 的生成步骤, 极大提升有效生成速度. 

KV Cache 在这里扮演了什么角色？在投机解码中, 大模型需要同时维护两个 KV Cache：一个是为已确认前缀准备的, 另一个是为候选序列准备的. 这使得 KV Cache 的管理变得更加复杂. 一些最新工作(如 Medusa、Lookahead Decoding)开始探索"复用 KV Cache 来加速 draft token 生成"的策略——与其让 draft model 从头计算, 不如利用大模型已有的 KV Cache 来直接预测下一个几个 token. 

这暗示了一个更深层的趋势：**KV Cache 不再仅仅是一个被动的存储结构, 而正在成为一个可以被主动利用的计算资源. **

### 8.3 未来方向：自适应 KV Cache

展望 2026 年及以后, KV Cache 优化的终极形态可能是**自适应(Adaptive)** 策略：模型根据输入内容的类型、任务的性质、当前可用的显存预算, **动态选择最优的压缩组合**. 

例如：
- 当处理代码生成任务时, 自动选择更保守的剪枝策略(因为代码需要精确的远距离引用). 
- 当处理创意写作任务时, 自动选择更激进的量化策略(因为创意任务对数值精度不敏感). 
- 当显存充足时, 保持高质量的全精度 KV; 当显存紧张时, 自动降级到 INT4 + H2O 剪枝. 

这种自适应策略需要模型具备"元认知"能力——不仅理解输入内容, 还理解自己的推理过程对 KV 精度的敏感度. 一些前沿工作已经开始探索通过轻量级的"KV 策略网络"来动态调控压缩参数, 这或许是 KV Cache 优化的下一个前沿.

## 8.4 工业前沿：DeepSeek-V4 的异构 KV Cache 管理与磁盘存储

DeepSeek-V4 的混合注意力机制(CSA + HCA + SWA)引入了多种类型的 KV 条目, 各自有不同的缓存大小和更新规则, 对传统的 PagedAttention 假设构成了根本挑战. 

### 异构 KV 条目的类型与特征

| KV 类型 | 来源 | 压缩率 | 更新规则 | 相对体积 |
|:--------|:-----|:------|:--------|:--------|
| CSA 压缩 KV | CSA 注意力层 | $1/m$ | 每 $m$ 个 token 生成 1 个条目 | $\sim 5$-$10\%$ |
| HCA 压缩 KV | HCA 注意力层 | $1/m'$ | 每 $m'$ 个 token 生成 1 个条目 | $\sim 2$-$5\%$ |
| CSA Indexer KV | Lightning Indexer | 独立维度 | 与主注意力不同的 embedding 维度 | $\sim 1$-$2\%$ |
| SWA KV | 滑动窗口注意力 | 窗口截断 | 仅保留最近 $n_{\text{win}}$ 个 token | $\sim 1$-$2\%$ |
| 未压缩尾部 | 压缩分支 | $1\times$ | 不足 $m$ 个 token 时暂存 | 微量 |

这种异构性打破了 PagedAttention 的两个基本假设：**各层缓存策略一致**(SWA 的淘汰策略与 CSA/HCA 不同)和**各层 block 大小一致**(高性能 attention kernel 的对齐要求不同). 

### 双区 KV Cache 布局

DeepSeek-V4 将 KV Cache 组织为两个物理区域：

**State Cache**：针对 SWA KV 设计. SWA 的缓存大小有限(仅 $n_{\text{win}}$ 个 token), 可视为状态空间模型的序列状态, 只依赖当前位置. 预分配固定大小的状态缓存池, 动态分配给各序列. 

**Classical KV Cache**：针对 CSA/HCA 压缩 KV 设计. CSA 压缩比为 $m$, HCA 压缩比为 $m'$. 为让两者在同一个 block 内对齐, 每个 block 覆盖 $\text{lcm}(m, m')$ 个原始 token, 产生：
- CSA 压缩条目：$k_1 = \text{lcm}(m, m') / m$ 个
- HCA 压缩条目：$k_2 = \text{lcm}(m, m') / m'$ 个

稀疏 attention kernel 需要协同设计以支持不同层中可变的 block token 数. 例如, 将 block padding 到 cache line 对齐可提升访存效率. 

### 磁盘 KV Cache 存储

为消除共享前缀请求的重复 prefill, DeepSeek-V4 实现了磁盘 KV Cache 存储. 

**CSA/HCA 压缩 KV 的存储**：直接将所有压缩 KV 条目存储到磁盘. 命中前缀时, 读取并复用前缀对应的压缩 KV 条目, 直到最后一个完整压缩 block. 对尾部不足 $m$ 个 token 的不完整 block, 需重计算以恢复未压缩 KV(因为未压缩 KV 不存盘). 

收益示例：系统提示词 18 token, $m=4$. 压缩后存盘 4 个条目(C0-C3), 覆盖 16 个 token. 后续请求只需重计算 2 个 token 的未压缩 KV, 节省 $16/18 \approx 89\%$ 的 prefill 计算. 前缀越长, 节省比例越接近 100%. 

**SWA KV 的三种存储策略**：SWA KV 未经压缩且存在于每一层, 体积约为压缩 KV 的 8 倍. DeepSeek-V4 提供三种策略：

| 策略 | 存储内容 | 命中时重计算量 | 写放大 | 适用场景 |
|:-----|:--------|:-------------|:------|:--------|
| Full Caching | 全部 SWA KV | 0 | $\sim 15\times$ | 读取密集型, SSD 寿命不敏感 |
| Periodic Checkpointing | 每 $p$ 个 token 存最近 $n_{\text{win}}$ 个 | 最多 $p$ 个 token | $\sim 5\times$ | 平衡读写(推荐) |
| Zero Caching | 不存 SWA KV | 最后 $n_{\text{win}} \cdot L$ 个 token | 0 | 超长前缀($>10$K), 计算充裕 |

Zero Caching 的原理：每层 SWA KV 只依赖上一层最近 $n_{\text{win}}$ 个 token 的 SWA KV. 对 $L$ 层模型, 利用 CSA/HCA 缓存, 只需重计算最后 $n_{\text{win}} \cdot L$ 个 token 即可恢复全部 SWA 状态. 在 100K token 前缀场景下, 仅需 prefill 最后 3840 token(假设 $n_{\text{win}}=64, L=60$), vs 完整 prefill 100K token → 节省 96%. 

### MLA 的 KV Cache 压缩效果：工业级实证

DeepSeek-V3 的 MLA(Multi-head Latent Attention)架构为 KV Cache 压缩提供了最激进的工业级方案. 通过将 Key 和 Value 压缩到低维潜在向量, MLA 将每 token 的 KV Cache 降至极致：

| 模型 | 架构 | 每 token KV Cache (BF16) | 相对比例 |
|:-----|:-----|:------------------------|:--------|
| DeepSeek-V3 | MLA | 70 KB | 1× |
| Qwen-2.5 72B | GQA | 327 KB | 4.7× |
| LLaMA-3.1 405B | GQA | 516 KB | 7.4× |

如表所示, DeepSeek-V3 的 MLA 将 KV Cache 压缩到同规模 GQA 模型的 **1/4.7 ~ 1/7.4**, 这一压缩比并非来自更激进的量化或剪枝, 而是来自架构层面的根本性重新设计. MLA 的核心洞察是：注意力计算真正需要的是 Query 与 Key 之间的相似度, 而非 Key 和 Value 本身的高维表示. 通过将 Key/Value 投影到一个远小于原始维度的低维潜在空间(latent space), MLA 在保留注意力表达能力的同时, 将存储需求压缩了一个数量级. 

这一设计的物理意义在于：KV Cache 的膨胀根源不是"需要存储太多信息", 而是"存储了太多冗余信息". GQA 通过减少 Key/Value 头的数量来压缩(从 MHA 的 $H$ 套减到 1 套或 $G$ 套), 但每个头仍然保持原始维度 $d_h$. MLA 则更进一步, 直接在特征维度上进行压缩, 将每 token 的 KV 表示从 $H \times d_h$ 维压缩到 $d_c$ 维($d_c \ll H \times d_h$). 在 100 万 token 长上下文场景下, DeepSeek-V4-Pro 的 KV Cache 仅占 DeepSeek-V3.2 的 10%, 使得百万级上下文推理在消费级硬件上成为可能. 

### 对 KV Cache 优化的启示

DeepSeek-V4 的 KV Cache 管理揭示了三个工业级设计原则：

1. **异构分层**：不同类型的 KV 需要不同的物理存储策略, 统一 block 设计是低效的. 未来的推理系统需要"类型感知"的内存管理器. 

2. **压缩与存储正交**：CSA/HCA 先在算法层面压缩序列维度, 磁盘存储再消除跨请求的重复 prefill. 两者叠加产生乘数效应. 

3. **计算-存储动态权衡**：Zero SWA Caching 在短前缀场景无优势(重计算量 > 前缀长度), 但在超长前缀场景收益巨大. 策略选择应基于前缀长度分布动态决策. 

## 9. 总结与参考文献 (References)

### 核心要点总结

1. **KV Cache 是 LLM 推理中的显存黑洞**：其大小与 batch size 和序列长度线性增长, 在长上下文和高并发场景下极易超过模型权重本身, 成为推理部署的首要瓶颈. 

2. **量化压缩(INT8/FP8)** 是最直接的压缩手段. FP8 因具备动态范围优势, 比 INT8 更适合处理 KV Cache 中天然存在的异常值. Key Cache 对精度更敏感, Value Cache 可以容忍更激进的压缩. 

3. **低秩压缩** 利用 KV Cache 在特征维度上的冗余, 通过投影矩阵将 $d_{head}$ 维向量压缩到更低维度. 信息保留率可以通过特征值分析严格量化. 

4. **动态剪枝(H2O / StreamingLLM / SnapKV)** 基于注意力模式的统计观察, 发现绝大多数 attention weight 集中在极少数 token 上, 因此可以安全丢弃大量"无关"token 的 KV, 将序列维度的复杂度从 $O(L)$ 降到 $O(1)$ 或 $O(k)$. 

5. **跨层共享** 利用相邻层 attention pattern 的高度相关性, 通过共享 KV 投影矩阵或 KV Cache 本身, 将层维度的存储减半. 

6. **组合策略** 可以将原始 KV Cache 压缩到 5%-10% 的大小, 使得在消费级硬件上运行长上下文推理成为可能. 但不同压缩手段之间存在复杂的交互效应, 需要仔细校准. 

7. **每种压缩策略都有其边界条件**：量化在长上下文中有累积误差; H2O 类方法对需要全局信息回溯的任务有害; 跨层共享在浅层和深层的精度损失不均匀; GQA/MQA 架构改变了优化的边际收益.
