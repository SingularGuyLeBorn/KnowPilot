# 6.4 KV 缓存与内存优化

## 1. 演进脉络与技术演进

LLM 推理的每次生成步骤需要计算当前 token 与所有历史 token 的 Attention 分数. 2020 年之前的实现不缓存 Key 和 Value, 每步重新计算全部历史 K/V, 计算复杂度 $O(L^2)$, 在 $L > 2048$ 时即不可行. KV Cache 在 2020 年后成为 Transformer 推理的标准配置, 将复杂度降至 $O(L)$, 但将内存瓶颈从计算侧转移至显存侧. 

2022 年至 2023 年初, 工业界关注点集中于 Attention 计算的 IO 效率. FlashAttention 通过 tiling 和重计算将 Attention 的内存复杂度从 $O(N^2)$ 压至接近 $O(N)$, 但其优化对象是 Attention 计算过程中的中间激活矩阵, 而非跨步缓存的 KV Cache 本身. 

2023 年成为 KV Cache 管理的范式跃迁之年. vLLM 在 SOSP 2023 提出 PagedAttention, 将操作系统虚拟内存的分页思想引入 GPU 显存管理, 首次系统性地解决了动态分配、内存碎片和跨序列共享三个工程难题. 同年, StreamingLLM 发现 Attention Sink 现象, 证明并非所有历史 KV 都必要, 为后续压缩技术奠定了实证基础. 

2024 年, 压缩技术爆发. H2O(NeurIPS 2024)通过累积 Attention Score 识别 Heavy Hitter, KVQuant 系统研究 INT8/FP8 量化的粒度与异常值处理, LoRA-KV 系列工作探索低秩投影的信息保留率边界. SnapKV 和 PyramidKV 进一步将动态剪枝精细化到层级别和头级别. 

2024 年至 2025 年, 架构重构成为主线. GQA 和 MQA 将 Key/Value 头数从与 Query 头数等比降至 1/8 甚至 1, 在模型设计层面压缩 KV Cache. DeepSeek-V2/V3 提出的 MLA(Multi-head Latent Attention)通过低秩联合压缩将 KV Cache 降至传统 MHA 的约 1/4, 且不损失模型质量. 

当前(2025-2026)的主流范式是组合优化：分页管理解决分配效率, 量化压缩解决数值精度冗余, 动态剪枝解决序列长度冗余, 架构重构解决设计层面的固有冗余. 四者正交叠加, 共同将 KV Cache 从推理瓶颈转化为可控资源. 

| 阶段 | 时间 | 核心问题 | 代表技术 | 压缩维度 |
|------|------|----------|----------|----------|
| 引入缓存 | 2020-2022 | 避免重复计算历史 K/V | KV Cache | 无(基线) |
| IO 优化 | 2022-2023 | Attention 计算内存效率 | FlashAttention | 计算过程激活值 |
| 内存管理 | 2023 | 动态分配、碎片、共享 | PagedAttention | 分配效率 |
| 内容压缩 | 2023-2024 | KV 本身占用过大 | H2O, StreamingLLM, KVQuant | 精度/维度/序列 |
| 架构重构 | 2024-2025 | 从设计层面减少 KV | GQA, MQA, MLA | 结构 |
| 组合策略 | 2025-2026 | 多技术协同与自适应 | 自适应 KV Cache | 全维度叠加 |

## 2. 工业价值与应用场景

KV Cache 优化直接决定 LLM 服务的经济可行性. 训练成本是一次性的; 推理成本是持续性的, 占在线服务总运营支出的 70% 以上. KV Cache 的大小与 batch size 和序列长度线性增长, 在长上下文和高并发场景下极易超过模型权重本身, 成为推理部署的首要瓶颈. 

以 Llama-3-70B 为例, $B=32, L=8192$ 时 KV Cache 占用约 80 GiB; $L=32768$ 时膨胀至 320 GiB; $B=128$ 时达 1.25 TiB. 单卡 A100/H100 显存仅 80 GiB. 这意味着：不做任何优化的单卡连中等并发的中等长度对话都承载不了. 

KV Cache 的物理约束直接映射到三个核心商业指标：

- **最大 batch size**：更大的 batch 意味着更高的 GPU 利用率, 但每个样本都需要独立的 KV Cache 副本. 
- **最大上下文长度**：长上下文能力是当前 LLM 竞争的核心卖点, 但每翻一倍上下文长度, KV Cache 翻一倍. 
- **并发用户数**：在线服务的 QPS 直接受限于同时能塞进显存的请求数量. 

| 场景 | 典型配置 | KV Cache 原始大小 | 核心痛点 | 首要优化方向 |
|------|----------|-------------------|----------|--------------|
| 在线聊天 | B=64, L=4K | 40-60 GB | 高并发显存容量 | PagedAttention + 量化 |
| 长文档分析 | B=8, L=128K | 200+ GB | 序列长度线性膨胀 | 动态剪枝 + 低秩压缩 |
| 代码补全 | B=128, L=8K | 300+ GB | 批处理与长上下文叠加 | 组合压缩 + 连续批处理 |
| 多轮对话 | B=32, L=32K | 300+ GB | 动态长度导致碎片 | PagedAttention + H2O |
| RAG 前缀共享 | B=16, L=16K | 80+ GB | 重复计算共享前缀 | PagedAttention 引用计数 |

从成本角度量化：假设某在线服务月 GPU 租金为 100 万元, KV Cache 优化将显存占用降低 50%, 意味着同等硬件支持 2 倍并发, 或同等并发所需 GPU 减半. 月运营成本直接下降 30-50 万元. 对于 C 端产品, 这意味着更低的延迟、更长的记忆能力、更便宜的订阅价格. 对于 B 端 API 服务, 这意味着更具竞争力的定价和更高的利润率. 

## 3. 物理直觉与核心机制

KV Cache 存储的是每个历史 token 在每一层、每一头的 Key 和 Value 向量. 这些向量是中间激活值, 不是模型权重. 模型权重在加载后固定不变; KV Cache 随序列增长和用户增加而线性膨胀. 在自回归生成的第 $t$ 步, Attention 输出为历史 Value 的加权和：

$$o_t = \sum_{i=1}^{t} \alpha_t(i) V_i \tag{1} $$

其中 $\alpha_t(i)$ 是第 $t$ 个 token 对第 $i$ 个历史 token 的 attention weight. 如果不缓存 $K_i$ 和 $V_i$, 每步需要重新计算 $1$ 到 $t$ 的所有 K/V, 计算量 $O(t \cdot d_{model})$ 每步, 总复杂度 $O(L^2 \cdot d_{model})$. 缓存后每步仅计算当前 token 的 K/V, 复杂度降至 $O(L \cdot d_{model})$. 

PagedAttention 的核心直觉来自操作系统虚拟内存. 传统 KV Cache 为每个序列预分配连续显存块, 导致三个问题：短序列浪费预分配空间(内部碎片)、不同长度序列交错释放产生外部碎片、共享前缀(如系统 Prompt)无法物理复用. PagedAttention 将 KV Cache 切分为固定大小的 Block(通常为 16 个 token), 通过 Block Table 维护逻辑块索引到物理块地址的映射. Block 不需要物理连续, 按需分配, 通过引用计数实现物理共享. 

KV Cache 压缩的直觉来自四个正交维度：

| 维度 | 操作对象 | 核心思想 | 压缩率来源 | 代表方法 |
|------|----------|----------|------------|----------|
| 精度 | 单个数值 | 用更少 bit 表示同一数值 | $P_{orig} / P_{comp}$ | INT8, FP8, INT4 |
| 维度 | 单个 token 的向量 | 用更低维空间近似高维向量 | $D_{head} / r$ | 低秩投影, MLA |
| 序列 | token 集合 | 丢弃低重要性 token | $L_{kept} / L_{orig}$ | H2O, StreamingLLM |
| 结构 | 层间冗余 | 相邻层共享 KV 投影矩阵或 Cache | $N_{group} / N_{layer}$ | Cross-Layer Sharing |

四个维度相互独立, 可以叠加. 低秩投影将 $d_{head}$ 从 128 降到 64(维度压缩 2x), INT8 量化将每数值从 2 字节降到 1 字节(精度压缩 2x), H2O 只保留 20% token(序列压缩 5x), 跨层共享将层数减半(结构压缩 2x). 组合压缩率为各维度压缩率的乘积：$2 \times 2 \times 5 \times 2 = 40$x. 

## 4. 数学推导与公式对比

标准 Multi-Head Attention 的 KV Cache 显存占用由六个因子相乘决定：

$$M_{KV}^{MHA} = 2 \times B \times L \times N_{layer} \times N_{head} \times D_{head} \times P \tag{2} $$
因子 2 对应 Key 和 Value 两个张量, $P$ 为每个数值的字节数(FP16 对应 $P=2$, FP32 对应 $P=4$, INT8/FP8 对应 $P=1$). 

Grouped Query Attention 将 $N_{head}$ 替换为 $N_{kv\_head}$(Key/Value 头数, 通常为 Query 头数的 1/8 或 1/4), KV Cache 的计算公式变为：

$$M_{KV}^{GQA} = 2 \times B \times L \times N_{layer} \times N_{kv\_head} \times D_{head} \times P \tag{3} $$

通过减少 KV 头数, GQA 将内存占用降低为原来的 $N_{kv\_head} / N_{head}$. Multi-Query Attention 进一步将 $N_{kv\_head}$ 降至 1, 达到极致压缩：

$$M_{KV}^{MQA} = 2 \times B \times L \times N_{layer} \times 1 \times D_{head} \times P \tag{4} $$
PagedAttention 的内存效率通过碎片率量化. 设系统中序列平均长度为 $L_{avg}$, 最大长度为 $L_{max}$. 传统预分配策略按 $L_{max}$ 为每个序列分配空间：

$$M_{prealloc} = 2 \times B \times L_{max} \times N_{layer} \times N_{kv\_head} \times D_{head} \times P \tag{5} $$

PagedAttention 按需分配, 实际占用：

$$M_{paged} = 2 \times B \times L_{avg} \times N_{layer} \times N_{kv\_head} \times D_{head} \times P + M_{metadata} \tag{6} $$
$M_{metadata}$ 为 Block Table 的元数据开销(每个物理块一个整数引用计数, 通常可忽略). 碎片率定义为浪费空间占预分配空间的比例：

$$\text{Fragmentation} = \frac{M_{prealloc} - M_{paged}}{M_{prealloc}} = 1 - \frac{L_{avg}}{L_{max}} \tag{7} $$

在线服务的序列长度通常服从重尾分布(大量短请求, 少量极长请求), $L_{avg} \ll L_{max}$, 碎片率可达 30%-60%. 

对称 INT8 量化的数学形式. 对张量 $x$, 计算最大绝对值 $\alpha = \max(|x|)$, 缩放因子 $s = \alpha / 127$, 量化映射：

$$x_q = \text{round}\left(\text{clip}\left(\frac{x}{s}, -128, 127\right)\right) \tag{8} $$
反量化重构：$\hat{x} = x_q \cdot s$. 由于 round 操作引入的误差在每个量化步长内近似均匀分布, 单值量化误差方差为：

$$\mathbb{E}[(x - \hat{x})^2] \approx \frac{s^2}{12} = \frac{\alpha^2}{127^2 \times 12} \tag{9} $$

该误差与缩放因子 $s$ 的平方成正比. 当 KV Cache 中存在异常值(outliers, 绝对值极大的元素)时, 全局 scale 必须放大以覆盖异常值, 导致绝大多数正常数值被压缩到粗糙的格点上, 精度严重损失. FP8(E4M3/E5M2)通过指数位提供动态范围, 在覆盖异常值的同时保留小数值精度, 更适合 KV Cache 的实际分布. 

低秩压缩利用 KV Cache 在特征维度上的冗余. 对 Key Cache $K \in \mathbb{R}^{L \times D_{head}}$ 计算协方差矩阵 $\Sigma_K = \frac{1}{L} K^T K$, 特征值分解 $\Sigma_K = U \Lambda U^T$. 若前 $r$ 个特征值占据总方差的绝大部分：

$$\eta = \frac{\sum_{i=1}^{r} \lambda_i}{\sum_{i=1}^{D_{head}} \lambda_i} > \tau \tag{10} $$
则 $K$ 的有效信息维度仅为 $r$, 可通过投影矩阵 $W_{proj} \in \mathbb{R}^{D_{head} \times r}$ 压缩：

$$K_{compressed} = K \cdot W_{proj} \in \mathbb{R}^{L \times r} \tag{11} $$

## 5. 算力成本与数值走查

以 Llama-3-70B 为基准模型：$N_{layer}=80, N_{head}=8, D_{head}=128, P=2$. 场景参数：$B=32, L=8192$. 

$$M_{KV} = 2 \times 32 \times 8192 \times 80 \times 8 \times 128 \times 2 = 85{,}899{,}345{,}920 \text{ bytes} \approx 80 \text{ GiB} \tag{12} $$
模型权重占用：$70 \times 10^9 \times 2 = 140 \text{ GB}$. KV Cache 已达权重的 57%. 若序列拉长至 $L=32768$, KV Cache 膨胀至 320 GiB, 超过权重本身. 若 batch size 扩大至 $B=128$, KV Cache 达 1.25 TiB. 

| 优化策略 | 参数调整 | KV Cache 大小 | 相对原始 | 信息损失风险 |
|----------|----------|---------------|----------|--------------|
| 原始 FP16 | 无 | 80.0 GiB | 100% | 无 |
| GQA (8->1 KV head) | $N_{kv\_head}=1$ | 10.0 GiB | 12.5% | 极低(架构原生) |
| INT8 对称量化 | $P=1$ | 40.0 GiB | 50% | 低(per-channel) |
| FP8 E4M3/E5M2 | $P=1$ | 40.0 GiB | 50% | 低(动态范围优势) |
| H2O 保留 20% | $L \to 0.2L$ | 16.0 GiB | 20% | 中(全局回溯任务有损) |
| StreamingLLM | sink(4)+window(512) | ~1.0 GiB | ~1.3% | 中(依赖 sink 容量) |
| 低秩 50% | $D_{head}: 128 \to 64$ | 40.0 GiB | 50% | 低(信息保留率 >95%) |
| 跨层共享 2x | 每 2 层共享 | 40.0 GiB | 50% | 低-中(浅层损失更大) |
| PagedAttention | 按需分配 | ~48-56 GiB | 60-70% | 无(管理优化) |
| MLA (DeepSeek-V2) | $D_{latent} \ll D_{head}$ | ~20 GiB | ~25% | 极低(训练适配) |
| 组合：GQA+INT8+H2O | 多策略叠加 | ~5 GiB | ~6% | 中(需任务适配) |

128K 上下文场景 ($B=16$)：

$$M_{KV}^{(128K)} = 2 \times 16 \times 131072 \times 80 \times 8 \times 128 \times 2 = 512 \text{ GiB} \tag{13} $$

单卡 80GB A100 需要 7 卡才能容纳 KV Cache. 应用 GQA($1/8$) + INT8($1/2$) + H2O($1/5$) 组合后：

$$M_{compressed}^{(128K)} = 512 \times 0.125 \times 0.5 \times 0.2 = 6.4 \text{ GiB} \tag{14} $$
从 512 GiB 压缩到 6.4 GiB, 压缩率 80x. 单卡即可承载, 且余量可用于增大 batch size 或运行更大模型. 

带宽瓶颈的数值分析. Decoder 阶段每次只生成一个 token, 但需要从 HBM 读取整个 KV Cache 到 SM 的共享内存参与 Attention 计算. 以 $B=32, L=8192$ 的 Llama-3-70B 为例, 每层需要读取：

$$M_{read}^{per\_layer} = 2 \times 32 \times 8192 \times 8 \times 128 \times 2 = 1{,}073{,}741{,}824 \text{ bytes} = 1 \text{ GB} \tag{15} $$

80 层共需从 HBM 读取 80 GB 的 KV Cache. A100 的 HBM 带宽为 2 TB/s, 纯读取 KV Cache 耗时约 40 ms. 这还不包括模型权重读取(140 GB, 约 70 ms)和实际矩阵计算. 若 KV Cache 压缩 10 倍, 读取时间降至 4 ms, TPOT(Time Per Output Token)从受带宽限制转为受计算限制, GPU Tensor Core 利用率显著提升. 

## 6. 极简实现与源码映射

以下代码实现 PagedAttention 风格的块级 KV Cache 管理器核心逻辑, 包含物理块池、Block Table、引用计数、按需分配和序列 fork(用于 beam search 或并行采样). 完整量化、低秩投影和动态剪枝的 PyTorch 实现见 [./6.4.2-KVCache压缩与优化技术.md](./6.4.2-KVCache压缩与优化技术.md). PagedAttention 的 CUDA kernel 级优化和 vLLM 集成细节见 [./6.4.1-PagedAttention原理.md](./6.4.1-PagedAttention原理.md). 

```python
import torch
from typing import Dict, List, Tuple, Optional

class BlockManager:
    """
    PagedAttention 风格的块级 KV Cache 管理器. 
    将 KV Cache 划分为固定大小的物理块, 通过 Block Table 映射逻辑位置到物理地址. 
    """
    def __init__(self, block_size: int, num_blocks: int, num_layers: int,
                 num_heads: int, head_dim: int, dtype: torch.dtype = torch.float16):
        self.block_size = block_size      # 每个物理块容纳的 token 数(通常为 16)
        self.num_layers = num_layers
        self.num_heads = num_heads
        self.head_dim = head_dim
        
        # 物理块池: (num_blocks, num_layers, 2, num_heads, block_size, head_dim)
        # dim=2 对应 Key(0) 和 Value(1)
        self.kv_blocks = torch.zeros(
            num_blocks, num_layers, 2, num_heads, block_size, head_dim,
            dtype=dtype, device="cuda"
        )
        
        # 空闲块索引列表
        self.free_blocks: List[int] = list(range(num_blocks))
        
        # 序列 ID -> Block Table(逻辑块索引 -> 物理块索引的列表)
        self.block_tables: Dict[int, List[int]] = {}
        
        # 物理块引用计数(用于共享前缀的生命周期管理)
        self.ref_count: List[int] = [0] * num_blocks
    
    def allocate(self, seq_id: int, num_tokens: int) -> List[int]:
        """为新序列分配物理块, 返回分配的物理块索引列表. """
        num_blocks_needed = (num_tokens + self.block_size - 1) // self.block_size
        if len(self.free_blocks) < num_blocks_needed:
            raise RuntimeError("Out of memory: insufficient free blocks")
        
        blocks = [self.free_blocks.pop() for _ in range(num_blocks_needed)]
        for b in blocks:
            self.ref_count[b] += 1
        self.block_tables[seq_id] = blocks
        return blocks
    
    def append(self, seq_id: int,
               new_kv: Tuple[torch.Tensor, torch.Tensor]) -> bool:
        """
        将新 token 的 KV 追加到序列末尾. 
        若当前逻辑块的最后一个物理块已满, 自动分配新块. 
        """
        k_new, v_new = new_kv
        # new_kv 形状: (num_layers, num_heads, new_len, head_dim)
        num_new = k_new.shape[2]
        
        blocks = self.block_tables[seq_id]
        total_capacity = len(blocks) * self.block_size
        current_len = self._get_seq_len(seq_id)
        
        # 容量不足时扩容
        if current_len + num_new > total_capacity:
            need = ((current_len + num_new - total_capacity + self.block_size - 1)
                    // self.block_size)
            if len(self.free_blocks) < need:
                return False  # OOM, 无法分配新块
            for _ in range(need):
                b = self.free_blocks.pop()
                self.ref_count[b] += 1
                blocks.append(b)
        
        # 按 token 位置写入物理块
        for pos in range(num_new):
            global_pos = current_len + pos
            block_idx = global_pos // self.block_size
            offset = global_pos % self.block_size
            phys_block = blocks[block_idx]
            
            self.kv_blocks[phys_block, :, 0, :, offset, :] = k_new[:, :, pos, :]
            self.kv_blocks[phys_block, :, 1, :, offset, :] = v_new[:, :, pos, :]
        
        return True
    
    def get_kv(self, seq_id: int) -> Tuple[torch.Tensor, torch.Tensor]:
        """按逻辑顺序拼接物理块, 返回完整 KV Cache 张量. """
        blocks = self.block_tables[seq_id]
        k_list, v_list = [], []
        for phys in blocks:
            k_list.append(self.kv_blocks[phys, :, 0, :, :, :])
            v_list.append(self.kv_blocks[phys, :, 1, :, :, :])
        
        k = torch.cat([
            b.view(self.num_layers, self.num_heads, -1, self.head_dim)
            for b in k_list
        ], dim=2)
        v = torch.cat([
            b.view(self.num_layers, self.num_heads, -1, self.head_dim)
            for b in v_list
        ], dim=2)
        return k, v
    
    def fork(self, parent_id: int, child_id: int) -> None:
        """
        创建序列副本(用于 beam search 或并行采样). 
        子序列与父序列共享物理块, 引用计数 +1, 无需物理拷贝. 
        """
        parent_blocks = self.block_tables[parent_id]
        self.block_tables[child_id] = parent_blocks.copy()
        for b in parent_blocks:
            self.ref_count[b] += 1
    
    def free(self, seq_id: int) -> None:
        """释放序列. 引用计数归零的物理块归还到空闲池. """
        if seq_id not in self.block_tables:
            return
        for b in self.block_tables[seq_id]:
            self.ref_count[b] -= 1
            if self.ref_count[b] == 0:
                self.free_blocks.append(b)
        del self.block_tables[seq_id]
    
    def _get_seq_len(self, seq_id: int) -> int:
        """获取序列当前长度(简化实现, 实际应由外部显式跟踪). """
        return len(self.block_tables.get(seq_id, [])) * self.block_size
```

该实现的核心设计点有三：第一, 物理块不保证逻辑连续, Block Table 通过索引列表维护映射关系, 消除外部碎片; 第二, `fork` 操作仅增加引用计数, 多个序列共享同一物理前缀时内存零拷贝; 第三, `append` 按需扩容, 避免预分配造成的内部碎片. 生产级实现(如 vLLM)在此基础上增加了 CUDA kernel 优化(支持不连续块上的 Attention 计算)、前缀缓存(Prefix Caching)和连续批处理(Continuous Batching)集成. 

## 7. 边界条件与失效模式

量化压缩在长序列中存在累积误差. 单个 token 的量化误差 $\delta K_i$ 通过 Attention 计算传播到后续所有步骤. 第 $t$ 步的 attention score 变为：

$$\hat{\alpha}_t(i) = \frac{\exp(Q_t (K_i + \delta K_i)^T / \sqrt{d})}{\sum_j \exp(Q_t K_j^T / \sqrt{d})} \tag{16} $$
Softmax 的指数放大效应使得微小误差在 attention score 的相对排序中被放大. $\delta K_i$ 不仅改变 $\hat{\alpha}_t(i)$, 还通过 $o_t$ 影响下一层的输入表示, 进而改变下一层生成的 KV, 误差开始级联传播. 

INT8 量化在 4K 上下文中 perplexity 差异通常 $<1\%$, 在 128K 上下文中上升至 $5\%-10\%$. 法律文书中的精确条款引用、长文档中的数值计算、学术论文中的公式推导等任务对累积误差最为敏感. 实践中, Key Cache 的量化应比 Value Cache 更保守(Key 决定 attention pattern 的排序, Value 的误差被加权平均后部分抵消). 

H2O 和 StreamingLLM 类动态剪枝方法依赖一个根本性假设：过去被高度关注的 token 在未来也容易被关注. 该假设在以下任务中失效：

| 任务类型 | 失效表现 | 物理根因 |
|----------|----------|----------|
| 长文档摘要 | 前 90% 生成关注局部细节, 最后 10% 需回溯文档开头核心论点 | attention score 时间分布极不均匀 |
| 多轮对话指代消解 | 第 1 轮关键信息在第 10 轮后因累积 score 被稀释而丢弃 | 近期局部信息淹没历史全局信息 |
| 跨函数代码生成 | 文件末尾函数引用开头定义的类, 中间实现细节占据 attention 主导 | 中间 token 的局部 attention 权重过高 |
| 精确数值推理 | 长链式计算中早期数字 token 被剪枝, 导致后续计算引用错误 | 数字 token 的 attention weight 分布稀疏 |

StreamingLLM 通过强制保留 sink tokens(序列最前面的 1-4 个 token)缓解全局信息丢失. 但 sink token 的信息容量有限, 无法承载复杂的多主题文档或长链式对话历史. 

PagedAttention 的 block size 存在工程权衡. 块过小(如 1 个 token)导致 Block Table 元数据膨胀、显存访问不连续(GPU 无法有效合并内存事务)、CUDA kernel 启动开销增大. 块过大(如 256 个 token)导致内部碎片增加——序列末尾未用完的块空间被浪费, 且共享粒度变粗(无法细粒度共享前缀). vLLM 默认 block size 为 16, 是在元数据开销、碎片率和共享粒度之间的经验平衡点. 对于平均序列长度极短(< 32 token)的场景, block size 应降至 8 或 4. 

跨层共享的精度损失在浅层和深层分布不均匀. 浅层(前 1/3 层)的输入表示 $X^{(l)}$ 和 $X^{(l+1)}$ 差异较大, 相邻层 attention pattern 正在从原始输入中"发现"不同类型的局部结构, 共享引入的误差更显著. 深层(后 1/3 层)的表示已进入相对稳定的高维流形, 相邻层趋于相似. 经验策略：浅层使用独立 KV Cache 或 2 层一组的小共享组, 深层使用 4-8 层一组的大共享组. 

组合策略存在交互陷阱. 量化 + 低秩叠加时, 低秩投影矩阵改变了数值分布范围, 原始量化 scale 不再适用, 需要重新在投影后的数据上校准 scale. 低秩 + 剪枝叠加时, 投影矩阵 $W_{proj}$ 若基于完整数据训练, 剪枝后的数据分布偏移可能导致投影方向次优. 实践中应在剪枝后的数据分布上重新计算或微调投影矩阵. 

## 8. 技术前瞻与演进

MLA(Multi-head Latent Attention)代表了 KV Cache 优化的架构级终极方向. 传统 MHA 中, 每个 token 每层需要存储 $N_{head}$ 个 Key 向量和 $N_{head}$ 个 Value 向量, 总维度 $2 \times N_{head} \times D_{head}$. MLA 通过一个低维的 latent 向量 $c \in \mathbb{R}^{d_c}$ 联合编码 Key 和 Value 的信息, 其中 $d_c \ll N_{head} \times D_{head}$. 推理时, latent 向量通过解耦的投影矩阵分别恢复为 Key 和 Value：

$$K = c \cdot W_{dek}^T, \quad V = c \cdot W_{dev}^T \tag{17} $$

DeepSeek-V2 中, $d_c$ 的设计使得每层每个 token 的 KV Cache 降至传统 MHA 的约 1/4, 且通过训练阶段的低秩约束保持了模型质量. MLA 将 KV Cache 压缩从"推理时的后处理优化"提升为"训练时的架构设计约束", 从根本上改变了优化范式. 

自适应 KV Cache(Adaptive KV Cache)是 2026 年的前沿方向. 其核心目标是根据输入内容类型、任务性质和当前显存预算, 动态选择压缩策略的组合与强度. 实现路径包括两类：

| 路径 | 机制 | 优势 | 挑战 |
|------|------|------|------|
| 规则状态机 | 基于任务类型(代码/写作/摘要)和序列长度选择预设压缩配置 | 实现简单、可解释性强 | 无法处理混合内容输入 |
| 轻量策略网络 | 在每层附加一个小型 MLP, 根据当前 hidden state 预测最优压缩参数 | 细粒度、内容自适应 | 训练数据难以构造、推理开销 |

硬件-算法协同设计将 KV Cache 优化推向新维度. NVIDIA Hopper/Blackwell 架构的原生 FP8 Tensor Core 支持降低了量化推理的转换开销. 未来专用加速器可能直接支持：可变精度 KV Cache(不同层自动选择 INT8/FP8/FP16)、硬件级块表管理(Block Table 由硬件 MMU 维护)、以及将 KV Cache 从 HBM 迁移到更廉价的存储层级(CPU 内存或 NVMe SSD). 后者的核心思想是：仅将当前计算需要的 KV Block 保留在 GPU HBM 中, 其余块按 LRU 策略换出到主机内存, 通过 PCIe/NVLink 按需分页加载. 这在理论上可将上下文窗口扩展至"无限", 代价是极端长上下文下的访问延迟. 

混合 GPU-CPU 注意力机制(HGCA)将跨层级 KV 缓存管理从理论构想推进为工程现实. 现有方案存在二元对立：将 KV 缓存完全卸载到 CPU 内存虽可扩展容量, 但每次注意力计算需通过 PCIe 将完整 KV 缓存传输到 GPU, 在 16 通道 PCIe 4.0 上带宽仅 32 GB/s, 成为严重瓶颈; 稀疏注意力虽只保留高注意力分数的 KV 子集在 GPU 内, 但关键 KV 的误判会直接损害模型准确性. HGCA(2025)提出了一种非对称的混合计算范式：对保留在 GPU 内存中的近期 KV 条目执行**密集注意力**(保证核心上下文精度), 同时对卸载到 CPU 内存中的历史 KV 条目执行**并行稀疏注意力**(利用 CPU 算力处理长尾上下文), 最终通过**对数求和指数(log-sum-exp)融合**高效合并两个注意力输出, 最小化跨总线数据传输. 

HGCA 的关键创新在于细粒度的**每头稀疏化策略**(per-head sparsity). 不同注意力头对历史上下文的依赖模式差异显著：某些头(如局部语法头)主要关注邻近 token, 而另一些头(如全局主题头)需要回溯远距离信息. HGCA 为每个头独立维护稀疏化预算, 对 CPU 侧的历史 KV 按头级别选择显著条目, 而非全局统一剪枝. 这保留了 CPU 稀疏注意力的上下文相关性, 同时针对 CPU 执行优化了计算图(减少不规则内存访问, 提高 SIMD 利用率). 

实验表明, HGCA 在多种模型和工作负载上实现了接近完整注意力的准确性(显著优于统一稀疏基线), 同时支持更长的序列和更大的批处理大小, 且**无需模型重新训练**即可无缝集成到现有 LLM 框架. 这为消费级 GPU 上的长上下文推理提供了一条新路径：通过非对称计算(GPU 做密集局部注意力 + CPU 做稀疏全局注意力)突破单卡显存墙, 而非单纯依赖硬件升级. 

推测解码(Speculative Decoding)与 KV Cache 的协同是另一演进方向. Medusa、Lookahead Decoding 等方法不再依赖独立的 draft model, 而是利用大模型已有的 KV Cache 直接预测未来多个 token. KV Cache 从"被动的历史存储"转化为"主动的预测资源", 其管理策略需要同时支持已确认前缀和候选分支的两套缓存体系. 

**TurboQuant：旋转 + Lloyd-Max 量化的 KV 压缩新范式**. Google 在 ICLR 2026 提出的 TurboQuant 代表了 KV Cache 量化的最新突破. 其核心思路包含三步：

1. **随机旋转**：用一个正交矩阵旋转 KV 向量, 让每个坐标的分布变为已知的 Beta 分布. 旋转消除了 KV 向量各维度之间的相关性, 使每个维度可以独立量化. 
2. **Lloyd-Max 量化**：由于分布已知, 可以预计算信息论最优的量化码本(codebook), 无需像传统 per-channel 量化那样每组存储 min/max. 这显著降低了量化元数据的存储开销. 
3. **QJL 残差修正**：用 1-bit 的 Johnson-Lindenstrauss 投影修正量化误差, 在几乎不增加存储的情况下大幅提升重建精度. 

原始论文实现了 6x KV Cache 压缩、8x 注意力加速、零精度损失, 但仅压缩了 Key, Value 保持 FP16. 社区复现进一步实现了 K+V 双压缩, 将显存节省从 37% 提升到 73-99%. TurboQuant 的关键优势在于"分布已知"——通过旋转将任意分布转化为标准分布, 使最优量化器可以离线预计算, 推理时只需查表. 

**Flash Decoding：解码阶段的并行度解放**. Flash Attention 在 Prefill 阶段表现出色(Q/K/V 长度相同, 可分块并行), 但在 Decode 阶段面临并行度瓶颈——每个新 token 的 Query 只有一个, 而 KV Cache 长度可达数万, 小 batch 下 GPU 计算资源严重闲置. 

Flash Decoding 的核心创新是在 **KV 维度上分块并行**：将长序列的 KV Cache 切分为多个块, 每个块与一个 thread block 绑定, 多个 thread block 同时计算局部 attention, 最后通过 online softmax 的扩展规则合并结果. 这实现了两个层面的并行：

- **块内并行**：每个 KV 块内部的计算充分利用 Tensor Core
- **块间并行**：多个 KV 块同时处理, 隐藏内存访问延迟

实验表明, Flash Decoding 在 Decode 阶段可将吞吐量提升 2-8 倍(取决于序列长度), 且与 Flash Attention 的 KV Cache 格式完全兼容, 无需修改模型结构即可即插即用.
