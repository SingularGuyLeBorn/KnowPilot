# 6.4.1 PagedAttention 深度技术指南：LLM KV Cache 高效内存管理与 vLLM 实现解析


## 1. LLM 推理基础与 KV Cache (Fundamentals of LLM Inference and KV Cache)

大型语言模型的核心通常是基于 Transformer 架构构建的. 在其自回归生成过程中, 为了高效地预测下一个词元 (Token), 模型需要回顾所有先前已处理的词元所包含的上下文信息. 这些上下文信息以键 (Key) 和值 (Value) 向量的形式被缓存起来, 即我们常说的 **KV 缓存**.

### 1.1 LLM 推理的两个阶段 (Two Stages of LLM Inference)

LLM 的推理过程通常可以划分为两个主要阶段：

- **Prefill 阶段 (预填充)** : 在此阶段, 模型接收整个用户输入的提示 (Prompt) 序列作为一个整体输入. 此时, 所有输入词元会并行地通过模型计算, 并生成对应的 KV 缓存. 这些 KV 缓存会被保存起来, 以供后续的生成阶段使用. 例如, 如果 Prompt 包含 3 个词元, Prefill 阶段结束后, 这 3 个词元的所有 KV 值都已准备就绪并存储在缓存中.

- **Decode 阶段 (自回归生成)** : 顾名思义, 此阶段是模型逐个生成响应词元的过程. 每生成一个新词元, 模型都会利用当前已生成的词元和所有历史词元的 KV 缓存来预测下一个词元. 新生成的词元的 KV 值也会被添加到 KV 缓存中. 例如, 在生成第 4 个词元时, 模型会与缓存中第 0 到第 3 个词元的 KV 值计算注意力, 然后将自身的 KV 值也存入缓存. 由于 Decode 阶段是串行逐个生成词元的, 它无法像 Prefill 阶段那样进行大规模的并行计算, 因此在 LLM 推理中, Decode 阶段的耗时通常是更大的.

### 1.2 KV Cache 的作用与挑战 (Role and Challenges of KV Cache)

KV 缓存的存在显著加速了 LLM 的推理, 避免了在生成每个新词元时重复计算历史词元的注意力键和值. 然而, 随着 Prompt 长度的增加以及生成序列的延长, KV 缓存也会随之增大. 例如, 对于一个 130 亿参数的 OPT 模型, 单个词元的 KV 缓存可能需要约 800 KB 的空间, 而一个长达 2048 个词元的序列可能需要高达 1.6 GB 的 KV 缓存. 考虑到现有 GPU 的内存容量通常在几十 GB 级别, 这意味着 GPU 内存中大部分空间可能被 KV 缓存占据, 从而对 GPU 显存造成巨大压力.

**引导性问题**: 为什么 KV 缓存在 LLM 推理中如此重要？它在 Transformer 的自注意力机制中是如何产生的？

#### 1.2.1 数学背景 (Mathematical Background)

在 Transformer 模型的自注意力层中, 对于输入的一系列隐藏状态 $(h_1, \dots, h_T)$, 模型会通过线性变换生成查询 (Query)、键 (Key) 和值 (Value) 向量：

$q_i = W_q h_i, \quad k_i = W_k h_i, \quad v_i = W_v h_i$

其中 $W_q, W_k, W_v$ 是可学习的权重矩阵. 键向量 $k_i$ 和值向量 $v_i$ 构成了 KV 缓存的核心. 在计算第 $i$ 个词元的注意力输出 $o_i$ 时, 其查询向量 $q_i$ 会与所有先前词元(包括自身)的键向量 $k_j$ 计算注意力得分, 然后用这些得分作为权重对值向量 $v_j$ 进行加权平均：

$a_{ij} = \frac{\exp(q_i^T k_j / \sqrt{d})}{\sum_{t=1}^i \exp(q_i^T k_t / \sqrt{d})}, \quad o_i = \sum_{j=1}^i a_{ij} v_j$

这里的 $d$ 是键向量的维度. 每次计算 $o_i$ 时, 都需要访问 $k_1, \dots, k_i$ 和 $v_1, \dots, v_i$. 因此, 将这些 $k_j$ 和 $v_j$ 缓存起来, 避免重复计算, 便是 KV Cache 的核心思想.

## 2. 传统 KV Cache 管理的局限性 (Limitations of Traditional KV Cache Management)

在 LLM 推理服务中, 传统的 KV Cache 管理方法往往效率低下, 导致显著的内存浪费和吞吐量限制. 这主要源于其对内存的**连续性要求**和**静态预分配策略**.

### 2.1 连续内存分配策略 (Contiguous Memory Allocation Strategy)

当前大多数深度学习框架中的操作符都假定张量(包括 KV Cache)存储在连续的内存空间中. 因此, 传统的 LLM 服务系统也沿袭了这一做法, 为每个请求的 KV Cache 分配一块连续的内存区域. 然而, LLM 的输出长度是不可预测的, 这就带来了固有的挑战.

### 2.2 三大内存浪费来源 (Three Major Sources of Memory Waste)

由于输出长度的未知性, 传统的系统通常会为每个请求静态地预分配一块内存, 其大小基于请求可能的最大序列长度(例如, 2048 个词元). 这种做法导致了以下三种主要的内存浪费形式：

1. **内部碎片 (Internal Fragmentation)** :

- **定义**: 指的是系统为请求分配了内存, 但其中一部分并未被实际使用. 例如, 一个请求的最大序列长度被设置为 2048 词元, 但实际只生成了 500 词元, 那么剩余的 1548 词元对应的空间就成为了内部碎片.

- **问题**: 这部分未使用的空间即使理论上是空闲的, 也无法被其他请求利用, 因为它属于当前请求的预分配区域.

1. **外部碎片 (External Fragmentation)** :

- **定义**: 当内存中存在大量不连续的、小的空闲块, 这些小块的总和可能足以满足一个新请求的内存需求, 但由于没有一个连续的空闲块足够大, 新请求无法被分配内存. 这种情况就像仓库里有很多小箱子, 但没有一个大箱子能装下你的大件物品.

- **问题**: 传统的系统为不同请求预留不同大小的连续内存块, 当请求完成释放内存后, 会留下大小不一的空闲块, 这些零散的空闲块难以被高效利用, 进一步限制了批处理的规模.

1. **预留浪费 (Reservation Waste)** :

- **定义**: 指的是为未来可能生成的词元而提前预留的内存槽位. 虽然这些槽位最终可能会被使用, 但在请求的整个生命周期内, 它们被独占地保留, 阻止了其他请求对这些空间的利用, 尤其是在预留空间较大且长时间未被填满时.

**视觉化描述**: 如下图所示, 传统系统中, 请求 A 和请求 B 的 KV Cache 分别被预分配了固定大小的连续内存空间. 其中：浅色块代表已使用的 KV Cache, 中色块是已预留但尚未填满的槽位(预留碎片), 深色块是序列结束后仍未使用的预留空间(内部碎片), 而灰色块则是由于不连续分配导致无法被利用的空闲空间(外部碎片). 这种模式使得实际用于存储词元状态的 KV Cache 空间, 在现有系统中可能低至 **20.4% - 38.2%**.

### 2.3 缺乏内存共享机制 (Lack of Memory Sharing)

除了碎片化问题, 传统系统还无法有效地利用内存共享的机会. 在 LLM 服务中, 存在多种解码算法, 如**并行采样 (Parallel Sampling)** 和**束搜索 (Beam Search)** , 它们会为单个请求生成多个输出序列. 这些序列通常共享相同的前缀 (例如, 共同的 Prompt). 然而, 由于每个序列的 KV Cache 都被存储在独立的连续内存空间中, 这些系统无法实现内存共享, 导致相同前缀的 KV Cache 被重复计算和存储, 进一步加剧了内存浪费.

## 3. PagedAttention 核心思想：虚拟内存与分页 (PagedAttention: Inspired by Virtual Memory and Paging)

为了克服传统 KV Cache 管理的固有缺陷, PagedAttention 算法应运而生. 它的设计灵感直接来源于操作系统中**虚拟内存**和**分页技术**的成熟理念. 要理解 PagedAttention, 我们首先需要回顾一下操作系统内存管理的发展历程.

### 3.1 操作系统虚拟内存回顾 (A Review of Operating System Virtual Memory)

想象一下, 计算机中的程序需要访问内存来存储数据和指令.

1. **不使用虚拟内存 (Direct Physical Addressing)** :
在最原始的计算机系统中, 程序直接操作物理内存地址. 这就像一个单片机, 程序精确地知道要使用内存中的哪个具体位置. 如果只有一个程序运行, 这倒没什么问题. 但当多个程序需要同时运行时, 麻烦就来了：每个程序在分配内存时, 都需要小心翼翼地避免与其它程序占用的物理地址冲突. 这导致不同程序之间耦合度极高, 开发难度大, 且难以实现隔离和保护.

2. **分段式内存管理 (Segmented Memory Management)** :
为了解决上述问题, 操作系统引入了**虚拟内存 (Virtual Memory)** 的概念. 每个程序都被分配一个独立的**虚拟地址空间**, 程序在开发和运行时都只使用虚拟地址, 仿佛自己独占了整个内存. 操作系统则负责将程序的虚拟地址映射到真实的物理内存地址上.
在分段式管理中, 虚拟内存被划分为多个逻辑段(如代码段、数据段、堆栈段等), 每个段可以有不同的大小, 并被映射到物理内存中的**连续区域**.
然而, 分段式管理仍然存在缺陷：

- **外部碎片**: 尽管物理内存总空间可能充足, 但由于其被分割成大小不一的空闲段, 一个大的新程序可能无法找到足够大的连续空间来加载, 即使总空闲内存足够. 这就像一个大包裹需要一个完整的货车车厢, 即使有多个小货车车厢空着也无济于事.

- **交换开销**: 当物理内存不足时, 操作系统可能需要将某个程序的一个或多个整个段从物理内存**交换 (Swap)** 到磁盘上, 以腾出空间给新程序, 然后再将其换回. 如果段很大, 且内存与磁盘之间的带宽有限, 这个交换过程会非常耗时, 导致系统卡顿.

1. **分页式内存管理 (Paged Memory Management)** :
为了进一步解决碎片化和交换效率问题, 分页式内存管理被提出. 其核心思想是：

- **固定大小的块**: 将物理内存划分为固定大小的块, 称为**页帧 (Page Frame)** . 虚拟内存也以相同大小的块进行划分, 称为**页 (Page)** .

- **按需加载**: 程序不再需要一次性将所有代码和数据加载到物理内存. 相反, 只有当程序实际需要访问某个页时, 对应的页才会被从磁盘加载到物理内存中的一个空闲页帧, 并建立映射关系.

- **逻辑连续, 物理非连续**: 对于程序而言, 它看到的虚拟地址空间是连续的. 但在物理内存中, 这些虚拟页对应的页帧可以是非连续的. 操作系统通过**页表 (Page Table)** 来维护这种逻辑页到物理页帧的映射关系. 这就像一本书的章节在目录上是连续的, 但在图书馆中, 这些章节可能散落在不同书架的不同书籍中.

**总结**: 分页式管理通过将内存细粒度化、按需分配和允许物理非连续存储, 极大地提高了内存的利用率, 减少了碎片, 并提升了交换效率.

### 3.2 PagedAttention 的借鉴与创新 (PagedAttention: Borrowing and Innovating)

PagedAttention 正是借鉴了操作系统分页式内存管理的精髓, 将其应用于 LLM 推理中的 KV Cache 管理. 它将 KV Cache 的动态管理问题转化为一个类似操作系统内存分页的问题.

- **核心类比 (Core Analogy)** :

- LLM 的**请求 (Request)** : 类比操作系统中的**进程 (Process)** .- 请求的 **KV Cache 序列**: 类比进程的**虚拟内存空间**.- KV Cache 中的 **Token**: 类比虚拟内存中的**字节 (Byte)** .- PagedAttention 中的 **逻辑块 (Logical Block)** : 类比操作系统虚拟内存中的**页 (Page)** .- PagedAttention 中的 **物理块 (Physical Block)** : 类比操作系统物理内存中的**页帧 (Page Frame)** .- PagedAttention 中的 **块表 (Block Table)** : 类比操作系统的**页表 (Page Table)** .

- **KV Cache 的块化 (Blocking KV Cache)** :
  PagedAttention 不再将整个 KV Cache 视为一个连续的整体, 而是将其划分为固定大小的**KV 块 (KV Blocks)** . 每个块包含固定数量 Token 的键和值向量. 在 vLLM 中, 默认的块大小为 16, 即一个块可以存储 16 个 Token 的 K/V 值. 这种固定大小的块设计是解决外部碎片问题的关键.

- **逻辑与物理分离 (Logical-Physical Separation)** :
  每个请求的 KV Cache 在逻辑上表现为一系列连续的**逻辑块**. 但在 GPU 的物理显存中, 这些逻辑块可以被映射到**非连续的物理块**上. 这种分离由**块表 (Block Table)** 来维护, 块表记录了每个逻辑块对应的物理块 ID 以及该物理块当前被填充的槽位数.

- **按需分配 (On-Demand Allocation)** :
  vLLM 不会像传统系统那样为请求预分配最大长度的 KV Cache 空间. 相反, 它会根据实际需要动态分配物理块. 当 Prefill 阶段完成后, 或者 Decode 阶段当前逻辑块已满时, vLLM 会分配新的物理块来存储新生成的 KV Cache. 这种“用多少占多少”的策略, 从根本上消除了内部碎片和预留浪费.

1. **Decode 阶段 - 生成第 1 个词元**:

- **计算注意力**: 模型计算新词元 "fathers" 的注意力. 在逻辑上, 模型看到的是连续的 Token 序列. 实际上, vLLM 后台通过块表从非连续的物理块中获取数据进行计算. 这样, 每个请求都仿佛在连续且充足的空间中操作.

1. **Decode 阶段 - 生成第 2 个词元**: 流程类似, 持续动态分配和填充.

## 4. PagedAttention 的实现与高级特性 (PagedAttention Implementation and Advanced Features)

PagedAttention 不仅通过其基础的分页机制解决了内存碎片问题, 还通过引入引用计数、写时复制等高级特性, 极大地增强了 KV Cache 的共享能力, 从而在复杂解码场景和高并发服务中展现出卓越的性能.

### 4.1 KV Cache 块管理 (KV Cache Block Management)

PagedAttention 的核心在于对 KV Cache 的块级管理.

- **逻辑块与物理块 (Logical Blocks and Physical Blocks)** :

- **逻辑块**: 代表了序列 KV Cache 在逻辑上连续的视图. 对应用程序(如 LLM 推理计算)而言, 它仿佛访问的是一段连续的内存区域.

- **物理块**: 是 GPU 显存中实际存储 KV Cache 数据的固定大小的连续内存区域. 它们在物理地址上不要求连续. 每个物理块的大小是固定的, 例如 vLLM 默认的 **块大小 (Block Size)** 为 16 个 Token 的 KV 值. 选择合适的块大小至关重要：过小可能无法充分利用 GPU 并行性; 过大则会增加内部碎片.

- **块表 (Block Table)** :

- 块表是 PagedAttention 实现逻辑与物理分离的关键数据结构. 它维护着每个序列的逻辑块到物理块的映射关系. 每个条目记录了逻辑块对应的物理块 ID, 以及该物理块中已填充的 Token 槽位数. 在 KV Cache Manager 中, 块表是管理内存分配和回收的依据.

- **引用计数 (Reference Count)** :

- 为了实现物理块的共享, 每个物理块都关联一个**引用计数**. 这个计数器记录了有多少个逻辑块(来自不同序列或同一序列的不同分支)正在引用同一个物理块.- 当一个物理块的引用计数大于 1 时, 表示它正在被多个序列共享.- 当一个物理块的引用计数减少到 0 时, 意味着没有任何序列再使用它, 此时该物理块可以被释放或放入空闲池中等待复用.

- **写时复制 (Copy-on-Write, CoW)** :

- CoW 是在共享内存场景下确保数据正确性的重要机制. 当多个逻辑块共享同一个物理块时, 如果其中一个逻辑块需要修改(即写入新的 KV 值), CoW 机制会被触发.

- **触发条件**: PagedAttention 检测到待写入的物理块的引用计数大于 1.

- **执行过程**: PagedAttention 不会直接修改共享的物理块, 而是会为即将修改的逻辑块分配一个新的物理块, 并将旧物理块的内容复制到新物理块中. 然后, 修改操作在新物理块上进行, 同时旧物理块的引用计数减 1. 这就像分叉进程时, 父子进程共享内存, 直到子进程尝试写入时, 才会复制一份私有副本.

### 4.2 多种解码策略下的内存共享 (Memory Sharing with Various Decoding Strategies)

PagedAttention 的块级管理和 CoW 机制使其能够高效支持多种复杂的 LLM 解码策略, 大幅提升内存共享效率.

#### 4.2.1 并行采样 (Parallel Sampling)

- **场景描述**: 在并行采样中, 用户对同一输入 Prompt 请求生成多个独立的输出样本. 例如, 生成多个代码建议或回答, 用户可以从中选择最佳项.

- **PagedAttention 如何优化**:

- **Prompt 共享**: 由于所有样本都源自相同的 Prompt, 它们的 Prompt 部分的 KV Cache 是完全相同的. PagedAttention 允许这些不同样本的逻辑块(对应于 Prompt 部分)映射到**同一个物理块**. 这些共享的物理块的引用计数会相应增加.

- **CoW 机制**: 当不同样本开始生成各自不同的 Token 时, 它们会在其对应的逻辑块中写入新的 KV 值. 如果这个逻辑块当前映射的物理块是共享的(引用计数大于 1), CoW 机制就会被触发. 系统会为要写入的样本分配一个新的物理块, 复制旧内容, 然后写入新数据, 并减少旧物理块的引用计数.

- **优势**: 大幅减少了长 Prompt 在并行采样场景下的内存占用, 特别是当 Prompt 占据了 KV Cache 的主要部分时, 内存节省效果更为显著.

#### 4.2.2 束搜索 (Beam Search)

- **场景描述**: 束搜索 (Beam Search) 是一种常用的解码算法, 用于在每一步生成中保留概率最高的 $k$ 个候选序列(束宽为 $k$), 而不是只选择一个最优路径. 它常用于机器翻译等任务, 以寻找最佳输出.

- **PagedAttention 如何优化**:

- **动态共享**: 束搜索比并行采样具有更复杂的共享模式. 除了初始 Prompt 部分, 不同束候选序列之间还可能共享更长的公共前缀. PagedAttention 能够动态地识别并共享这些公共前缀所对应的物理块.

- **淘汰与回收**: 随着解码的推进, 一些束候选序列会因为概率较低而被淘汰. PagedAttention 会及时释放这些被淘汰序列所占用的逻辑块, 并减少对应物理块的引用计数. 当引用计数降为 0 时, 物理块会被回收, 供其他请求复用. 这类似于操作系统中进程树的动态变化.

- **优势**: 显著减少了束搜索过程中频繁的 KV Cache 复制开销, 尤其是在束宽较大时, 内存节省和吞吐量提升更为明显.

#### 4.2.3 共享前缀 (Shared Prefix)

- **场景描述**: 在许多 LLM 应用中, 特别是使用 Prompt Engineering 时, 多个用户请求可能会共享一个长任务描述、指令或示例 (System Prompt).

- **PagedAttention 如何优化**: LLM 服务提供商可以预先计算并缓存这些共享前缀的 KV Cache. vLLM 可以方便地维护一组用于预定义共享前缀的物理块, 就像操作系统管理共享库一样. 当用户请求包含这些共享前缀时, 其对应的逻辑块可以直接映射到缓存的物理块, 只有用户特定输入的 KV Cache 需要重新计算.

- **优势**: 大幅减少了对相同 System Prompt 的重复计算和内存占用, 提高了长 Prompt 场景下的效率.

#### 4.2.4 混合解码方法 (Mixed Decoding)

- **特点**: PagedAttention 能够同时高效处理具有不同解码偏好(如贪婪解码、Top-K 采样、Beam Search)的请求. 这是现有系统难以高效实现的能力.

- **原理**: PagedAttention 通过统一的块表和逻辑-物理映射层, 向 LLM 及其执行内核隐藏了不同序列之间复杂的内存共享模式. 无论底层共享如何, 内核只需处理物理块 ID 列表, 从而提高了批处理的灵活性和整体吞吐量.

### 4.3 请求调度与抢占 (Request Scheduling and Preemption)

当请求流量超过系统处理容量时, vLLM 必须有效地管理队列并分配资源. PagedAttention 的动态内存管理与灵活的调度抢占策略相结合, 确保了系统的高效运行和公平性.

- **调度原则 (Scheduling Principles)** :

- **先来先服务 (First-Come-First-Serve, FCFS)** : vLLM 采用 FCFS 策略来确保请求的公平处理, 防止饥饿现象(即某些请求长时间得不到处理).

- **后到先抢占 (Preemption for Later Arrivals)** : 当 GPU 显存资源不足时, 为了确保最早到达的请求能够尽快完成, vLLM 会选择最新到达的、优先级较低的请求进行抢占, 即暂停其推理并释放其占用的 KV Cache 资源. 这样做可以避免所有请求因争夺资源而都无法完成, 提高整体完成率.

- **内存不足时的处理 (Handling Out-of-Memory Situations)** :
当 GPU 物理块空间耗尽, 无法存储新生成的 KV Cache 时, vLLM 会触发抢占机制. 对于被抢占的请求, vLLM 提供了两种恢复机制:

1. **Swapping (交换策略)** :

- **机制**: 类似于操作系统将不活跃的页面交换到磁盘, vLLM 会将被抢占请求的 KV Cache 从 GPU 显存复制到 **CPU 内存**中.

- **逐出策略**: PagedAttention 采取“**全有或全无 (All-or-Nothing)** ”的逐出策略, 即要么将整个序列(或同一请求的序列组)的所有 KV Cache 块都交换出去, 要么都不交换. 这是因为序列的所有块通常是同时被访问的.

- **恢复**: 当 GPU 显存资源充足时, 被交换到 CPU 内存的 KV Cache 块会被重新加载回 GPU 显存, 然后请求继续处理.

- **限制**: 交换到 CPU RAM 的块数量不会超过 GPU RAM 中分配给 KV Cache 的总物理块数, 因此 CPU RAM 上的交换空间受限于 GPU 内存.

1. **Recomputation (重计算策略)** :

- **机制**: 对于某些类型的任务(例如并行采样中生成单个样本的任务), vLLM 在抢占时可以直接丢弃其 KV Cache 物理块, 而不是进行交换. 被抢占的请求会被重新放回等待队列.

- **恢复**: 当轮到这些请求再次被调度时, 其 KV Cache 会从头开始重新计算. 值得注意的是, 重新计算的延迟通常会显著低于原始 Prefill 阶段的延迟, 因为它可以将用户 Prompt 和已生成的 Token 连接起来作为新的 Prompt, 在一次 Prefill 阶段迭代中生成所有必要的 KV Cache.

- **vLLM 默认策略**: 在 vLLM 的早期版本中, 对于单个序列(n=1), Recomputation 是默认的抢占恢复模式, 因为它通常具有较低的开销. 但对于包含多个序列的序列组(如 Beam Search), 由于其复杂性, 通常仍采用 Swapping 模式.

- **权衡与选择**: Swapping 和 Recomputation 各有优劣. Swapping 避免了重复计算, 但受限于 PCIe 带宽和 CPU 内存的容量及速度; Recomputation 节省了内存传输, 但增加了计算量. 块大小对这两种策略的性能有影响：小块大小会导致大量小的传输, 使 Swapping 开销变大, 此时 Recomputation 可能更优; 大块大小则相反. 对于中等块大小(16 到 64), 两者性能相当.

**引导性问题**: 抢占机制对于 LLM 服务在高负载情况下的稳定性和用户体验有何重要意义？在实际部署中, 选择 Swapping 还是 Recomputation 应该考虑哪些因素？

### 4.4 分布式执行 (Distributed Execution)

许多大型语言模型的参数量巨大, 常常超过单个 GPU 的内存容量. 因此, 在实际部署中, 通常需要将模型参数分散到多个 GPU 上, 并以**模型并行 (Model Parallelism)** 的方式执行. 这就要求内存管理器能够处理分布式环境下的 KV Cache.

- **模型并行与张量并行 (Model Parallelism and Tensor Parallelism)** :

- **模型并行**: 将模型层或层内的算子分布到多个设备上.

- **张量并行 (Tensor Parallelism)** : 是一种常见的模型并行策略, 例如 Megatron-LM 风格的张量并行. 它将模型的线性层(如 Attention 机制中的 $W_q, W_k, W_v$ 矩阵)沿某个维度进行分割, 分配给不同的 GPU. 每个 GPU 负责计算模型的一部分, 并通过 All-Reduce 等通信原语同步中间结果. 在 Attention 机制中, 通常在 Attention Head 的维度上进行分割.

- **PagedAttention 在分布式环境中的运作**:

- **中央调度器**: vLLM 采用一个**中央调度器 (Central Scheduler)** 来协调分布式 GPU Worker 的执行. 这个调度器负责计算和管理所有 GPU 上 KV Cache 的逻辑块到物理块的映射表 (Block Tables).

- **共享映射**: 尽管模型分布在多个 GPU 上, 但每个 GPU Worker 仍然处理相同的输入 Token 集合, 并需要访问相同位置的 KV Cache. 因此, vLLM 的中央调度器管理一个**统一的 Block Table**, 并在每次迭代开始时将其广播给所有 GPU Worker.

- **部分 KV Cache 存储**: 即使所有 GPU Worker 共享相同的物理块 ID, 每个 Worker 也只存储其负责的 Attention Head 部分的 KV Cache. 例如, 如果 Attention Head 分割为 8 份, 每张卡只存储 1/8 的 KV Cache.

- **减少内存管理同步开销**: 这种设计使得 GPU Worker 之间在内存管理层面无需频繁同步. 它们只需要在每个解码迭代开始时从调度器接收完整的内存管理信息(包括输入 Token ID 和更新后的 Block Table), 即可独立执行计算. 这大大降低了分布式推理中的通信开销, 提高了效率.

## 5. vLLM 核心实现：自动前缀缓存 (Automatic Prefix Caching in vLLM)

除了 PagedAttention 带来的 KV Cache 内存高效利用, vLLM 还进一步引入了**自动前缀缓存 (Automatic Prefix Caching)** 功能, 其核心思想是 **RadixAttention**, 旨在最大化 KV Cache 的复用, 尤其是在具有相同长 System Prompt 或多轮对话的场景中, 显著降低首 Token 时延.

### 5.1 背景与目标 (Background and Goal)

在 LLM 推理应用中, 存在以下常见场景：

- **长 System Prompt**: 不同的请求可能使用相同的长 System Prompt.

- **多轮对话**: 每一轮对话都需要依赖所有历史轮次的上下文, 即历史对话的 KV Cache 在后续每一轮中都需要被重新计算.
如果能够将这些共享或历史的 KV Cache 保存下来, 并供后续请求复用, 将极大地降低首 Token 的计算耗时 (Time-To-First-Token, TTFT), 提升用户体验. 这就是自动前缀缓存的目标.

### 5.2 RadixAttention 原理 (Principles of RadixAttention)

RadixAttention 最初在 SGLang 论文《SGLang: Efficient Execution of Structured Language Model Programs》中被提出, 用于实现 **Automatic KV Cache Reuse**.

- **概念**: RadixAttention 使用 **基数树 (Radix Tree)** 而不是传统的前缀树 (Prefix Tree) 来管理 KV Cache 的复用. 基数树的一个显著特点是其节点不仅可以代表单个元素, 还可以代表一个**变长的序列**.

- **动态分裂**: 在必要时, 基数树中的一个大节点可以动态地分裂成小节点, 以适应动态的共享前缀需求.

- **缓存范围**: RadixAttention 会缓存**Prefill 阶段**产生的 KV Cache (即 Prompt 部分) 和 **Generated 阶段**产生的 KV Cache (即模型生成的部分). 这种全面的缓存机制最大程度地增加了 KV Cache 被新请求复用的几率. 在多轮对话场景中, 这意味着可以基本消除历史轮次中生成对话的重新计算开销.

### 5.3 vLLM 的 Hash RadixAttention 实现 (vLLM's Hash RadixAttention Implementation)

vLLM 实现了 RadixAttention 算法, 但巧妙地使用 **Hash 码**作为物理 KV Block 的唯一标识, 从而简化了工程实现, 我们称之为 **Hash RadixAttention**.

- **Hash 码作为唯一标识**: 在 vLLM 中, 通过 Prompt 中的 token_ids 来计算 Hash 值作为 Cache Block 的唯一标识. 即使不同请求的逻辑索引 logical_idx 可能重复, 但实际用于生成 Hash 码的 token_ids 是不同的, 确保了不同 Prompt 的 Cache Block 能够获得唯一的 Hash 码.

- **Hash Prefix Tree (哈希前缀树)** : 这种基于 Hash 码的实现方式, 实际上具备了**前缀树 (Prefix Tree)** 的功能. 由于每个 Block 的 Hash 码都**依赖于此前所有 Block 的 **token_ids, 因此这个唯一的 Hash 码实际上也**代表着一种唯一的前缀关系**. 这确保了从 Prefix Cache KV Blocks 中取出的一系列 Block 具有**相同上下文**.

### 5.4 Prefix/Generate 阶段 Hash 码处理 (Hash Code Handling in Prefix/Generate Stages)

在 PagedAttention 中, Hash 码的处理方式针对 Prefill 和 Generate 阶段有所不同.

- **Prefill 阶段**: 所有的 token_ids 在推理开始前就已已知, 因此可以直接通过 token_ids 获取对应的 Block Hash 码.

- **Generate 阶段**: 由于无法事先知道 Generate 阶段的 token_ids (它们尚未生成), 但又必须先分配 Block 来保存生成的 KV Cache. vLLM 的处理方式是：

- **分配 Fake Hash (伪哈希)** : 对于 Generate 阶段所需的 Block, vLLM 会先为其分配一个临时性的 fake hash.

- **更新 Hash 码**: 在进行 Token Generation 后, 当该 Block 被生成的 Token 填满时, vLLM 会根据实际生成的 token_ids 来**更新**其真实的 Hash 码. 这个更新过程通过调用 _promote_last_block 函数, 重新计算 new_hash 并使用 gpu_allocator.update_hash 更新 last block 的 Hash 码.

### 5.5 Prefix + Generated KV Caching 调度逻辑 (CachedBlockAllocator)

vLLM 的 Prefix Caching 功能不仅缓存了 Prefill 阶段的 KV Cache, 还缓存了 Generated 阶段的 KV Cache, 这在多轮对话场景中具有更大的首 Token 时延优势. 这主要通过 CachedBlockAllocator 类实现, 其核心机制包括：

- **通用缓存功能**: CachedBlockAllocator 实现的是通用的缓存功能, **不区分 KV Cache 是来自 Prefill 还是 Generate 阶段**. 只要产生了 KV Cache, 它就会被放入 cached_blocks 表中进行缓存, 其中 Key 是 block_hash, Value 是 block_id.- ref_count** (引用计数)** :

- 这是 CachedBlockAllocator 中的一个重要概念, 表示 Block 被引用的次数.- 当 ref_count >= 1 时, Block 会被保留在 cached_blocks 中, 表示它正在被某个或多个活跃请求使用.- 当 ref_count 变为 0 时, 表示没有活跃请求再引用该 Block. 此时, Block 会从 cached_blocks 中移除, 并被添加到 LRUEvictor 的 free_table 中, 等待被真正逐出或被新的请求复用. **需要注意的是, **ref_count = 0** 并不代表立即释放显存, 只有当显存不足时, LRUEvictor 才会真正执行逐出操作.**

- LRUEvictor** (最近最少使用逐出器)** :

- LRUEvictor 管理着 free_table, 其中包含了 ref_count 为 0 的 Block.- add(): 将 ref_count 为 0 的 Block 添加到 free_table 中. 这些 Block 的 computed 标志仍为 True, 表示它们已计算并可能被复用.- remove(): 当有新请求的 Hash 码命中 free_table 中的 Block 时, 该 Block 会被取出复用, 其 ref_count 增加 1.- evict(): 当显存不足时, LRUEvictor 会执行真正的逐出功能. 它会从 free_table 中选择最长时间未被访问的 Block (LRU 策略, 通过 OrderedDict 实现) 进行逐出. 如果存在多个符合条件的 Block, 会优先逐出 num_hashed_tokens 数量最大的 Block, 因为这表示其被复用的可能性较小.

- allocate()** 分支分析**:

- **命中 Generated KV Cache (在 **evictor** 中)** : 如果 Hash 码命中了 LRUEvictor 的 free_table, 说明该 Block 曾经被生成过, 并且目前未被活跃请求引用. 这通常发生在 Generated KV Cache 上. Block 会被取出复用, ref_count 增加 1.

- **未命中任何 Cache**: 如果 Hash 码既不在 cached_blocks 也不在 evictor 中, 则直接分配新的 Block, 并将其 ref_count 设置为 1.

- **命中 Prefix KV Cache (在 **cached_blocks** 中)** : 如果 Hash 码命中了 cached_blocks, 说明该 Block 正在被活跃请求引用(或之前被 Prefill 过). Block 会被获取, ref_count 增加 1. 具有相同 System Prompt 的 Prefix KV Cache 会因为 ref_count >= 1 而长时间驻留在 cached_blocks 中, 从而实现高效复用.

### 5.6 边界情况思考 (Edge Cases)

在 PagedAttention 的实际实现中, 有一些边界情况可能导致 KV Cache 无法被命中复用：

1. **Last Block 未被用满**:

- vLLM 对 last block(序列的最后一个 Block)采用了 **Copy-on-Write (CoW)** 策略. 当 last block 中的槽位未被完全用满时, 它不会被不同的序列共享.- 如果 _is_last_block_full 结果为 False, 该 last block 就不会通过 _promote_last_block 函数根据实际生成的 token_ids 更新其 Hash 码, 而是保持其 fake hash. 新请求的 Hash 码是基于实际 token_ids 计算的, 因此与 fake hash 不同, 导致该 last block 无法被后续请求作为 Cache 命中复用.

1. **请求的 Token 数小于 **block_size:

- 当一个请求的 Prefill 阶段或整个序列 (Prompt + Generated) 的 Token 数小于 block_size 时, 其 last block 也是 first block, 因此也符合上述 (1) 的情况. 这意味着即使是相同的短请求在第二次到达 vLLM 后, 也可能无法命中 Cache.

### 5.7 多轮对话中的应用分析 (Application in Multi-Turn Conversations)

自动前缀缓存, 尤其是 vLLM 实现的包含 Prefix 和 Generated KV Cache 的版本, 在多轮对话场景中具有极其重要的应用价值.

- **挑战**: 随着对话轮次的增加, 每次新轮次的对话 Prompt 会越来越长. 如果没有 KV Cache 复用, 每次都需要重新计算所有历史上下文的 KV Cache, 导致首 Token 时延随对话轮次线性增长.

- **仅 Prefix Caching 的优化**: 如果只缓存 Prompt 部分 (Only Prefix KV Caching), 那么在每轮对话中, 上一轮的生成结果 (Generated Tokens) 仍然需要在本轮 Prefill 阶段被重新计算. 虽然这比完全不缓存要好, 但仍然存在显著的 Recompute 开销.

- **Prefix + Generated KV Caching 的优势**: vLLM 的实现同时缓存了 Prefix 和 Generated KV Cache. 在忽略边界情况的前提下, 这基本可以消除历史轮次中生成对话的重新计算. 每轮对话中, 只有当前轮次的 Prompt 部分需要在 Prefill 阶段进行计算, 历史轮次的 Prefix + Generated KV Cache 都将通过缓存命中复用.

图 11: Prefix + Generated KV Caching 在多轮对话中消除历史上下文的重计算.

- **实际应用考量**: 为了充分利用 vLLM 的自动前缀缓存特性, 需要考虑以下几点：

- **负载均衡**: 尽量保证每个模型服务实例的负载均衡, 避免 QPS 过高导致频繁逐出 Cache.

- **会话路由**: 同一个对话 Session 的不同轮次请求应**尽量路由到同一个模型服务实例**. 因为实例之间的 KV Cache 是不共享的, 只有在相同的实例上才能实现历史缓存的复用.

- **System Prompt 复用**: 具有相同或相似长 System Prompt 的请求, 应尽可能发送到同一个模型服务实例进行推理, 以最大化共享前缀的 Cache 命中率.

### 5.8 Prefix Prefill Kernel 与 Attention Kernel 区别 (Prefix Prefill Kernel vs. Attention Kernel)

在启用 Prefix Caching 后, 传统的 Attention Kernel 不再适用 Prefill 阶段的计算.

- **传统 Attention Kernel 假设**: 常规 Attention Kernel 通常假定 Query 序列长度 ($Q_{len}$) 等于 Key-Value 序列长度 ($KV_{len}$), 且两者都等于 Prompt 长度 ($Prompt_{len}$).

- **Prefix Caching 下的变化**: 在 Prefix Caching 的背景下, 当前请求的 Prompt 中会有部分 KV Cache 被命中, 无需重复计算. 这意味着, 只有未命中部分的 Query 参与计算, 导致 $Q_{len} < Prompt_{len}$. 然而, 每个 Query 仍然需要与所有历史上下文的 KV 值进行 Attention 计算, 因此 $KV_{len}$ 依然等于 $Prompt_{len}$. 简而言之, 此时出现了 $Q_{len} < KV_{len} = Prompt_{len}$ 的情况.

- **专用 Kernel**: 为了高效处理 Q_len < KV_len 的情况, vLLM 引入了专门的 **Prefix Prefill Kernel**. 该 Kernel 通常基于 Triton 实现, 针对这种特殊的 Attention 计算模式进行优化.

## 6. vLLM 应用实践与性能表现 (vLLM Application and Performance)

vLLM 作为 PagedAttention 的主要实现者, 其易用性和卓越性能使其成为 LLM 推理服务领域的佼佼者.

### 6.1 如何开启 PagedAttention/Prefix Caching (How to Enable)

在 vLLM 中开启 PagedAttention (默认已开启, 因为它是核心机制) 和 Automatic Prefix Caching 非常简单:

- **离线推理 (Offline Inference)** : 在实例化 LLM 对象时设置 enable_prefix_caching=True.

- **在线服务化 (Online Serving)** : 在启动 vLLM API 服务器时添加 --enable-prefix-caching 参数.

### 6.2 性能评估 (Performance Evaluation)

vLLM 在各项性能指标上均展现出显著优势, 尤其是在吞吐量和内存利用率方面.

- **吞吐量提升 (Throughput Improvement)** :

- vLLM 相比于传统系统 (如 FasterTransformer) 和其他先进系统 (如 Orca) 能够实现 **2 到 4 倍** 的吞吐量提升. 这种提升对于更长的序列、更大的模型和更复杂的解码算法尤为明显.- 在实际工作负载下, vLLM 可以维持更高的请求速率, 且保持更低的归一化延迟, 尤其是在请求速率超过服务系统容量的临界点之后, vLLM 的优势更加突出.

- **内存利用率 (Memory Utilization)** :

- 传统系统由于内存碎片和过度预留, KV Cache 的有效利用率可能低至 20%-40%. PagedAttention 将这一效率提升至接近 **96%**, 实现了**近乎零浪费**的 KV Cache 存储.- 在 GPU 内存分布中, 模型参数占据固定部分, 而 KV Cache 是动态变化且占据大量空间的. PagedAttention 能够平滑 KV Cache 的增长曲线, 使得相同内存大小下可以支持更大的批处理规模.

- **块大小 (Block Size) 的影响**:

- 块大小的选择是 PagedAttention 性能的关键因素.

- **块太小**: 可能导致 GPU 并行读取和处理 KV Cache 的利用率不足.

- **块太大**: 会增加内部碎片化, 降低内存共享的概率.

- **vLLM 默认值**: vLLM 默认块大小设置为 16. 实践证明, 这个值足够大以有效利用 GPU 并行性, 同时又足够小以避免显著的内部碎片化, 适用于大多数工作负载.

- **重计算与交换的性能对比**:

- **Swapping (交换)** : 当块大小较小时, 交换会带来过多的 PCIe 带宽开销, 因为涉及大量小数据传输.

- **Recomputation (重计算)** : 开销在不同块大小下相对稳定, 不使用 KV 块传输. 因此, 对于小块大小, 重计算效率更高.- 对于中等大小的块 (16 到 64), 两种方法的整体性能相当.

### 6.3 PagedAttention 内核开销讨论 (Discussion on PagedAttention Kernel Overhead)

虽然 PagedAttention 带来了巨大的吞吐量提升, 但其动态块映射也引入了一些额外的内核开销. 例如, 与高度优化的 FasterTransformer 实现相比, PagedAttention 的注意力内核可能表现出 20%-26% 的延迟增加. 这是因为需要额外的操作, 如访问块表、执行额外的分支以及处理可变长度序列. 然而, 这种开销通常是可接受的, 因为它主要影响注意力模型部分, 而不是模型中的其他算子(如线性算子), 且 PagedAttention 在端到端性能上仍显著优于传统方法.

## 7. 相关工作与其他优化 (Related Works and Other Optimizations)

PagedAttention 并非唯一致力于优化 LLM 推理内存管理和吞吐量的技术. 许多其他研究和系统也提出了不同的思路.

- **其他 Prefix Caching 方案 (Other Prefix Caching Schemes)** :

- **Prompt Cache**: 专注于缓存 Prompt 本身的 KV Cache.

- **Shared Prefixes**: 强调在多个请求之间共享共同的前缀.

- **ChunkAttention**: 一种基于分块的注意力机制, 也关注 Prefix 优化.
这些方案通常只聚焦于 Prefix 本身, 即属于“Only Prefix Caching”的情形, 其通用性和复用范围可能不及 vLLM 的 Hash RadixAttention 实现, 后者包含了 Prefill 和 Generated KV Cache 的缓存.

- **vAttention (Virtual Attention)** :

- **核心思路**: vAttention 提出了一种在连续虚拟内存中存储 KV Cache 而无需提前提交物理内存的系统. 它旨在在 PagedAttention 的思想基础上, 通过利用底层 CUDA 虚拟内存 API 实现更细粒度的物理内存分配, 并保持 KV Cache 在逻辑上的连续性, 从而可以直接使用 FlashAttention 等无需修改的 GPU Kernel.

- **创新点**:

- **细粒度物理内存分配**: 修改开源 CUDA 统一虚拟内存驱动程序, 支持 64KB 到 256KB 更细粒度的物理内存分配, 解决 PageAttention 中 2MB 最小物理内存分配粒度可能导致的容量浪费.

- **隐藏内存分配延迟**: 将内存分配与计算重叠, 提前执行操作, 并延迟内存回收, 以隐藏 CUDA API 调用带来的高延迟.

- **PyTorch 接口扩展**: 扩展 PyTorch 缓存分配器, 允许为张量保留虚拟内存缓冲区, 而无需提前提交物理内存.

- **优势**: vAttention 宣称可以无需修改 Attention Kernel 代码, 无缝接入 FlashAttention 等, 性能比 vLLM 高出 1.97 倍 (在特定测试下). 它实现了 PagedAttention 的内存碎片化和动态化管理功能, 同时保留了连续内存访问的高效率.
