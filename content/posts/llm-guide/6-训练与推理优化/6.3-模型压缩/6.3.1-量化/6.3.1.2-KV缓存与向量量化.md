# 6.3.1.2 KV缓存与向量量化

## 1. 演进脉络与技术演进

KV Cache 压缩的历史比权重量化更短，因为直到 2023 年长上下文推理(32K 以上)成为刚需，KV Cache 的显存占用才超过模型权重本身，成为推理部署的首要瓶颈. 在此之前，工程界将权重量化的成熟方案直接套用到 KV Cache 上，发现精度迅速崩塌——因为 KV Cache 不是训练后固定的权重，而是随序列动态增长的激活值，其分布特性与权重截然不同. 

**第一代(2023 年初)：对称逐张量量化**. 直接将 LLM.int8() 的 absmax 对称量化应用于 KV Cache，在整个序列的所有 token 上共享单一缩放因子. 这种方法在 INT8 下对短序列(< 2048)尚可接受，但在长序列下，Key 的通道级异常值会迫使缩放因子过大，导致绝大多数正常值被压缩到 1-2 个有效位，注意力分数严重失真. 

**第二代(2024 年)：非对称分层量化(KIVI)** . Liu et al. 的 KIVI 发现 Key 和 Value 具有完全不同的分布特性：Key 存在显著的通道级异常值(某些 head_dim 维度的值比其他维度大 10-100 倍)，而 Value 的分布相对均匀但在不同 token 间差异大. KIVI 提出 Key 使用 per-channel(逐通道)非对称量化、Value 使用 per-token(逐 token)非对称量化的混合策略，首次在不微调模型的情况下将 KV Cache 压缩到 2-4 bit 并保持可接受的精度. 

**第三代(2024-2025 年)：投影与向量量化**. 标量量化的精度天花板来自缩放因子的存储开销——当分组很细时，缩放因子本身可能占据 1-2 bit 的等效带宽. QJL(Quantized Johnson-Lindenstrauss)通过随机投影将 Key 向量压缩为 1-bit 符号向量，完全消除了缩放因子的存储开销. 随后 RaBitQ 和 TurboQuant 将随机旋转与最优标量量化器结合，将向量量化推向理论最优界. RaBitQ/TurboQuant 的完整数学推导见 [6.3.1.3 极低比特向量压缩](./6.3.1.3-极低比特向量压缩.md)，本节聚焦于从 absmax 到 QJL 的工程实践演进. 

## 2. 工业价值与应用场景

KV Cache 是长序列推理中最坚硬的显存瓶颈. 以 Llama-3-8B 为例，在 batch=32、seq_len=4096 的 FP16 推理下，KV Cache 显存占用与模型权重本身相当. 

| 组件 | 计算式 | FP16 显存占用 |
|------|--------|--------------|
| 模型权重 | $8\text{B} \times 2\text{ bytes}$ | 16.0 GB |
| Key Cache | $32 \times 8 \times 128 \times 4096 \times 32 \times 2$ | 8.0 GB |
| Value Cache | $32 \times 8 \times 128 \times 4096 \times 32 \times 2$ | 8.0 GB |
| KV Cache 总计 | — | 16.0 GB |
| 激活值与开销 | — | ~4-8 GB |
| **总计** | — | **~36-40 GB** |

单张 A100 40GB 在该配置下已无法容纳; 即使 A100 80GB，留给更大 batch 或更长序列的空间也极为有限. 将 KV Cache 从 FP16 压缩到 INT4，显存从 16 GB 降至 4 GB，意味着 batch size 可从 32 提升到 128，吞吐量提升 4 倍; 或者在单张 RTX 4090(24 GB)上运行 batch=32、seq=4096 的推理成为可能. 

工业界的落地节奏清晰：vLLM 在 0.4.0 版本集成 FP8 KV Cache; llama.cpp 的 GGUF 格式支持 Q4_K_M 和 Q5_K_M 的 KV 量化; Qwen3 的官方实现采用 Key INT2 + Value INT4 的混合策略，在 128K 上下文下将 KV Cache 从 48 GB 压缩到约 6 GB. 

## 3. 物理直觉与核心机制

### 3.1 为什么 KV Cache 比权重量化更难

权重量化面对的是训练后固定的参数矩阵. 量化算法可以用 128-1024 个样本的校准集反复试探最优缩放因子，甚至使用 OBS 补偿机制调整未量化的权重来抵消误差. KV Cache 则完全不同：它在推理过程中逐 token 在线生成，量化必须在生成的同时完成，没有"校准集"可言. 

更深层的困难来自激活值的分布特性. 权重矩阵经过训练后通常近似高斯分布，均值为 0，方差稳定; 而 KV Cache 是 LayerNorm/RMSNorm 的输出再经过投影的结果，继承了激活值的三个顽固特征：

1. **通道级异常值(Channel-wise Outliers)** ：Key 的某些 head_dim 维度存在幅度异常大的值，通常是其他维度的 10-100 倍. 这些异常值如同人群中的"巨人"——如果量化方案不为它们单独分配空间，整个动态范围会被拉宽，绝大多数正常值的精度被压缩到 1-2 位. 

2. **时序分布漂移**：早期 token(如 BOS)的 KV 向量与后期 token 的分布可能显著不同. sink token 现象表明，模型对某些初始 token 赋予极高的注意力权重，这些 token 的 Key 向量在特定通道上形成异常聚类. 

3. **Key 与 Value 的不对称性**：Key 在 attention 中参与指数运算($\exp(QK^T/\sqrt{d})$)，微小的量化误差会被 softmax 的指数函数非线性放大; Value 在 attention 后做加权平均，误差会被 softmax 权重线性传播. 这决定了 Key 对量化精度更敏感，需要更细的粒度或更高的位宽. 

### 3.2 量化粒度的选择逻辑

KV Cache 的量化粒度选择是一个精密的工程权衡. per-tensor(整层共享)最简单但精度最差; per-channel(每个 head_dim 独立)能捕捉 Key 的通道异常值，但每个通道需要独立存储缩放因子; per-token(每个 token 独立)适合 Value 的 token 间差异，但缩放因子数量与序列长度成正比; per-group(分组)在两者之间折中，但分组大小的选择直接影响存储-精度 trade-off. 

KIVI 的核心洞察是：**不需要对 Key 和 Value 使用相同的量化策略**. Key 的异常值是"结构性"的(固定在某些通道)，用 per-channel 解决; Value 的差异是"时序性"的(不同 token 不同)，用 per-token 解决. 这种非对称设计将两种分布特性解耦处理，避免了单一策略的互相拖累. 

## 4. 数学推导与公式对比

### 4.1 KV Cache 显存计算

KV Cache 的总显存由层数、KV 头数、头维度、序列长度、batch size 和精度共同决定：

$$
M_{\text{KV}} = 2 \cdot n_{\text{layers}} \cdot n_{\text{kv\_heads}} \cdot d_{\text{head}} \cdot s \cdot B \cdot \text{bytes} \tag{1}
$$
其中 $n_{\text{layers}}$ 为模型层数(Llama-3-8B 为 32); $n_{\text{kv\_heads}}$ 为 GQA 中的 KV 头数(Llama-3-8B 为 8，注意力头数为 32); $d_{\text{head}}$ 为每个头的维度(Llama-3-8B 为 128). $s$ 为序列长度(已生成 token 数)，$B$ 为 batch size; $\text{bytes}$ 为每个元素的存储字节数(FP16=2，INT8=1，INT4=0.5). 系数 2 表示 Key Cache 与 Value Cache 各占一份. 

$n_{\text{layers}} \cdot n_{\text{kv\_heads}} \cdot d_{\text{head}}$ 为单层单 token 的 KV 向量维度，Llama-3-8B 下为 $32 \times 8 \times 128 = 32768$ 个元素; $s \cdot B$ 为推理过程中需要缓存的总 token 数. 整体乘积给出所有层、所有 token 的 KV 元素总数，再乘以 2 和 $\text{bytes}$ 得到总字节数. 如式 (1) 所示，KV Cache 显存与序列长度和 batch size 线性增长，这是长上下文推理的根本瓶颈. 

### 4.2 Absmax 对称量化

对称量化假设数据关于 0 对称分布，缩放因子由绝对值最大值决定：

$$
s = \frac{\max(|\mathbf{x}|)}{2^{b-1} - 1} \tag{2}
$$
$$
x_q = \text{round}\left(\text{clamp}\left(\frac{x}{s}, -(2^{b-1}), 2^{b-1}-1\right)\right) \tag{3}
$$
其中 $\mathbf{x}$ 为待量化的 KV 向量或矩阵; $s$ 为对称缩放因子(scale). $b$ 为目标位宽，如 INT4 时 $b=4$; $2^{b-1}-1$ 为 $b$ 位有符号整数的最大正值(INT4 为 7，INT8 为 127). $\text{clamp}(v, a, b)$ 将 $v$ 截断到 $[a, b]$ 区间. 

式 (2) 用数据绝对值最大值除以最大整数值，将浮点动态范围线性映射到整数域，零点固定为 0，计算时无需零点偏移. 式 (3) 先将浮点值按比例缩放到整数域，再四舍五入离散化，最后截断防止溢出; $\text{round}$ 是信息损失的唯一来源. 反量化重建为 $\hat{x} = x_q \cdot s$，重建误差范围为 $[-s/2, s/2]$. 

对称量化的优势是计算简单，反量化时无需零点偏移; 劣势是当数据分布明显不对称时(如 ReLU 输出始终非负)，负值范围的量化级别被浪费. 

### 4.3 非对称量化(Min-Max)

非对称量化引入零点偏移，充分利用全部量化级别：

$$
s = \frac{\max(\mathbf{x}) - \min(\mathbf{x})}{2^b - 1}, \quad z = \text{round}\left(\frac{\min(\mathbf{x})}{s}\right) \tag{4}
$$
$$
x_q = \text{round}\left(\frac{x - z}{s}\right) \tag{5}
$$
其中 $\min(\mathbf{x})$ 和 $\max(\mathbf{x})$ 分别为数据的最小值和最大值; $z$ 为零点偏移(zero-point)，将数据最小值映射到整数 0. $2^b - 1$ 为 $b$ 位无符号整数的最大值(INT4 为 15). 

式 (4) 的缩放因子将数据的实际动态范围 $[\min, \max]$ 线性映射到 $[0, 2^b-1]$，相比对称量化多利用了 $2^{b-1}$ 个额外量化级别; 零点偏移确保数据的最小值恰好映射到整数 0，消除对称量化中对负值范围的浪费. 式 (5) 先减去零点偏移，再按缩放因子量化，适用于分布明显偏向正或负的数据; 反量化为 $\hat{x} = (x_q + z) \cdot s$. 

### 4.4 KIVI 的分层非对称量化

KIVI 针对 Key 和 Value 的不同分布特性采用不同的量化粒度. 

**Key 的 Per-Channel 量化**：对每个 head_dim 通道，跨所有 token 计算独立的缩放因子和零点. 

$$
s_j = \frac{\max_{i}(\mathbf{K}_{i,j}) - \min_{i}(\mathbf{K}_{i,j})}{2^{b_k} - 1}, \quad z_j = \text{round}\left(\frac{\min_{i}(\mathbf{K}_{i,j})}{s_j}\right) \tag{6}
$$
其中 $\mathbf{K} \in \mathbb{R}^{s \times d_{\text{head}}}$ 为单 head、单 batch 元素的 Key 矩阵; $j$ 为通道(channel)索引，取值范围 $1 \dots d_{\text{head}}$; $i$ 为 token 索引，取值范围 $1 \dots s$. $b_k$ 为 Key 的目标位宽，KIVI 使用 $b_k = 2$ 或 $4$; $s_j, z_j$ 为第 $j$ 个通道的独立缩放因子和零点偏移. 

$\max_{i}(\mathbf{K}_{i,j})$ 为第 $j$ 个通道在所有 token 上的最大值; Key 的异常值具有通道结构性，同一通道在所有 token 上同步异常. 跨 token 计算 min/max 确保该通道的动态范围被精确捕捉，不因其他通道的异常值而被压制. 每个通道存储一个 $s_j$ 和一个 $z_j$(各 2 bytes，FP16)，对于 $d_{\text{head}}=128$ 仅增加 512 bytes，相对于 Key 矩阵本身可忽略. 

**Value 的 Per-Token 量化**：对每个 token，在其 head_dim 维度上计算独立的缩放因子和零点. 

$$
s_i = \frac{\max_{j}(\mathbf{V}_{i,j}) - \min_{j}(\mathbf{V}_{i,j})}{2^{b_v} - 1}, \quad z_i = \text{round}\left(\frac{\min_{j}(\mathbf{V}_{i,j})}{s_i}\right) \tag{7}
$$
其中 $\mathbf{V} \in \mathbb{R}^{s \times d_{\text{head}}}$ 为单 head、单 batch 元素的 Value 矩阵; $i$ 为 token 索引，$j$ 为通道索引. $b_v$ 为 Value 的目标位宽，KIVI 使用 $b_v = 4$; $s_i, z_i$ 为第 $i$ 个 token 的独立缩放因子和零点偏移. 

$\max_{j}(\mathbf{V}_{i,j})$ 为第 $i$ 个 token 在所有通道上的最大值; Value 的分布差异主要体现为 token 间的差异，同一 token 的各通道值域相近. Per-token 量化让每个 token 的 Value 向量独立映射到满量程，避免了长序列中早期 token 与后期 token 的动态范围冲突. 存储开销与序列长度成正比：$s$ 个 token 需要 $s$ 组 $(s_i, z_i)$; 当 $s=4096$ 时，额外存储约 $4096 \times 2 \times 2 = 16$ KB per head，相对于 Value 矩阵的 1 MB(INT4)增加约 1.6%. 

### 4.5 量化后的 Attention 计算

量化 KV Cache 后，注意力计算在反量化后的近似值上进行：

$$
\text{Attention}(Q, \hat{K}, \hat{V}) = \text{softmax}\left(\frac{Q \hat{K}^T}{\sqrt{d_{\text{head}}}}\right) \hat{V} \tag{8}
$$
其中 $Q \in \mathbb{R}^{s_q \times d_{\text{head}}}$ 为当前步的 Query 矩阵; $\hat{K}, \hat{V}$ 为反量化后的 Key 和 Value 矩阵. $s_q$ 为当前生成的 token 数(解码步长通常为 1). 

$Q \hat{K}^T$ 使用反量化后的 Key 计算近似注意力分数; 量化误差 $\Delta K = K - \hat{K}$ 会线性传播到分数中，即 $Q \hat{K}^T = QK^T - Q\Delta K^T$. $\text{softmax}$ 将分数差异指数化; 如式 (8) 所示，Key 的量化误差在 softmax 中被指数放大，这是 Key 需要更精细量化的数学根源. $\hat{V}$ 的量化误差在加权平均中线性传播，对最终输出的影响相对温和. 

### 4.6 QJL：消除缩放因子开销的投影量化

标量量化(无论对称或非对称)都需要存储缩放因子和零点. 当分组粒度很细(如 per-token 或 per-channel)时，这些元数据的存储开销可能达到 1-2 bit/元素的等效带宽. QJL 通过 Johnson-Lindenstrauss 随机投影完全消除这一开销. 

对 Key 向量 $\mathbf{k} \in \mathbb{R}^d$，QJL 变换为：

$$
\mathcal{H}_S(\mathbf{k}) = \text{sign}(\mathbf{S}\mathbf{k}) \tag{9}
$$
其中 $\mathbf{S} \in \mathbb{R}^{m \times d}$ 为随机高斯投影矩阵，元素 i.i.d. 采样自 $\mathcal{N}(0, 1)$; $\text{sign}(\cdot)$ 为符号函数，输出 $\{-1, +1\}^m$. $m$ 为投影维度，通常 $m = O(\varepsilon^{-2} \log n)$，$n$ 为序列长度; $\mathcal{H}_S(\mathbf{k})$ 为量化后的 1-bit 签名向量. 

$\mathbf{S}\mathbf{k}$ 将 $d$ 维 Key 向量投影到 $m$ 维随机子空间; JL 引理保证投影后的内积以高概率保持原始内积的相对关系. $\text{sign}(\cdot)$ 将每个投影坐标量化为 1-bit，仅保留方向信息; 这是信息损失最剧烈的量化，但 JL 引理保证方向信息足以估计内积. 如式 (9) 所示，QJL 不需要计算或存储任何缩放因子或零点——投影矩阵 $\mathbf{S}$ 在推理前随机生成一次，全局共享. 

查询向量 $\mathbf{q}$ 同样投影(但不取符号)，内积估计为：

$$
\langle \mathbf{q}, \mathbf{k} \rangle \approx \frac{\sqrt{\pi/2}}{m} \|\mathbf{k}\|_2 \cdot \langle \mathbf{S}\mathbf{q}, \mathcal{H}_S(\mathbf{k}) \rangle \tag{10}
$$
其中 $\|\mathbf{k}\|_2$ 为原始 Key 向量的 $L_2$ 范数，每个 token 存储一个 float32(4 bytes)，相对于 Key 向量的 $d \times 2$ bytes 可忽略. $\langle \mathbf{S}\mathbf{q}, \mathcal{H}_S(\mathbf{k}) \rangle$ 为投影后查询向量与 1-bit 签名向量的内积，可用位运算加速; $\sqrt{\pi/2}/m$ 为归一化系数，由标准高斯分布的绝对值期望推导而来，确保估计器无偏. 

$\|\mathbf{k}\|_2$ 补偿了 sign 量化丢失的幅度信息; 仅存储范数(而非完整向量)就将位宽从 $d \times 16$ bit 降至 $m \times 1 + 32$ bit. $\langle \mathbf{S}\mathbf{q}, \mathcal{H}_S(\mathbf{k}) \rangle$ 可通过 SIMD popcount 指令高效计算，将浮点内积转化为二进制向量的汉明距离计算. 如式 (10) 所示，QJL 的总存储约为 $m$ bit + 32 bit 每 token; 当 $d=128, m=128$ 时，等效位宽约 3 bit(含范数)，实现了 5 倍以上的压缩比. 

QJL 的局限性在于它只压缩 Key，Value 仍需单独量化. 在后续工作中，TurboQuant 将 QJL 与 Lloyd-Max 最优量化器结合，同时对 K 和 V 做向量量化，将压缩比推向 8-16 倍. TurboQuant 和 RaBitQ 的完整数学框架(随机旋转、Beta 分布、Lloyd-Max 迭代)在 [6.3.1.3 极低比特向量压缩](./6.3.1.3-极低比特向量压缩.md) 中详述. 

## 5. 算力成本与数值走查

以 Llama-3-8B 的真实配置做端到端数值走查：$n_{\text{layers}}=32$，$n_{\text{kv\_heads}}=8$，$d_{\text{head}}=128$，batch=32，seq=4096. 

### 5.1 显存占用对比

如式 (1) 所示，FP16 KV Cache 总显存为：

$$
M_{\text{KV}} = 2 \times 32 \times 8 \times 128 \times 4096 \times 32 \times 2 = 17{,}179{,}869{,}184 \text{ bytes} = 16.0 \text{ GB} \tag{11}
$$

该计算以 Llama-2-70B 的典型配置为例，展示了 FP16 精度下 KV Cache 占用的显存规模，为后续量化压缩方案提供了基准参照. 

| 方案 | Key 位宽/粒度 | Value 位宽/粒度 | KV Cache 显存 | 相对 FP16 |
|------|--------------|----------------|--------------|----------|
| FP16 基准 | 16 / — | 16 / — | 16.0 GB | 1.00× |
| Absmax INT8 | 8 / per-tensor | 8 / per-tensor | 8.0 GB | 0.50× |
| KIVI (4+4) | 4 / per-channel | 4 / per-token | 4.0 GB | 0.25× |
| KIVI-v2 (2+4) | 2 / per-channel | 4 / per-token | 3.0 GB | 0.19× |
| QJL | ~3 / JL投影 | 4 / per-token | ~2.5 GB | 0.16× |
| 向量量化 (K+V) | 4 / vector | 4 / vector | 4.0 GB | 0.25× |

上表中的"向量量化 (K+V)"指 RaBitQ/TurboQuant 类方法对 Key 和 Value 同时做 4-bit 向量量化. 注意 KIVI-v2 的 Key 2-bit 方案在部分模型上会导致 perplexity 上升 5-10%，需要配合分组缩放(group-wise per-channel)来缓解. 

### 5.2 带宽与延迟收益

解码阶段的注意力计算受内存带宽限制——GPU 的计算单元大部分时间处于等待 KV Cache 数据的状态. A100 的 HBM 带宽为 2.0 TB/s. 

| 方案 | KV Cache 加载量 | 加载时间(A100) | 相对延迟 |
|------|----------------|-----------------|---------|
| FP16 | 16.0 GB | 8.0 ms | 1.00× |
| INT8 | 8.0 GB | 4.0 ms | 0.50× |
| KIVI 4-bit | 4.0 GB | 2.0 ms | 0.25× |
| QJL (~3-bit) | ~2.5 GB | ~1.3 ms | ~0.16× |

上表的"加载时间"仅指 KV Cache 从 HBM 加载到 SM 的时间，不包含计算时间. 在 batch=32 的解码阶段，注意力计算的 arithmetic intensity 较低，内存带宽是瓶颈，因此带宽节省直接转化为延迟收益. 实测中，KIVI 4-bit 在长序列(> 4096)解码中可实现 1.5-2.0 倍的端到端加速. 

### 5.3 缩放因子开销的精确核算

量化方案的总存储不仅包括量化后的数据，还包括缩放因子等元数据. 以 KIVI 的 Key per-channel INT4 为例：

- Key 数据：$32 \times 8 \times 4096 \times 128 \times 0.5 \text{ bit} = 2.0 \text{ GB}$
- 缩放因子：$32 \times 8 \times 128 \times 2 \text{ bytes (FP16)} = 64 \text{ KB}$(每个通道一个 scale + zero_point)
- 元数据占比：$64 \text{ KB} / 2.0 \text{ GB} \approx 0.003\%$

对于 Value 的 per-token INT4：

- Value 数据：$32 \times 8 \times 4096 \times 128 \times 0.5 \text{ bit} = 2.0 \text{ GB}$
- 缩放因子：$32 \times 8 \times 4096 \times 2 \times 2 \text{ bytes} = 4.0 \text{ MB}$(每个 token 一个 scale + zero_point)
- 元数据占比：$4.0 \text{ MB} / 2.0 \text{ GB} = 0.2\%$

当位宽进一步降低(如 Key INT2)或分组粒度变细(如 group_size=32)时，元数据占比会显著上升. QJL 的核心价值在于完全消除了这一开销. 

## 6. 极简实现与源码映射

以下给出 KIVI 风格的 KV Cache per-channel + per-token INT4 量化的 PyTorch 实现(约 90 行)，精确映射到上方的数学公式. 

```python
import torch
import math

# 对应公式 (4)(5): 非对称量化
# x: 输入张量, dim: 计算 min/max 的维度, b: 位宽
def quantize_asymmetric(x: torch.Tensor, dim: int, b: int):
    x_min = x.min(dim=dim, keepdim=True)[0]
    x_max = x.max(dim=dim, keepdim=True)[0]
    scale = (x_max - x_min).clamp_min(1e-5) / (2 ** b - 1)
    zero_point = torch.round(x_min / scale).to(torch.int32)
    x_q = torch.round((x - x_min) / scale).clamp(0, 2 ** b - 1)
    return x_q.to(torch.uint8), scale.squeeze(dim), zero_point.squeeze(dim)

# 对应公式 (4)(5) 反量化
def dequantize_asymmetric(x_q: torch.Tensor, scale: torch.Tensor, 
                          zero_point: torch.Tensor):
    return x_q.float() * scale + zero_point.float() * scale

# 对应公式 (6): Key 的 per-channel 量化
# K: (batch, n_kv_heads, seq_len, head_dim)
# 在 seq_len 维度(dim=2)计算每个 head_dim 通道的 min/max
def quantize_key_per_channel(K: torch.Tensor, b: int = 4):
    K_q, scale, zp = quantize_asymmetric(K, dim=2, b=b)
    return K_q, scale, zp  # scale/zp shape: (B, H, D)

# 对应公式 (7): Value 的 per-token 量化
# V: (batch, n_kv_heads, seq_len, head_dim)
# 在 head_dim 维度(dim=-1)计算每个 token 的 min/max
def quantize_value_per_token(V: torch.Tensor, b: int = 4):
    V_q, scale, zp = quantize_asymmetric(V, dim=-1, b=b)
    return V_q, scale, zp  # scale/zp shape: (B, H, S)

# 对应公式 (8): 使用量化 KV Cache 的注意力
def attention_quantized_kv(Q, K_q, V_q, k_scale, k_zp, v_scale, v_zp):
    B, H_q, S_q, D = Q.shape
    _, H_kv, S_kv, _ = K_q.shape
    # GQA: Q 头数可能大于 KV 头数，需要广播
    if H_q != H_kv:
        reps = H_q // H_kv
        K_q = K_q.unsqueeze(2).repeat(1, 1, reps, 1, 1).flatten(1, 2)
        V_q = V_q.unsqueeze(2).repeat(1, 1, reps, 1, 1).flatten(1, 2)
        k_scale = k_scale.unsqueeze(2).repeat(1, 1, reps, 1).flatten(1, 2)
        k_zp = k_zp.unsqueeze(2).repeat(1, 1, reps, 1).flatten(1, 2)
        v_scale = v_scale.unsqueeze(2).repeat(1, 1, reps, 1).flatten(1, 2)
        v_zp = v_zp.unsqueeze(2).repeat(1, 1, reps, 1).flatten(1, 2)
    
    # 反量化 Key: (B, H_q, S_kv, D)
    K_hat = dequantize_asymmetric(K_q.float(), k_scale.unsqueeze(2), k_zp.unsqueeze(2))
    # 反量化 Value
    V_hat = dequantize_asymmetric(V_q.float(), v_scale.unsqueeze(-1), v_zp.unsqueeze(-1))
    
    scores = torch.matmul(Q, K_hat.transpose(-2, -1)) / math.sqrt(D)
    attn = torch.softmax(scores, dim=-1)
    out = torch.matmul(attn, V_hat)
    return out

# === 数值走查: Llama-3-8B 配置 ===
B, H_kv, D, S = 32, 8, 128, 4096
H_q = 32  # GQA: 4 个 Q head共享 1 个 KV head

# 模拟 FP16 KV Cache
K_fp16 = torch.randn(B, H_kv, S, D, dtype=torch.float16)
V_fp16 = torch.randn(B, H_kv, S, D, dtype=torch.float16)

# KIVI 量化: Key per-channel 4-bit, Value per-token 4-bit
K_q, k_s, k_z = quantize_key_per_channel(K_fp16.float(), b=4)
V_q, v_s, v_z = quantize_value_per_token(V_fp16.float(), b=4)

# 显存对比
kv_fp16_gb = 2 * B * H_kv * S * D * 2 / (1024**3)
kv_kivi_gb = (K_q.numel() * 0.5 + V_q.numel() * 0.5 +
              k_s.numel() * 2 + v_s.numel() * 2 +
              k_z.numel() * 2 + v_z.numel() * 2) / (1024**3)
print(f"FP16 KV Cache: {kv_fp16_gb:.2f} GB")
print(f"KIVI 4-bit KV Cache: {kv_kivi_gb:.2f} GB")
print(f"压缩比: {kv_fp16_gb / kv_kivi_gb:.2f}x")

# 精度验证: 与 FP16 注意力的输出差异
Q = torch.randn(B, H_q, 1, D)  # 解码阶段，当前步 1 个 token
out_fp16 = attention_quantized_kv(
    Q, K_fp16.repeat(1, H_q//H_kv, 1, 1).float(),
    V_fp16.repeat(1, H_q//H_kv, 1, 1).float(),
    torch.zeros(B, H_q, D), torch.zeros(B, H_q, D),
    torch.zeros(B, H_q, S), torch.zeros(B, H_q, S)
)
out_q = attention_quantized_kv(Q, K_q, V_q, k_s, k_z, v_s, v_z)
rel_err = (out_fp16 - out_q).abs().mean() / out_fp16.abs().mean()
print(f"注意力输出相对误差: {rel_err:.4%}")
```

**源码映射说明**

| 代码行 | 对应公式 | 说明 |
|--------|----------|------|
| `quantize_asymmetric` | 式 (4)(5) | 计算非对称量化的缩放因子、零点偏移和量化值 |
| `quantize_key_per_channel` | 式 (6) | Key 在 seq_len 维度(dim=2)计算 per-channel 参数 |
| `quantize_value_per_token` | 式 (7) | Value 在 head_dim 维度(dim=-1)计算 per-token 参数 |
| `dequantize_asymmetric` | 式 (4)(5) 反变换 | $(x_q + z) \cdot s$ 重建浮点值 |
| `attention_quantized_kv` | 式 (8) | 反量化后执行标准注意力计算 |
| GQA 广播逻辑 | — | H_q/H_kv 倍重复 KV head 以匹配 Query head |

## 7. 边界条件与失效模式

### 7.1 短序列的元数据开销反超

当序列长度 $s$ 很短时(如 $s < 64$)，per-token 量化的缩放因子存储量 $s \times 2 \times 2$ bytes 相对于量化节省的空间可能不成比例. 以 Value INT4 per-token 为例：

- 数据节省：$s \times d \times (16-4) / 8$ bytes
- 元数据开销：$s \times 2 \times 2$ bytes(scale + zero_point，各 FP16)
- 盈亏平衡点：$s \times d \times 1.5 < s \times 4$，即 $d < 2.67$

由于 $d_{\text{head}}$ 通常为 64-128，盈亏平衡从未触发. 但如果在极细粒度下使用 per-group(如 group_size=1)，元数据开销将压倒数据节省. **物理根源**：量化收益与元素数量成正比，元数据开销与分组数量成正比，存在一个临界分组大小使净收益为 0. 

### 7.2 Sink Token 的异常值摧毁通道精度

LLM 推理中存在 attention sink 现象：BOS token 或某些特定 token 的 Key 向量在少数通道上具有异常大的值. Per-channel 量化会将这些异常值纳入该通道的 min/max 范围，导致同一通道中其他正常 token 的精度被压缩到 1-2 位. 实验表明，在 Llama-2-70B 中，移除 sink token 的异常值后，per-channel INT4 的 perplexity 可从上升 3% 改善到上升 0.5%. 

**物理根源**：Sink token 的 Key 向量在特定通道上的值是模型训练涌现的结构性特征(用于聚合全局信息)，与正常 token 的局部分布不在同一个数量级. Min-max 映射被迫为这一个"巨人"牺牲整个通道的精度. 

**修复方向**：对 sink token 使用 FP16 存储; 或引入 outlier-aware 的截断量化(如将超出 $3\sigma$ 的值截断后再量化). 

### 7.3 GQA 下量化误差的头间广播放大

GQA(Grouped Query Attention)中，$H_q / H_{\text{kv}}$ 个 Query head 共享同一个 KV head. 量化误差在同一个 KV head 内被多个 Query head 复用，导致误差在输出端被叠加. 例如 GQA 比例为 4:1 时，一个 KV head 的量化误差会同时影响 4 个 Query head 的注意力输出. 

**物理根源**：GQA 的设计初衷是用 KV head 共享换取显存节省，量化在此基础上进一步压缩，误差不再被"稀释"到多个独立的 KV head 中，而是被多个 Query head 直接复用. 

**修复方向**：对共享 KV head 使用更高的位宽(如 6-bit 或 INT8); 或在误差敏感的层(如浅层)恢复 FP16 KV Cache. 

### 7.4 Key 低比特下的 Softmax 指数放大

当 Key 压缩到 INT2(仅 4 个量化级别)时，量化误差 $\Delta K$ 在注意力分数中表现为 $Q \Delta K^T$. 经过 softmax 的指数运算：

$$
\text{softmax}(x + \epsilon)_i = \frac{e^{x_i + \epsilon_i}}{\sum_j e^{x_j + \epsilon_j}} \tag{12}
$$

即使 $\epsilon$ 很小，指数函数的爆炸性增长也会放大领先者的优势. 实验表明，INT2 Key 在温度系数较低(< 1.0)的模型上会导致约 5-10% 的 token 漂移，但在温度系数较高(> 1.2)的模型上影响较小——因为较高的温度系数本身就在"平滑"注意力分布，降低了量化误差对 softmax 输出的敏感度. 

**物理根源**：Softmax 的指数运算将加性误差转化为乘性失真，且对概率分布的尖锐模式特别敏感. Key 的低比特量化改变了注意力分数的相对排序，导致 softmax 输出的概率质量重新分配. 

## 8. 技术前瞻与演进

### 8.1 动态精度与 Token 重要性分层

并非所有 token 对当前生成的贡献均等. Sink token 和最近局部窗口(如最后 1024 个 token)对预测影响最大，应使用高精度(4-bit 或 FP8); 中间历史 token 的影响衰减，可用 2-bit; 超出 attention 窗口的 token 可直接丢弃. 结合 SnapKV 的稀疏化策略，可将长序列的等效 KV Cache 再压缩 2-4 倍而不损失下游任务精度. 

### 8.2 FP8 KV Cache 的硬件原生支持

NVIDIA H100 的 Transformer Engine 原生支持 FP8(E4M3 格式)存储和计算. 与 INT8 相比，FP8 的动态范围更大(可表示约 $10^{-2}$ 到 $10^1$)，天然适应激活值的长尾分布，无需复杂的缩放因子管理. H100 上 FP8 KV Cache 的带宽节省与 INT8 相当，但省去了 per-channel/per-token 缩放因子的存储和计算开销. 随着 Blackwell 架构的普及，FP8 KV Cache 预计将在 2026 年成为生产环境的默认配置. 

### 8.3 量化与 PagedAttention 的融合

vLLM 的 PagedAttention 将 KV Cache 按 block(通常为 16 或 32 个 token)分块管理，不同序列的 block 可在物理内存中共享. 量化可以在 block 粒度上执行：每个 block 内的 KV 向量共享同一组量化参数，block 之间独立量化. 这种融合既保留了 PagedAttention 的内存复用优势，又获得了 KV 量化的压缩收益. vLLM 0.6.0 的路线图已明确将 block-wise KV 量化列为高优先级特性. 

### 8.4 跨层差分编码

相邻层的 KV 向量在语义上高度相关——第 $l$ 层和第 $l+1$ 层的 Key 向量余弦相似度通常超过 0.85. 存储层间差分 $\Delta \mathbf{k}_l = \mathbf{k}_{l+1} - \mathbf{k}_l$ 而非绝对值，可将差分向量的动态范围压缩 50% 以上. 差分编码与量化的联合使用，等效提升 0.5-1 个比特的有效精度，或在不降低精度的情况下将位宽再降低 1 bit.
