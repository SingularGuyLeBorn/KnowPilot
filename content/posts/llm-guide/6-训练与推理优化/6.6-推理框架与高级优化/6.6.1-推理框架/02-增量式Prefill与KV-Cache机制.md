# 增量式 Prefill 与 KV Cache 机制：动态检索增强推理的技术基石

> 本文深入解析 Search-R1 等交互式检索增强推理系统的底层机制：KV Cache 的动态增长、增量式 Prefill、以及为什么不缓存 Q 和 QK^T. 

---

## 1. 推理联机搜索过程

### 1.1 交错式搜索交互

Search-R1 的核心工作流程：**不是重新执行生成, 而是暂停当前生成、执行搜索、将搜索结果作为上下文追加, 然后继续生成**. 

想象 LLM 在写作文时遇到知识盲点, 主动"暂停写作", 去"查阅百科全书", 然后把"查阅到的段落"贴到草稿纸上, 再基于新信息继续写作. 

**单轮搜索的完整步骤**：

1. **生成搜索查询**(Action)：
   - LLM 生成特殊标记：`<search>`
   - 生成具体查询：如 `Britney Spears birthplace`
   - 生成结束标记：`</search>`
   - 暂停文本生成

2. **执行搜索**(Environment)：
   - 外部系统拦截 `<search>...</search>` 之间的文本
   - 发送给搜索引擎
   - 返回 Top-K 段落

3. **注入搜索结果**(Environment)：
   - 格式化检索结果：`<information>...</information>`
   - 直接追加到当前生成序列末尾

4. **继续生成**(Continue)：
   - LLM 基于更新后的完整上下文继续自回归解码
   - 可能发起新一轮搜索, 或直接生成 `<answer>`

### 1.2 与传统 RAG 的对比

| 特性 | Search-R1(交错式) | 传统 RAG(分离式) |
|:-----|:-------------------|:------------------|
| 工作流程 | Reason → Act → Reason → Act → ... | Retrieve → Generate |
| 检索时机 | 动态、多轮、按需触发 | 静态、单轮、开始前完成 |
| 查询生成 | LLM 自主产生, 基于当前推理状态 | 由输入问题改写或直接使用 |
| 灵活性 | 高, 可迭代深化查询 | 低, 第一次检索不准则全局受影响 |

## 2. 增量计算新增的检索信息上下文

### 2.1 核心概念：KV Cache

**Attention 计算**：LLM 的注意力机制在计算某个位置的输出时, 需要查看序列中所有之前的 token. 对于每个 token, 模型会计算其 Key(K)和 Value(V)向量. 

**KV Cache 的作用**：在生成过程中(Decode 阶段), 已经生成的 token 的 K 和 V 向量是固定不变的. KV Cache 就是将这些 K 和 V 向量缓存起来, 避免在生成每个新 token 时都重新计算整个历史序列的 K 和 V. 

### 2.2 Search-R1 的生成与搜索循环

| 步骤 | 操作 | 模式 | KV Cache 变化 |
|:-----|:-----|:-----|:--------------|
| 初始生成 | 生成推理和搜索标记 | Decode | 逐步追加 |
| 搜索执行 | 外部系统调用搜索引擎 | - | 暂停, 保持不变 |
| 注入信息 | 插入检索到的文本块 | **增量式 Prefill** | 并行计算新文本的 K/V, 追加到 Cache |
| 继续生成 | 基于新上下文继续输出 | Decode | 从 Cache 末尾继续追加 |

**注入检索信息的过程类似于一次针对特定片段的 Prefill**. 它不是从头开始的 Prefill, 而是在已有生成基础上的增量式 Prefill.

## 3. 增量式 Prefill 过程

### 3.1 标准 Prefill vs 增量式 Prefill

**标准 Prefill**：
- 触发时机：处理初始用户输入时
- 初始状态：KV Cache 为空
- 计算内容：计算整个输入提示的 K 和 V
- 输出：填充初始的 KV Cache

**增量式 Prefill**(Search-R1)：
- 触发时机：在生成过程中插入新的已知文本块(如检索结果)
- 初始状态：KV Cache 已包含之前生成的历史
- 计算内容：**仅计算新增文本块的 K 和 V**
- 输出：将新 K/V 追加到现有 KV Cache 之后

### 3.2 KV Cache 是一层都要缓存吗？

**是的**, 在标准的 Transformer Decoder 架构中, KV Cache 需要在**每一层**进行缓存. 

原因在于：
- LLM 由多个(如 32 层)Transformer Block 堆叠而成
- 每一层都有自己的 Key(K)和 Value(V)投影矩阵
- 同一 token 在不同层产生的 K 和 V 向量是完全不同的
- 生成第 t 个新 token 时, 需要与序列中前 t-1 个 token 在第 n 层产生的 K、V 向量进行注意力计算

Cache 的结构通常是一个大小为 `[num_layers, 2, batch_size, num_heads, seq_len, head_dim]` 的张量. 

---

## 4. 为什么不缓存 K、V 之外的中间计算结果？

### 4.1 为什么不缓存 Q(Query)？

Query 是**当前正在生成的 token** 独有的. 在 Decode 阶段, 每次只生成一个新 token, 它的 Q 向量只使用一次(用于与所有历史 K 计算注意力), 然后就被丢弃了. 缓存 Q 没有价值, 因为下一个 token 会有全新的 Q. 

### 4.2 为什么不缓存 QK^T(注意力分数)或 QK^TV(注意力输出)？

这些中间结果**依赖于当前 Query**：
- $QK^T$ 是当前 Q 与所有历史 K 的点积
- $QK^TV$ 是注意力加权后的输出

当生成下一个 token 时, Q 变了, $QK^T$ 和 $QK^TV$ 必须重新计算. 缓存它们没有意义. 

### 4.3 总结与类比

| 缓存对象 | 是否缓存 | 原因 |
|:--------|:--------|:-----|
| K(Key) |  | 历史 token 的 K 固定不变, 可被未来所有 token 复用 |
| V(Value) |  | 历史 token 的 V 固定不变, 可被未来所有 token 复用 |
| Q(Query) |  | 每个新 token 有唯一的 Q, 只用一次 |
| QK^T |  | 依赖当前 Q, 下个 token 的 Q 不同 |
| QK^TV |  | 依赖当前 Q, 下个 token 的 Q 不同 |

**类比**：KV Cache 就像写论文时的"参考文献卡片". 你已经读过的资料(历史 token)的要点(K, V)被记在卡片上, 后面写作时随时查阅. 但你当前正在写的这句话(当前 Q)和它与参考资料的关系(QK^T)是临时的, 不需要记录. 

---

## 5. 总结

增量式 Prefill 机制是实现 Search-R1 这类多轮交互系统的技术基石. 它巧妙地在计算效率和功能灵活性之间取得了平衡：

1. **动态增长**：KV Cache 随生成过程动态增长, 支持任意长度的交互历史
2. **增量更新**：新增信息只需局部 Prefill, 无需重新处理整个历史序列
3. **选择性缓存**：只缓存 K/V, 避免缓存依赖于当前 Query 的中间结果
