# LLM Serving 与 PagedAttention 深度解析

> 本文深入解析 SOSP '23 标志性工作 PagedAttention 及其在 vLLM 中的工程实现, 涵盖 LLM 推理的计算模式、内存瓶颈、迭代级调度与分页式 KV Cache 管理. 

---

## 1. LLM 推理的计算模式

### 1.1 两阶段执行模型

LLM 推理划分为两个阶段：

**Prefill 阶段(预填充)** ：
- 一次性并行处理用户输入的完整 Prompt 序列 $x_1, x_2, \ldots, x_n$
- 计算并输出第一个生成的 token
- 同时生成所有输入 token 的 KV Cache($k_1/v_1, \ldots, k_n/v_n$)
- **计算特性**：Attention 为矩阵乘矩阵(GEMM), Compute-bound

**Decode 阶段(解码)** ：
- 输入当前最新生成的单个 token $x_{n+t}$
- 结合显存中缓存的所有历史 KV 向量, 预测下一个 token $x_{n+t+1}$
- **计算特性**：Attention 退化为矩阵乘向量(GEMV), 但每次必须读取完整 KV Cache
- **瓶颈**：Memory-bound, 访存开销 dominates

### 1.2 KV Cache 的内存挑战

以 LLaMA-13B 为例：
- 层数：40
- 隐藏维度：5120
- KV head 数：40
- 序列长度：4096
- 数据类型：FP16

KV Cache 显存 = $2 \times 40 \times 5120 \times 4096 \times 2 \text{ bytes} \approx 2.6 \text{ GB}$

这仅是**单个请求**的显存消耗. 在 Batch 场景下, 显存需求随 Batch Size 和序列长度线性增长, 迅速成为系统瓶颈.

## 2. 现有系统的内存浪费

### 2.1 三种内存浪费来源

**内部碎片化(Internal Fragmentation)** ：
- 为每个请求预分配最大可能长度的连续显存块
- 实际序列长度通常远小于最大值, 造成大量未使用空间

**外部碎片化(External Fragmentation)** ：
- 不同长度的序列交错分配释放, 导致显存中出现大量不连续的小块
- 即使总空闲显存足够, 也无法满足大连续块的分配请求

**预留碎片化(Reservation Fragmentation)** ：
- 系统需要为每个请求预留未来增长的显存空间
- 这部分预留空间在大部分时间内处于空闲状态

### 2.2 现有方案的局限

- **FasterTransformer**：静态内存池, 无法适应动态变化的序列长度
- **TensorRT-LLM**：预先分配最大长度, 内部碎片化严重
- **Hugging Face TGI**：简单的内存复用, 外部碎片化未解决

## 3. PagedAttention：分页式 KV Cache 管理

### 3.1 核心思想

借鉴操作系统虚拟内存的**分页机制**：
- 将 KV Cache 分割为固定大小的**页(Page/Block)** 
- 页在逻辑上连续, 物理上可以离散分布
- 通过页表映射逻辑地址到物理地址

### 3.2 关键设计

**固定大小的块(Block)** ：
- 每个块存储固定数量的 token 的 KV 向量(如 16 个 token)
- 块大小是超参数, 需权衡内存管理开销与碎片化程度

**页表(Block Table)** ：
- 每个请求维护一个页表, 记录其逻辑块到物理块的映射
- 逻辑上连续的块在物理上可以不连续

**按需分配(On-Demand Allocation)** ：
- 仅在需要时分配新块, 而非预分配最大长度
- 新 token 的 KV 向量写入当前块的空闲槽位
- 块满时分配新块, 通过页表链接

### 3.3 内存效率分析

| 方案 | 内部碎片 | 外部碎片 | 预留碎片 | 总内存效率 |
|:-----|:--------|:--------|:--------|:----------|
| 静态预分配 | 高 | 无 | 高 | 低 |
| 动态连续分配 | 低 | 高 | 中 | 中 |
| **PagedAttention** | **低** | **无** | **无** | **高** |

## 4. 迭代级调度(Continuous Batching)

### 4.1 机制解析

在**迭代(Token)级别**进行批处理：
1. 每次迭代生成一个 token 后, 将已完成的请求从 Batch 中移除
2. 立即将队列中的新请求加入 Batch
3. 新请求的 Prefill 阶段与现有请求的 Decode 阶段**混合执行**

### 4.2 与 PagedAttention 的协同

- **内存复用**：请求完成后, 其占用的物理块立即释放, 可分配给新请求
- **动态 Batch Size**：Batch Size 随时间动态变化, 最大化 GPU 利用率
- **抢占与恢复**：高优先级请求可以抢占低优先级请求的块, 被抢占的请求块写入 CPU 内存, 恢复时读回

## 5. vLLM 的架构与优化

### 5.1 核心组件

**Scheduler**：
- 维护请求队列, 根据优先级和 SLO 选择进入 Batch 的请求
- 决定何时进行抢占和恢复

**Block Allocator**：
- 管理物理块的分配、释放、复制(用于 Beam Search 和并行采样)
- 实现 Copy-on-Write：多个请求共享同一块时, 写操作触发复制

**Worker**：
- 执行实际的模型推理(Prefill/Decode)
- 通过 CUDA kernel 实现高效的分页 Attention 计算

### 5.2 高级特性

**并行采样(Parallel Sampling)** ：
- 同一 Prompt 的多个采样输出共享 Prefill 阶段的 KV Cache
- 仅复制 Block Table, 物理块共享, 直到出现分歧(Copy-on-Write)

**Beam Search**：
- 多个 Beam 共享大部分 KV Cache
- 仅需为分叉的 Beam 复制少量块

**前缀缓存(Prefix Caching)** ：
- 缓存常见前缀(如系统提示)的 KV Cache
- 新请求匹配缓存前缀时, 直接复用, 跳过 Prefill

## 6. 性能与影响

### 6.1 吞吐量提升

vLLM 在多个基准测试上相比现有系统实现 **2-4× 吞吐量提升**：
- 在 ShareGPT 数据集上, vLLM 的吞吐量是 FasterTransformer 的 2.5×
- 在延迟约束下(P99 < 1s), vLLM 支持的并发请求数是 TGI 的 3×

### 6.2 产业影响

PagedAttention 已成为 LLM 推理的**事实标准**：
- vLLM 成为最广泛使用的开源推理框架
- 商业推理服务(Together AI、Fireworks、Anyscale)均基于 vLLM
- 后续工作(SGLang、LMDeploy)均继承并扩展了 PagedAttention 思想

## 7. 局限与未来方向

### 7.1 当前局限

- **块大小固定**：无法适应不同层、不同头的不同 KV 维度
- **CPU 内存卸载**：长序列场景下的抢占-恢复开销较大
- **多 GPU 扩展**：跨节点的块管理尚未完全优化

### 7.2 未来方向

- **自适应块大小**：根据模型层、头的特性动态调整块大小
- **磁盘卸载**：将不常用的块卸载到 NVMe SSD, 扩展有效内存容量
- **推测解码集成**：将 PagedAttention 与 speculative decoding 结合, 进一步降低延迟
