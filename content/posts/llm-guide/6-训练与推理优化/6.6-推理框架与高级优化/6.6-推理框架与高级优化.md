# 6.2 推理优化

## 1. 背景与核心痛点

在大语言模型的全生命周期中, 训练是一次性的资本投入, 而推理则是持续不断的运营成本. 一个部署在公有云上、服务百万日活用户的对话系统, 其推理成本往往占到总运营成本的 70% 以上. 这使得推理优化不仅是一个纯技术问题, 更是一个直接关系到商业可行性的经济命题. 

从计算图的角度看, 大模型推理可以清晰地划分为两个阶段：**预填充阶段(Prefill Phase)** 和 **解码阶段(Decode Phase)** . 预填充阶段接收用户的完整输入 prompt, 并行计算所有输入 token 的 Key 和 Value 向量, 并生成第一个输出 token; 解码阶段则进入自回归循环, 每次只生成一个 token, 且每个新 token 的生成都依赖于所有历史 token 的 KV Cache. 这两个阶段的瓶颈截然不同：预填充阶段通常是**计算密集型(Compute-bound)** , 其耗时与输入序列长度的平方成正比; 而解码阶段则往往是**内存带宽密集型(Memory-bound)** , 因为每次迭代都需要从显存中读取庞大的模型权重和 KV Cache, 而实际的矩阵乘法量却很小. 

传统深度学习推理框架(如早期版本的 ONNX Runtime、PyTorch JIT)是为 CNN、Transformer Encoder 等"一次前向传播即可完成"的模型设计的, 它们在应对自回归解码的动态性、变长序列、大体积 KV Cache 等问题时显得力不从心. 正是在这一背景下, 专门针对大模型推理优化的技术栈在过去几年内迅速成熟, 从底层的 CUDA kernel 融合到上层的调度策略, 形成了一套完整的工程方法论. 

## 2. 为什么重要

推理优化的重要性可以通过一组具体数字来感知. 假设我们有一个 70B 参数的模型, 以 FP16 精度部署在单张 A100(80GB)上：

- 模型权重本身占用约 140GB, 远超单卡显存, 因此至少需要 2 张 A100 做张量并行; 
- 当 batch size = 1、序列长度为 4096 时, KV Cache 约占用 5.5GB; 
- 当 batch size 提升到 32 时, KV Cache 膨胀到约 176GB, 成为显存的主要消耗者. 

如果没有推理优化, 这个模型在服务层面几乎不可行. 而通过量化将权重压缩到 INT4(体积减少 75%)、通过 PagedAttention 管理 KV Cache、通过连续批处理提升 GPU 利用率, 我们完全有可能在单张 A100 上实现可用的吞吐量. 这些优化技术的叠加, 使得从 GPT-3 到 ChatGPT 的"产品化跃迁"成为可能. 

## 3. 直觉类比：餐厅厨房的生产线

想象一家繁忙的餐厅厨房, 厨师(GPU)需要根据顾客的点单(推理请求)制作菜品(生成文本). 传统推理就像这样一种工作方式：每来一桌客人, 厨房就单独为他们准备所有食材、从头开始做菜, 即使多桌客人点了相同的开胃菜(共享的系统 prompt), 也要重复准备. 当某桌客人菜很多、某桌客人菜很少时, 厨师必须等所有人都吃完才能收拾桌子, 效率极低. 

现代推理优化则像是引入了一套精益生产体系：

- **算子融合** 相当于把"切菜、腌制、下锅"三个步骤合并成一个流畅动作, 减少了来回拿取食材的时间; 

- **PagedAttention** 相当于把厨房储物柜变成了灵活的分格收纳盒, 按需分配、按需回收, 不再有空间浪费; 

- **连续批处理** 相当于不等待整桌客人吃完才接待下一桌, 而是每做好一道菜就端出去, 空出的位置马上给新客人; 

- **前缀缓存** 相当于把点击率最高的开胃菜提前批量做好, 来客直接端上; 

- **投机解码** 相当于学徒先按照常规做法快速炒一份草稿, 主厨只需尝一口判断是否合格, 合格就直接上桌. 

## 4. 算子融合与图优化

### 4.1 算子融合(Kernel Fusion)

在 GPU 上执行神经网络时, 一个常常被低估的开销是 **kernel 启动开销** 和 **显存读写开销**. 现代深度学习框架(PyTorch、TensorFlow)将模型表达为一张计算图, 图中的每个节点对应一个独立的 CUDA kernel. 如果严格按照"一个算子一个 kernel"的方式执行, 那么即使每个 kernel 的计算量很小, 频繁的 kernel 启动和数据在显存中的来回搬运也会成为显著瓶颈. 

算子融合的核心思想是：**将计算图中相邻的、数据依赖关系简单的多个算子合并为一个单一的 CUDA kernel, 从而减少对全局显存的读写次数**. 

#### 4.1.1 Attention 融合：FlashAttention 与 FlashDecoding

Attention 计算的标准实现涉及多个独立的矩阵操作：计算 $QK^T$、应用 softmax、与 $V$ 相乘. 在传统的 PyTorch 实现中, 这些操作需要多次读写显存, 生成大规模的中间结果矩阵(如 $N \times N$ 的注意力分数矩阵). 

FlashAttention 通过精细的 **tiling(分块)** 和 **在线 softmax(online softmax)** 技术, 将整个 Attention 计算融合到一个 kernel 中. 其核心洞察在于：softmax 操作可以分解为局部的指数计算和归一化因子, 不需要一次性看到完整的注意力分数矩阵. 通过将 $Q$、$K$、$V$ 矩阵切分成小块加载到快速的 SRAM(共享内存)中, FlashAttention 在 SRAM 内完成分块的 Attention 计算, 仅输出最终的 $O$ 矩阵到全局显存, 避免了中间结果矩阵的显存占用. 

FlashAttention 的算法流程可以概括为：外层按块遍历查询矩阵 $Q$, 内层按块遍历键值矩阵 $K,V$, 在每个小块内依次完成局部分数计算、在线 softmax 统计更新和输出累加, 避免将任何中间矩阵写回全局显存：

$$
\text{for each block } i \text{ of } Q: \tag{1}
$$
$$
\quad \text{for each block } j \text{ of } K, V: \tag{2}
$$

在内循环中, 首先计算当前 $Q$ 块与 $K$ 块的局部注意力分数：
$$
\quad \quad S_{ij} = Q_i K_j^T \tag{3}
$$

将局部最大值与历史全局最大值比较, 更新当前所见的全局最大值, 用于保证后续指数计算的数值稳定性：
$$
\quad \quad m_{ij}^{\text{new}} = \max(m_{ij}^{\text{old}}, \max(S_{ij})) \tag{4}
$$

以新的全局最大值为基准计算当前块的指数化分数, 所有元素被平移至非正区间, 避免指数溢出：
$$
\quad \quad P_{ij} = \exp(S_{ij} - m_{ij}^{\text{new}}) \tag{5}
$$

全局指数和通过将旧统计量按新旧最大值的差距重新标度后, 与当前块的行和相加得到：
$$
\quad \quad l_{ij}^{\text{new}} = l_{ij}^{\text{old}} \cdot \exp(m_{ij}^{\text{old}} - m_{ij}^{\text{new}}) + \text{rowsum}(P_{ij}) \tag{6}
$$

输出累加器先按指数和的更新比例 rescale, 再融入当前块的加权值向量贡献：
$$
\quad \quad O_i = O_i \cdot \frac{l_{ij}^{\text{old}} \cdot \exp(m_{ij}^{\text{old}} - m_{ij}^{\text{new}})}{l_{ij}^{\text{new}}} + \frac{P_{ij} V_j}{l_{ij}^{\text{new}}} \tag{7}
$$

其中 $m$ 是在线维护的每行最大值, $l$ 是在线维护的每行指数和. 如式 (14)–式 (17) 所示, 整个计算在 SRAM 内闭环完成, 无需将中间注意力矩阵写回 HBM. 通过这种方式, FlashAttention 将 Attention 的显存复杂度从 $O(N^2)$ 降低到 $O(N)$, 同时由于大量减少了 HBM 访问量, 实际运行速度也有数倍提升. 

FlashDecoding 是 FlashAttention 在推理解码阶段的延伸. 解码阶段中 $Q$ 矩阵只有一行(只生成一个 token), 但 $K$、$V$ 矩阵非常长(所有历史 token). FlashDecoding 针对这一特殊形状进一步优化了并行策略, 将长序列的 $K$、$V$ 拆分给多个 warp 并行处理, 然后汇总结果, 显著降低了长序列解码的延迟. 

#### 4.1.2 LayerNorm 融合与残差连接融合

Transformer 块中的 LayerNorm、残差连接(Residual Connection)和激活函数通常也是独立的算子. 以标准的前向传播为例：

$$
\text{out} = \text{LayerNorm}(x + \text{Attention}(x)) \tag{8}
$$
传统实现会先计算 $\text{Attention}(x)$, 写回显存, 然后计算加法, 再写回显存, 最后计算 LayerNorm. 融合后的 kernel 可以一次性读取 $x$, 在寄存器中完成 Attention 输出、加法、LayerNorm 的全部计算, 只写回最终结果. 

NVIDIA 的 FasterTransformer 和 TensorRT-LLM 都对 Transformer 层进行了深度融合, 将一个完整的 Transformer 层(Attention + FFN + LayerNorm + Residual)压缩到极少数的几个 kernel 中. 

### 4.2 图优化(Graph Optimization)

图优化是在计算图层面进行的静态分析和变换, 不需要修改模型权重, 完全是"零成本"的优化. 

#### 4.2.1 常量折叠(Constant Folding)

如果计算图中的某些节点只依赖于常量输入(如模型中的位置编码偏置、固定的 mask 矩阵), 图优化器会在编译时预先计算这些节点的输出, 用常量节点替换原计算子图. 例如, Transformer 中 $1/\sqrt{d_k}$ 的缩放因子可以在编译时直接算出具体数值, 避免每次前向传播都做一次除法. 

#### 4.2.2 死代码消除(Dead Code Elimination)

训练阶段的计算图中常常包含推理时不需要的节点, 例如 dropout、梯度计算相关的辅助节点、损失函数节点等. 通过静态分析找出对最终输出无贡献的"死代码"并将其移除, 可以减小计算图规模、降低 kernel 启动开销. 

#### 4.2.3 算子替换与布局优化

图优化器还会进行一系列等价变换：

- **算子替换**：将 `Conv + BN + ReLU` 替换为融合后的 `ConvBiasReLU`; 将多个连续的 `Transpose` 合并为一个; 

- **布局优化**：根据后端硬件的特性选择最优的张量布局(NCHW vs NHWC、Row-major vs Column-major), 减少数据重排的开销; 

- **内存复用分析**：通过生命周期分析, 让不再需要的张量占用的显存被后续计算复用, 降低峰值显存占用. 

TensorRT 的图优化器是业界的标杆. 它会对 ONNX 格式的模型进行数十种优化 pass 的迭代执行, 最终生成高度优化的引擎. TensorRT-LLM 在此基础上进一步针对大模型的特点进行了定制, 例如对 Multi-Head Attention 的特定融合模式、对 GEMM(General Matrix Multiply)的自动选择最优的 CUDA kernel 实现等. 

## 5. 动态批处理(Continuous Batching / In-flight Batching)

### 5.1 传统静态批处理的困境

在早期的推理服务中, batching 是提升吞吐量的主要手段：将多个独立的推理请求打包成一个 batch, 一起送入模型进行前向计算. 然而, 这种 **静态批处理(Static Batching)** 存在一个致命缺陷：batch 中所有请求必须"同进同出". 

考虑一个 batch 中有三个请求, 分别需要生成 10、50、100 个 token. 在静态批处理中, 即使第一个请求在 10 步后就完成了, 它的 GPU 资源仍然被占用, 必须等到第三个请求完成 100 步后才能释放. 这意味着 GPU 在相当长的时间内只为少数活跃请求工作, 大量计算资源被"已完成但未释放"的请求浪费. 

从数学上看, 静态批处理的 GPU 利用率可以近似表示为：

$$
\eta_{\text{static}} = \frac{\sum_{i=1}^{B} T_i}{B \cdot \max(T_1, T_2, \ldots, T_B)} \tag{9}
$$

其中 $B$ 是 batch size, $T_i$ 是第 $i$ 个请求需要生成的 token 数. 当各请求的 $T_i$ 差异很大时, 这个比率可以低至 20-30%. 

### 5.2 连续批处理的原理

**连续批处理(Continuous Batching)** , 也称为 **In-flight Batching**, 彻底改变了这一模式. 它的核心思想是：**将 batching 的粒度从"请求级别"降低到"token 级别"**. 在每一步解码迭代中, 系统只处理当前 batch 中所有尚未完成的请求的一个 token; 一旦某个请求生成了结束符(EOS)或达到最大长度, 它立即被移出 batch, 同时新的请求被动态插入. 

这样, GPU 在每一步都处于"满负荷"状态——只要有待处理的 token, 就有计算在进行. 从数学上看, 连续批处理的利用率接近：

$$
\eta_{\text{continuous}} \approx \frac{\sum_{i} T_i}{\sum_{t=1}^{T_{\max}} B_t} \tag{10}
$$
其中 $B_t$ 是第 $t$ 步时 batch 中活跃请求的数量. 由于完成的请求被及时替换, $B_t$ 在整个过程中保持相对稳定, 利用率可以提升到 80-90% 以上. 

### 5.3 实现中的关键挑战

连续批处理的实现并不简单, 它面临几个工程挑战：

**KV Cache 的动态管理**：当新请求插入 batch 时, 需要为其分配 KV Cache 空间; 当请求完成时, 需要回收其 KV Cache. 这要求底层有一个高效的动态内存分配器——这正是 PagedAttention 的价值所在. vLLM 的 Block Manager 可以在 token 级别快速分配和释放固定大小的块, 完美适配连续批处理的需求. 

**注意力掩码的动态构造**：不同请求的长度不同, 合并成一个 batch 时需要构造一个组合的注意力掩码(attention mask), 确保每个 token 只能看到正确的历史上下文. 在 CUDA kernel 中动态构造这种"不规则"的掩码有额外的开销, 需要专门的 kernel 优化. 

**请求调度的公平性与优先级**：在实际服务中, 不同请求可能有不同的优先级(付费用户 vs 免费用户)、不同的延迟要求(实时对话 vs 离线批处理). 调度器需要在吞吐量、延迟、公平性之间做出权衡. ORCA 论文中提出的迭代级调度(iteration-level scheduling)和 vLLM 中的各种调度策略(如最早截止时间优先、最大输出长度优先)都是对这一问题的回答.

## 6. 推理引擎对比：vLLM、TensorRT-LLM、TGI、SGLang、lmdeploy

大模型推理的复杂性催生了一批专门的推理引擎. 这些引擎在功能定位、优化策略和适用场景上各有侧重. 

### 6.1 vLLM

**vLLM** 是由伯克利大学 Sky Computing Lab 开发的开源推理引擎, 也是目前社区最活跃、生态最丰富的推理框架之一. 它的核心创新是 **PagedAttention**, 将操作系统虚拟内存的分页机制引入 KV Cache 管理. 

**核心特性**：
- **PagedAttention**：通过固定大小的 block(默认 16 tokens)管理 KV Cache, 消除显存碎片, 支持动态分配和内存共享; 

- **Continuous Batching**：迭代级调度, 支持请求动态进出; 

- **广泛的后端支持**：支持 PyTorch、FlashAttention、XFormers 等多种 Attention 后端; 

- **丰富的 API**：提供 OpenAI 兼容的 REST API, 支持多 LoRA 切换、前缀缓存、投机解码等高级特性. 

**适用场景**：通用大模型推理服务, 特别是需要高吞吐量、动态负载、快速迭代的场景. 由于其纯 Python 的灵活性和活跃的社区, vLLM 非常适合研究和原型开发. 

**局限性**：相比 TensorRT-LLM 等编译型引擎, vLLM 的延迟优化不够极致, 在极致低延迟场景下可能不是最优选择. 

### 6.2 TensorRT-LLM

**TensorRT-LLM** 是 NVIDIA 推出的官方推理引擎, 基于 TensorRT 编译器框架, 专门针对 NVIDIA GPU 进行了深度优化. 

**核心特性**：
- **极致的 kernel 优化**：利用 TensorRT 的编译器和 NVIDIA 的 cuBLAS、cuDNN 库, 生成高度优化的 CUDA kernel; 

- **丰富的量化支持**：原生支持 INT8、FP8、INT4(AWQ、GPTQ)量化, 且量化后的 kernel 经过专门调优; 

- **多 GPU 并行**：内置张量并行(TP)和流水线并行(PP)支持, 通信优化深入到底层; 

- **In-flight Batching**：支持迭代级调度, 且 batching 逻辑与 kernel 执行深度耦合, 调度开销极小. 

**适用场景**：生产环境中对延迟和吞吐量都有极高要求的部署, 特别是已使用 NVIDIA GPU(A100、H100、H200)的场景. TensorRT-LLM 是 NVIDIA Triton Inference Server 的推荐后端, 与 NVIDIA 的完整推理栈集成度最高. 

**局限性**：仅支持 NVIDIA GPU; 编译过程较慢(需要将模型编译为 engine); 灵活性较低, 新增自定义算子或模型架构的难度较大. 

### 6.3 TGI(Text Generation Inference)

**TGI** 是 Hugging Face 推出的推理服务框架, 设计目标是让 Hugging Face 生态的模型能够快速部署为生产级推理服务. 

**核心特性**：
- **与 Hugging Face 生态深度集成**：支持 Transformers 库中的绝大多数模型架构, 新模型上架后往往可以"零修改"部署; 

- **Safetensors 支持**：默认使用 safetensors 格式加载模型, 加载速度快且安全; 

- **基础的 batching 和流式输出**：支持静态批处理和流式(streaming)输出; 

- **Rust 实现的 HTTP 服务**：高性能的网络层, Python 处理模型推理. 

**适用场景**：需要快速部署 Hugging Face 模型的场景, 特别是对模型多样性要求高、不想花时间做模型适配的场景. 

**局限性**：相比 vLLM 和 TensorRT-LLM, TGI 在吞吐量优化方面相对保守, 缺乏 PagedAttention 和连续批处理等高级特性(注：最新版本已逐步引入这些特性, 但成熟度和优化深度仍不及前两者). 

### 6.4 SGLang

**SGLang** 是由伯克利大学 LMSYS 团队推出的新一代推理框架, 其设计哲学是：**将大模型视为可编程的计算单元, 通过结构化生成(Structured Generation)和运行时优化提升效率**. 

**核心特性**：
- **RadixAttention**：SGLang 对 PagedAttention 的扩展, 专门优化前缀缓存. 它将 KV Cache 管理组织成一个 radix tree(基数树), 自动检测和复用请求之间的公共前缀; 

- **结构化生成 DSL**：提供 Pythonic 的 DSL, 允许用户描述复杂的输出结构(如 JSON Schema、正则表达式约束), 框架自动优化生成过程; 

- **高效的 Runtime**：用 C++ 实现了核心运行时, 调度开销极低; 

- **多模型协作**：原生支持一个 pipeline 中多个模型的协作推理(如 speculative decoding 中的 draft + target). 

**适用场景**：需要大量前缀复用的场景(如多轮对话、RAG、Few-shot Prompting), 以及对输出结构有严格约束的应用(如代码生成、数据提取). 

**局限性**：相对较新, 生态和社区规模不如 vLLM; 某些高级功能需要特定的模型支持. 

### 6.5 lmdeploy

**lmdeploy** 是商汤科技推出的推理引擎, 主打"轻量化部署"和国产硬件适配. 

**核心特性**：
- **TurboMind 推理引擎**：C++ 实现的高性能推理后端, 支持 Continuous Batching 和 FlashAttention; 

- **广泛的量化方案**：支持 AWQ、GPTQ、KV Cache 量化等多种压缩方案; 

- **国产硬件支持**：除 NVIDIA GPU 外, 还支持华为昇腾、海光 DCU 等国产 AI 加速器; 

- **多模态推理**：原生支持 LLaVA 等视觉-语言模型的推理. 

**适用场景**：需要国产硬件适配的场景, 以及需要在资源受限环境(如边缘设备)中部署大模型的场景. 

**局限性**：在纯 NVIDIA 生态中的优化深度和社区活跃度不及 vLLM 和 TensorRT-LLM.

## 7. 前缀缓存(Prefix Caching)

### 7.1 核心动机

在许多实际应用场景中, 多个推理请求共享相同的前缀文本. 最典型的例子包括：

- **多轮对话**：每一轮对话的历史上下文就是共享前缀; 

- **RAG(检索增强生成)** ：多个查询可能使用相同的检索文档作为上下文; 

- **Few-shot Prompting**：多个样本共享相同的指令和示例模板; 

- **系统 Prompt**：所有用户请求都包含相同的系统级指令. 

如果不做优化, 这些共享前缀的 KV Cache 会被重复计算和存储多次, 造成巨大的计算和显存浪费. 

### 7.2 实现原理

前缀缓存的基本思路是：**将共享前缀的 KV Cache 预先计算并持久化存储, 后续请求直接使用缓存结果, 跳过前缀部分的计算**. 

SGLang 的 **RadixAttention** 是前缀缓存的一个优雅实现. 它将 KV Cache 块组织成一个 radix tree(基数树, 即压缩前缀树)：

- 树的每个节点代表一个 KV Cache 块(16 或 32 tokens); 
- 从根节点到某个叶节点的路径代表一个完整序列的 KV Cache; 
- 当新请求到达时, 系统在 radix tree 中查找最长匹配的前缀路径, 直接复用匹配部分的 KV Cache; 
- 对于未匹配的部分, 再执行正常的预填充计算. 

这种设计天然支持前缀的**部分重叠**. 例如, 请求 A 的前缀是 "请总结以下文章：{文章1}", 请求 B 的前缀是 "请总结以下文章：{文章2}", 两者只有 "请总结以下文章：" 这部分重叠, RadixAttention 仍能复用这部分的 KV Cache. 

vLLM 在 0.3.0 版本后也引入了类似的前缀缓存支持, 通过 block 级别的引用计数实现前缀共享. 当多个序列共享相同的初始 block 时, 这些 block 的引用计数增加, 实际的物理显存只存储一份. 

### 7.3 性能收益

前缀缓存的收益与共享前缀的长度和请求的重叠度直接相关. 在 RAG 场景中, 如果所有请求都基于同一份 2000 token 的文档, 前缀缓存可以将预填充阶段的计算量减少到接近零, 首 token 延迟(TTFT)降低 90% 以上. 在多轮对话场景中, 随着对话轮数增加, 共享前缀越来越长, 缓存的价值也越来越大. 

## 8. 解码策略优化

解码策略决定模型如何从概率分布中选择下一个 token, 它直接影响生成文本的质量、多样性和速度. 

### 8.1 Beam Search

**Beam Search** 是一种系统性的解码方法, 它在每一步维护 $k$ 个最可能的候选序列(beam width = $k$), 而不是像贪心搜索那样只保留最优的一个. 

在每一步, Beam Search 对当前 $k$ 个候选序列分别扩展 vocabulary 中所有可能的下一个 token, 得到 $k \times V$ 个候选($V$ 为词表大小), 然后从中选择概率最高的 $k$ 个作为下一步的候选序列. 

候选序列的评分通常使用对数概率之和, 并加上长度惩罚(length penalty)来避免过度偏好短序列：

$$
\text{score}(y) = \frac{1}{|y|^\alpha} \sum_{t=1}^{|y|} \log P(y_t | y_{<t}) \tag{11}
$$

其中 $\alpha$ 是长度惩罚系数, 通常设为 0.6-1.0. 当 $\alpha = 1$ 时, 评分就是平均对数概率; 当 $\alpha < 1$ 时, 允许模型生成更长的序列. 

**Beam Search 的优势与局限**：

- 优势：生成的文本在概率上更为"安全", 语法和事实准确性通常较高; 适合机器翻译、摘要等需要高准确度的任务. 
- 局限：生成的文本往往缺乏多样性, 不同 beam 之间的结果高度相似; 在大模型中, Beam Search 甚至可能导致"退化"现象——生成重复、机械化的文本. 研究表明, 对于开放式生成任务(如对话、创意写作), Beam Search 的效果往往不如采样方法. 

### 8.2 Top-k Sampling

**Top-k Sampling** 解决了贪心搜索和纯随机采样之间的张力. 它首先将下一个 token 的概率分布限制在概率最高的 $k$ 个 token 中, 然后在这 $k$ 个 token 上重新归一化概率并进行采样. 

形式化地, 令 $V^{(k)}$ 为概率最高的 $k$ 个 token 的集合：

$$
P'(y_t = w) = \begin{cases} \frac{P(w | y_{<t})}{\sum_{w' \in V^{(k)}} P(w' | y_{<t})} & w \in V^{(k)} \\ 0 & w \notin V^{(k)} \end{cases} \tag{12}
$$
Top-k Sampling 的优势在于简单直观, 通过调整 $k$ 可以控制输出的随机性. $k$ 越小, 输出越"保守"; $k$ 越大, 输出越"多样". 

然而, Top-k Sampling 存在一个明显缺陷：**固定的 $k$ 无法适应不同时间步概率分布的"平坦度"**. 在某些时间步, 概率分布可能非常尖锐(一个 token 的概率接近 1), 此时即使 $k=50$, 也只有前几个 token 有显著概率; 在另一些时间步, 概率分布可能非常平坦, $k=50$ 会包含大量低概率的"噪声" token. 

### 8.3 Top-p Sampling(Nucleus Sampling)

**Top-p Sampling**(又称 Nucleus Sampling)是 Top-k Sampling 的优雅改进. 它不再固定保留 $k$ 个 token, 而是保留概率累积和达到阈值 $p$ 的最小 token 集合：

$$
V^{(p)} = \{ w' \mid \sum_{w \in V, P(w) \geq P(w')} P(w) \leq p \} \tag{13}
$$

也就是说, 我们从概率最高的 token 开始向下累加, 直到累加和超过 $p$ 为止, 这些 token 构成采样空间. 

Top-p Sampling 的优势在于**自适应**：当概率分布尖锐时, $V^{(p)}$ 自动变小, 只保留高概率 token; 当概率分布平坦时, $V^{(p)}$ 自动变大, 允许更多样化的选择. 这种自适应性使得 Top-p 成为当前大模型推理中最常用的采样策略. 实践中, $p$ 通常设为 0.9 或 0.95. 

### 8.4 Temperature

**Temperature** 是对概率分布的"锐化"或"平滑"操作, 它不改变采样的 token 集合, 而是改变集合内 token 的相对概率. 

给定原始概率分布 $P(w)$, 经过 temperature $T$ 调整后的概率为：

$$
P_T(w) = \frac{\exp(z_w / T)}{\sum_{w'} \exp(z_{w'} / T)} \tag{14}
$$
其中 $z_w$ 是模型输出的 logits. 

- 当 $T < 1$(如 $T = 0.7$)时, 概率分布变得更尖锐, 高概率 token 的优势被放大, 输出更加"确定"和"保守"; 
- 当 $T = 1$ 时, 概率分布保持不变; 
- 当 $T > 1$(如 $T = 1.5$)时, 概率分布变得更平坦, 低概率 token 有更多机会被选中, 输出更加"随机"和"有创意". 

Temperature 通常与 Top-p 结合使用：先用 Temperature 调整分布形状, 再用 Top-p 截断尾部噪声. 

### 8.5 数值走查：Temperature 的效果

假设模型对三个 token(A, B, C)输出的 logits 为 $[2.0, 1.0, 0.5]$. 温度参数 $T$ 通过缩放 logits 来控制生成分布的尖锐程度——$T < 1$ 增强高概率 token 的优势, $T > 1$ 则平滑分布、增加低概率 token 的机会. 

**Step 1：计算原始概率($T = 1$)** 

$$
\text{softmax}([2.0, 1.0, 0.5]) = [0.576, 0.212, 0.128] \tag{15}
$$
此式描述了变量之间的定量关系, 其物理意义将在下文详细阐述. 
**Step 2：$T = 0.5$(锐化)** 

降低温度使 logits 差异放大, 分布更集中：

$$
[2.0/0.5, 1.0/0.5, 0.5/0.5] = [4.0, 2.0, 1.0] \tag{16}
$$
$$
\text{softmax}([4.0, 2.0, 1.0]) = [0.844, 0.114, 0.042] \tag{17}
$$

A 的概率从 57.6% 提升到 84.4%, 分布明显更集中. 

**Step 3：$T = 2.0$(平滑)** 

$$
[2.0/2.0, 1.0/2.0, 0.5/2.0] = [1.0, 0.5, 0.25] \tag{18}
$$
$$
\text{softmax}([1.0, 0.5, 0.25]) = [0.455, 0.277, 0.215] \tag{19}
$$

A 的概率下降到 45.5%, C 的机会显著增加. 

| Temperature | P(A) | P(B) | P(C) | 分布特征 |
|------------|------|------|------|---------|
| 0.5 | 0.844 | 0.114 | 0.042 | 尖锐, 几乎确定选 A |
| 1.0 | 0.576 | 0.212 | 0.128 | 原始分布 |
| 2.0 | 0.455 | 0.277 | 0.215 | 平坦, 三者机会接近 |

### 8.6 解码策略的工程优化

从工程角度看, 不同的解码策略对推理效率有不同影响：

- **贪心解码** 最简单, 不需要维护多个候选, 内存和计算开销最小; 

- **Beam Search** 需要维护 $k$ 个并行的 KV Cache, 显存占用增加 $k$ 倍, 且不同 beam 之间无法共享计算; 

- **采样方法(Top-k / Top-p)** 在单次采样时与贪心解码的计算量相同, 但如果需要生成多个独立样本(如 temperature sampling 的多次运行), 可以通过 batch 并行执行来提升吞吐量. 

## 9. 量化推理

量化推理是将模型权重(以及可能的激活值)从高精度浮点数转换为低精度表示, 从而减小模型体积、降低显存带宽需求、提升计算吞吐量. 在大模型推理中, 量化已经从"可选优化"变成了"必需配置"——没有量化, 许多模型甚至无法在消费级硬件上运行. 

### 9.1 GPTQ(General-purpose Post-Training Quantization)

GPTQ 是一种逐层进行的训练后量化方法, 由 Frantar et al. 在 2022 年提出, 专门用于将大模型压缩到 INT4 精度. 

#### 9.1.1 核心思想

量化的本质是用有限个离散值近似连续分布的权重. 对于一个权重矩阵 $W$, 量化过程可以表示为：

$$
\hat{W} = s \cdot \text{round}\left(\frac{W}{s}\right) \tag{20}
$$
其中 $s$ 是缩放因子(scale), 决定了量化的步长. 对于逐层量化, 我们希望找到 $\hat{W}$, 使得该层的输出与原始输出之间的差异最小：

$$
\min_{\hat{W}} \| WX - \hat{W}X \|_2^2 \tag{21}
$$

其中 $X$ 是该层的输入激活值(使用校准数据集中的一批样本计算). 

#### 9.1.2 最优脑量化(Optimal Brain Quantization)

GPTQ 的核心算法基于 **OBS(Optimal Brain Surgeon)** 的思想. OBS 最初用于网络剪枝, 其基本思想是：给定一个已经训练好的网络, 如果要将某个权重 $w_i$ 设为 0(或量化为某个固定值), 应该同时调整其他未改变的权重来补偿这个变化带来的误差. 

数学上, OBS 通过二阶泰勒展开近似目标函数的变化：

$$
\Delta L \approx \frac{1}{2} \sum_{i,j} H_{ij} \Delta w_i \Delta w_j \tag{22}
$$
其中 $H$ 是 Hessian 矩阵. 为了找到最优的补偿, OBS 求解：

$$
\Delta w = -\frac{w_q}{[H^{-1}]_{qq}} H^{-1}_{\\cdot,q} \tag{23}
$$

这意味着当我们量化第 $q$ 个权重时, 所有其他权重都应该按 $H^{-1}$ 的第 $q$ 列进行补偿调整. 

GPTQ 将这一思想应用到量化中, 通过逐列处理权重矩阵, 每量化一个权重就立即用 OBS 规则更新同一列中尚未量化的权重, 从而逐步累积最小化量化误差. 

#### 9.1.3 GPTQ 的工程实践

GPTQ 的吸引力在于其极高的压缩比和相对合理的精度损失. 一个 70B 参数的模型, FP16 下需要 140GB 显存; 使用 GPTQ INT4 量化后, 权重仅占用约 40GB, 配合 4GB 左右的 KV Cache, 单张 A100 即可运行. 

AutoGPTQ 和 ExLlama 是两个广泛使用的 GPTQ 实现. 它们不仅实现了基础算法, 还针对 INT4 矩阵乘法开发了高度优化的 CUDA kernel, 使得量化后的模型在实际推理中确实能体现出速度提升(而非仅仅是体积缩小). 

### 9.2 AWQ(Activation-aware Weight Quantization)

AWQ 由 Lin et al. 在 2023 年提出, 其关键洞察是：**并非所有权重的通道(channel)对模型输出的影响都相同. 保护那些对激活值敏感的权重通道, 可以显著减少量化误差**. 

#### 9.2.1 核心洞察

考虑线性层 $Y = XW$. 如果我们按通道(列)对 $W$ 进行量化, 不同通道的量化误差对最终输出的影响取决于对应的输入激活值 $X$ 的幅度. 如果某个通道的输入激活值普遍很大, 那么该通道的量化误差会被放大; 反之, 如果输入激活值很小, 量化误差的影响也相对较小. 

AWQ 通过分析校准数据上的激活值分布, 识别出"重要"的权重通道, 并通过逐通道的缩放(scaling)来保护这些通道：

$$
\hat{W} = s \cdot \text{round}\left(\frac{W}{s}\right), \quad s \text{ 为逐通道的缩放因子} \tag{24}
$$
具体来说, AWQ 为每个输入通道引入一个缩放因子 $s$, 通过优化这些缩放因子来最小化量化后的输出误差. 这个优化过程不需要反向传播, 而是基于激活值统计量的解析解, 因此速度非常快. 

#### 9.2.2 AWQ 与 GPTQ 的对比

| 特性 | GPTQ | AWQ |
|------|------|-----|
| 量化粒度 | 逐层/逐组 | 逐通道 |
| 优化目标 | 最小化层输出误差 | 保护激活敏感的通道 |
| 校准数据 | 需要 | 需要 |
| 量化精度 | INT4/GPTQ | INT4/INT3 |
| 典型效果 | 4-bit 下 ppl 略有上升 | 4-bit 下 ppl 通常优于 GPTQ |
| 推理速度 | 需要专门 kernel | 需要专门 kernel |

在实践中, AWQ 在 4-bit 量化下通常能获得比 GPTQ 更好的困惑度(perplexity), 但两者的差距因模型而异. TensorRT-LLM 和 vLLM 都原生支持这两种量化格式. 

### 9.3 SmoothQuant

SmoothQuant 由 Xiao et al. 在 2022 年提出, 专门解决 **权激活量化(Weight-Activation Quantization)** 的难题. 

#### 9.3.1 问题背景

仅对权重进行量化(Weight-only Quantization)虽然能减小模型体积, 但由于计算时仍需要将权重反量化为 FP16 再与 FP16 激活值相乘, 其加速效果受限于内存带宽. 真正的加速来自于权激活量化——让权重和激活值都以 INT8 参与矩阵乘法, 利用 GPU Tensor Core 的 INT8 计算能力. 

然而, 激活值的量化比权重量化困难得多. 权重在训练后通常服从相对规则的高斯分布, 容易找到合适的缩放因子; 而激活值在不同样本、不同层之间的动态范围差异巨大, 存在所谓的 **异常值(outlier)** 问题——少数激活通道的值特别大, 导致简单的逐张量量化严重损失精度. 

#### 9.3.2 数学等价变换

SmoothQuant 的核心是一个数学等价变换. 对于线性层 $Y = XW$, 我们引入一个逐通道的缩放向量 $s$：

$$
Y = XW = (X \cdot \text{diag}(s)^{-1}) \cdot (\text{diag}(s) \cdot W) = \tilde{X} \cdot \tilde{W} \tag{25}
$$

其中 $\tilde{X} = X \cdot \text{diag}(s)^{-1}$ 是缩放后的激活值, $\tilde{W} = \text{diag}(s) \cdot W$ 是缩放后的权重. 

通过选择合适的 $s$, 我们可以"平滑"激活值的分布(使其更易于量化), 同时将量化的难度"迁移"到权重上. 由于权重的量化本身相对容易, 这种迁移显著提升了整体的量化精度. 

具体地, SmoothQuant 选择 $s$ 使得：

$$
s_j = \frac{\max(|X_j|)^\alpha}{\max(|X_j|)^{1-\alpha} \cdot \max(|W_j|)^\alpha} \tag{26}
$$
当 $\alpha = 1$ 时, 所有缩放都应用于激活值; 当 $\alpha = 0$ 时, 所有缩放都应用于权重. 实践中 $\alpha = 0.5$ 是一个好的平衡点. 

#### 9.3.3 SmoothQuant 的工程价值

SmoothQuant 的突破性在于：**它是第一个在大模型上实现了真正的 INT8 权激活量化且精度损失极小的方法**. 在 OPT-175B 和 BLOOM-176B 上, SmoothQuant 的 INT8 量化模型与 FP16 基线相比, 困惑度差异小于 1%. 这意味着我们可以在 NVIDIA GPU 的 INT8 Tensor Core 上运行大模型, 获得接近 2 倍的吞吐量提升. 

### 9.4 GGUF/GGML

**GGUF**(GPT-Generated Unified Format, 前身为 GGML)是 Georgi Gerganov 为 llama.cpp 开发的量化格式, 专门面向 CPU 推理和边缘设备部署. 

#### 9.4.1 k-quant 方法

GGUF 的核心是 **k-quant** 系列量化方案, 它不是在全矩阵上使用单一的量化策略, 而是将权重矩阵划分为大小为 $k$(通常是 256)的块, 在每个块内独立选择最优的量化参数. 这种细粒度的量化允许 GGUF 在模型大小和质量之间提供丰富的权衡选项：

| 量化类型 | 位宽 | 说明 |
|---------|------|------|
| Q8_0 | 8-bit | 每块 8 位, 质量最高, 速度较快 |
| Q6_K | 6-bit | 混合精度, 对重要层用更高精度 |
| Q5_K_M | 5-bit | 中等质量, 较好的大小/质量平衡 |
| Q4_K_M | 4-bit | 最常用的配置, 质量可接受 |
| Q3_K_M | 3-bit | 较小体积, 质量有所下降 |
| Q2_K | 2-bit | 最小体积, 仅适用于对质量不敏感的场景 |

k-quant 的另一个创新是 **混合精度**：对注意力层的权重(通常对精度更敏感)使用更高的位宽, 对 FFN 层的权重使用更低的位宽. 这种非均匀量化策略使得 GGUF 在极低位宽下仍能保持相对可接受的性能. 

#### 9.4.2 适用场景

GGUF 主要用于以下场景：
- **本地 CPU 推理**：在没有 GPU 或 GPU 显存不足的设备上运行大模型; 

- **边缘设备**：手机、树莓派等资源极度受限的环境; 

- **快速原型验证**：不需要极致性能, 只需要快速验证模型行为的场景. 

llama.cpp 和 Ollama 是 GGUF 格式的主要消费者, 它们利用 CPU 的 SIMD 指令(如 AVX2、AVX-512、ARM NEON)实现了相当高效的推理. 

### 9.5 量化推理的代码示例

以下是一个简化的 AWQ 风格的逐通道量化实现：

```python
import torch
import torch.nn as nn

def fake_quantize_per_channel(weight, bits=4):
    """
    逐通道对称量化(伪量化, 用于演示原理)
    weight: [out_features, in_features]
    """
    # 计算每个输出通道的缩放因子
    max_val = weight.abs().max(dim=1, keepdim=True)[0]  # [out_features, 1]
    qmax = 2 ** (bits - 1) - 1
    scale = max_val / qmax  # [out_features, 1]
    
    # 量化：round(weight / scale)
    weight_quant = torch.clamp(torch.round(weight / scale), -qmax - 1, qmax)
    
    # 反量化(模拟推理时的操作)
    weight_dequant = weight_quant * scale
    
    return weight_dequant, scale

class QuantizedLinear(nn.Module):
    def __init__(self, in_features, out_features, bits=4):
        super().__init__()
        self.weight = nn.Parameter(torch.randn(out_features, in_features))
        self.bits = bits
        self.scale = None
    
    def quantize(self):
        # 使用校准数据确定最优 scale(简化版)
        _, self.scale = fake_quantize_per_channel(self.weight, self.bits)
    
    def forward(self, x):
        # 推理时：先反量化权重, 再做 FP16 矩阵乘法
        # 实际工程中这里会用专门的 INT4/INT8 GEMM kernel
        weight_dequant, _ = fake_quantize_per_channel(self.weight, self.bits)
        return F.linear(x, weight_dequant)
```

## 10. 局限性与边界条件

推理优化技术并非万能, 每种技术都有其适用的边界和内在局限：

**算子融合的局限**：过度融合可能导致寄存器压力增大, 反而降低性能. 某些融合模式可能只适用于特定的输入形状(如固定的序列长度), 在动态形状场景下需要退回到通用实现. 

**连续批处理的调度开销**：虽然理论上利用率可以接近 100%, 但频繁的请求进出会引入调度开销和 KV Cache 管理的复杂度. 在请求到达率极低(如每秒 1 个请求)的场景下, 连续批处理的优势无法体现, 反而可能因为 batch 不足而导致延迟升高. 

**前缀缓存的命中率依赖**：前缀缓存的收益高度依赖于工作负载的特征. 如果所有请求的前缀完全不重叠(如完全随机的独立问题), 前缀缓存没有任何收益. 此外, 缓存本身需要显存来存储 KV Cache, 在显存极度紧张时可能需要权衡缓存大小. 

**量化推理的精度-速度权衡**：量化 inevitably 引入精度损失. 对于需要高可靠性的任务(如医疗诊断、金融分析), 4-bit 量化可能风险过高. 此外, 某些架构(如 MoE 模型)的量化效果不如 dense 模型稳定, 因为专家路由的稀疏性对量化误差更敏感. 

**推理引擎的锁定效应**：TensorRT-LLM 等编译型引擎虽然性能极致, 但一旦模型架构发生变化(如新增一个自定义层), 就需要重新编译引擎, 灵活性大打折扣. 对于快速迭代的研究场景, 这种锁定效应可能是不可接受的. 

## 11. 演进与承上启下

推理优化领域正在快速演进, 几个值得关注的方向包括：

**投机解码的成熟化**：从独立草稿模型到 Medusa、Lookahead Decoding 等"无草稿模型"方案, 投机解码正在降低部署门槛. 未来我们可能会看到更多"分层"的解码架构——大模型负责"关键决策", 小模型或 lookup 表负责"常规填充". 

**FP8 的普及**：随着 NVIDIA Hopper 及更新架构的 GPU 成为主流, FP8 推理将从实验室走向生产环境. FP8 相比 INT8 的优势在于对异常值的天然容忍, 有望在几乎无损精度的情况下实现 2 倍加速. 

**端侧推理的突破**：Apple Neural Engine、Qualcomm Hexagon 等 NPU 的算力正在快速提升, 配合 GGUF/MLX 等边缘优化框架, 在手机端运行 7B 甚至 13B 模型正在成为可能. 

**推理与训练的融合优化**：一些前沿工作开始探索"推理感知训练"——在训练阶段就考虑到后续推理的量化、剪枝等约束, 使得模型天生就更易于压缩和加速. 

在下一节(6.3 模型压缩)中, 我们将从更宏观的角度审视模型压缩方法论, 包括量化、剪枝、知识蒸馏、低秩分解和神经架构搜索. 这些技术与本节讨论的推理优化紧密交织：量化既是模型压缩的核心手段, 也是推理加速的关键路径; 知识蒸馏产生的小模型本身就是推理优化的终极方案.
