# 01 · 投机解码：Draft-then-Verify 的数学保证与 2-3x 推理加速

## 1. 背景与核心痛点 (Background & Pain Points)

### 1.1 自回归生成的宿命：一个 token 一步 forward

如果你曾经观察过 GPT-4、Claude 或 DeepSeek 在屏幕上"打字"的过程, 你会发现一个令人不安的事实：这些模型生成文字的速度, 慢得不像一个纯软件系统, 而像是一个人类在键盘上一个字母一个字母地敲击. 一个拥有万亿参数、在千亿 token 上训练出来的庞然大物, 为什么在输出阶段表现得如此"笨拙"？

答案藏在自回归(Autoregressive)生成的数学结构里. 设我们的语言模型为 $P(x_1, x_2, \ldots, x_T)$, 根据概率的链式法则, 它可以被分解为：

$$P(x_1, x_2, \ldots, x_T) = \prod_{t=1}^{T} P(x_t \mid x_1, x_2, \ldots, x_{t-1}) \tag{1} $$

这个公式看起来简洁优雅, 但它暗含了一个极其残酷的约束：**位置 $t$ 的 token $x_t$ 的条件概率, 依赖于所有之前位置的 token 作为输入**. 这意味着, 在生成第 100 个 token 之前, 你必须先把前 99 个 token 全部确定下来. 没有任何并行空间——你不能像并行计算矩阵乘法那样同时猜测第 100、第 101、第 102 个 token 是什么, 因为第 101 个 token 的输入上下文里必须包含第 100 个 token 的值, 而在你算出第 100 个 token 之前, 这个值根本不存在. 

这种串行依赖是刻在自回归模型 DNA 里的. 每一次生成一个 token, 模型都需要执行一次完整的 forward pass：输入当前的 token 序列, 经过 Embedding 层、若干 Transformer 块(每个块包含 Multi-Head Self-Attention 和 FFN), 最后从输出层的 logits 中采样出一个 token. 对于参数量为 $N$、上下文长度为 $L$ 的 Transformer 模型, 单次 forward pass 的计算复杂度大约是 $O(N + L^2 \cdot d)$, 其中 $L^2$ 来自 Attention 机制中 Query 矩阵与 Key 矩阵的逐点乘积. 当你要生成 $T$ 个 token 时, 总计算量大约是 $T$ 次这样的 forward pass 之和. 

### 1.2 解码阶段的压倒性占比

在大语言模型的完整推理 pipeline 中, 我们通常区分两个阶段：**预填充阶段(Prefill)** 和**解码阶段(Decoding)** . 

预填充阶段发生在用户提交一个 prompt 之后. 模型需要将整个 prompt 的 token 序列一次性输入网络, 计算每一层的 Key 和 Value 激活, 并将它们缓存到 KV Cache 中. 这个阶段虽然计算量巨大(因为 prompt 可能长达数千 token), 但它只执行一次. 而且, 由于整个 prompt 是已知的, Attention 计算可以在一定程度上并行化——所有 token 的 Query 可以同时计算, 与对应的 Key 做矩阵乘法. 

解码阶段则是噩梦的开始. 一旦预填充完成, 模型进入逐 token 生成的循环. 假设用户要求模型写一篇 2000 字的论文, 大约对应 3000-4000 个 token. 在解码阶段, 模型需要执行 3000-4000 次 forward pass. 每次 forward pass 的输入长度虽然在增长(从 1 个新 token 逐步增加到 3000 个), 但关键在于：**你无法跳过任何一步**. 

在实际工程 profiling 中, 解码阶段通常占据端到端推理时间的 **70%-90%**, 甚至更高. 对于长文本生成任务(如小说续写、代码生成、长文档摘要), 这个比例会进一步攀升. 预填充阶段可能只需要几百毫秒, 而解码阶段可能需要数十秒甚至数分钟. 这就是为什么用户在使用大模型时会感到"它在思考"——实际上它只是在执行成千上万次无法并行的矩阵运算. 

### 1.3 为什么加速解码如此困难

面对解码阶段的性能瓶颈, 工程师们尝试了无数种优化手段, 但几乎每一种都在自回归的数学铁律面前碰了壁. 

**KV Cache 优化**(如 PagedAttention)通过更高效的内存管理来减少显存碎片和内存带宽压力, 但它并没有减少 forward pass 的次数——你仍然需要执行 $T$ 次 forward pass. 

**量化(Quantization)** 将模型的权重从 FP16 压缩到 INT8 或 INT4, 降低了单次 forward pass 的计算量和内存带宽需求, 但同样没有改变"$T$ 次串行 forward"的根本结构. 它让每一步跑得更快, 但步数不变. 

**连续批处理(Continuous Batching)** 通过动态调度多个请求来提高 GPU 利用率, 让 GPU 在解码不同序列时"不闲着". 但这是一种系统层面的调度优化, 针对的是吞吐率(Throughput)而非单个请求的延迟(Latency). 对于单个用户等待自己的请求完成, 它无能为力. 

**Early Exit** 等方法试图在模型中间层就做出预测, 跳过后续层的计算. 但这种方法往往以牺牲精度为代价, 而且只有当模型对当前 token"非常有信心"时才能触发, 适用场景极其有限. 

问题的核心在于：如果我们不改变"一次只生成一个 token"的范式, 那么无论单次 forward pass 优化得多好, 生成的总步数始终是 $T$. 而真正令人绝望的是, 这 $T$ 步是**严格串行**的——第 $t+1$ 步必须等待第 $t$ 步的输出, 没有任何并行化的空间. 

### 1.4 算术分析：$T$ 次 $O(L^2)$ Attention 的残酷现实

让我们做一个更精确的算术分析. 假设我们要生成 $T$ 个新 token, 模型的隐藏维度为 $d$, 层数为 $L_{layer}$. 

在第 $t$ 步解码时($1 \leq t \leq T$), 输入序列的长度为 $L_{prefix} + t - 1$, 其中 $L_{prefix}$ 是 prompt 的长度. Attention 计算中, Query 的维度是 $(1, d)$(因为我们只计算最新一个 token 的 Query), 而 Key/Value Cache 的维度是 $(L_{prefix} + t - 1, d)$. 因此 Attention 的计算量为 $O((L_{prefix} + t - 1) \cdot d)$. 

在整个解码阶段, Attention 的总计算量为：

$$\sum_{t=1}^{T} O\left((L_{prefix} + t) \cdot d\right) = O\left(T \cdot L_{prefix} \cdot d + T^2 \cdot d\right) \tag{2} $$
当 $T$ 较大时(例如 $T = 4096$), $T^2$ 项会产生巨大的计算开销. 而且这仅仅是 Attention 部分——FFN 层的计算量同样不可忽视. 

更关键的是**内存墙(Memory Wall)** 问题. 在解码阶段, 模型参数量太大无法全部驻留在 GPU 的 L2 Cache 中, 每次 forward pass 都需要从 HBM(高带宽显存)中读取权重矩阵. 即使使用了 KV Cache, 每次 forward 仍然需要读取全部的模型权重. 对于 70B 参数的模型, 单次 forward pass 需要从显存读取约 140GB 的数据(FP16). 如果生成 4000 个 token, 总数据搬运量达到 560TB——这是推理延迟的真正瓶颈所在. 

**家谱定位**：投机解码(Speculative Decoding)正是诞生于这样的绝望背景之下. 它不试图优化单次 forward pass 的速度, 也不试图压缩模型权重, 而是从根本上挑战了"一次只能生成一个 token"的自回归铁律. 它提出了一种极其优雅的思路：如果我们无法并行地"确定"未来的 token, 那能不能先"猜测"未来的 token, 然后一次性验证这些猜测的正确性？

这个思路的天才之处在于, 验证过程是**可以并行化**的——大模型可以一次性查看一段候选 token 序列, 判断其中哪些是正确的. 只要猜测的准确率足够高, 我们就能用一次 forward pass 的代价, 换取多个 token 的确认, 从而突破自回归生成的串行瓶颈. 

## 2. 为什么重要 (Significance)

### 2.1 推理加速领域的里程碑式突破

投机解码是自 2023 年 DeepMind 发表《Fast Inference from Transformers via Speculative Decoding》以来, 大语言模型推理加速领域最具影响力的算法突破之一. 在它之前, 推理加速的主流思路集中在"让每一步更快"：量化、剪枝、高效的 Attention 实现、更好的内存管理. 这些优化虽然有效, 但都是渐进式的改进——通常只能带来 10%-50% 的加速, 而且往往伴随着精度损失或额外的工程复杂度. 

投机解码带来了一个质的飞跃：**在不改变模型输出分布、不损失任何精度的前提下, 实现 2-3 倍的推理加速**. 这不是通过压缩模型或降低计算精度实现的, 而是通过改变解码的"控制流"——从"生成-生成-生成"变成"猜测-验证-接受". 这种加速是算法层面的, 意味着它与量化、PagedAttention 等其他优化手段**正交叠加**：你可以在一个已经量化过的模型上使用投机解码, 在一个已经部署了 PagedAttention 的系统中集成投机解码, 从而获得乘积效应的加速. 

### 2.2 已被主流推理框架全面采纳

一项技术的真正价值, 最终要通过工程落地来检验. 投机解码之所以重要, 还因为它已经被当今最主流的生产级推理框架广泛采纳：

- **vLLM**：作为开源社区最流行的 LLM 推理引擎, vLLM 在 0.3.0 版本之后就集成了投机解码支持, 并与自家的 PagedAttention 无缝结合. 用户可以通过简单的配置参数启用 draft model 进行投机解码. 

- **TensorRT-LLM**：NVIDIA 官方的高性能推理框架, 将投机解码作为核心特性之一进行了深度优化, 充分利用了 Tensor Core 的并行计算能力来加速验证阶段. 

- **SGLang**：由 LMSYS 团队开发的结构化生成框架, 不仅支持传统的 draft model 投机解码, 还创新性地将投机解码与结构化输出(如 JSON Schema)结合, 进一步提升了特定场景下的加速比. 

- **Text Generation Inference (TGI)** ：HuggingFace 的生产级推理框架, 同样提供了投机解码的内置支持. 

- **Medusa**：作为一个独立的推理加速框架, Medusa 将投机解码的思想推向了新的高度——完全不需要第二个 draft model, 而是通过给目标模型增加多个解码头来实现多 token 预测. 

这种广泛的框架支持意味着, 投机解码已经从一个研究论文中的理论构想, 变成了生产环境中的标准配置. 当你在生产环境中部署 LLM 服务时, 启用投机解码几乎是一个"默认就该打开"的选项. 

### 2.3 开启了"多 token 预测"的新范式

投机解码的深远影响不仅在于它自身的加速效果, 更在于它开启了一个全新的研究范式：**多 token 预测(Multi-Token Prediction)** . 传统的自回归训练目标是最小化下一个 token 的预测交叉熵, 而投机解码告诉我们：如果模型能同时预测未来多个 token, 推理速度就可以大幅提升. 

这一洞见催生了后续一系列重要工作：从 Medusa 的多解码头架构, 到原生在训练阶段就加入多 token 预测目标的 MTP(Multi-Token Prediction)训练方法, 再到 Moonshot AI 的 Moonlight 等模型将多 token 预测作为核心训练目标. 可以说, 投机解码不仅是一个推理加速技巧, 更是推动大语言模型从"单步预测"走向"多步规划"的催化剂. 

## 3. 直觉类比 (Intuition)

### 3.1 标准解码：一个字一个字地写文章

想象你是一位极度严谨的作家, 正在撰写一篇重要的学术论文. 你的写作方式是：每写一个字, 都要停下来, 把前面所有的文字重新读一遍, 深思熟虑之后, 才决定下一个字该写什么. 而且你有一个奇怪的规定——你必须确定了第 $n$ 个字之后, 才能开始思考第 $n+1$ 个字是什么. 

这就是标准自回归解码的真实写照. 模型就像这位作家, 每生成一个 token 都要做一次完整的"重读+思考"(forward pass). 如果论文有 5000 字, 模型就要做 5000 次这样的重读和思考. 更糟的是, 随着文章越来越长, 每次重读所需的时间也越来越长(因为 Attention 需要处理更长的上下文). 

### 3.2 投机解码：草稿助手与主编的协作

现在让我们换一种工作方式. 你请来了一位实习生(草稿助手, Draft Model)和一位资深主编(目标模型, Target Model). 

**工作流程是这样的**：

1. **草稿阶段**：你先给实习生一个主题大纲, 让他快速地、不用太思考地写出一整段文字. 实习生写得很快, 因为他不需要像主编那样字斟句酌——他只需要凭直觉把大致意思写出来. 假设他一口气写了 $k$ 个字. 

2. **审阅阶段**：你把实习生写的这 $k$ 个字连同前面已经定稿的文字一起交给主编. 主编的阅读方式非常高效——他不是逐字阅读, 而是**一次性浏览整段**, 然后对每一个字做出判断："这个字放在这里是否合适？"

3. **接受/退回**：主编的审阅规则很特殊：
   - 如果主编认为第 1 个字是对的, 他就接受它; 
   - 如果第 1 个字是对的, 他才会看第 2 个字; 
   - 如果第 2 个字也是对的, 他继续看第 3 个字; 

- **一旦遇到第一个错误的字**, 主编会停下来, 不再看后面的字(因为后面的字是基于这个错误字写的, 即使碰巧对也不能保留). 
   - 对于那个错误的字, 主编不会直接丢弃——他会根据自己对上下文的理解, **重新选择一个更合适的字**来替代它. 

4. **循环**：从主编停下来的位置开始, 重复整个过程——再让实习生写 $k$ 个字, 主编再一次性审阅. 

这个类比的关键洞察在于：**主编的"一次性审阅 $k$ 个字"是并行的**. 虽然主编的判断能力(模型质量)比实习生强得多, 计算代价也高得多, 但由于他可以并行判断 $k$ 个字, 只要实习生的草稿质量足够高(接受率 $\alpha$ 足够大), 整体的写作速度就能显著提升. 

### 3.3 关键细节：为什么只退回到第一个错字

在这个类比中有一个非常关键但容易忽略的细节：为什么主编遇到第一个错字就停下来, 而不是继续看后面的字？

答案是**条件依赖**. 假设实习生的草稿是：

```
位置 1: "深"  (正确)
位置 2: "度"  (正确)
位置 3: "学"  (错误, 应该是"习")
位置 4: "习"  (碰巧对了, 但它是基于位置 3 的"学"写的)
位置 5: "模"  (基于错误上下文写的)
```

虽然位置 4 的"习"字本身是对的, 但它是基于位置 3 的"学"字推导出来的. 在真实的语言模型中, $P(x_4 \mid \text{"深度学"})$ 和 $P(x_4 \mid \text{"深度习"})$ 是完全不同的分布. 因此, 一旦位置 3 错了, 位置 4 及之后的所有 token 都必须被视为无效, 不能保留. 

这就是为什么投机解码的接受机制是"顺序验证、首次拒绝即停止"——这不仅仅是工程上的简化, 而是由概率的链式法则严格决定的数学必然. 

### 3.4 为什么这个方案不会降低文章质量

你可能会担心：让"水平较差"的实习生写草稿, 会不会导致最终文章的质量下降？

这就是投机解码最精妙的地方：**不会**. 因为主编(目标模型)对每一个被接受的 token 都进行了严格的"质量把关". 一个 token 被接受, 当且仅当主编认为它在自己的分布下是一个合理的采样结果. 如果实习生的某个 token 恰好与主编会采样的 token 一致, 那就接受它; 如果不一致, 主编会按照自己的分布重新采样一个替代 token. 

从数学上讲, 这个机制保证了一个惊人的性质：**投机解码生成的整个序列的分布, 与目标模型标准自回归解码生成的序列分布完全一致**. 也就是说, 用户完全感知不到任何质量差异——既不会更好, 也不会更差, 但速度却快了 2-3 倍. 这种"免费午餐"在计算机科学中极其罕见. 

## 4. 数学推导与公式对比 (Mathematical Rigor)

### 4.1 核心思想：Draft-then-Verify

让我们从符号定义开始, 逐步建立投机解码的完整数学框架. 

设目标模型(大模型)为 $M_{target}$, 草稿模型(小模型)为 $M_{draft}$. 给定前缀序列 $x_{<t} = (x_1, x_2, \ldots, x_{t-1})$, 两个模型分别定义了下一个 token 的条件概率分布：

- 目标模型的真实分布：$p(x \mid x_{<t})$
- 草稿模型的近似分布：$q(x \mid x_{<t})$

在标准自回归解码中, 目标模型每次只采样一个 token：$x_t \sim p(\cdot \mid x_{<t})$. 要生成 $k$ 个 token, 需要 $k$ 次串行的 forward pass. 

投机解码的核心创新在于以下三步流程：

**步骤 1：Draft(起草)** 

草稿模型 $M_{draft}$ 利用其更快的推理速度, 自回归地生成 $k$ 个候选 token：

$$\hat{x}_t \sim q(\cdot \mid x_{<t}) \tag{3} $$
$$\hat{x}_{t+1} \sim q(\cdot \mid x_{<t}, \hat{x}_t) \tag{4} $$
基于上述分析, 建立如下数学关系：
$$\hat{x}_{t+k-1} \sim q(\cdot \mid x_{<t}, \hat{x}_t, \ldots, \hat{x}_{t+k-2}) \tag{5} $$
得到候选序列 $\hat{x}_{t:t+k-1} = (\hat{x}_t, \hat{x}_{t+1}, \ldots, \hat{x}_{t+k-1})$. 这一步完全由草稿模型独立完成, 不需要目标模型参与. 

**步骤 2：Verify(验证)** 

这是整个算法最关键的一步. 我们将前缀 $x_{<t}$ 和草稿候选 $\hat{x}_{t:t+k-1}$ 拼接起来, 一次性输入目标模型 $M_{target}$：

$$\text{input} = [x_1, x_2, \ldots, x_{t-1}, \hat{x}_t, \hat{x}_{t+1}, \ldots, \hat{x}_{t+k-1}] \tag{6} $$

目标模型执行**一次** forward pass, 同时为这 $k$ 个位置中的每一个计算其真实的条件概率分布：

$$p_i(x) = p(x \mid x_{<t}, \hat{x}_t, \ldots, \hat{x}_{t+i-1}) \quad \text{for } i = 0, 1, \ldots, k-1 \tag{7} $$
注意这里的下标含义：$p_0$ 是在前缀 $x_{<t}$ 下对位置 $t$ 的预测分布, $p_1$ 是在前缀 $x_{<t}, \hat{x}_t$ 下对位置 $t+1$ 的预测分布, 以此类推. 由于这些分布都依赖于草稿 token, 它们是**条件于草稿序列**的分布. 

**步骤 3：Accept/Reject(接受/拒绝)** 

对于每个位置 $i$($0 \leq i \leq k-1$), 我们比较目标分布 $p_i$ 和草稿分布 $q_i$ 在该位置候选 token 上的概率值. 定义接受概率：

$$\alpha_i = \min\left(1, \frac{p_i(\hat{x}_{t+i})}{q_i(\hat{x}_{t+i})}\right) \tag{8} $$

其中 $q_i(\hat{x}_{t+i})$ 是草稿模型在生成该 token 时的原始采样概率(需要在 draft 阶段保存下来). 

验证从 $i=0$ 开始顺序进行：
- 以概率 $\alpha_0$ 接受 $\hat{x}_t$. 如果被接受, 继续验证 $i=1$; 
- 以概率 $\alpha_1$ 接受 $\hat{x}_{t+1}$. 如果被接受, 继续验证 $i=2$; 
- 依此类推, 直到某个位置 $j$ 被拒绝, 或者所有 $k$ 个 token 都被接受. 

**如果被拒绝**：假设位置 $j$ 是第一个被拒绝的位置. 我们不保留 $\hat{x}_{t+j}$, 而是从一个**修正分布(Modified Distribution)** 中采样替代 token：

$$x_{t+j} \sim p'_j(x) = \text{normalize}\left(\max(0, p_j(x) - q_j(x))\right) \tag{9} $$
更精确地写为：

$$p'_j(x) = \frac{\max(0, p_j(x) - q_j(x))}{1 - \sum_{x'} \min(p_j(x'), q_j(x'))} \tag{10} $$

这个修正分布的物理意义是：从目标分布中"减去"草稿分布已经覆盖的部分, 只保留目标分布独有的概率质量. 

验证过程停止后, 所有被接受的 token 加上可能的一个重新采样的 token, 构成这一轮投机解码的输出. 下一轮从最后一个被接受(或重新采样)的位置之后继续. 

### 4.2 接受/拒绝的数学机制：为什么这个接受率能保持目标分布？

这是投机解码最核心的数学保证, 也是整篇文章中最值得深入理解的部分. 我们需要证明：**上述 Draft-then-Verify 过程产生的 token 分布, 与目标模型标准自回归采样产生的分布完全一致**. 

#### 4.2.1 单步分析：一个位置上的接受/拒绝

让我们先聚焦于单个位置 $i$. 给定前缀(可能包含之前被接受的草稿 token), 草稿模型建议了一个 token $\hat{x}$, 其采样概率为 $q(\hat{x})$. 目标模型在该位置的真实分布为 $p(x)$. 

投机解码在这个位置定义了一个随机过程, 产生输出 token $x^*$. 我们需要证明 $x^* \sim p(x)$. 

这个过程的定义是：
1. 以概率 $\alpha = \min(1, p(\hat{x}) / q(\hat{x}))$ 接受 $\hat{x}$, 即 $x^* = \hat{x}$; 
2. 以概率 $1 - \alpha$ 拒绝 $\hat{x}$, 并从修正分布 $p'(x) \propto \max(0, p(x) - q(x))$ 中采样 $x^*$. 

我们分两种情况讨论：

**情况 A：$p(\hat{x}) \geq q(\hat{x})$**

此时 $\alpha = 1$, 即草稿 token 被**必然接受**. 

为什么会这样？因为当目标模型认为 $\hat{x}$ 的概率不低于草稿模型的估计时, 说明草稿模型在这个 token 上"偏保守"了——目标模型认为这个 token 比草稿模型认为的更合理(或至少一样合理). 在这种情况下, 直接接受它不会破坏目标分布. 

**情况 B：$p(\hat{x}) < q(\hat{x})$**

此时 $\alpha = p(\hat{x}) / q(\hat{x}) < 1$, 即草稿 token 以 $p/q$ 的概率被接受. 

为什么要用这个概率？让我们计算输出 token $x^*$ 等于某个特定值 $v$ 的总概率. 

**子情况 B1：$v = \hat{x}$(输出等于草稿 token)** 

$$P(x^* = \hat{x}) = \alpha = \frac{p(\hat{x})}{q(\hat{x})} \tag{11} $$
但这只是接受概率. 我们需要考虑 $\hat{x}$ 本身是从 $q$ 中采样的随机变量. 不过在这个单步分析中, 给定 $\hat{x}$ 已经被采样, 条件概率就是 $\alpha$. 

**子情况 B2：$v \neq \hat{x}$(输出不等于草稿 token)** 

$v$ 被输出的唯一途径是被拒绝后从修正分布中采样得到. 给定拒绝发生(概率 $1 - \alpha$), 从 $p'$ 中采样到 $v$ 的概率为：

$$p'(v) = \frac{p(v) - q(v)}{\sum_{x'} \max(0, p(x') - q(x'))} \tag{12} $$

注意这里的分子：当 $v \neq \hat{x}$ 且 $p(v) \geq q(v)$ 时, 分子为 $p(v) - q(v)$; 如果 $p(v) < q(v)$, 则分子为 0(因为 max 操作), 这意味着 $v$ 不可能从修正分布中被采样到. 

但这看起来有问题：如果 $p(v) < q(v)$, $v$ 岂不是永远不会被输出了？不对, 因为我们还需要考虑 $v$ 本身作为草稿 token 被采样的情况. 如果 $v$ 恰好是被草稿模型采样的那个 token(即 $\hat{x} = v$), 那么它会在情况 B1 中被以概率 $\alpha$ 接受. 

#### 4.2.2 完整的分布等价性证明

现在我们给出严格的证明. 对于任意特定的输出 token $v$, 计算它在投机解码下单步输出的总概率 $P_{spec}(v)$, 并证明它等于 $p(v)$. 

草稿 token $\hat{x}$ 是从 $q$ 中采样的. 对于固定的 $v$, $P_{spec}(v)$ 可以分解为两部分：

$$P_{spec}(v) = \underbrace{P(\hat{x} = v) \cdot P(\text{accept} \mid \hat{x} = v)}_{\text{草稿就是 } v \text{ 且被接受}} + \underbrace{P(\text{reject } \hat{x}) \cdot P(x^* = v \mid \text{reject})}_{\text{拒绝草稿后从修正分布采样到 } v} \tag{13} $$
**第一项**：$P(\hat{x} = v) = q(v)$. 接受概率取决于 $p(v)$ 和 $q(v)$ 的大小关系：
- 如果 $p(v) \geq q(v)$：$\alpha = 1$, 第一项为 $q(v) \cdot 1 = q(v)$
- 如果 $p(v) < q(v)$：$\alpha = p(v) / q(v)$, 第一项为 $q(v) \cdot \frac{p(v)}{q(v)} = p(v)$

**第二项**：需要计算拒绝概率和修正分布的概率. 拒绝概率为：

$$P(\text{reject}) = \mathbb{E}_{\hat{x} \sim q}[1 - \alpha(\hat{x})] = \sum_{x'} q(x') \left(1 - \min\left(1, \frac{p(x')}{q(x')}\right)\right) \tag{14} $$
此式描述了变量之间的定量关系, 其物理意义将在下文详细阐述. 
当 $p(x') \geq q(x')$ 时, $1 - \alpha = 0$; 当 $p(x') < q(x')$ 时, $1 - \alpha = 1 - p(x')/q(x') = (q(x') - p(x')) / q(x')$. 

基于上述分析, 建立如下数学关系：

$$P(\text{reject}) = \sum_{x': p(x') < q(x')} q(x') \cdot \frac{q(x') - p(x')}{q(x')} = \sum_{x'} \max(0, q(x') - p(x')) \tag{15} $$
这是 $q$ 超过 $p$ 的"超额质量"总量. 由于 $p$ 和 $q$ 都是概率分布(总和为 1), 可以证明：

$$\sum_{x'} \max(0, q(x') - p(x')) = \sum_{x'} \max(0, p(x') - q(x')) = \frac{1}{2} \sum_{x'} |p(x') - q(x')| \tag{16} $$

这个量等于总变差距离(Total Variation Distance)的一半, 记为 $D_{TV}(p, q)$. 它精确度量了两个概率分布之间的"不重叠程度". 

当拒绝事件发生时, 目标模型需要从修正分布 $p'$ 中采样. 修正分布在 $v$ 处的概率正比于 $p$ 超过 $q$ 的部分：

$$p'(v) = \frac{\max(0, p(v) - q(v))}{\sum_{x'} \max(0, p(x') - q(x'))} = \frac{\max(0, p(v) - q(v))}{P(\text{reject})} \tag{17} $$
因此第二项对目标分布的贡献为：

$$P(\text{reject}) \cdot p'(v) = \max(0, p(v) - q(v)) \tag{18} $$
此式描述了变量之间的定量关系, 其物理意义将在下文详细阐述. 
**合并两项**：

现在将接受分支和拒绝分支合并, 分两种情况讨论：

**如果 $p(v) \geq q(v)$**：
- 第一项 = $q(v)$(因为 $\alpha = 1$, 草稿 $v$ 必然被接受)
- 第二项 = $p(v) - q(v)$(因为 $\max(0, p(v) - q(v)) = p(v) - q(v)$)
- 总和 = $q(v) + (p(v) - q(v)) = p(v)$ ✓

**如果 $p(v) < q(v)$**：
- 第一项 = $p(v)$(因为 $\alpha = p(v)/q(v)$, $q(v) \cdot p(v)/q(v) = p(v)$)
- 第二项 = $0$(因为 $\max(0, p(v) - q(v)) = 0$)
- 总和 = $p(v) + 0 = p(v)$ ✓

**结论**：对于任意 $v$, $P_{spec}(v) = p(v)$. 这严格证明了投机解码的每一步输出都遵循目标模型的真实分布. $lacksquare$

#### 4.2.3 修正分布的深入理解

修正分布 $p'(x) \propto \max(0, p(x) - q(x))$ 有一个非常优美的物理解释. 让我们思考：当草稿 token $\hat{x}$ 被拒绝时, 目标模型对什么 token "最有意见"？

答案是：那些目标模型认为概率很高、但草稿模型却严重低估了的 token. 具体来说, 如果 $p(v) > q(v)$, 说明目标模型认为 $v$ 比草稿模型认为的更合理——草稿模型"漏掉"了这个好选项. 修正分布正好把这些被低估的 token 挑出来, 按它们被低估的程度($p(v) - q(v)$)重新分配概率质量. 

反之, 如果 $p(v) \leq q(v)$, 说明草稿模型对 $v$ 的估计至少不比目标模型更悲观. 这样的 token 不会出现在修正分布中, 因为它们没有被"低估". 

从信息论的角度看, 修正分布捕获了目标分布相对于草稿分布的"正剩余信息"——即目标模型知道但草稿模型不知道的关于下一个 token 的信息. 

### 4.3 期望加速比推导

分布等价性保证了投机解码不会降低输出质量, 但工程上我们更关心的是：它到底能快多少？

#### 4.3.1 单次投机迭代的期望接受数

在一次 Draft-then-Verify 迭代中, 草稿模型生成了 $k$ 个候选 token. 设第 $i$ 个位置的接受概率为 $\alpha_i$($i = 1, 2, \ldots, k$). 注意 $\alpha_i$ 本身是随机变量, 因为它依赖于前面位置的草稿 token 是否被接受. 但为了简化分析, 我们通常假设所有位置的接受概率近似相等, 记为 $\alpha$. 这个假设在实际中大致成立, 尤其是当草稿模型和目标模型"风格一致"时. 

定义 $A$ 为单次迭代中被接受的 token 数量. $A$ 的分布为：

- $P(A \geq 1) = \alpha_1$
- $P(A \geq 2) = \alpha_1 \cdot \alpha_2$(第 1 个和第 2 个都被接受)
- $P(A \geq i) = \prod_{j=1}^{i} \alpha_j$

因此期望接受数为：

$$\mathbb{E}[A] = \sum_{i=1}^{k} P(A \geq i) = \sum_{i=1}^{k} \prod_{j=1}^{i} \alpha_j \tag{19} $$
在等接受概率假设下($\alpha_j = \alpha$ 对所有 $j$)：

$$\mathbb{E}[A] = \sum_{i=1}^{k} \alpha^i = \alpha \cdot \frac{1 - \alpha^k}{1 - \alpha} \tag{20} $$

当 $k \to \infty$ 时(即草稿无限长), 期望接受数收敛到：

$$\mathbb{E}[A]_{k=\infty} = \frac{\alpha}{1 - \alpha} \tag{21} $$

#### 4.3.2 理想情况分析

在理想情况下, 草稿模型与目标模型完全一致($q = p$), 则每个位置的接受概率 $\alpha_i = 1$. 此时：

$$\mathbb{E}[A] = \sum_{i=1}^{k} 1 = k \tag{22} $$

所有 $k$ 个草稿 token 都被接受. 由于验证阶段只需要**一次**目标模型的 forward pass(虽然这次 forward 的序列长度是 $k$, 但可以通过高效的并行 Attention 优化), 而标准解码需要 $k$ 次 forward pass, 因此理论加速比为 $k$ 倍. 

当然, $q = p$ 的假设意味着草稿模型和目标模型是同一个模型, 这失去了使用草稿模型的意义. 但它给出了加速比的上界. 

#### 4.3.3 现实情况分析

在实际中, 草稿模型通常是目标模型的一个"缩小版"(如用 7B 模型作为 70B 模型的草稿, 或用目标模型的前几层作为草稿). 接受率 $\alpha$ 通常在 0.6 到 0.8 之间, 具体取决于任务的难度和草稿模型的质量. 

基于上述分析, 建立如下数学关系：

$$\mathbb{E}[A] = 0.7 + 0.49 + 0.343 + 0.2401 + 0.16807 \approx 1.94 \tag{23} $$
这意味着平均每次迭代大约能接受 2 个 token. 如果验证的计算开销与一次标准解码相当(实际情况中, 一次长度为 $k$ 的验证 forward 通常比 $k$ 次单 token forward 更快, 因为可以复用 KV Cache 并进行矩阵并行), 那么加速比大约为 1.94 倍. 

#### 4.3.4 最优草稿长度 $k^*$ 的推导

草稿长度 $k$ 的选择是一个关键的工程超参数. $k$ 太小, 加速效果不明显; $k$ 太大, 验证的计算开销增加, 而且被拒绝时浪费的草稿计算也更多. 

设：
- $c_d$：草稿模型生成一个 token 的平均时间
- $c_t$：目标模型验证一个长度为 $k$ 的序列的时间
- $\alpha$：接受率(假设各位置相同)

在单次迭代中：
- 草稿阶段耗时：$k \cdot c_d$
- 验证阶段耗时：$c_t(k)$(通常是 $k$ 的次线性函数, 因为有并行性)
- 期望产出：$\mathbb{E}[A] = \sum_{i=1}^{k} \alpha^i$

每 token 的期望耗时为：

$$\text{Cost per token} = \frac{k \cdot c_d + c_t(k)}{\sum_{i=1}^{k} \alpha^i} \tag{24} $$

标准解码每 token 的耗时约为 $c_t(1)$(一次目标模型的单 token forward). 加速比为：

$$\text{Speedup}(k) = \frac{c_t(1) \cdot \sum_{i=1}^{k} \alpha^i}{k \cdot c_d + c_t(k)} \tag{25} $$
为了找到最优 $k^*$, 我们可以对这个函数求导并令导数为零. 但由于 $c_t(k)$ 的具体形式依赖于硬件和实现, 解析解通常难以得到. 在实际工程中, 最优 $k$ 通常通过网格搜索或在线自适应调整来确定. 

不过我们可以得到一个直观的结论：当草稿模型非常快($c_d \ll c_t$)且接受率较高($\alpha \to 1$)时, 应该选择较大的 $k$; 当草稿模型较慢或接受率较低时, 应该选择较小的 $k$. 

#### 4.3.5 加速比的理论上界

投机解码的加速比有一个理论上界, 由草稿模型和目标模型的计算效率比以及接受率共同决定. 

设目标模型生成一个 token 的计算量为 $C_{target}$, 草稿模型为 $C_{draft}$. 在理想情况下($\alpha = 1$), 加速比为：

$$\text{Speedup}_{ideal} = \frac{C_{target}}{C_{draft} + C_{target}/k} \approx \frac{C_{target}}{C_{draft}} \tag{26} $$

当 $k$ 很大时, 加速比趋近于目标模型与草稿模型的计算量比. 这意味着：**投机解码的加速比不可能超过目标模型和草稿模型的计算能力比**. 如果你用一个和原模型一样大的模型做草稿, 即使接受率是 100%, 你也只是把计算从一个模型转移到了另一个模型, 没有净加速. 

### 4.4 改进变体：从 Draft Model 到 Multi-Head

传统投机解码需要一个独立的草稿模型, 这带来了额外的显存开销和模型管理复杂度. 研究者们在此基础上发展出了一系列变体, 试图消除对独立草稿模型的依赖. 

#### 4.4.1 Medusa：目标模型自身的多解码头

Medusa 的核心洞察非常直接：**既然草稿模型的目的是预测未来 token, 为什么不直接在目标模型上增加几个额外的解码头, 让它们同时预测未来多个位置的 token？**

具体而言, Medusa 在目标模型的最后一层隐藏状态之上, 添加了 $k$ 个独立的线性投影头(heads). 设最后一层的隐藏状态为 $h \in \mathbb{R}^d$：

- 头 0(原始头)：预测下一个 token, $\text{logits}_0 = W_0 \cdot h$
- 头 1：预测下下个 token, $\text{logits}_1 = W_1 \cdot h$
- 头 2：预测下下下个 token, $\text{logits}_2 = W_2 \cdot h$
- $\vdots$
- 头 $k$：预测第 $k$ 个未来 token, $\text{logits}_k = W_k \cdot h$

这些额外的头在训练阶段通过辅助损失函数进行训练. 给定一个训练序列 $(x_1, x_2, \ldots, x_T)$, Medusa 的损失为：

$$\mathcal{L}_{Medusa} = \mathcal{L}_{next} + \lambda \sum_{i=1}^{k} \mathcal{L}_{i} \tag{27} $$
其中 $\mathcal{L}_{next}$ 是标准的下一个 token 预测交叉熵, $\mathcal{L}_i$ 是第 $i$ 个未来位置的预测交叉熵, $\lambda$ 是平衡系数. 

**Medusa 与传统投机解码的关键差异**：

| 特性 | 传统投机解码 | Medusa |
|------|-------------|--------|
| 草稿来源 | 独立的草稿模型 $M_{draft}$ | 目标模型自身的多个解码头 |
| 显存开销 | 需要加载两个完整模型 | 仅增加少量头参数($k \times d \times |V|$) |
| 分布一致性 | 数学严格保证 | 训练目标近似, 非严格保证 |
| 训练成本 | 无需额外训练 | 需要微调训练额外头 |
| 灵活性 | 可任意更换草稿模型 | 头与目标模型绑定 |

Medusa 的验证过程与传统投机解码类似：用 $k$ 个头分别生成 $k$ 个候选 token, 然后用目标模型的原始头(或一个完整的 forward pass)并行验证. 由于所有头共享同一个骨干网络, draft 阶段几乎没有额外开销. 

#### 4.4.2 树形注意力(Tree Attention)：分支验证的优雅结构

传统投机解码中, 草稿序列是一条线性的链. 但 Medusa 的多个头可以产生多个独立的候选序列, 这些序列可以组织成一棵树状结构. 例如：

- 头 1 预测位置 $t+1$ 的 top-2 候选：$A$ 或 $B$
- 如果位置 $t+1$ 是 $A$, 头 2a 预测位置 $t+2$ 的候选：$C$ 或 $D$
- 如果位置 $t+1$ 是 $B$, 头 2b 预测位置 $t+2$ 的候选：$E$ 或 $F$

这样就形成了一棵候选树：

```
        [前缀]
       /      \
      A        B
     / \      / \
    C   D    E   F
```

树形注意力(Tree Attention)允许目标模型在一次 forward pass 中并行验证这棵候选树上的所有路径. 通过精心设计的 Attention Mask, 模型可以让每个候选节点只看到其祖先节点(即前缀 + 该路径上之前的候选), 而看不到其他分支的节点. 

这种结构的优势在于：即使某条路径在早期就被拒绝, 其他路径仍然可以继续验证, 提高了"命中"的概率. 树形注意力的数学复杂度与树的宽度和深度有关, 但通过 CUDA Kernel 优化, 它在实际中可以非常高效. 

#### 4.4.3 Lookahead Decoding：从历史中寻找未来

Lookahead Decoding 是另一种不需要草稿模型的投机解码变体. 它的核心思想是：**如果当前正在生成的内容, 与 prompt 中或之前已生成内容中的某个片段相似, 那么我们可以直接"借用"那个后续片段作为草稿**. 

具体实现上, Lookahead Decoding 维护一个 n-gram 池. 在解码过程中：
1. 提取当前最近生成的 $n$ 个 token 作为 query n-gram; 
2. 在 n-gram 池中查找匹配的 n-gram; 
3. 如果找到匹配, 将匹配 n-gram 后面的连续 token 作为草稿候选; 
4. 用目标模型验证这些草稿 token. 

这种方法对于重复性较高的文本特别有效, 如代码生成(循环体、模板代码)、列表生成、格式化输出等. 它的优点是**完全不需要额外的模型或训练**, 只需要维护一个 n-gram 缓存. 缺点是对于创新性强的文本(如创意写作、开放式问答)效果有限, 因为很难找到匹配的 n-gram. 

#### 4.4.4 Prompt Lookup Decoding：在 Prompt 中挖掘投机机会

Prompt Lookup Decoding 进一步将 Lookahead 的思想局限在 prompt 范围内. 它假设：**用户提供的 prompt 中可能包含了与生成内容相关的片段**. 例如, 在代码补全任务中, prompt 中可能已经有类似的函数定义; 在文档问答中, prompt 中可能已经有答案的相关句子. 

算法流程如下：
1. 将 prompt 切分为 token 序列并索引; 
2. 在解码过程中, 用当前生成的前缀在 prompt 中查找最长匹配; 
3. 将 prompt 中匹配位置之后的 token 作为草稿候选; 
4. 验证并接受/拒绝. 

这种方法对于长上下文任务尤其有效, 因为长 prompt 提供了更多的"借用"机会. 而且它的实现极其轻量——几乎只需要一个字符串匹配算法(如后缀数组或 Trie)就能工作. 



## 5. 数值走查 (Numerical Example)

数学推导虽然严谨, 但对于许多读者来说, 具体的数字往往比抽象的符号更能建立直觉. 本节我们将构造一个极度简化的场景, 手动走一遍投机解码的完整流程, 让每一个公式都"落地"为可触摸的计算结果. 

### 5.1 场景构造：一个三 token 词表的微观世界

假设我们的词表只有三个 token：$\mathcal{V} = \{A, B, C\}$. 这是一个极度简化的设定, 但它足以展示投机解码的所有核心机制. 

给定某个前缀(具体是什么不重要, 我们关注的是条件分布), 草稿模型 $q$ 和目标模型 $p$ 在该位置的概率分布如下：

| Token | $q$ (草稿模型) | $p$ (目标模型) |
|-------|---------------|---------------|
| A     | 0.5           | 0.6           |
| B     | 0.3           | 0.2           |
| C     | 0.2           | 0.2           |

草稿模型从 $q$ 中采样, 假设采样结果是 token "A"(这在 $q$ 中是最可能的结果, 概率为 0.5). 

### 5.2 接受概率计算

根据投机解码的接受机制：

$$\alpha = \min\left(1, \frac{p(A)}{q(A)}\right) = \min\left(1, \frac{0.6}{0.5}\right) = \min(1, 1.2) = 1 \tag{28} $$

接受概率为 1, 即**必然接受**. 这符合我们的直觉：目标模型认为 "A" 的概率(0.6)比草稿模型认为的(0.5)还要高. 也就是说, 草稿模型在这个 token 上"偏保守"了, 目标模型完全认可这个选择. 

### 5.3 拒绝场景分析：如果草稿采样到 "B"

让我们换一个场景, 假设草稿模型这次采样到的是 "B"(概率为 0.3)：

$$\alpha = \min\left(1, \frac{p(B)}{q(B)}\right) = \min\left(1, \frac{0.2}{0.3}\right) = \min(1, 0.667) \approx 0.667 \tag{29} $$
接受概率为 2/3. 这意味着：
- 以 66.7% 的概率接受 "B"; 
- 以 33.3% 的概率拒绝 "B", 并从修正分布中重新采样. 

为什么不是必然拒绝？因为虽然目标模型认为 "B" 的概率(0.2)低于草稿模型的估计(0.3), 但这并不意味着 "B" 一定是错的——它仍然在目标分布中占有 20% 的概率质量. 投机解码的机制是：以与两个分布的"置信度比值"成比例的概率来接受它. 

### 5.4 修正分布的采样

假设 "B" 被拒绝了(发生了 33.3% 概率的事件). 我们需要从修正分布中采样替代 token. 

修正分布的定义为：

$$p'(x) = \frac{\max(0, p(x) - q(x))}{\sum_{x'} \max(0, p(x') - q(x'))} \tag{30} $$

首先计算分子 $\max(0, p(x) - q(x))$：

| Token | $p(x)$ | $q(x)$ | $p(x) - q(x)$ | $\max(0, p - q)$ |
|-------|--------|--------|---------------|-----------------|
| A     | 0.6    | 0.5    | 0.1           | **0.1**         |
| B     | 0.2    | 0.3    | -0.1          | 0               |
| C     | 0.2    | 0.2    | 0.0           | 0               |

分母(归一化因子)为：$0.1 + 0 + 0 = 0.1$

因此修正分布为：

| Token | $p'(x)$ |
|-------|---------|
| A     | 0.1 / 0.1 = **1.0** |
| B     | 0 / 0.1 = 0.0       |
| C     | 0 / 0.1 = 0.0       |

修正分布将所有概率质量都赋予了 "A"！这非常合理：目标模型相对于草稿模型, 唯一"更有信心"的 token 就是 "A"(目标认为 0.6 > 草稿认为 0.5). 对于 "B", 草稿模型高估了它; 对于 "C", 两个模型看法一致. 因此当 "B" 被拒绝时, 唯一合理的替代就是 "A". 

### 5.5 输出分布的等价性验证

现在我们来验证这个单步过程的输出分布确实等于 $p$. 

输出 token $x^*$ 的来源有两条路径：

**路径 1：草稿采样到 $v$, 且被接受**

对于每个 $v \in \{A, B, C\}$：

- **$v = A$**：$q(A) = 0.5$, $\alpha_A = 1$, 贡献 = $0.5 \times 1 = 0.5$
- **$v = B$**：$q(B) = 0.3$, $\alpha_B = 2/3$, 贡献 = $0.3 \times 2/3 = 0.2$
- **$v = C$**：$q(C) = 0.2$, $\alpha_C = \min(1, 0.2/0.2) = 1$, 贡献 = $0.2 \times 1 = 0.2$

**路径 2：草稿采样到某个 $\hat{x}$, 被拒绝, 修正采样到 $v$**

我们需要计算总的拒绝概率, 以及修正采样到每个 $v$ 的概率. 

总的拒绝概率：
$$P(\text{reject}) = q(A)(1-\alpha_A) + q(B)(1-\alpha_B) + q(C)(1-\alpha_C) \tag{31} $$
$$= 0.5 \times 0 + 0.3 \times \frac{1}{3} + 0.2 \times 0 = 0.1 \tag{32} $$

注意这个值正好等于 $\sum_{x'} \max(0, p(x') - q(x')) = 0.1$(即之前计算的归一化因子), 验证了理论的一致性. 

修正分布我们已计算：$p'(A) = 1.0$, $p'(B) = 0$, $p'(C) = 0$. 

因此路径 2 对输出分布的贡献为：
- $v = A$：$P(\text{reject}) \times p'(A) = 0.1 \times 1.0 = 0.1$
- $v = B$：$P(\text{reject}) \times p'(B) = 0.1 \times 0 = 0$
- $v = C$：$P(\text{reject}) \times p'(C) = 0.1 \times 0 = 0$

**合并两条路径**：

| Token | 路径 1 贡献 | 路径 2 贡献 | 总输出概率 $P_{spec}(v)$ | 目标分布 $p(v)$ |
|-------|------------|------------|------------------------|----------------|
| A     | 0.5        | 0.1        | **0.6**                | 0.6 ✓          |
| B     | 0.2        | 0          | **0.2**                | 0.2 ✓          |
| C     | 0.2        | 0          | **0.2**                | 0.2 ✓          |

完美匹配！这个具体的数值例子严格验证了我们之前通过符号推导证明的结论：投机解码的输出分布与目标模型的自回归分布完全一致. 

### 5.6 期望加速比的数值计算

现在我们来计算不同接受率和草稿长度组合下的期望加速比. 

假设：
- 目标模型单次单 token forward 耗时为 1 单位
- 草稿模型单次单 token forward 耗时为 0.1 单位(草稿模型比目标模型快 10 倍)
- 验证一个长度为 $k$ 的序列耗时为 $0.5 + 0.1k$ 单位(有固定开销 + 线性增长)

标准解码生成 $k$ 个 token 的总耗时：$k \times 1 = k$ 单位. 

投机解码单次迭代耗时：$k \times 0.1 + (0.5 + 0.1k) = 0.2k + 0.5$ 单位. 

单次迭代期望产出：$\mathbb{E}[A] = \sum_{i=1}^{k} \alpha^i$. 

**表格：不同 $\alpha$ 和 $k$ 下的期望加速比**

| 接受率 $\alpha$ | 草稿长度 $k$ | 期望接受数 $\mathbb{E}[A]$ | 迭代耗时 | 每 token 耗时 | 加速比 |
|----------------|-------------|------------------------|---------|-------------|--------|
| 0.90           | 3           | $0.9 + 0.81 + 0.729 = 2.439$ | 1.1     | 0.451         | **2.22x** |
| 0.90           | 5           | $0.9 + 0.81 + 0.729 + 0.656 + 0.590 = 3.686$ | 1.5     | 0.407         | **2.46x** |
| 0.90           | 8           | $\approx 4.78$              | 2.1     | 0.439         | **2.28x** |
| 0.75           | 3           | $0.75 + 0.5625 + 0.422 = 1.734$ | 1.1     | 0.634         | **1.58x** |
| 0.75           | 5           | $0.75 + 0.5625 + 0.422 + 0.316 + 0.237 = 2.288$ | 1.5     | 0.656         | **1.52x** |
| 0.75           | 8           | $\approx 2.50$              | 2.1     | 0.840         | **1.19x** |
| 0.60           | 3           | $0.6 + 0.36 + 0.216 = 1.176$ | 1.1     | 0.935         | **1.07x** |
| 0.60           | 5           | $0.6 + 0.36 + 0.216 + 0.130 + 0.078 = 1.384$ | 1.5     | 1.084         | **0.92x** |
| 0.60           | 8           | $\approx 1.50$              | 2.1     | 1.400         | **0.71x** |

从这张表格中我们可以提炼出几个关键洞察：

1. **高接受率至关重要**：当 $\alpha = 0.90$ 时, 即使草稿长度只有 3, 也能获得超过 2 倍的加速. 而当 $\alpha$ 降到 0.60 时, 投机解码反而可能变慢(加速比 < 1). 

2. **草稿长度存在最优值**：对于固定的接受率, 加速比并不是随 $k$ 单调递增的. 以 $\alpha = 0.90$ 为例, $k = 5$ 时加速比最高(2.46x), 而 $k = 8$ 时反而下降到 2.28x. 这是因为草稿阶段的线性开销($k \cdot c_d$)和验证阶段的线性增长最终超过了额外草稿 token 带来的边际收益. 

3. **存在一个"死亡区域"**：当接受率过低时(如 $\alpha = 0.60, k = 8$), 投机解码不仅不能加速, 反而会显著拖慢推理. 这是因为大量计算浪费在了生成和验证最终被拒绝的草稿 token 上.

## 6. 简化实现 (PyTorch Code)

理论必须落地为代码才有工程价值. 本节提供两个简化但完整的 PyTorch 实现：一个是传统投机解码的主循环, 另一个是 Medusa 风格的多解码头实现. 代码力求在 100 行左右展示核心逻辑, 并在注释中严格标注对应的数学公式. 

### 6.1 传统投机解码主循环

```python
import torch
import torch.nn.functional as F

def speculative_decoding_step(
    prefix_ids,           # [1, L] 已确认的前缀 token IDs
    draft_model,          # 草稿模型(小模型)
    target_model,         # 目标模型(大模型)
    k=4,                  # 草稿长度
    temperature=1.0,      # 采样温度
):
    """
    单次投机解码迭代：Draft -> Verify -> Accept/Reject
    对应论文：Fast Inference from Transformers via Speculative Decoding
    返回：新接受的 token IDs, 以及下一轮的前缀
    """
    device = prefix_ids.device
    
    # ========== 步骤 1: Draft(起草)==========
    # 对应数学公式: \hat{x}_t \sim q(\cdot | x_{<t})
    draft_ids = prefix_ids.clone()
    draft_tokens = []      # 保存采样的 token IDs
    draft_probs = []       # 保存采样概率 q(\hat{x})
    
    draft_model.eval()
    with torch.no_grad():
        for _ in range(k):
            logits = draft_model(draft_ids).logits[:, -1, :]  # 只取最后一个位置
            probs = F.softmax(logits / temperature, dim=-1)
            
            # 采样一个 token
            next_token = torch.multinomial(probs, num_samples=1)  # [1, 1]
            token_prob = probs.gather(-1, next_token)  # q(\hat{x})
            
            draft_tokens.append(next_token)
            draft_probs.append(token_prob)
            draft_ids = torch.cat([draft_ids, next_token], dim=1)
    
    # draft_ids 现在包含: [前缀, \hat{x}_1, \hat{x}_2, ..., \hat{x}_k]
    # draft_tokens: list of [1,1] tensors
    # draft_probs: list of [1,1] tensors (采样概率 q)
    
    # ========== 步骤 2: Verify(验证)==========
    # 对应数学公式: p_i(x) = p(x | x_{<t}, \hat{x}_1, ..., \hat{x}_{i-1})
    target_model.eval()
    with torch.no_grad():
        # 一次性输入前缀 + k 个草稿 token
        # target_logits 的形状: [1, L+k, V]
        target_logits = target_model(draft_ids).logits
        
        # 取草稿位置的 logits(即前缀之后的 k 个位置)
        # target_logits[:, L-1:L-1+k, :] 对应 k 个位置
        L = prefix_ids.shape[1]
        draft_logits = target_logits[:, L-1:L-1+k, :]  # [1, k, V]
        draft_target_probs = F.softmax(draft_logits / temperature, dim=-1)
    
    # ========== 步骤 3: Accept/Reject(接受/拒绝)==========
    # 对应数学公式: \alpha_i = min(1, p_i(\hat{x}_i) / q_i(\hat{x}_i))
    accepted_tokens = []
    n_accepted = 0
    
    for i in range(k):
        # 目标模型对第 i 个草稿 token 的概率
        p_i = draft_target_probs[0, i, draft_tokens[i][0, 0]]  # p_i(\hat{x}_i)
        q_i = draft_probs[i][0, 0]  # q_i(\hat{x}_i)
        
        # 计算接受率
        alpha = torch.min(torch.tensor(1.0, device=device), p_i / q_i)
        
        # 以概率 alpha 接受
        if torch.rand(1, device=device).item() < alpha.item():
            accepted_tokens.append(draft_tokens[i])
            n_accepted += 1
        else:
            # ========== 被拒绝：从修正分布采样 ==========
            # 对应数学公式: p'(x) \propto max(0, p(x) - q(x))
            # 这里我们需要在当前前缀下重新计算目标分布和草稿分布
            
            current_prefix = torch.cat([prefix_ids] + accepted_tokens, dim=1)
            
            with torch.no_grad():
                # 重新计算两个模型在当前前缀下的分布
                q_logits = draft_model(current_prefix).logits[:, -1, :]
                q_dist = F.softmax(q_logits / temperature, dim=-1)
                
                p_logits = target_model(current_prefix).logits[:, -1, :]
                p_dist = F.softmax(p_logits / temperature, dim=-1)
            
            # 修正分布
            corrected = torch.clamp(p_dist - q_dist, min=0.0)
            if corrected.sum() > 0:
                corrected = corrected / corrected.sum()
                replacement = torch.multinomial(corrected, num_samples=1)
            else:
                # 退化情况：直接从 p 采样
                replacement = torch.multinomial(p_dist, num_samples=1)
            
            accepted_tokens.append(replacement)
            break  # 第一个拒绝就停止
    
    # 如果所有 k 个都被接受, 需要从目标模型额外采样一个
    if n_accepted == k:
        current_prefix = torch.cat([prefix_ids] + accepted_tokens, dim=1)
        with torch.no_grad():
            extra_logits = target_model(current_prefix).logits[:, -1, :]
            extra_probs = F.softmax(extra_logits / temperature, dim=-1)
            extra_token = torch.multinomial(extra_probs, num_samples=1)
        accepted_tokens.append(extra_token)
    
    # 构建新的前缀
    if accepted_tokens:
        new_prefix = torch.cat([prefix_ids] + accepted_tokens, dim=1)
    else:
        new_prefix = prefix_ids
    
    return new_prefix, n_accepted


def speculative_generate(prompt_ids, draft_model, target_model, max_new_tokens=100, k=4):
    """
    完整的投机解码生成循环
    """
    prefix = prompt_ids
    total_accepted = 0
    n_iterations = 0
    
    while prefix.shape[1] - prompt_ids.shape[1] < max_new_tokens:
        prefix, n_acc = speculative_decoding_step(prefix, draft_model, target_model, k=k)
        total_accepted += n_acc
        n_iterations += 1
        
        if n_acc == 0:
            # 退化情况：没有任何 token 被接受, 回退到标准解码一步
            with torch.no_grad():
                logits = target_model(prefix).logits[:, -1, :]
                probs = F.softmax(logits, dim=-1)
                token = torch.multinomial(probs, num_samples=1)
            prefix = torch.cat([prefix, token], dim=1)
    
    return prefix
```

### 6.2 Medusa 风格多解码头实现

```python
import torch
import torch.nn as nn
import torch.nn.functional as F

class MedusaHead(nn.Module):
    """
    Medusa 解码头：从隐藏状态预测第 i 个未来 token
    对应论文：Medusa: Simple LLM Inference Acceleration Framework
    """
    def __init__(self, hidden_size, vocab_size, num_layers=1):
        super().__init__()
        layers = []
        for _ in range(num_layers):
            layers.extend([
                nn.Linear(hidden_size, hidden_size),
                nn.GELU(),
            ])
        layers.append(nn.Linear(hidden_size, vocab_size))
        self.head = nn.Sequential(*layers)
    
    def forward(self, hidden_states):
        """
        hidden_states: [batch, seq_len, hidden_size]
        返回: [batch, seq_len, vocab_size] 的 logits
        """
        return self.head(hidden_states)


class MedusaModel(nn.Module):
    """
    包装器：为目标模型添加 k 个 Medusa 解码头
    """
    def __init__(self, base_model, num_heads=4):
        super().__init__()
        self.base_model = base_model
        self.num_heads = num_heads
        hidden_size = base_model.config.hidden_size
        vocab_size = base_model.config.vocab_size
        
        # k 个独立的解码头, 分别预测未来 1, 2, ..., k 个 token
        self.medusa_heads = nn.ModuleList([
            MedusaHead(hidden_size, vocab_size) for _ in range(num_heads)
        ])
    
    def forward(self, input_ids, return_medusa=True):
        """
        前向传播同时计算原始 logits 和 Medusa logits
        """
        outputs = self.base_model(input_ids, output_hidden_states=True)
        hidden_states = outputs.hidden_states[-1]  # 最后一层 [batch, seq, hidden]
        
        if not return_medusa:
            return outputs.logits, None
        
        # 每个头预测对应未来位置的 logits
        # medusa_logits[i] 的形状: [batch, seq, vocab]
        # 注意：位置 t 的隐藏状态预测的是位置 t+i+1 的 token
        medusa_logits = [head(hidden_states) for head in self.medusa_heads]
        
        return outputs.logits, medusa_logits
    
    def generate_draft(self, prefix_ids, temperature=1.0):
        """
        使用 Medusa 头生成草稿 token
        注意：这里简化为贪心采样, 实际可用 top-k / top-p
        """
        self.eval()
        with torch.no_grad():
            base_logits, medusa_logits = self.forward(prefix_ids, return_medusa=True)
            
            # 取最后一个位置的隐藏状态对应的预测
            drafts = []
            for i, m_logits in enumerate(medusa_logits):
                probs = F.softmax(m_logits[:, -1, :] / temperature, dim=-1)
                token = torch.argmax(probs, dim=-1, keepdim=True)  # 贪心
                drafts.append(token)
        
        return drafts  # list of [batch, 1] token IDs


def medusa_verify(base_logits, medusa_tokens, medusa_logits_list, temperature=1.0):
    """
    Medusa 验证过程：并行验证多个头生成的草稿 token
    
    base_logits: [batch, seq, vocab] 目标模型在完整序列上的 logits
    medusa_tokens: list of [batch, 1] 草稿 token
    medusa_logits_list: list of [batch, seq, vocab] Medusa 头的 logits
    """
    device = base_logits.device
    accepted = []
    
    # 这里简化处理：直接用原始头的对应位置 logits 验证
    # 实际实现中需要构建树形 attention mask 来并行验证
    for i, draft_tok in enumerate(medusa_tokens):
        # 计算目标分布中该 token 的概率
        pos = base_logits.shape[1] - len(medusa_tokens) + i
        if pos < 0:
            break
            
        p_dist = F.softmax(base_logits[:, pos, :] / temperature, dim=-1)
        p_token = p_dist.gather(-1, draft_tok)
        
        # Medusa 头的采样概率(这里简化为最大概率)
        m_dist = F.softmax(medusa_logits_list[i][:, -1, :] / temperature, dim=-1)
        q_token = m_dist.gather(-1, draft_tok)
        
        alpha = torch.min(torch.tensor(1.0, device=device), p_token / (q_token + 1e-10))
        
        if torch.rand(1, device=device).item() < alpha.item():
            accepted.append(draft_tok)
        else:
            # 从修正分布采样或直接从 p_dist 采样
            # 简化：直接从 p_dist 采样
            replacement = torch.multinomial(p_dist, num_samples=1)
            accepted.append(replacement)
            break
    
    return accepted
```

> **代码-理论对应说明**：
> - `speculative_decoding_step` 中的 `for _ in range(k)` 循环严格对应数学公式中的 $k$ 次草稿采样 $\hat{x}_t \sim q(\cdot \mid x_{<t})$. 
> - `alpha = min(1, p_i / q_i)` 直接实现了接受率公式 $\alpha_i = \min(1, p_i(\hat{x}_i) / q_i(\hat{x}_i))$. 
> - `corrected = clamp(p_dist - q_dist, min=0)` 是修正分布 $p'(x) \propto \max(0, p(x) - q(x))$ 的直接代码实现. 
> - 当 `n_accepted == k` 时额外采样一个 token, 是因为如果所有 $k$ 个草稿都被接受, 目标模型实际上只"确认"了这些 token, 并没有在最后一个被接受位置之后进行真正的"生成". 按照原始论文的建议, 此时需要从目标模型额外采样一个 token 来保证过程的完整性. 

## 7. 局限性与边界条件 (Limitations & Boundary Conditions)

世界上没有包治百病的算法, 投机解码虽然在理论上优美、在实践中有效, 但它同样面临着一系列严峻的局限性和边界条件. 深入理解这些限制, 对于在生产环境中正确部署和调优投机解码至关重要. 

### 7.1 草稿模型的选择困境：分布兼容性

投机解码的加速效果高度依赖于接受率 $\alpha$, 而接受率又直接取决于草稿模型 $q$ 与目标模型 $p$ 的分布相似度. 如果两个模型的分布差异过大, $p(\hat{x}) / q(\hat{x})$ 的比率会频繁地远小于 1, 导致大量的拒绝和修正采样, 反而拖慢推理速度. 

**兼容性的数学度量**：

接受率与两个分布之间的"重叠度"密切相关. 在期望意义上, 单步接受概率可以近似表示为：

$$\mathbb{E}[\alpha] \approx 1 - D_{TV}(p, q) \tag{33} $$
其中 $D_{TV}(p, q) = \frac{1}{2} \sum_x |p(x) - q(x)|$ 是总变差距离. 这意味着两个分布越接近(总变差距离越小), 期望接受率越高. 当 $p = q$ 时, $D_{TV} = 0$, 接受率为 100%. 

**工程实践中的困境**：

1. **模型族匹配**：通常草稿模型需要与目标模型属于同一个模型族(如都用 Llama 架构), 并且在相似的数据上训练. 你不能用一个在代码上训练的模型去为文学创作的模型做草稿——它们的分布差异太大. 

2. **规模比例**：草稿模型通常需要比目标模型小一个数量级(如 7B 为 70B 做草稿). 但"小"本身并不能保证分布相似. 一个训练不充分的小模型, 其分布可能与目标模型差异极大. 

3. **任务敏感性**：不同任务对分布差异的敏感度不同. 在开放式创意写作中, 接受率可能只有 0.5-0.6; 而在代码生成或结构化输出中, 如果草稿模型也经过代码训练, 接受率可能达到 0.8-0.9. 

4. **动态变化**：即使在同一个任务中, 接受率也会随着生成内容的推进而动态变化. 例如, 在生成一段技术性内容时接受率可能很高, 但突然转到引用一段诗歌时, 接受率可能骤降. 

### 7.2 短序列的 Overhead 陷阱

投机解码并非在所有场景下都能获益. 对于一个只生成 5-10 个 token 的短序列(如简单的问答、分类任务), 投机解码的固定开销可能超过其收益. 

**固定开销来源**：

1. **Draft 阶段**：即使只需要生成几个 token, 草稿模型仍然需要先执行一次 forward pass 来生成候选. 

2. **验证阶段**：验证需要一次目标模型的 forward pass, 其序列长度虽然可以并行处理, 但 Attention 的内存访问模式与单 token 解码不同. 

3. **调度开销**：投机解码引入了额外的控制流(接受/拒绝判断、修正采样、状态管理), 这些在短序列上构成了不可忽略的比例. 

**经验法则**：

在实际工程中, 投机解码通常只在预期生成长度超过 20-30 个 token 时启用. 对于更短的序列, 直接回退到标准自回归解码往往更划算. 一些先进的推理系统(如 vLLM)会在运行时动态检测生成长度, 并自适应地决定是否启用投机解码. 

### 7.3 确定性任务的"悬崖效应"

投机解码在开放式、高熵的生成任务(如创意写作、头脑风暴)中表现良好, 因为这类任务中"合理"的 token 很多, 草稿模型很容易猜中目标模型也会选择的 token. 但在**低熵、高确定性**的任务中, 投机解码可能遭遇"悬崖效应". 

**代码生成中的语法边界**：

考虑代码生成场景. 假设目标模型正在生成一个 Python 函数的括号结构：

```python
def calculate(x, y):
```

在 `y)` 之后, 下一个 token **几乎必然**是 `:`(冒号), 然后是换行符和缩进. 这是一个极低熵的决策点——目标模型对这个位置的概率分布可能集中在 `:` 上(概率 0.99). 

但如果草稿模型在这个位置上稍有偏差——比如它猜测下一个 token 是空格或者注释符号——那么接受率 $p(:)/q(:)$ 可能会非常小(因为 $q(:)$ 可能只有 0.3 或更低), 导致该位置几乎必然被拒绝. 而一旦拒绝, 修正采样几乎必然还是采样到 `:`(因为这是目标分布的众数), 这意味着我们浪费了一次草稿-验证的完整循环, 只为确认一个本可以直接生成的 token. 

在代码生成、数学公式生成、结构化数据输出等场景中, 这种"语法边界"频繁出现. 每一个括号、引号、缩进、关键字都是低熵的决策点, 草稿模型稍有偏差就会触发拒绝, 导致投机解码在这些高频边界上反复"摔跤". 

### 7.4 显存开销：双模型的代价

传统投机解码需要同时在 GPU 显存中驻留两个模型：目标模型和草稿模型. 对于当前的大模型部署环境, 这是一个不可忽视的成本. 

**显存计算的残酷现实**：

- 一个 70B 参数的 FP16 模型需要约 140GB 显存(仅权重); 
- 如果使用 7B 参数的草稿模型, 额外需要约 14GB 显存; 
- 加上 KV Cache、激活值、优化器状态等, 总显存需求可能超过 200GB. 

在大多数生产环境中, GPU 显存是稀缺资源. 部署投机解码意味着你可能需要将 batch size 减半, 或者使用更多的 GPU 来做模型并行——这些都会抵消一部分加速收益. 

**缓解方案**：

1. **量化草稿模型**：将草稿模型量化为 INT8 或 INT4, 大幅降低其显存占用. 由于草稿模型本身对精度要求不如目标模型严格, 量化带来的分布偏移通常在可接受范围内. 

2. **CPU Offloading**：将草稿模型放在 CPU 内存甚至 SSD 上, 只在需要时将其激活加载到 GPU. 虽然这会增加草稿阶段的延迟, 但如果草稿模型足够小且 CPU-GPU 传输带宽足够, 总体仍然可能获益. 

3. **Medusa 方案**：如 4.4 节所述, Medusa 完全避免了第二个模型, 只增加了少量解码头参数(通常不到原模型参数量的 1%), 显存开销极小. 

### 7.5 Batch Size > 1 时的复杂性爆炸

当 batch size 大于 1 时(即同时处理多个用户的请求), 投机解码的复杂性显著增加. 

**问题 1：不同序列的接受长度不同**

在一个 batch 中, 序列 A 可能接受了 4 个草稿 token, 序列 B 可能只接受了 1 个, 序列 C 可能全部被拒绝. 这意味着三个序列在验证后的长度不一致. 为了继续下一轮解码, 我们需要对 batch 进行重新对齐(padding 或 truncation), 这引入了额外的内存拷贝和计算开销. 

**问题 2：验证阶段的 Attention Mask 复杂度**

在 batch 验证时, Attention Mask 需要同时处理多个序列的不同草稿长度. 树形注意力在 batch 场景下的实现尤其复杂——不同序列可能生成不同形状的候选树, 难以高效地 batch 在一起. 

**问题 3：KV Cache 的管理噩梦**

每个序列在解码过程中都有自己的 KV Cache. 当序列的接受长度不同时, 它们的 KV Cache 增长速度也不同. 在连续批处理(Continuous Batching)系统中, 新请求不断加入、已完成请求不断退出, 不同长度的 KV Cache 使得内存管理变得极其复杂. 

目前主流框架(如 vLLM)通过以下方式缓解这些问题：
- **固定草稿长度**：强制所有序列使用相同的 $k$, 简化 batching; 

- ** Early Stopping**：如果 batch 中大多数序列都已拒绝, 提前终止当前迭代; 

- **独立调度**：将投机解码的验证阶段与正常的连续批处理解耦, 用独立的调度器管理. 

但无论如何, batch size > 1 时的投机解码, 其实现复杂度和 overhead 都远超单序列场景, 这是生产部署中需要谨慎评估的重要因素. 

## 8. 演进与承上启下 (Evolution & Segue)

### 8.1 与 PagedAttention 的深度融合

投机解码和 PagedAttention 是解决 LLM 推理瓶颈的两个正交方向：PagedAttention 优化内存管理和 batch 调度, 投机解码优化单序列的解码效率. 两者的结合是当前推理引擎的标配. 

在 vLLM 的架构中, PagedAttention 负责将 KV Cache 分块存储在非连续的物理内存页中, 支持高效的内存共享和动态分配. 当投机解码验证阶段需要并行处理长度为 $k$ 的序列时, PagedAttention 可以灵活地为草稿 token 分配临时的内存页, 验证完成后只保留被接受 token 对应的页, 释放被拒绝 token 占用的页. 

这种结合的深层意义在于：**投机解码打破了"单 token 单步"的串行假设, 而 PagedAttention 打破了"连续内存分配"的空间假设**. 两者共同推动了 LLM 推理从"朴素的串行+连续"范式, 向"并行的、内存灵活的多 token 推理"范式演进. 

### 8.2 与量化技术的正交叠加

量化技术(如 GPTQ、AWQ、FP8)通过降低权重精度来减少内存带宽和计算量. 投机解码与量化是正交的：

- 你可以将目标模型量化为 INT8, 同时保持草稿模型为 FP16; 
- 或者将草稿模型进一步量化为 INT4, 使其体积和推理速度都大幅优化; 
- 甚至可以将 Medusa 的额外解码头也进行量化. 

在实际部署中, 一个常见的配置是：目标模型用 INT8 量化保证精度, 草稿模型用 INT4 量化追求极致速度. 由于草稿模型的输出只用于"猜测", 其精度损失对最终输出质量的影响经过目标模型的验证后被完全消除. 

### 8.3 从外挂投机到内建多 token 预测

投机解码及其变体(Medusa、Lookahead)本质上都是**外挂式**的优化：它们在模型训练完成之后, 通过额外的机制来加速推理. 但研究界已经开始探索更根本的解决方案——**在训练阶段就让模型内建多 token 预测的能力**. 

**Multi-Token Prediction (MTP)** 训练目标已经在一些最新的模型中被采用. 与传统的"预测下一个 token"不同, MTP 要求模型同时预测未来 $k$ 个 token. 这意味着模型的隐藏状态被训练成包含了对未来多个位置的信息, 从而在推理时可以直接输出多个 token, 而不需要外挂的草稿模型或多解码头. 

Moonshot AI 的 Moonlight 模型、DeepSeek 的某些变体, 都在训练阶段引入了 MTP 目标. 这种"原生多 token 预测"的模型在推理时可以自然地一次输出多个 token, 投机解码的 draft 阶段被内建到了模型自身的参数中. 

**未来方向的展望**：

1. **端到端的多 token 架构**：未来的 Transformer 变体可能会从根本上改变"一次预测一个 token"的架构设计. 例如, 通过层次化的预测结构(先预测句子骨架, 再填充细节), 或者通过更激进的并行解码架构. 

2. **自适应投机深度**：当前投机解码使用固定的草稿长度 $k$. 未来的系统可能会在每个解码步骤动态调整 $k$——在高确定性区域(如代码语法边界)使用较小的 $k$ 甚至关闭投机, 在高熵区域(如开放式写作)使用较大的 $k$. 

3. **与推理时计算扩展的结合**：随着 Test-Time Compute(如 Chain-of-Thought、Self-Consistency)成为提升模型能力的主流手段, 推理阶段的计算量进一步爆炸. 投机解码及其后续演进, 将在控制推理成本方面扮演越来越重要的角色. 

投机解码不只是一个加速技巧, 它代表了 LLM 推理领域从"忠实复现自回归"到"智能地打破自回归"的范式转变. 下一篇文章, 我们将深入探讨 **Medusa 的完整训练与推理体系**, 剖析树形注意力的 CUDA Kernel 级优化, 以及如何在生产环境中将投机解码的加速比推向理论上限. 

## 9. 总结与参考文献 (References)

### 9.1 核心要点总结

1. **自回归的串行瓶颈**：标准解码每步只产 1 个 token, $T$ 次 forward pass 的总计算量和内存带宽开销构成了 LLM 推理延迟的主要来源. 

2. **Draft-then-Verify 的优雅思想**：先用小模型快速猜测未来 $k$ 个 token, 再用大模型一次性并行验证. 关键是验证机制保证输出分布严格等于大模型的自回归分布——**零精度损失**. 

3. **接受/拒绝的数学核心**：接受率 $\alpha_i = \min(1, p_i(\hat{x}_i)/q_i(\hat{x}_i))$ 配合修正分布 $p' \propto \max(0, p - q)$, 严格保持了目标分布, 这是投机解码"免费午餐"的数学根基. 

4. **期望加速比**：在高接受率($\alpha \approx 0.8-0.9$)和合适的草稿长度($k \approx 4-6$)下, 实际加速比可达 2-3 倍. 但存在"死亡区域"——低接受率或过大 $k$ 会导致减速. 

5. **变体演进**：从传统的双模型投机, 到 Medusa 的单模型多解码头, 再到 Lookahead 和 Prompt Lookup 的无模型方案, 投机解码的家族在不断降低部署门槛. 

6. **工程边界**：草稿模型选择、短序列 overhead、确定性任务的悬崖效应、显存开销、batch > 1 的复杂性, 都是生产部署中必须面对的工程挑战.
