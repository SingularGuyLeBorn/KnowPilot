# 6.2.1 大语言模型显存与计算量深度解析：推理与训练优化指南


## 1  大模型显存占用基础

理解大模型的显存占用首先需要掌握数据在计算机中的存储方式以及模型参数的构成. 显存是GPU上的高速存储器，其容量直接决定了模型能够加载和处理的数据规模.

### 1.1 数据精度与显存换算

数据精度指的是用于表示一个数值的比特 (bit) 数量. 不同的精度类型决定了每个参数占用的存储空间，进而影响总体的显存需求.

- **位、字节与内存单位换算**:- 1 Byte (字节) = 8 bits (比特)- 1 KB (千字节) = 1,024 Bytes- 1 MB (兆字节) = 1,024 KB- 1 GB (千兆字节) = 1,024 MB

这意味着，一个含有 10 亿参数 (1 Billion, 1B) 的模型，如果每个参数都是 32 位 (4 Bytes)，那么仅加载模型本身就需要 $10^9 \times 4 \text{ Bytes} \approx 4 \text{ GB}$ 的显存.

- **常见浮点数精度类型**:- **FP32 (Single-Precision Floating-Point, 单精度浮点数)** : 使用 32 位表示，其中 1 位符号位，8 位指数位，23 位尾数位. 具有高精度和宽动态范围. 每个参数占用 4 Bytes.

- **FP16 (Half-Precision Floating-Point, 半精度浮点数)** : 使用 16 位表示，1 位符号位，5 位指数位，10 位尾数位. 相较于 FP32，其范围和精度都有所降低，但显存占用减半. 每个参数占用 2 Bytes.

- **BF16 (Brain Floating Point 16)** : 由 Google Brain 团队提出，同样使用 16 位表示，但其 1 位符号位，8 位指数位，7 位尾数位. BF16 的指数位与 FP32 相同，因此具有与 FP32 相似的数值范围，但在精度上低于 FP16. 这使其在深度学习训练中能够更好地处理大范围的梯度值，减少溢出问题. 每个参数占用 2 Bytes.

- **TF32 (TensorFloat-32)** : 并非标准的 32 位浮点数，它使用 19 位，其中 1 位符号位，8 位指数位，10 位尾数位. TF32 是 NVIDIA 针对其 Tensor Core 优化的数据类型，在保持 FP32 动态范围的同时，提供接近 FP16 的计算速度.

- **INT8 / INT4 (8位/4位整型)** : 用于模型量化 (Quantization)，可以显著减少模型大小. INT8 每个参数占用 1 Byte，INT4 每个参数占用 0.5 Bytes. 量化虽然降低了显存需求，但可能会对模型性能造成轻微损失.

- **浮点数二进制表示原理**:
浮点数的十进制值可以通过以下公式计算:

$\text{Value} = (-1)^{\text{sign}} \times 2^{\text{exponent} - \text{bias}} \times 1.\text{mantissa}$

其中:

- sign: 符号位，0 表示正数，1 表示负数.- exponent: 指数部分转换成的十进制值.- bias: 指数偏移量，例如 FP16 为 15，FP32 为 127，BF16 为 127.- 1.mantissa: 尾数部分，通常采用隐式小数的表示法，即默认存在一个隐含的 1，例如 1.0100000 表示 $1 + 0 \times 2^{-1} + 1 \times 2^{-2} = 1.25$.

**BF16 示例 (以原始材料中提供的计算为例)** :
假设有一个 BF16 的二进制数 0 10000011 0100000:

1. **符号位 (sign)** : 第 1 位是 0，表示正数.

2. **指数位 (exponent)** : 接下来 8 位是 10000011，转换为十进制是 131. BF16 的指数偏移量为 127.

$\text{实际指数} = 131 - 127 = 4$

1. **尾数位 (mantissa)** : 最后的 7 位是 0100000. 转换为小数部分 $0.25$ ($0 \times 2^{-1} + 1 \times 2^{-2} + 0 \times 2^{-3} \dots$).

2. **计算最终数值**: 将所有部分代入公式:

$\text{Value} = (-1)^0 \times 2^4 \times 1.25 = 1 \times 16 \times 1.25 = 20$

这个 BF16 二进制数 0 10000011 0100000 对应的十进制数是 20.

### 1.2 显存占用七大组成部分概述

在 GPU 上，大模型运行时的显存占用通常可以细分为以下七个主要部分:

1. **模型参数 (Model Parameters)** : 模型本身的权重和偏置项. 这部分是模型显存占用的基础，其大小由模型参数量和数据精度共同决定.

2. **梯度 (Gradients)** : 在训练过程中，为了更新模型参数，需要计算每个参数的梯度. 梯度的数量与模型参数数量相同，其精度通常与模型参数或优化器相关.

3. **优化器状态 (Optimizer States)** : 优化器 (如 AdamW, SGD) 在训练过程中会维护额外的状态变量，用于参数更新. 例如，AdamW 会维护每个参数的一阶矩 (momentum) 和二阶矩 (variance) 估计.

4. **中间激活 (Intermediate Activations)** : 在模型的前向传播过程中，每层的输出 (激活值) 需要被存储，以便在反向传播时计算梯度. 这部分显存占用与批量大小 (batch size) 和序列长度 (sequence length) 成正比.

5. **KV Cache (Key-Value Cache)** : 在自回归推理过程中，为了避免重复计算 Attention 机制中的键 (Key) 和值 (Value) 向量，它们会被缓存起来. KV Cache 的大小与序列长度、批量大小、模型层数和隐藏层维度等因素有关.

6. **功能性显存 (Functional Memory)** : 指的是 GPU 运行时环境、CUDA/cuDNN 库、以及其他框架或系统级别所需的显存. 这部分通常是固定的，且相对较小.

7. **临时显存 (Temporary Memory)** : 用于存储一些临时变量、缓冲区等，在计算完成后会立即释放. 例如，Beam Search 算法在推理时会额外占用一部分临时显存.

大模型推理时的显存占用主要由模型权重和 KV Cache 构成，同时还需考虑少量激活值和其他临时变量.

### 2.1  模型权重显存

模型权重是推理时显存占用的最基础部分，其大小取决于模型参数量 ($\Phi$) 和所采用的数据精度 (n Byte/参数).

$\text{模型权重显存 (Bytes)} = \Phi \times n$

不同精度下的占用量:

| dtype | 1B (10亿) 参数所需占用显存 |
| float32 (全精度) | 4 GB |
| bf16/fp16 (半精度) | 2 GB |
| int8 | 1 GB |
| int4 | 0.5 GB |
经验法则: 通常认为，如果模型参数量为 X (例如 7B)，采用 FP16 精度，则推理时加载模型至少需要 2X GB 显存.

### 2.2 KV Cache (键值缓存)

在自回归生成任务中，Transformer 模型会逐个生成 Token. 每生成一个 Token，都需要重新计算自注意力，涉及到当前 Token 与所有已生成 Token 之间的 Key (键) 和 Value (值) 向量. 为了避免重复计算，这些 K 和 V 向量会被缓存起来，这便是 KV Cache. KV Cache 显著提升了推理速度，但代价是会占用大量显存.

- **概念解释**:- **多头注意力 (Multi-Head Attention, MHA)** : 将查询 (Q)、键 (K) 和值 (V) 投影到多个子空间中，然后并行执行注意力计算，再将结果拼接.

- **分组 KV 头 (Grouped Key-Value Heads, GQA)** : GQA 是一种优化 MHA 的技术，它允许多个 Query 头共享同一组 Key 和 Value 头. 这种方式可以减少 KV Cache 的显存需求，因为它减少了需要缓存的 K 和 V 向量的总数. 如果模型配置中包含 num_key_value_heads 参数，通常 num_key_value_heads < num_attention_heads.

- **单 Token KV Cache 显存计算公式**:
假设:- hidden_size: 模型隐藏层维度.- num_attention_heads: 注意力头数量.- num_key_value_heads: KV 头数量 (如果未分组，等于 num_attention_heads).- num_hidden_layers: Transformer 层数.- kvcache_dtype_byte: KV Cache 数据类型对应的字节数 (例如 FP16/BF16 为 2 Bytes).- cache_num: K 和 V 两个缓存矩阵，所以为 2.

单个 Token 占用的 KV Cache 显存大小计算公式如下:

$\text{OneTokenKVMem} = \left(\frac{\text{hidden_size}}{\text{num_attention_heads}} \times \text{num_key_value_heads}\right) \times \text{num_hidden_layers} \times \text{kvcache_dtype_byte} \times \text{cache_num}$

需要注意的是， (hidden_size / num_attention_heads) 是单个注意力头的维度. 如果采用分组 KV 头，实际缓存的 K 和 V 维度是 (hidden_size / num_attention_heads) * num_key_value_heads.

在给定显卡总容量和模型大小的情况下，可以估算出模型所能支持的最大批量大小.

**估算方法**:

### 2.4 其他推理显存开销

除了模型权重和 KV Cache，推理过程中还会产生一些其他显存开销:

- **激活值**: 虽然推理时激活值通常不像训练时那样成为主要瓶颈，但在前向传播过程中，仍需存储一些中间激活值，尤其是在批量大小较大或 Beam Search 等复杂解码策略下.

- **临时变量**: 例如 Beam Search 算法会生成多个候选序列，这些都需要额外的显存空间. 经验估算，前向计算的开销通常在模型权重的 20% 左右.

因此，推理时的总显存占用大约是模型权重显存的 1.2 倍.

## 3  大模型训练显存占用

大模型的训练过程比推理消耗更多的显存，因为它不仅需要存储模型参数，还需要存储梯度、优化器状态和激活值.

![](https://i-blog.csdnimg.cn/blog_migrate/9acfebfe247239ef9872868327f158a1.gif)

### 3.1  核心组成部分

训练显存的主要组成部分如下:

1. **模型权重 (Model Parameters)** :
与推理时相同，根据模型参数量和精度占用显存.

- FP32: $\Phi \times 4 \text{ Bytes}$- FP16/BF16: $\Phi \times 2 \text{ Bytes}$

1. **梯度 (Gradients)** :
每个参数都有一个对应的梯度. 梯度的显存占用通常与模型权重的显存占用相同. 在混合精度训练中，梯度通常存储为 FP16.

- FP32 训练: $\Phi \times 4 \text{ Bytes}$- FP16/混合精度训练: $\Phi \times 2 \text{ Bytes}$

1. **优化器状态 (Optimizer States)** :
不同优化器维护的状态变量不同，因此显存占用也不同.

**AdamW/Adam**: 最常用，需要维护每个参数的一阶动量 (momentum) 和二阶动量 (variance) 估计. 这些状态通常以 FP32 精度存储，即使模型权重是 FP16. 因此，对于每个模型参数，AdamW 需要额外存储 2 个 FP32 状态变量，加上模型权重本身(若按FP32计)，总显存占用为模型参数量的 **3 倍** (2个状态变量 * 4 Bytes/变量 + 1个FP32模型权重 * 4 Bytes/权重). 如果模型权重本身是 FP16，那么优化器状态会是全精度模型权重的 **3 倍** (
$2 \times 4 \text{ Bytes/param} + 1 \times 2 \text{ Bytes/param}$
 for model param itself, which means $10 \text{ Bytes/param}$ total if model is FP16 and optimizer is FP32).显存占用 (FP32 AdamW, 模型 FP16): 
$\Phi \times (2 \times 4 + 2) \text{ Bytes} = \Phi \times 10 \text{ Bytes}$
显存占用 (FP32 AdamW, 模型 FP32): 
$\Phi \times (2 \times 4 + 4) \text{ Bytes} = \Phi \times 12 \text{ Bytes}$
- **bitsandbytes 优化的 AdamW**: 通过量化优化器状态，可以显著减少显存. 例如，将动量和二阶动量以 INT8 存储，每个参数只需占用 2 Bytes (1 Byte for momentum, 1 Byte for variance).

- **SGD**: 相对简单，通常只维护一个状态变量 (动量，如果启用的话). 其优化器状态的显存占用与模型权重显存相同 (即每个参数 1 个状态变量).显存占用 (FP32 SGD, 模型 FP16): 
$\Phi \times (1 \times 4 + 2) \text{ Bytes} = \Phi \times 6 \text{ Bytes}$
显存占用 (FP32 SGD, 模型 FP32): 
$\Phi \times (1 \times 4 + 4) \text{ Bytes} = \Phi \times 8 \text{ Bytes}$

**总结**: 使用 Adam 优化器训练 1B 全精度模型，所占显存约为 
$4\text{G (模型)} + 4\text{G (梯度)} + 8\text{G (优化器)} = 16\text{G}$
.
### 3.2 激活值 (Activations)

激活值是在前向传播过程中各层输出的中间结果，用于反向传播计算梯度. 激活值的显存占用量与模型的层数、设计、批量大小和序列长度成正比.

- **激活重计算 (Activation Checkpointing)** :
由于激活值在训练中会占用大量显存，激活重计算 (Activation Recomputation / Activation Checkpointing) 成为了一种重要的优化策略. 其核心思想是在前向传播时只存储部分关键层的激活值，而在反向传播需要时，再重新计算其他层的激活值. 这是一种以计算时间换取显存空间的策略.

- **无重计算**: 存储所有中间激活值.

$\text{内存} = (\text{s} \times \text{b} \times \text{h} \times \text{l}) \times \left(10 + \frac{24}{\text{t}} + 5 \times \frac{\text{a} \times \text{s}}{\text{h} \times \text{t}}\right) \text{ Bytes}$

- **选择性重计算**: 存储部分关键层的激活值.

$\text{内存} = (\text{s} \times \text{b} \times \text{h} \times \text{l}) \times \left(10 + \frac{24}{\text{t}}\right) \text{ Bytes}$

- **全部重计算**: 只存储输入和输出，反向传播时重新计算所有中间激活值.

$\text{内存} = 2 \times (\text{s} \times \text{b} \times \text{h} \times \text{l}) \text{ Bytes}$

其中:

- s: Token 长度 (序列长度)- b: 每个 GPU 的批量大小 (Batch Size)- h: 隐藏层维度 (Hidden Size)- l: 模型层数 (Number of Hidden Layers)- a: Transformer 模型中注意力头 (attention heads) 的个数- t: 张量并行度 (Tensor Parallelism)，如果无张量并行，则为 1.

激活重计算虽然节省显存，但会增加计算量，其上界是额外增加一次前向传播的计算成本.

### 3.3 训练数据与其他开销

- **训练数据**: 原始的输入数据在加载到 GPU 后也会占用显存. 批量大小越大、上下文长度越长，这部分显存占用越多.

- **其他**: CUDA 核心、驱动程序、操作系统等也会占用一部分显存.

**说明**:

- **全量微调**: 需要更新所有模型参数，因此显存开销最大，是模型参数量、梯度、优化器状态和激活值之和.

- **混合精度训练**: 通常指前向传播使用 FP16/BF16，反向传播使用 FP32 梯度和优化器状态. 这样可以兼顾速度和精度，显著降低显存需求.

- **LoRA**: 通过引入少量可训练的低秩适配器矩阵来微调大模型，冻结大部分原始模型参数. 这样只有 LoRA 适配器相关的参数、梯度和优化器状态需要存储和更新，极大节省显存.

- **QLoRA**: 在 LoRA 的基础上，将原始大模型的权重进一步量化到 4 位 (通常是 NF4 格式)，并将其存储在 GPU 或 CPU 上，从而进一步降低原始模型权重占用的显存.

- **PPO/DPO**: 在强化学习训练 (如 PPO, DPO) 中，除了目标 Actor 模型，可能还需要加载 Reference 模型、Reward 模型和 Critic 模型 (PPO) 或仅 Reference 模型 (DPO). 这些额外的模型会进一步增加显存占用.

## 4  大模型计算量估算 (FLOPs)

除了显存，计算量 (FLOPs, Floating-Point Operations) 是衡量模型运行效率和预估时延的关键指标. FLOPs 越高，完成计算所需的时间可能越长.

### 4.1 FLOPs 概念及重要性

- **FLOPs (floating-point operations)** : 指浮点运算次数，是一个衡量计算量的单位.

- **FLOPS (floating-point operations per second)** : 指每秒浮点运算次数，是衡量算力的单位.
虽然两者在很多地方常混用，但 FLOPs 是运算次数，FLOPS 是运算速度.

### 4.2 Transformer 层计算量构成

Transformer 模型的计算量主要集中在以下三个核心模块:

1. **自注意力机制 (Self-Attention)** :

- Q, K, V 投影: 输入 [S, H] 与权重 [H, H] 相乘 (近似 $3 \times S \times H \times 2H = 6SH^2$).- Q 与 K 转置相乘: 输入 [S, H] 与 [H, S] 相乘 (近似 $2S^2H$).- 注意力权重与 V 相乘: 输入 [S, S] 与 [S, H] 相乘 (近似 $2S^2H$).- 输出投影 (Output Projection): 输入 [S, H] 与权重 [H, H] 相乘 (近似 $2SH^2$).
因此，自注意力层的总计算量近似为 $8SH^2 + 4S^2H$.

1. **前馈网络 (Feed-Forward Network, FFN)** :
FFN 通常包含一个升维层、一个激活函数 (如 GELU/SiLU) 和一个降维层. 若考虑一个升维层 (输入 [S, H], 输出 [S, I]) 和一个降维层 (输入 [S, I], 输出 [S, H])，以及一个 gate 层 (通常与升维层参数量相同)，其计算量近似为 $2SHI + 2SHI + 2SIH = 6SHI$.

2. **LM Head (语言模型头)** :
最后的线性层，将隐藏层输出投影到词表维度. 计算量近似为 $2SHV$.

### 4.3 Prefill (编码阶段) 计算量

Prefill 阶段 (也称为编码阶段或首 Token 阶段) 是模型接收完整输入序列并生成首个输出 Token 的过程. 在这个阶段，输入序列长度 (S) 通常较长.

每个 Transformer 层的计算量为 $8SH^2 + 4S^2H + 6SHI$.
加上最后的 lm_head 层，Prefill 的总计算量为:

$\text{FLOPs}_{\text{Prefill}} = L(8SH^2 + 4S^2H + 6SHI) + 2SHV$

其中:

- L: Transformer 层数 (num_hidden_layers)- S: 输入序列长度 (input_length)- H: 隐藏层大小 (hidden_size)- I: FFN 中间层大小 (通常 $I \approx 4H$ 或根据模型配置)- V: 词表大小 (vocab_size)

### 4.4 Decode (生成阶段) 计算量

Decode 阶段是模型逐个生成后续 Token 的过程. 与 Prefill 阶段不同，Decode 阶段的输入序列长度每次为 1 (生成一个新 Token). 然而，自注意力机制需要考虑已缓存的所有历史 Key 和 Value.

- **自注意力层**:- Q, K, V 投影 (输入长度为 1): 近似 $6H^2$.- Q 与 K 转置相乘 (当前 Q 与缓存的 K): 输入 [1, H] 与 [H, T] (T 为当前序列总长度) 相乘，近似 $2HT$.- 注意力权重与 V 相乘: 输入 [1, T] 与 [T, H] 相乘，近似 $2HT$.
因此，自注意力层的计算量近似为 $8H^2 + 4HT$.

- **FFN 层**: 与 Prefill 类似，但输入长度为 1: 近似 $6HI$.

每个 Transformer 层的计算量为 $8H^2 + 4HT + 6HI$.
加上最后的 lm_head 层，Decode 的总计算量为:

$\text{FLOPs}_{\text{Decode}} = L(8H^2 + 4HT + 6HI) + 2HV$
其中 T 是当前序列的总长度 (Prefill 长度 + 已生成长度). 如果要计算 K 个 Decode 过程的平均耗时，T 可以取平均序列长度，例如 
$T = \frac{\text{input_length} + (\text{input_length} + \text{output_length})}{2}$

### 4.5 FLOPs 与实际时延差异分析

理论计算量 (FLOPs) 并不等同于实际运行时间. 实际时延受多种因素影响，导致理论值与实测值之间存在差异.

- **峰值算力与实际利用率**:
AI 芯片的标称算力通常是峰值算力. 实际运行中，芯片的算力利用率往往低于峰值，例如在 60% 左右.

- **Prefill 阶段**通常涉及大矩阵乘法 (GEMM, General Matrix Multiply)，可以充分利用 GPU 的计算能力，因此算力利用率较高.

- **Decode 阶段** (特别是 batch_size=1 时) 主要涉及小矩阵运算，甚至退化为矩阵向量乘法 (GEMV, General Matrix-Vector Multiply). GEMV 运算的算力利用率通常较低，可能不到 20%，因为 GPU 大部分时间都在等待数据搬运 (访存受限). 这解释了为何 Decode 阶段的理论计算量虽小，但实际时延可能远高于预期.

- **非计算耗时因素**:
除了核心计算时间，时延还包括:- **内存搬运时间**: 数据从主存到显存，或在显存内部不同区域之间的传输时间.

- **软件栈之间的数据传输时间**: 在复杂的推理框架中，数据在不同软件模块之间传递的开销.

- **其他开销**: 例如操作系统的调度、驱动程序开销等.
对于计算时间短的场景 (如 Decode 阶段)，这些非计算环节的时延占比会显著增加.

- **分布式推理中的通信开销**:
在分布式推理场景下，模型被分割到多个设备上，因此设备间的数据通信成为新的时延来源. 通信算子 (如 All-to-All) 的耗时往往较大，例如在 MoE (Mixture-of-Experts) 结构模型中，通信可能占总时延的 30%.

**示例计算与实测对比**:

- **Decode 预估**: 0.85ms，远小于实测的 18.77ms.

## 5 总结 (Conclusion)

大语言模型的显存与计算量估算在实际部署和训练中至关重要. 通过深入理解模型参数、数据精度、KV Cache、梯度、优化器状态和激活值等核心组成部分的显存占用机制，我们可以有效进行资源规划. 尤其是在推理阶段，KV Cache 的管理对显存效率至关重要; 而在训练阶段，混合精度、LoRA/QLoRA 等参数高效微调技术以及激活重计算是缓解显存压力的关键策略. 此外，精确估算 FLOPs 虽能提供理论计算量，但实际时延还需考虑硬件利用率、内存带宽和通信开销等因素. 掌握这些知识将帮助工程师更好地优化大模型的性能和成本效率.
