## 1. 为什么文本RoPE不能直接用于图像

文本序列是一维的：位置索引 $m$ 单调递增, 相邻token在空间上也相邻. 图像/视频数据则不同：

1. **多维位置**：图像像素有 $(h, w)$ 二维坐标, 视频有 $(t, h, w)$ 三维坐标
2. **模态碰撞**：若将 $(t, h, w)$ 展平为 $m' = t \cdot T + h \cdot H + w \cdot W$, 不同模态(文本vs图像)可能映射到同一索引
3. **结构丢失**：$(t, h, w)$ 的三维拓扑关系被压扁为一维顺序

## 2. 2D-RoPE：Qwen VL视觉塔

### 2.1 对半分配策略

将head_dim分为两段, 分别编码行和列信息. 

**配置走查**(Qwen2-VL, head_dim = 128)：

| 索引区间 | 内容 | 对应维度 |
|---------|------|---------|
| emb[0:32] | row的32个频率分量 | 行坐标 $r$ |
| emb[32:64] | col的32个频率分量 | 列坐标 $c$ |
| emb[64:96] | 再次row(与0:32相同) | 行坐标 $r$ |
| emb[96:128] | 再次col(与32:64相同) | 列坐标 $c$ |

**实现逻辑**：
- 前64个维度(emb[0:64])与后64个维度(emb[64:128])构成`rotate_half`所需的配对
- 配对 $[0, 64]$ 和 $[32, 96]$ 由行坐标决定相位
- 配对 $[16, 80]$ 和 $[48, 112]$ 等由列坐标决定相位

```python
def compute_2d_rope_embeddings(
    row_indices: torch.Tensor,
    col_indices: torch.Tensor,
    head_dim: int = 128,
    base: float = 10000.0
) -> tuple[torch.Tensor, torch.Tensor]:
    """计算2D RoPE嵌入(基于Qwen VL实现)
    
    维度分配：head_dim // 2 = 64对, 前32对编码row, 后32对编码col. 
    由于rotate_half配对方式, 实际构造为：
        freqs[:64] = row_freqs(复制一份以匹配配对)
        freqs[64:] = col_freqs(复制一份以匹配配对)
    """
    half_dim = head_dim // 2
    quarter_dim = head_dim // 4
    
    inv_freq = 1.0 / (base ** (torch.arange(0, half_dim, 2, dtype=torch.float) / half_dim))
    
    row_freqs = row_indices[:, None].float() * inv_freq[None, :quarter_dim]
    col_freqs = col_indices[:, None].float() * inv_freq[None, :quarter_dim]
    
    freqs = torch.cat([
        torch.cat([row_freqs, col_freqs], dim=-1),
        torch.cat([row_freqs, col_freqs], dim=-1)
    ], dim=-1)
    
    return freqs.cos(), freqs.sin()
```

**失效模式**：对极端宽高比的图像(如 $1 \times 10000$ 的条码), 对半分配会导致高度方向被过度编码、宽度方向编码不足, 位置碰撞概率增加.

## 3. 3D-RoPE：M-RoPE

### 3.1 时间维度的引入

Qwen2-VL采用的M-RoPE(Multimodal RoPE)将通道维切分为三段, 分别用 $t, h, w$ 的索引旋转：

$$\text{angle}_t = t \cdot \theta_i, \quad \text{angle}_h = h \cdot \theta_i, \quad \text{angle}_w = w \cdot \theta_i \tag{3.1}$$

**配置走查**(Qwen2-VL, head_dim = 64, mrope_section = [16, 24, 24])：

| 通道区间 | 长度 | 旋转轴 | 说明 |
|---------|------|-------|------|
| [0, 16) | 16 | 时间 $t$ | 编码帧间关系 |
| [16, 40) | 24 | 高度 $h$ | 编码垂直空间关系 |
| [40, 64) | 24 | 宽度 $w$ | 编码水平空间关系 |

注意：16+24+24=64(= head_dim / 2), 实际嵌入维度需翻倍以匹配rotate_half. 

### 3.2 Interleaved-MRoPE：频率分布均衡化

M-RoPE的**频谱不均问题**：时间维度分配前16个通道(全为高频), 宽度维度分配后24个通道(全为低频). 导致时间维度长程分辨力不足、宽度维度短程分辨力过剩. 

**Interleaved-MRoPE的解决方案**：轮询分配每个旋转对到三个轴：

$$\text{axis}(j) = j \bmod 3, \quad j = 0, 1, \ldots, \frac{d}{2} - 1 \tag{3.2}$$

映射规则：$0 \to t$, $1 \to h$, $2 \to w$. 

**优势**：每个维度的高低频率均匀交错分布, 避免频谱偏斜. 时间维度既有高频对(短程敏感)也有低频对(长程区分). 

```python
def apply_interleaved_mrope(freqs: torch.Tensor, mrope_section: list[int]) -> torch.Tensor:
    """将分段M-RoPE频率重排为交错格式
    
    输入 freqs: (3, batch, seq_len, head_dim//2)
        freqs[0]: 时间维度的频率
        freqs[1]: 高度维度的频率
        freqs[2]: 宽度维度的频率
    
    输出: (batch, seq_len, head_dim//2), 按 j%3 的轮询规则交错组合. 
    """
    freqs_t = freqs[0].clone()
    
    for dim, offset in enumerate((1, 2), start=1):
        length = mrope_section[dim] * 3
        idx = slice(offset, length, 3)
        freqs_t[..., idx] = freqs[dim, ..., idx]
    
    return freqs_t
```

**性能影响**：交错重排的额外内存拷贝约为 $O(2 \times B \times L \times d)$. 在batch=1、L=32768、d=64的场景下, 额外拷贝约4MB, 耗时<0.1ms. 

### 3.3 纯文本退化

当输入为纯文本(无图像/视频)时, 将 $(t, h, w)$ 都设为文本位置 $m$, 三个维度的旋转角度均为 $m \cdot \theta_i$. 所有通道的旋转角度相同, 等价于标准1D-RoPE. 

这是M-RoPE设计的关键兼容性保证：同一套代码同时支持纯文本和多模态输入, 无需条件分支.
