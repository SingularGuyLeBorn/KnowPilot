# MoE 系统优化综述

> 本文从系统层面梳理 MoE 模型的部署与优化策略, 涵盖显存充足与受限两大场景下的并行策略、卸载策略、通信优化和计算优化. 

---

## 1. 系统部署的两大场景

### 1.1 显存充足场景

**专家并行(Expert Parallelism, EP)** ：
- 将专家分配到多个 GPU 上
- 每个 GPU 只存放部分专家
- 通过 All-to-All 通信将 token 发送到目标专家

**优势**：
- 单卡显存需求降低
- 专家计算并行化

**挑战**：
- All-to-All 通信开销大
- 负载不均衡导致部分 GPU 空闲

### 1.2 显存受限场景

**卸载策略(Offloading)** ：
- 将不活跃专家卸载到 CPU 内存或磁盘
- 需要时动态加载到 GPU
- 适用于单卡或少量 GPU 部署

**优势**：
- 支持超大模型在有限显存上运行
- 灵活性高

**挑战**：
- 加载延迟高
- 需要精细的预取策略

---

## 2. 并行策略详解

### 2.1 数据并行 + 专家并行

**ZeRO-EP 混合**：
- 数据并行维度：复制专家参数
- 专家并行维度：分散专家到不同节点
- 梯度聚合：先在 EP 组内聚合, 再在 DP 组间聚合

### 2.2 张量并行 + 专家并行

- Attention 层：张量并行(TP)
- MoE 层：专家并行(EP)
- 组合：TP 处理共享参数, EP 处理专家参数

---

## 3. 通信优化

### 3.1 All-to-All 优化

**问题**：MoE 的 All-to-All 通信量与 batch size × 专家数量成正比. 

**优化方向**：
- **细粒度调度**：按 expert 分组发送, 减少单次通信量
- **拓扑感知**：优先将 token 发送到同一节点的专家
- **双缓冲**：通信与计算重叠

### 3.2 通信-计算重叠

**核心思想**：在通信的同时进行其他计算. 

实现方式：
1. 预取下一 batch 需要的专家权重
2. 在 All-to-All 通信时计算非 MoE 层
3. 流水线并行：前向传播与反向传播重叠

---

## 4. 计算优化

### 4.1 专家计算优化

**Group GEMM**：
- 不同专家的输入拼接成一个大矩阵
- 一次性执行矩阵乘法
- 减少 kernel 启动开销

**动态批处理**：
- 根据实际激活的专家动态调整 batch size
- 避免为未激活专家分配计算资源

### 4.2 负载均衡

**问题**：某些专家被频繁激活, 成为瓶颈. 

**解决方案**：
- **辅助损失**：在训练时加入负载均衡损失
- **容量因子**：限制每个专家处理的 token 数量
- **随机路由**：添加随机噪声打破平局

---

## 5. 访存优化

### 5.1 专家权重管理

**按需加载**：
- 只加载当前 step 需要的专家
- 使用 LRU 缓存管理专家权重

**权重共享**：
- 多个专家共享部分参数
- 减少总参数量和访存量

### 5.2 激活值优化

**选择性重计算**：
- 重计算注意力输出, 释放 KV Cache
- 以计算换显存

**激活值压缩**：
- 使用 FP8/BF16 存储激活值
- 降低显存占用

---

## 6. 总结

MoE 系统优化的核心矛盾：**稀疏激活的灵活性 vs 硬件的并行性要求**. 

| 优化维度 | 关键技术 | 目标 |
|:--------|:--------|:-----|
| 并行 | EP + TP + PP | 扩展性 |
| 通信 | All-to-All 优化 + 重叠 | 降低延迟 |
| 计算 | Group GEMM + 动态批处理 | 提升吞吐 |
| 访存 | 按需加载 + 选择性重计算 | 节省显存 |
| 负载 | 辅助损失 + 容量因子 | 均衡负载 |


## 7. SonicMoE：IO 与 Tile 感知的极致优化

### 7.1 核心观察

SonicMoE 提出三个关键观察：
1. **细粒度 MoE 需要更大的激活内存**：激活大小与激活的专家数量呈线性关系
2. **计算强度降低, IO 成本增加**：细粒度和高稀疏性使 MoE 更趋向内存密集型
3. **Grouped GEMM 分块量化效应**：高度稀疏时, 固定 Tile 大小导致计算资源浪费

### 7.2 激活内存优化

**目标**：在不增加 FLOPs 的情况下, 以尽可能小的激活内存占用实现 MoE 训练. 

MoE 总 FLOPs 为 $(6+12) \cdot T \cdot n \cdot K \cdot d$. 为了保持总计算量不变, $n \cdot K$ 必须是常数——增加专家数量就必须减少每个专家的容量. 

标准实现中, 反向传播需要缓存中间激活(尺寸 $T \cdot K \cdot d$), 当 $K$ 增加时内存线性增长. 

**SonicMoE 的三项技术**：
1. **识别并避免缓存巨型激活**：反向传播不依赖存储在 HBM 的 $X_e$ 和 $Y$
2. **Operator Fusion 消除中间产物**：将 HBM load 和 Gather 融合成一个 Kernel, 不显式构建 $X_e$
3. **数学等价替换绕开依赖**：通过 $dS = \langle dA, A' \rangle$ 路径计算 $dS$, 无需存储 $Y$

### 7.3 算子融合

**Gather 与 HBM Load 融合**：
- 通过 `cp.async` 指令异步加载 token index 对应的内容到 SMEM
- 支持 contiguously-packed 或非连续输入
- 通过 prefetch 和 producer 协同获取优化同步索引

**Epilogue 融合**：
- 前向传播：在 GEMM Epilogue 后直接在寄存器中做 SwiGLU 计算
- 反向传播：将 $dH$、$dS$、$d\text{SwiGLU}$ 三个关键梯度计算融合到一个 Kernel

### 7.4 计算与 IO 重叠优化

**Hopper GPUs 上的 PingPong Warpgroup 调度**：
- 计算 $dH$ Epilogue 时的 TMA load 与其他操作并行
- 前向 $Y$ 计算与反向 $\tilde{X}$ 的 TMA store：先用 TMA 将结果连续存储到 HBM 临时区域, 再启动独立 Kernel 完成 Scatter
- 避免融合 Scatter-Store 对主 GEMM 的阻塞

### 7.5 Top-K 排序 Kernel 设计

**问题**：PyTorch 的 `torch.topk` 占据路由计算时间的 40%. 

**解决方案**：
- 设计适用于 $E \leq 4096$、$K \leq 16$ 的专用内核
- 使用**双调排序(Bitonic Sort)** 作为核心算法
- 打包索引保证排序稳定性
- 利用寄存器和 Warp-Shuffle 提升内存效率

### 7.6 Token 舍入算法

**问题**：训练非常稀疏的 MoE 时, 每个专家分配的 token 数量变少, 无法被 Tile 大小整除, 造成填充浪费. 

**Token Rounding Routing**：
1. 计算初始分配和候选池：执行标准 Top-K 路由, 对每个专家的所有 token 路由得分排序
2. 舍入决策与执行：就近舍入到最接近 Tile 大小的倍数, 向上或向下调整分配

### 7.7 实验结果

| 指标 | SonicMoE vs ScatterMoE | SonicMoE vs MoMoE |
|:-----|:----------------------|:------------------|
| 峰值内存(7B, n=256)| -45% | 更显著 |
| TFLOPs(1.4B/7B)| +40% | +40% |
| 正向传播吞吐量 | >500 TFLOPs | 远超 baseline |
