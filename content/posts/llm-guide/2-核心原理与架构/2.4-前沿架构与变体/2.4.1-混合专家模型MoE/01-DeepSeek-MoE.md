本篇就从 v1 都 v3 梳理他们从最早的尝试到大规模训练都有哪些改变

### DeepSeek MoE V1 (16B 总参数 - 2.8B 激活参数)

DeepSeek MoE V1 的核心思想是在标准的 Transformer 架构中, 将前馈网络 (Feed-Forward Network, FFN) 部分替换为一个专家混合 (Mixture of Experts, MoE)层. 这个 MoE 层包含两种类型的专家: **共享专家 (Shared Experts)** 和 **路由专家 (Routed Experts)** .

![](./01-DeepSeek-MoE-images/image_0.png)

图 1: DeepSeek MoE V1 架构示意图.

如上图所示, 对于每个输入的 token, 它会同时被所有共享专家处理, 并由一个**路由器 (Router)** 选择一部分路由专家进行处理. 这种设计的动机是让共享专家学习所有 token 都需要的通用知识, 而让路由专家学习更细粒度的, 特定领域的知识. V1 版本包含 **2 个共享专家**和 **64 个路由专家**, 路由器会从中为每个 token 选择**得分最高的 4 个**进行激活.

#### 标准 Top-k 路由机制 (Standard, top-k routing)

路由器的决策过程通过以下公式实现:

$h'_t = u_t + \sum_{i=1}^{N_s} \text{FFN}_i^{(s)}(u_t) + \sum_{i=1}^{N_r} g_{i,t} \text{FFN}_i^{(r)}(u_t)$

$g_{i,t} =
\begin{cases}\ns_{i,t}, & \text{if } s_{i,t} \in \text{Topk}(\{s_{j,t} | 1 \le j \le N_r\}, K_r) \\
0, & \text{otherwise}
\end{cases}$

$s_{i,t} = \text{Softmax}_i(u_t^T e_i)$

- **变量解释**:

- $h'_t$: MoE 层的最终输出.- $u_t$: MoE 层的输入 token 表示. 最终的输出 $h'_t$ 通过残差连接加入了原始输入 $u_t$.- $\text{FFN}_i^{(s)}$: 第 $i$ 个**共享专家**. 总共有 $N_s$ 个 (V1中 $N_s=2$). 所有输入 $u_t$ 都会经过所有共享专家的计算.- $\text{FFN}_i^{(r)}$: 第 $i$ 个**路由专家**. 总共有 $N_r$ 个 (V1中 $N_r=64$).- $s_{i,t}$: 路由器为输入 $u_t$ 对第 $i$ 个路由专家计算出的**路由分数 (或权重)** . 这是通过计算 $u_t$ 与专家 $i$ 的可训练嵌入向量 $e_i$ 的点积, 再经过 Softmax 函数得到的. Softmax 确保所有专家的分数和为 1.- $g_{i,t}$: 最终用于加权求和的**门控值 (gating value)** . Topk 函数会从所有 $N_r$ 个专家的分数 $\{s_{j,t}\}$ 中选出最高的 $K_r$ 个 (V1中 $K_r=4$). 只有被选中的专家的 $g_{i,t}$ 等于其原始分数 $s_{i,t}$, 其他专家的 $g_{i,t}$ 都为 0.- 这个过程确保了对于每个 token, 只有 $K_r$ 个路由专家被激活, 实现了模型的稀疏性.

#### 标准辅助损失函数 (Standard Aux-loss balancing)

为了防止路由器总是将 token 发送给少数几个受欢迎的专家, 导致负载不均, 训练时引入了一个辅助损失函数来鼓励负载均衡.

$L_{\text{ExpBal}} = \alpha_1 \sum_{i=1}^{N_r} f_i P_i$

$f_i = \frac{1}{K_r T} \sum_{t=1}^{T} \mathbb{1}(\text{Token } t \text{ selects Expert } i)$

$P_i = \frac{1}{T} \sum_{t=1}^{T} s_{i,t}$

- **变量解释**:

- $L_{\text{ExpBal}}$: 专家平衡辅助损失.- $\alpha_1$: 一个超参数, 用于控制该损失项的权重.- $T$: 一个 batch 中的 token 总数.- $f_i$: 第 $i$ 个专家在 batch 中被**选择的频率**. 它计算了路由到专家 $i$ 的 token 数量占总路由数量的比例.- $P_i$: 第 $i$ 个专家在 batch 中获得的**平均路由分数**.- 通过最小化 $L_{\text{ExpBal}}$ (即 $f_i$ 和 $P_i$ 的乘积), 模型被激励去让更多的专家被利用起来, 并且路由器的打分也趋向于更均匀.

### DeepSeek MoE V2 (236B 总参数 - 21B 激活参数)

V2 版本将模型规模大幅提升, 拥有 **2 个共享专家**和 **160 个路由专家**, 每次激活 **10 个**. 随着模型和专家数量的急剧增加, 训练的系统级挑战, 特别是设备间的通信开销, 变得至关重要. V2 的主要创新点在于优化路由策略以降低通信成本.

![](./01-DeepSeek-MoE-images/image_1.png)

图 2: DeepSeek MoE V2 架构示意图.

相比 V1,V2 在架构上没有任何区别,只是专家数量变化了 . 损失函数也是一样的,但他们增加了一个非常巧妙的方法用于解决细粒度专家的缺点

当专家粒度过小,数量很多,那么就会有很多活跃专家,每次输入都需要将 token 通过路由发送给不同设备上的大量专家,这会导致巨大的通信开销

DeepSeek 团队采用了Top-M 设备路由 (Top-M device routing)去解决这个事情(果然是 Infra 的神！但我其实更期待 Google 团队的 Infra 经验分享)

#### Top-M 设备路由 (Top-M device routing)

当专家被分散到大量不同的计算设备 (如 GPU) 上时, 如果一个 token 需要的专家分布在很多设备上, 就会产生巨大的通信开销. Top-M 路由正是为了解决这个问题.

- **核心思想**: 在为 token 选择 Top-K 专家之前, **首先选择 Top-M 个设备**. 具体来说, 路由器先计算出所有专家的分数, 然后将分数按设备进行汇总, 选出总分最高的 M 个设备. 之后, 路由选择被限制在这 M 个设备内部进行 Top-K 专家选择. 这样, 每个 token 最多只需要与 M 个设备通信, 极大地控制了通信带宽.

#### 通信平衡损失 (Communication balancing loss)

除了专家负载, 设备间的通信负载也需要平衡. V2 引入了一个新的损失项来平衡设备间 "输入" (token 发往设备) 和 "输出" (计算结果返回) 的通信量.

$L_{\text{CommBal}} = \alpha_2 \sum_{j=1}^{D} l_j \cdot r_j$

$l_j = \frac{1}{MT} \sum_{t=1}^{T} \mathbb{1}(\text{Token } t \text{ is sent to Device } j)$

$r_j = \frac{1}{T} \sum_{t=1}^{T} \sum_{i \in \text{Experts on Device } j} s_{i,t}$

- **变量解释**:

- $L_{\text{CommBal}}$: 通信平衡损失.- $D$: 设备总数.- $l_j$: 发送到设备 $j$ 的**token 比例**, 代表了输入通信负载.- $r_j$: 路由到设备 $j$ 上所有专家的**总分数占比**, 代表了输出通信负载.- 这个损失函数鼓励模型将 token 和路由权重均匀地分布到不同的设备上.

### DeepSeek MoE V3 (671B 总参数 - 37B 激活参数)

V3 是目前规模最大的版本, 拥有 **1 个共享专家**和 **258 个路由专家**, 每次激活 **8 个**. V3 在路由算法和平衡策略上做了进一步的优化.

![](./01-DeepSeek-MoE-images/image_2.png)

图 3: DeepSeek MoE V3 架构示意图.

#### Sigmoid+Softmax TopK 路由

V3 调整了门控值的计算方式, 引入了 Sigmoid 函数, 并且只对被选中的专家进行归一化.

$h'_t = u_t + \sum_{i=1}^{N_s} \text{FFN}_i^{(s)}(u_t) + \sum_{i=1}^{N_r} g_{i,t} \text{FFN}_i^{(r)}(u_t)$

$g_{i,t} = \frac{g'_{i,t}}{\sum_{j=1}^{N_r} g'_{j,t}}$

$g'_{i,t} =
\begin{cases}\ns_{i,t}, & \text{if } s_{i,t} \in \text{Topk}(\{s_{j,t} | 1 \le j \le N_r\}, K_r) \\
0, & \text{otherwise}
\end{cases}$

$s_{i,t} = \text{Sigmoid}(u_t^T e_i)$

- **主要变化**:

- 专家分数 $s_{i,t}$ 由原始的点积得分经过 **Sigmoid** 函数得到, 而不是直接送入 Softmax. Sigmoid 函数将得分映射到 (0, 1) 区间, 相比 Softmax 行为更平滑, 数值更稳定.- 最终的门控值 $g_{i,t}$ 通过对被选中的 Top-K 个专家的分数 $g'_{i,t}$ 进行归一化 (相当于一个 Softmax) 得到. 这样保证了最终用于加权的系数之和为 1.

#### 无辅助损失的序列级均衡 (Aux-loss-free + seq-wise aux)

V3 抛弃了 V1/V2 中的显式辅助损失函数, 转而采用一种无损失的均衡策略. 此外, 它还考虑了在单个序列级别上进行均衡.

- **无辅助损失**: 虽然图中公式未详细展开, 但其思想是通过为每个专家引入一个可动态调整的偏置项 (bias) $b_i$ 来实现均衡. 当一个专家被过度使用时, 其偏置 $b_i$ 会被调低, 从而降低其后续被选中的概率, 反之亦然. 这样可以在没有额外损失项的情况下实现负载均衡.

- **序列级均衡**: V1/V2 的均衡策略是基于整个 batch 的. 但在推理时, 模型可能接收到一些分布外的序列, 导致某些专家在处理该序列时被集中调用. 序列级均衡旨在确保在单个序列的处理过程中, 专家的使用也是均衡的, 从而增强模型在推理时的鲁棒性.

### 深度解析 DeepSeek V3 的注意力机制变体: MLA (Multi-Head Latent Attention)

在追求更大模型和更长上下文处理能力的道路上, 一个核心的系统瓶颈是**KV 缓存 (KV Cache)** . 在标准 Transformer 的自回归解码 (即逐字生成) 过程中, 为了避免重复计算, 模型需要存储过去所有 token 的键 (Key, K) 和值 (Value, V) 向量. 当上下文窗口变长时, 这个缓存会消耗巨量的显存 (VRAM), 成为限制模型推理能力的主要障碍.

DeepSeek V3 引入的 MLA (Multi-Head Latent Attention) 是一种创新的注意力机制, 其根本目标就是**压缩 KV 缓存**, 从而在不显著影响模型性能的前提下, 大幅减少推理时的内存占用.

![](https://i.imgur.com/your_image_mla.png)![](./01-DeepSeek-MoE-images/image_4.png)

图 4: Multi-Head Latent Attention (MLA) 架构示意图.

MLA 的核心思想是**不直接缓存高维的 K 和 V 向量, 而是先将它们投影到一个共享的, 维度更低的 "潜空间" (latent space), 然后只缓存这个低维的 "潜激活向量" (latent activation)** . 在实际需要进行注意力计算时, 再从这个压缩后的潜向量中将 K 和 V "重建" 出来.

这个过程可以分解为以下几个关键步骤:

#### 1. 降维: 从输入生成潜激活向量

对于注意力层的每一个输入 token 的隐藏状态 $h_t$, MLA 首先通过一个线性投影将其压缩成一个低维的**潜向量** $c_t^{KV}$.

$c_t^{KV} = W^{DKV} h_t$

- **变量解释**:

- $h_t$: 注意力层的输入隐藏状态, 代表了模型在第 $t$ 个位置对 token 的完整语义理解, 维度较高.- $W^{DKV}$: 一个可训练的**降维投影矩阵 (Down-projection Key-Value matrix)** . 它的作用就是将高维的 $h_t$ "压缩" 成低维的 $c_t^{KV}$.- $c_t^{KV}$: 生成的**潜激活向量**. 这个向量的维度远小于原始的 K 和 V 向量. **这正是 MLA 实际存入 KV 缓存中的内容**, 也是实现内存优化的关键.

#### 2. 升维: 从潜向量重建 K 和 V

当需要计算注意力分数时, 模型需要完整的 K 和 V 向量. MLA 通过另外两个投影矩阵, 从缓存的潜向量 $c_t^{KV}$ 中 "恢复" 或 "上采样" 出 K 和 V.

$k_t^c = W^{UK} c_t^{KV}$

$v_t^c = W^{UV} c_t^{KV}$

![](./01-DeepSeek-MoE-images/image_5.png)

- **变量解释**:

- $W^{UK}, W^{UV}$: 两个可训练的**升维投影矩阵 (Up-projection Key/Value matrices)** . 它们分别负责从共享的潜向量 $c_t^{KV}$ 中重建出键向量 $k_t^c$ 和值向量 $v_t^c$.- $k_t^c, v_t^c$: 重建后的键向量和值向量. 它们将用于后续与查询向量 (Query, Q) 的注意力计算.

#### 3. 计算效率的巧思: 矩阵合并

一个非常明显的问题是: MLA 引入了额外的升维矩阵乘法 ($W^{UK}$ 和 $W^{UV}$), 这是否会增加计算开销? DeepSeek 的设计非常巧妙地规避了这个问题.

在标准的注意力计算中, 查询向量 Q 也是由输入 $h_t$ 通过一个投影矩阵 $W^Q$ 得到的 ($q_t = W^Q h_t$). 注意力分数的核心是计算 Q 和 K 的点积. 在 MLA 中, 这个操作变为计算 $q_t$ 和 $k_t^c$ 的点积.

这个技巧的关键在于, **升维矩阵 **$W^{UK}$** 可以在计算注意力分数时, 与查询 (Query) 的投影矩阵 **$W^Q$** 在数学上预先合并成一个新矩阵**. 因此, 实际计算中并没有增加矩阵乘法的步骤, 只是用一个合并后的新矩阵替代了原来的 $W^Q$ 矩阵. 这样一来, MLA 就在几乎不增加计算复杂度的前提下, 实现了 KV 缓存的压缩.

#### 4. 解决与 RoPE 的兼容性问题

旋转位置编码 (RoPE) 是现代大模型中广泛使用的技术, 它通过 "旋转" Q 和 K 向量来注入位置信息. 这个旋转操作发生在 Q 和 K 经过各自的投影矩阵之后, 点积计算之前.

这就带来一个问题: RoPE 的旋转操作正好插入到了 $W^Q$ 和 $W^{UK}$ 之间, 打破了它们能够直接合并的代数结构.

- **不使用 RoPE**: 两个矩阵可以合并, $⟨h W^Q, W^{UK} c_t^{KV}⟩ = ⟨h W^Q W^{UK}, c_t^{KV}⟩$- **使用 RoPE**: $⟨h W^Q R_q, W^{UK} c_t^{KV} R_k⟩$, 旋转矩阵 $R_q$ 和 $R_k$ 阻碍了合并.

DeepSeek 的解决方案是: **保留 K 向量中的少数几个维度不参与潜空间压缩, 专门用于执行 RoPE 的旋转操作**, 而其余大部分维度则正常通过 MLA 进行压缩. 这是一种务实的折衷方案, 既享受了 MLA 带来的绝大部分内存收益, 又保留了 RoPE 的关键能力.

#### 附: 一个具体的计算示例

为了让这个过程更具体, 让我们假设一些实际的维度参数.

- **模型隐藏层维度** ($d_{model}$): 512- **注意力头数** ($N_{heads}$): 8- **每个头的维度** ($d_{head}$): $d_{model} / N_{heads} = 512 / 8 = 64$- **潜空间维度** ($d_{latent}$): 16 (这是一个关键参数, 必须远小于 $d_{head}$)

现在, 让我们追踪一个 token (输入为 $h_t$) 在一个注意力头中的计算流程.

**传统注意力机制**:

1. 为该注意力头生成 K, V 向量:

- $k_t = W^K h_t$, 这里的 $W^K$ 是一个 [64, 512] 的矩阵, 生成的 $k_t$ 是一个 [64, 1] 的向量.- $v_t = W^V h_t$, 这里的 $W^V$ 是一个 [64, 512] 的矩阵, 生成的 $v_t$ 是一个 [64, 1] 的向量.

1. **需要缓存的内容**: 向量 $k_t$ 和 $v_t$.

2. **缓存大小 (单个头, 单个token)** : $64 (\text{for } k_t) + 64 (\text{for } v_t) = 128$ 个浮点数.

**使用 MLA 的注意力机制**:

1. **步骤 1: 降维 (压缩)** 

- 计算潜向量 $c_t^{KV} = W^{DKV} h_t$.- 这里的 $W^{DKV}$ 是一个 [16, 512] 的矩阵.- 输入 $h_t$ 是 [512, 1] 的向量.- 生成的 $c_t^{KV}$ 是一个 [16, 1] 的向量.

1. **需要缓存的内容**: 只有潜向量 $c_t^{KV}$.

2. **缓存大小 (单个头, 单个token)** : **16 个浮点数**.

3. **步骤 2: 升维 (重建, 仅在需要计算时)** 

- 当需要计算注意力时, 从缓存中取出 $c_t^{KV}$.- $k_t^c = W^{UK} c_t^{KV}$. 这里的 $W^{UK}$ 是 [64, 16] 的矩阵, 重建出的 $k_t^c$ 是 [64, 1] 的向量.- $v_t^c = W^{UV} c_t^{KV}$. 这里的 $W^{UV}$ 是 [64, 16] 的矩阵, 重建出的 $v_t^c$ 是 [64, 1] 的向量.- 重建出的 $k_t^c$ 和 $v_t^c$ 的维度与传统注意力机制完全相同, 可以无缝参与后续计算.

#### **对比与总结**

在这个例子中:

- 传统方法需要为每个 token 的每个头缓存 **128** 个浮点数.- MLA 只需要缓存 **16** 个浮点数.

**缓存大小被压缩到了原来的 **$16 / 128 = 1/8$. 这是一个巨大的优化. 对于一个拥有 8 个头的模型, 每个 token 的总 KV 缓存大小从 8 * 128 = 1024 降低到了 8 * 16 = 128. 当处理数千个 token 的长序列时, 节省的显存量是非常可观的, 这也正是 MLA 技术的核心价值所在.

### DeepSeek V3 的推理加速技术: MTP (Multi-token Prediction)

标准的自回归语言模型 (Autoregressive LLM) 在生成文本时效率较低, 因为它们必须逐个 token 生成: 计算出第 N 个 token 后, 才能将其作为输入来计算第 N+1 个 token. 这个过程是严格串行的, 限制了生成速度. MTP 的核心目标就是打破这一瓶颈, **通过训练一些小型的、轻量级的 "预测模块", 让模型具备一次性预测未来多个 token 的能力**, 从而在推理时实现加速.

![](./01-DeepSeek-MoE-images/image_6.png)

图 5: 左侧为 DeepSeek v3 的 MTP 训练框架, 右侧为 EAGLE 论文中类似 MTP 的技术在推理时的应用 (speculative decoding).

#### MTP 的训练机制 (图左侧)

MTP 的思想是在训练主模型的同时, "附带" 训练一个或多个小模型.

1. **主模型 (Main Model)** :

- 这是一个标准的、深层的大语言模型 (由 L 个 Transformer Block 组成).- 它的任务是进行常规的**下一词元预测 (Next Token Prediction)** . 即给定输入序列 t₁, t₂, t₃, t₄, 它的目标是预测出 t₂, t₃, t₄, t₅.- 其损失函数为 $L_{Main}$.

1. **MTP 模块 (MTP Module)** :

- 这些是**非常轻量级**的模型, 例如, 可能只包含一个 Transformer Block.- 它们的目标是**预测更远的未来**. MTP Module 1 预测 "下下个" token (t_{i+2}), MTP Module 2 预测 "下下下个" token (t_{i+3}), 以此类推.- 为了效率, 这些 MTP 模块与主模型**共享**参数最重的部分: **词嵌入层 (Embedding Layer)** 和 **输出头 (Output Head)** .

#### MTP 模块如何工作

MTP 模块的输入设计非常巧妙, 它利用了主模型的信息和 "teacher forcing" (即使用真实标签作为下一时间步的输入) 的思想.

$h_i^k = M_k[\text{RMSNorm}(h_i^{k-1}); \text{RMSNorm}(\text{Emb}(t_{i+k}))]$

$p_{i+k+1}^k = \text{OutHead}(h_i^k)$

- **变量解释**:

- $k$: 代表这是第 $k$ 个 MTP 模块 (例如, $k=1$ 对应 MTP Module 1).- $h_i^{k-1}$: 表示在位置 $i$ 时, 上一个阶段 (第 $k-1$ 个模块) 输出的隐藏状态. 对于第一个 MTP 模块 ($k=1$), $h_i^0$ 就是**主模型 (Main Model)** 在位置 $i$ 的最终输出隐藏状态.- $\text{Emb}(t_{i+k})$: 这是**未来第 k 个真实 token** ($t_{i+k}$) 的词嵌入. 这是一个关键的 "提示" 信息.- $[ ; ]$: 表示**拼接 (concatenation)** 操作.- $M_k$: 第 $k$ 个 MTP 模块 (即那个轻量级的 Transformer Block).- $h_i^k$: 第 $k$ 个 MTP 模块在处理完拼接后的输入后, 得到的输出隐藏状态.- $p_{i+k+1}^k$: 最终的预测概率分布. 它是通过将 $h_i^k$ 送入共享的 OutHead 得到的, 目标是预测真实 token ($t_{i+k+1}$)

**举例说明 MTP Module 1 (**$k=1$**) 的工作流程**:
为了预测$t_{i+2}$ , MTP Module 1 会接收两个输入:

1. 主模型在位置 $i$ 的输出 $h_i^0$.2. 真实 token $t_{i+1}$的词嵌入.
它将这两者拼接起来, 通过自己的单层 Transformer 进行处理, 最终预测 $t_{i+2}$ . 这种方式让 MTP 模块学会了 "在已知主模型意图和下一个真实词的情况下, 推断再下一个词" 的能力.

#### MTP 的推理应用: 推测解码 (Speculative Decoding)

虽然 MTP 在训练时看起来很复杂, 但其真正的威力体现在推理阶段, 通常与一种叫做 "推测解码" 的技术相结合 (如图右侧 EAGLE 论文的示意图).

1. **起草 (Draft)** : 使用一个或多个快速的 MTP 模块 (现在作为 "草稿模型") 并行地、推测性地生成未来几个 token 的一个序列草稿. 例如, 生成 "make", "help", "with", "you".

2. **验证 (Verify)** : 将原始输入和整个草稿序列 ("How can I" + "make help with you") 一次性送入强大的**主模型 (Main Model)** . 主模型只需一次前向传播, 就可以并行地计算出在每个位置上它自己会生成的 token.

3. **接受/拒绝**: 比较主模型的预测和草稿模型的草稿.

- 如果主模型的预测与草稿一致 (例如, 主模型在 "I" 后面也想生成 "make"), 则接受草稿 token.- 这个过程一直持续, 直到发现一个不匹配的 token.- 接受所有匹配的 token, 用主模型的正确预测替换第一个不匹配的 token, 然后丢弃草稿中剩余的部分.

这个过程之所以能加速, 是因为主模型的一次并行前向传播可以一次性验证多个 token, 这远比主模型自己逐个生成这些 token 要快得多.

#### DeepSeek V3 的实际做法

图中有一个非常关键的注释: **"(But they only do MTP with one token ahead)"**.

这意味着, 尽管 MTP 的理论框架可以支持预测未来很多步, 但在 DeepSeek V3 的最终实现中, 他们**只训练并使用了一个 MTP 模块 (MTP Module 1)** . 这个模块的任务是预测 "下下个" token (t_{i+2}). 在推理时, 这允许他们进行每次能 "推测" 一个额外 token 的简化版推测解码, 从而在实践中获得稳定有效的加速效果.

### 从 V1 到 V3 的演进总结

DeepSeek MoE 架构的演进清晰地展示了从一个基础可行的 MoE 模型到一个高度优化的, 能够支撑超大规模训练和推理的系统性解决方案的路径.

- **Motivation 与改变**:

- **从 V1 到 V2**: 核心动机是**解决超大规模训练带来的系统瓶颈**. 当专家数量从 64 增加到 160 时, 跨设备通信成为主要矛盾. 为此, V2 引入了 **Top-M 设备路由**来限制通信范围, 并增加了**通信平衡损失**来均衡网络负载. 这标志着模型设计从纯粹的算法层面深入到了软硬件协同的系统工程层面.

- **从 V2 到 V3**: 核心动机是**进一步提升算法的稳定性和推理的鲁棒性**, 并继续优化关键资源(如内存)的利用率. V3 将路由算法优化为更稳定的 **Sigmoid + Softmax TopK** 形式, 并采用更高效的**无辅助损失均衡策略**. 引入**序列级均衡**是为了应对推理时可能出现的极端情况. 同时, V3 引入了 **MLA 注意力机制**, 旨在解决大模型在长序列推理中面临的 KV 缓存瓶颈, 显示了对模型全方位性能的极致追求.

- **效果**: 这一系列演进使得 DeepSeek 团队能够成功地训练和部署从百亿到近七千亿参数的 MoE 模型, 并在性能上取得了领先. 每一步迭代都针对前一版本在更大规模下暴露出的问题, 提出了精准且有效的解决方案, 体现了在构建前沿大模型过程中的深刻洞察和卓越工程能力.
