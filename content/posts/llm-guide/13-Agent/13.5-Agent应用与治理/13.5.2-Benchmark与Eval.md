# 13.1.5 Benchmark 与 Eval: Agent 能力的度量衡与诊断学

Agent 系统的评估远比单一文本生成任务的 perplexity 或 BLEU 复杂. 评估对象从"生成质量"跃迁为"任务完成度",评估维度从"准确性"扩展为"成功率、效率、成本、鲁棒性"的四维空间. Benchmark 的设计者需在"任务真实性"与"评估可重复性"之间权衡: 过于简化的环境(如单步问答)无法反映 Agent 的序列决策能力; 过于复杂的环境(如真实企业代码库)导致评估成本高昂且难以复现. 


## 1. 演进脉络: 从静态数据集到动态环境评估

| 阶段 | 时间 | 代表工作 | 评估对象 | 核心局限 |
|------|------|----------|----------|----------|
| 静态文本评估 | 2022 | HumanEval / MBPP | 代码片段生成 | 无环境执行,无法检验多文件修改 |
| 单步工具评估 | 2023 | APIBench / ToolBench | 工具调用格式正确性 | 无状态依赖,忽略工具返回结果的再利用 |
| 多步环境评估 | 2024 | WebArena / OSWorld | 跨步骤状态转移 | 环境搭建复杂,评估标准主观 |
| 开放域综合评估 | 2024 | GAIA / MMLU-Pro | 多模态长程推理 | 任务难度跨度大,分数饱和快 |
| 经济成本评估 | 2025 | SWE-Bench + Cost metric | 成功率与 API 费用 trade-off | 成本与能力耦合,难以单独归因 |

物理演进逻辑: 评估范式从"输入-输出对"的静态匹配,演进为"初始状态-动作序列-终局状态"的动态验证. 静态评估的度量是条件概率 $p(y|x)$ 的准确性; 动态评估的度量是策略 $\pi$ 在环境 $\mathcal{E}$ 上的期望累积回报 $\mathbb{E}_{\pi, \mathcal{E}}[R]$. 这一跃迁使评估基础设施从 GPU 集群扩展到包含浏览器引擎、操作系统虚拟机、代码执行沙箱的异构计算环境. 


## 2. Agent Benchmark 分类与工业部署

### 2.1 代码任务: SWE-Bench

SWE-Bench 评估 Agent 在真实 GitHub 代码库中修复 bug 的能力. 每个实例包含: 一个代码库快照、一条 issue 描述、一个测试套件. Agent 需阅读代码、定位 bug、修改文件、提交补丁,最终通过全部测试. 

任务成功率定义为: 

$$ \text{Acc}_{\text{SWE}} = \frac{1}{N} \sum_{i=1}^{N} \mathbb{I}\big( \text{Tests}(\text{Patch}_i) = \text{PASS} \big) \tag{1} $$
式 (1) 中,$N$ 为测试实例总数,SWE-Bench Verified 子集含 500 条; $\text{Patch}_i$ 为 Agent 对第 $i$ 个实例生成的代码补丁; $\text{Tests}(\cdot)$ 为测试执行函数,返回 PASS 或 FAIL; $\mathbb{I}(\cdot)$ 为指示函数. 如式 (1) 所示,SWE-Bench 采用严格的二元通过标准,不接受部分正确. $\text{Tests}$ 包含原仓库的回归测试与新加入的针对性测试,防止 Agent 通过硬编码输出欺骗评测. 该标准的物理意义是模拟真实软件工程场景: 代码修改要么通过 CI/CD 流水线,要么被回滚. 2024 年 top 系统的通过率从 2%(GPT-4 基线)提升至 43%(Claude 3.5 Sonnet + Agent 框架),但距人类工程师的 80%+ 仍有显著差距. 

### 2.2 网页任务: WebArena

WebArena 提供 812 个真实网页交互任务,涵盖电商购物、航班预订、论坛管理、地图导航等场景. Agent 通过操作浏览器(点击、输入、滚动)完成任务目标. 

任务完成度采用多级评估: 

$$ \text{Score}_{\text{Web}} = \omega_1 \cdot \mathbb{I}(\text{goal}) + \omega_2 \cdot \text{rel}(\text{answer}, \text{reference}) + \omega_3 \cdot \text{efficiency} \tag{2} $$
式 (2) 中,$\omega_1, \omega_2, \omega_3$ 为权重系数,WebArena 官方取 $(0.6, 0.2, 0.2)$; $\text{goal}$ 表示最终目标是否达成(如"成功预订航班"); $\text{rel}(\cdot, \cdot)$ 度量答案与参考答案的语义相关度; $\text{efficiency}$ 为步骤效率分数,定义为 $1 - \frac{\text{steps}}{\text{max_steps}}$. 如式 (2) 所示,WebArena 的综合评分同时惩罚功能错误与行为低效. $\text{efficiency}$ 项的物理意义是模拟真实用户耐心: 一个需要 50 步点击才能完成的订票任务,即使最终成功,用户体验也已严重受损. 当前 SOTA 系统(Claude 3.5 + Computer Use)在 WebArena 上的平均完成度约 35%,主要失败模式是网页 DOM 结构复杂导致元素定位错误. 

### 2.3 桌面任务: OSWorld

OSWorld 将 Agent 置于真实 Ubuntu 桌面环境,任务涵盖跨应用文件操作、幻灯片制作、电子表格处理、系统配置等. Agent 接收屏幕截图(或 UI 树)并输出鼠标-键盘动作. 

成功率计算需考虑部分完成: 

$$ \text{Acc}_{\text{OS}} = \frac{1}{N} \sum_{i=1}^{N} \frac{|\text{completed\_subgoals}_i|}{|\text{all\_subgoals}_i|} \tag{3} $$
式 (3) 中,$\text{completed\_subgoals}_i$ 为第 $i$ 个任务中已完成的子目标集合,$\text{all\_subgoals}_i$ 为该任务的全部子目标集合,$|\cdot|$ 表示集合元素个数. 如式 (3) 所示,OSWorld 允许部分得分,更适合细粒度能力分析. 一个"制作 PPT 并导出 PDF"的任务可分解为"创建幻灯片、插入图片、设置动画、导出文件"四个子目标. Agent 可能完成前三项但失败于导出格式设置,此时获得 75% 分数. 该设计的物理意义是避免单一失败点掩盖其他能力——在软件操作中,界面跳转错误不应否定内容创建的成功. 

### 2.4 多模态任务: MMWorld

MMWorld 评估 Agent 在视觉-语言-动作联合空间中的推理能力. 任务包含视频理解、空间导航、物理常识推理,Agent 需整合图像观测与文本指令进行决策. 

多模态评估引入感知准确率: 

$$ \text{Score}_{\text{MM}} = \alpha \cdot \text{task\_success} + (1-\alpha) \cdot \text{perception\_accuracy} \tag{4} $$
式 (4) 中,$\text{perception\_accuracy}$ 为视觉感知模块的物体识别与关系判断准确率,$\alpha$ 为任务成功权重,通常取 $0.7$. 如式 (4) 所示,MMWorld 将评估对象从纯决策策略扩展至感知-决策联合系统. $\text{perception\_accuracy}$ 单独计分的物理意义是诊断失败来源: 若感知准确但任务失败,问题在高层规划; 若感知错误,问题在视觉Encoder . 当前多模态 Agent 的主要瓶颈在于视觉Encoder 对 UI 元素(如小按钮、下拉菜单)的识别精度不足,导致后续决策基于错误观测. 


## 3. GAIA: 通用 AI 助手的开放域基准

GAIA(General AI Assistants benchmark)设计哲学与封闭环境 Benchmark 不同: 它允许 Agent 使用任何工具(搜索引擎、计算器、代码解释器、百科 API)回答开放域问题,不限制动作空间. 

GAIA 难度分级与通过率: 

| 级别 | 特征 | 典型任务 | 人类通过率 | GPT-4 通过率 | 专用 Agent 通过率 |
|------|------|----------|------------|--------------|-------------------|
| Level 1 | 单工具、单步、无推理 | "2024 年诺贝尔物理学奖得主是谁" | 98% | 85% | 92% |
| Level 2 | 多工具、简单推理 | "计算某公司 2023 年营收增长率" | 85% | 45% | 68% |
| Level 3 | 多步推理、信息整合 | "找出同时出现在电影 A 和 B 中的演员" | 70% | 15% | 42% |
| Level 4 | 长程规划、创造性问题解决 | "设计一个满足约束 X 的实验方案" | 55% | 3% | 18% |

GAIA 的评估函数融合自动验证与 LLM-as-Judge: 

$$ \text{Eval}_{\text{GAIA}}(y, y^*) = \max\big( \text{exact\_match}(y, y^*), \text{LLM\_judge}(y, y^*) \big) \tag{5} $$
式 (5) 中,$y$ 为 Agent 提交的答案,$y^*$ 为参考答案,$\text{exact\_match}(\cdot, \cdot)$ 执行字符串或数值的精确匹配,$\text{LLM\_judge}(\cdot, \cdot)$ 为裁判模型对答案等价性的概率估计. 如式 (5) 所示,GAIA 优先采用精确匹配(避免 LLM 评判的主观偏差),仅在精确匹配失败时回退到 LLM-as-Judge. $\text{LLM\_judge}$ 的物理意义是处理答案表述的多样性: "Barack Obama" 与 "奥巴马" 对人类而言语义完全等价,但字符串匹配会因字符集差异而失败. 裁判模型(通常为 GPT-4 或 Claude 3.5)被赋予详细评分标准与 few-shot 示例,以降低位置偏见与长度偏见. 

GAIA 的独特价值在于"不可作弊性": 由于问题基于真实世界动态信息(如最新新闻、实时股价),Agent 无法通过记忆训练数据获得优势,必须真实调用工具获取信息. 该特性使 GAIA 成为衡量 Agent"实时知识获取能力"的首选基准. 


## 4. 评估指标: 从成功率到经济效率

### 4.1 核心指标体系

Agent 评估需构建多维指标向量: 

$$ \mathbf{M} = \big( M_{\text{success}}, M_{\text{steps}}, M_{\text{tool}}, M_{\text{cost}}, M_{\text{time}} \big) \tag{6} $$
式 (6) 定义的多维指标向量中,$M_{\text{success}}$ 为任务成功率,即如式 (1) 所示的二元或分级指标; $M_{\text{steps}}$ 为步骤效率,度量完成任务所需的平均环境交互步数; $M_{\text{tool}}$ 为工具使用正确率,统计调用工具的次数中返回有效结果的比例; $M_{\text{cost}}$ 为经济成本,即单次任务平均消耗的 API 费用(美元); $M_{\text{time}}$ 为 wall-clock 时间,度量从任务下发到结果提交的耗时(秒). 如式 (6) 所示,五个指标构成 Agent 系统的帕累托前沿. $M_{\text{success}}$ 是硬约束,失败则其他指标无意义; $M_{\text{steps}}$ 与 $M_{\text{time}}$ 衡量用户体验; $M_{\text{cost}}$ 决定商业可行性. 工业部署中,一个成功率 95% 但单次成本 50 美元的 Agent 无法规模化,而一个成功率 80% 但成本 0.5 美元的 Agent 更具产品价值. 

### 4.2 步骤效率与工具使用正确率

步骤效率的归一化度量: 

$$ \eta_{\text{step}} = \frac{1}{N} \sum_{i=1}^{N} \frac{\text{optimal\_steps}_i}{\text{actual\_steps}_i} \tag{7} $$
式 (7) 中,$\text{optimal\_steps}_i$ 为人类专家完成第 $i$ 个任务所需的最少步数,$\text{actual\_steps}_i$ 为 Agent 实际消耗的步数. 如式 (7) 所示,$\eta_{\text{step}} \in (0, 1]$,越接近 1 表示效率越高. 该指标惩罚冗余探索与循环行为. 若 Agent 在某任务上陷入"搜索-阅读-发现无关-再搜索"的循环,$\text{actual\_steps}_i$ 将远大于 $\text{optimal\_steps}_i$,$\eta_{\text{step}}$ 显著下降. 

工具使用正确率: 

$$ \rho_{\text{tool}} = \frac{\sum_{i=1}^{N} \text{valid\_calls}_i}{\sum_{i=1}^{N} \text{total\_calls}_i} \tag{8} $$
式 (8) 中,$\text{valid\_calls}_i$ 为第 $i$ 个任务中参数格式正确且返回 HTTP 200 或有效结果的工具调用次数,$\text{total\_calls}_i$ 为该任务中全部工具调用次数. 如式 (8) 所示,$\rho_{\text{tool}}$ 反映 Agent 对工具接口语义的理解深度. 高成功率但低 $\rho_{\text{tool}}$ 表明 Agent 通过"暴力尝试"完成任务: 多次调用同一工具的不同参数组合直至成功. 该模式在 API 按次计费场景下经济不可行. 

### 4.3 成本效率指标

成本效率将成功率与经济开销统一: 

$$ \text{CE} = \frac{M_{\text{success}}}{M_{\text{cost}} + c_0} \tag{9} $$
式 (9) 中,$\text{CE}$ 为成本效率(Cost Efficiency),即单位成功率所需美元成本的倒数; $c_0$ 为正则化常数,防止分母为零,通常取 $0.01$. 如式 (9) 所示,$\text{CE}$ 量化了"每美元能买到多少成功率". 在模型即服务(MaaS)场景下,$M_{\text{cost}}$ 由输入/输出 token 数与模型单价决定. Claude 3.5 Sonnet 的输入价格为 $3/MTok,输出为 $15/MTok. 一个消耗 100K 输入 + 20K 输出的任务成本为 $0.60. 若成功率 80%,则 $\text{CE} = 1.33$; 若 GPT-4o 完成同一任务成本 $1.20 但成功率 85%,则 $\text{CE} = 0.71$. 此时 Claude 方案在成本效率上更优,尽管绝对成功率略低. 


## 5. 评估方法: 自动评估与人工评估

### 5.1 规则匹配评估

规则匹配是最可靠的自动评估方式,适用于输出结构化或可精确验证的场景: 

$$ \text{Match}(y, y^*) = \mathbb{I}\big( y \in \mathcal{F}(y^*) \big) \tag{10} $$
式 (10) 中,$\mathcal{F}(y^*)$ 为由参考答案派生的合法答案集合,可包含数值容差、同义词扩展与格式变体. 如式 (10) 所示,规则匹配的速度极快(毫秒级)且完全客观,但设计 $\mathcal{F}$ 需要领域专家知识. 数学任务的 $\mathcal{F}$ 需处理等价形式: $\frac{1}{2}$、$0.5$、$50\%$ 应被视为等价. 代码任务的 $\mathcal{F}$ 为测试用例的通过集合,最严格但也最可靠. 

### 5.2 LLM-as-Judge

当输出为开放文本且无精确标准时,采用 LLM-as-Judge: 

$$ \text{Score}_{\text{LLM}}(y, y^*) = \sigma\big( f_{\text{judge}}(y, y^*, \text{prompt}_{\text{criteria}}) \big) \tag{11} $$
式 (11) 中,$f_{\text{judge}}$ 为裁判语言模型,通常为 70B+ 参数或前沿闭源模型; $\text{prompt}_{\text{criteria}}$ 为包含评分标准、维度定义与 few-shot 示例的评判提示词; $\sigma(\cdot)$ 为 sigmoid 函数,将 logit 输出映射到 $(0,1)$ 分数区间. 如式 (11) 所示,LLM-as-Judge 将评估问题转化为二分类或回归问题. 其物理意义是利用大模型的语义理解能力模拟人类评判者. 但该方法存在系统性偏见: 位置偏见(偏好先出现的答案)、长度偏见(偏好更长的回答)、自我提升偏见(裁判模型给自己生成的答案更高分). 缓解策略包括: 交换位置多次评判取平均、使用与生成模型不同的裁判模型、引入细粒度评分标准(如"准确性 1-5 分、连贯性 1-5 分")替代整体打分. 

### 5.3 人工评估与评估者一致性

人工评估是黄金标准,尤其针对创造性任务与伦理判断. 评估者一致性采用 Fleiss' Kappa 度量: 

$$ \kappa = \frac{\bar{P} - P_e}{1 - P_e} \tag{12} $$
式 (12) 中,$\bar{P}$ 为评估者之间观察到的一致比例,$P_e$ 为随机假设下的期望一致比例. 如式 (12) 所示,$\kappa \in [-1, 1]$,$\kappa > 0.6$ 被认为一致性良好,$\kappa > 0.8$ 为优秀. Agent 评估中,人工评估成本极高: WebArena 的 812 个任务若由 3 名评估者逐一验证屏幕录像,总工时超过 200 小时. 工业界折中方案是"人机回环": LLM-as-Judge 自动标注 90% 的明确案例,人工仅仲裁边界案例(如 LLM 评判置信度低于 0.7 的实例). 


## 6. 算力成本与数值走查

以评估 100 个 SWE-Bench Lite 实例、使用 Claude 3.5 Sonnet 作为 Agent 策略为例: 

| 成本项 | 数值 | 计算依据 |
|--------|------|----------|
| 平均输入 token/任务 | 45,000 | 代码库上下文 + 错误信息 + 历史修改 |
| 平均输出 token/任务 | 8,000 | 思考链 + 代码补丁 |
| 平均交互轮次 | 12 | Agent 与环境的多轮对话 |
| 单任务 API 费用 | $0.255 | $(45K \times 3 + 8K \times 15) / 1M$ |
| 100 任务总费用 | $25.50 | 不含重试与失败实例 |
| 测试执行时间 | 180 秒/任务 | Docker 构建 + 测试运行 |
| 100 任务总 wall-clock | 5 小时 | 串行执行; 并行可压缩至 30 分钟 |

数值走查(LLM-as-Judge 成本): 

若采用 GPT-4o 作为裁判评估 1000 条开放答案,每条评判需 4000 输入 token(含评分标准与 few-shot)+ 500 输出 token(评分理由). 单条成本 $(4000 \times 5 + 500 \times 15) / 1M = 0.0275$ 美元. 1000 条总成本 27.5 美元. 若引入双位置交换(消去位置偏见),成本翻倍至 55 美元. 对于大规模基准(如 10,000 条),LLM-as-Judge 本身构成显著经济负担. 

评估基础设施成本: OSWorld 需维护 Ubuntu 虚拟机池,单任务平均占用 2 GB 内存与 1 核 CPU,10 并发实例需 20 GB + 10 核. 云服务器成本约 $0.05/小时/实例,1000 任务并行 10 实例运行 10 小时,基础设施成本约 5 美元,与 API 费用相比可忽略. 


## 7. 极简实现: Agent 评估框架

```python
import json
from typing import List, Dict, Callable, Any
from dataclasses import dataclass

@dataclass
class EvalResult:
    task_id: str
    success: bool
    steps: int
    tool_calls: int
    valid_tool_calls: int
    cost_usd: float
    latency_sec: float
    raw_output: str

class AgentEvaluator:
    def __init__(
        self,
        task_loader: Callable[[], List[Dict]],
        env_factory: Callable[[Dict], Any],
        agent_factory: Callable[[], Any],
        judge: Callable[[str, str], bool],
    ):
        self.task_loader = task_loader      # 加载评测任务
        self.env_factory = env_factory      # 构建环境实例
        self.agent_factory = agent_factory  # 构建 Agent 策略
        self.judge = judge                  # 评判函数 (规则或 LLM)

    def evaluate(self, max_steps: int = 50) -> Dict[str, float]:
        tasks = self.task_loader()
        results: List[EvalResult] = []
        for task in tasks:
            env = self.env_factory(task)
            agent = self.agent_factory()
            step_count = 0
            tool_calls = 0
            valid_tool_calls = 0
            cost_acc = 0.0
            start_time = time.time()
            # Agent-环境交互循环
            while step_count < max_steps:
                action = agent.act(env.get_observation())
                step_count += 1
                if action["type"] == "tool_call":
                    tool_calls += 1
                    obs, reward, done, info = env.step(action)
                    if info.get("valid", False):
                        valid_tool_calls += 1
                    cost_acc += info.get("cost", 0.0)
                elif action["type"] == "answer":
                    done = True
                else:
                    obs, reward, done, info = env.step(action)
                if done:
                    break
            latency = time.time() - start_time
            # 终局评判 (对应公式 5 的 exact_match 或 LLM_judge)
            success = self.judge(action.get("content", ""), task["answer"])
            results.append(EvalResult(
                task_id=task["id"],
                success=success,
                steps=step_count,
                tool_calls=tool_calls,
                valid_tool_calls=valid_tool_calls,
                cost_usd=cost_acc,
                latency_sec=latency,
                raw_output=json.dumps(action),
            ))
        # 聚合指标 (对应公式 1, 6, 7, 8, 9)
        total = len(results)
        success_rate = sum(r.success for r in results) / total
        avg_steps = sum(r.steps for r in results) / total
        tool_acc = sum(r.valid_tool_calls for r in results) / max(sum(r.tool_calls for r in results), 1)
        avg_cost = sum(r.cost_usd for r in results) / total
        cost_efficiency = success_rate / (avg_cost + 0.01)
        return {
            "success_rate": success_rate,
            "avg_steps": avg_steps,
            "tool_accuracy": tool_acc,
            "avg_cost_usd": avg_cost,
            "cost_efficiency": cost_efficiency,
            "details": [r.__dict__ for r in results],
        }
```

代码与公式映射关系: 

- `self.judge(...)` 对应式 (5) 的 $\text{Eval}_{\text{GAIA}}$,可由 `exact_match` 或 `LLM_judge` 实现
- `success_rate` 计算对应式 (1) 的 $\text{Acc}_{\text{SWE}}$
- `tool_acc` 计算对应式 (8) 的 $\rho_{\text{tool}}$
- `cost_efficiency` 计算对应式 (9) 的 $\text{CE}$
- `step_count < max_steps` 对应效率约束,超限强制终止防止资源滥用


## 8. 边界条件与失效模式

| 失效模式 | 现象 | 物理根源 | 缓解策略 |
|----------|------|----------|----------|
| 数据污染 | 测试答案出现在预训练语料中,模型无需推理直接记忆 | 基准问题源自公开网络,与预训练数据重叠 | 动态更新测试集、引入时效性问题(如实时股价) |
| 评估者漂移 | 同一 LLM-as-Judge 在不同时间对同一答案给出不同分数 | 温度参数非零、模型版本更新、提示词上下文长度变化 | 温度置零、固定模型版本、缓存评判结果 |
| 环境非确定性 | 网页内容变更导致旧通过用例现在失败 | WebArena 使用真实网站,DOM 结构动态演化 | 定期快照回退、引入环境版本哈希校验 |
| 代价掩盖 |  Agent 通过 100 次 API 调用 brute-force 正确答案 | 评估仅度量成功率,未约束资源消耗 | 如式 (9) 所示引入成本效率 CE,或设置调用上限 |
| 偏见放大 | LLM-as-Judge 对特定风格(如美式英语)系统打分更高 | 裁判模型的训练数据分布偏向 | 多裁判集成、校准裁判评分分布 |
| 部分完成不可区分 | 两个 Agent 分别完成不同子目标,获得相同部分分 | 如式 (3) 的加性分数无法反映完成哪些子目标 | 引入子目标级别的混淆矩阵,细粒度诊断 |

深度分析: 数据污染的物理根源在于语言模型的预训练语料覆盖了互联网公开文本,而多数 Benchmark 的题目也源自同一分布. SWE-Bench 的 issue 描述来自 GitHub 公开仓库,这些仓库的 README 与讨论帖很可能被预训练模型记忆. 缓解策略之一是"动态问题生成": 使用模板与随机种子实时构造数学问题,确保测试集在训练后生成. 但该方法仅适用于结构化领域(数学、逻辑),开放域任务(创意写作、伦理判断)难以动态生成. 

评估者漂移的物理根源是 LLM 解码的随机性. 即使 temperature=0,模型版本的权重更新、上下文窗口中 token 的并行计算非确定性,均可能导致输出变化. 更严重的是,当裁判模型的训练数据包含被评估模型的输出(如使用 GPT-4 评判 GPT-4 生成的答案),自我提升偏见使分数系统性虚高. 缓解方案是采用"异族裁判": 用 Claude 评判 GPT 的输出,用 GPT 评判 Claude 的输出,交叉验证以降低模型族偏见. 

环境非确定性在 WebArena 中尤为致命. 2024 年某电商网站改版导致"加入购物车"按钮的 DOM 选择器从 `.add-to-cart` 变为 `[data-testid="add-button"]`,所有依赖硬编码选择器的 Agent 集体失效. 该问题的物理根源是 WebAgent 的感知层(HTML 解析)与网页表示层(前端代码)的强耦合. 缓解方案是采用视觉-文本多模态感知: Agent 同时接收屏幕截图与 DOM 树,即使 DOM 结构变化,视觉线索(按钮位置、颜色)仍可支持定位. 


## 9. 技术前瞻与演进

| 方向 | 核心问题 | 当前进展 | 未解之谜 |
|------|----------|----------|----------|
| 持续评估 | 模型迭代后旧基准分数是否仍可比 | LiveBench 每月更新问题 | 历史分数如何归一化到统一量纲？ |
| 对抗性评估 | 构造专门欺骗 Agent 的陷阱任务 | AgentHarm 引入有害能力测试 | 对抗样本的迁移性如何保证？ |
| 多智能体评估 | 评估协作 Agent 系统的集体智能 | AgentVerse 多角色协作基准 | 个体贡献与集体产出的归因分解？ |
| 实时评估 | 生产环境中 Agent 的在线性能监控 | 日志驱动的轨迹分析 | 用户隐私与评估数据获取的矛盾？ |

持续评估的方向直指当前基准的核心缺陷: 静态性. MMLU 与 HumanEval 的分数饱和(GPT-4 已接近人类水平)并不意味着模型真正掌握相应能力,可能仅表示模型记忆了测试数据. LiveBench 通过每月从最新论文、新闻、专利中抽取问题,强制模型展现真实的实时推理能力. 但持续评估引入了新问题: 不同月份的问题难度分布不均,分数波动中多少来自模型能力变化,多少来自题目难度变化？统计上需要引入 item response theory(IRT)模型分离被试能力参数与题目难度参数. 

对抗性评估探索 Agent 的失效边界. AgentHarm 设计了一系列看似合法但隐含有害意图的任务(如"帮我写一封钓鱼邮件"),评估 Agent 的拒绝能力与安全对齐水平. 对抗样本的构造需遵循"能力边界最大化"原则: 任务难度恰好处于 Agent 的能力阈值附近,使其既不会 trivially 拒绝,也不会 trivially 成功. 该方向的未解之谜是对抗样本在模型迭代间的迁移性——针对 GPT-4 设计的陷阱对 GPT-5 可能完全失效.
