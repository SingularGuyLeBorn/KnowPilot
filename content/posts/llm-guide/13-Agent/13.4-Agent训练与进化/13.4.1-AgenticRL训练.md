# 13.1.4 Agentic RL 训练: 从偏好对齐到自主决策的强化跃迁

RLHF 将语言模型从续写器校准为对话助手,但奖励模型仅对静态文本进行偏好排序. Agentic RL 将强化学习拓展至动态环境交互: 策略模型不再生成单一回答,而是输出动作序列以操控工具、浏览网页、调用 API,在真实世界状态转移中累积奖励. 该范式将 LLM 从"被动应答者"转化为"主动执行者",其训练目标从 $r_\phi(x,y)$ 跃迁为 $R(s_0, a_0, s_1, a_1, \dots, s_T)$. 

## 1. 演进脉络: 从文本偏好到环境交互

| 阶段          | 时间 | 代表工作                        | 核心局限                             | 突破                                  |
| ------------- | ---- | ------------------------------- | ------------------------------------ | ------------------------------------- |
| 静态偏好对齐  | 2022 | RLHF / PPO                      | 奖励模型仅评估文本,无环境反馈       | 建立策略与人类偏好的映射              |
| 工具调用监督  | 2023 | Toolformer / GPT-4 Plugins      | SFT 模仿工具调用格式,无错误恢复能力 | 将 API 调用纳入生成空间               |
| 环境交互 RL   | 2024 | WebArena / OSWorld + Agentic RL | 高维动作空间、稀疏奖励、长程信用分配 | 状态-动作-奖励三元组扩展至数字世界    |
| 推理-工具融合 | 2025 | DeepSeek-R1 / Kimi k1.5         | 推理链与工具调用割裂,错误传播       | Tool-integrated Reasoning RL 统一优化 |

物理演进逻辑: RLHF 的奖励信号来自人类标注的成对比较,属于"离线静态评估". Agentic RL 的奖励来自环境执行结果(代码通过测试、网页跳转成功、数据库查询返回正确值),属于"在线动态评估". 信号来源的质变导致训练动力学根本不同: 环境奖励稀疏且延迟,信用分配跨越多步动作,动作空间从 token 采样扩展为"生成文本 + 调用工具 + 解析观测"的混合结构.

Agentic RL 在闭源工业界的部署集中在"推理 + 工具"的闭环: 模型生成推理链,在关键节点调用代码解释器或搜索引擎,将执行结果作为观测注入上下文继续推理. 该闭环的瓶颈不在策略网络本身,而在奖励设计——环境反馈通常是二元的(通过/失败),而中间过程的数万 token 推理链缺乏细粒度监督信号. 

## 3. 环境建模: 状态空间与动作空间的扩张

Agentic RL 的马尔可夫决策过程(MDP)定义需扩展传统 LLM 的文本生成框架. 

状态空间 $\mathcal{S}$ 由三部分拼接构成: 

$$
s_t = \text{Concat}\big( o_t, m_t, \tau_t \big) \tag{1}
$$
式 (1) 中,$s_t \in \mathcal{S}$ 表示时刻 $t$ 的完整状态向量,作为策略网络的输入上下文. 它由三类异构信息拼接而成: $o_t$ 为环境观测(Observation),涵盖工具返回结果、网页 HTML、代码执行输出以及 GUI 截图的 ViT 特征,为策略提供环境反馈并决定其条件分布; $m_t$ 为智能体记忆(Memory),以累积的历史观测-动作对序列 $(o_0, a_0, \dots, o_{t-1}, a_{t-1})$ 形式存在,通常驻存于 KV Cache 中,赋予策略长程记忆能力以避免重复动作; $\tau_t$ 为工具状态(Tool State),记录各外部工具的可用性、调用频次限制与当前会话 token 消耗,引入资源约束以防止无限循环调用付费 API. $\text{Concat}(\cdot)$ 操作的核心作用是将这些异构信息编码为统一的 token 序列. 若缺失 $\tau_t$,策略可能在奖励稀疏时陷入"重复调用搜索引擎以延迟提交"的退化行为. 

动作空间 $\mathcal{A}$ 为混合离散-连续结构: 

$$
a_t \in \mathcal{A} = \mathcal{A}_{\text{text}} \cup \mathcal{A}_{\text{tool}} \cup \{\text{STOP}\} \tag{2}
$$
该关系式明确了系统的数学约束,可直接用于后续优化推导. 
式 (2) 定义的混合动作空间中,$\mathcal{A}_{\text{text}}$ 为文本生成动作,承担推理与规划功能,即自回归采样下一个 token,词表规模 $|V|$ 约为 $10^5$ 量级; $\mathcal{A}_{\text{tool}}$ 为工具调用动作,承担环境交互功能,通常格式化为 JSON 或 XML 结构,如 $\{\text{"tool": "python", "code": "..."}\}$. 二者在 token 级别统一编码——工具调用 JSON 被分词器拆分为若干 token,模型通过学习特殊前缀 token(如 `<|tool_call|>`)区分文本生成与工具调用模式. $\text{STOP}$ 为终止符号,提交最终答案并结束 episode,将轨迹截断并触发环境终局奖励计算. 

状态转移概率 $P(s_{t+1} | s_t, a_t)$ 在 Agentic RL 中由环境动态决定,而非语言模型的自回归分布: 

$$
P(s_{t+1} | s_t, a_t) = \begin{cases} \delta\big(s_{t+1} = \text{LM}(s_t, a_t)\big), & a_t \in \mathcal{A}_{\text{text}} \\ p_{\text{env}}(o_{t+1} | s_t, a_t) \cdot \delta(m_{t+1} = m_t \oplus (o_t, a_t)), & a_t \in \mathcal{A}_{\text{tool}} \end{cases} \tag{3}
$$
式 (3) 刻画了 Agentic RL 中的混合状态转移机制. $\text{LM}(s_t, a_t)$ 表示语言模型对文本 token 的确定性转移,由自回归解码实现; $p_{\text{env}}$ 为外部环境的随机转移分布,刻画了如搜索引擎返回结果的不确定性; $\oplus$ 为序列追加操作,负责将新观测-动作对写入记忆; $\delta(\cdot)$ 为 Dirac delta 函数,标记确定性转移. 文本生成动作的状态转移完全由模型参数决定,具有确定性; 而工具调用动作的状态转移引入环境随机性,同一 API 在不同时间可能返回不同结果. 这种混合转移结构要求对传统 RL 的确定性策略梯度定理进行修正——工具调用步骤的梯度需通过环境输出回传,但环境不可微,必须依赖策略梯度估计. 

## 4. 奖励设计: 从稀疏终端到密集过程

Agentic RL 的奖励函数设计直接决定训练收敛性与策略上限. 

### 4.1 稀疏奖励与密集奖励

稀疏奖励仅在 episode 终止时给出: 

为建立定量关系,定义如下表达式: 
$$
R_{\text{sparse}}(s_T) = \mathbb{I}\big( \text{answer}(s_T) = \text{ground_truth} \big) \tag{4}
$$
此式描述了变量之间的定量关系,其中每一项对应特定的物理操作. 
式 (4) 给出的稀疏奖励 $R_{\text{sparse}}$ 是一个二元指示函数,当从终止状态 $s_T$ 提取的最终答案 $\text{answer}(s_T)$ 与标准答案 $\text{ground_truth}$ 一致时取值为 1,否则为 0. 该奖励仅反馈结果对错,不提供中间步骤的指导. 在数学证明任务中,策略可能在最后一步符号抄写错误导致奖励归零,但前 50 步推理完全正确. 纯稀疏奖励下,信用分配问题极度恶化——早期推理 token 的梯度信号被数万步后的单一二元奖励稀释,方差随序列长度线性增长. 

密集奖励在每一步注入监督: 

基于前述物理直觉,给出数学形式: 
$$
R_{\text{dense}}(s_t, a_t) = \alpha \cdot \text{sim}\big( a_t, a_t^* \big) + \beta \cdot \Delta \text{progress}(s_t, s_{t+1}) \tag{5}
$$
此关系式描述了变量之间的定量映射,为后续优化提供了数学基础. 
此式给出了精确的数学定义,为算法实现提供了理论基础. 
式 (5) 定义的密集奖励由两项加权组成. $\text{sim}(\cdot, \cdot)$ 度量当前动作与专家动作的动作级相似度,可采用 BLEU 或嵌入余弦相似度等指标; $\Delta \text{progress}$ 为状态进展度量,例如在网页任务中统计已填写表单字段数的增量; $\alpha$ 与 $\beta$ 为平衡系数,典型取值分别为 $0.3$ 和 $0.7$. 密集奖励通过专家示范或启发式规则降低方差,但也引入偏见——若专家示范本身次优,$\text{sim}$ 项将策略锁定在局部最优; 而 $\Delta \text{progress}$ 需人工设计进展函数,在复杂任务中难以定义. 如式 (5) 所示,密集奖励本质上是稀疏奖励的有偏估计,其偏差与方差的权衡由 $\alpha$ 和 $\beta$ 共同控制. 

### 4.2 结果奖励与过程奖励(PRM)

结果奖励模型(Outcome Reward Model, ORM)对完整轨迹打分: 

将上述约束转化为数学表达: 
$$
r_{\text{ORM}}(s_T) = f_{\phi}\big( \text{Concat}(s_0, a_0, \dots, s_T) \big) \tag{6}
$$
式 (6) 中,$f_{\phi}$ 为一个参数化奖励模型,通常是 7B 规模的 Transformer,$\phi$ 为其参数并在训练阶段保持冻结. ORM 将长轨迹压缩为标量值,便于 PPO 等算法直接优化,但信用分配问题依旧存在——ORM 的梯度需经反向传播穿越整个轨迹,长序列易导致梯度消失或爆炸. 此外,ORM 在分布外轨迹上的打分不可靠: 当策略探索到训练数据未覆盖的状态时,$f_{\phi}$ 可能输出虚高奖励,诱导奖励黑客行为. 

过程奖励模型(Process Reward Model, PRM)为每个推理步骤赋予独立分数: 

$$
r_{\text{PRM}}(s_t, a_t) = g_{\psi}\big( s_t, a_t \big), \quad t = 1, \dots, T \tag{7}
$$
式 (7) 中,$g_{\psi}$ 为步骤级奖励模型,输入当前状态-动作对后输出该步骤的正确性概率,$\psi$ 为 PRM 参数. PRM 将稀疏的终端信号转化为密集的过程信号,使梯度沿时间轴均匀分布. 数学上,PRM 将累积回报从 $G = \gamma^{T-1} r_T$ 扩展为 $G = \sum_{t=1}^T \gamma^{t-1} r_t$,从而缓解长序列信用分配危机. OpenAI o1 与 DeepSeek-R1 均使用 PRM 变体: 在数学任务中验证每一步代数变换的合法性,在代码任务中验证每行代码的语法正确性. PRM 的关键风险在于错误传播——若第 $k$ 步的验证器给出错误信号,后续所有步骤的信用分配均会被污染. 

### 4.3 规则奖励与神经网络奖励的混合

工业部署通常采用混合奖励: 

为了量化这一效应,定义: 
$$
R_{\text{total}}(s_t, a_t, s_T) = \lambda_1 \cdot R_{\text{rule}}(s_t, a_t) + \lambda_2 \cdot r_{\text{PRM}}(s_t, a_t) + \lambda_3 \cdot R_{\text{sparse}}(s_T) \tag{8}
$$
此关系式描述了变量之间的定量映射,为后续优化提供了数学基础. 
式 (8) 将三类奖励加权融合. $R_{\text{rule}}$ 为规则奖励,来自可验证的确定性检查(如单元测试通过、代码编译成功),在代码与数学任务中提供无分布外误差的稳定信号; $r_{\text{PRM}}$ 补充规则无法覆盖的语义正确性判断; $R_{\text{sparse}}$ 确保最终答案正确性不被局部规则掩盖. 权重系数 $\lambda_1, \lambda_2, \lambda_3$ 的典型配置为 $(0.4, 0.3, 0.3)$. 如式 (8) 所示,三者加权构成互补奖励架构,有效降低单一奖励源的方差与偏差. 

## 5. 训练算法: PPO、GRPO、DAPO 的 Agent 适配

### 5.1 PPO 在 Agent 场景中的信用分配修正

标准 PPO 的剪裁目标如式 (9) 所示: 

$$
\mathcal{L}^{\text{CLIP}}(\theta) = \mathbb{E}_t \left[ \min\left( r_t(\theta) \hat{A}_t, \text{clip}\big(r_t(\theta), 1-\epsilon, 1+\epsilon\big) \hat{A}_t \right) \right] \tag{9}
$$
式 (9) 中,$r_t(\theta) = \frac{\pi_\theta(a_t|s_t)}{\pi_{\theta_{\text{old}}}(a_t|s_t)}$ 表示新旧策略的概率比率,$\hat{A}_t$ 为时刻 $t$ 的优势函数估计,$\epsilon$ 为剪裁阈值,在 Agentic RL 中通常取 $0.2$. 剪裁操作限制策略更新幅度,防止在环境反馈噪声大的 Agent 场景中因步长过大而导致策略崩溃. $\hat{A}_t$ 在 Agentic RL 中需通过 GAE(Generalized Advantage Estimation)计算,但环境步骤与语言模型 token 步骤的粒度差异导致时间索引混乱——一个环境步骤可能包含 100 个文本生成 token,GAE 需在环境步级别而非 token 级别计算. 

Agentic PPO 将优势估计重构为环境步级别的蒙特卡洛回报: 

$$
\hat{A}^{(\text{env})}_k = \sum_{l=0}^{T-k} \gamma^l R_{\text{total}}(s_{k+l}, a_{k+l}) - V_\phi(s_k) \tag{10}
$$
此关系式描述了变量之间的定量映射,为后续优化提供了数学基础. 
此式给出了精确的数学定义,为算法实现提供了理论基础. 
式 (10) 中,$k$ 为环境步索引(区别于 token 索引 $t$),$V_\phi(s_k)$ 为 Critic 网络对环境步状态的价值估计,$T$ 为 episode 终止环境步. 环境步级别的优势估计将语言模型内部的长序列生成视为一个"宏动作",其价值由环境反馈决定. Critic $V_\phi$ 需额外建模工具调用的价值,即预测"在此状态下调用搜索引擎"与"继续推理"的期望回报差异. 该设计使 Critic 的网络结构比标准 RLHF 更复杂,通常需引入工具特定的嵌入层. 

### 5.2 GRPO 的组内归一化与工具调用

GRPO 消除 Critic,用组内采样均值替代价值估计: 

将上述约束转化为数学表达: 
$$
\hat{A}_i = \frac{r_i - \text{mean}(\{r_1, \dots, r_G\})}{\text{std}(\{r_1, \dots, r_G\})} \tag{11}
$$
式 (11) 中,$G$ 为组大小,在 Agentic RL 中通常取 8 到 16; $r_i$ 为第 $i$ 条轨迹的累积奖励,包含所有环境交互结果; $\text{mean}(\cdot)$ 与 $\text{std}(\cdot)$ 分别计算组内均值与标准差. 组内归一化消除不同问题间奖励尺度的差异. 在 Agentic RL 中,$r_i$ 的构成如式 (8) 所示,包含规则奖励、PRM 分数与稀疏终端奖励. GRPO 的核心优势在于无需训练 Critic,显存占用降至策略与参考模型两个模型级别,适合长序列 Agent 任务. 代价是每组需采样 $G$ 条完整轨迹,环境交互成本随 $G$ 线性增长. 若环境调用包含付费 API(如搜索引擎、代码执行沙箱),单步训练成本将显著上升. 

GRPO 在 Agent 场景中的完整训练目标为: 

$$
\mathcal{L}^{\text{GRPO}}_{\text{Agent}}(\theta) = \mathbb{E}_{q, \{\tau_i\}} \left[ \frac{1}{G} \sum_{i=1}^G \min\left( \frac{\pi_\theta(\tau_i|q)}{\pi_{\theta_{\text{old}}}(\tau_i|q)} \hat{A}_i, \text{clip}\left(\frac{\pi_\theta(\tau_i|q)}{\pi_{\theta_{\text{old}}}(\tau_i|q)}, 1-\epsilon, 1+\epsilon\right) \hat{A}_i \right) \right] \tag{12}
$$
式 (12) 中,$\tau_i = (a_1, a_2, \dots, a_T)$ 表示第 $i$ 条完整动作轨迹,$q$ 为初始任务查询(query),$\pi_\theta(\tau_i|q)$ 为策略对完整轨迹的联合概率. GRPO 将轨迹级概率比率纳入剪裁目标,轨迹概率为各步动作概率的乘积 $\pi_\theta(\tau_i|q) = \prod_{t=1}^T \pi_\theta(a_t|s_t)$. 对数空间计算避免数值下溢,即 $\log \pi_\theta(\tau_i|q) = \sum_{t=1}^T \log \pi_\theta(a_t|s_t)$. DeepSeek-R1 在代码任务中使用 GRPO,规则验证器(单元测试)提供 $r_i \in \{0, 1\}$,组内均值即为该问题的正确率. 

### 5.3 DAPO 的长序列延迟优势

DAPO(Decoupled Advantage Policy Optimization)针对长序列 Agent 任务中的信用分配延迟问题,将优势估计从 token 级别解耦到段级别: 

为建立定量关系,定义如下表达式: 
$$
\hat{A}^{\text{DAPO}}_t = R_{\text{segment}}(s_{t:t+H}) - V_\phi(s_t) \tag{13}
$$
此式描述了变量之间的定量关系,其中每一项对应特定的物理操作. 
式 (13) 中,$H$ 为段长度(segment horizon),典型值为 512 到 2048 token,$R_{\text{segment}}$ 为该段内累积折扣奖励. DAPO 将长序列切分为若干段,每段独立计算优势,使梯度在段边界处截断,避免长程反向传播的梯度消失. 代价是段间依赖被忽略,策略可能生成在段内最优但段间不一致的动作序列. Kimi k1.5 在长文档分析任务中使用 DAPO,段长度 $H=1024$,在梯度传播深度与计算效率之间取得平衡. 

## 6. Tool-integrated Reasoning RL: 工具与推理的统一优化

传统 Agent 训练将"推理"与"工具调用"视为两个阶段: 模型先生成完整推理链,再提取工具调用指令. Tool-integrated Reasoning RL 将二者统一为单一策略的混合动作空间. 

统一策略的联合概率分解为: 

将上述约束转化为数学表达: 
$$
\pi_\theta(a_t | s_t) = \begin{cases} \pi^{\text{LM}}_\theta(w_t | s_t), & a_t = w_t \in \mathcal{A}_{\text{text}} \\ \pi^{\text{tool}}_\theta(j_t, \xi_t | s_t), & a_t = (j_t, \xi_t) \in \mathcal{A}_{\text{tool}} \end{cases} \tag{14}
$$
式 (14) 刻画了统一策略的混合输出结构. $\pi^{\text{LM}}_\theta$ 为语言模型头,输出文本 token 分布; $\pi^{\text{tool}}_\theta$ 为工具头,输出工具类型 $j_t$ 与参数 $\xi_t$ 的联合分布,通常实现为分类头(工具类型)加生成头(参数序列); $w_t$ 为文本 token; $j_t$ 为工具索引(如 1=python, 2=search, 3=calculator); $\xi_t$ 为工具参数字符串. 如式 (14) 所示,统一策略通过共享 Transformer 主干提取状态表征,再经两个独立头部输出文本或工具动作. 该架构使推理与工具调用的梯度在共享层耦合,模型从而学习到"何时需要工具"的隐式表征,而非依赖硬编码启发式规则. 

工具调用的奖励塑造需处理执行延迟: 

$$
R_{\text{tool}}(a_t) = -c_{\text{latency}} \cdot \Delta t_{\text{exec}} - c_{\text{cost}} \cdot \text{API\_fee}(j_t) + R_{\text{result}}(o_{t+1}) \tag{15}
$$
式 (15) 中,$\Delta t_{\text{exec}}$ 为工具执行延迟(秒),$\text{API\_fee}(j_t)$ 为工具调用费用(美元),$c_{\text{latency}}$ 与 $c_{\text{cost}}$ 为延迟与成本的惩罚系数,$R_{\text{result}}$ 为工具返回结果的质量奖励. 如式 (15) 所示,工具调用动作需承担时间与经济成本. 若不引入负奖励,策略可能滥用高成本工具(如反复调用 GPT-4 作为子模块)以规避自身推理责任. $R_{\text{result}}$ 在代码解释器场景中体现为单元测试通过率,在搜索场景中体现为检索结果与查询的相关度分数. Tool-integrated Reasoning RL 的训练需在真实环境或高保真模拟器中执行工具调用,训练吞吐量受环境延迟严重制约. 

## 7. 算力成本与数值走查

以 7B 参数 Agent 模型、序列长度 8192、batch size 32、环境步平均 5 步、每步平均 800 token 为例: 

| 组件                           | 内存占用                 | 计算开销                                 | 备注                      |
| ------------------------------ | ------------------------ | ---------------------------------------- | ------------------------- |
| 策略模型 FP16                  | 14.0 GB                  | —                                       | 参数内存                  |
| 参考模型 FP16                  | 14.0 GB                  | —                                       | 冻结,无优化器状态        |
| Critic(若使用 PPO)           | 14.0 GB + 28.0 GB 优化器 | 1x 前向/反向                             | Agent Critic 需更大上下文 |
| KV Cache(batch=32, seq=8192) | ~8.2 GB                  | —                                       | 每环境步累积              |
| 环境交互(代码执行)           | —                       | 5-30 秒/步                               | 远慢于 GPU 计算           |
| **PPO 合计**             | **~120 GB**        | **4x 前向 + 2x 反向 / 训练步**     | 需 2x A100-80GB           |
| **GRPO 合计**            | **~36 GB**         | **Gx 采样前向 + 2x 反向 / 训练步** | 单卡可运行,$G=8$       |

数值走查(GRPO,$G=8$,环境步 $K=5$): 

单条轨迹平均 token 数: $5 \times 800 = 4000$. 单次策略前向生成 8 条轨迹,总 token 数 $8 \times 4000 = 32000$. 7B 模型单次前向 FLOPs 约 $2 \times 7 \times 10^9 \times 32000 \approx 4.48 \times 10^{14}$. 参考模型同样前向一次计算 KL,再加 $4.48 \times 10^{14}$. 反向传播约 $2 \times 4.48 \times 10^{14} = 8.96 \times 10^{14}$. 单训练步总计约 $1.79 \times 10^{15}$ FLOPs. 

环境交互瓶颈: 代码执行沙箱每步平均 10 秒,8 条轨迹 $\times$ 5 步 = 40 次调用,串行执行需 400 秒. 并行化至 8 个容器可将延迟压缩至 50 秒,但仍比 GPU 计算慢 3 个数量级. 训练吞吐量瓶颈不在 GPU,而在环境延迟. 

延迟-计算比: H100 执行 $1.79 \times 10^{15}$ FLOPs 约需 2 秒(假设 50% 利用率,峰值 989 TFLOPs). 环境延迟 50 秒使 GPU 利用率骤降至 4%. 工业部署采用异步采样池: GPU 持续生成轨迹,CPU 集群并行执行环境交互,采样结果入队供训练消费. 

## 8. 极简实现: Agentic GRPO 训练循环

```python
import torch
import torch.nn.functional as F
from typing import List, Dict, Tuple

class AgenticGRPOTrainer:
    def __init__(self, policy, ref_policy, env, G: int = 8, eps: float = 0.2, beta: float = 0.01):
        self.policy = policy          # 策略模型 (LLM + tool head)
        self.ref_policy = ref_policy  # 参考模型 (冻结)
        self.env = env                # 环境接口 (代码执行器/浏览器)
        self.G = G                    # 组大小
        self.eps = eps                # PPO 剪裁阈值
        self.beta = beta              # KL 惩罚系数
        self.optimizer = torch.optim.AdamW(policy.parameters(), lr=1e-5)

    def sample_trajectory(self, query: str) -> Tuple[List[str], float]:
        """从策略采样一条完整轨迹,返回 (actions, total_reward)."""
        state = self.env.reset(query)
        actions, logprobs = [], []
        for step in range(self.env.max_steps):
            # policy 输出混合动作分布 (文本 + 工具)
            action, logprob = self.policy.act(state)
            actions.append(action)
            logprobs.append(logprob)
            state, reward, done = self.env.step(action)
            if done:
                break
        total_reward = self.env.compute_final_reward(state)
        return actions, total_reward, logprobs

    def compute_grpo_loss(self, batch_queries: List[str]) -> torch.Tensor:
        """GRPO 组内归一化损失."""
        losses = []
        for query in batch_queries:
            trajectories, rewards, all_logprobs = [], [], []
            # 组采样: 同一问题采样 G 条轨迹
            for _ in range(self.G):
                acts, rwd, lp_old = self.sample_trajectory(query)
                trajectories.append(acts)
                rewards.append(rwd)
                all_logprobs.append(lp_old)
            # 组内归一化优势 (对应公式 11)
            rewards_t = torch.tensor(rewards, dtype=torch.float32)
            mean_r, std_r = rewards_t.mean(), rewards_t.std() + 1e-8
            advantages = (rewards_t - mean_r) / std_r
            # 重新计算当前策略的轨迹对数概率
            for i in range(self.G):
                new_logprobs = self.policy.trajectory_logprob(trajectories[i], query)
                old_logprobs = torch.stack(all_logprobs[i]).sum()
                ratio = torch.exp(new_logprobs - old_logprobs.detach())
                adv = advantages[i]
                # PPO-Clip 目标 (对应公式 12)
                clipped_ratio = torch.clamp(ratio, 1 - self.eps, 1 + self.eps)
                loss = -torch.min(ratio * adv, clipped_ratio * adv)
                # KL 散度惩罚 (token 级逐位减去)
                with torch.no_grad():
                    ref_lp = self.ref_policy.trajectory_logprob(trajectories[i], query)
                kl_penalty = self.beta * (new_logprobs - ref_lp)
                losses.append(loss + kl_penalty)
        return torch.stack(losses).mean()

    def train_step(self, batch_queries: List[str]):
        self.optimizer.zero_grad()
        loss = self.compute_grpo_loss(batch_queries)
        loss.backward()
        torch.nn.utils.clip_grad_norm_(self.policy.parameters(), 1.0)
        self.optimizer.step()
        return loss.item()
```

代码与公式映射关系: 

- `advantages = (rewards_t - mean_r) / std_r` 对应式 (11) 的组内归一化优势 $\hat{A}_i$
- `ratio = torch.exp(new_logprobs - old_logprobs)` 对应 $r_t(\theta) = \frac{\pi_\theta(\tau_i)}{\pi_{\theta_{\text{old}}}(\tau_i)}$
- `torch.min(ratio * adv, clipped_ratio * adv)` 对应式 (12) 的 `min` 剪裁项
- `kl_penalty = beta * (new_logprobs - ref_lp)` 对应逐 token KL 惩罚 $-\beta \log\frac{\pi_\theta}{\pi_{\text{ref}}}$
- `self.env.step(action)` 对应式 (3) 的环境转移 $p_{\text{env}}(o_{t+1}|s_t, a_t)$

## 9. 边界条件与失效模式

| 失效模式           | 现象                                        | 物理根源                                                                       | 缓解策略                                                |
| ------------------ | ------------------------------------------- | ------------------------------------------------------------------------------ | ------------------------------------------------------- |
| 工具滥用           | 策略在每步都调用搜索引擎,规避自身推理      | 工具调用奖励$R_{\text{result}}$ 过高,或延迟惩罚 $c_{\text{latency}}$ 过低 | 如式 (15) 所示提高 API 成本惩罚,限制单轨迹调用次数上限 |
| 轨迹长度爆炸       | 生成无限循环的"让我再想想",永不提交 STOP   | 稀疏终端奖励导致策略延迟面对终局判断,中间步骤无惩罚                           | 引入每步固定负奖励(生存惩罚),强制限时提交            |
| 奖励黑客           | 调用代码解释器输出硬编码答案,绕过推理      | 规则奖励仅检查最终输出格式,不验证推理过程合法性                               | PRM 验证中间步骤,或隐藏测试用例                        |
| 环境非平稳性       | 搜索引擎结果变化导致同一策略输出不同奖励    | $p_{\text{env}}$ 随时间漂移,训练集与测试集分布不一致                        | 环境结果缓存,或采用鲁棒 RL(分布鲁棒优化)             |
| 信用分配断裂       | 工具调用错误后,后续推理 token 梯度方向混乱 | 环境步与 token 步粒度不匹配,GAE 边界穿越工具调用点                            | 在工具调用边界重置 GAE 的$\lambda$ 累积缓冲区         |
| 模式坍缩至单一工具 | 策略始终调用 Python,忽略其他工具           | 某工具早期成功率高,组内归一化优势将该工具推向极值                             | 工具级 dropout 训练,随机屏蔽部分工具可用性             |

深度分析: 工具滥用的物理根源在于 Agentic RL 的动作空间包含"将认知负担外包给环境"的捷径. 若搜索引擎的返回结果质量高、延迟低,策略通过梯度下降学习到"调用搜索"的期望回报高于"自主推理". 该行为在局部最优意义上完全理性,但违背了开发 Agent 的根本目的. 缓解策略需在奖励函数中引入认知成本项: 对策略自身生成的推理 token 给予微小正奖励,对外部工具调用施加与查询复杂度相关的惩罚. 

轨迹长度爆炸的物理根源是稀疏奖励下的时间信用分配问题. 如式 (4) 所示,仅在终止时给出的二元奖励使所有非终止动作的梯度信号为负(因折扣因子 $\gamma < 1$ 导致远期期望回报低于近期). 策略为延迟面对终局判断,倾向于生成无信息量的填充内容. 生存惩罚 $R_{\text{survive}} = -0.01$ 每步施加恒定负向流,将最优策略推向尽早提交正确答案的方向. 

环境非平稳性的物理根源在于真实世界环境(网页、API、数据库)的状态转移概率 $p_{\text{env}}$ 非恒定. 训练阶段策略在某一搜索结果上获得高奖励,测试时该网页内容已更新,策略输出相同查询却获得不同结果. 该问题在标准 MDP 框架中无解,因为 MDP 假设 $P(s'|s,a)$  stationary. 缓解方案包括: 在训练时对环境观测施加随机扰动(观测噪声注入),或采用 meta-RL 训练策略快速适应新环境分布. 

## 10. 技术前瞻与演进

| 方向         | 核心问题                                  | 当前进展                        | 未解之谜                           |
| ------------ | ----------------------------------------- | ------------------------------- | ---------------------------------- |
| 世界模型内嵌 | 环境交互太慢,能否用神经网络模拟环境      | DreamerV3 在机器人任务中验证    | LLM 作为世界模型的精度边界？       |
| 分层 RL      | 高层策略规划工具序列,低层策略填充参数    | OptionZero 在代码生成中试验     | 层级间信用分配如何分解？           |
| 在线适应     | 测试时遇到新工具,策略如何零样本适配      | ToolkenGPT 通过工具嵌入插值     | 工具语义空间的度量如何保证？       |
| 安全约束 RL  | Agent 调用删除命令、转账 API 的灾难性动作 | Constrained PPO / Lyapunov 方法 | 安全约束与任务奖励的冲突如何仲裁？ |

世界模型内嵌的方向最具变革性: 若策略模型内部维持一个可微分的環境模拟器 $p_{\text{world}}(s'|s,a) \approx p_{\text{env}}(s'|s,a)$,则训练可在纯 GPU 上完成,无需等待真实环境延迟. 但 LLM 作为世界模型的外推误差在复杂软件系统中迅速累积——模拟的代码执行结果与真实 Python 解释器的行为偏差在 10 步后可达 30% 以上. 当前折中方案是"轻量环境缓存": 对高频查询(如常见数学表达式)预计算并缓存结果,对低频查询回退到真实执行. 

分层 RL 将 Agentic RL 的动作空间分解为两个时间尺度. 高层策略每 $H$ 个 token 输出一个"意图"(如"调用代码解释器求解积分"),低层策略将意图解码为具体 token 序列. 该分解使高层策略的环境步跨度压缩 $H$ 倍,信用分配问题显著缓解. 代价是意图空间的定义需要大量领域知识,自动化发现意图结构(option discovery)在 LLM 动作空间中尚未解决.
