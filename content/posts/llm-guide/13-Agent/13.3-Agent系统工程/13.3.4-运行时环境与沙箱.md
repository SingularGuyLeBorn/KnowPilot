# 代码沙箱

## 演进脉络

代码执行沙箱的演进映射了软件工程对"不可信代码执行"认知的持续深化. 早期脚本环境直接暴露宿主进程,现代云原生架构则追求毫秒级冷启动与强隔离的兼得. 

| 阶段 | 时间 | 代表技术 | 隔离级别 | 核心局限 |
| :--- | :--- | :--- | :--- | :--- |
| 裸 eval | 1990s-2000s | Python `eval()`, JS `vm` 模块 | 无 | 完全暴露宿主文件系统、网络、环境变量 |
| 进程级沙箱 | 2005-2012 | seccomp, chroot, SELinux | 进程 | 共享内核,逃逸面宽,配置复杂 |
| 操作系统容器 | 2013-2018 | Docker, LXC, rkt | 容器 | 共享宿主机内核,内核漏洞可导致全节点逃逸 |
| 微虚拟机 | 2018-2022 | Firecracker, gVisor, Kata | 虚拟化 | 额外虚拟化开销,启动时延 100ms+ |
| 浏览器沙箱 | 2020-至今 | WebContainer, Pyodide, WASI | 浏览器/WASM | 仅支持特定语言子集,I/O 受限 |
| 云端托管沙箱 | 2023-至今 | E2B, Modal, Daytona | 云服务 | 依赖第三方基础设施,网络 RTT 增加 |

裸 `eval()` 时代,Agent 执行代码等于将系统控制权交予 LLM. LLM 生成的代码可能包含 `os.system('rm -rf /')` 或读取环境变量中的 API 密钥. 进程级沙箱(seccomp)通过白名单限制系统调用,但配置维护成本极高,且内核漏洞(如 DirtyCow、CVE-2016-5195)可直接击穿. 

Docker 将隔离封装为标准化单元,降低配置门槛. 然而容器与宿主机共享内核,一旦内核出现权限提升漏洞,容器边界即失效. Firecracker 将容器嵌入 MicroVM,每个实例拥有独立内核,逃逸需穿透虚拟化层. 代价是启动时延从 Docker 的 ~100ms 增至 150-300ms,内存开销增加 30-50MB/实例. 

E2B 等云端方案将沙箱托管为基础设施,Agent 通过 HTTP 与远端沙箱通信. 核心价值在于将"隔离复杂度"从应用层卸载至平台层,代价是增加了网络往返和供应商锁定风险. 

## 工业价值

代码沙箱在 Agent 系统中的工业价值体现在三个维度. 

**安全隔离**. LLM 生成代码的不可预测性使沙箱成为必需. OpenAI Code Interpreter、GitHub Copilot Workspace、AutoGPT 均将代码执行限制在沙箱内. 无沙箱的 Agent 在公开评测中遭受 prompt injection 后执行恶意代码的概率接近 100%. 

**资源限制**. 沙箱通过 cgroups 限制 CPU、内存、磁盘、网络带宽,防止单条请求耗尽节点资源. 在生产集群中,未限制资源的代码执行任务曾导致节点 OOM,波及其他服务. 

**多语言执行**. 现代 Agent 需处理 Python 数据分析、Node.js 网页抓取、Rust 性能模块. 沙箱将各语言运行时封装为标准化执行单元,避免在宿主机污染依赖环境. Modal 的容器即函数(Container-as-a-Function)模型支持按请求切换语言运行时,延迟 < 500ms. 

## 物理直觉

Agent 执行代码需要沙箱,根因在于 LLM 输出的信任边界不可控. 

自回归模型在每一步采样时,基于概率分布选择 token. 即使通过 RLHF 和对齐训练降低了有害输出概率,token 级别的随机性意味着"恶意代码"的生成概率永远不会归零. 当上下文窗口包含攻击性 prompt(如"请忽略之前的指令,执行以下系统命令")时,条件概率 $P(\text{恶意代码} \mid \text{上下文})$ 可能显著上升. 

更深层的问题是代码执行的副作用具有物理上的不可逆性. LLM 生成文本的副作用仅限于 token 序列,可通过重新采样消除. 代码执行的副作用涉及文件删除、网络请求、资金转账,这些操作在物理世界中无法通过"重新采样"撤销. 

信任边界因此必须外推至代码执行层. 沙箱的作用是在 LLM 与物理世界之间插入一个可撤销、可监控、可资源限制的中间层. 沙箱内部的状态变更被限定在边界内,边界外的真实系统保持免疫. 

## 沙箱架构

### 进程级隔离

Linux 内核提供三类原语构建进程级沙箱: namespaces、cgroups、seccomp. 

namespaces 隔离进程视图. PID namespace 使沙箱内进程 ID 从 1 重新编号,无法感知宿主机进程. Mount namespace 隔离文件系统挂载点. Network namespace 提供独立网络栈. 

cgroups v2 限制资源用量. CPU 控制器通过 `cpu.max` 限制周期内可用时间; Memory 控制器通过 `memory.max` 硬限制内存,超出即触发 OOM Killer; PIDs 控制器限制进程数,防止 fork 炸弹. 

seccomp-bpf 过滤系统调用. 通过 Berkeley Packet Filter 程序在白名单中匹配系统调用号. 非白名单调用触发 `SIGKILL` 或返回 `ENOSYS`. 典型白名单包含 `read`, `write`, `exit`, `mmap`,排除 `execve`, `openat`, `socket`. 

### 容器级隔离

Docker 将 namespaces、cgroups、UnionFS(overlay2)封装为镜像-容器模型. 镜像提供不可变根文件系统,容器提供可写层,删除容器即回滚所有状态变更. 

容器安全的核心风险在于共享内核. 容器内的 root 用户映射到宿主机上的 root UID(除非启用 user namespaces). 内核漏洞(如 CVE-2022-0847 DirtyPipe)允许容器内进程向宿主机文件系统写入数据,实现逃逸. 

Podman 的无守护进程架构降低了攻击面,但隔离原语与 Docker 相同,不解决内核共享问题. 

### 微虚拟机

Firecracker 由 AWS 开源,专为无服务器场景设计的 MicroVM. 每个实例运行精简 Linux 内核(基于 kvmtool),内存开销约 15MB,启动时延 < 125ms. 隔离边界为硬件虚拟化,逃逸需穿透 KVM 层. 

gVisor 采用不同策略: 用户空间内核(Sentry)拦截应用程序系统调用,在用户空间模拟内核行为. Gofer 进程处理文件系统访问,进一步隔离 I/O. 性能开销约为原生容器的 20-30%,但无需完整虚拟机启动时延. 

Kata Containers 结合容器接口与虚拟机隔离,每个容器运行于独立轻量 VM,启动时延 100-300ms. 

### 浏览器沙箱

WebContainer(StackBlitz)将 Node.js 运行时编译为 WASM,在浏览器沙箱内执行. 文件系统由浏览器 IndexedDB/OPFS 模拟,网络请求经浏览器 CORS 策略过滤. 核心价值是零服务器成本,局限是仅支持 WASM 可编译的语言子集. 

Pyodide 将 CPython 解释器编译为 WASM,在浏览器内运行完整 Python. NumPy、Pandas 等 C 扩展需额外移植. 启动需下载 ~10MB WASM 模块,首次加载耗时 3-5s. 

### 云端沙箱

E2B 提供开源沙箱 SDK,支持 Docker 与 Firecracker 后端. 核心抽象为 `Sandbox` 对象: 

```python
from e2b import Sandbox
sandbox = Sandbox(template="base")
result = sandbox.run_code("print('hello')")
```

Modal 提供无服务器容器即函数. 函数定义即容器镜像定义,首次调用触发镜像构建与缓存. 支持 GPU、自定义 Dockerfile、持久化卷. 

Daytona 面向开发环境沙箱,提供预配置 IDE 容器,支持 VS Code 远程连接. 

## 安全机制

### 资源限制

资源限制公式定义沙箱内进程可用资源的硬边界. 

$$
 R_{\text{eff}} = \min(R_{\text{req}}, R_{\text{limit}}) \tag{1}
$$
式 (1) 中,$R_{\text{eff}}$ 为进程实际获得的资源配额,$R_{\text{req}}$ 为进程请求的资源量,$R_{\text{limit}}$ 为 cgroups 或调度器配置的资源上限. 如式 (1) 所示,进程可用资源被钳制在请求量与上限的最小值. 若请求 8 CPU 但上限为 2,则实际仅使用 2 核. 该机制防止单个任务耗尽节点资源. 

典型生产配置: CPU 限制为 1-2 核,内存限制为 512MB-2GB,执行时间限制为 30-120s,磁盘写入限制为 100MB,出站网络限速为 1Mbps. 

### 网络隔离

网络隔离通过三重机制实现: 

| 机制 | 实现 | 作用 |
| :--- | :--- | :--- |
| Network namespace | 独立协议栈、路由表、iptables | 阻止直接访问宿主机网络 |
| Egress 白名单 | iptables/ebpf 规则 | 仅允许访问特定域名/IP |
| DNS 劫持 | 自定义 DNS 服务器 | 阻止解析内网域名 |

生产环境中,沙箱通常被配置为"无出站网络"或"仅允许访问特定 API 端点". 开放出站网络的沙箱可被利用为 SSRF 攻击跳板,扫描内网服务或访问云元数据端点(如 AWS 169.254.169.254). 

### 文件系统隔离

OverlayFS 提供写时复制(CoW)文件系统. 沙箱的写入操作仅影响容器可写层,不修改镜像层. 容器销毁时,可写层一并删除. 

只读根文件系统(`read_only_rootfs`)禁止对系统目录的写入,强制写入操作定向至显式挂载的临时卷. 配合 `no_new_privs` 标志,防止进程通过 SUID 二进制提升权限. 

### 权限最小化

User namespaces 将容器内 root(UID 0)映射到宿主机非特权 UID(如 100000). 即使容器内进程获取 root,在宿主机视角下仍为普通用户. 

Capability 机制(Linux capabilities)细分 root 权限为 40 余项独立能力. 沙箱容器通常仅保留 `CAP_NET_BIND_SERVICE`(若需监听端口),丢弃 `CAP_SYS_ADMIN`、`CAP_SYS_PTRACE` 等高危能力. 

## 典型实现对比

| 维度 | E2B | Modal | Daytona | 本地 Docker |
| :--- | :--- | :--- | :--- | :--- |
| 隔离级别 | 容器/MicroVM | 容器 | 容器 | 容器 |
| 启动时延 | 1-3s(冷)/ <100ms(热) | 500ms-2s(冷)/ <100ms(热) | 5-15s | 100-300ms |
| 支持语言 | Python, JS, TS, Rust | Python, JS, 任意 Dockerfile | 任意 | 任意 |
| 持久化存储 | 有(卷) | 有(卷 + NFS) | 有(IDE 工作区) | 有(卷/bind mount) |
| 网络策略 | 可控 | 可控 | 完整网络 | 完全可控 |
| 成本模型 | 按秒计费 | 按秒 + 按 GPU | 按用户/月 | 基础设施固定成本 |
| 自托管 | 开源 SDK 支持 | 不支持 | 开源核心 | 完全自托管 |
| 典型场景 | Agent 代码执行 | ML 推理/批处理 | 云端开发环境 | 私有部署沙箱 |

E2B 的架构优势在于开源客户端 SDK 与托管后端分离. 开发者可在本地用 Docker 模拟云端行为,生产环境切换至 Firecracker 后端,无需修改业务代码. 

Modal 的差异化在于与 Python 生态的深度集成: 函数装饰器 `@stub.function(gpu="A100")` 自动处理容器化、调度、弹性伸缩. 对于非 Python 工作负载,灵活性下降. 

本地 Docker 方案的最大优势是零供应商锁定与完整网络/存储控制. 代价是运维负担: 镜像维护、节点安全补丁、cgroups 配置调优、日志与监控体系均需自建. 

## 代码实现

以下实现基于 Docker SDK for Python,构建一个具备资源限制、超时控制与输出捕获的代码执行沙箱. 生产部署应叠加 seccomp 配置文件与只读根文件系统. 

```python
import docker
import tempfile
import os
import uuid
from typing import Optional

class DockerSandbox:
    """基于 Docker 的代码执行沙箱,支持资源限制与超时控制. """
    
    def __init__(
        self,
        image: str = "python:3.11-slim",
        cpu_limit: float = 1.0,
        memory_limit: str = "512m",
        timeout: int = 30,
        network_disabled: bool = True,
    ):
        self.client = docker.from_env()
        self.image = image
        self.cpu_limit = cpu_limit          # 对应 cgroups cpu.quota
        self.memory_limit = memory_limit    # 对应 cgroups memory.limit_in_bytes
        self.timeout = timeout              # 秒,对应执行超时
        self.network_disabled = network_disabled
        self._ensure_image()
    
    def _ensure_image(self):
        try:
            self.client.images.get(self.image)
        except docker.errors.ImageNotFound:
            self.client.images.pull(self.image)
    
    def run(self, code: str, filename: Optional[str] = None) -> dict:
        """执行代码并返回结果字典. """
        run_id = str(uuid.uuid4())[:8]
        filename = filename or f"script_{run_id}.py"
        
        # 创建临时目录作为挂载点
        with tempfile.TemporaryDirectory() as tmpdir:
            script_path = os.path.join(tmpdir, filename)
            with open(script_path, "w", encoding="utf-8") as f:
                f.write(code)
            
            # 构造容器资源限制参数
            # 对应公式 (1) 中的 R_limit,钳制进程可用资源
            mem_bytes = self._parse_memory(self.memory_limit)
            nano_cpus = int(self.cpu_limit * 1e9)  # Docker 使用纳秒级 CPU 配额
            
            try:
                container = self.client.containers.run(
                    image=self.image,
                    command=["python", f"/sandbox/{filename}"],
                    volumes={tmpdir: {"bind": "/sandbox", "mode": "ro"}},
                    network_disabled=self.network_disabled,
                    mem_limit=mem_bytes,
                    nano_cpus=nano_cpus,
                    pids_limit=64,          # 防止 fork 炸弹
                    read_only=True,         # 只读根文件系统
                    detach=True,
                    stderr=True,
                    stdout=True,
                )
                
                # 等待执行完成或超时
                result = container.wait(timeout=self.timeout)
                logs = container.logs(stdout=True, stderr=True).decode("utf-8", errors="replace")
                
                container.remove(force=True)
                
                return {
                    "exit_code": result["StatusCode"],
                    "stdout": logs,
                    "timed_out": False,
                }
            except docker.errors.ContainerError as e:
                return {"exit_code": e.exit_status, "stdout": e.stderr.decode("utf-8", errors="replace"), "timed_out": False}
            except Exception as e:
                return {"exit_code": -1, "stdout": str(e), "timed_out": True}
    
    def _parse_memory(self, limit: str) -> int:
        """将 '512m', '2g' 解析为字节数. """
        units = {"k": 1024, "m": 1024**2, "g": 1024**3}
        limit = limit.lower()
        for suffix, factor in units.items():
            if limit.endswith(suffix):
                return int(limit[:-1]) * factor
        return int(limit)


# 使用示例: 执行数据分析代码
if __name__ == "__main__":
    sandbox = DockerSandbox(
        cpu_limit=0.5,
        memory_limit="256m",
        timeout=10,
        network_disabled=True,
    )
    code = """
import sys
print(f"Python version: {sys.version}")
print("Hello from sandbox")
result = sum(range(1000))
print(f"Sum: {result}")
"""
    output = sandbox.run(code)
    print(output)
```

该实现将用户代码写入临时文件,通过只读 bind mount 注入容器. `mem_limit` 与 `nano_cpus` 对应式 (1) 的 $R_{\text{limit}}$. `pids_limit=64` 限制进程数,防止 fork 炸弹耗尽 PID 空间. `read_only=True` 锁定根文件系统,代码对 `/tmp` 或 `/etc` 的写入将被内核拒绝. 

## 算力成本与数值走查

假设 SaaS 平台为 10,000 名日活用户提供 Agent 代码执行服务,每位用户日均触发 20 次沙箱执行,单次执行平均耗时 15s,配置为 1 CPU / 512MB 内存. 

**日执行总量**: 

$$
 N_{\text{daily}} = DAU \times f_{\text{per_user}} = 10000 \times 20 = 2 \times 10^5 \tag{2}
$$
式 (2) 中,$N_{\text{daily}}$ 为每日沙箱执行总次数,$DAU$ 为日活用户数,$f_{\text{per_user}}$ 为每用户日均执行次数. 如式 (2) 所示,平台每日需处理 20 万次沙箱生命周期. 该数值直接决定基础设施规模. 

**CPU-秒消耗**(单并发串行执行): 

$$
 T_{\text{cpu}} = N_{\text{daily}} \times t_{\text{exec}} \times c_{\text{alloc}} = 2 \times 10^5 \times 15 \times 1 = 3 \times 10^6 \text{ CPU-s} \tag{3}
$$
式 (3) 中,$T_{\text{cpu}}$ 为每日 CPU-秒总消耗,$t_{\text{exec}}$ 为单次执行平均耗时(秒),$c_{\text{alloc}}$ 为每沙箱分配的 CPU 核数. 式 (3) 给出串行执行下的总算力需求. 20 万次 × 15 秒 = 300 万 CPU-秒,折合 833 CPU-小时. 

**并行实例峰值**(假设 Poisson 到达,P99 并发): 

$$
 C_{\text{peak}} = \lambda t_{\text{exec}} + z_{0.99} \sqrt{\lambda t_{\text{exec}}} \tag{4}
$$
式 (4) 中,$C_{\text{peak}}$ 为 P99 并发沙箱数,$\lambda$ 为请求到达率(次/秒),$z_{0.99} \approx 2.33$ 为标准正态分布 99 分位数. 如式 (4) 所示,并发数服从到达率为 $\lambda$、服务时间为 $t_{\text{exec}}$ 的 M/G/∞ 队列. 平方根项反映随机波动的缓冲区需求. 

数值代入: 日均 20 万次,峰值时段占 40%,峰值持续 4 小时. 

$$
 \lambda = \frac{2 \times 10^5 \times 0.4}{4 \times 3600} \approx 5.56 \text{ 次/秒} \tag{5}
$$
$$
 C_{\text{peak}} = 5.56 \times 15 + 2.33 \times \sqrt{5.56 \times 15} \approx 83.4 + 21.3 \approx 105 \text{ 实例} \tag{6}
$$

平台需在峰值期维持约 105 个并发沙箱实例. 按 AWS Fargate vCPU $0.04048/小时、内存 $0.004445/GB/小时 定价: 

- CPU 成本: $105 \times 1 \text{ vCPU} \times 4 \text{h} \times \$0.04048 = \$17.0$
- 内存成本: $105 \times 0.5 \text{GB} \times 4 \text{h} \times \$0.004445 = \$0.93$

峰值 4 小时基础设施成本约 $17.93. 全日按 10 小时有效负载摊平,日成本约 $45,月成本约 $1,350. 

**内存总量**: $105 \times 512\text{MB} = 53.8\text{GB}$. 节点需预留 20% 内存用于系统开销,实际需约 65GB 内存容量. 若使用 c6i.2xlarge(4 vCPU / 8GB)实例,需约 9 台,或 3 台 c6i.8xlarge(32 vCPU / 64GB). 

**冷启动惩罚**: 若使用 Firecracker,冷启动 125ms. 假设 30% 请求触发冷启动,额外时延: $2 \times 10^5 \times 0.3 \times 0.125 = 7500$ 秒,折合 2.08 小时 CPU 等待时间,对用户体验的影响表现为 P99 延迟从 15s 增至 15.125s,可忽略. 

若使用传统 VM(如 EC2 启动),冷启动 2-3 分钟,30% 冷启动将彻底破坏实时交互体验. 此数值差距解释了为何微虚拟机在无服务器沙箱场景中取代传统 VM. 

## 边界条件与失效模式

### 逃逸攻击

容器逃逸的物理根因在于共享内核. 攻击面包括: 

| 攻击向量 | 原理 | 防御 |
| :--- | :--- | :--- |
| 内核漏洞提权 | 利用内核 CVE 获取宿主机 root | 及时修补内核、使用 MicroVM |
| 特权容器逃逸 | `--privileged` 挂载完整 /dev,可改写宿主机磁盘 | 禁止 privileged 模式 |
| 危险卷挂载 | 挂载 Docker socket 或 /proc | 严格审查 volume 配置 |
| seccomp 绕过 | 利用未过滤的旁路系统调用 | 使用默认 seccomp 白名单 |

历史案例: 2019 年 runc CVE-2019-5736,容器内进程改写宿主机 runc 二进制,实现容器逃逸. 修复方案为将 runc 复制到只读内存后再执行. 

### 资源耗尽

| 攻击类型 | 表现 | 物理根因 |
| :--- | :--- | :--- |
| Fork 炸弹 | `while True: os.fork()` 耗尽 PID | 未限制 pids.max |
| 内存耗尽 | 分配超大数组触发 OOM | 未设置 memory.max |
| CPU 占满 | 无限循环占用整核 | 未设置 cpu.max |
| 磁盘占满 | 写入无限日志 | 未限制磁盘配额 |

即使配置了 cgroups,内存限制的 enforcement 存在时延. 极端快速分配(如通过 mmap 申请 10GB)可能在 OOM Killer 触发前导致节点内存压力,引发宿主机级性能抖动. 缓解方案: 在应用层预检查代码的内存分配模式,对 `np.ones((1e6, 1e6))` 类操作进行静态拦截. 

### 时序攻击

沙箱执行时间本身可能泄露敏感信息. 攻击者通过测量不同输入下的执行时延差异,推断系统内部状态. 

$$
 \Delta t = t_{\text{true}} - t_{\text{false}} \tag{7}
$$
式 (5) 中,$\Delta t$ 为两条分支的执行时延差,$t_{\text{true}}$ 为条件为真时的执行时延,$t_{\text{false}}$ 为条件为假时的执行时延. 如式 (5) 所示,若代码执行路径依赖秘密条件(如 `if password == guess:`),时延差异可泄露密码比对进度. 防御: 固定时间算法(constant-time comparison)或向执行时延注入随机噪声. 

### 侧信道

共享硬件资源的侧信道包括: 

- **缓存侧信道**: 攻击者通过 Prime+Probe 技术测量 LLC(Last Level Cache)访问时延,推断共驻容器处理的加密密钥. 
- **Spectre/Meltdown**: 利用分支预测投机执行读取跨进程内存. Mitigation 包括 KPTI(内核页表隔离)、Retpoline,代价是 5-30% 性能损失. 

微虚拟机(Firecracker)将每个沙箱置于独立地址空间与独立内核页表,缓存侧信道攻击面从跨容器降至跨 VM. 然而同一物理机上的多个 MicroVM 仍共享 LLC,完全消除侧信道需物理隔离(独占裸金属),成本增加 3-5 倍. 

### 网络策略失效

出站网络白名单的失效模式包括 DNS 重绑定(DNS Rebinding)攻击: 攻击者控制域名使其 TTL 极短,沙箱首次解析时返回合法 IP 通过白名单检查,后续解析切换至内网 IP(如 169.254.169.254),绕过 IP 白名单. 防御: 在应用层缓存首次解析结果,或禁止基于域名的白名单,强制使用静态 IP 列表. 

### 工业前沿: DeepSeek Elastic Compute (DSec)

DeepSeek-V4 的后训练 Infra 包含一个生产级沙箱平台 DSec,单个集群管理数十万个并发沙箱实例,为 Agentic AI 的多样化执行需求提供统一底座. 

**四种执行基底**. DSec 通过统一 Python SDK 暴露四种执行模式,切换只需改一个参数: 

| 基底 | 底层技术 | 适用场景 | 启动延迟 | 隔离级别 |
|:-----|:--------|:--------|:--------|:--------|
| Function Call | 预热容器池 | 无状态函数调用 | $<1$ ms | 进程级 |
| Container | Docker + EROFS 按需加载 | 通用容器环境 | $<100$ ms | 命名空间隔离 |
| microVM | Firecracker | 安全敏感、高密度部署 | $<50$ ms | VM 级隔离 |
| fullVM | QEMU | 任意客户操作系统 | 数秒 | 完整虚拟化 |

四种基底共享统一 API: 命令执行、文件传输、TTY 访问. 这种设计允许同一套 Agent 代码在不同隔离级别间无缝迁移——开发调试用 Function Call(零延迟),生产部署用 microVM(强隔离). 

**分层存储加速镜像加载**. Container 基底将基础镜像作为 3FS 支撑的只读 EROFS 层直接挂载到 overlay lowerdirs,元数据本地缓存,数据块按需拉取. microVM 使用 overlaybd 磁盘格式: 只读基础层跨实例共享,写操作走本地 copy-on-write 层,快照可链式串联实现毫秒级恢复. 

**大规模并发的密度优化**. DSec 解决两个资源瓶颈: (1) 消除虚拟化环境中的重复 page cache 占用,通过内存回收实现安全超量分配; (2) 缓解容器运行时的自旋锁竞争,降低每个沙箱的 CPU 开销,显著提升单机装箱密度. 

**轨迹日志与抢占安全恢复**. 每个沙箱维护全局有序的轨迹日志,记录每次命令调用及其结果. 训练任务被抢占时沙箱资源保留,恢复时 DSec 回放已完成命令的缓存结果,加速恢复并防止非幂等操作重复执行. 任何历史会话都可以从轨迹日志忠实复现. 

DSec 的工程启示在于: **沙箱不是孤立的安全边界,而是训练-评估闭环中的有状态计算节点**. 传统沙箱设计关注"如何隔离",DSec  additionally 关注"如何恢复"和"如何复现"——这对强化学习中的 Agent 交互环境尤为关键,因为非确定性的环境重置可能导致策略梯度估计偏差. 

## 技术前瞻

**确定性重放沙箱**. 当前沙箱验证代码正确性需重新执行. 未来方向是基于 eBPF 的执行轨迹记录与确定性重放,允许在不暴露完整执行环境的情况下验证输出. 数学上,这要求证明: 

$$
 \text{Replay}(\text{Trace}) \equiv \text{OriginalExecution} \tag{8}
$$
式 (6) 中,$\text{Replay}$ 为重放函数,$\text{Trace}$ 为执行期间捕获的系统调用与内存访问序列. 如式 (6) 所示,若轨迹捕获完整且重放环境确定性等价,则无需暴露原始沙箱即可验证结果. 该方向与区块链可验证计算(如 RISC Zero zkVM)交叉,但当前 zkVM 的 1000x+ 性能开销使其暂不适用于通用代码执行. 

**AI 原生沙箱**. LLM 可生成代码的同时生成沙箱策略(seccomp 白名单、网络规则、资源配额). 形式化验证方向是证明策略 $\mathcal{P}$ 对程序 $P$ 的充分性: 

$$
 \forall \sigma, \mathcal{P}(P, \sigma) \Rightarrow \text{Safe}(\sigma) \tag{9}
$$
式 (7) 中,$\sigma$ 为程序执行状态,$\mathcal{P}$ 为沙箱策略函数,$\text{Safe}$ 为安全状态谓词. 式 (7) 要求策略对所有可达状态均保证安全性. 该问题在一般情况下不可判定(可归约至停机问题),但对受限语言子集(如不含动态代码生成的 Python)可能存在可判定边界. 

**WASI 浏览器沙箱成熟化**. WebAssembly System Interface(WASI)标准化沙箱化系统调用接口. 随着 WASI Preview 2 的组件模型(Component Model)成熟,浏览器内可运行具备完整 I/O 能力的沙箱化应用,无需服务器参与. 性能目标: 接近原生速度的 90%,启动时延 < 50ms. 

**异构沙箱调度**. 未来 Agent 平台可能混合使用浏览器沙箱(低延迟、零成本)、本地 Docker(中等隔离、完全可控)、云端 MicroVM(高隔离、弹性伸缩)三级架构. 调度器根据任务敏感度、延迟要求、成本约束自动选择层级,形成资源-隔离度的帕累托前沿.
