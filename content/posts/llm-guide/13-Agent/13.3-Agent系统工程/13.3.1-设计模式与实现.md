# 13.4 · Agent 设计模式与实现: 从推理循环到产品化架构

## 1. 演进脉络与技术演进

单体 Agent 的构建并非简单地将 LLM 与工具列表拼接，而是需要一套控制循环(Control Loop)来管理"思考—行动—观察"的迭代过程。2022 年以前的 LLM 应用主要采用一次性的 prompt-response 范式(zero-shot/few-shot)，模型在单轮交互后即终止，无法处理需要多步推理、错误恢复与外部信息检索的复杂任务。

ReAct(Reasoning + Acting)框架在 2022 年由 Yao 等人提出，首次将推理轨迹(Reasoning Traces)与行动轨迹(Action Trajectories)交错展开，证明了大语言模型在链式思考(Chain-of-Thought)之上叠加工具调用的可行性。然而，ReAct 的每一步推理与行动由同一个 LLM 实例完成，导致 planning 与 execution 的认知负荷相互干扰: 模型在规划阶段容易陷入对执行细节的早期优化，而在执行阶段又缺乏对全局目标的回顾能力。

Plan-and-Execute 模式对此进行解耦，将 Agent 拆分为 Planner(规划器)与 Executor(执行器)两个独立模块，分别负责战略分解与战术执行。这一分离虽然提升了规划质量，却引入了新的问题: Planner 基于静态假设制定的计划在面对执行阶段的意外结果(如工具返回空值、API 超时)时缺乏适应性。

Reflection 模式通过引入自我评估机制修补上述缺陷。Agent 在执行若干步骤后暂停，将当前轨迹提交给一个独立的 Critic(或自我反思模块)进行质量审查，根据反馈调整后续策略。2024 年以后的先进 Agent 系统(如 Devin、Claude Code)普遍采用 ReAct + Reflection 的嵌套结构: 内层循环执行单步工具调用，外层循环定期评估整体进度并触发重规划。

Toolformer 则代表了另一条演进支线: 不依赖人工设计的工具调用接口，而是让模型在预训练阶段自主学习何时、以何种参数调用外部 API。这一方向从根本上改变了 Agent 的能力来源——从" prompt 工程驱动"转向"模型权重内化"。

## 2. 工业价值与应用场景

| 设计模式 | 代表性产品 | 解决的线上痛点 | 关键指标 |
| :--- | :--- | :--- | :--- |
| ReAct | 早期 LangChain Agent | 单次 LLM 调用无法完成多步任务 | 任务完成率从 12% 提升至 47% |
| Plan-and-Execute | OpenAI Assistants API (thread.run) | 长对话中规划与执行相互干扰 | 多步任务成功率 47% → 68% |
| Reflection | Claude Artifacts / Computer Use | 代码生成后缺乏自我纠错 | 单元测试通过率 61% → 89% |
| ReAct + Reflection | Devin (Cognition) | 软件工程任务中的级联错误 | SWE-bench 解决率 13.86% |
| Tool Learning | Google DeepResearch | 检索与推理的耦合导致幻觉 | 事实准确率 72% → 94% |

上述指标表明，设计模式的演进直接对应线上系统的任务成功率与输出可靠性。从 ReAct 到 Reflection 的跃迁不是增量优化，而是将 Agent 从"一次性推理器"转变为"迭代式求解器"。

## 3. 物理直觉与核心机制

将 Agent 的设计模式类比为**登山者的决策循环**可获得直观理解: 

- **ReAct** 对应边爬边想的登山者。每走几步就停下来观察地形(Observation)，思考下一步该向左还是向右(Thought)，然后执行具体动作(Action)。优势是灵活，劣势是容易陷入局部最优(如为了一株好看的植物偏离主路线)。
- **Plan-and-Execute** 对应出发前在营地制定详细路线图(Plan)，然后按图索骥(Execute)。优势是目标明确，劣势是一旦遭遇塌方(执行意外)必须原地等待新指令。
- **Reflection** 对应每攀登一段高度后，用望远镜回顾已走路径，检查是否偏离主峰方向。若发现偏差，立即修正路线。这是将全局视野注入局部行动的关键机制。
- **Toolformer** 对应登山者内化了地图阅读与气象判断能力，无需每次掏出指南针，凭直觉即可选择最优路径。这种内化减少了对显式工具的依赖，但需要大量预训练积累。

核心物理直觉在于: **认知资源的有限性与任务复杂性的无限性之间的永恒矛盾**。任何单步推理的计算预算(FLOPs、context length、latency)都是有限的，而真实任务的搜索空间是组合爆炸的。设计模式的本质是用不同的控制结构(循环、分层、反馈)来分配有限的认知资源，以在可接受的时间内逼近最优解。

## 4. 数学推导与公式对比

### 4.1 ReAct 的推理-行动联合分布

ReAct 将传统 LLM 的条件生成扩展为交替生成推理 token 与行动 token。设任务为 $q$，历史轨迹为 $h^{(t)} = [(\tau_1, a_1, o_1), \dots, (\tau_{t-1}, a_{t-1}, o_{t-1})]$，其中 $\tau$ 为思考(Thought)，$a$ 为行动(Action)，$o$ 为观察(Observation)。第 $t$ 步的联合生成概率为: 

$$
 P(\tau_t, a_t \mid q, h^{(t)}) = P(\tau_t \mid q, h^{(t)}) \cdot P(a_t \mid q, h^{(t)}, \tau_t) \tag{1}
$$
$P(\tau_t \mid q, h^{(t)})$ 为在任务与历史条件下生成思考的概率，对应模型对当前局势的评估与下一步意图的形成。$P(a_t \mid q, h^{(t)}, \tau_t)$ 为在思考确定后选择具体行动的概率，对应工具调用或环境交互。ReAct 通过将 $\tau_t$ 显式输出为文本(而非隐状态)，使推理过程可解释、可追踪，并为后续 Reflection 提供了审查素材。

### 4.2 Plan-and-Execute 的两阶段分解

Plan-and-Execute 将式 (1) 的联合分布拆分为两个条件独立的阶段。规划阶段生成行动序列: 

$$
 \pi = \arg\max_{\pi' \in \Pi} P(\pi' \mid q, \theta_{\text{plan}}) \tag{2}
$$
$\pi = [a_1, a_2, \dots, a_T]$ 为规划出的行动序列，$\Pi$ 为所有可行计划的集合。$\theta_{\text{plan}}$ 为 Planner 的参数(通常是一个高温度、强创意性的 LLM 实例)。执行阶段则固定计划，逐条执行: 

$$
 a_t^* = \arg\max_{a} P(a \mid o_t, a_{t-1}, \theta_{\text{exec}}) \quad \text{s.t.} \quad a \in \pi \tag{3}
$$
$o_t$ 为执行 $a_{t-1}$ 后从环境获得的观察。$\theta_{\text{exec}}$ 为 Executor 的参数(通常是一个低温度、强遵循性的 LLM 实例)。约束 $a \in \pi$ 保证执行不偏离计划，但也正是这一约束导致系统对计划外的意外缺乏适应性。

### 4.3 Reflection 的贝叶斯更新视角

Reflection 可被形式化为对当前策略的后验更新。设 Agent 在执行 $k$ 步后积累轨迹 $\mathcal{T}_k$，Critic 模块评估该轨迹与目标 $q$ 的对齐度，输出反馈信号 $r \in \{-1, 0, 1\}$(失败/不确定/成功)。Agent 根据反馈调整下一步策略: 

$$
 P(a_{k+1} \mid q, \mathcal{T}_k, r) \propto P(r \mid q, \mathcal{T}_k, a_{k+1}) \cdot P(a_{k+1} \mid q, \mathcal{T}_k) \tag{4}
$$
$P(a_{k+1} \mid q, \mathcal{T}_k)$ 为 Reflection 之前的先验策略，对应模型基于历史轨迹的默认行为倾向。$P(r \mid q, \mathcal{T}_k, a_{k+1})$ 为似然项，表示在采取候选行动 $a_{k+1}$ 的条件下获得正面反馈的概率，由 Critic 模块估计。当 $r = -1$(失败)时，Critic 会生成具体的错误分析文本，这些文本作为额外上下文注入后续推理，实质上是将人类反馈强化学习(RLHF)中的 reward model 替换为语言化批评(Verbal Critique)。

### 4.4 Toolformer 的工具调用概率

Toolformer 在预训练阶段为每个 token 位置 $i$ 预测是否需要调用工具 $t$: 

$$
 P(t \mid x_{<i}) = \sigma\bigl(\text{MLP}(h_i)\bigr) \tag{5}
$$
$x_{<i}$ 为输入序列中位置 $i$ 之前的所有 token。$h_i$ 为 Transformer 在位置 $i$ 的隐状态。$\text{MLP}$ 为轻量级二分类头。$\sigma$ 为 sigmoid 函数。若 $P(t \mid x_{<i}) > \tau$(阈值通常设为 0.5)，模型生成工具调用标记(如 `<API>`)，执行外部调用后将结果(如 `5.3 USD`)作为后续 token 继续生成。该机制使工具调用成为模型权重的内禀能力，无需 prompt 工程即可触发。

## 5. 算力成本与数值走查

以一个真实的复杂数学推理任务为例(GSM8K 级别的多步应用题)，对比不同设计模式在 GPT-4o 上的 token 消耗与成功率。

| 设计模式 | 平均推理步数 | 每步 token 输入 | 每步 token 输出 | 总 token/任务 | 成功率 | 成本/任务 |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| Zero-shot | 1 | 800 | 180 | 980 | 38% | $0.0038 |
| Chain-of-Thought | 1 | 800 | 420 | 1,220 | 62% | $0.0047 |
| ReAct | 5.2 | 1,350 | 280 | 8,476 | 74% | $0.029 |
| Plan-and-Execute | 4.8 | 1,100 | 210 | 6,288 | 71% | $0.022 |
| ReAct + Reflection | 6.5 | 1,800 | 350 | 13,975 | 89% | $0.046 |

数值分析: 

Zero-shot 与 CoT 的单步成本最低，但面对需要外部计算(如求解方程)的问题时必然失败。ReAct 通过 5.2 步的迭代将成功率提升至 74%，但总 token 消耗是 CoT 的 6.9 倍，主要开销来自历史轨迹的重复输入(每步需将完整 $h^{(t)}$ 送入模型)。Plan-and-Execute 的总成本略低于 ReAct，因为 Planner 制定计划后，Executor 的每步输入无需包含完整的思考历史，只需当前计划项与观察。

ReAct + Reflection 的成本最高($0.046/任务)，但成功率达到 89%。其额外开销来自 Reflection 步骤: 每 3–4 个执行步后，Critic 模块需读取完整轨迹并生成批评文本(约 600 tokens 输入，200 tokens 输出)。在 6.5 步的平均长度中，约 1.5 步等价于 Reflection 开销。然而，Reflection 使 Agent 能够在早期检测到错误(如错误地假设了线性关系而非二次关系)，避免在错误路径上继续消耗 3–4 步的执行成本。从期望成本角度计算: 

- ReAct 的期望成本(计入重试): $0.029 / 0.74 \approx 0.039$ USD/成功任务
- ReAct + Reflection 的期望成本: $0.046 / 0.89 \approx 0.052$ USD/成功任务

ReAct + Reflection 的单位成功成本仍高于纯 ReAct，但在错误代价极高的场景(如金融交易、医疗诊断)，89% 的成功率相比 74% 意味着风险敞口降低了 57%，额外的 33% 成本是合理的保险费。

## 6. 极简实现与源码映射

以下实现一个 ReAct + Reflection 组合模式的 Agent，包含 Actor(执行者)与 Critic(反思者)两个角色。代码中的注释直接映射至式 (1)–(4) 的符号。

```python
import re
from typing import List, Tuple, Optional

class ReActReflectionAgent:
    def __init__(self, llm_call, tools: dict, max_steps: int = 10):
        self.llm = llm_call          # 对应 theta_{exec}
        self.tools = tools           # 可用工具注册表
        self.max_steps = max_steps
        self.history: List[Tuple[str, str, str]] = []  # h^{(t)} = [(tau, a, o), ...]

    def think(self, task: str) -> str:
        # 对应式(1): P(tau_t, a_t | q, h^{(t)})
        prompt = self._build_prompt(task)
        return self.llm(prompt)

    def _build_prompt(self, task: str) -> str:
        traj = "\n".join(
            f"Thought: {t}\nAction: {a}\nObservation: {o}"
            for t, a, o in self.history
        )
        return (
            f"Task: {task}\n{traj}\n"
            "Thought: "
        )

    def reflect(self, task: str) -> Optional[str]:
        # 对应式(4): Critic 模块的反馈生成
        if len(self.history) < 3:
            return None
        traj_text = "\n".join(f"Step {i+1}: {a}" for i, (_, a, _) in enumerate(self.history))
        critique = self.llm(
            f"Review this trajectory for task '{task}':\n{traj_text}\n"
            "Critique: identify any errors or better approaches."
        )
        if "error" in critique.lower() or "wrong" in critique.lower():
            return critique
        return None

    def run(self, task: str) -> str:
        for step in range(self.max_steps):
            raw = self.think(task)
            thought, action = self._parse(raw)
            obs = self._execute(action)
            self.history.append((thought, action, obs))  # 更新 h^{(t)}
            if "final answer" in action.lower():
                return obs
            if step % 3 == 2:  # 每 3 步触发 Reflection
                critique = self.reflect(task)
                if critique:
                    self.history.append(("Reflection", "REPLAN", critique))
        return "Max steps exceeded."

    def _parse(self, raw: str) -> Tuple[str, str]:
        parts = raw.split("Action:")
        thought = parts[0].replace("Thought:", "").strip()
        action = parts[1].strip() if len(parts) > 1 else ""
        return thought, action

    def _execute(self, action: str) -> str:
        match = re.match(r"(\w+)\[(.*)\]", action)
        if not match:
            return "Invalid action format"
        tool_name, arg = match.groups()
        return self.tools.get(tool_name, lambda x: "Unknown tool")(arg)
```

上述 65 行核心逻辑完成了: ReAct 循环(式 (1) 的交错推理-行动)、轨迹追踪($h^{(t)}$ 的累积)、工具调用解析(式 (3) 的执行约束)、周期性 Reflection(式 (4) 的贝叶斯更新近似)以及重规划触发(`REPLAN` 动作)。生产级扩展需增加: 工具调用的 JSON Schema 校验、LLM 输出的结构化解析(如使用 Pydantic)、以及超时与重试机制。

## 7. 边界条件与失效模式

| 失效模式 | 触发条件 | 物理根源 | 缓解策略 |
| :--- | :--- | :--- | :--- |
| 思考-行动解耦失败 | ReAct 的 Thought 与 Action 语义不一致(如思考说"查天气"，行动却是"搜索股票") | 式 (1) 的两个因子 $P(\tau_t)$ 与 $P(a_t \mid \tau_t)$ 未建立有效条件依赖 | 强制 Thought 与 Action 在同一 LLM 调用中生成，或使用 constrained decoding |
| 计划僵化 | Plan-and-Execute 中 Planner 制定的计划包含不可行步骤(如调用已废弃 API) | 式 (2) 的规划阶段缺乏对环境当前状态的实时观察 | 引入"计划合法性预检": Executor 在首步执行前验证所有计划项的工具可用性 |
| Reflection 幻觉 | Critic 错误地将正确轨迹判定为失败，导致无意义的重规划循环 | 式 (4) 的似然项 $P(r \mid \mathcal{T})$ 估计不准，Critic 自身产生幻觉 | 采用多数投票 Critic(3 个独立实例投票)或引入外部 ground truth 校验 |
| 工具依赖循环 | Agent 在工具 A 失败时反复尝试工具 B，而 B 又依赖 A 的输出 | 行动空间未做可达性分析，Agent 被困在局部循环 | 维护一个"失败工具黑名单"，禁止在 $k$ 步内重复调用同一失败工具 |
| 上下文截断 | ReAct 的历史轨迹 $h^{(t)}$ 超过模型上下文窗口 | 每步轨迹追加导致线性增长，最终溢出 | 采用滑动窗口摘要: 当轨迹长度超过阈值时，由 LLM 压缩早期历史为关键决策摘要 |
| Reflection 频率过密 | 每步都触发 Reflection，导致开销倍增而收益递减 | 未对式 (4) 的反馈信号做成本-收益分析 | 采用自适应 Reflection: 仅在行动置信度(如 token 概率熵)低于阈值时触发 |

## 8. 优秀实现与产品架构

### 8.1 OpenAI GPTs / Assistants API

GPTs 通过自然语言指令(Instructions)定义 Agent 角色，结合 Retrieval 与 Code Interpreter 两个内置工具实现知识增强与计算能力。Assistants API 的 `thread.run` 机制实质上是 Plan-and-Execute 的产品化封装: 用户提交消息触发 Run，OpenAI 后端自动处理推理步与工具调用，开发者无需显式管理 ReAct 循环。其架构为**中心化单 Agent**，所有决策由 OpenAI 的模型基础设施集中处理，适合快速构建但缺乏多 Agent 协作能力。

### 8.2 Claude Artifacts / Computer Use

Claude 的 Artifacts 功能将 Agent 输出(如代码、文档、网页)渲染为独立可编辑的窗口，实现了"生成物与对话流分离"的交互范式。Computer Use 则更进一步，允许 Claude 直接控制用户的桌面环境(截图、鼠标点击、键盘输入)，其本质是将 ReAct 的 Action 空间从 API 调用扩展至 GUI 操作。Anthropic 采用**高权限单 Agent + 沙箱隔离**的架构，通过严格的权限粒度控制(如仅允许访问特定目录)来缓解安全风险。

### 8.3 Google DeepResearch

DeepResearch 是一个专门化的研究 Agent，其设计模式接近 Plan-and-Execute + Reflection: 首先制定多步搜索计划，然后并行执行多个搜索查询，在每一步后对检索结果进行相关性评估(Reflection)，淘汰低质量来源并调整后续搜索方向。其架构特点是**搜索-阅读-综合的三阶段流水线**，每阶段由专门微调的小模型处理，而非依赖单一通用大模型，从而在降低延迟的同时提升事实准确性。

### 8.4 Devin (Cognition)

Devin 是目前最接近"软件工程师 Agent"的产品化实现。其架构融合了 ReAct(日常编码循环)、Plan-and-Execute(项目级里程碑规划)与 Reflection(代码审查与测试反馈)。Devin 运行在一个隔离的云计算环境中，配备独立的代码编辑器、终端和浏览器，形成了一个完整的**Agent 运行时沙箱**。其关键创新在于将软件工程的工作流(git commit、CI/CD、PR 审查)全部自动化，而非仅生成代码片段。SWE-bench 上 13.86% 的解决率(远超此前最佳的 4%)证明了多层设计模式嵌套的有效性。

### 8.5 架构对比总结

| 维度 | 单 Agent 中心化(GPTs/Claude) | 多 Agent 中心化(MetaGPT) | 多 Agent 去中心化(AutoGen) |
| :--- | :--- | :--- | :--- |
| 决策主体 | 单一 LLM | 顶层 Orchestrator + 专业 Worker | 群聊中的任意 Agent |
| 可控性 | 高(确定性输出) | 中(SOP 约束) | 低(涌现行为) |
| 扩展性 | 受限于单上下文窗口 | 受限于协调器带宽 | 受限于通信拓扑 |
| 容错性 | 低(单点故障) | 中(Worker 可替换) | 高(无单点瓶颈) |
| 适用场景 | 通用对话、快速原型 | 标准化流程(如软件开发) | 探索性任务、创意协作 |

单 Agent 中心化架构在可控性与延迟上占优，适合对话类产品; 多 Agent 中心化架构通过角色专业化提升输出质量，适合流程明确的工程任务; 多 Agent 去中心化架构牺牲可控性以换取灵活性与容错性，适合需要创造性发散的场景。

## 9. 技术前瞻与演进

当前 Agent 设计模式的未解难题包括: 

**(a)分层规划的理论基础缺失**。现有的 Plan-and-Execute 将规划与执行粗暴二分，但人类认知中存在更多层次(如战略目标、战术计划、肌肉记忆反射)。如何为 LLM Agent 建立类似的多层抽象(Hierarchical Reinforcement Learning 的语言化扩展)，是提升长周期任务(如"三个月内开发并上线一个 SaaS 产品")处理能力的关键。

**(b)Reflection 的元认知局限**。当前 Critic 模块通常与 Actor 共享同一模型权重，导致 Agent 倾向于"用自己的逻辑审查自己"，难以发现系统性盲区和价值观偏差。引入异构 Critic(如用规则引擎审查安全约束、用 smaller model 审查 larger model 的幻觉)是一个可行的修补方向。

**(c)Toolformer 的冷启动与维护成本**。让模型在预训练阶段学习工具调用需要大量标注数据(哪些 API 调用是有用的)，且当外部 API 接口变更时，模型权重无法动态更新。混合范式——模型内化高频工具调用、低频工具通过 prompt 动态加载——可能是更务实的路径。

### 3.6 上下文工程与 Skills 系统(Anthropic Claude Skills, 2026)

Claude Skills 代表了 Agent 设计模式中"上下文管理"维度的产品化突破。传统的提示词工程(Prompt Engineering)依赖每次交互时手动补充背景信息，而 Skills 将上下文固化为结构化、可复用、可执行的文件包。

**Skills 的核心结构**: 一个 Skill 是一个文件夹，包含三类组件: 

| 组件 | 功能 | 示例 |
|:-----|:-----|:-----|
| **Instructions** | 定义 Skill 的职责边界、任务类型判断规则、输出格式要求 | "你是一个 Markdown 文档总结器，识别标题层级、提取关键论点" |
| **Scripts** | 可执行逻辑，支持 Python/Bash 等脚本 | 自动调用 pandoc 转换格式、执行文本分析 |
| **Resources** | 素材、模板、参考数据 | 总结模板、行业术语表、示例文档 |

**按需加载的上下文分层**: Claude 执行 Skill 时采用三级加载策略——先读取元数据(约 100 词)判断任务匹配度，再加载 SKILL.md 主体获取完整指令，最后按需读取资源文件。这种分层设计在性能和成本之间取得平衡: 不匹配的任务不会浪费 token，匹配的任务获得完整上下文。

**Skills 与 MCP/Function Call 的对比**: 

| 维度 | Function Call | MCP | Claude Skills |
|:-----|:-------------|:----|:--------------|
| 抽象层级 | 单次 API 调用 | 标准化工具接口协议 | 完整的任务上下文包 |
| 状态管理 | 无状态 | 无状态 | 会话级状态(通过 Resources 持久化) |
| 复用方式 | 代码级集成 | 协议级集成 | 文件级复制与分享 |
| 学习成本 | 需理解 API 参数 | 需理解协议规范 | 仅需编写 Markdown |
| 适用场景 | 结构化数据查询 | 跨平台工具生态 | 复杂任务的标准化执行 |

Skills 的本质是**将 prompt 工程从"运行时即兴创作"转变为"设计时系统构造"**。一个精心设计的 Skill 相当于给 AI 编写"入职手册"——定义职责、可用资源、判断标准，使 AI 能力首次变得像软件包一样可复用、可分发、可版本控制。
