# 13.3 Agent 系统工程

## 1. 演进脉络与技术坐标

单体 Agent 的构建并非简单地将 LLM 与工具列表拼接，而是需要一套控制循环来管理"思考—行动—观察"的迭代过程。2022 年以前的 LLM 应用主要采用一次性的 prompt-response 范式; ReAct 框架首次将推理轨迹与行动轨迹交错展开; Plan-and-Execute 模式将 Agent 拆分为 Planner 与 Executor; Reflection 模式引入自我评估机制。2024 年以后，先进 Agent 系统普遍采用 ReAct + Reflection 的嵌套结构。

当任务复杂度超出单个 Agent 的能力边界时，多 Agent 系统(MAS)成为必然选择。从 2023 年 AutoGen 的群聊模式到 2024 年 LangGraph 的状态图编排，MAS 的演进主线是从"对话编排"到"状态图编排"再到"工作流即代码"。

## 3. 工业价值与落地现状

以 SWE-Bench 为例，2023 年 GPT-4 的直接解决率不足 2%，而通过 Agent 框架的增强，OpenHands 在 2025 年初达到了 53.3% 的解决率。MetaGPT 的 SOP 角色分工将端到端代码生成通过率从 23% 提升至 85%。MAS 的核心工业价值不在于单个 Agent 的智商提升，而在于通过任务分解与并行化降低系统的整体 latency 与错误传播范围。

## 4. 技术前瞻

当前系统工程的未解难题包括: 通信协议的语义压缩(Agent 间使用自然语言通信，信息密度远低于结构化协议)、动态拓扑重组(现有框架的通信拓扑多为静态配置，无法自适应调整)、以及经济调度与算力预算(MAS 的总成本并非单调递减，需要在线的成本-收益模型动态决定最优 Agent 数量与拓扑结构)。
