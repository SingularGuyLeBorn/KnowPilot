# 13.3.3 · 多 Agent 系统: 从单体智能到群体协作

## 1. 演进脉络与技术演进

单体 LLM Agent 在 2023 年通过 ReAct 和 Toolformer 范式证明了其具备基本的推理与工具调用能力，但很快触碰到三重重约束: 

**(a)上下文窗口瓶颈**。当任务涉及数百个工具调用、多轮文件操作与跨域知识检索时，单个 Agent 的上下文长度(即使 128K 或 200K)被系统 prompt、历史轨迹和中间观察结果迅速耗尽，导致关键信息被挤出 attention 窗口，推理质量断崖式下跌。

**(b)能力耦合与职责泛化**。要求一个 Agent 同时精通代码生成、金融建模、法律合规审查和 UI 设计，本质上是将互异的专业知识压入同一组权重参数。LLM 在训练时对这些领域的数据分布存在显著差异，强行统一会导致每个领域的输出质量均被稀释。

**(c)单点故障与串行延迟**。单体 Agent 一旦在某个推理步陷入循环(例如反复调用同一失败工具)，整个任务链即告瘫痪。且所有子任务必须串行执行，无法利用多核 CPU 或分布式 GPU 的并行算力。

多 Agent 系统(Multi-Agent System, MAS)的设计初衷正是为了瓦解上述约束。其思想渊源可追溯至 1980 年代的分布式人工智能(Distributed AI)和 1990 年代的软件 Agent 理论(如 Wooldridge 的 BDI 模型)，但直到 LLM 具备足够强的自然语言理解与生成能力后，Agent 之间的通信才无需依赖严格的符号化协议(如 KQML/KIF)，转而使用自然语言作为通用接口，从而大幅降低了系统构建门槛。

从 2023 年 AutoGen 的发布到 2024 年 LangGraph 的成熟，MAS 的演进主线呈现出从"对话编排"到"状态图编排"再到"工作流即代码"的跃迁。早期的 AutoGen 依赖群聊(Group Chat)和角色扮演(Role Play)实现协作，灵活性高但可控性弱; LangGraph 引入显式的状态机与条件边，使 Agent 之间的流转可被精确追踪与回滚; MetaGPT 则将软件工程中的 SOP(Standard Operating Procedure)编码为 Agent 协作规范，将自然语言任务自动拆解为 UML 图、接口定义和测试用例。

## 2. 工业价值与应用场景

MAS 已在多个高复杂度场景落地，其带来的核心指标改进如下表所示: 

| 场景                    | 单体 Agent 瓶颈                                       | MAS 解决方案                                                            | 关键指标改进                                                           |
| :---------------------- | :---------------------------------------------------- | :---------------------------------------------------------------------- | :--------------------------------------------------------------------- |
| 自动化软件工程          | 单 Agent 无法同时维护架构设计、编码、测试、文档一致性 | MetaGPT 的 SOP 角色分工: 产品经理、架构师、工程师、测试员各司其职       | 端到端代码生成通过率从 23% 提升至 85%(HumanEval 扩展集)                |
| 深度研究(Deep Research) | 单次搜索-阅读-综合的 token 消耗超出上下文限制         | 检索 Agent、摘要 Agent、综合 Agent 分阶段流水线作业                     | 单任务处理文档量从 20 篇扩展至 200+ 篇，幻觉率下降 40%                 |
| 金融风控报告生成        | 多源异构数据(财报、舆情、宏观指标)耦合度过高          | 数据清洗 Agent、因子计算 Agent、合规审查 Agent、报告撰写 Agent 层级协作 | 报告生成延迟从 45 min 降至 8 min(4 路并行)，合规漏检率从 12% 降至 1.5% |
| 多模态内容生产          | 文案、配图、配音、剪辑由同一 Agent 输出，风格割裂     | 文案 Agent、文生图 Agent、TTS Agent、视频编排 Agent 对等协作            | 端到端生产周期从 6 h 降至 35 min                                       |

上述指标表明，MAS 的核心工业价值不在于单个 Agent 的智商提升，而在于通过**任务分解**与**并行化**降低系统的整体 latency 与错误传播范围。

## 3. 物理直觉与核心机制

将多 Agent 系统类比为一个**交响乐团**可获得直观理解: 

- **指挥(Orchestrator)** 对应层级式架构中的顶层 Agent，负责将总谱(用户目标)拆分为各声部的分谱(子任务)，并控制演奏节奏(同步点)。
- **声部首席(Section Leaders)** 对应专业 Agent，只负责自己擅长的乐器(领域)，无需理解其他声部的全部细节，只需在指定小节进入(接口契约)。
- **乐谱架(Shared Blackboard)** 对应黑板系统，任何乐手都可以瞥见当前全局演奏进度(共享状态)，从而决定自己的下一个动作。
- **即兴对话(Jam Session)** 对应对等式架构，乐手之间实时互相聆听并调整，无需指挥干预，适合探索性任务但缺乏确定性保证。

关键物理直觉在于: **通信带宽与认知局部性之间的权衡**。当 Agent 数量 $N$ 增加时，全连接通信的复杂度为 $O(N^2)$，而信息熵(系统状态的不确定性)若不加以约束将指数级膨胀。因此，任何 MAS 架构本质上都是在用某种拓扑结构(层级、流水线、星型)对通信图做稀疏化，以控制状态空间的爆炸。

## 4. 数学推导与公式对比

### 4.1 任务分解与专业化收益

设用户提交的总任务为 $T$，其复杂度(以所需推理步数或 token 消耗度量)为 $C(T)$。若将 $T$ 分解为 $n$ 个互斥子任务 $\{t_1, t_2, \dots, t_n\}$，每个子任务由专门化 Agent $A_i$ 处理，则系统总成本可建模为: 

$$
C_{\text{MAS}} = \sum_{i=1}^{n} C(t_i) + C_{\text{comm}}(n) + C_{\text{coord}}(n) \tag{1}
$$

其中 $C(t_i)$ 为子任务执行成本，$C_{\text{comm}}(n)$ 为 Agent 间通信开销，$C_{\text{coord}}(n)$ 为协调器决策开销。由于专业化 Agent 在特定领域的数据分布上具有更高的后验概率精度，单个 $C(t_i)$ 显著低于通用 Agent 处理对应片段的成本。若分解合理，满足 $\sum C(t_i) \ll C(T)$，则即使引入额外的通信与协调开销，仍有 $C_{\text{MAS}} < C(T)$。

$\sum C(t_i)$ 为各子任务在专门化 Agent 下的推理成本之和，典型场景下可比单体 Agent 降低 40%–70%。$C_{\text{comm}}(n)$ 为 Agent 间消息传递的 token 开销，与拓扑结构密切相关: 全连接为 $O(n^2)$，层级式为 $O(n)$，流水线式为 $O(n)$。$C_{\text{coord}}(n)$ 为协调器进行任务分配与冲突仲裁的计算成本，通常可忽略(单次 LLM 调用)或摊薄到多个任务周期中。

### 4.2 通信机制的形式化描述

Agent $A_i$ 在时刻 $k$ 的状态可表示为三元组 $s_i^{(k)} = (b_i^{(k)}, m_i^{(k)}, \theta_i)$，其中 $b_i^{(k)}$ 为内部信念状态(Belief)，$m_i^{(k)}$ 为待处理消息队列，$\theta_i$ 为角色参数(系统 prompt 与工具集)。在消息传递模型中，Agent $j$ 向 Agent $i$ 发送消息的动作定义为: 

$$
m_{j \to i}^{(k)} = \text{LLM}_{\theta_j}\bigl(s_j^{(k)}, \text{task}(i)\bigr) \tag{2}
$$

$\text{LLM}_{\theta_j}$ 表示由角色参数 $\theta_j$ 参数化的大语言模型。$s_j^{(k)}$ 为发送方在时刻 $k$ 的内部状态，包含历史观察与推理轨迹。$\text{task}(i)$ 为发送方对 $A_i$ 所需执行任务的描述，可能包含输入数据、期望输出格式与约束条件。此式表明，Agent 之间的通信内容并非预先编码的固定格式，而是由发送方根据其当前状态和接收方角色动态生成的自然语言指令，这是 LLM-based MAS 与传统分布式系统的根本差异。

在黑板系统(Blackboard)模型中，共享状态 $B^{(k)}$ 的更新规则为: 

$$
B^{(k+1)} = B^{(k)} \cup \bigcup_{i \in \mathcal{A}_{\text{active}}} \Delta_i^{(k)} \tag{3}
$$

$B^{(k)}$ 为时刻 $k$ 的全局黑板状态，通常以键值对、结构化 JSON 或向量数据库形式存储。$\mathcal{A}_{\text{active}}$ 为当前活跃 Agent 集合。$\Delta_i^{(k)}$ 为 Agent $A_i$ 在本轮迭代中对黑板状态的增量贡献，可能包含新提取的事实、更新后的置信度分数或部分解。黑板系统的核心优势在于解耦了 Agent 之间的直接依赖: $A_i$ 无需知道 $A_j$ 的存在，只需读取黑板上的相关内容即可推进自身任务。

### 4.3 协调与共识机制

任务分配问题可建模为二分图匹配。设任务集合为 $\mathcal{T} = \{t_1, \dots, t_n\}$，Agent 能力矩阵为 $M \in \mathbb{R}^{n \times m}$，其中 $M_{ij}$ 表示 Agent $A_j$ 对任务 $t_i$ 的胜任度评分(由 LLM 自我评估或历史准确率统计得出)。最优分配 $x^*$ 为: 

$$
x^* = \arg\max_{x \in \{0,1\}^{n \times m}} \sum_{i=1}^{n} \sum_{j=1}^{m} M_{ij} x_{ij} \quad \text{s.t.} \quad \sum_{j} x_{ij} = 1, \; \sum_{i} x_{ij} \leq 1 \tag{4}
$$

$x_{ij}$ 为二元决策变量，$x_{ij}=1$ 表示任务 $t_i$ 分配给 Agent $A_j$。约束 $\sum_j x_{ij}=1$ 保证每个任务恰好被一个 Agent 执行。约束 $\sum_i x_{ij} \leq 1$ 保证每个 Agent 在单轮中最多承担一个任务(避免过载)。该整数线性规划(ILP)在 $n,m \leq 100$ 时可由匈牙利算法在 $O(n^3)$ 内精确求解; 在更大规模下，采用贪心近似或基于 LLM 的启发式分配。

对于需要共识的场景(如多个评审 Agent 对同一方案投票)，采用加权多数投票机制。设 $N$ 个 Agent 对候选方案 $p$ 投出置信度加权票 $v_i \in \{-1, 1\}$，权重 $w_i$ 与其历史准确率正相关，则共识得分为: 

$$
S(p) = \sum_{i=1}^{N} w_i v_i \tag{5}
$$

$v_i = 1$ 表示赞成，$v_i = -1$ 表示反对。$w_i = \alpha \cdot \text{Acc}_i + (1-\alpha) \cdot \text{Recency}_i$，其中 $\text{Acc}_i$ 为 Agent $i$ 在该类任务上的历史准确率，$\text{Recency}_i$ 为近期表现的指数衰减权重。$S(p) > 0$ 时方案通过，$S(p) < 0$ 时驳回，$S(p) = 0$ 时触发仲裁 Agent 进行深度分析。该机制将传统分布式共识算法(如 Raft、PBFT)中的确定性投票替换为基于语言模型置信度的柔性投票，牺牲绝对一致性以换取对模糊问题的适应性。

## 5. 算力成本与数值走查

以一个真实的软件工程自动化场景为例，对比单体 Agent 与 MAS 的 token 消耗与 wall-clock 时间。任务: 从需求文档生成一个 Python Web API(含数据库模型、路由、单元测试、API 文档)。

| 指标                      | 单体 Agent         | 多 Agent 系统(MetaGPT 风格 4 角色) |
| :------------------------ | :----------------- | :--------------------------------- |
| 单次任务总 token 输入     | 185K               | 320K(含角色间通信)                 |
| 单次任务总 token 输出     | 42K                | 58K                                |
| 峰值上下文长度            | 128K(触顶截断风险) | 48K(单 Agent 最大)                 |
| Wall-clock 时间(单线程)   | 8 min 30 s         | 12 min(含通信开销)                 |
| Wall-clock 时间(4 路并行) | 不适用             | 4 min 20 s                         |
| 代码一次性通过率          | 31%                | 79%                                |
| 平均重试轮次              | 3.2                | 0.8                                |

数值分析: 

单体 Agent 的总 token 消耗(227K)看似低于 MAS(378K)，但其 128K 的上下文窗口在任务后半段触发了关键历史信息的截断(如最初的需求约束)，导致输出偏离原始目标，需 3.2 轮重试才能收敛。MAS 虽然总 token 更高，但每个 Agent 的上下文始终维持在 48K 以内，信息保真度高，且 4 路并行将 wall-clock 时间压缩至单体的 51%。若按 GPT-4o 的定价(输入 $2.50/1M tokens，输出 $10.00/1M tokens)计算: 

- 单体成本: $(185 \times 2.5 + 42 \times 10) / 1000 = 0.88$ USD/次(未计入重试)
- 实际单体成本(含 3.2 轮重试): $0.88 \times 3.2 \approx 2.82$ USD
- MAS 成本(含 0.8 轮重试): $(320 \times 2.5 + 58 \times 10) / 1000 \times 1.8 \approx 2.49$ USD

MAS 在单位任务上不仅更快，综合成本(计入重试)反而更低。当任务规模扩大至 10 个并发子任务时，MAS 的并行优势使吞吐量提升接近线性($8\times$–$9\times$)，而单体 Agent 的串行瓶颈导致 throughput 仅提升 $1.2\times$。

## 6. 极简实现与源码映射

以下实现一个层级式多 Agent 协调器，包含一个 Orchestrator 和三个专业 Worker(Coder、Reviewer、Tester)，使用 Python 3.11+ 语法。代码中的注释直接映射至式 (1)–(5) 的符号。

```python
import asyncio
from typing import Callable, Dict, List
from dataclasses import dataclass, field

@dataclass
class Message:
    sender: str
    receiver: str
    content: str
    msg_type: str = "default"  # task | review | test | result

@dataclass
class Agent:
    name: str
    role_prompt: str
    tools: List[Callable] = field(default_factory=list)
    inbox: asyncio.Queue = field(default_factory=asyncio.Queue)

    async def think(self, ctx: str) -> str:
        # 对应式(2): m_{j->i} = LLM_{theta_j}(s_j, task(i))
        # 实际生产环境替换为真实 LLM API 调用
        return f"[{self.name}] processed: {ctx[:60]}..."

class Orchestrator:
    def __init__(self, agents: Dict[str, Agent]):
        self.agents = agents
        self.blackboard: Dict[str, str] = {}  # 对应式(3): B^{(k)}

    async def dispatch(self, task: str) -> Dict[str, str]:
        # 对应式(4): 任务分配(此处简化为固定映射)
        pipeline = [
            ("coder", "Implement feature based on: " + task),
            ("reviewer", "Review the following code: "),
            ("tester", "Write tests for: "),
        ]
        for agent_name, prompt_prefix in pipeline:
            agent = self.agents[agent_name]
            # 从黑板读取前置输出作为当前输入
            prev = self.blackboard.get("last_output", "")
            full_prompt = f"{prompt_prefix}\n{prev}"
            msg = Message("orchestrator", agent_name, full_prompt, "task")
            await agent.inbox.put(msg)
            result = await agent.inbox.get()
            self.blackboard["last_output"] = await agent.think(result.content)
        return self.blackboard

    async def run_parallel(self, tasks: List[str]) -> List[Dict]:
        # 利用 asyncio 实现式(1)中的并行化收益
        return await asyncio.gather(*(self.dispatch(t) for t in tasks))

async def main():
    coder = Agent("coder", "You are a senior Python engineer.")
    reviewer = Agent("reviewer", "You are a code reviewer focused on safety.")
    tester = Agent("tester", "You are a QA engineer.")
    orch = Orchestrator({"coder": coder, "reviewer": reviewer, "tester": tester})
    results = await orch.run_parallel(["Build a REST API for user auth"])
    print(results)

if __name__ == "__main__":
    asyncio.run(main())
```

上述代码在 50 行核心逻辑内完成了: Agent 定义(角色参数 $\theta_i$)、消息队列($m_i^{(k)}$)、黑板系统($B^{(k)}$)、流水线调度(任务分配与顺序约束)、以及并行分发(`asyncio.gather` 对应式 (1) 的并行项)。生产级扩展需将 `think` 方法替换为真实 LLM 调用，并引入重试、超时与状态持久化。

## 7. 边界条件与失效模式

| 失效模式     | 触发条件                                                                   | 物理根源                                                                           | 缓解策略                                                         |
| :----------- | :------------------------------------------------------------------------- | :--------------------------------------------------------------------------------- | :--------------------------------------------------------------- |
| 通信熵爆炸   | Agent 数量$n > 20$ 且采用全连接拓扑                                      | 式 (1) 中$C_{\text{comm}}(n)$ 以 $O(n^2)$ 增长，token 开销超过子任务节省的收益 | 强制层级拓扑或引入"小组长"中间层，将通信复杂度压至$O(n)$       |
| 任务耦合循环 | 子任务$t_i$ 的输出是 $t_j$ 的输入，且 $t_j$ 的输出又反向依赖 $t_i$ | 分解阶段未识别循环依赖，导致死锁或无限循环                                         | 在调度前构建 DAG(有向无环图)，检测环并强制打破(引入虚拟合并节点) |
| 共识僵局     | 评审 Agent 对关键设计决策的投票持续$S(p) = 0$                            | 各 Agent 的后验概率分布差异过大，缺乏共享先验                                      | 引入仲裁 Agent(权重$w_{\text{arb}} \gg w_i$)或回退至人类审核   |
| 黑板污染     | 某个故障 Agent 向黑板写入低质量或矛盾信息                                  | 式 (3) 的$B^{(k)}$ 更新缺少过滤机制，错误信息被后续 Agent 消费并放大             | 为黑板写入增加置信度阈值，或设置只读/读写权限矩阵                |
| 角色同质化   | 多个 Agent 的$\theta_i$ 过于相似，导致冗余计算而非真正分工               | 系统 prompt 设计未体现足够的能力差异                                               | 为每个 Agent 配置专属 few-shot 示例与工具集，强化角色边界        |
| 级联错误     | 上游 Agent 输出微小错误(如变量名笔误)，下游 Agent 未校验直接传播           | 缺乏跨 Agent 的契约验证(schema check)                                              | 在消息传递层引入 Pydantic/JSON Schema 校验，强制类型安全         |

## 8. 典型框架对比

| 框架      | 架构范式            | 通信模型                 | 协调机制                     | 适用场景                       | 成熟度 |
| :-------- | :------------------ | :----------------------- | :--------------------------- | :----------------------------- | :----- |
| AutoGen   | 对等式 + 可选层级   | 对话(Conversable)        | 群聊管理器(GroupChatManager) | 快速原型、探索性对话           | 高     |
| MetaGPT   | 严格层级            | SOP 消息流               | 顺序执行 + 角色间交付物检查  | 软件工程自动化                 | 中     |
| CrewAI    | 对等式 + 任务流水线 | 委托(Delegation)         | 任务依赖图 + 人类反馈回路    | 商业分析、内容创作             | 高     |
| LangGraph | 状态图(任意拓扑)    | 状态通道(State Channels) | 显式条件边 + 循环/分支       | 复杂工作流、需要精确控制与回滚 | 高     |

AutoGen 的群聊模式在灵活性上最优，但缺乏对长流程的显式状态管理，适合 5 轮以内的短对话链。MetaGPT 的 SOP 模式将软件工程方法论固化到 Agent 行为中，对需求明确、流程标准化的任务(如 CRUD 应用开发)效果极佳，但对探索性任务(如开放式研究)过度约束。LangGraph 的图模型提供了最强的可控性，其状态快照机制支持类似数据库事务的回滚，是金融、医疗等对正确性要求极高场景的首选。

## 9. 技术前瞻与演进

当前 MAS 的未解难题集中在三个维度: 

**(a)通信协议的语义压缩**。Agent 之间使用自然语言通信，信息密度远低于结构化协议。一条 500 token 的消息可能只传递了一个布尔决策。未来方向是训练专门的"Agent 间语言"(Inter-Agent Language)——一种比自然语言更紧凑、比 protobuf 更灵活的中间表示，由 LLM 原生读写。

**(b)动态拓扑重组**。现有框架的通信拓扑(层级、图、星型)多为静态配置。当任务 midway 发现需要引入新角色(如"安全审计 Agent")或合并两个低效 Agent 时，系统无法自适应调整。借鉴生物群体智能(如蚁群算法)的动态角色转换机制是一个潜在修补方向。

**(c)经济调度与算力预算**。MAS 的总成本(如式 (1) 所示)并非单调递减。当 $n$ 过大时，通信与协调开销将压倒专业化收益。需要建立在线的成本-收益模型，根据任务的实时复杂度动态决定最优 Agent 数量与拓扑结构，而非由人类预先设定。
