# Agent 工具调用演进：从 Function Calling 到 Skill

> 本文梳理大模型 Agent 工具调用机制的三代演进——从 Function Calling 的直接调用,到 MCP 的标准化协议,再到 Skill 的封装化能力单元,分析各代方案的设计权衡、适用场景与架构演进逻辑. 

---

## 1. 第一代：Function Calling——工具调用的起点

### 1.1 实现机制

Function Calling(函数调用)是 OpenAI 在 2023 年 GPT-4 中首次引入的工具调用机制,其核心思想是：**让 LLM 在生成自然语言的同时,能够输出结构化的函数调用请求**. 

具体流程：
1. **Schema 注册**：开发者将可用工具的接口定义(函数名、参数类型、描述)以 JSON Schema 格式注册到模型上下文. 
2. **意图识别**：模型根据用户输入判断是否需要调用工具,以及调用哪个工具. 
3. **参数填充**：模型生成符合 Schema 的函数调用 JSON,包含函数名和参数值. 
4. **外部执行**：宿主程序解析 JSON,调用实际函数,将结果返回给模型. 
5. **结果整合**：模型根据函数返回值生成最终回答. 

### 1.2 优势与局限

**优势**：
- 实现简单,单次 HTTP 请求即可完成工具调用
- 模型无关——任何支持 JSON 输出的 LLM 都可使用
- 延迟低,无额外协议层开销

**局限**：
- **碎片化**：每新增一个工具都需要修改 Schema 并重新部署
- **不可复用**：工具定义与具体模型/应用强耦合
- **维护成本高**：N 个工具 × M 个模型 = N×M 个适配工作
- **无状态**：每次调用都是独立的,无法维护跨调用的上下文状态

## 2. 第二代：MCP——工具调用的标准化生态

### 2.1 核心设计

MCP(Model Context Protocol)由 Anthropic 于 2024 年提出,旨在解决 Function Calling 的碎片化问题. 其核心理念是：**通过标准化协议,将工具提供方与工具消费方解耦**. 

MCP 的三层架构：

| 层级 | 职责 | 示例 |
|:-----|:-----|:-----|
| **Transport Layer** | 通信传输(stdio、SSE、HTTP) | 本地进程通信、远程服务调用 |
| **Protocol Layer** | 消息格式与生命周期管理 | 工具发现、调用、错误处理 |
| **Capability Layer** | 能力定义(Tools、Resources、Prompts) | 文件系统访问、数据库查询、API 调用 |

### 2.2 MCP 生态

MCP 的标准化催生了工具生态的爆发：
- **工具提供方**：独立开发 MCP Server,暴露标准化接口(如文件系统、Git、数据库、搜索引擎)
- **工具消费方**：任何支持 MCP 的 Client(Claude Desktop、Cursor、Windsurf 等)可零代码集成这些工具
- **中间件**：MCP Registry 提供服务发现,MCP Bridge 实现协议转换

### 2.3 权衡：MCP 的"重"与 Function Calling 的"轻"

| 维度 | Function Calling | MCP |
|:-----|:----------------|:----|
| 架构复杂度 | 轻量,单次调用 | 重,需维护 Server/Client 连接 |
| 标准化程度 | 低,各平台 Schema 不一 | 高,统一协议 |
| 生态复用 | 无,工具与应用强耦合 | 强,一次开发处处可用 |
| 延迟 | 低(单次推理) | 中(额外协议层 + 服务发现) |
| 状态管理 | 无状态 | 可维护会话级状态 |
| 适用场景 | 快速原型、本地控制 | 生态集成、企业级部署 |

**OpenClaw 的选择**：以 OpenClaw 为代表的个人 Agent 选择回归 Function Calling,核心考量是响应速度与系统简洁性. 对于本地运行的个人 Agent,MCP 的协议开销和服务发现延迟是不可接受的——用户要求"现在打开我的备忘录",不应等待 MCP Server 的握手过程.

## 3. 第三代：Skill——从"能调用"到"可交付"

### 3.1 核心思想

Skill 代表了工具调用范式的第三次跃迁：**从"调用单个工具"升级到"执行完整任务"**. 如果说 Function Calling 是给 AI 一把锤子,MCP 是给 AI 一个工具箱,那么 Skill 就是给 AI 一本"维修手册"——不仅提供工具,还提供何时用、怎么用、用完后如何处理的标准流程. 

Skill 的典型结构：

```
skill-name/
├── SKILL.md          # 能力定义：职责边界、输入输出、执行流程
├── scripts/          # 可执行逻辑
│   ├── preprocess.py
│   └── validate.sh
└── resources/        # 素材、模板、参考数据
    ├── template.md
    └── examples.json
```

### 3.2 Think Tool 与推理显性化

Anthropic 提出的 **Think Tool** 是 Skill 思想的前身. 它不给模型增加外部能力,而是让模型在推理过程中显式地"停下来思考"——将中间推理过程输出到一个特殊的工具调用中,再基于思考结果继续生成. 

这一设计的洞察是：模型的推理能力并不受限于参数知识,而受限于**推理过程的可见性和可编辑性**. Think Tool 让模型的思考过程从隐式变为显式,从而可以通过外部干预(如人工修正、规则检查)提升推理质量. 

### 3.3 三代范式对比

| 范式 | 抽象层级 | 复用方式 | 学习成本 | 代表产品 |
|:-----|:--------|:--------|:--------|:--------|
| Function Calling | 单次 API 调用 | 代码级复制 | 低 | GPT-4 Function Calling |
| MCP | 标准化工具接口 | 协议级集成 | 中 | Claude Desktop, Cursor |
| Skill | 完整任务能力包 | 文件级分享 | 低(仅需 Markdown) | Claude Skills, OpenClaw |

### 3.4 从"能调用"迈向"可交付"

Skill 的真正价值在于**能力的工程化封装**. 一个精心设计的 Skill 不仅是工具集合,更是：
- **知识沉淀**：将领域专家的经验固化为可执行流程
- **质量保障**：内置输入验证、错误处理、边界检查
- **可维护性**：版本控制、文档化、可测试
- **可复用性**：一次开发,跨模型、跨平台使用

这意味着 AI 能力第一次可以像软件包一样被**版本控制、单元测试、持续集成**——从" prompt 工程"的 craftsmanship 升级为"能力工程"的 industrialization.

## 4. 未来方向

**协议统一与分层共存**. 未来不太可能出现"一统天下"的单一方案,而是 Function Calling / MCP / Skill 在各自适用场景中分层共存：
- Function Calling 负责本地、低延迟、简单工具调用
- MCP 负责跨服务、跨团队、生态级工具集成
- Skill 负责复杂任务的标准化执行与知识沉淀

### 3.5 Claude Skills 的工程实践

Anthropic 推出的 Claude Skills 将 Skill 概念落地为**上下文工程的工程化实现**. 其核心设计是一个文件夹结构,包含三类文件：

- **Instructions(SKILL.md)** ：定义 Skill 的职责、任务类型判断逻辑、可用资源和执行流程
- **Scripts**：存放具体逻辑脚本(如 Python、Shell),实现自动化操作
- **Resources**：素材、模板、数据文件等静态资源

**按需加载机制**是 Claude Skills 的关键优化：
1. 首先加载元数据(约 100 词),快速判断是否需要该 Skill
2. 然后读取 SKILL.md 主体,获取完整指令
3. 最后按需读取具体资源文件

这种分层加载策略显著降低了 token 消耗和响应延迟. 

**使用示例**：以官方 `markdown-summarizer` Skill 为例,用户只需将 Skill 文件夹放入 skills 目录,然后输入"总结这个 Markdown 文档",Claude 即自动加载 Skill 并执行结构化总结——识别标题、摘要、分级内容、列表和代码块. 

**当前局限**：
- 部分 Skill 依赖系统级工具(如 `apt-get`),跨平台兼容性待提升
- 缺乏内置持久化存储,长期状态追踪需外挂数据库

## 4. 未来方向

**协议统一与分层共存**. 未来不太可能出现"一统天下"的单一方案,而是 Function Calling / MCP / Skill 在各自适用场景中分层共存：
