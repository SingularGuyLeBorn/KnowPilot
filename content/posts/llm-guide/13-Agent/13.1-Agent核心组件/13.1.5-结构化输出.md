# 结构化输出

## 演进脉络

LLM 输出格式的演进,本质是"人类可读性"向"机器可解析性"妥协的过程. 早期模型被训练为生成流畅散文,下游系统被迫用脆弱的正则表达式提取信息. 现代基础设施将结构化约束推进至模型解码层,使格式正确率从 70% 提升至 99.9%. 

| 阶段 | 时间 | 代表技术 | 约束位置 | 格式正确率(JSON) | 核心局限 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| 自由文本 | 2018-2020 | GPT-2, GPT-3 | 无 | < 30% | 完全依赖下游解析,格式不可控 |
| 提示工程 | 2020-2021 | Few-shot JSON 示例 | Prompt | 60-75% | 复杂模式易漏字段、出尾随逗号 |
| 正则约束 | 2021-2022 | Regex-guided decoding | 后处理 | 70-80% | 仅能在生成后过滤,无法预防错误 |
| JSON Schema | 2022-2023 | OpenAI function calling | API 层 | 85-95% | API 封装黑盒,无法自定义Decoder   |
| CFG 约束解码 | 2023-2024 | Outlines, LMQL, Guidance | Decoder   | 95-99.5% | 增加解码步计算开销 10-30% |
| 原生结构化 | 2024-至今 | OpenAI structured output, Claude artifacts | 模型/训练 | 99.5%+ | 供应商锁定,开源模型支持滞后 |

GPT-3 时代,开发者通过在 prompt 中插入 `"请严格按 JSON 格式返回,不要包含 markdown 代码块"` 试图约束输出. 该方案在简单键值对上有效,面对嵌套数组、可选字段、Union 类型时,模型仍会生成 Markdown 围栏代码块(```json)、尾随逗号、未转义换行符等非法语法. 

OpenAI 于 2023 年 6 月发布 function calling API,允许调用者声明 JSON Schema,模型在输出层强制匹配. 该功能并非修改Decoder  ,而是在采样后对 token 序列进行合法性校验与重试,时延波动较大. 

Outlines、Guidance 等库将约束下移至Decoder  : 在每一步预测时,仅允许推进 JSON 语法有限状态机的合法 token. 错误 token 在 logits 阶段即被屏蔽,生成过程本身保证语法正确. 代价是每步需对完整词表执行 mask 计算,增加 10-30% 的解码延迟. 

## 工业价值

结构化输出是 LLM 从聊天玩具升级为生产系统的关键基建. 

**机器可读与下游集成**. API、数据库、前端组件均要求强类型输入. 自由文本的输出需经过 NLP 解析器或二次 LLM 提取,引入延迟与错误级联. 结构化输出使 LLM 直接成为类型化流水线的一环,与 Pydantic、Zod、TypeScript 类型系统无缝衔接. 

**类型安全**. 在 Python 生态中,Pydantic 模型可在运行时验证字段类型、范围、正则模式. `response_model=UserProfile` 将 LLM 输出直接反序列化为类型化对象,在接口边界处捕获类型错误,而非在下游业务逻辑中引发 `KeyError` 或 `TypeError`. 

**减少解析错误**. OpenAI 统计表明,function calling 将开发者解析失败率从 15% 降至 < 2%. 在需要高可靠性的场景(如金融数据提取、医疗记录结构化),2% 与 15% 的差距决定产品可用性. 

**确定性接口**. 结构化输出使 LLM 的行为可契约化. 两个版本的模型若均遵守同一 JSON Schema,下游系统无需修改即可升级模型. 无结构约束时,模型版本迭代可能改变输出散文格式,导致解析器失效. 

## 物理直觉

LLM 难以保证输出格式的物理根因在于自回归采样的 token 级不可控性. 

自回归模型在每一步生成 token $x_t$ 时,基于前文 $x_{<t}$ 计算条件概率分布 $P(x_t \mid x_{<t})$. 该分布覆盖整个词表(通常 32,000 至 200,000 token). 温度参数 $T$ 与 top-p 采样引入随机性,使低概率 token 仍有非零选中概率. 

JSON 语法要求在 `"key":` 之后出现 `"` 开启字符串值. 若某步采样中 `"` 的 logits 排名较低,而温度 $T$ 设置较高,`{`, `[`, `t`(true)等非法 token 可能被选中. 此时模型已"物理上"偏离合法 JSON 轨道,后续所有 token 均建立在非法前缀之上. 

传统方案依赖 prompt 请求模型"不要犯错". 这相当于要求一个物理系统(自回归采样器)违背其内在动力学规律(概率分布). 约束解码的物理意义是在采样动力学中引入外部势场: 将非法 token 的概率质量设为零,强制轨迹始终保持在合法语法流形上. 

## JSON 输出

### JSON Schema 约束

JSON Schema 定义了输出的结构契约: 字段名、类型、是否必需、嵌套层级、枚举值. OpenAI function calling 与 Claude structured output 将 Schema 编译为模型可见的约束. 

以用户画像提取为例: 

```json
{
  "type": "object",
  "properties": {
    "name": {"type": "string"},
    "age": {"type": "integer", "minimum": 0},
    "skills": {"type": "array", "items": {"type": "string"}}
  },
  "required": ["name", "age"]
}
```

API 层在生成后使用 JSON Schema 验证器(如 `jsonschema`)检查输出. 验证失败时触发重试,最多 $N_{\text{retry}}$ 次. 该方案的正确率上限受限于重试次数与模型本身的格式理解能力. 

### 语法约束解码

约束解码(Constrained Decoding)将验证从"生成后"前移至"生成中". 核心思想是在每一步计算允许的 token 集合 $V_{\text{valid}}(x_{<t})$,将 logits 中不属于该集合的项置为 $-\infty$. 

$$
 \tilde{P}(x_t \mid x_{<t}) = \frac{\exp(z_{x_t} / T)}{\sum_{v \in V_{\text{valid}}} \exp(z_v / T)} \cdot \mathbb{I}(x_t \in V_{\text{valid}}) \tag{1}
$$
式 (1) 中,$\tilde{P}$ 为约束后的条件概率,$z_v$ 为 token $v$ 的 logit,$T$ 为温度参数,$V_{\text{valid}}$ 为当前步骤语法允许的 token 集合,$\mathbb{I}(\cdot)$ 为指示函数(合法 token 取 1,非法取 0). 如式 (1) 所示,分子保留合法 token 的 logit 并经温度缩放; 分母仅对合法 token 求和,实现概率重归一化. 指示函数 $\mathbb{I}(x_t \in V_{\text{valid}})$ 将非法 token 的概率质量强制置零. 该操作在物理上等价于在自回归动力系统中引入无穷大势垒,禁止轨迹穿越至非法区域. 

$V_{\text{valid}}$ 的构造依赖于上下文无关文法(CFG)或有限状态机(FSM). 对于 JSON 对象,若当前已生成 `{"name": "`,FSM 处于"字符串值"状态,允许的下一字符为任意 Unicode 字符或 `"`(结束字符串). `}` 或 `,` 在该状态下非法. 

### Outlines、LMQL、Guidance 原理

| 库 | 约束表达方式 | Decoder  集成 | 状态机实现 | 性能特征 |
| :--- | :--- | :--- | :--- | :--- |
| Outlines | Pydantic / JSON Schema / Regex | 自定义 logits processor | FSM (interegular) | 预编译 FSM,每步 O(1) mask |
| LMQL | SQL-like 查询语言 | 修改 Transformers 生成循环 | LL(1) Parser | 解析器每步重计算,开销较大 |
| Guidance | Handlebars 模板 + 生成指令 | 自定义生成引擎 | PEG Parser | 支持交错控制流与生成,复杂度高 |
| SGLang | DSL + 运行时编译 | RadixAttention + FSM | FSM | 高吞吐,支持批量约束 |

Outlines 的核心架构分为三层: 

1. **约束编译器**: 将 JSON Schema、Regex 或 Pydantic 模型编译为正则表达式,再经 `interegular` 库转换为 FSM. 
2. **FSM 执行器**: 维护当前 FSM 状态. 每生成一个 token,将 token 解码为字符序列,驱动 FSM 状态转移. 
3. **Logits Processor**: 在模型前向传播后、采样前,查询 FSM 执行器获取合法下一 token 集合,构建二进制 mask 应用于 logits. 

该设计的关键优化在于 FSM 预编译与 token 前缀树索引. 传统方案每步需遍历全部词表验证合法性,复杂度 $O(|V| \cdot L)$,$|V|$ 为词表大小,$L$ 为 token 平均长度. Outlines 通过构建"FSM 状态 × token 首字符"的索引,将每步 mask 计算降至 $O(|V_{\text{candidate}}|)$,其中候选 token 数通常比全词表小 1-2 个数量级. 

### JSON 模式下的常见失败

| 失败类型 | 示例 | 物理根因 | 缓解策略 |
| :--- | :--- | :--- | :--- |
| 尾随逗号 | `{"a": 1,}` | FSM 未正确区分"最后一个元素"与"还有更多元素"状态 | 严格 JSON Schema(无尾随逗号模式) |
| 未转义字符 | `{"msg": "Line 1\nLine 2"}` | 换行符直接写入而非 `\\n` | FSM 中对控制字符强制转义 |
| 类型不匹配 | `{"count": "ten"}` | Schema 声明 `integer` 但模型生成字符串 | 约束解码在 `"ten"` 首字符处即屏蔽 `"` |
| 字段缺失 | `{"name": "Alice"}`(age 必需) | Schema 要求字段未在 prompt 中充分强调 | 将 required 字段注入 prompt + FSM 强制 |
| 数值溢出 | `{"score": 1e400}` | 超出 IEEE-754 双精度范围 | Schema 设置 `maximum` / `minimum` 并编译至 FSM |
| Markdown 围栏 | ```json\n{...}\n``` | 训练数据中 JSON 常包裹在 markdown 代码块内 | 约束解码直接禁止 `` ` `` token |

## XML 输出

### XML Schema 约束

XML Schema(XSD)定义了元素层级、属性类型、出现次数约束. 与 JSON Schema 相比,XML 的标记语法(`<tag>...</tag>`)在生成过程中提供更频繁的语法Checkpoint: 每个开启标签必须匹配关闭标签,FSM 可通过标签栈实时验证嵌套合法性. 

XML 约束解码的 FSM 需维护栈结构,状态空间大于 JSON. JSON 仅需跟踪"当前值类型"(对象/数组/字符串/数字),XML 需额外跟踪"已开启但未关闭的标签栈". 栈深度为 $D$ 时,FSM 状态数约为 $O(N_{\text{tags}} \cdot D)$,$N_{\text{tags}}$ 为不同标签数量. 

### React/Angular 式组件化输出

部分 Agent 框架(如 Vercel AI SDK、LangChain XML Agent)要求 LLM 直接生成可渲染的 JSX/XML 组件. 输出示例: 

```xml
<tool_call>
  <tool_name>weather_lookup</tool_name>
  <parameters>
    <city>Beijing</city>
    <date>2024-06-01</date>
  </parameters>
</tool_call>
```

组件化输出的优势在于可扩展性: 新增工具无需修改解析器,只需注册新标签的 Schema. 前端可直接将 XML 映射为虚拟 DOM 节点,实现 LLM 输出到 UI 的零胶水层转换. 

### XML 与 JSON 对比

| 维度 | JSON | XML |
| :--- | :--- | :--- |
| 可读性 | 低(嵌套深时括号混乱) | 中(标签自描述,但冗余) |
| FSM 状态空间 | 小(无标签栈) | 大(需维护开启标签栈) |
| 模式验证 | JSON Schema | XSD / DTD |
| 扩展性 | 差(字段名变更破坏解析) | 优(新标签可被旧解析器忽略) |
| 与前端集成 | 需 JSON.parse + 映射 | 可直接映射至虚拟 DOM |
| 注释支持 | 不支持 | 支持 `<!-- -->` |
| 属性 vs 子元素 | 仅键值对 | 属性(元数据)与子元素(数据)分离 |

JSON 在 API 集成中占主导,因其与 JavaScript/Python 对象模型同构. XML 在需要混合内容与标记的场景(如富文本生成、UI 组件树)中更具优势. 

## 数学推导

### 约束解码的数学形式

标准自回归解码将输出序列 $x_{1:T}$ 的联合概率分解为条件概率连乘: 

$$
 P(x_{1:T}) = \prod_{t=1}^{T} P(x_t \mid x_{<t}) \tag{2}
$$
式 (2) 中,$P(x_{1:T})$ 为序列联合概率,$P(x_t \mid x_{<t})$ 为第 $t$ 步的条件概率,$T$ 为序列总长度. 如式 (2) 所示,联合概率被分解为每步条件概率的乘积. 该分解基于概率链式法则,是自回归生成的数学基础. 

引入格式约束集合 $\mathcal{C}$(如 JSON Schema 定义的所有合法字符串集合),约束解码的目标是: 

$$
 x^* = \arg\max_{x_{1:T} \in \mathcal{C}} P(x_{1:T}) \tag{3}
$$
式 (3) 中,$x^*$ 为最优约束序列,$\mathcal{C}$ 为合法序列集合,由上下文无关文法或正则语言定义. 式 (3) 将搜索空间从全词表序列 $\mathcal{V}^T$ 限制至合法子集 $\mathcal{C} \subset \mathcal{V}^T$. 直接求解式 (3) 的复杂度为 $O(|\mathcal{C}|)$,而 $|\mathcal{C}|$ 随序列长度指数增长. 

约束解码的贪心近似在每步局部限制: 

$$
 x_t^* = \arg\max_{x_t \in V_{\text{valid}}(x_{<t})} P(x_t \mid x_{<t}) \tag{4}
$$
式 (4) 中,$x_t^*$ 为第 $t$ 步的约束最优 token,$V_{\text{valid}}(x_{<t})$ 为前缀 $x_{<t}$ 下语法允许的 token 子集. 如式 (4) 所示,约束解码不全局搜索最优序列,而是在每步将 argmax 限制在当前合法子集内. 该贪心策略保证生成序列的语法正确性,但可能牺牲语义最优性(局部合法选择可能导向全局死胡同). 

### 基于 FSM 的约束解码

将格式约束编译为确定性有限状态机(DFA)$\mathcal{M} = (Q, \Sigma, \delta, q_0, F)$,其中 $Q$ 为状态集,$\Sigma$ 为字符集,$\delta$ 为转移函数,$q_0$ 为初始状态,$F$ 为接受状态集. 

tokenizer 将字符序列映射为 token 序列,但 token 与字符边界不对齐: 单个 token 可能包含多个字符,单个字符可能被拆分为多个 token(如 UTF-8 多字节字符). 因此,FSM 状态转移需在字符级别执行,而 logits mask 在 token 级别应用. 

定义 token $v$ 解码后的字符序列为 $c_1 c_2 \cdots c_k$. token $v$ 从状态 $q$ 出发合法的充要条件是: 

$$
 \delta^*(q, c_1 c_2 \cdots c_k) \neq \emptyset \tag{5}
$$
式 (5) 中,$\delta^*$ 为 DFA 的扩展转移函数(处理字符串而非单字符),$c_1 \cdots c_k$ 为 token $v$ 解码后的字符序列. 如式 (5) 所示,token 合法当且仅当其完整字符序列驱动 DFA 到达某个有效状态(不必是接受状态,中间状态亦可). Outlines 在预计算阶段构建"状态-字符"转移表,运行时每步仅需查表即可确定合法 token 集合. 

对于词表中的每个 token $v$,预计算其所有可能的前缀状态路径. 若存在至少一条路径从当前状态 $q$ 出发可完整消耗 token $v$,则 $v \in V_{\text{valid}}$. 该预计算的词表索引使运行时 mask 构建降至亚毫秒级. 

约束解码的条件概率在 FSM 框架下重写为: 

$$
 P(x_t \mid x_{<t}, \mathcal{M}) = \frac{\exp(z_{x_t} / T)}{\sum_{v \in V_{\text{valid}}(q_t)} \exp(z_v / T)} \cdot \mathbb{I}(x_t \in V_{\text{valid}}(q_t)) \tag{6}
$$
式 (6) 中,$\mathcal{M}$ 为格式约束自动机,$q_t = \delta^*(q_0, \text{decode}(x_{<t}))$ 为前缀驱动后的当前 FSM 状态. 如式 (6) 所示,约束概率与式 (1) 形式相同,但合法集合 $V_{\text{valid}}$ 显式依赖于当前 FSM 状态 $q_t$. 随着序列生成,$q_t$ 沿 FSM 轨迹迁移,合法 token 集合动态变化. 当 $q_t \in F$(接受状态)时,生成可合法终止. 

## 代码实现

以下实现基于 Outlines,演示如何将 Pydantic 模型编译为 FSM 约束,并应用于 Hugging Face Transformers 模型. 生产部署应使用 `outlines.generate.json` 高层 API,此处展示底层 logits processor 以增强可解释性. 

```python
import torch
from transformers import AutoModelForCausalLM, AutoTokenizer, LogitsProcessorList
from pydantic import BaseModel, Field
from typing import List
import outlines
from outlines.fsm.guide import RegexGuide
from outlines.models.transformers import TransformerTokenizer

# 定义结构化输出模式(对应下游系统类型契约)
class UserProfile(BaseModel):
    name: str = Field(..., min_length=1, max_length=50)
    age: int = Field(..., ge=0, le=150)
    skills: List[str] = Field(..., min_length=0, max_length=10)
    email: str = Field(..., pattern=r"^[\w\.-]+@[\w\.-]+\.\w+$")

# 将 Pydantic 模型编译为正则表达式,再转换为 FSM
# 对应数学推导中的自动机 M
schema_regex = outlines.generate.regex(UserProfile)

# 加载模型与 tokenizer(示例使用小型模型,生产用 Qwen2.5 / Mistral)
model_name = "Qwen/Qwen2.5-0.5B-Instruct"
tokenizer = AutoTokenizer.from_pretrained(model_name, trust_remote_code=True)
model = AutoModelForCausalLM.from_pretrained(
    model_name,
    torch_dtype=torch.float16,
    device_map="auto",
    trust_remote_code=True,
)

# Outlines 包装 tokenizer 以统一接口
outlines_tokenizer = TransformerTokenizer(tokenizer)

# 编译 FSM Guide(对应式 (6) 中的 M)
guide = RegexGuide.from_regex(schema_regex, outlines_tokenizer)

class FSMLoigtsProcessor(torch.nn.Module):
    """
    基于 FSM 的 logits processor. 
    每步根据当前 FSM 状态构造合法 token mask,
    对应式 (6) 中的 I(x_t ∈ V_valid(q_t)). 
    """
    def __init__(self, guide: RegexGuide):
        super().__init__()
        self.guide = guide
        self.state = guide.initial_state  # q_0
    
    def __call__(self, input_ids: torch.LongTensor, scores: torch.FloatTensor) -> torch.FloatTensor:
        # 获取当前状态下合法 token 集合 V_valid(q_t)
        allowed_tokens = self.guide.get_next_instruction(self.state).tokens
        
        if allowed_tokens is not None and len(allowed_tokens) > 0:
            # 构建二进制 mask: 非法 token 概率质量置零
            mask = torch.full_like(scores, float("-inf"))
            mask[:, allowed_tokens] = 0.0
            scores = scores + mask  # 非法项变为 -inf
        
        return scores
    
    def advance(self, token_id: int):
        """根据采样的 token 驱动 FSM 状态转移. """
        self.state = self.guide.get_next_state(self.state, token_id)

# 构造 prompt
prompt = "Extract user profile from: Alice is 28, knows Python and Rust, alice@example.com"
messages = [
    {"role": "system", "content": "You are a structured data extractor."},
    {"role": "user", "content": prompt},
]
text = tokenizer.apply_chat_template(messages, tokenize=False, add_generation_prompt=True)
input_ids = tokenizer(text, return_tensors="pt").input_ids.to(model.device)

# 初始化 logits processor
fsm_processor = FSMLoigtsProcessor(guide)
logits_processors = LogitsProcessorList([fsm_processor])

# 手动生成循环(便于展示 FSM 状态转移,生产环境可用 model.generate)
generated = input_ids.clone()
max_new_tokens = 256

for _ in range(max_new_tokens):
    outputs = model(generated)
    next_token_logits = outputs.logits[:, -1, :]
    
    # 应用 FSM 约束 mask(对应式 (6))
    next_token_logits = fsm_processor(generated, next_token_logits)
    
    # 贪婪采样(温度 T=1 的 argmax)
    next_token = torch.argmax(next_token_logits, dim=-1, keepdim=True)
    
    # 驱动 FSM 状态转移
    fsm_processor.advance(next_token.item())
    
    generated = torch.cat([generated, next_token], dim=-1)
    
    # 检查 FSM 是否到达接受状态
    if fsm_processor.state in guide.final_states:
        break

# 解码并解析结果
result_text = tokenizer.decode(generated[0][input_ids.shape[1]:], skip_special_tokens=True)
print("Generated JSON:", result_text)

# 反序列化为 Pydantic 对象(运行时类型验证)
profile = UserProfile.model_validate_json(result_text)
print("Parsed profile:", profile.model_dump())
```

该实现将 Pydantic 模型编译为正则表达式,再经 `interegular` 构建 FSM. `FSMLoigtsProcessor` 在每步前向传播后查询当前 FSM 状态的合法 token 集合,将非法 token 的 logits 置为 $-\infty$,严格对应式 (6). 生成循环中 `advance` 方法驱动状态转移,确保生成轨迹始终停留在合法语法流形上. 

## 算力成本与数值走查

约束解码引入的计算开销主要来自 mask 构建与词表过滤. 以下对 Llama-3.1-8B(词表大小 $|V| = 128256$)进行数值走查. 

**无约束解码的单步 FLOPs**: 

$$
 \text{FLOPs}_{\text{base}} = 2 \cdot d_{\text{model}} \cdot |V| \tag{7}
$$
式 (7) 中,$\text{FLOPs}_{\text{base}}$ 为输出层投影的浮点运算量,$d_{\text{model}}$ 为模型隐藏层维度(Llama-3.1-8B 为 4096),$|V|$ 为词表大小. 如式 (7) 所示,输出层将隐藏状态 $h \in \mathbb{R}^{d_{\text{model}}}$ 投影至词表空间,计算矩阵-向量乘 $W_{\text{out}} \cdot h$,其中 $W_{\text{out}} \in \mathbb{R}^{|V| \times d_{\text{model}}}$. 运算量为 $2 \cdot |V| \cdot d_{\text{model}}$(乘加各一次). 

基于上述分析,建立如下数学关系: 

$$
 \text{FLOPs}_{\text{base}} = 2 \times 4096 \times 128256 \approx 1.05 \times 10^9 \text{ FLOPs} \tag{8}
$$

即每生成一个 token 需约 1 GFLOP 的输出层计算. 

**约束解码的额外开销**: Outlines 的 mask 构建在 CPU 上执行,需遍历候选 token 集合. 假设平均合法 token 数 $|V_{\text{valid}}| = 5000$(JSON 生成中期典型值,大量 token 因语法非法被屏蔽),预计算索引使每步仅需查表即可确定合法性,但 logits 张量仍驻留 GPU,需将 CPU 构建的 mask 上传至 GPU. 

GPU mask 应用为逐元素加法(scores + mask),计算量可忽略. 主要瓶颈在于 CPU-GPU 同步: 每步需将合法 token ID 列表从 CPU 传至 GPU. 列表长度 5000,int32 编码,数据量 $5000 \times 4 = 20$ KB. PCIe 4.0 x16 带宽 32 GB/s,传输延迟约 $0.6 \mu s$,远低于 GPU 内核执行时间. 

实际观测到的 10-30% 延迟增加主要来自: 

1. **Python 开销**: Outlines 的 FSM 状态查询在 Python 层执行,每步涉及多次字典查找. 
2. **非连续索引**: 合法 token 在词表中非连续分布,`torch.index_fill` 或高级索引操作在 GPU 上的效率低于连续切片. 

**批量生成的扩展**: 当 batch size $B = 32$ 时,每步需处理 32 个并行的 FSM 状态. 若各序列的约束 Schema 相同(如均生成 `UserProfile`),可共享同一 FSM 实例,仅需并行查询 32 个状态. Outlines 当前实现对此优化有限,SGLang 通过 RadixAttention 与批量 FSM 状态管理将 batch 约束解码的吞吐提升至接近无约束水平. 

**端到端成本估算**: 假设生成 200 token 的 JSON 对象,Llama-3.1-8B 在 A100 上的无约束吞吐约 80 token/s. 约束解码增加 20% 延迟,吞吐降至 67 token/s. 生成 200 token 耗时从 2.5s 增至 3.0s,单请求额外 0.5s GPU 占用. 按 AWS g5.xxlarge(A10G)$1.006/小时定价,单请求成本: 

$$
 \text{Cost}_{\text{req}} = \frac{3.0}{3600} \times 1.006 \approx \$0.00084 \text{ (约束)} \tag{9}
$$
$$
 \text{Cost}_{\text{base}} = \frac{2.5}{3600} \times 1.006 \approx \$0.00070 \text{ (无约束)} \tag{10}
$$

单请求成本差异 $0.00014,对于百万级日请求量的系统,日额外成本约 $140. 该成本在避免 15% 解析错误率导致的下游重试与人工介入面前通常可接受. 

## 边界条件与失效模式

### 模式过于复杂导致解码变慢

| 复杂度指标 | 阈值 | 症状 | 物理根因 |
| :--- | :--- | :--- | :--- |
| Schema 嵌套深度 | > 5 层 | FSM 状态爆炸 | 每层嵌套引入笛卡尔积状态数 |
| 可选字段数 | > 20 个 | 每步合法 token 集膨胀 | 可选分支增加 NFA 状态 |
| Union 类型数 | > 5 个 | 解析歧义 | `anyOf` 需并行维护多条 FSM 路径 |
| 正则模式复杂度 | 回溯型 | 编译超时 | 正则到 DFA 的子集构造法指数级 |

JSON Schema 的 `anyOf` 要求生成内容匹配多个子模式之一. FSM 编译时需构造并行的非确定性路径,每步维护活跃状态集合. 当 `anyOf` 分支数 $k = 5$ 时,活跃状态集合大小理论上界为 $2^k = 32$,实际中通常更小,但仍增加 mask 计算量. 

缓解策略: 将复杂 Schema 拆分为多轮调用. 例如,先提取 `name` 与 `age`,再单独提取 `skills` 数组. 每轮 Schema 简单,FSM 状态数可控,总延迟可能低于单轮复杂 Schema. 

### 嵌套层级过深

自回归模型的有效上下文长度有限. 当 JSON 嵌套深度超过模型训练时的典型深度(如 10 层),模型可能丢失层级对称性约束,生成 `[[[[[` 而无法正确关闭. 约束解码的 FSM 强制语法正确,但不保证语义合理. `[[[[[]]]]]` 在语法上合法,但语义上可能是错误的数组数量. 

深层嵌套还增加序列长度. 每层嵌套消耗 2 个 token(开启与关闭括号),20 层嵌套仅括号即 40 token. 在 4K 上下文限制下,深层对象可能截断内容. 

### Unicode 转义问题

JSON 字符串要求非 ASCII 字符转义为 `\uXXXX` 序列. tokenizer 可能将 `\u` 拆分为两个 token(`\` 与 `u`),或作为单个 token. 若 FSM 在字符级别要求 `\u` 后跟随 4 位十六进制,但 tokenizer 在 `\u` 处已生成单个 token,则 FSM 无法准确验证后续字符,导致非法 Unicode 转义序列漏出. 

更隐蔽的问题是代理对(surrogate pair). Emoji (U+1F600)在 UTF-16 中为代理对 `\uD83D\uDE00`. JSON 允许直接存储 UTF-8 或转义序列. 若模型生成 `\uD83D` 后未跟随 `\uDE00`,形成孤立的代理对前半部分,下游解析器可能抛出异常. 约束解码需显式处理代理对的成对约束,增加 FSM 复杂度. 

### 温度与约束冲突

温度 $T \to \infty$ 时,概率分布趋近均匀. 若合法 token 集合 $|V_{\text{valid}}|$ 极小(如仅 1 个 token),温度变化不影响采样结果. 但若合法集合较大且温度极高,模型可能在语义上等价的合法 token 间随机游走,生成冗长或次优表达. 例如,`"status": "active"` 与 `"status": "running"` 均合法,但语义不同,随机选择破坏下游一致性. 

### FSM 死胡同

贪心约束解码的局部最优可能导致全局死胡同: 当前选择的合法 token 使 FSM 进入某个状态后,后续无合法 token 可完成序列. 例如,JSON 数组要求至少一个元素,模型生成 `[` 后,下一步选择 `]`(空数组). 若 Schema 要求 `minItems: 1`,`]` 在当前状态下非法,但若 FSM 状态管理存在 bug,`]` 被误判为合法,后续生成将陷入无接受状态可达的困境. 

该问题在 LL(1) 文法中可通过预计算 FIRST/FOLLOW 集避免,但 JSON Schema 的 `minItems`、`minLength` 等数值约束将文法扩展为上下文相关,超出纯 CFG 的表达力. Outlines 等库对此类约束采用近似处理: 在 CFG 阶段允许空数组,生成后由 Pydantic 校验捕获,触发重试. 该混合策略牺牲了一部分"生成时绝对正确"的承诺. 

## 技术前瞻

**上下文相关文法的约束解码**. 当前 Outlines 基于正则/CFG,无法表达 `"数组长度必须为字段 X 的值"` 等上下文相关约束. 未来方向是将约束分解为两层: CFG 层保证语法,轻量级程序验证层保证上下文语义. 形式化框架为: 

$$
 \mathcal{C} = \mathcal{C}_{\text{CFG}} \cap \mathcal{C}_{\text{context}} \tag{11}
$$
式 (8) 中,$\mathcal{C}$ 为完整约束集合,$\mathcal{C}_{\text{CFG}}$ 为上下文无关语法约束,$\mathcal{C}_{\text{context}}$ 为上下文相关语义约束(如跨字段引用、数值范围). 如式 (8) 所示,完整约束是语法约束与语义约束的交集. Decoder  在 CFG 约束下生成候选,语义约束通过后续过滤或拒绝采样施加. 该两层架构在计算可处理性与表达力之间取得平衡. 

**模型内建结构化能力**. GPT-4o 与 Claude 3.5 的原生结构化输出通过训练数据增强与后训练对齐实现,无需外部约束Decoder  . 数学上,模型学习了条件分布 $P(x_t \mid x_{<t}, \text{Schema})$,将 Schema 作为隐变量内化至网络参数中. 该方向的开源复现需在预训练阶段注入大量结构化数据(JSON/XML/Protocol Buffers),并在 RLHF 阶段对格式错误施加强惩罚. 

**推测性约束解码**. 标准约束解码阻塞每步的 mask 计算. 推测解码(Speculative Decoding)通过小模型草稿+大模型验证的方式加速,但约束解码要求每步验证合法性,与批量化验证存在张力. 未来方向是让小模型在约束条件下生成草稿序列,大模型一次性验证整条序列的 FSM 兼容性,将每步 FSM 查询摊平至序列级别. 

**类型引导的束搜索**. 束搜索(Beam Search)保留 $k$ 条候选序列. 在约束解码框架下,每条候选维护独立的 FSM 状态. 当某条候选进入语义死胡同时(如无合法后续 token),该束被提前剪枝,释放计算资源至其他有希望的候选. 该策略将束搜索与 FSM 状态耦合,可能改善复杂 Schema 下的生成质量.
