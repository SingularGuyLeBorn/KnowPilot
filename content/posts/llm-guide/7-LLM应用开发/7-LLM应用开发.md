## 1. 章节定位与价值

一个拥有 175B 参数的 GPT-3 模型, 如果仅仅通过一个简单的文本框与用户交互, 它的价值是极其有限的. 真正让大语言模型产生巨大商业和社会价值的, 是围绕它构建的**应用层技术栈**——Prompt 工程让模型的输出更加可控和精确, RAG 让模型能够基于私有知识进行回答, Agent 让模型能够自主规划并调用工具完成复杂任务, Function Calling 和 MCP 则为模型与外部世界的交互提供了标准化的接口. 

本章的定位是知识库的"应用架构师手册". 它不关注模型内部的 Transformer 层如何计算, 而是关注**如何将一个训练好的模型"封装"成实际可用的产品**. 通过本章的学习, 你将掌握: 

1. **Prompt 工程的核心范式**: 从基础技巧到高级模式, 学会如何与模型进行高效的"沟通". 

2. **RAG 系统的完整构建**: 从数据准备到检索策略, 从生成优化到评估体系, 搭建一个生产级的知识问答系统. 

3. **Agent 系统的设计原理**: 理解规划、记忆、工具使用三大核心组件, 构建能够自主完成任务的智能体. 

4. **模型与外部世界的交互协议**: 深入理解 Function Calling 的实现机制和 MCP(模型上下文协议)的标准化愿景. 

5. **评估与部署的工程实践**: 建立科学的评估体系, 将应用稳定、高效地部署到生产环境. 

本章的内容组织遵循 LLM 应用开发的**完整 pipeline**: 从最基础的 Prompt 设计开始, 逐步叠加 RAG、Agent、工具调用等能力, 最终形成一个功能完备、可评估、可部署的现代化 AI 应用.

## 2. Prompt 工程: 与模型对话的艺术

Prompt 工程(Prompt Engineering)是 LLM 应用开发的最基础、也是最重要的技能. 它研究的是: 如何通过精心设计的输入文本, 引导模型产生期望的输出. Prompt 工程不需要修改模型参数, 不需要额外的训练数据, 仅仅通过改变输入的方式, 就能显著提升模型的表现. 这种"零成本"的优化方式, 使其成为所有 LLM 应用开发者的必修课. 

### 2.1 基础技巧

**明确指令(Clear Instructions)** 
模型不是读心者. 模糊的请求("写点关于人工智能的东西")会得到模糊的结果. 明确的指令("请用三段话解释深度学习中的反向传播算法, 要求第一段讲原理, 第二段讲计算过程, 第三段讲一个实际应用场景. 目标读者是计算机专业的大二学生. ")才能得到精确的回答. 

**角色设定(Role Prompting)** 
为模型设定一个角色, 可以显著影响其输出的风格、深度和视角. 例如, "你是一位有 20 年经验的资深后端架构师"会让模型在回答技术问题时更加注重工程实践和架构权衡; "你是一位耐心的小学老师"会让模型在解释概念时更加循序渐进、多用类比. 

**少样本提示(Few-Shot Prompting)** 
在请求中提供几个输入-输出示例, 让模型"模仿"示例中的模式. Few-Shot 特别适用于那些难以用自然语言精确描述规则、但可以通过示例传达意图的任务(如特定格式的文本转换、风格迁移、分类任务等). 示例的质量远比数量重要——2-3 个精心挑选的示例, 往往比 10 个随意的示例效果更好. 

**思维链提示(Chain-of-Thought, CoT)** 
在 Prompt 中明确要求模型"一步步思考"("Let's think step by step"), 可以显著提升模型在数学推理、逻辑推理、复杂问题求解等任务上的表现. CoT 的原理是强迫模型将内部的"隐式推理"过程"外显化", 从而减少跳步导致的错误. Self-Consistency CoT 进一步通过生成多条推理路径并投票表决, 进一步提升了推理的准确性. 

### 2.2 高级模式

**ReAct(Reasoning + Acting)** 
ReAct 是一种将推理(Reasoning)与行动(Acting)交织进行的 Prompt 模式. 模型被要求在生成最终答案之前, 先进行多轮的"思考 → 行动 → 观察"循环. 在"思考"步骤中, 模型分析当前状态和下一步需要什么信息; 在"行动"步骤中, 模型调用某个工具(如搜索、计算器、代码执行器); 在"观察"步骤中, 模型接收工具的返回结果, 并据此调整后续的思考. ReAct 是构建简单 Agent 的基础模式. 

**结构化输出(Structured Output)** 
现代 LLM(如 GPT-4、Claude 3.5)支持通过 JSON Schema 约束模型的输出格式, 使其返回结构化的数据(而非自由文本). 这对于需要将模型输出接入下游系统的应用场景至关重要. 例如, 要求模型以 JSON 格式返回提取的实体信息, 每个实体包含 name、type、confidence 字段. 

**系统提示词工程(System Prompt Engineering)** 
系统提示词(System Prompt)是定义模型全局行为的指令, 通常在所有用户消息之前发送. 一个精心设计的系统提示词可以设定模型的安全边界、输出风格、知识范围和行为准则. 系统提示词工程正在成为一门独立的技能——如何编写既不过度限制模型创造力、又能有效防止有害输出的系统提示词.

## 3. RAG: 检索增强生成

RAG(Retrieval-Augmented Generation)是当前 LLM 应用落地的**最重要技术范式**, 没有之一. 它的核心动机非常朴素: 大语言模型的知识是"静态的"(截止于训练数据的时间点)且"泛化的"(对特定领域的细节知识掌握不足). RAG 通过让模型在生成答案之前先检索相关的参考文档, 解决了这两个核心痛点. 

### 3.1 RAG 的核心 Pipeline

一个标准的 RAG 系统包含三个核心阶段: 

**索引阶段(Indexing)** 
将知识源(文档、网页、数据库记录等)转化为可检索的向量表示. 具体流程包括: 
- **文档加载与解析**: 支持 PDF、Word、Markdown、网页等多种格式, 处理表格、图片、代码块等复杂内容. 

- **文本分块(Chunking)** : 将长文档切分成适合 Embedding 模型处理的片段. 分块策略(固定长度、按段落、按语义、递归切分等)直接影响检索质量. 

- **Embedding 生成**: 使用预训练的 Embedding 模型(如 OpenAI text-embedding-3、BGE、E5、GTE 等)将文本块转换为高维向量. 

- **向量存储**: 将向量连同原始文本元数据存入向量数据库(如 FAISS、Chroma、Milvus、Pinecone、Weaviate 等). 

**检索阶段(Retrieval)** 
根据用户查询, 从向量数据库中找到最相关的文本块. 关键考虑因素包括: 
- **Embedding 模型选择**: 不同模型在不同领域和语言上的表现差异显著, 需要针对具体场景选择或微调. 

- **检索算法**: 除了最基础的余弦相似度, 还可以引入稀疏检索(BM25)、混合检索(Dense + Sparse)、多路召回等策略. 

- **重排序(Re-ranking)** : 使用更强大的 cross-encoder 模型对初筛结果进行精排, 显著提升 top-k 结果的精确率. 

- **查询改写(Query Rewriting/Expansion)** : 将用户的原始查询改写为更适合检索的形式(如消除歧义、扩展同义词、分解复杂查询). 

**生成阶段(Generation)** 
将检索到的上下文与用户查询组合成 Prompt, 送入 LLM 生成最终答案. 优化方向包括: 
- **Prompt 模板设计**: 如何组织检索到的文档、如何指示模型基于这些文档回答问题、如何处理文档之间的冲突信息. 

- **引用与溯源(Attribution)** : 要求模型在生成答案时标注信息来源, 提升答案的可信度和可验证性. 

- **拒绝回答机制**: 当检索结果与查询不相关或信息不足时, 模型应该学会说"我不知道", 而不是 hallucinate(编造答案). 

### 3.2 RAG 的进阶优化

**多模态 RAG**
将 RAG 从纯文本扩展到图像、音频、视频等多模态内容. 核心挑战在于: 如何为不同模态的内容生成统一的向量表示？如何处理"跨模态查询"(如用户用文字查询描述一张图片的内容)？

**图 RAG(Graph RAG)** 
Microsoft 提出的 Graph RAG 将传统的"向量相似度检索"升级为"知识图谱检索". 它首先使用 LLM 从文档中提取实体和关系, 构建知识图谱; 在回答问题时, 先在图谱中进行社区发现和摘要生成, 再基于这些结构化信息生成答案. Graph RAG 在处理需要综合多文档信息、进行全局性回答的复杂查询时, 表现优于传统 RAG. 

**自适应 RAG(Adaptive RAG)** 
不是所有查询都需要检索. 自适应 RAG 通过一个小型分类器或 LLM 自身, 先判断一个查询是否需要检索辅助: 简单的事实性问题可以直接由模型回答, 复杂的、时效性强的、或者涉及特定领域知识的查询才触发检索流程. 这减少了不必要的检索开销, 提升了响应速度.

## 4. Agent: 自主规划与行动

如果说 RAG 是"给模型一本参考书, 让它开卷考试", 那么 Agent 就是"给模型一个目标, 让它自己想办法完成". Agent(智能体)代表了大语言模型应用的**最前沿形态**, 它的核心特征是**自主性**——能够根据目标进行规划、调用工具、观察环境反馈、并迭代调整策略, 直到任务完成. 

### 4.1 Agent 的核心组件

**规划(Planning)** 
规划是 Agent 的"大脑", 负责将复杂的目标分解为可执行的步骤. 规划的方法包括: 
- **单路径规划**: 如 Chain-of-Thought、ReAct, 按顺序执行一系列步骤. 

- **多路径规划**: 如 Tree-of-Thought(ToT), 在每一步探索多个可能的行动分支, 通过评估选择最优路径. 

- **分层规划**: 如 Hierarchical Task Network(HTN), 将高层目标递归分解为子目标, 直到达到原子动作. 

- **反思与修正**: 如 Reflexion、Self-Refine, 让 Agent 在执行过程中评估自己的表现, 识别错误并进行修正. 

**记忆(Memory)** 
记忆让 Agent 能够利用历史信息来指导当前决策. 记忆通常分为: 
- **短期记忆(Short-term Memory)** : 当前对话的上下文, 通常通过滑动窗口或摘要机制管理. 

- **长期记忆(Long-term Memory)** : 跨越多次对话的持久化知识, 通常以向量数据库(存储语义记忆)或图数据库(存储关系记忆)的形式存在. 

- **工作记忆(Working Memory)** : 当前任务相关的临时信息, 如正在处理的中间结果、需要关注的约束条件等. 

**工具使用(Tool Use)** 
工具是 Agent 与外部世界交互的"双手". 常见的工具类型包括: 
- **信息检索工具**: 搜索引擎、数据库查询、API 调用. 

- **计算工具**: 计算器、代码解释器(Python REPL)、数学求解器. 

- **执行工具**: 文件系统操作、邮件发送、日历管理、代码部署. 

- **感知工具**: 图像识别、语音识别、传感器数据读取. 

### 4.2 多 Agent 系统

当单个 Agent 的能力不足以完成复杂任务时, 可以构建多 Agent 协作系统. 在这种架构中, 不同的 Agent 扮演不同的角色(如项目经理、程序员、测试员、设计师), 通过消息传递进行协作. 多 Agent 系统的核心设计挑战包括: 

- **角色定义与分工**: 如何划分任务边界, 避免重复工作或遗漏. 

- **通信协议**: Agent 之间如何交换信息——是直接传递结构化数据, 还是通过自然语言对话？

- **冲突解决**: 当不同 Agent 的结论冲突时, 如何进行仲裁和整合. 

- **人机协作**: 在什么节点引入人类的监督和确认, 以平衡自动化程度和风险控制. 

代表性的多 Agent 框架包括 AutoGen(Microsoft)、CrewAI、LangGraph 等.

## 5. Function Calling 与 MCP: 模型与外部世界的标准化接口

Agent 需要调用工具, 但"如何调用"这个问题的答案一直在演变. Function Calling 和 MCP 代表了这一演变的两个重要里程碑. 

### 5.1 Function Calling

Function Calling(函数调用/工具调用)是由 OpenAI 在 GPT-4 中引入的机制, 它允许模型在需要时输出一个结构化的函数调用请求(包含函数名和参数), 而非直接的文本回复. 应用程序接收到这个请求后, 执行对应的函数, 并将结果返回给模型, 模型再基于结果继续生成回复. 

**核心机制**
Function Calling 的实现依赖于模型在训练阶段学习到的"模式识别"能力: 模型学会了在特定情境下(如用户询问天气、需要进行计算、需要查询数据库)生成特定格式的 JSON 对象, 而不是自由文本. 这个 JSON 对象遵循预定义的 Schema, 包含 `name`(函数名)和 `arguments`(参数键值对)两个字段. 

**工程实践要点**
- **Schema 设计**: 函数定义的 Schema 直接影响模型调用的准确性. 参数的描述(description)必须清晰、无歧义, 枚举类型(enum)比开放式字符串更可靠. 

- **错误处理**: 模型可能调用不存在的函数、传入错误的参数类型、或者在不需要调用函数时强行调用. 应用层必须有健壮的错误处理和重试机制. 

- **安全性**: 模型调用的函数可能具有副作用(如发送邮件、删除文件、转账). 必须对函数权限进行严格的沙箱隔离和人工确认. 

### 5.2 MCP(模型上下文协议)

MCP(Model Context Protocol)是 Anthropic 在 2024 年底推出的**开放协议标准**, 旨在为 AI 模型与外部数据源、工具之间的连接提供一个统一的、可扩展的接口. 如果说 Function Calling 是"每个模型厂商自己定义的一套工具调用格式", 那么 MCP 就是"一套跨厂商的标准化协议". 

**MCP 的核心架构**
MCP 采用客户端-服务器架构: 
- **MCP 服务器**: 封装了特定的数据源或工具(如本地文件系统、GitHub API、Slack、数据库), 通过标准化的协议暴露其能力. 

- **MCP 客户端**: 集成在 AI 应用(如 Claude Desktop、Cursor、各类 Agent 框架)中, 负责与 MCP 服务器通信, 将服务器的"工具声明"转化为模型可理解的 Function Calling Schema. 

**MCP 的价值**
- **解耦**: 工具开发者只需要按照 MCP 协议实现一次, 就可以被所有支持 MCP 的客户端使用, 无需为每个应用单独适配. 

- **可发现性**: MCP 服务器通过协议自动声明自己提供哪些工具、每个工具的参数 Schema 是什么, 客户端可以动态发现和调用. 

- **安全性**: MCP 协议内置了权限控制和本地执行机制, 敏感操作(如访问本地文件)在本地沙箱中执行, 降低了安全风险. 

MCP 正在快速成为 LLM 应用生态系统中的"USB-C 接口"——统一的、开放的、即插即用的连接标准.

## 6. 评估与部署

### 6.1 评估体系

"你无法优化你无法衡量的东西. "建立科学的评估体系是 LLM 应用开发的关键环节. 

**RAG 评估**
RAG 系统的评估通常关注三个维度: 
- **检索质量**: 检索到的文档是否与查询相关？(Recall@K、Precision@K、MRR、NDCG)
- **生成质量**: 生成的答案是否准确、连贯、与检索到的文档一致？(Faithfulness、Answer Relevance)
- **端到端质量**: 系统作为一个整体是否满足用户需求？(用户满意度、任务完成率)

常用的 RAG 评估框架包括 RAGAS、ARES、TruLens 等, 它们使用 LLM 作为评判者(LLM-as-a-Judge)来自动化评估过程. 

**Agent 评估**
Agent 的评估更加复杂, 因为它涉及多步骤的决策和工具调用. 评估维度包括: 
- **任务完成率**: Agent 是否成功完成了给定的目标？

- **效率**: 完成任务所需的步骤数、工具调用次数、Token 消耗量. 

- **正确性**: 每一步的决策是否正确？工具调用的参数是否准确？

- **鲁棒性**: 面对错误信息、工具故障、模糊指令时, Agent 的恢复能力如何？

**人工评估与自动评估的平衡**
自动化评估(如使用 GPT-4 作为评判者)可以大规模、低成本地运行, 但可能存在评判偏差(Judge Bias). 人工评估更可靠, 但成本高昂. 实践中通常采用"自动评估筛选 + 人工评估验证"的组合策略. 

### 6.2 部署实践

**架构模式**
- **直接调用 API**: 最简单的方式, 适用于原型验证和轻量级应用. 直接使用 OpenAI、Anthropic、Google 等厂商提供的 API. 

- **自托管开源模型**: 使用 vLLM、SGLang、TensorRT-LLM 等引擎在自有 GPU 上部署开源模型. 适用于数据隐私要求高、或者需要精细控制推理行为的场景. 

- **混合架构**: 对通用查询调用商业 API, 对敏感/特定领域查询调用本地部署的模型. 通过路由层(Router)根据查询特征进行分发. 

**生产环境 checklist**
- [ ] **输入输出过滤**: 部署内容安全过滤器, 防止有害输入和输出的传播. 
- [ ] **速率限制与配额管理**: 防止恶意用户或 bug 导致 API 费用爆炸. 
- [ ] **可观测性**: 记录请求日志、延迟指标、错误率、Token 消耗量, 设置告警阈值. 
- [ ] **A/B 测试与灰度发布**: 新模型或新 Prompt 上线前, 先在小流量上验证效果. 
- [ ] **回滚机制**: 当线上模型出现严重问题时, 能够快速切换回稳定版本. 
- [ ] **成本监控**: 实时跟踪推理成本, 识别异常消耗并优化.
