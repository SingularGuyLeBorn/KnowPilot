# 7.2 · RAG 检索增强生成: 从 Dense Retrieval 到 Agentic RAG

## 1. 背景与核心痛点 (Background & Pain Points)

### 1.1 家谱定位: LLM 应用范式的三阶演进

大语言模型自 GPT-3 起掀起了一场认知革命, 但当我们试图将其从"聊天玩具"推向"生产力工具"时, 四座无形的大山横亘在前. **检索增强生成(Retrieval-Augmented Generation, RAG)** 正是为了翻越这四座大山而被提出的第三代应用范式. 

让我们先快速回顾 LLM 应用范式的三阶演进, 以定位 RAG 在技术家谱中的坐标: 

- **第一代: Zero-Shot / Few-Shot Prompting(2020-2022)** . 研究者发现, 通过在 prompt 中给出几个示例(in-context learning), 模型就能完成从未见过的任务. 这极其优雅, 但完全受限于模型预训练时"见过什么". 一旦问题涉及训练数据截止日期之后的事件, 或者极其细分的垂直领域知识, 模型就会开始"一本正经地胡说八道". 

- **第二代: Fine-Tuning(2022-2023)** . 通过监督微调(SFT)或参数高效微调(LoRA、QLoRA、Prefix Tuning 等), 将领域知识"烧录"进模型参数. Fine-Tuning 确实能让模型掌握特定风格和专业术语, 但它有三个致命缺陷: 一是**成本高昂**, 每次知识更新都需要重新训练; 二是**知识更新滞后**, 无法实时反映最新信息; 三是**可解释性极差**, 你无法追问模型"这个结论是从哪里来的", 因为知识已经以分布式的方式固化在数十亿参数的权重矩阵中. 

- **第三代: RAG(2023-至今)** . RAG 的核心理念极其简单却极其深刻: **不要让模型去死记硬背外部知识, 而是让模型在需要时去"查资料"**. 模型本身只负责理解问题和综合信息, 而知识的存储、更新、检索被解耦到外部系统中. 这种架构上的解耦, 从根本上解决了前三代范式的核心痛点. 

**核心动机**: RAG 的诞生并非为了替代 Fine-Tuning, 而是为了在"知识密集型任务"这一特定战场上, 提供一种成本更低、更新更快、来源可溯的替代方案. 

### 1.2 痛点一: 幻觉(Hallucination)——最危险的敌人

LLM 的幻觉问题是指模型会生成看似语法通顺、逻辑连贯, 但实际上与事实严重不符的内容. 这种错误不是简单的"不知道", 而是"自信满满地胡说". 

幻觉产生的数学根源在于语言模型的训练目标. 自回归语言模型通过最大化训练数据中的条件概率来优化: 

$$
 \mathcal{L}_{LM} = - \sum_{t=1}^{T} \log P(w_t | w_1, w_2, \ldots, w_{t-1}; \theta) \tag{1}
$$

注意, 这个目标函数**从未要求模型输出"真"的东西**, 它只要求模型输出"像训练数据"的东西. 当模型遇到一个它在训练集中从未见过或记忆模糊的事实时, 为了维持高概率输出, 它会利用其在数十亿参数中学到的"语法模式和统计关联"去"拼凑"一个听起来最合理的答案. 这个拼凑过程没有任何事实校验机制. 

在医疗、法律、金融等高风险领域, 一个关于药物剂量或法规条款的幻觉可能带来灾难性后果. 2023 年, 美国一位律师使用 ChatGPT 撰写法庭文书, 结果引用了多个根本不存在的判例, 导致被法官处罚——这就是 RAG 要解决的现实问题的极端写照. 

### 1.3 痛点二: 知识截止日期(Knowledge Cutoff)——时间的囚徒

每个预训练模型都有一个隐形的"知识截止日期". GPT-4 的知识截止到 2023 年 4 月, Claude 3 截止到 2024 年初. 这意味着模型对训练数据截止日期之后发生的任何事情都一无所知. 

这个问题的严重性正在指数级增长. 在 2024-2025 年, AI 领域本身的发展速度就使得任何超过三个月的知识都可能过时. 一个关于"最新发布的大模型性能对比"的问题, 对于知识截止在半年前的模型来说, 是完全无法回答的. 

Fine-Tuning 看似能解决这一问题, 但代价是: 每次世界发生新事件, 你都需要收集新数据、重新训练模型、重新部署服务. 对于需要实时股价、实时新闻、实时产品库存的企业应用来说, 这是不可接受的. 

### 1.4 痛点三: 领域专业知识的缺失(Domain Gap)——通才的盲区

通用大模型在海量互联网文本上训练, 掌握了广泛但浅薄的知识. 然而, 当你向它询问企业内部的私有文档(如某公司的内部 API 文档、医院的患者病历、律所的保密卷宗)时, 模型完全无能为力. 这些私有数据从未出现在公开训练集中, 也不可能被泄露到预训练模型中. 

即便对于公开的专业领域, 通用模型也常常表现不佳. 例如, GPT-4 在回答中国最新税法细则、某小众编程语言的最新版本特性、或者某个冷门学术领域的最新论文结论时, 准确率会急剧下降. 这不是因为模型"不够聪明", 而是因为预训练数据的分布在长尾领域天然稀疏. 

### 1.5 痛点四: 可溯源性与可验证性(Verifiability)——信任的基石

在企业级部署中, "模型为什么给出这个答案"是一个不可回避的问题. 当模型通过 Fine-Tuning 将知识内化到参数中时, 它的决策过程是一个黑盒. 你无法向客户或监管机构证明"这个答案是有依据的". 

RAG 通过将生成过程建立在**显式检索到的文档片段**之上, 从根本上改变了这一局面. 如果模型在回答时明确标注了"根据文档 A 的第 3 节和文档 B 的第 7 节", 用户和审计人员就可以直接检查这些原始出处. 这种可溯源性不是锦上添花的功能, 而是 LLM 从"玩具"走向"关键基础设施"的入场券.

## 2. 为什么重要 (Significance)

### 2.1 RAG: 2023-2026 年企业级 LLM 应用的事实标准

RAG 的重要性不需要抽象的论证, 只需要看工业界的实际部署数据即可说明问题. 

根据 LangChain 2024 年度状态报告, 在对全球超过 1,300 名开发者的调研中, **超过 78% 的生产级 LLM 应用采用了某种形式的 RAG 架构**. 在 NVIDIA 2024 年的 RAG 基准测试白皮书中, RAG 被明确定义为"企业从原型验证迈向生产部署的必经桥梁". 微软的 Copilot、Google 的 NotebookLM、以及几乎所有企业内部的"知识问答助手", 其底层无一例外都是 RAG 的变体. 

为什么 RAG 能在如此短的时间内成为事实标准？因为它精准地击中了企业部署 LLM 时的"不可能三角": **成本、准确性、实时性**. 企业既希望获得 LLM 强大的语言理解和生成能力, 又不愿承受幻觉风险, 还需要系统能实时反映最新的内部文档和政策变化. RAG 以极低的额外成本(主要是向量数据库的存储和检索开销)同时满足了这三个需求. 

### 2.2 RAG vs Fine-Tuning: 成本、更新速度与可解释性的三重胜利

很多初学者会问: 既然想让模型掌握新知识, 为什么不直接 Fine-Tune 模型？让我们从三个维度进行严格对比: 

**第一维度: 经济成本**. Fine-Tuning 一个 70B 参数模型, 即使是使用 QLoRA 进行参数高效微调, 也需要至少一张 A100 80GB 显卡运行数小时, 每次知识更新都要重复这一过程. 而 RAG 的"知识更新"仅仅是将新文档切分、计算 embedding、写入向量数据库——这个过程可以在 CPU 上完成, 成本下降了两个数量级以上. 对于拥有百万级文档的企业知识库, Fine-Tuning 的成本是完全不可接受的. 

**第二维度: 更新速度**. Fine-Tuning 的更新周期以"天"或"周"为单位(数据收集、训练、评估、部署). RAG 的更新周期以"秒"或"分钟"为单位——一篇新文档发布, 其 embedding 可以在写入数据库后的毫秒级被检索到. 对于新闻摘要、实时客服、股价分析等对时效性要求极高的场景, RAG 是唯一可行的方案. 

**第三维度: 可解释性与可控性**. Fine-Tuned 模型的知识是隐式的, 你无法知道某个输出是基于哪条训练数据. 如果模型输出了错误信息, 你只能重新检查整个训练集并重新训练. 而 RAG 的每一步都是显式的: 检索到了哪些文档、相似度分数是多少、哪些文档被用于生成答案——全部可审计、可干预. 如果某篇文档含有错误信息, 你只需将其从向量数据库中删除或更新, 问题立即解决, 无需重新训练任何模型. 

| 维度 | Fine-Tuning | RAG |
|------|------------|-----|
| 训练成本 | 高(GPU 小时级) | 极低(CPU 分钟级) |
| 知识更新延迟 | 天/周级 | 秒/分钟级 |
| 知识来源可溯 | 不可溯(黑盒参数) | 完全可溯(显式文档) |
| 幻觉控制能力 | 弱(知识固化在参数中) | 强(基于检索上下文生成) |
| 适合场景 | 风格迁移、行为对齐 | 知识问答、实时信息检索 |

**需要强调的是, RAG 和 Fine-Tuning 并非互斥关系**. 在工业界的最佳实践中, 往往采用"RAG + Light Fine-Tuning"的混合策略: 用 RAG 提供外部知识, 用轻量级 Fine-Tuning 调整模型的回答风格、输出格式和特定行为的偏好. 这种组合策略兼顾了灵活性、准确性和成本效益, 代表了当前 LLM 应用工程的最优解. 

### 2.3 学术研究的前沿焦点

RAG 不仅是工程界的宠儿, 也是学术研究最活跃的领域之一. 从 2020 年 Facebook AI 提出 ORIGINAL RAG 论文(Lewis et al., *Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks*)至今, arXiv 上以"Retrieval-Augmented"为关键词的论文数量呈指数增长. 研究领域从基础的 Dense Retrieval, 一路演进至 Self-RAG、Corrective RAG、Speculative RAG、GraphRAG、Agentic RAG 等前沿变体. 每一代变体都在解决前一代在特定边界条件下的退化问题. 

可以毫不夸张地说, **理解 RAG 就是理解当前 LLM 应用工程的核心骨架**. 无论后续技术如何演进, "检索 + 生成"这一基本范式在可预见的未来内都将保持不变.

## 3. 核心架构与数学 (Mathematical Rigor)

### 3.1 基础 RAG Pipeline: 索引、检索、生成的三步舞

一个标准的 RAG 系统由三个连续阶段构成: 索引(Indexing)、检索(Retrieval)和生成(Generation). 让我们用数学语言精确描述这个 pipeline. 

#### 3.1.1 索引阶段: 将非结构化知识转化为可检索的向量空间

索引阶段的目标是将原始文档集合 $\mathcal{D} = \{d_1, d_2, \ldots, d_N\}$ 转化为一个支持高效相似度搜索的数据结构. 

**步骤 1: 文档切分(Chunking)** 

原始文档通常太长, 无法直接 embedding. 我们需要将文档切分为语义相对独立的片段(chunks): 

$$
 d_i \rightarrow \{c_{i,1}, c_{i,2}, \ldots, c_{i,m_i}\} \tag{2}
$$
其中每个 chunk $c_{i,j}$ 是一个文本片段, 通常长度在 100-1000 个 token 之间. 切分策略的选择极其关键, 我们将在 3.2 节详细讨论. 

**步骤 2: Embedding 编码**

使用一个预训练的 embedding 模型(bi-encoder)将每个 chunk 编码为稠密向量: 

$$
 \mathbf{e}_{i,j} = \text{Embed}(c_{i,j}) \in \mathbb{R}^d \tag{3}
$$

其中 $d$ 是向量维度, 常见取值有 384、768、1024、1536 或 4096. embedding 模型的选择直接决定了语义空间的构造质量. 

**步骤 3: 向量存储与索引构建**

将所有 chunk 向量存入向量数据库(Vector Store), 并构建近似最近邻(Approximate Nearest Neighbor, ANN)索引: 

$$
 \mathcal{V} = \{ (\mathbf{e}_{i,j}, \text{meta}_{i,j}) \}_{i,j} \tag{4}
$$
其中 $\text{meta}_{i,j}$ 包含原始文本、来源文档 ID、页码等元数据, 用于生成阶段的上下文拼接和答案溯源. 

#### 3.1.2 检索阶段: 从向量空间中召回相关信息

给定用户查询 $q$, 检索阶段的目标是找到与查询最相关的 $k$ 个文档 chunks. 

**步骤 1: 查询编码**

将查询通过同一个 embedding 模型编码为查询向量: 

$$
 \mathbf{q} = \text{Embed}(q) \in \mathbb{R}^d \tag{5}
$$

注意, query 和 document 使用**同一个 bi-encoder** 进行编码. 这是 dense retrieval 的核心设计. 

**步骤 2: 相似度计算与 Top-k 检索**

计算查询向量与所有文档 chunk 向量的相似度. 最常用的度量是余弦相似度: 

$$
 \text{sim}(\mathbf{q}, \mathbf{e}_{i,j}) = \frac{\mathbf{q} \cdot \mathbf{e}_{i,j}}{\|\mathbf{q}\| \|\mathbf{e}_{i,j}\|} = \cos\theta \tag{6}
$$
余弦相似度的取值范围是 $[-1, 1]$, 但由于 embedding 模型通常输出非负值(或在归一化后), 实际范围更接近 $[0, 1]$. 值越接近 1, 表示语义越相关. 

然后返回相似度最高的 $k$ 个 chunks: 

$$
 \mathcal{R}_k(q) = \text{Top-}k_{(i,j)} \, \text{sim}(\mathbf{q}, \mathbf{e}_{i,j}) \tag{7}
$$

这里的 Top-k 操作由 ANN 索引(如 FAISS、Milvus、Pinecone 等)高效完成, 时间复杂度从暴力搜索的 $O(Nd)$ 降低到 $O(d \log N)$ 甚至更低. 

#### 3.1.3 生成阶段: 让 LLM 在检索证据上综合作答

检索到的 chunks 被组织成上下文(context), 与用户原始查询一起输入 LLM: 

$$
 \text{Context} = \text{Concat}(c_{r_1}, c_{r_2}, \ldots, c_{r_k}) \tag{8}
$$
此式描述了变量之间的定量关系, 其物理意义将在下文详细阐述. 
然后构造 prompt: 

```
基于以下参考信息回答问题. 如果参考信息不足以回答, 请明确说明. 

[参考信息]
{Context}

[问题]
{q}

[回答]
```

在 RAG 的生成阶段, LLM 将检索到的 Top-k chunks 拼接为外部上下文, 与用户查询一并输入模型. LLM 基于条件概率逐 token 生成答案: 

$$
 P(a | q, \text{Context}; \theta_{LLM}) = \prod_{t=1}^{T} P(a_t | q, \text{Context}, a_1, \ldots, a_{t-1}; \theta_{LLM}) \tag{9}
$$

此式将答案生成建模为在查询和检索上下文条件下的自回归序列分解. 这里的关键在于, LLM 的生成不再仅依赖于其预训练知识, 而是**显式地以检索到的外部上下文为条件**. 只要检索到的上下文是准确的, 即使 LLM 原本不知道答案, 也能基于上下文正确回答. 这一机制使得 RAG 系统在知识更新频率高、领域专业性强的场景中(如医疗诊断、法律咨询)具有显著优势.

### 3.2 文档切分策略: 语义完整性与检索精度的博弈

文档切分(Chunking)是 RAG 系统中**最容易被低估、却对最终效果影响最大**的环节之一. 一个糟糕的切分策略可以在瞬间毁掉整个系统的检索质量. 

#### 3.2.1 固定长度切分(Fixed-Size Chunking)

最简单粗暴的方法: 按照固定的 token 数或字符数切分文档, 每 chunk 长度为 $L$(例如 512 tokens), 相邻 chunks 之间可有 $O$ 个 token 的重叠(overlap). 

$$
 c_j = d[ j \cdot (L - O) : j \cdot (L - O) + L ] \tag{10}
$$
**优点**: 实现简单, 计算速度快, 每个 chunk 的长度均匀可控, 便于 batch embedding. 

**致命缺陷**: **语义断裂**. 假设文档中有一段话: "Transformer 架构的核心是 self-attention 机制. Self-attention 允许模型在处理序列时同时关注所有位置的信息. "如果切分点恰好落在两句之间, 前半句在一个 chunk 中, 后半句在下一个 chunk 中, 那么检索到前半句的用户将只看到"Transformer 架构的核心是 self-attention 机制", 而丢失了最关键的"允许模型同时关注所有位置"这一解释. 更糟的是, 如果查询是"self-attention 的作用是什么？", 包含"self-attention 允许模型……"的那个 chunk 可能因为缺少了"Transformer 的核心是"这一前置语境, 而与查询的语义匹配度降低, 导致检索失败. 

#### 3.2.2 递归切分(Recursive Chunking)

递归切分是一种层次化的切分策略, 优先尝试在更大的语义边界上切分, 如果 chunk 仍然太大, 则在更细的粒度上继续切分: 

1. **第一层**: 按段落(由空行分隔)切分. 

2. **第二层**: 如果某段落仍超过目标长度, 按句子(由句号、问号、感叹号分隔)切分. 

3. **第三层**: 如果某句子仍超过目标长度, 按固定长度切分. 

$$
 \text{Split}(\text{text}, \text{separators}=["\\n\\n", "\\n", ".", " "], \text{chunk_size}=L) \tag{11}
$$

递归切分保持了更高层次的语义完整性: 它优先保证段落级别和句子级别的完整性, 只在迫不得已时才在词语中间切断. LangChain 的 `RecursiveCharacterTextSplitter` 就是这一策略的经典实现. 

#### 3.2.3 语义切分(Semantic Chunking)

递归切分虽然比固定长度切分更尊重自然边界, 但它仍然是**基于启发式规则**的, 没有真正"理解"文本内容. 语义切分则更进一步: 它使用 embedding 模型来判断切分点两侧的文本语义是否发生了显著变化. 

具体做法: 将文本按句子切分后, 计算相邻句子对的 embedding 相似度: 

$$
 s_i = \text{sim}(\text{Embed}(\text{sent}_i), \text{Embed}(\text{sent}_{i+1})) \tag{12}
$$
如果 $s_i < \tau$(某个阈值), 说明句子 $i$ 和句子 $i+1$ 之间的语义发生了"跳变", 这里就是一个理想的切分点. 

$$
 \text{Boundary}_i = \mathbb{1}[s_i < \tau] \tag{13}
$$

语义切分试图在"语义低洼地带"(两个主题之间的过渡区域)下刀, 使得每个 chunk 内部的语义高度一致, 而不同 chunks 之间的语义边界清晰. 这种方法特别适合处理结构松散的长文档(如访谈记录、会议转录). 

#### 3.2.4 Agentic 切分与智能边界检测

更前沿的切分方法甚至使用 LLM 本身来判断切分策略. 例如, 你可以先将文档输入 LLM, 让它输出一个结构化的目录大纲; 然后按照大纲的层级结构进行切分. 这种方法虽然计算成本高, 但对于需要极致质量的场景(如法律合同分析、医学文献检索), 其收益是巨大的. 

#### 3.2.5 最优 Chunk Size 的权衡公式

Chunk size 的选择不是越大越好, 也不是越小越好, 而是一个多目标优化问题. 设 chunk size 为 $L$, 我们需要在以下三个目标之间权衡: 

**目标 1: 检索精度**. 较小的 chunks 通常具有更聚焦的语义, 与查询的匹配更精准. 如果 chunk 太大, 其中会混入大量与查询无关的信息, 导致 embedding 的"语义中心"被稀释, 相似度计算时被无关内容"拉偏". 

**目标 2: 上下文完整性**. 较大的 chunks 保留了更多的上下文, 使得 LLM 在生成答案时能看到更完整的背景信息. 如果 chunk 太小, LLM 可能只看到"因为 A, 所以 B"中的"所以 B", 却不知道"A"是什么. 

**目标 3: Embedding 模型的上下文限制**. 不同的 embedding 模型有不同的最大输入长度. 例如, OpenAI 的 `text-embedding-3-small` 支持 8191 tokens, 而 sentence-transformers 的一些轻量模型只支持 256 tokens. 超过长度限制的文本会被截断, 导致信息丢失. 

**经验法则**: 对于大多数通用场景, chunk size 在 **200-500 tokens** 之间是一个良好的起点, overlap 设置为 chunk size 的 **10%-20%**. 对于需要精确事实检索的场景(如客服 FAQ), 可以减小到 **100-200 tokens**; 对于需要理解长篇论述的场景(如学术论文综述), 可以增大到 **1000+ tokens**, 但此时必须配合重排序(Reranking)和后检索压缩技术.

### 3.3 Embedding 与检索: Dense、Sparse 与 Hybrid 的三重奏

检索是 RAG 的"心脏". 检索质量的上限直接决定了整个系统的回答质量上限. 当前的主流检索方法分为三大类: Dense Retrieval、Sparse Retrieval 和 Hybrid Retrieval. 

#### 3.3.1 Dense Retrieval: Bi-Encoder 与语义空间

Dense Retrieval 的核心思想是将查询和文档映射到同一个低维稠密向量空间中, 然后通过向量相似度来衡量语义相关性. 

**架构: Bi-Encoder**

Bi-Encoder 由**两个共享参数**的Encoder 组成(实际上是同一个模型分别编码 query 和 document): 

$$
 \mathbf{q} = f_\theta(q), \quad \mathbf{d} = f_\theta(d) \tag{14}
$$
注意 query 和 document 是**独立编码**的——document 的编码不依赖于 query, 反之亦然. 这使得文档向量可以**离线预计算**并存储, 只有查询向量需要在线计算. 这种"解耦编码"是 dense retrieval 能够高效处理大规模文档集的关键. 

**相似度计算**

在检索阶段, 相似度通过向量内积或余弦相似度计算: 

向量内积直接计算 query 和 document 的逐元素乘积之和, 值越大表示语义越相似: 

$$
 \text{sim}(q, d) = \mathbf{q}^\top \mathbf{d} \quad \text{(dot product)} \tag{15}
$$

若 embedding 向量已经做了 L2 归一化, 内积与余弦相似度等价. 对于未归一化的向量, 余弦相似度通过除以模长消除向量尺度差异, 只保留方向信息: 

$$
 \text{sim}(q, d) = \frac{\mathbf{q}^\top \mathbf{d}}{\|\mathbf{q}\| \|\mathbf{d}\|} \quad \text{(cosine similarity)} \tag{16}
$$

当 $\|\mathbf{q}\| = \|\mathbf{d}\| = 1$ 时, 两者输出完全一致. 余弦相似度的优势在于对向量的绝对尺度不敏感, 只关心方向的一致性. 

训练良好的 embedding 模型会将语义相似的文本映射到向量空间中彼此靠近的区域. 例如, "如何训练神经网络？"和"深度学习模型优化方法"在向量空间中的距离会很近, 尽管它们几乎没有共享词汇. 这种"语义层面的泛化"是 dense retrieval 相比传统关键词检索的最大优势. 

**训练目标**: 对比学习(Contrastive Learning)

Embedding 模型通常通过对比学习来训练. 对于每个查询 $q$, 存在一个正样本文档 $d^+$(相关的文档)和多个负样本文档 $d^-_1, d^-_2, \ldots$(不相关的文档). 训练目标是让正样本的相似度远高于负样本: 

$$
 \mathcal{L}_{\text{contrastive}} = - \log \frac{\exp(\text{sim}(q, d^+) / \tau)}{\exp(\text{sim}(q, d^+) / \tau) + \sum_{i} \exp(\text{sim}(q, d^-_i) / \tau)} \tag{17}
$$

其中 $\tau$ 是温度系数, 控制分布的尖锐程度. 分母中的求和包含了一个批次(batch)中的所有负样本. 这个 loss 就是 InfoNCE, 它直接优化"在候选集中正确识别出正样本"的能力, 与检索任务的目标完全一致. 

#### 3.3.2 Sparse Retrieval: BM25 与 TF-IDF 的复兴

在深度学习兴起之前, 信息检索领域长期由基于词项频率的稀疏方法统治. 这些方法至今仍在 RAG 中扮演重要角色, 尤其是在处理包含罕见专有名词的查询时. 

**TF-IDF**

TF-IDF(Term Frequency-Inverse Document Frequency)为每个文档-词项对赋予一个权重: 

$$
 \text{TF-IDF}(t, d) = \text{tf}(t, d) \times \text{idf}(t) \tag{18}
$$
其中: 
- $\text{tf}(t, d)$ 是词项 $t$ 在文档 $d$ 中的频率(通常取对数平滑: $\log(1 + \text{count}(t, d))$). 
- $\text{idf}(t) = \log \frac{N}{|\{d : t \in d\}|}$ 是逆文档频率, 衡量词项的区分度. 常见词(如"的"、"the")出现在几乎所有文档中, idf 值极低; 罕见专有名词只出现在少数文档中, idf 值很高. 

文档和查询都表示为稀疏的高维向量(维度等于词汇表大小), 然后计算余弦相似度. 

**BM25**

BM25(Best Match 25)是 TF-IDF 的进阶版本, 引入了对文档长度和词项饱和效应的修正: 

$$
 \text{BM25}(q, d) = \sum_{t \in q} \text{idf}(t) \cdot \frac{\text{tf}(t, d) \cdot (k_1 + 1)}{\text{tf}(t, d) + k_1 \cdot \left(1 - b + b \cdot \frac{|d|}{\text{avgdl}}\right)} \tag{19}
$$

让我们逐项拆解这个公式: 
- $\text{idf}(t)$: 同 TF-IDF, 衡量词项的稀缺性和区分度. 
- $\text{tf}(t, d)$: 词项在文档中的原始频率. 
- $k_1$: 控制词频饱和的参数. 当 $k_1 \approx 1.2$ 时, 一个词在文档中出现 10 次和 100 次的得分差异会趋于平缓, 避免了"堆砌关键词"就能高排名的弊端. 
- $b$: 控制文档长度归一化的参数(通常设为 0.75). $|d|$ 是当前文档长度, $\text{avgdl}$ 是平均文档长度. 
- $\frac{|d|}{\text{avgdl}}$: 文档长度归一化因子. 长文档天然更容易包含更多查询词, 这一项对长文档进行惩罚, 确保短而精的文档不会被埋没. 

**BM25 的物理意义**: BM25 本质上是在问"查询中的每个词在文档中出现的频率有多高？这些词在其他文档中是否罕见？文档是不是过长导致这些词的出现不那么有含金量？"这三个问题的加权组合. 

**Dense vs Sparse 的互补性**: 
- Dense retrieval 擅长捕捉**语义相关性**(如"神经网络"和"深度学习模型"). 
- Sparse retrieval 擅长捕捉**词汇精确匹配**(如产品型号"RTX-4090"、人名"Yoshua Bengio"、错误代码"0x80070005"). 

当一个查询同时包含语义概念和精确词汇时(如"Yoshua Bengio 提出的神经网络架构是什么？"), dense retrieval 可能遗漏包含精确人名但表述不同的文档, 而 sparse retrieval 可能遗漏用不同词汇表达同一概念的文档. 这就是为什么我们需要 Hybrid Retrieval. 

#### 3.3.3 Hybrid Retrieval: 融合两种世界的力量

Hybrid Retrieval 将 dense retrieval 和 sparse retrieval 的结果进行融合, 兼顾语义泛化和词汇精确匹配的优势. 

**线性加权融合**

最简单的方法是对两个检索系统的分数进行线性加权: 

$$
 \text{score}_{\text{hybrid}}(q, d) = \alpha \cdot \text{score}_{\text{dense}}(q, d) + (1 - \alpha) \cdot \text{score}_{\text{sparse}}(q, d) \tag{20}
$$
其中 $\alpha \in [0, 1]$ 是融合权重, 通常通过验证集调优(常见取值在 0.5-0.8 之间, 取决于任务对语义匹配 vs 词汇匹配的依赖程度). 

**RRF(Reciprocal Rank Fusion)** 

更鲁棒的方法是不直接融合分数(因为两种检索系统的分数尺度不同, 不可比), 而是融合排名(rank): 

$$
 \text{RRF}(d) = \sum_{r \in R} \frac{1}{k + r(d)} \tag{21}
$$

其中 $R$ 是所有检索系统的排名结果集合, $r(d)$ 是文档 $d$ 在某个检索系统中的排名(从 1 开始), $k$ 是一个常数(通常取 60, 用于平滑低排名的影响). RRF 的优势在于它完全不受原始分数尺度的影响, 只关心"排第几名", 对不同检索系统的异构输出极其鲁棒. 

#### 3.3.4 重排序(Rerank): Cross-Encoder 的精排艺术

无论是 dense retrieval 还是 sparse retrieval, 它们都必须面对一个工程约束: **检索需要极快的速度**(毫秒级), 因此不能使用过于复杂的模型. 这导致初步检索(retrieval)的结果往往是"还不错, 但不够精确"的. 

重排序(Reranking)就是解决这个问题的: 先用快速的 bi-encoder 召回前 $K$ 个候选(例如 $K=100$), 然后用一个更强大但更慢的模型对这 $K$ 个候选进行精细排序, 最终选出最好的 $k$ 个(例如 $k=5$). 

**Cross-Encoder 架构**

与 Bi-Encoder 分别编码 query 和 document 不同, Cross-Encoder 将 query 和 document **拼接后一起输入**模型: 

$$
 \text{score}(q, d) = f_\phi([q; \text{SEP}; d]) \tag{22}
$$
其中 $[\cdot; \cdot]$ 表示拼接, SEP 是分隔符. Cross-Encoder 的自注意力机制允许 query 的每个 token 直接 attend 到 document 的每个 token, 从而实现深度的细粒度交互. 

**为什么 Cross-Encoder 更准确, 但也更慢？**

Bi-Encoder 的相似度计算是 $O(d)$ 的向量点积, 百万级文档可以在毫秒内完成. 而 Cross-Encoder 需要对每一对 $(q, d)$ 分别做一次前向传播, 计算复杂度是 $O(L_q \cdot L_d)$, 其中 $L_q$ 和 $L_d$ 分别是 query 和 document 的序列长度. 这意味着 Cross-Encoder 无法用于全量检索, 只能用于对已经召回的小候选集进行精排. 

**重排序的数学表述**: 

$$
 \mathcal{R}_{\text{final}}(q) = \text{Top-}k_{d \in \mathcal{R}_K(q)} \, f_\phi([q; \text{SEP}; d]) \tag{23}
$$

工业界常用的开源重排序模型包括 BGE-Reranker、jina-reranker-v2 等. 实验表明, 加入一个高质量的重排序步骤, 可以将检索准确率提升 10%-30%(以 NDCG@10 衡量).

### 3.4 高级 RAG 技术: 超越基础 Pipeline

基础 RAG pipeline 在实际部署中会遇到各种边界问题, 研究者和工程师们提出了一系列高级技术来应对. 

#### 3.4.1 Query 重写(Query Rewriting): 弥合表达鸿沟

用户的原始查询往往不是检索的最佳形式. 例如: 
- 用户问: "那台红色笔记本多少钱？"(在一个多轮对话中, "那台红色笔记本"指代的是前文提到的某个具体型号). 
- 用户问: "这东西怎么用？"(完全缺乏上下文). 
- 用户问: "怎么让模型变聪明？"(口语化、不精确). 

Query Rewriting 使用 LLM 将用户的原始查询改写成更适合检索的形式. 常见的改写策略包括: 

**(1)Query Expansion(查询扩展)** 

让 LLM 生成原始查询的多个同义表述或更具体的变体: 

$$
 q \rightarrow \{q, q_1^{\text{exp}}, q_2^{\text{exp}}, \ldots\} \tag{24}
$$
然后分别检索这些变体, 合并结果. 这种方法增加了检索的召回率(Recall), 但可能引入噪声. 

**(2)假设文档嵌入(HyDE, Hypothetical Document Embedding)** 

HyDE 是一种极具创造力的策略, 由 Gao et al. (2022) 提出. 它的核心洞见是: **直接 embedding 查询的效果, 往往不如 embedding 查询对应的"假设答案"**. 

具体步骤: 
1. 使用 LLM 根据查询 $q$ 生成一个"假设答案" $d^{\text{hyp}}$(不保证正确, 只需要看起来合理). 
2. 将假设答案进行 embedding: $\mathbf{e}^{\text{hyp}} = \text{Embed}(d^{\text{hyp}})$. 
3. 用 $\mathbf{e}^{\text{hyp}}$ 在向量数据库中检索真实文档. 

**物理直觉**: 用户查询通常很短(几个词到一句话), 信息量稀疏. 而一个假设答案虽然可能包含错误信息, 但它的语义密度和结构更接近真实文档. 在向量空间中, 假设答案的向量方向往往比原始查询向量更靠近真实相关文档. 

$$
 \mathcal{R}(q) = \text{Top-}k \, \text{sim}(\text{Embed}(\text{LLM}(q)), \mathbf{e}_{i,j}) \tag{25}
$$

HyDE 在查询极度简短或口语化的场景下效果尤为显著, 但在需要精确词汇匹配的场景(如查找特定型号的产品)中可能适得其反. 

**(3)多查询分解(Multi-Query Decomposition)** 

对于复杂的多条件查询, LLM 可以将其分解为多个子查询: 

$$
 q_{\text{complex}} \rightarrow \{q_1, q_2, \ldots, q_m\} \tag{26}
$$
每个子查询独立检索, 最终合并结果. 例如, "比较 Python 和 JavaScript 在异步编程上的差异"可以分解为"Python 异步编程特点"和"JavaScript 异步编程特点"两个子查询. 

#### 3.4.2 多路检索(Multi-Channel Retrieval)

单一检索通道往往有其盲区. 多路检索同时使用多个独立的检索通道, 然后融合它们的结果: 

$$
 \mathcal{R}_{\text{multi}}(q) = \text{Fusion}(\mathcal{R}_{\text{vector}}(q), \mathcal{R}_{\text{keyword}}(q), \mathcal{R}_{\text{graph}}(q), \ldots) \tag{27}
$$

常见的检索通道包括: 
- **向量检索(Dense)** : 语义匹配. 

- **关键词检索(Sparse)** : 精确词汇匹配. 

- **图检索(Graph)** : 基于实体关系的遍历(GraphRAG). 

- **全文检索(Full-Text)** : Elasticsearch 等传统搜索引擎. 

- **SQL 检索**: 针对结构化数据(表格、数据库)的精确查询. 

每一路检索都在解决其他路的盲区. 例如, 用户问"请列出 2024 年发布且评分超过 4.5 的 Python 库", 其中"2024 年"和"评分 > 4.5"适合用结构化查询(SQL), 而"Python 库"适合用向量检索. 

#### 3.4.3 Self-RAG: 让模型自己决定是否需要检索

Self-RAG(Asai et al., 2023)将检索决策权从固定的 pipeline 移交给了 LLM 本身. 它的核心思想是: **并非每个问题都需要检索**. 如果 LLM 已经确定自己对某个问题了如指掌, 强制检索只会浪费计算资源并可能引入干扰信息. 

Self-RAG 在生成过程中引入特殊的"反思 token"(reflection tokens): 

- **Retrieve token**: 模型在生成每个段落前, 先判断"我是否需要检索外部信息？"输出 `[Retrieve]` 或 `[No Retrieve]`. 

- **IsRel token**: 如果执行了检索, 模型判断"检索到的文档与当前问题相关吗？"输出 `[Relevant]` 或 `[Irrelevant]`. 

- **IsSup token**: 模型判断"我生成的内容是否被检索到的文档所支持？"输出 `[Fully Supported]`、`[Partially Supported]` 或 `[No Support]`. 

- **IsUse token**: 模型判断"这个信息对回答问题有用吗？"输出 `[Useful]` 或 `[Not Useful]`. 

训练时, 这些反思 token 通过人工标注或自动评估进行监督; 推理时, 模型自发生成这些 token 来引导检索和生成行为. 

**Self-RAG 的数学表述**: 

设模型在生成第 $t$ 个 token 时的隐藏状态为 $\mathbf{h}_t$. 模型首先输出一个检索决策: 

$$
 P(\text{Retrieve} | \mathbf{h}_t) = \sigma(\mathbf{w}_r^\top \mathbf{h}_t) \tag{28}
$$
如果需要检索, 则执行检索并更新上下文; 如果不需要, 则直接基于内部知识继续生成. 这种自适应机制使得系统在处理简单事实性问题时可以直接回答, 在处理复杂或不确定问题时自动求助外部知识, 实现了效率与准确性的动态平衡.

### 3.5 评估指标: 如何量化 RAG 系统的质量

没有评估就没有优化. RAG 系统的评估分为三个层次: 检索质量、生成质量和端到端质量. 

#### 3.5.1 检索指标

**Recall@k**: 在前 $k$ 个检索结果中, 包含至少一个相关文档的比例. 

$$
 \text{Recall@}k = \frac{1}{|Q|} \sum_{q \in Q} \mathbb{1}[\exists d^+ \in \mathcal{R}_k(q)] \tag{29}
$$

Recall@k 衡量的是"能不能找到"相关文档. 在 RAG 中, 由于 LLM 具有较强的综合和推理能力, 通常 Recall 比 Precision 更重要——只要相关文档被包含在前 $k$ 个结果中, LLM 往往能从中提取正确答案. 

**MRR(Mean Reciprocal Rank)** : 相关文档在排名中的位置的倒数的平均值. 

$$
 \text{MRR} = \frac{1}{|Q|} \sum_{q \in Q} \frac{1}{\text{rank}_q(d^+)} \tag{30}
$$
如果相关文档排在第一位, 贡献 1; 排在第二位, 贡献 0.5; 排得越靠后, 贡献越小. MRR 同时考虑了"找不找得到"和"排得靠不靠前". 

**NDCG@k(Normalized Discounted Cumulative Gain)** : 考虑多等级相关度的排名指标. 

$$
 \text{DCG@}k = \sum_{i=1}^{k} \frac{2^{\text{rel}_i} - 1}{\log_2(i + 1)} \tag{31}
$$

其中 $\text{rel}_i$ 是第 $i$ 个文档的相关度等级(如 0=不相关, 1=部分相关, 2=完全相关). 分母 $\log_2(i+1)$ 是对排名的折扣因子——排得越靠前, 权重越大. NDCG 通过除以理想排名下的 DCG 进行归一化. NDCG 是信息检索领域最权威的排名质量指标, 同时考虑了相关度和排序位置. 

#### 3.5.2 生成指标

**Faithfulness(忠实度)** : 生成答案中的陈述有多少比例可以被检索到的上下文所支持. 

$$
 \text{Faithfulness} = \frac{|\{\text{claim} \in a : \text{claim is supported by } \mathcal{R}_k(q)\}|}{|\{\text{claim} \in a\}|} \tag{32}
$$
Faithfulness 通常由另一个 LLM(作为 judge)来自动评估: 将检索上下文和生成答案输入 judge LLM, 让它逐句判断每个陈述是否有上下文支持. 高 Faithfulness 意味着模型没有"自由发挥", 严格基于检索到的证据作答. 

**Answer Relevance(答案相关性)** : 生成答案与原始查询的匹配程度. 即使答案是忠实的(有文档支持), 它也可能答非所问. Answer Relevance 由 judge LLM 评估答案是否直接回应了查询的核心意图. 

#### 3.5.3 端到端指标: RAGAS 框架

RAGAS(Retrieval-Augmented Generation Assessment, Es et al., 2023)是一个专门用于评估 RAG 系统的开源框架, 它将上述指标整合为四个核心维度: 

1. **Faithfulness**: 答案是否被上下文支持. 

2. **Answer Relevance**: 答案是否与问题相关. 

3. **Context Relevance**: 检索到的上下文是否包含回答问题的最少必要信息(惩罚包含过多无关信息的上下文). 

4. **Context Recall**: 检索到的上下文是否包含了 ground-truth 答案所需的所有信息. 

RAGAS 的评估完全自动化, 不需要人工标注的答案, 这使得它非常适合在 CI/CD pipeline 中持续监控 RAG 系统的性能退化.

## 4. 数值走查 (Numerical Example)

为了将上述理论转化为可触摸的计算过程, 让我们构造一个极其简化的场景, 手动走一遍 RAG 的核心计算流程. 

### 4.1 场景设定: 一个小型技术文档集

假设我们有以下 4 篇关于机器学习优化的微型文档(已预先切分为 chunks): 

| Doc ID | Chunk Text |
|--------|-----------|
| D1 | "梯度下降通过计算损失函数对参数的导数来更新权重" |
| D2 | "Adam 优化器结合了动量和自适应学习率, 收敛速度快" |
| D3 | "随机梯度下降每次只用一个样本来估计梯度, 计算效率高" |
| D4 | "Transformer 使用自注意力机制并行处理序列中的所有位置" |

用户查询: $q = $ "哪种优化算法收敛最快？"

### 4.2 手动计算 BM25 分数

首先, 我们对文档和查询进行中文分词. 为简化, 假设我们按字分词: 

- D1: ["梯","度","下","降","通","过","计","算","损","失","函","数","对","参","数","的","导","数","来","更","新","权","重"]
- D2: ["A","d","a","m","优","化","器","结","合","了","动","量","和","自","适","应","学","习","率","收","敛","速","度","快"]
- D3: ["随","机","梯","度","下","降","每","次","只","用","一","个","样","本","来","估","计","梯","度","计","算","效","率","高"]
- D4: ["T","r","a","n","s","f","o","r","m","e","r","使","用","自","注","意","力","机","制","并","行","处","理","序","列","中","的","所","有","位","置"]

查询 q: ["哪","种","优","化","算","法","收","敛","最","快"]

**Step 1: 计算 IDF**

文档总数 $N = 4$. 统计每个查询词出现在多少篇文档中: 

| 词项 | 包含该词的文档数 $n_t$ | IDF = $\log\frac{N - n_t + 0.5}{n_t + 0.5}$ |
|------|----------------------|-------------------------------------------|
| 优 | D2 | $n=1$, idf = $\log\frac{4-1+0.5}{1+0.5} = \log\frac{3.5}{1.5} = \log(2.33) \approx 0.847$ |
| 化 | D2 | $n=1$, idf $\approx 0.847$ |
| 收 | D2 | $n=1$, idf $\approx 0.847$ |
| 敛 | D2 | $n=1$, idf $\approx 0.847$ |
| 梯 | D1, D3 | $n=2$, idf = $\log\frac{4-2+0.5}{2+0.5} = \log\frac{2.5}{2.5} = \log(1.0) = 0$ |
| 度 | D1, D3 | $n=2$, idf = $0$ |

为简化计算, 我们假设所有查询词在各自文档中的 tf 均为 1(即每个词在文档中只出现一次). 参数取标准值 $k_1 = 1.2$, $b = 0.75$. 平均文档长度 $\text{avgdl} \approx 24$(4 篇文档平均约 24 个字). 

**Step 2: 计算各文档的 BM25 分数**

对于 D2("Adam 优化器结合了动量和自适应学习率, 收敛速度快"), 文档长度约 24: 

查询词"优"、"化"、"收"、"敛"均出现在 D2 中. 以"优"为例: 

$$
 \text{term score} = \text{idf}(\text{优}) \cdot \frac{1 \cdot (1.2 + 1)}{1 + 1.2 \cdot (1 - 0.75 + 0.75 \cdot \frac{24}{24})} = 0.847 \cdot \frac{2.2}{1 + 1.2 \cdot 1.0} = 0.847 \cdot \frac{2.2}{2.2} = 0.847 \tag{33}
$$

四个相关词的总分约为 $4 \times 0.847 = 3.388$. 

对于 D1, 查询中只有"梯"、"度"出现, 但这两个词的 idf = 0(因为出现在两篇文档中, 区分度为零). 因此 D1 的 BM25 分数为 0. 

对于 D3, 同样只有"梯"、"度"出现, BM25 分数为 0. 

对于 D4, 没有任何查询词出现, BM25 分数为 0. 

**BM25 排名**: D2 (3.39) >> D1 = D3 = D4 (0)

BM25 正确地识别出 D2 是最相关的文档, 因为它包含了"优化"和"收敛"这两个查询中的关键词. 

### 4.3 Dense Retrieval 的简化演示

为展示 dense retrieval, 我们构造一个极度简化的 3 维 embedding 空间: 

| Chunk | Embedding (3-dim) | 语义标签 |
|-------|------------------|---------|
| D1 | $[0.8, 0.1, 0.0]$ | 梯度下降 |
| D2 | $[0.2, 0.9, 0.1]$ | Adam 优化器 |
| D3 | $[0.7, 0.2, 0.0]$ | 随机梯度下降 |
| D4 | $[0.1, 0.1, 0.9]$ | Transformer |

查询 "哪种优化算法收敛最快？"的 embedding 假设为 $\mathbf{q} = [0.3, 0.85, 0.1]$(更接近优化器话题). 

计算余弦相似度(为简化, 假设向量已归一化, 点积即余弦相似度): 

- $\text{sim}(q, D1) = 0.3 \times 0.8 + 0.85 \times 0.1 + 0.1 \times 0.0 = 0.24 + 0.085 = 0.325$
- $\text{sim}(q, D2) = 0.3 \times 0.2 + 0.85 \times 0.9 + 0.1 \times 0.1 = 0.06 + 0.765 + 0.01 = 0.835$
- $\text{sim}(q, D3) = 0.3 \times 0.7 + 0.85 \times 0.2 + 0.1 \times 0.0 = 0.21 + 0.17 = 0.380$
- $\text{sim}(q, D4) = 0.3 \times 0.1 + 0.85 \times 0.1 + 0.1 \times 0.9 = 0.03 + 0.085 + 0.09 = 0.205$

**Dense Retrieval 排名**: D2 (0.835) > D3 (0.380) > D1 (0.325) > D4 (0.205)

Dense retrieval 同样正确地将 D2 排在首位, 并且额外识别出 D3(随机梯度下降)也是相关的, 因为它和 D1(梯度下降)共享相似的语义方向. 而 BM25 由于"梯度"一词的 idf 为 0, 完全没有区分 D1 和 D3 的能力. 这展示了两种检索方法在行为上的本质差异. 

### 4.4 Chunk Size 对检索质量的影响对比

假设我们有一篇长文档, 内容如下: 

> "Transformer 模型由 Vaswani 等人在 2017 年提出. 其核心创新是自注意力机制(Self-Attention). 自注意力允许模型在处理序列时同时关注所有位置的信息, 而不需要像 RNN 那样按顺序处理. 这种并行性使得 Transformer 在长序列上的训练速度远超 RNN. 此外, Transformer 还引入了多头注意力(Multi-Head Attention), 通过多组不同的注意力权重并行计算, 捕捉不同子空间的信息. Transformer 已经成为现代大语言模型的基础架构, 包括 GPT 系列、BERT、T5 等都基于这一架构. "

**实验 A: Chunk Size = 50 tokens**

切分结果(简化示意): 
- Chunk A1: "Transformer 模型由 Vaswani 等人在 2017 年提出. 其核心创新是自注意力机制"
- Chunk A2: "(Self-Attention). 自注意力允许模型在处理序列时同时关注所有位置的信息"
- Chunk A3: ", 而不需要像 RNN 那样按顺序处理. 这种并行性使得 Transformer 在长序列上的"
- Chunk A4: "训练速度远超 RNN. 此外, Transformer 还引入了多头注意力(Multi-Head Attention)"

**实验 B: Chunk Size = 150 tokens**

切分结果: 
- Chunk B1: "Transformer 模型由 Vaswani 等人在 2017 年提出...训练速度远超 RNN. "
- Chunk B2: "此外, Transformer 还引入了多头注意力...都基于这一架构. "

现在用户查询: "自注意力的作用是什么？"

在实验 A 中, Chunk A2 几乎完全围绕"自注意力允许模型在处理序列时同时关注所有位置的信息"这一核心命题, 与查询的匹配度极高. 检索系统能精准地召回 A2. 

在实验 B 中, Chunk B1 包含了关于 Transformer 提出者、自注意力、RNN 对比、并行性等多个主题. 虽然它确实包含了自注意力的信息, 但由于混入了许多其他主题的语义, 其 embedding 向量的"语义纯度"被稀释了. 在向量空间中, B1 的方向可能既指向"自注意力", 又指向"RNN"和"并行计算", 导致与纯"自注意力"查询的相似度不如 A2 高. 

**结论**: 较小的 chunk size 通常带来更高的检索精度(因为语义更聚焦), 但可能牺牲上下文完整性(因为每个 chunk 包含的信息更少). LLM 在基于 A2 生成答案时, 可能知道"自注意力允许同时关注所有位置", 但不知道这是 Transformer 相对于 RNN 的核心优势——这个背景信息在 A1 中. 这就是 chunk size 权衡的本质.

## 5. 简化实现 (Simplified Implementation)

以下是一个约 100 行的完整 RAG pipeline Python 实现, 涵盖文档切分、embedding、FAISS 向量检索和生成阶段. 代码自包含, 只需安装 `faiss-cpu` 和 `sentence-transformers` 即可运行. 

```python
"""
简化 RAG Pipeline 实现
涵盖: 文档切分 -> Embedding -> FAISS 检索 -> Prompt 构造 -> 答案生成
"""
import numpy as np
import faiss
from sentence_transformers import SentenceTransformer

# ============================================================
# 1. 文档切分: 递归按段落/句子切分
# ============================================================
def recursive_split(text: str, max_length: int = 100, overlap: int = 20) -> list[str]:
    """
    递归切分文本: 优先按段落(\n\n), 其次按句子(.?!), 最后按固定长度
    对应 3.2 节讨论的 Recursive Chunking 策略
    """
    separators = ["\n\n", ".", "!", "?", " "]
    
    # 如果文本本身就在限制内, 无需切分
    if len(text) <= max_length:
        return [text]
    
    # 尝试用分隔符切分
    for sep in separators:
        parts = text.split(sep)
        if len(parts) > 1:
            chunks = []
            current = ""
            for part in parts:
                candidate = current + (sep if current else "") + part
                if len(candidate) <= max_length:
                    current = candidate
                else:
                    if current:
                        chunks.append(current.strip())
                    current = part
            if current:
                chunks.append(current.strip())
            
            # 添加 overlap: 每个 chunk 末尾包含下一个 chunk 的开头部分
            if overlap > 0 and len(chunks) > 1:
                overlapped = []
                for i, chunk in enumerate(chunks):
                    if i > 0:
                        prefix = chunks[i-1][-overlap:]
                        chunk = prefix + chunk
                    overlapped.append(chunk)
                return overlapped
            return chunks
    
    # 兜底: 直接按固定长度切分
    return [text[i:i+max_length] for i in range(0, len(text), max_length - overlap)]


# ============================================================
# 2. 索引阶段: Embedding + FAISS 向量存储
# ============================================================
class VectorStore:
    """
    基于 FAISS 的向量数据库, 支持 add 和 search
    对应 3.1.1 节和 3.3.1 节的 Dense Retrieval 实现
    """
    def __init__(self, embedding_model_name: str = "BAAI/bge-small-zh-v1.5"):
        # 加载 Bi-Encoder embedding 模型
        # 对应数学公式: e_{i,j} = Embed(c_{i,j})
        self.encoder = SentenceTransformer(embedding_model_name)
        self.dim = self.encoder.get_sentence_embedding_dimension()
        self.index = faiss.IndexFlatIP(self.dim)  # 内积索引(假设向量已归一化, 等价于余弦相似度)
        self.chunks = []  # 存储原始文本, 用于结果溯源
        
    def add_documents(self, documents: list[str]):
        """将文档切分、编码后存入向量索引"""
        all_chunks = []
        for doc in documents:
            chunks = recursive_split(doc, max_length=120, overlap=20)
            all_chunks.extend(chunks)
        
        # 编码所有 chunks
        # 对应数学公式: e_{i,j} = Embed(c_{i,j})
        embeddings = self.encoder.encode(all_chunks, normalize_embeddings=True)
        embeddings = np.array(embeddings, dtype=np.float32)
        
        # 添加到 FAISS 索引
        self.index.add(embeddings)
        self.chunks.extend(all_chunks)
        print(f"已索引 {len(all_chunks)} 个 chunks, 向量维度 {self.dim}")
    
    def search(self, query: str, top_k: int = 3) -> list[tuple[str, float]]:
        """
        检索 Top-k 相关 chunks
        对应数学公式: R_k(q) = Top-k sim(q, e_{i,j})
        """
        # 编码查询
        q_embedding = self.encoder.encode([query], normalize_embeddings=True)
        q_embedding = np.array(q_embedding, dtype=np.float32)
        
        # FAISS 搜索: 返回相似度分数和索引
        scores, indices = self.index.search(q_embedding, top_k)
        
        results = []
        for score, idx in zip(scores[0], indices[0]):
            if idx >= 0:  # FAISS 返回 -1 表示未找到
                results.append((self.chunks[idx], float(score)))
        return results


# ============================================================
# 3. RAG Pipeline: 检索 + 生成
# ============================================================
class SimpleRAG:
    """简化 RAG Pipeline"""
    def __init__(self, vector_store: VectorStore):
        self.vector_store = vector_store
    
    def retrieve(self, query: str, top_k: int = 3) -> list[tuple[str, float]]:
        """检索阶段: 对应 3.1.2 节"""
        return self.vector_store.search(query, top_k)
    
    def construct_prompt(self, query: str, retrieved_chunks: list[tuple[str, float]]) -> str:
        """构造带上下文的 prompt: 对应 3.1.3 节"""
        context = "\n\n".join([
            f"[相关度 {score:.3f}] {chunk}" 
            for chunk, score in retrieved_chunks
        ])
        
        prompt = f"""基于以下参考信息回答问题. 如果参考信息不足以回答, 请明确说明. 

[参考信息]
{context}

[问题]
{query}

[回答]
"""
        return prompt
    
    def generate(self, query: str, top_k: int = 3) -> dict:
        """
        完整的 RAG 流程
        实际工程中, 这里会调用 LLM API; 为演示, 我们返回构造好的 prompt
        """
        retrieved = self.retrieve(query, top_k)
        prompt = self.construct_prompt(query, retrieved)
        
        return {
            "query": query,
            "retrieved_chunks": retrieved,
            "prompt": prompt,
            "answer_placeholder": "[此处由 LLM 基于 prompt 生成答案]"
        }


# ============================================================
# 4. 运行示例
# ============================================================
if __name__ == "__main__":
    # 示例文档集
    documents = [
        "Transformer 模型由 Vaswani 等人在 2017 年提出. 其核心创新是自注意力机制(Self-Attention). "
        "自注意力允许模型在处理序列时同时关注所有位置的信息, 而不需要像 RNN 那样按顺序处理. "
        "这种并行性使得 Transformer 在长序列上的训练速度远超 RNN. ",
        
        "Adam 优化器是一种自适应学习率优化算法, 由 Kingma 和 Ba 在 2014 年提出. "
        "它结合了动量法和 RMSProp 的优点, 通过维护一阶矩和二阶矩估计来动态调整每个参数的学习率. "
        "Adam 在实践中收敛速度快, 对超参数的选择不敏感, 是深度学习中最常用的优化器之一. ",
        
        "检索增强生成(RAG)是一种将外部知识检索与大语言模型生成相结合的技术架构. "
        "RAG 通过向量检索找到与用户查询相关的文档片段, 然后将这些片段作为上下文提供给 LLM, "
        "从而显著降低幻觉并提高答案的事实准确性. "
    ]
    
    # 构建索引
    store = VectorStore()
    store.add_documents(documents)
    
    # 构建 RAG pipeline
    rag = SimpleRAG(store)
    
    # 测试查询
    queries = [
        "自注意力的作用是什么？",
        "Adam 优化器有什么特点？",
        "什么是 RAG？"
    ]
    
    for q in queries:
        result = rag.generate(q, top_k=2)
        print("=" * 60)
        print(f"查询: {q}")
        print(f"检索结果:")
        for chunk, score in result["retrieved_chunks"]:
            print(f"  [score={score:.3f}] {chunk[:60]}...")
        print(f"\n构造的 Prompt 长度: {len(result['prompt'])} 字符")
```

**代码对应理论说明**: 

| 代码段 | 对应章节 | 数学公式 |
|--------|---------|---------|
| `recursive_split` | 3.2 文档切分策略 | $d_i \rightarrow \{c_{i,j}\}$ |
| `encoder.encode` | 3.1.1 Embedding | $\mathbf{e}_{i,j} = \text{Embed}(c_{i,j})$ |
| `IndexFlatIP` + `search` | 3.3.1 Dense Retrieval | $\text{sim}(\mathbf{q}, \mathbf{e}) = \mathbf{q}^\top \mathbf{e}$ |
| `construct_prompt` | 3.1.3 生成阶段 | $P(a \| q, \text{Context}; \theta_{LLM})$ |

## 6. 局限性与边界条件 (Limitations & Boundary Conditions)

RAG 并非银弹. 理解它的局限性, 对于设计可靠的生产系统至关重要. 

### 6.1 检索失败: 当 Query 与文档语义不匹配时

RAG 系统的质量上限受限于检索阶段的质量. 如果检索阶段失败了——即真正的相关文档没有被召回——那么无论后端的 LLM 多么强大, 都无法给出正确答案. 

**检索失败的典型场景**: 

**(1)词汇不匹配(Vocabulary Mismatch)** 

用户用了一套词汇, 而文档使用了完全不同的词汇来表达相同的概念. 例如: 
- 用户查询: "如何防止神经网络过拟合？"
- 相关文档: "Dropout 是一种正则化技术, 通过在训练时随机忽略部分神经元来减少共适应. "

文档中没有出现"过拟合"这个词. 纯 sparse retrieval(如 BM25)会完全错过这篇文档. Dense retrieval 虽然有一定机会通过语义相似度捕获(因为"防止过拟合"和"正则化技术"在语义空间中距离较近), 但如果 embedding 模型的训练数据中"正则化"和"过拟合"的共现不够强, dense retrieval 也可能失败. 

**(2)抽象查询 vs 具体文档**

用户问了一个高度抽象的问题, 而文档中的信息非常具体. 例如: 
- 用户查询: "我们公司去年营收增长的原因是什么？"
- 相关文档: "Q3 亚太地区销售额同比增长 34%, 主要得益于新产品线的推出. "

文档中没有任何地方提到"营收增长的原因"这一抽象概念. 它只是在陈述一个事实. 要让检索系统将这两者关联起来, 需要 embedding 模型具备非常强的推理级语义理解能力——而当前的开源 embedding 模型在这方面仍有明显不足. 

**(3)否定与条件查询**

用户查询包含否定词或复杂条件时, 检索系统往往"看不懂": 
- 用户查询: "哪种优化器**不**需要手动设置学习率？"
- 不相关文档: "SGD 需要仔细调整学习率, 否则可能发散. "
- 相关文档: "Adam 优化器自适应地调整学习率, 无需手动设置. "

Embedding 模型通常对否定词不敏感, "不需要手动设置学习率"和"需要手动设置学习率"在向量空间中可能非常接近, 导致不相关文档被错误召回. 

**根本数学原因**: 当前主流的 embedding 模型基于对比学习训练, 优化的目标是"相关文本对的向量距离近, 不相关文本对的向量距离远". 但对比学习的训练数据通常不包含否定、条件、因果等复杂逻辑结构的显式标注, 导致模型学到的语义空间是"浅层语义"的——它能判断"苹果"和"水果"相关, 但难以处理"不是苹果"这种否定语义. 

### 6.2 上下文窗口限制: 塞不下是所有检索系统的宿命

即使检索阶段完美召回了所有相关文档, 这些文档的总长度可能远超 LLM 的上下文窗口限制. 

**数学约束**: 设 LLM 的上下文窗口为 $C$ tokens, 检索到的 $k$ 个 chunks 的总长度为 $L_{\text{chunks}}$, 系统 prompt 和输出预留长度为 $L_{\text{system}}$. 必须满足: 

$$
 L_{\text{chunks}} + L_{\text{system}} + L_{\text{output}} \leq C \tag{34}
$$
当 $L_{\text{chunks}} > C$ 时, 我们不得不进行**上下文压缩**(Context Compression): 

- **截断(Truncation)** : 直接丢弃超出窗口的文档. 简单粗暴, 但可能丢失关键信息. 

- **摘要压缩(Summarization Compression)** : 使用一个轻量级模型对检索到的文档进行逐段摘要, 只保留精华. 增加了计算开销, 且摘要过程本身可能引入信息损失. 

- **Map-Reduce 策略**: 将文档分批处理, 每批独立生成部分答案, 最后汇总. 这种策略显著增加了 token 消耗和延迟. 

即使使用上下文窗口长达 128K 甚至 1M token 的模型(如 GPT-4 Turbo、Kimi、Claude 3), "塞不下"的问题依然没有根除. 原因有两个: 一是**检索噪声**——检索到的文档中往往混入了大量低相关度的内容, 这些内容不仅占用宝贵的上下文窗口, 还可能干扰 LLM 的判断(即引入"干扰信息"); 二是**注意力稀释**——即便技术上能塞进去 100 篇文档, LLM 的注意力机制在处理超长序列时, 对远距离信息的敏感度会下降(这在学术上被称为"Lost in the Middle"现象), 导致中间位置的文档被"忽略". 

### 6.3 多跳推理: RAG 的阿喀琉斯之踵

多跳推理(Multi-Hop Reasoning)是指回答一个问题需要综合多个独立文档中的信息, 并通过推理链条将它们连接起来. 

**典型例子**: 
- 文档 A: "爱因斯坦于 1921 年获得诺贝尔物理学奖. "
- 文档 B: "1921 年的诺贝尔物理学奖颁奖典礼在斯德哥尔摩举行. "
- 用户查询: "爱因斯坦获得诺贝尔奖的颁奖典礼在哪里举行？"

要回答这个问题, 模型需要: 
1. 从文档 A 中推断出"爱因斯坦获得诺贝尔奖的年份是 1921 年". 
2. 用这个年份去文档 B 中查找对应信息. 
3. 综合得出"在斯德哥尔摩". 

**标准 RAG 的失败模式**: 当查询"爱因斯坦获得诺贝尔奖的颁奖典礼在哪里举行？"被 embedding 后, 它在向量空间中更接近"爱因斯坦"和"诺贝尔奖"相关的文档(即文档 A), 而与"颁奖典礼地点"相关的文档 B 可能因为不包含"爱因斯坦"这个关键词, 相似度排名较低, 无法进入 Top-k. 即便文档 B 被召回, 标准 RAG 的 prompt 构造方式只是将所有文档简单拼接, LLM 需要自己在文档之间建立跨文档的推理关联——而对于长文档集合, 这种关联的成功率很低. 

**根本局限**: 基础 RAG 的检索是"单轮、单查询"的. 它没有显式的机制来规划"需要先查什么, 再查什么"的推理路径. 后续演进的 GraphRAG 和 Agentic RAG 正是为了解决这个问题. 

### 6.4 索引更新延迟与数据新鲜度

RAG 的另一个工程现实是: 索引的建立不是免费的, 索引的更新也不是瞬时的. 

当新文档加入知识库时, 需要经过切分、embedding、写入向量数据库这一系列操作. 对于百万级文档的知识库, 全量重建索引可能需要数小时. 增量更新虽然快, 但 ANN 索引(如 HNSW、IVF)在频繁增量写入后可能需要周期性重建以维持搜索质量. 

更棘手的是**数据删除和修改**. 如果一篇已经被索引的文档中的某个事实被更正了, 你需要: 
1. 识别并删除该文档对应的所有旧 chunks. 
2. 重新切分更新后的文档. 
3. 重新计算 embedding 并写入索引. 

如果这一步做得不够严谨(例如旧 chunks 没有被完全删除), 就会出现"新旧版本信息共存"的混乱状态, 导致 LLM 基于过时的上下文生成错误答案. 这在需要高数据一致性的金融、医疗场景中是不可接受的.

## 7. 演进与承上启下 (Evolution & Segue)

面对上述局限性, RAG 技术在过去两年中经历了迅猛的演进. 每一条演进路径都在试图攻克基础 RAG 的某个特定软肋. 

### 7.1 Corrective RAG: 当检索质量不佳时自动纠错

Corrective RAG(CRAG)在标准 pipeline 中加入了一个**检索质量评估模块**. 在检索完成后、生成开始前, 系统先判断检索到的文档是否足够可靠. 如果检索质量评分低于阈值, 系统会自动触发 web 搜索等备用检索通道, 或者重写查询后重新检索. 这种"自我纠错"机制显著提高了系统在边缘查询上的表现. 

### 7.2 Speculative RAG: 用小模型做草稿, 大模型做精修

Speculative RAG 借鉴了 Speculative Decoding 的思想: 使用一个轻量级的"草稿 RAG"模型快速生成初始答案, 然后让大模型在验证阶段检查这个草稿是否需要修正. 这种方法在保持答案质量的同时, 将延迟降低了 30%-50%. 

### 7.3 GraphRAG: 用知识图谱捕获多跳关系

GraphRAG(由微软研究院提出)是应对多跳推理问题的重要方向. 它不再将文档存储为孤立的 chunks, 而是: 

1. **提取阶段**: 使用 LLM 从文档中提取实体(如"爱因斯坦"、"诺贝尔奖"、"斯德哥尔摩")和关系(如"获得"、"在……举行"). 

2. **建图阶段**: 构建一个知识图谱, 节点是实体, 边是关系. 

3. **检索阶段**: 用户查询触发图遍历(Graph Traversal), 沿着关系边找到多跳关联的实体和文档. 

GraphRAG 将多跳推理从 LLM 的"隐式推理"转变为知识图谱的"显式遍历", 大幅提高了可解释性和准确性. 但代价是建图成本高、图谱维护复杂, 且对于非结构化叙述性文本, 实体和关系的提取本身就会引入误差. 

### 7.4 Agentic RAG: 从 Pipeline 到智能体

**Agentic RAG 是当前最前沿、也最具潜力的演进方向**. 它将 RAG 从一个固定的线性 pipeline 转变为一个具备决策能力的智能体(Agent). 

在 Agentic RAG 中: 
- **路由(Routing)** : Agent 根据查询的类型决定走哪条路径. 事实性问题走 RAG 检索, 数学计算问题走代码解释器, 创意写作问题直接调用 LLM. 

- **工具调用(Tool Use)** : Agent 不仅检索文档, 还可以调用计算器、数据库、API、搜索引擎等多种工具. 这本质上就是 Function Calling 的泛化. 

- **规划(Planning)** : 对于复杂查询, Agent 会制定多步计划. 例如, "比较我们公司和竞争对手上季度的营收增长"需要先检索本公司财报, 再检索竞争对手财报, 然后计算增长率, 最后综合比较. 

- **反思(Reflection)** : Agent 在生成答案后会自我检查, 判断答案是否充分、是否有遗漏的信息来源、是否需要进一步检索. 

Agentic RAG 的架构与我们在后续章节中要深入探讨的 **Function Calling** 和 **MCP(Model Context Protocol)** 紧密相连. Function Calling 为 LLM 提供了调用外部工具的规范接口, 而 MCP 则定义了标准化的上下文交换协议, 使得 LLM Agent 能够无缝接入异构的数据源和工具生态.

## 8. 总结与参考文献 (References)

### 8.1 核心要点总结

1. **RAG 的本质是架构解耦**: 将知识的存储/更新与知识的理解/生成分离, 使得 LLM 能够动态获取最新、最相关的信息, 同时保持低成本和高可解释性. 

2. **检索质量决定系统上限**: 无论 LLM 多么强大, 如果检索阶段召回了错误的文档, 生成阶段的答案必然出错. Dense Retrieval、Sparse Retrieval、Hybrid Retrieval 和 Reranking 的组合使用, 是当前工程实践中的最佳方案. 

3. **Chunking 是被低估的核心环节**: 切分策略和 chunk size 的选择直接影响检索精度和上下文完整性之间的平衡. 递归切分和语义切分优于固定长度切分, 但需要根据具体场景调优. 

4. **评估必须覆盖检索和生成两个层面**: Recall@k 衡量检索的召回能力, Faithfulness 衡量生成的事实准确性, RAGAS 提供了端到端的自动化评估框架. 

5. **RAG 正在向 Agentic 演进**: 从固定 pipeline 到自适应检索(Self-RAG), 再到具备规划、路由和工具调用能力的 Agentic RAG, 检索增强生成正在与 Agent 技术深度融合.
