# 7.6 上下文工程

## 1. 演进脉络

LLM 应用开发的早期阶段, Prompt 工程几乎等同于"写好一条指令". 开发者将问题封装进一段精心设计的文本, 模型基于单轮输入生成输出. 这种模式的物理约束极为明显: 上下文窗口仅容纳 2048 或 4096 个 token, 对话历史一旦超出窗口边界,  earliest messages 被直接截断丢弃. 

多轮对话产品的普及催生了第二代需求. 聊天机器人需要维持连续对话的连贯性, 开发者引入滑动窗口机制, 保留最近 $k$ 轮对话作为上下文. 滑动窗口的局限在对话轮次增加后暴露: 关键信息(如用户此前声明的偏好、约束条件)因超出窗口而丢失, 模型被迫"遗忘"已确认的事实. 

RAG 系统的兴起将上下文管理推向第三阶段. 检索模块将外部知识注入 prompt, 但检索结果与原始对话争夺有限的 token 配额. Stuff、Map-Reduce、Refine 等策略被提出以在信息完整性与窗口约束之间做工程权衡. 此阶段的核心矛盾从"如何写好 prompt"转变为"如何在有限的 token 预算内编排最相关的信息". 

Agent 系统的复杂化催生了第四阶段. ReAct、ToT 等规划模式要求模型在推理过程中调用工具、观察结果、迭代决策. 每一步工具的返回输出都会膨胀上下文, 多轮推理的 token 消耗呈线性甚至超线性增长. Context Engineering 作为独立的技术域在此阶段确立: 它不再只是 prompt 设计的附属品, 而是决定应用能力边界的核心工程学科. 

当前演进的前沿聚焦在动态上下文检索与自适应记忆机制. 基于信息熵的 token 筛选、层次化上下文模板、语义感知的分层记忆等方案, 试图在固定窗口的物理约束下最大化有效信息的保留密度. 

| 阶段 | 时间线 | 核心矛盾 | 代表性技术 |
| :--- | :--- | :--- | :--- |
| 单轮 Prompt | 2020-2021 | 指令表达能力不足 | Few-Shot, CoT |
| 滑动窗口对话 | 2021-2022 | 长对话中的信息遗忘 | 滑动窗口, 对话摘要 |
| RAG 上下文编排 | 2022-2023 | 检索结果与对话争夺 token | Stuff, Map-Reduce, Refine |
| Agent 上下文管理 | 2023-2024 | 工具调用膨胀与推理深度 | ReAct, 工作记忆, 向量记忆 |
| 动态自适应上下文 | 2024-至今 | 固定窗口与信息密度的根本性冲突 | 选择性保留, 层次化模板, 记忆分层 |

## 2. 工业价值

上下文窗口是 LLM 应用落地的首要物理瓶颈, 其约束直接决定应用的能力边界. 

**客服系统的首问解决率**与客户历史对话的保留长度强相关. 某头部电商平台 A/B 测试显示: 将上下文窗口从 4K 扩展至 32K 并引入智能摘要后, 复杂售后场景的首问解决率从 61% 提升至 84%, 人工介入率下降 37%. P99 响应延迟从 2.8s 增至 4.1s, 但综合客服成本下降 43%. 

**代码助手**的跨文件编辑能力受限于上下文对项目结构信息的容纳量. GitHub Copilot 的内部评估表明, 当上下文能够包含当前编辑文件的上下游 3 个相关文件时, 跨文件引用的准确率从 52% 跃升至 79%. 上下文工程在此场景中的核心任务不是"塞入更多代码", 而是"在 token 预算内选择最相关的代码片段". 

**RAG 系统的检索质量**与上下文注入策略直接挂钩. 生产环境数据显示, 采用动态上下文检索(根据查询复杂度自适应调整检索 chunk 数量)的系统, 相比固定注入 5 个 chunk 的基线, Answer Relevance 评分提升 0.18(7 分制), 同时平均 token 消耗降低 22%. 

**Agent 系统的任务完成率**受上下文膨胀导致的注意力稀释影响. LangChain 社区的一项大规模基准测试显示, 执行超过 10 步的复杂任务时, 未做上下文优化的 Agent 任务完成率从初始的 78% 衰减至 31%, 而采用分层记忆与选择性保留的优化版本衰减幅度控制在 12% 以内. 

| 应用场景 | 上下文瓶颈指标 | 优化后的典型增益 |
| :--- | :--- | :--- |
| 客服对话 | 历史对话截断导致重复询问 | 首问解决率 +23pp |
| 代码助手 | 跨文件引用准确率 | 跨文件引用准确率 +27pp |
| RAG 问答 | 检索 chunk 与查询相关性 | Answer Relevance +0.18 |
| 多步 Agent | 长推理链中的注意力稀释 | 任务完成率衰减从 47pp 降至 12pp |

## 3. 物理直觉

上下文管理困难的根源, 嵌入在 Transformer 架构的物理属性之中. 

**注意力机制的二次复杂度**构成第一重约束. Self-Attention 的计算量随序列长度 $n$ 呈 $O(n^2)$ 增长. 当上下文从 4K 扩展至 128K, 注意力计算的 FLOPs 增长 1024 倍. 这不仅是算力问题, 更是内存带宽问题: KV Cache 的显存占用同样线性增长, 128K 上下文在 batch_size=32、fp16 精度下的 KV Cache 占用超过 80 GB, 超出单卡 A100 的显存容量. 

**信息过载**构成第二重约束. 人类在处理长文档时会自然进行"扫读"和"跳读", 将注意力集中在关键段落. Transformer 的注意力机制在理论上具备全局感知能力, 但实证研究表明, 长序列中的中间位置 token 被关注到的概率显著低于首尾位置——这一现象被称为"Lost in the Middle". 模型并非真正"阅读"了所有上下文, 而是在信息密度的稀释中逐渐丢失对中段信息的敏感度. 

**语义漂移**构成第三重约束. 多轮对话中, 话题的渐进式偏移导致早期对话的语义相关性随时间衰减. 用户在第 1 轮询问"Python 的 GIL 机制", 第 5 轮可能已转移到"异步编程的最佳实践". 如果上下文管理策略不做语义相关性的动态筛选, 早期关于 GIL 的技术细节会持续占据 token 配额, 稀释当前话题的有效信息密度. 

这三重约束的共同作用, 使得上下文工程的本质问题可以表述为: **在固定长度 $L$ 的 token 预算下, 如何选择并编排信息集合 $S$, 使得模型在生成回复时能够最大化利用 $S$ 中的有效信息**. 这是一个带约束的优化问题, 其目标函数涉及信息相关性、时效性、冗余度等多个维度. 

## 4. 数学推导

### 4.1 注意力计算的二次复杂度

Transformer Decoder  中, 单层的 Self-Attention 计算量由查询(Query)、键(Key)、值(Value)三个矩阵的交互决定: 

$$
 \text{FLOPs}_{\text{attn}} = 4 \cdot n^2 \cdot d_{\text{head}} \cdot h + 2 \cdot n \cdot d_{\text{model}}^2 \tag{1}
$$
其中 $n$ 为序列长度(上下文中的 token 数量); $d_{\text{head}}$ 为每个注意力头的维度; $h$ 为注意力头数, 满足 $h \cdot d_{\text{head}} = d_{\text{model}}$; $d_{\text{model}}$ 为模型的隐藏层维度. 第一项 $4 \cdot n^2 \cdot d_{\text{head}} \cdot h$ 涵盖注意力矩阵 $QK^T$ 的乘法与 Softmax 加权求和的总开销, 其中 $n^2$ 项来源于每个 token 必须与序列中全部 $n$ 个 token 计算注意力分数, 这是二次复杂度的核心来源, 系数 4 则同时计入乘法与加法运算. 第二项 $2 \cdot n \cdot d_{\text{model}}^2$ 对应线性投影层($Q, K, V$ 的生成及输出投影)的开销, 此项随 $n$ 线性增长, 在长序列场景中相对于 $n^2$ 项通常可忽略, 但在短序列中往往占主导地位. 

如式 (1) 所示, 当 $n = 4096$、$d_{\text{model}} = 4096$、$h = 32$ 时, 第一项约为 $8.6 \times 10^9$ FLOPs, 第二项约为 $1.3 \times 10^8$ FLOPs, 注意力计算占总 FLOPs 的 98.5%. 将 $n$ 扩展至 131072(128K), 第一项膨胀至 $8.8 \times 10^{15}$ FLOPs, 增长约 1024 倍. 

### 4.2 上下文压缩的信息论基础

上下文压缩的目标是用更短的文本 $C$ 保留原始上下文 $X$ 的最大信息量. 从信息论视角, 最优压缩策略最小化 $X$ 与 $C$ 之间的互信息损失: 

$$
 \mathcal{L}_{\text{compress}} = I(X; Y) - I(C; Y) + \lambda \cdot |C| \tag{2}
$$
其中 $X$ 为原始上下文(完整对话历史或文档); $C$ 为压缩后的上下文(摘要或筛选后的子集); $Y$ 为目标输出, 即模型需要生成的回复; $I(\cdot; \cdot)$ 为互信息, 衡量两个随机变量之间的统计依赖程度; $|C|$ 为压缩后文本的 token 长度; $\lambda$ 为压缩率惩罚系数, 控制保留信息量与长度之间的权衡. 第一项 $I(X; Y)$ 表示原始上下文与目标输出之间的互信息, 代表理想情况下生成回复所需的全部信息上限, 此项在压缩前即为常量. 第二项 $I(C; Y)$ 衡量压缩后上下文与目标输出之间的互信息, 其值越大说明 $C$ 保留了对生成 $Y$ 越有用的信息, $I(X; Y) - I(C; Y)$ 因此定义为压缩导致的信息损失. 第三项 $\lambda \cdot |C|$ 是对压缩文本长度的惩罚, $\lambda$ 的物理含义是单位 token 的边际成本——在固定窗口约束下, 每多用一个 token 做摘要, 就少一个 token 容纳其他信息, 因此可将其理解为机会成本. 

### 4.3 语义检索的相似度评分

RAG 系统中, 检索模块通过向量相似度将查询 $q$ 与文档 chunk $d_i$ 匹配. 最常用的评分函数是余弦相似度: 

$$
 \text{score}(q, d_i) = \frac{\mathbf{e}_q \cdot \mathbf{e}_{d_i}}{\|\mathbf{e}_q\| \cdot \|\mathbf{e}_{d_i}\|} = \cos(\theta) \tag{3}
$$
其中 $\mathbf{e}_q \in \mathbb{R}^D$ 为查询 $q$ 经 Embedding 模型编码后的稠密向量, $D$ 为向量维度; $\mathbf{e}_{d_i} \in \mathbb{R}^D$ 为第 $i$ 个文档 chunk 的稠密向量; $\|\mathbf{e}_q\|$ 为查询向量的 L2 范数; $\theta$ 为两个向量在 $D$ 维空间中的夹角. 分子 $\mathbf{e}_q \cdot \mathbf{e}_{d_i}$ 是两个向量的点积, 衡量它们在方向一致时的投影重叠程度, 点积越大意味着语义相似度越高. 分母 $\|\mathbf{e}_q\| \cdot \|\mathbf{e}_{d_i}\|$ 是归一化因子, 用于消除向量模长的影响——未归一化的点积会天然偏向模长大的向量, 而归一化后的余弦相似度 $\cos(\theta)$ 仅反映方向一致性, 其值域为 $[-1, 1]$. 在语义检索场景中, 高质量 Embedding 模型(如 text-embedding-3-large、BGE-M3)的向量通常分布在正象限, 实际值域集中在 $[0, 1]$, 值越接近 1 表示语义相关性越强. 

生产环境中常采用带有温度缩放的评分函数增强区分度: 

$$
 \text{score}_{\tau}(q, d_i) = \frac{\exp(\text{score}(q, d_i) / \tau)}{\sum_{j=1}^{K} \exp(\text{score}(q, d_j) / \tau)} \tag{4}
$$
其中 $\tau$ 为温度系数, 控制概率分布的锐化或平滑程度; $K$ 为候选 chunk 的总数; $j$ 为求和遍历索引. 分子内部先对相似度分数执行温度缩放 $\text{score}(q, d_i) / \tau$: 当 $\tau < 1$ 时分数差异被放大, top-k 结果的区分度增强; 当 $\tau > 1$ 时分布趋于平缓. 随后通过指数函数 $\exp(\cdot)$ 将线性分数映射到正实数域, 确保非负性并为 Softmax 做准备. 分母 $\sum_{j=1}^{K} \exp(\text{score}(q, d_j) / \tau)$ 为归一化因子, 使所有候选 chunk 的评分之和为 1, 形成合法的概率分布. 整体而言, 该函数将原始相似度分数转换为概率分布, 便于在上下文注入时进行加权采样或确定性截断. 

### 4.4 多轮对话的上下文累积与截断策略

设对话历史包含 $T$ 轮, 第 $t$ 轮的用户输入为 $u_t$, 模型回复为 $r_t$. 完整上下文为 $H_T = [(u_1, r_1), (u_2, r_2), \dots, (u_T, r_T)]$. 在窗口约束 $L$ 下, 需选择子集 $S \subseteq H_T$ 使得总长度不超过 $L$. 

滑动窗口截断策略的数学表达为: 

$$
 S_{\text{window}} = \{ (u_t, r_t) \mid T - k < t \leq T \} \tag{5}
$$
其中 $T$ 为对话总轮次; $k$ 为滑动窗口保留的最近轮次数; $t$ 为轮次索引. 窗口的下界 $T - k$ 决定了起始轮次, 所有索引小于等于 $T - k$ 的对话轮次均被截断丢弃; 上界 $t \leq T$ 则保证窗口始终终止于当前最新一轮. 该策略仅保留最近 $k$ 轮对话, 截断决策复杂度为 $O(1)$, 但由于完全以时间近邻性作为筛选标准, 未考虑信息重要性, 因此存在关键信息丢失的风险. 

基于重要性的自适应截断策略引入每轮对话的重要性权重 $w_t$: 

$$
 S_{\text{adaptive}} = \underset{S \subseteq H_T, \, |S| \leq L}{\arg\max} \sum_{(u_t, r_t) \in S} w_t \cdot \text{rel}(u_t, q) \tag{6}
$$
其中 $w_t$ 为第 $t$ 轮对话的内在重要性权重, 基于语义关键度、包含实体数量、用户强调标记等信号计算; $\text{rel}(u_t, q)$ 为第 $t$ 轮用户输入与当前查询 $q$ 的语义相关性分数, 通常由余弦相似度度量; $|S|$ 为选中对话子集的总 token 长度; $L$ 为上下文窗口的 token 上限. 重要性加权 $w_t$ 的作用是识别包含关键约束、用户偏好或事实确认的高价值对话轮次——例如用户声明"我只接受 Python 方案"的轮次应获得高权重. 相关性加权 $\text{rel}(u_t, q)$ 则负责筛选与当前查询语义相关的历史对话, 若查询转向新话题, 早期不相关的轮次应被降权或剔除. $\arg\max$ 操作在约束 $|S| \leq L$ 下寻求加权相关性的最大化, 该问题在一般形式下属于 NP-hard 的背包问题变体, 但实践中通常采用贪心算法近似求解. 

### 4.5 层次化上下文的结构熵

层次化上下文模板将信息按重要性分层组织. 设上下文包含 $M$ 个信息块, 每个块 $m$ 的优先级为 $p_m$, 信息量为 $I_m$. 层次化编排的目标是最大化结构化的信息效用: 

$$
 U = \sum_{m=1}^{M} p_m \cdot I_m \cdot \mathbb{1}_{[\sum_{j=1}^{m} l_j \leq L]} \tag{7}
$$
其中 $p_m$ 为第 $m$ 个信息块的优先级系数, 通常遵循系统指令高于用户偏好、用户偏好高于检索文档、检索文档高于历史对话的层级; $I_m$ 为第 $m$ 个信息块的信息量, 可用 token 级别的自信息或语义密度估计; $l_j$ 为第 $j$ 个信息块的 token 长度; $\mathbb{1}_{[\cdot]}$ 为指示函数, 条件满足时取值为 1, 否则为 0; $L$ 为总 token 预算. 优先级加权信息量 $p_m \cdot I_m$ 确保高优先级块——例如安全约束或输出格式要求——即使绝对信息量较小, 也会在编排中被优先保留. 截断指示 $\mathbb{1}_{[\sum_{j=1}^{m} l_j \leq L]}$ 表示当累积 token 消耗超出预算 $L$ 时, 后续信息块被完全截断, 体现了层次化模板"先保核心、后保外围"的物理逻辑. 整体而言, 层次化模板的编排本质上是按优先级排序后的贪心填充, 确保在预算耗尽前最高优先级的信息被完整保留. 

## 5. 核心技术与方法

### 5.1 长上下文压缩策略

LLM 应用处理长文档时, 核心挑战在于如何将超出窗口限制的文本送入模型. LangChain 社区提出的三种经典策略构成了工程实践的基准. 

**Stuff 策略**

将所有相关文本直接拼接进 prompt. 物理逻辑最简单: 零信息损失、零额外推理开销. 适用条件: 总文本长度 $< L - |\text{instruction}|$. 在 Llama-3-8B(8K 上下文)的场景中, Stuff 策略仅能处理约 6000 个 token 的文档内容. 一旦超出, 直接触发截断或报错. 

**Map-Reduce 策略**

将长文档切分为 $N$ 个 chunk, 每个 chunk 独立送入模型生成局部答案(Map 阶段), 再将所有局部答案聚合为最终输出(Reduce 阶段). Map 阶段的并行度为 $N$, 总 token 消耗约为 $N \cdot (L_{\text{chunk}} + L_{\text{answer}})$. Reduce 阶段的输入为 $N$ 个局部答案的拼接, 若其总长度超出窗口, 需递归执行 Reduce. 

Map-Reduce 的信息损失发生在两个环节: chunk 边界处的语义割裂(一个句子被截断到两个 chunk), 以及 Reduce 阶段对局部答案的二次压缩. 工程上通常采用重叠切分(overlap)缓解边界问题, 典型重叠率为 10%-20%. 

**Refine 策略**

迭代式压缩: 初始轮次将第一个 chunk 与查询送入模型生成初始答案; 后续每一轮将当前答案与下一个 chunk 送入模型, 要求模型基于新信息 refine(优化)答案. Refine 的 token 消耗约为 $N \cdot (L_{\text{answer}} + L_{\text{chunk}})$, 与 Map-Reduce 同级, 但串行执行导致延迟增加 $N$ 倍. 

Refine 的优势在于答案的连贯性: 每一轮都在前一轮的基础上增量更新, 避免了 Map-Reduce 中局部答案风格不一致的问题. 缺陷是错误传播——早期 chunk 引入的错误会在后续轮次中被累积放大. 

| 策略 | 并行度 | 信息损失点 | 延迟特征 | 适用场景 |
| :--- | :--- | :--- | :--- | :--- |
| Stuff | $O(1)$ | 截断丢弃 | 最低 | 短文档、低延迟需求 |
| Map-Reduce | $O(N)$ | Chunk 边界割裂、Reduce 截断 | 中(Map 可并行) | 独立段落、可并行处理 |
| Refine | $O(1)$ | 错误传播、早期信息遗忘 | 高(串行 $N$ 轮) | 强连贯性需求、叙事类文本 |

### 5.2 选择性上下文保留

选择性保留的核心思想是: 并非所有 token 对生成目标输出的贡献均等, 可以通过重要性评分筛选出"高价值"token, 剔除冗余信息. 

**基于自注意力权重的筛选**

在模型前向传播过程中, 计算每个 token 对最终输出 token 的注意力权重之和. 权重低于阈值 $\theta$ 的 token 被视为"低贡献"并被剔除. 设第 $i$ 个 token 的注意力权重和为 $A_i = \sum_{j} \alpha_{ij}$, 则保留集合为 $R = \{ i \mid A_i \geq \theta \}$. 

此方法的物理直觉是: 注意力权重直接量化了 token 间的信息流动强度. 但缺陷在于, 注意力权重是查询依赖的——同一个 token 在不同查询下的重要性可能截然不同, 离线计算的权重不具备泛化性. 

**基于信息熵的 token 筛选**

使用小型语言模型(如 DistilBERT、TinyLLaMA)计算每个 token 的条件熵, 熵值越高的 token 携带的信息量越大. 设 token $x_i$ 的条件熵为 $H(x_i \mid x_{<i})$, 保留概率与 $H(x_i \mid x_{<i})$ 正相关. 

此方法的工程优势在于: 小型模型的推理成本远低于主模型, 可在预处理阶段离线完成. 某开源实现(LLMLingua)的评估显示, 在保留 20% token 的情况下, 下游任务的性能衰减控制在 3% 以内. 

### 5.3 动态上下文检索

动态上下文检索解决 RAG 系统中"注入多少检索内容"的自适应问题. 固定注入 top-k 个 chunk 的策略在简单查询中浪费 token, 在复杂查询中信息不足. 

**基于查询复杂度的自适应检索**

定义查询复杂度分数 $C(q)$, 综合以下信号: 

- 查询长度: 长查询通常需要更多上下文支撑. 
- 实体密度: 包含多个实体(人名、地名、专业术语)的查询可能涉及多文档信息. 
- 问题类型: "为什么"/"如何"类问题比"是什么"类问题需要更深入的上下文. 

检索数量 $k$ 与复杂度分数正相关: $k = \max(k_{\min}, \min(k_{\max}, \lfloor \alpha \cdot C(q) \rfloor))$. 典型配置: $k_{\min} = 2$, $k_{\max} = 10$, $\alpha = 0.5$. 

**基于语义覆盖率的截断**

在检索到候选 chunk 后, 不直接按排名截断, 而是计算已选 chunk 的语义覆盖率. 设候选 chunk $d_i$ 与查询的相似度为 $s_i$, 已选集合为 $D_{\text{sel}}$. 新 chunk $d_j$ 的边际贡献定义为: 

$$
 \Delta_j = s_j - \max_{d_i \in D_{\text{sel}}} \text{score}(d_j, d_i) \tag{8}
$$
其中 $s_j$ 为候选 chunk $d_j$ 与查询 $q$ 的相似度分数; $\max_{d_i \in D_{\text{sel}}} \text{score}(d_j, d_i)$ 表示 $d_j$ 与已选集合中最相似 chunk 的相似度; $\Delta_j$ 为 $d_j$ 的边际语义贡献, 若 $d_j$ 与已选 chunk 高度相似, $\Delta_j$ 将接近 0, 说明其提供的新信息有限. 查询相关性 $s_j$ 确保候选 chunk 与当前问题相关; 冗余度 $\max \text{score}(d_j, d_i)$ 衡量 $d_j$ 与已选 chunk 的语义重叠, 值越高意味着冗余越大. 边际增益 $\Delta_j$ 仅在 $d_j$ 既与查询相关、又与已选集合不重复时才显著大于 0, 因此按 $\Delta_j$ 降序选择 chunk, 可在有限预算内最大化语义覆盖. 

### 5.4 记忆机制的分层架构

Agent 系统的记忆机制通常采用三层架构, 每层对应不同的时间尺度和存储介质. 

**短期记忆(Short-term Memory)** 

承载当前对话的近期轮次, 通常以原始文本形式存储在内存中. 物理实现为滑动窗口或循环缓冲区, 容量 $L_{\text{stm}} \leq L$(模型上下文窗口). 短期记忆的更新频率为每轮对话一次, 生命周期随会话结束而终结. 

**中期记忆(Medium-term Memory)** 

承载当前会话中提炼出的关键事实、用户偏好和约束条件. 物理实现为结构化摘要(如 JSON 格式的 key-value 对), 由模型在对话过程中动态提取和更新. 中期记忆的容量独立于 token 预算, 因为它以压缩后的结构化形式存在, 通常在 200-500 个 token 量级. 

**长期记忆(Long-term Memory)** 

跨越多个会话的持久化知识, 以向量数据库(语义记忆)和图数据库(关系记忆)的形式存储. 长期记忆的写入发生在会话结束时: 将对话摘要编码为向量, 连同时间戳和主题标签存入向量数据库. 读取时, 将当前查询编码为向量, 检索最相似的历史记忆. 

**向量记忆的检索时效性**

长期记忆面临时效性衰减问题. 3 个月前的用户偏好可能已被更新, 直接检索会导致过时信息干扰. 引入时间衰减因子的评分函数: 

$$
 \text{score}_{\text{memory}}(q, m_i) = \cos(\mathbf{e}_q, \mathbf{e}_{m_i}) \cdot e^{-\beta \cdot \Delta t_i} \tag{9}
$$
其中 $m_i$ 为第 $i$ 条历史记忆; $\mathbf{e}_{m_i}$ 为记忆 $m_i$ 的向量表示; $\Delta t_i$ 为记忆 $m_i$ 距离当前的绝对时间差(单位: 天); $\beta$ 为衰减系数, 控制记忆遗忘速度, 例如 $\beta = 0.01$ 表示记忆约 69 天后 relevance 衰减至 50%; $e^{-\beta \cdot \Delta t_i}$ 为指数衰减因子, 模拟人类记忆的遗忘曲线. 语义相关性 $\cos(\mathbf{e}_q, \mathbf{e}_{m_i})$ 提供基础检索分数, 衡量查询与记忆的语义匹配度; 时间衰减 $e^{-\beta \cdot \Delta t_i}$ 则对旧记忆施加惩罚, $\Delta t_i$ 越大, 衰减因子越小, 旧记忆的评分被相应压低. 整体评分在语义相关性的基础上引入时间维度, 优先返回既相关又新鲜的记忆. 

### 5.5 上下文感知的 Agent 规划

ReAct(Reasoning + Acting)模式中的上下文管理面临独特挑战: 每一轮"思考 → 行动 → 观察"都会将新的观察和思考追加到上下文, 导致序列长度线性增长. 

未经优化的 ReAct 在执行 $N$ 步后, 上下文长度约为: 

$$
 L_N = L_0 + N \cdot (L_{\text{think}} + L_{\text{action}} + L_{\text{obs}}) \tag{10}
$$
其中 $L_0$ 为初始 prompt 长度, 包含系统指令、任务描述与工具定义; $L_{\text{think}}$ 为每轮思考文本的平均长度; $L_{\text{action}}$ 为每轮行动(工具调用)的 JSON 长度; $L_{\text{obs}}$ 为每轮观察(工具返回结果)的平均长度; $N$ 为已执行的步数. 初始长度 $L_0$ 是固定开销, 与执行步数无关; 累积开销项 $N \cdot (L_{\text{think}} + L_{\text{action}} + L_{\text{obs}})$ 是 ReAct 上下文膨胀的核心来源——每步思考、行动与观察三阶段的文本都被追加到上下文, $N$ 步后呈线性增长. 若观察结果 $L_{\text{obs}}$ 较大(例如搜索结果包含多个网页的原始内容), 累积速度将显著加快. 

工程优化策略包括: 

1. **观察结果摘要**: 工具返回的原始观察不直接追加, 而是先经小型模型摘要为 1-2 句话的关键发现. 
2. **思考压缩**: 将多步思考的冗余表述(如重复的计划确认)压缩为结构化状态标记. 
3. **步数截断**: 当 $N$ 超过阈值 $N_{\max}$ 时, 将早期步骤的详细记录替换为高层摘要, 仅保留最近 $k$ 步的完整细节. 

## 6. 算力成本与数值走查

以一个生产级 RAG 客服系统为例, 走查上下文工程决策对算力和成本的影响. 

**系统配置**

- 模型: GPT-4o, 上下文窗口 128K, 输出限制 4K. 
- 平均查询长度: 50 tokens. 
- 检索 chunk 大小: 300 tokens/个. 
- 日均查询量: 100,000 次. 
- API 定价: 输入 $5 / 1M tokens, 输出 $15 / 1M tokens. 

**场景 A: 固定注入 top-5 chunk**

每查询的输入 token 数: $50 + 5 \times 300 = 1550$ tokens. 假设平均输出 200 tokens. 

每查询成本: $1550 \times 5 / 10^6 + 200 \times 15 / 10^6 = 0.00775 + 0.003 = \$0.01075$. 

日成本: $100{,}000 \times \$0.01075 = \$1{,}075$. 

**场景 B: 动态自适应检索(平均注入 3.2 个 chunk)** 

通过查询复杂度分析, 简单查询注入 2 个 chunk, 复杂查询注入 6-8 个 chunk, 加权平均 3.2 个. 

每查询输入 token 数: $50 + 3.2 \times 300 = 1010$ tokens. 

每查询成本: $1010 \times 5 / 10^6 + 200 \times 15 / 10^6 = 0.00505 + 0.003 = \$0.00805$. 

日成本: $100{,}000 \times \$0.00805 = \$805$. 

动态策略的日节省: $\$1{,}075 - \$805 = \$270$, 年节省约 $\$98{,}550$(约 25%). 

**场景 C: 引入选择性上下文保留(压缩率 40%)** 

在场景 B 基础上, 对检索到的 chunk 进行选择性压缩, 剔除冗余 token, 平均压缩率 40%. 压缩后每个 chunk 平均 180 tokens. 

每查询输入 token 数: $50 + 3.2 \times 180 = 626$ tokens. 

每查询成本: $626 \times 5 / 10^6 + 200 \times 15 / 10^6 = 0.00313 + 0.003 = \$0.00613$. 

日成本: $100{,}000 \times \$0.00613 = \$613$. 

相比场景 A 的日节省: $\$1{,}075 - \$613 = \$462$(约 43%). 

**场景 D: 长上下文直接推理(128K 全量文档)** 

若业务需求要求模型直接阅读整份用户手册(约 80K tokens)而非检索: 

每查询输入 token 数: $50 + 80{,}000 = 80{,}050$ tokens. 

每查询成本: $80{,}050 \times 5 / 10^6 + 200 \times 15 / 10^6 = 0.40025 + 0.003 = \$0.40325$. 

日成本: $100{,}000 \times \$0.40325 = \$40{,}325$. 

长上下文直接推理的成本是场景 C 的 65.8 倍. 这一数量级差异解释了为什么 RAG 仍是当前 LLM 应用的主流范式——上下文工程的经济价值不仅体现在性能, 更直接体现在运营成本. 

| 场景 | 平均输入 tokens | 单次成本 | 日成本(10万查询) | 相对基线 |
| :--- | :--- | :--- | :--- | :--- |
| A: 固定 top-5 | 1,550 | $0.01075 | $1,075 | 1.00x |
| B: 动态检索 | 1,010 | $0.00805 | $805 | 0.75x |
| C: 动态 + 压缩 | 626 | $0.00613 | $613 | 0.57x |
| D: 全量长上下文 | 80,050 | $0.40325 | $40,325 | 37.5x |

**显存占用走查**

在自托管场景(Llama-3-70B, batch_size=16, fp16)中, KV Cache 的显存占用为: 

$$
 M_{\text{KV}} = 2 \cdot n \cdot l \cdot d_{\text{model}} \cdot \text{batch_size} \cdot \text{bytes_per_param} \tag{11}
$$
其中 $n$ 为序列长度(上下文 token 数); $l$ 为 Transformer 层数, Llama-3-70B 取 80 层; $d_{\text{model}}$ 为隐藏层维度, 取 8192; $\text{batch_size}$ 为批次大小, 此处为 16; $\text{bytes_per_param}$ 为每个参数的存储字节数, fp16 精度下为 2 字节; 系数 2 表示同时存储 Key 和 Value 两个矩阵. $n \cdot l \cdot d_{\text{model}}$ 给出单层单序列的 KV Cache 元素数, 乘以 2 是因为 Key 与 Value 各需一份缓存; 再乘以 $\text{batch_size}$ 对应同时处理 16 条序列的并行开销; 最后乘以 2 字节得到 fp16 精度下的总显存占用. 

基于上述分析, 建立如下数学关系: 

$$
 M_{\text{KV}} = 2 \times 8192 \times 80 \times 8192 \times 16 \times 2 = 343 \, \text{GB} \tag{12}
$$

单条 8K 序列的 KV Cache 即占用 343 GB, 远超单卡 A100(80 GB). 即使采用 8 卡张量并行, 每卡仍需 43 GB, 几乎占满显存. 将 $n$ 扩展至 128K: 

$$
 M_{\text{KV}} = 2 \times 131072 \times 80 \times 8192 \times 16 \times 2 = 5.5 \, \text{TB} \tag{13}
$$

这解释了为什么长上下文推理需要 16-32 卡的多节点并行, 以及为什么上下文压缩在自托管场景中的经济价值甚至高于 API 场景. 

## 7. 极简实现

以下实现一个带分层记忆的上下文管理器, 支持短期记忆的滑动窗口、中期记忆的结构化摘要、以及长期记忆的向量检索. 代码基于 PyTorch 和 NumPy, 可直接嵌入 Agent 框架. 

```python
import numpy as np
from typing import List, Dict, Tuple
from datetime import datetime

class ContextManager:
    """分层记忆上下文管理器, 对应公式 (5)(6)(7)(9)"""
    def __init__(
        self,
        window_size: int = 6,
        max_tokens: int = 4096,
        decay_beta: float = 0.01
    ):
        self.window_size = window_size      # 对应式 (5) 的 k
        self.max_tokens = max_tokens        # 对应式 (7) 的 L
        self.decay_beta = decay_beta        # 对应式 (9) 的 beta
        self.short_term: List[Tuple[str, str]] = []
        self.medium_term: Dict[str, str] = {}
        self.long_term: List[Dict] = []

    def add_turn(self, user_msg: str, assistant_msg: str) -> None:
        """追加对话到短期记忆, 溢出部分转入中期记忆"""
        self.short_term.append((user_msg, assistant_msg))
        if len(self.short_term) > self.window_size:
            evicted = self.short_term.pop(0)
            self._update_medium_memory(evicted)

    def _update_medium_memory(self, turn: Tuple[str, str]) -> None:
        """将淘汰轮次提炼为结构化摘要, 对应式 (6) 的重要性提取"""
        user_msg, _ = turn
        if "偏好" in user_msg or "喜欢" in user_msg:
            key = f"pref_{len(self.medium_term)}"
            self.medium_term[key] = user_msg

    def add_long_term_memory(self, content: str, embedding: np.ndarray) -> None:
        """写入长期记忆, embedding 归一化对应式 (3) 的分母归一化"""
        self.long_term.append({
            "content": content,
            "embedding": embedding / np.linalg.norm(embedding),
            "timestamp": datetime.now()
        })

    def retrieve_relevant_memories(
        self, query_embed: np.ndarray, top_k: int = 3
    ) -> List[str]:
        """检索长期记忆, 直接实现式 (9): score = cos_sim * exp(-beta * delta_t)"""
        if not self.long_term:
            return []
        q_norm = query_embed / np.linalg.norm(query_embed)
        scores, now = [], datetime.now()
        for mem in self.long_term:
            sim = float(np.dot(q_norm, mem["embedding"]))  # 式 (3) 余弦相似度
            delta_t = (now - mem["timestamp"]).days
            decay = np.exp(-self.decay_beta * delta_t)     # 式 (9) 时间衰减
            scores.append(sim * decay)
        top_indices = np.argsort(scores)[-top_k:][::-1]
        return [self.long_term[i]["content"] for i in top_indices]

    def build_context(self, current_query: str, query_embed: np.ndarray) -> str:
        """层次化填充, 优先级对应式 (7): 中期 > 长期 > 短期"""
        parts, used_tokens = [], 0
        if self.medium_term:
            medium_str = "[偏好]\n" + "\n".join(
                f"- {k}: {v}" for k, v in self.medium_term.items()
            )
            parts.append(medium_str)
            used_tokens += len(medium_str.split())
        relevant = self.retrieve_relevant_memories(query_embed, top_k=2)
        if relevant:
            mem_str = "[历史]\n" + "\n".join(f"- {m}" for m in relevant)
            parts.append(mem_str)
            used_tokens += len(mem_str.split())
        recent_str = "[对话]\n"
        for u, a in self.short_term:                     # 式 (5) 滑动窗口
            turn_text = f"User: {u}\nAssistant: {a}\n"
            turn_tokens = len(turn_text.split())
            if used_tokens + turn_tokens > self.max_tokens:
                break                                    # 式 (7) 的截断指示函数
            recent_str += turn_text
            used_tokens += turn_tokens
        parts.append(recent_str)
        parts.append(f"[查询]\n{current_query}")
        return "\n\n".join(parts)
```

代码与公式的映射关系: 

- `self.short_term` 的溢出处理对应式 (5) 的滑动窗口截断. 
- `_update_medium_memory` 对应式 (2) 中 $I(C; Y)$ 的结构化摘要策略. 
- `retrieve_relevant_memories` 中的 `sim * decay` 直接实现式 (9) 的 $\cos(\mathbf{e}_q, \mathbf{e}_{m_i}) \cdot e^{-\beta \cdot \Delta t_i}$. 
- `build_context` 的分层填充逻辑对应式 (7) 的优先级排序贪心策略. 

## 8. 边界条件与失效模式

### 8.1 上下文截断导致的关键信息丢失

**触发条件**: 对话轮次超出滑动窗口容量, 或检索 chunk 的总长度超出 token 预算. 

**物理机制**: 如式 (5) 所示, 滑动窗口策略以时间近邻性作为唯一筛选标准, 完全忽略信息的重要性. 用户在对话第 1 轮声明的约束("预算是 1 万元")可能在第 10 轮因超出窗口而被截断, 模型在第 11 轮生成超出预算的方案. 

**深层根因**: 时间近邻性与信息重要性在统计上并非强相关. 关键约束往往在对话早期建立, 而后续对话围绕该约束展开. 滑动窗口的 FIFO 机制与这一语义结构存在根本性错配. 

**修复方向**: 引入如式 (6) 所示的重要性加权机制, 对包含约束声明、用户确认、否定意见的轮次赋予高 $w_t$, 使其在截断决策中被优先保留. 

### 8.2 多轮对话中的语义漂移和话题跳跃

**触发条件**: 用户在多轮对话中自然切换话题, 或模型对早期话题的"执着"导致回复偏离用户当前意图. 

**物理机制**: 如式 (1) 所示, Transformer 的注意力机制对所有历史 token 赋予非零权重. 即使当前查询与早期话题无关, 早期 token 仍会通过注意力分数对当前生成产生微弱但系统性的偏置. 累积多轮后, 这种偏置导致模型"过度关联"历史话题. 

**深层根因**: 模型的上下文感知能力缺乏"话题边界检测"机制. 人类对话中, 话题切换通常伴随明确的信号("换个话题"、"另外"), 但模型无法自动识别这些信号并隔离不相关的历史上下文. 

**修复方向**: 在对话管理中引入话题分割模块, 检测话题切换信号, 当话题发生跳跃时, 将旧话题的上下文按式 (9) 的向量表示归档至长期记忆, 仅在当前话题的短窗口内保留相关轮次. 

### 8.3 Lost in the Middle

**触发条件**: 上下文长度超过 4K-8K 时, 模型对序列中段信息的提取能力显著下降. 

**物理机制**: Xiao et al. (2023) 的实证研究表明, 当正确答案位于长文档的中间位置时, 模型的准确率相比位于首尾时下降 20-40 个百分点. 如式 (1) 所示, 注意力计算的二次复杂度导致首尾位置的 token 获得更高的平均注意力权重, 中段 token 被"淹没". 

**深层根因**: 位置编码的远端衰减与注意力权重的 softmax 归一化共同作用. 在极长序列中, 中段位置的位置编码信号相对弱化, 同时 softmax 的归一化效应使得数量众多的中段 token 分摊了有限的注意力"质量". 

**修复方向**: 

1. **重排序注入**: 如式 (8) 所示, 计算检索 chunk 的边际语义贡献 $\Delta_j$, 将高贡献 chunk 置于 prompt 的首尾两端, 低贡献 chunk 置于中段. 
2. **分步检索**: 不一次性注入所有 chunk, 而是先注入 top-1 chunk 生成初始答案, 再基于答案与剩余 chunk 的匹配度做二次 refine. 

### 8.4 记忆检索的时效性与相关性权衡

**触发条件**: 长期记忆中存在语义相关但已过时的事实. 

**物理机制**: 如式 (9) 所示, 时间衰减因子 $e^{-\beta \cdot \Delta t}$ 引入了一个超参数 $\beta$. $\beta$ 过大导致新鲜但弱相关的记忆被优先选择; $\beta$ 过小导致旧但强相关的记忆持续占据检索结果. 最优的 $\beta$ 值高度依赖于具体领域的知识更新速度——技术文档的 $\beta$ 应大于文学作品. 

**深层根因**: 时间衰减模型(指数衰减)是对人类遗忘曲线的简化近似, 真实场景中的信息时效性并非单调递减. 某些事实(如用户的出生地)是静态的, 不应随时间衰减; 某些事实(如软件版本信息)在更新后应立即失效. 

**修复方向**: 为记忆引入类型标签(静态事实 / 动态事实 / 版本信息), 对不同类型应用不同的衰减函数, 或完全禁用静态事实的时间衰减. 

## 9. 技术前瞻

**上下文窗口的物理极限与架构变革**

当前上下文窗口的扩展受限于 $O(n^2)$ 的注意力复杂度. 线性注意力(Linear Attention)、状态空间模型(Mamba)、以及 RWKV 等替代架构试图将复杂度降至 $O(n)$. 若这些架构在保持 Transformer 级别生成质量的同时实现线性复杂度, 128K 乃至 1M 上下文的实时推理将成为常态, Context Engineering 的焦点将从"压缩与筛选"转向"组织与导航". 

**自适应上下文学习**

未来的模型可能具备"上下文感知元学习"能力: 在推理过程中动态判断哪些历史信息需要保留、哪些可以丢弃, 而非依赖外部的上下文管理器. 这类似于人类的工作记忆机制——大脑自动将无关信息转移至长期记忆或完全遗忘. 实现这一能力需要在训练阶段引入上下文长度变化的课程学习, 以及特殊的损失函数激励模型进行信息筛选. 

**多模态上下文的统一编排**

当前 Context Engineering 主要聚焦文本 token. 随着 GPT-4o、Gemini 等原生多模态模型的普及, 上下文将同时包含图像、音频、视频的时间序列. 不同模态的信息密度差异巨大: 一张 1024x1024 的图像在 ViT 编码后仅产生 256 个 patch token, 而同等信息量的文本可能需要数千个 token. 多模态上下文编排需要在跨模态的"信息-Token 效率比"上做新的工程权衡. 

**可解释的上下文决策**

生产系统需要解释"为什么模型选择了这些上下文". 当前的选择性保留和动态检索多为黑盒操作. 未来的 Context Engineering 框架需要提供可解释性接口: 展示每段被保留/丢弃上下文的具体理由(相关性分数、重要性评分、时间衰减因子), 以支持调试、合规审计和用户对系统的信任建立.
