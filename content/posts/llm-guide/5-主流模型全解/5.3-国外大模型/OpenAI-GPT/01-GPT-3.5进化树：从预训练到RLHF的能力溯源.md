## 1. 演进脉络：从 GPT-3 到 ChatGPT 的进化树

### 1.1 GPT-3.5 家族谱系

```
2020.07  davinci (初代 GPT-3, 175B)
    │
    ├── 2021.07  code-cushman-001 (12B, 代码微调)
    │
    ├── 2022.03  davinci-instruct-beta (指令微调初版)
    │       └── 2022.04  text-davinci-001
    │
    └── 2022.04  code-davinci-002 (最强基础模型)
            │
            ├── 2022.05  text-davinci-002 (SFT 指令微调)
            │       └── 2022.11  text-davinci-003 (SFT + RLHF)
            │
            └── 2022.11  ChatGPT (SFT + RLHF, 对话优化)
```

**核心结论**：指令微调和 RLHF **不注入新能力，只解锁已有能力**. 所有能力在 code-davinci-002 阶段已存在，后续调整是"能力分配"而非"能力创造". 

### 1.2 能力来源总览

| 能力 | 来源阶段 | 关键数据/技术 | 代表模型 |
|------|---------|-------------|---------|
| 语言生成 | 预训练 | 3000亿 token 语言建模 | davinci |
| 世界知识 | 预训练 | C4 + WebText2 + Books + Wikipedia | davinci |
| 上下文学习 (ICL) | 预训练 | 同任务数据在 batch 中连续排列 | davinci |
| 遵循指令 | 指令微调 | 77K 人工标注指令 | text-davinci-001 |
| 零样本泛化 | 指令微调 | 指令数量超过临界规模 | text-davinci-002 |
| 复杂推理/CoT | **代码训练** | 159G 代码数据 (28% 总数据) | code-davinci-002 |
| 对话能力 | RLHF | 人类偏好排序 + PPO | ChatGPT |
| 安全/拒绝 | RLHF | 安全标注 + 拒绝训练 | ChatGPT |

## 2. 初代 GPT-3：预训练的三重能力

### 2.1 基础能力矩阵

初代 GPT-3 (davinci) 展示了三种核心能力：

1. **语言生成**：遵循 prompt 生成补全——来自语言建模目标
2. **上下文学习 (In-Context Learning)**：给定几个示例后泛化到新用例——来源尚不明确，可能与训练时同任务数据在 batch 中连续排列有关
3. **世界知识**：事实性知识 + 常识——来自 3000 亿单词的训练语料

### 2.2 初代 GPT-3 有多强？

**悖论**：初代 GPT-3 在今天看来"不咋地"(许多任务不如 T5)，但后来的实验证明它蕴含了**极强的潜力**. 

**证据**：
- Meta 开源的 OPT 试图复现初代 GPT-3，能力与当今标准形成尖锐对比
- 但 code-davinci-002 及后续模型解锁了这些潜力，证明能力已存在于基础模型中

**关键参数**：
- 模型大小：1750 亿参数(主要用于存储知识)
- 训练数据：3000 亿 token(60% C4 + 22% WebText2 + 16% Books + 3% Wikipedia)
- 上下文窗口：2048

## 3. 代码训练：复杂推理的"神奇副产物"

### 3.1 Code-Davinci-002 的关键突破

code-davinci-002 是第一版深度融合代码训练和指令微调的模型，展现出初代 GPT-3 不具备的三种能力：

1. **响应人类指令**：生成符合指令要求的答案(而非训练集中的常见句子)
2. **泛化到未见过任务**：在新指令上自动生成有效回答
3. **利用思维链 (Chain-of-Thought) 进行复杂推理**：逐步推导解决数学、逻辑问题

### 3.2 为什么代码训练催生推理能力？

**相关性证据**(非因果证明)：

| 模型 | 代码训练数据 | 思维链能力 |
|------|-------------|-----------|
| 初代 GPT-3 | 无 | 无 |
| text-davinci-001 | 无 | 极弱 |
| code-cushman-001 (12B) | 有 | 有 |
| PaLM (540B) | 5% | 有 |
| code-davinci-002 | 28% | **强** |

**直觉解释**：
- **面向过程编程** $\approx$ 人类逐步解决任务的过程
- **面向对象编程** $\approx$ 将复杂任务分解为多个简单任务
- **代码的长距离依赖**：括号匹配、函数引用需要比自然语言更长的上下文

### 3.3 代码训练 vs 指令微调的分工

| 能力 | 代码训练 | 指令微调 |
|------|---------|---------|
| 复杂推理/CoT | **主因** | 副作用 |
| 遵循指令 | 无 | **主因** |
| 零样本泛化 | 无 | **主因**(需足够数据量) |
| 长距离依赖 | **有** | 无 |

**数据量对比**：
- 指令微调数据：~77K(预训练的 0.004%)—— 只能解锁能力，无法注入
- 代码训练数据：159G(预训练的 28%)—— 足够大，可能直接注入能力

## 4. RLHF：对齐人类价值观的"催化剂"

### 4.1 RLHF 触发的四种能力

| 能力 | 表现 | 机制 |
|------|------|------|
| **翔实回应** | 回答更长、更详细 | 人类偏好长答案 |
| **公正回应** | 多方利益事件中给出平衡观点 | 安全标注训练 |
| **拒绝不当问题** | 拒绝有害/偏见请求 | 内容过滤器 + RLHF |
| **拒绝知识范围外问题** | "我不知道" | **最神奇：模型隐式区分已知/未知** |

### 4.2 对齐税 (Alignment Tax)

**现象**：指令微调后的模型在基准测试上性能**低于** code-davinci-002. 

**原因**：为了生成符合人类期望的回答，模型牺牲了部分原始能力：

| 模型 | 上下文学习 | 零样本能力 | 对话能力 |
|------|-----------|-----------|---------|
| code-davinci-002 | **最强** | 强 | 无 |
| text-davinci-002 | 减弱 | **增强** | 无 |
| text-davinci-003 | 恢复部分 | 进一步增强 | 弱 |
| ChatGPT | **极弱** | 强 | **最强** |

**ChatGPT 的特殊取舍**：牺牲几乎所有上下文学习能力，换取：
- 建模对话历史
- 增加对话信息量
- 拒绝知识范围外问题

### 4.3 RLHF 的局限

**关键洞察**：RLHF 不注入"知道不知道"的能力——这种能力已存在于基础模型中，RLHF 只是**解锁**了表达它的方式. 

**证据**：RLHF 的数据量/计算量远小于预训练，不可能注入全新的元认知能力.

## 5. GPT-3.5 的"不能"

### 5.1 形式推理的边界

GPT-3.5 **不能**进行严格的形式推理：

- **数学证明**：中间步骤不能跳、不能模糊、不能错
- **一阶逻辑**：严格的符号推演

**模糊性光谱**：

| 类型 | 示例 | GPT-3.5 表现 |
|------|------|-------------|
| (a) 完全模糊 | 开放式创作 | 优秀 |
| (b) 有点逻辑 | 烹饪步骤、证明思路 | 良好 |
| (c) 严格无歧义 | 三位数加法、定理证明 | **失败** |

**务实方案**：将严格推理卸载到符号系统(如 Python)，语言模型负责高层规划. 

### 5.2 实时知识更新

- **模型信念顽固**：错误信念难以纠正(如坚持 3599 是质数)
- **知识有时间截断**：无法回答训练截止日期后的事件
- **无互联网检索**：虽然 WebGPT 已验证该能力，但未在 GPT-3.5 中开放

**未来方向**：将知识卸载到外部检索系统，模型专注推理，参数可大幅缩减.

## 6. 对开源复现的启示

### 6.1 能力复现路线图

```
Phase 1: 预训练
  └─→ 需要足够大的基础模型(≥100B)
  └─→ 语料需包含代码(≥5%)
  └─→ 上下文学习能力可能来自 batch 构造

Phase 2: 代码训练
  └─→ 代码数据量需达到显著比例(~30%)
  └─→ 面向过程和面向对象代码都有益

Phase 3: 指令微调
  └─→ 数据量不需很大(~100K 级)，但需覆盖多样任务
  └─→ 关键是指令数量超过临界规模，而非单个指令的质量

Phase 4: RLHF
  └─→ 需要高质量的人类偏好数据
  └─→ 对齐税不可避免，需在性能和安全间权衡
```

### 6.2 关键假设与验证方向

| 假设 | 证据强度 | 验证方法 |
|------|---------|---------|
| 代码训练是 CoT 的主因 | 中(相关性) | 对比 T5 vs FlanT5 vs CodeT5 |
| 指令微调只解锁不注入 | 强 | 数据量分析 + 消融实验 |
| code-davinci-002 基于更大模型 | 弱(猜测) | 对比 OPT-175B 微调效果 |
| 上下文学习来自 batch 构造 | 中 | 分析预训练数据的任务聚类 |

## 7. 边界条件与失效模式

| 场景 | 症状 | 根因 | 缓解 |
|------|------|------|------|
| 对齐过度 | 模型拒绝回答无害问题 | 安全训练过于保守 | 调整 RLHF 奖励函数的拒绝阈值 |
| 对齐不足 | 生成有害/偏见内容 | 安全数据覆盖不全 | 增加对抗性安全训练 |
| 知识幻觉 | 编造事实但语气自信 | 参数化知识的固有缺陷 | 结合 RAG 检索增强 |
| 推理跳步 | 数学问题中间步骤错误 | 语言模型不擅长严格符号推演 | 使用外部工具(Python/计算器) |
| 上下文遗忘 | 长对话中丢失早期信息 | 上下文窗口有限 + 注意力稀释 | 摘要机制 + 外部记忆 |
