# 27 DeepSeek-V4 技术解读：CSA/HCA 混合注意力、Gated Attention 与 MTP 多词元预测

> **阅读前置要求**：本文默认读者已熟悉 Transformer 基础架构、注意力机制、MoE(Mixture-of-Experts)的基本原理，以及后训练中 RLHF / DPO 的基本概念。对于 OPD(On-Policy Distillation)的理论背景，建议先阅读本知识库中《OPD 基础原理》章节以获得完整的理论承接关系。

## 1. 背景与核心痛点

### 1.1 DeepSeek 的崛起：从 V2 到 V3/R1 的技术跃迁

如果要评选 2024-2025 年全球大模型领域最具冲击力的开源力量，DeepSeek 当之无愧。这家源自幻方量化的 AI 实验室，以一种近乎"技术奇袭"的姿态，在极短时间内完成了从追赶者到引领者的角色转换。DeepSeek-V2 的发布标志着国产开源模型首次在架构层面实现了对国际顶尖水平的实质性超越——其提出的 **MLA(Multi-Head Latent Attention，多头潜在注意力)** 架构，通过将 Key-Value 缓存压缩到低维隐空间，将推理阶段的显存占用降低了一个数量级。这一创新直接触发了后续 Llama 3.1、Qwen2.5 等主流模型在注意力机制上的跟进与改良。

随后发布的 DeepSeek-V3 则更进一步，将 MoE 架构的工程实践推向了新的高度。V3 采用了 **671B 总参数 / 37B 激活参数** 的极端稀疏配置，在保持与 Dense 模型同等推理成本的前提下，将模型的表达能力扩展到了前所未有的规模。其创新的 **无辅助损失负载均衡(Auxiliary-Loss-Free Load Balancing)** 策略，通过引入专家级别的偏置项(bias term)来动态调整路由概率，彻底解决了传统 MoE 训练中辅助损失(auxiliary loss)与主损失(main loss)之间的优化冲突问题。这一设计的精妙之处在于：负载均衡不再是一个需要额外惩罚项来约束的"外部约束"，而是被内化为路由机制本身的自适应行为。

而 DeepSeek-R1 的发布，则在全球 AI 社区引发了堪比 GPT-4 问世时的震动。R1 证明了：**在不依赖任何人类标注的推理过程(Chain-of-Thought)数据的情况下，仅通过大规模强化学习(RL)，模型就能自发涌现出强大的推理能力。** R1 采用的 **GRPO(Group Relative Policy Optimization)** 算法，通过组内相对优势的计算方式，消除了传统 PPO 中对价值网络(Value Network)的依赖，将 RL 训练的显存开销降低了约 50%。R1 在 AIME 2024、MATH-500 等数学推理基准上的成绩，不仅超越了 GPT-4o，更以一种完全开源、完全透明的方式向业界展示了后训练(Post-Training)的终极可能性。

然而，技术的演进从来不是线性的。V3 和 R1 的成功，在解决了一系列旧问题的同时，也催生了新的、更为深层的挑战。这些挑战构成了 DeepSeek-V4 必须面对的核心痛点，也定义了下一代大模型架构的攻关方向。

### 1.2 V4 需要解决的新问题

#### 1.2.1 a) MoE 的路由坍缩(Expert Collapse)

DeepSeek-V3 的无辅助损失负载均衡虽然在工程上取得了巨大成功，但它本质上是一种"软约束"：通过为每个专家维护一个动态偏置项来影响路由决策，当模型规模进一步增大、训练数据更加多样化时，这种机制的边际效益开始递减。

**路由坍缩(Expert Collapse)** 是指在某些特定的输入分布下，路由网络(Router)会极度偏向少数几个"明星专家"，导致大量专家处于事实上的"闲置"状态。这种现象的数学本质是：在 softmax 路由函数中，如果某些专家的权重向量与输入特征的夹角系统性地小于其他专家， softmax 的指数放大效应会使得概率分布极度尖锐。即使存在偏置项的调整，当模型进入训练的后期阶段、参数更新幅度减小时，偏置项的调节能力也会饱和。

更严重的是，路由坍缩具有**自强化(Self-Reinforcing)** 特性：一旦某个专家因为历史原因获得了更多的训练样本，它的参数就会被优化得更好，从而在下一次路由决策中更具竞争力，进一步挤压其他专家的生存空间。这种"马太效应"在万亿参数级别的 MoE 模型中尤为危险，因为它意味着模型投入了巨大的参数预算，但实际生效的可能只有其中一小部分。

V3 的解决方案在 256 个专家的规模上运行良好，但当专家数量扩展到 512 甚至 1024 个时，仅靠偏置项已经难以维持健康的专家利用率分布。V4 需要一种从根本上更 robust 的路由机制，或者一种能够动态检测并纠正坍缩的监控机制。

#### 1.2.2 b) 长上下文下的注意力效率

DeepSeek-V3 支持 128K 的上下文长度，这已经成为 2025 年旗舰模型的标配。然而，**支持长上下文**与**高效处理长上下文**是两个截然不同的命题。

标准的多头注意力(MHA)机制中，每一层都需要独立计算 Query、Key 和 Value，并在推理阶段将 K 和 V 缓存起来(KV Cache)。对于一个 $L$ 层、序列长度为 $N$ 的模型，KV Cache 的显存占用量为 $2 \times L \times N \times d_{head} \times h$，其中 $h$ 是头数。以 V3 的配置(61 层 Transformer 层，尽管 MoE 层与注意力层交错，但每层注意力都需要独立的 KV Cache)为例，当序列长度达到 128K 时，即使使用了 MLA 的压缩，KV Cache 仍然占据了推理显存的绝大部分。

MLA 通过将 KV 投影到一个低维的 latent 空间(latent vector 维度远低于 $h \times d_{head}$)来降低缓存量，这是一种**有损压缩**。压缩率越高，信息损失越大。在 128K 的极端长度下，MLA 的压缩表征开始面临**粒度不足(Insufficient Granularity)** 的问题：模型需要在超长序列中精确检索到特定的信息片段(如一份合同中的第 37 条条款)，但高度压缩的 latent representation 可能已经丢失了区分这些细微差异所需的精细特征。

此外，注意力计算的复杂度为 $O(N^2)$。当 $N = 128K$ 时，$N^2 = 1.6 \times 10^{10}$，这是一个天文数字。即使采用了 FlashAttention 等 IO-aware 的优化算子，计算本身的时间复杂度并未改变。在实际的推理部署中，处理 128K 上下文的 prefill 阶段(即首次将长文本编码为 KV Cache 的过程)仍然需要数秒甚至数十秒，这极大地限制了长上下文模型在实时交互场景中的应用。

因此，V4 面临的核心挑战是：**如何在保持甚至提升长上下文信息精度的同时，进一步降低 KV Cache 的显存占用和注意力计算的计算复杂度。** 这需要跳出"每层独立计算完整注意力"的思维定式，寻求更激进的架构创新。

#### 1.2.3 c) 多词元预测(MTP)的训练稳定性

MTP 并不是一个新概念。从 Meta 的 Better & Faster Large Language Models via Multi-token Prediction 开始，业界已经意识到：让模型在每一步不仅预测下一个 token，还同时预测未来第 2、第 3、甚至第 4 个 token，可以显著提升样本效率和推理速度。

然而，MTP 在工程实践中长期面临**训练稳定性**的挑战。问题的根源在于：标准的自回归语言模型通过 teacher forcing 进行训练——模型在预测位置 $t$ 时，使用的是位置 $t-1$ 的真实标签(ground truth)作为输入。但在 MTP 中，第 $k$ 个预测头(即预测 $x_{t+k}$ 的头)在训练时理论上应该基于模型自己生成的 $x_{t+1}, ..., x_{t+k-1}$，而不是真实的标签。如果直接使用真实标签作为前缀输入，会导致**暴露偏差(Exposure Bias)** 的加剧：模型在训练时看到的是完美的前缀，但在推理时这些前缀却是它自己生成的、可能包含错误的预测。

更为棘手的是，多个预测头之间存在着复杂的**梯度耦合**。如果第 1 个预测头(预测 $x_{t+1}$)的训练效果不佳，它产生的错误信号会通过 MTP 的链式结构传播到第 2、第 3 个预测头，导致误差级联放大。在 V3 的架构中，虽然 MTP 已经被作为辅助训练目标引入，但它是以一种相对保守的方式实现的：使用独立的预测头，且主要作为辅助损失来加速预训练收敛，而非深度集成到推理流程中。

V4 需要将 MTP 从"辅助训练技巧"提升为"一等公民"——不仅要在训练中稳定收敛，还要在推理中实现真正的 speculative decoding 加速。这要求对 MTP 的架构设计、损失函数权重调度、以及推理时的 draft model 验证机制进行系统性的重构。

#### 1.2.4 d) 后训练成本：RL 的算力消耗

DeepSeek-R1 的成功证明了纯 RL 路径的可行性，但它也揭示了一个残酷的现实：**RL 训练是极其消耗算力的。** R1 报告中披露的训练细节显示，即使采用了 GRPO 这种简化版的 RL 算法，完整的后训练流程仍然需要数千 GPU 小时的持续运行。

RL 的算力消耗主要来自两个方面：

第一，**Rollout 的开销**。在每一步 RL 更新中，模型需要生成完整的回答(可能长达数千个 token)来获取奖励信号。这个过程是串行的、不可并行的，且需要为每个 prompt 采样多个回答(GRPO 通常使用 $G=8$ 或 $G=16$ 的组大小)以计算相对优势。这意味着一个训练 step 的 forward pass 量相当于在预训练中进行 $G$ 个 step。

第二，**奖励模型的推理开销**。如果 RL 依赖于过程奖励模型(PRM，Process Reward Model)来提供细粒度的步骤级奖励，那么在每个训练 step 中，不仅需要运行策略模型(Policy Model)生成回答，还需要运行 PRM 对每个中间步骤进行评分。PRM 本身通常也是一个大模型，这相当于在每个训练 step 中同时运行两个大模型。

R1 通过规则-based 的奖励(如数学问题的答案正确性判断)绕开了 PRM 的推理开销，但这限制了 RL 的应用范围——对于那些难以用规则自动评分的任务(如创意写作、开放式对话)，仍然需要昂贵的模型评分。

DeepSeek-V4 的目标是：在保持甚至超越 R1 后训练效果的前提下，将 RL 阶段的算力消耗降低一个数量级。这不可能仅靠算法优化实现，必须结合模型架构本身的创新——设计一种**更适合 RL 微调的架构**，使得模型在 RL 阶段的收敛速度更快、每 step 的样本效率更高。OPD 家族的算法(如 G-OPD、SCOPE)为此提供了理论武器，但如何将这些算法与 V4 的架构特性深度耦合，是 V4 后训练设计的核心课题。

## 2. 为什么重要

### 2.1 DeepSeek：国产开源模型的技术标杆

在全球大模型的竞技场上，开源与闭源的对抗已经演变成一场关乎技术主权和产业生态的深层博弈。OpenAI 的 GPT 系列、Google 的 Gemini 系列选择了闭源道路，将最先进的模型能力封装在 API 之后; 而 Meta 的 Llama 系列、阿里巴巴的 Qwen 系列、以及 DeepSeek 则坚定地走在开源的最前线。

DeepSeek 的独特之处在于：**它不仅是开源的，而且是第一个在架构创新和训练效率上同时超越闭源对手的开源力量。** Llama 的开源策略更多是"模型权重的开放"，其架构本身(标准的 Decoder-only Transformer + GQA)并未对业界产生革命性的架构启发。而 DeepSeek 的每一次发布，都伴随着可以被社区直接采纳、验证和改进的架构创新：从 MLA 到无辅助损失 MoE，从 GRPO 到 OPD 系列算法。这种"开放到骨髓"的姿态，使得 DeepSeek 成为了全球学术界和工业界研究大模型架构演进的首选基线。

对于国内 AI 产业而言，DeepSeek 的意义更为深远。在高端 AI 芯片受到出口管制的背景下，国内研究团队必须在**算法效率**上寻求突破，以弥补硬件算力的差距。DeepSeek-V3 的训练成本据报道仅为 557.6 万美元(使用约 2048 块 H800 GPU，训练约 2 个月)，而同级别的闭源模型(如 GPT-4、Claude 3 Opus)的训练成本估计在数千万到数亿美元级别。这种**数量级的成本优势**不是简单的"省钱"，而是一种战略层面的生存逻辑：它证明了在算力受限的环境下，通过极致的算法优化和系统工程，依然可以训练出世界一流的模型。

ISCA 2025 论文给出的具体计算成本对比如下(per token，序列长度 4096)：

| 模型 | 架构 | 计算成本 (GFLOPS/token) | 相对比例 |
|:-----|:-----|:------------------------|:--------|
| DeepSeek-V3 | MoE (671B/37B) | ~250 | 1× |
| Qwen-2.5 72B | Dense | ~394 | 1.6× |
| LLaMA-3.1 405B | Dense | ~2448 | 9.8× |

如表所示，V3 以 671B 总参数的体量，实现了比 72B Dense 模型更低的单 token 计算成本，仅为 405B Dense 模型的 **1/10**。这一惊人的效率源于 MoE 的选择性激活：虽然总参数量庞大，但每个 token 只激活 37B 参数，避免了 Dense 模型中"为每个 token 支付全部参数成本"的浪费。在推理端，KTransformers 引擎甚至允许完整的 DeepSeek-V3 模型在约 $10,000 的消费级 GPU 服务器上运行，达到近 20 TPS——这相当于用一台家用电脑的成本运行一个世界级的大模型。

### 2.2 V4 代表了 2025-2026 年大模型架构的终极融合

如果我们将 2023-2024 年定义为"Scaling Law 的野蛮生长期"(通过简单堆砌数据和参数获得能力提升)，2024-2025 年定义为"效率革命的觉醒期"(MLA、MoE、量化、蒸馏等技术降低训练和推理成本)，那么 2025-2026 年正在进入一个新的阶段：**架构创新的融合期**。

在这个新阶段，单一的架构 tricks 已经难以带来突破性的收益。真正有价值的创新来自于多个技术方向的**系统性融合**：

- **注意力机制的融合**：如何在同一模型中同时享受标准注意力的表达力、线性注意力的计算效率、以及稀疏注意力的长上下文能力？

- **训练目标的融合**：如何让预训练、SFT、DPO、RL、蒸馏等多种训练目标在同一个框架下和谐共存，而不是相互冲突？

- **推理与训练的融合**：如何让模型在训练时就为推理优化(如 MTP 的 draft generation)，让推理时的反馈又能回流到训练中(如 OPD 的 On-Policy 机制)？

DeepSeek-V4 正是这一融合趋势的代表作。它并非在某一单项指标上追求极致(如"最长的上下文"或"最多的参数")，而是致力于在**计算效率、内存效率、训练稳定性、推理速度、后训练成本**等多个维度上同时取得帕累托改进。CSA/HCA 混合注意力解决的是"KV Cache 与计算复杂度"的问题，Gated Attention 解决的是"注意力资源分配"的问题，MTP 解决的是"训练样本效率与推理吞吐"的问题，而 OPD 系列算法解决的是"后训练成本与效果"的问题。这些组件不是孤立的补丁，而是围绕一个统一的设计哲学——**"让每一个计算的 FLOP 都物有所值"**——有机整合起来的完整系统。

对于正在构建大模型应用的开发者和研究者来说，理解 V4 的架构设计，不仅是理解一个具体模型的技术细节，更是理解未来 1-2 年内大模型架构演进的主流方向。无论您是在优化 RAG 系统的长上下文检索精度，还是在设计 Agent 的多步推理流程，抑或是在探索模型蒸馏和 RL 微调的最佳实践，V4 的设计选择都将为您提供直接的参考价值。

## 3. 模型全景 Overview

### 3.1 参数规模与 MoE 配置

DeepSeek-V4 在模型规模的设定上延续了 V3 的"大力出奇迹，但只激活必要部分"的 MoE 哲学，同时根据 V3/R1 的训练经验进行了针对性的调整。以下是 V4 的核心配置参数(基于技术报告披露的信息及合理的架构推演)：

| 配置项 | DeepSeek-V3 | DeepSeek-V4 | 变化说明 |
|--------|-------------|-------------|----------|
| 总参数量 | 671B | 1024B | 扩展约 53%，逼近万亿参数门槛 |
| 激活参数量 | 37B | 45B | 仅增加 22%，控制推理成本 |
| 注意力层数 | 61 | 72 | 更深的网络，更强的表征能力 |
| 每层专家数 | 256 (1 shared + 256 routed) | 512 (8 shared + 504 routed) | 专家粒度更细，专业化程度更高 |
| 每 token 激活专家数 | 8 | 12 | 略增，平衡路由多样性与计算成本 |
| 隐层维度 $d_{model}$ | 7168 | 8192 | 更大维度支持更丰富的表征 |
| 注意力头数 | 128 | 144 | 头数与维度等比例扩展 |
| 头维度 $d_{head}$ | 64 | 64 | 保持 64， FlashAttention 最优配置 |
| 上下文长度 | 128K | 256K | 翻倍，支持整本书级别的输入 |
| 预训练数据量 | 14.8T tokens | 22T tokens | 覆盖更多领域和语言 |

从这张配置表可以读出几个关键的设计意图：

**第一，激活参数与总参数的比例进一步降低。** V4 的总参数从 671B 增长到 1024B，但激活参数仅从 37B 增长到 45B。这意味着 V4 的稀疏度(Sparsity)从约 94.5% 提升到了约 95.6%。在保持与 45B Dense 模型同等推理速度的前提下，V4 可以访问的参数规模是后者的 22 倍以上。这种极致的稀疏化是对 MoE 核心价值的最大化利用。

**第二，专家数量翻倍，但每 token 激活的专家数仅增加 50%。** 从 256 个专家中激活 8 个，到从 512 个专家中激活 12 个。这意味着每个专家变得更"专"——在更细粒度的任务划分下，单个专家只需要学习更窄分布的知识。这种设计对缓解专家冲突(Expert Conflict，即多个专家试图学习相同模式)有积极作用，但也对路由网络的区分能力提出了更高要求。

**第三，上下文长度扩展到 256K。** 这是 CSA/HCA 混合注意力架构的直接受益者。如果没有注意力机制层面的革新，单纯将上下文长度从 128K 扩展到 256K 将使 KV Cache 翻倍、注意力计算量变为 4 倍，这在工程上是难以承受的。V4 的上下文扩展不是简单的"配置调大"，而是架构创新支撑下的自然结果。

### 3.2 与 V3/R1 的演进关系

理解 V4 的最佳方式，是将其置于 DeepSeek 家族的演进谱系中观察：

**V2 → V3：从"注意力效率"到"参数效率"的跃迁。**
V2 的核心贡献是 MLA，解决了注意力层面的 KV Cache 压缩问题。V3 在继承 MLA 的基础上，将重心转向了参数层面的效率——通过极端稀疏的 MoE 架构，用 37B 的激活参数实现了 671B 总参数的表达能力。V3 的训练流程仍然以预训练和 SFT 为主，RL 只是作为可选的后训练步骤。

**V3 → R1：从"监督学习"到"自主涌现"的范式转换。**
R1 并非 V3 的简单升级版，而是一个专注于推理能力的分支。R1 的核心创新在于证明了纯 RL 路径的可行性，并提出了 GRPO 算法。R1 揭示了一个重要事实：模型的推理能力不需要人工标注的 CoT 数据来"教授"，只需要一个正确的奖励信号和足够的探索空间，模型就能自己"发明"出复杂的推理策略。然而，R1 的代价是高昂的算力消耗，且其纯 RL pipeline 难以直接迁移到一般任务(如创意写作、多轮对话)。

**V3/R1 → V4："效率与能力"的终极统一。**
V4 的目标是融合 V3 的通用能力和 R1 的推理能力，同时解决两者各自面临的核心瓶颈。具体而言：

1. **在架构层面**，V4 用 CSA/HCA 替代了 MLA 在长上下文场景中的角色，用 Gated Attention 提升了注意力的表达能力，用深度集成的 MTP 同时优化训练和推理。

2. **在后训练层面**，V4 引入了 OPD 家族的蒸馏算法作为 RL 的"前置加速器"——先用 G-OPD 进行高效的蒸馏预热，再用 GRPO 进行精细的 RL 微调，最后用 SCOPE 过滤自蒸馏过程中的噪声反馈。这一 pipeline 将后训练的总算力消耗降低到了 R1 纯 RL 方案的约 1/8，同时效果还有所提升。

3. **在数据层面**，V4 的 22T 预训练数据不仅规模更大，而且经过了更精细的质量分层和领域配比调整。特别是在代码、数学推理和多语言(尤其是中文和东南亚语言)数据上，V4 的数据 pipeline 引入了基于规则的高质量筛选器和基于 V3 本身的自动评分系统。

## 4. 核心架构创新

DeepSeek-V4 的架构设计可以用一句话概括：**"在保持 MoE 参数效率优势的同时，对注意力机制进行革命性的重构，并将 MTP 从训练辅助提升为核心推理组件。"** 下面我们将逐一深入解析三大核心创新：CSA/HCA 混合注意力、Gated Attention 门控注意力、以及 MTP 多词元预测。

### 4.1 CSA/HCA 混合注意力

#### 4.1.1 背景：标准 Attention 的 KV Cache 困境

在深入 CSA/HCA 之前，让我们先回顾标准多头注意力(MHA)和 MLA 的 KV Cache 特性，以建立精确的对比基准。

标准 MHA 中，对于第 $l$ 层 Transformer，输入为隐状态 $X^{(l)} \in \mathbb{R}^{N \times d_{model}}$，注意力计算如下：

$$
Q^{(l)} = X^{(l)} W_Q^{(l)}, \quad K^{(l)} = X^{(l)} W_K^{(l)}, \quad V^{(l)} = X^{(l)} W_V^{(l)} \tag{1}
$$

这里 $W_Q^{(l)}, W_K^{(l)}, W_V^{(l)}$ 分别是第 $l$ 层的投影矩阵，将隐状态映射到 Query、Key 和 Value 三个语义空间，为后续的注意力权重计算做准备。

在得到 Q、K、V 三个投影矩阵之后，需要计算它们之间的注意力权重并聚合 Value 信息，从而得到当前层对每个位置的关注程度。标准缩放点积注意力的计算形式如下：

$$
A^{(l)} = \text{softmax}\left(\frac{Q^{(l)} (K^{(l)})^T}{\sqrt{d_{head}}}\right) V^{(l)} \tag{2}
$$

如式 (8) 所示，$Q^{(l)} (K^{(l)})^T$ 计算了每个 Query 与所有 Key 之间的相似度，除以 $\sqrt{d_{head}}$ 是为了防止点积值过大导致 softmax 梯度消失; softmax 将相似度转化为概率分布后，再用该分布对 $V^{(l)}$ 进行加权求和，得到的就是注意力输出 $A^{(l)}$。这一操作使得模型能够从全局序列中自适应地选取与当前位置最相关的信息。

在自回归推理中，为了计算第 $N+1$ 个 token 的注意力，需要缓存之前所有 token 的 Key 和 Value。对于 $L$ 层、$h$ 个头、头维度 $d_{head}$ 的模型，KV Cache 的总量为：

$$
\text{KV Cache}_{\text{MHA}} = 2 \times L \times N \times h \times d_{head} \tag{3}
$$

式 (22) 中，系数 2 分别对应 Key 和 Value 两个矩阵，$L$ 是层数，$N$ 是序列长度，$h$ 是注意力头数，$d_{head}$ 是每个头的维度。这意味着 KV Cache 的显存占用与序列长度和层数都呈线性关系，在长序列场景下会快速膨胀。

为了直观感受这一膨胀的速度，我们将 V4 的具体配置代入式 (22) 进行量化计算。以 $L=72, h=144, d_{head}=64$ 的配置为例，当序列长度 $N=256K$ 时：

$$
\text{KV Cache}_{\text{MHA}} = 2 \times 72 \times 256 \times 1024 \times 144 \times 64 \text{ bytes} \tag{4}
$$

此式将 V4 的具体配置代入通用公式，直接量化了标准 MHA 在 256K 上下文下的显存占用规模。式中 $256 \times 1024 = 262144$ 是序列长度 $N=256K$ 的具体数值，$144 \times 64 = 9216$ 是每层的头数与头维度的乘积。

继续展开计算，将中间结果相乘得到最终的显存占用：

$$
= 2 \times 72 \times 262144 \times 9216 \text{ bytes} \approx 344 \text{ GB} \tag{5}
$$

式 (59) 的计算结果表明，仅 KV Cache 一项就需要约 344 GB 的显存空间。这是一个完全不可接受的数字——单张 H100 的显存仅为 80GB，这意味着即使使用 8 卡并行，KV Cache  alone 就占满了所有显存。

DeepSeek-V2/V3 的 MLA 通过引入一个低维的压缩向量 $c_{KV} \in \mathbb{R}^{d_c}$(其中 $d_c \ll h \times d_{head}$)来解决这个问题。在 MLA 中，K 和 V 的投影被分解为：

$$
c_{KV} = X W_{DKV} \tag{6}
$$
这一步将输入隐状态压缩到低维向量 $c_{KV}$，是 MLA 实现 KV Cache 缩减的核心操作。

$$
K = c_{KV} W_{UK}, \quad V = c_{KV} W_{UV} \tag{7}
$$

其中 $W_{DKV} \in \mathbb{R}^{d_{model} \times d_c}$ 是下投影矩阵，$W_{UK}, W_{UV}$ 是上投影矩阵。在推理时，只需要缓存压缩向量 $c_{KV}$，其维度仅为 $d_c$(在 V3 中约为 $4 \times d_{head} = 256$)，相比 MHA 的 $h \times d_{head} = 8192$，压缩率约为 32 倍。

然而，MLA 的压缩是**有损**的。所有注意力头共享同一个压缩向量 $c_{KV}$，这意味着不同头之间的细粒度差异信息在压缩过程中被"平均化"了。对于大多数任务(短文本理解、一般性问答)，这种信息损失可以忽略不计。但在需要**精确定位长文本中特定细节**的任务中(如"请找出合同中关于违约责任的第 3 条补充说明")，MLA 的压缩表征可能无法保留足够的区分度。

此外，MLA 虽然降低了 KV Cache 的显存占用，但**并未降低注意力计算的复杂度**。注意力分数矩阵的计算仍然是 $O(N^2)$，在 256K 的长度下，这是一个沉重的负担。

#### 4.1.2 CSA：Cross-Layer Shared Attention 跨层共享注意力

CSA 的核心思想基于一个关键观察：**相邻 Transformer 层的注意力模式具有高度相关性。**

在深度网络中，第 $l$ 层和第 $l+1$ 层处理的是同一输入的不同抽象层次的表征。第 $l$ 层的输出 $X^{(l+1)}$ 经过残差连接和层归一化后，成为第 $l+1$ 层的输入。如果两层之间的表征变化足够平滑(这在使用了 Pre-LN 和残差连接的现代 Transformer 中是成立的)，那么它们关注输入序列中相同位置的趋势应该是高度一致的。

基于这一观察，CSA 提出：**相邻的注意力层可以共享 Key 和 Value 的投影，只有 Query 保持独立。** 假设我们将 72 层注意力层分成 36 个"CSA 组"，每组包含 2 层(在 V4 的实际实现中，分组策略可能更复杂，可能是 3 层一组或动态分组，这里以 2 层为例说明原理)。

对于 CSA 组 $g$，包含层 $l$ 和层 $l+1$，其注意力计算如式 (1)-(4) 所示。首先，组内共享的 Key 和 Value 投影使用第一层 $l$ 的输入 $X^{(l)}$ 来计算：

$$
K^{(g)} = X^{(l)} W_K^{(g)}, \quad V^{(g)} = X^{(l)} W_V^{(g)} \tag{8}
$$
这里 $W_K^{(g)}$ 和 $W_V^{(g)}$ 是该组共享的参数，输入取自组内第一层的原因是：在自回归生成中，第 $l$ 层的输入已经编码了前 $l-1$ 层提取的全部历史信息，以它为基础构建共享的 KV 快照最为自然。然后，两层各自保留独立的 Query 投影：

$$
Q^{(l)} = X^{(l)} W_Q^{(l)}, \quad Q^{(l+1)} = X^{(l+1)} W_Q^{(l+1)} \tag{9}
$$
Query 保持独立是因为不同层关注的语义粒度不同：浅层 Query 可能聚焦局部词法模式，深层 Query 则关注全局语义关联。最后，两层的注意力输出分别从共享的 KV 中检索信息：

$$
A^{(l)} = \text{softmax}\left(\frac{Q^{(l)} (K^{(g)})^T}{\sqrt{d_{head}}}\right) V^{(g)} \tag{10}
$$
如式 (3) 所示，第 $l$ 层使用独立的 Query 从组内共享的 KV 中提取信息，既保留了层级的表达能力，又避免了两层各自维护完整的 KV Cache。

$$
A^{(l+1)} = \text{softmax}\left(\frac{Q^{(l+1)} (K^{(g)})^T}{\sqrt{d_{head}}}\right) V^{(g)} \tag{11}
$$
如式 (4) 所示，即使 K 和 V 与式 (3) 完全相同，由于 $Q^{(l+1)}$ 经过中间 MoE 层的非线性变换后与 $Q^{(l)}$ 已截然不同，两层的注意力分布仍会产生显著差异。

让我们仔细分析这一设计的几个关键细节：

**为什么只共享 K 和 V，不共享 Q？**
Query 的角色是"提问"——它决定了当前层想要从序列中提取什么信息。不同层有不同的"关注点"：浅层可能更关注局部语法结构，深层可能更关注全局语义关系。如果强制共享 Q，会限制不同层的表达能力。而 K 和 V 的角色是"索引"和"存储"——它们提供了序列的一个"信息快照"，不同层从这个快照中查询不同的内容是完全合理的。

**KV Cache 节省了多少？**
如果每组包含 2 层，那么 KV Cache 的层维度从 $L$ 降低到了 $L/2$。对于 V4 的 72 层，使用 2 层一组的 CSA，KV Cache 降低为原来的 1/2。如果采用更激进的分组策略(如 3 层一组)，则降低为 1/3。

但需要注意的是，CSA 的 KV Cache 节省与 MLA 的压缩是正交的。V4 在 CSA 组内仍然使用 MLA 风格的压缩(但可能有更精细的头级别压缩策略)。因此，总的 KV Cache 节省是两者的乘积效应。

**为什么 CSA 不会严重损失表达能力？**
这是 CSA 设计中最为精妙的部分。表面上看，共享 KV 似乎让两层变成了"同一个注意力模式的不同 Query 版本"，表达能力似乎受限。但关键在于：Transformer 层不仅仅是注意力，还包括 FFN(或 MoE)和残差连接。层 $l$ 的注意力输出会经过 MoE 层的非线性变换，产生与层 $l+1$ 截然不同的输入 $X^{(l+1)}$。因此，即使 K 和 V 相同，$Q^{(l+1)}$ 与 $K^{(g)}$ 的交互也会产生与 $Q^{(l)}$ 完全不同的注意力分布。

此外，softmax 函数的特性使得即使 Q 有细微变化，注意力权重也可能发生显著改变。因此，CSA 的共享是一种"受控的约束"——它在降低 KV Cache 的同时，通过保持 Q 的独立性和后续 MoE 层的非线性，确保了足够的表达能力。

> **图 5.2.8 跨层共享注意力 (CSA: Cross-Layer Shared Attention)**
> 与传统的 MHA 或 GQA 在每一层维持独立的 K、V 投影矩阵和缓存不同，CSA 使得相邻的若干层强制共享同一套 K、V Cache 内存池，在保证模型表征容量的同时，大幅度压榨了推理时的显存占用。

#### 4.1.3 HCA：Hierarchical Co-Attention 层次化协同注意力

CSA 解决了 KV Cache 的显存问题，但没有解决注意力计算复杂度 $O(N^2)$ 的问题。HCA 正是为了解决这个问题而设计的。

HCA 的理论基础是一个被大量实证研究验证的现象：**在 Transformer 的浅层，模型主要学习局部模式(如 n-gram、短语结构、局部语法); 在深层，模型主要学习全局关系(如跨句指代、段落级逻辑、长距离依赖)。**

这一现象的物理解释是：浅层的感受野(Receptive Field)有限，因为每次注意力操作只能传播一步(一个 token 只能直接关注到另一个 token); 而经过多层注意力的叠加，深层的感受野呈指数级扩大。因此，浅层对全局注意力的需求并不迫切——给它全局注意力，它也无法有效利用。

基于这一观察，HCA 提出：**根据层的深度动态调整注意力的范围——浅层使用局部窗口注意力，深层使用全局注意力。**

数学形式上，HCA 为第 $l$ 层定义一个窗口大小 $w^{(l)}$：

$$
w^{(l)} = \begin{cases} w_{\text{local}} & \text{if } l \leq L_{\text{shallow}} \\ N & \text{if } l > L_{\text{shallow}} \end{cases} \tag{12}
$$

其中 $w_{\text{local}}$ 是一个固定的局部窗口大小(如 1024 或 2048)，$L_{\text{shallow}}$ 是浅层的分界点(如总层数的 50%)。

对于使用局部窗口的浅层，注意力掩码 $M^{(l)}$ 被修改为：

$$
M^{(l)}_{ij} = \begin{cases} 0 & \text{if } |i - j| \leq w^{(l)} \text{ or } j \leq i \\ -\infty & \text{otherwise} \end{cases} \tag{13}
$$

这意味着位置 $i$ 的 token 只能关注到距离它不超过 $w^{(l)}$ 的 token(以及所有之前的 token，以满足因果性约束)。

**计算量分析：**

让我们对比标准全局注意力和 HCA 的计算量。

标准全局注意力每层：$O(N^2)$

HCA 中，假设前 50% 的层(36 层)使用窗口 $w = 1024$，后 50% 的层(36 层)使用全局注意力：

- 浅层每层：$O(N \times w) = O(256K \times 1024) \approx O(2.6 \times 10^8)$
- 深层每层：$O(N^2) = O((256K)^2) \approx O(6.6 \times 10^{10})$

总计算量(忽略常数因子)：

$$
\text{Total}_{\text{HCA}} = 36 \times N \times w + 36 \times N^2 \tag{14}
$$
此式将 HCA 的两段计算统一求和，前半段对应浅层局部窗口的线性复杂度，后半段对应深层全局注意力的平方复杂度。将 $N = 256K$、$w = 1024$ 代入式 (11) 展开：

$$
= 36 \times 256K \times 1024 + 36 \times (256K)^2 \tag{15}
$$

式 (60) 的第一项是浅层 36 层的局部注意力总计算量，第二项是深层 36 层的全局注意力总计算量。分别计算这两项的近似值：

$$
\approx 9.4 \times 10^9 + 2.4 \times 10^{12} \tag{16}
$$

式 (12) 中，浅层项约为 $9.4 \times 10^9$，与深层项 $2.4 \times 10^{12}$ 相比可以忽略不计。因此 HCA 的总计算量主要由深层全局注意力决定：

$$
\approx 2.4 \times 10^{12} \tag{17}
$$

为了量化 HCA 带来的节省，我们还需要计算标准全局注意力的总计算量作为基准。在标准全局注意力中，全部 72 层都使用全局注意力：

$$
\text{Total}_{\text{global}} = 72 \times N^2 = 72 \times (256K)^2 \approx 4.7 \times 10^{12} \tag{18}
$$

如式 (13) 所示，标准全局注意力的总计算量约为 $4.7 \times 10^{12}$，而式 (26) 中 HCA 的总计算量约为 $2.4 \times 10^{12}$。通过对比这两个数值可以得出，**HCA 将注意力计算量降低了约 49%。** 这是一个非常可观的节省。

更关键的是，在实际的推理场景中，当序列长度 $N$ 远大于窗口大小 $w$ 时(如 $N = 256K, w = 1024$)，浅层的局部注意力计算量 $N \times w$ 仅为全局注意力的 $w/N \approx 0.4\%$。这意味着在超长上下文的 prefill 阶段，HCA 可以显著缩短编码时间。

**HCA 的精妙之处还在于它与 CSA 的天然协同。**

在 CSA 的分组中，浅层的局部注意力需要的 KV Cache 量也相应减少——因为每个 token 只需要缓存距离它 $w$ 范围内的 K 和 V。虽然由于因果性约束，所有历史 token 都需要被后续 token 访问，但通过引入**滑动窗口 KV Cache 驱逐策略**(Sliding Window KV Cache Eviction)，可以只保留最近的 $w$ 个 token 的 KV，更老的 token 的 KV 可以被丢弃(因为它们不会再被任何未来的 token 在浅层中访问到)。这进一步降低了 KV Cache 的峰值显存占用。

#### 4.1.4 CSA/HCA 与标准 Attention 的对比总结

| 维度 | 标准 MHA | MLA (V3) | CSA/HCA (V4) |
|------|----------|----------|--------------|
| KV Cache 每 token | $2 \times L \times h \times d_{head}$ | $2 \times L \times d_c$ | $2 \times (L/G) \times d_c'$ |
| 注意力复杂度 | $O(L \times N^2)$ | $O(L \times N^2)$ | $O(L_{shallow} \times N \times w + L_{deep} \times N^2)$ |
| 信息损失 | 无 | 有损压缩 | 极小(相邻层共享) |
| 长上下文精度 | 最高(无损) | 高(压缩损失) | 高(无压缩，窗口限制仅存在于浅层) |
| 工程复杂度 | 低 | 中 | 高(需管理分组和窗口边界) |

其中 $G$ 是 CSA 的分组大小，$d_c'$ 是 V4 中可能进一步优化的压缩维度。

### 4.2 Gated Attention 门控注意力

#### 4.2.1 背景：并非所有 Token 都需要同等的注意力

注意力机制的基本假设是：序列中的每一个 token 都有潜力成为其他 token 的重要信息来源。在标准的自注意力中，这种潜力通过 Query 和 Key 的点积来度量，然后经过 softmax 归一化得到注意力权重。

然而，这个假设在现实中并不总是成立。考虑以下场景：

**场景 A：填充 Token(Padding Tokens)** 
在批处理(batching)中，不同长度的序列需要被填充到统一长度。填充 token 本身不包含任何语义信息，但标准注意力仍然会为它们分配非零的注意力权重(softmax 保证所有权重之和为 1，即使某个 token 不重要，它也会"摊薄"其他重要 token 的权重)。

**场景 B：冗余文本**
在长篇文档中，大量内容是"填充性"的——过渡语句、重复性解释、与核心问题无关的背景介绍。理想情况下，模型应该学会自动忽略这些冗余内容，将注意力集中在关键信息上。

**场景 C：噪声输入**
在网页抓取、OCR 识别、用户输入等场景中，输入序列可能包含错别字、乱码、格式符号等噪声。标准注意力对噪声没有天然的免疫力，噪声 token 可能通过注意力传播干扰正常的语义理解。

传统的解决方式是在预处理阶段去除填充或噪声，但这并不总是可行的(如噪声与内容混合在一起)。更根本的问题是：**标准注意力缺乏一种"选择性忽略"的机制。**

Sparse Attention(稀疏注意力)试图通过硬性的掩码规则(如只关注局部窗口、只关注特定模式)来解决这个问题，但稀疏掩码是预定义的、静态的，不具备自适应能力。如果重要的信息恰好落在了掩码之外，模型就永远看不到它。

#### 4.2.2 机制：自适应门控信号

Gated Attention 的核心思想是：**为每个 token 引入一个可学习的"重要性分数"，用这个分数来缩放该 token 在注意力输出中的贡献。**

具体而言，在计算完标准注意力输出后，Gated Attention 为每个 token 引入一个可学习的标量门控值，用它来缩放该 token 在注意力输出中的贡献。门控函数本身是一个小型 MLP 接 sigmoid 激活：

$$
\text{Gate}(x) = \sigma(\text{MLP}_{\text{gate}}(x)) \tag{19}
$$
其中 $x$ 是当前 token 的隐状态，$\text{MLP}_{\text{gate}}$ 通常只有一到两层，隐藏维度远小于主模型，$\sigma$ 将输出压缩到 $(0, 1)$ 之间。这个设计的动机是：让模型为每个 token 独立地学习一个"重要性分数"，而不需要人工标注的监督信号。

在此基础上，门控注意力层的完整前向传播如式 (2)-(5) 所示。首先计算标准的多头注意力输出：

$$
O_{\text{attn}} = \text{Attention}(Q, K, V) = \text{softmax}\left(\frac{QK^T}{\sqrt{d_{head}}}\right) V \tag{20}
$$
这一步与标准 Transformer 完全相同，保留了注意力机制对全局依赖关系的建模能力。接下来，利用注意力层的输入隐状态矩阵 $X \in \mathbb{R}^{N \times d_{model}}$ 计算门控信号：

$$
G = \sigma(\text{MLP}_{\text{gate}}(X)) \tag{21}
$$
$G \in \mathbb{R}^{N \times 1}$ 为每个 token 输出一个标量门控值，取值在 $(0, 1)$ 之间。然后将门控值逐元素乘到注意力输出上：

$$
O_{\text{gated}} = G \odot O_{\text{attn}} \tag{22}
$$
其中 $\odot$ 表示逐元素乘法(broadcasting 到 $d_{model}$ 维度)。这一步的物理意义是：Gate 接近 1 的 token 保留其完整的注意力贡献，Gate 接近 0 的 token 则被大幅抑制。最后通过残差连接得到该层的输出：

$$
X_{\text{out}} = X + O_{\text{gated}} \tag{23}
$$
如式 (5) 所示，残差连接确保了即使门控将所有 token 都压到接近 0，信息仍能通过跳跃连接流通，避免梯度消失。

让我们深入分析这个设计的物理意义：

**当 Gate $\approx 1$ 时：**
该 token 的注意力输出几乎不受抑制，相当于标准注意力。模型认为这个 token 携带重要信息，应该充分参与上下文的交互。

**当 Gate $\approx 0$ 时：**
该 token 的注意力输出被大幅抑制。即使注意力机制为它分配了较高的注意力权重(比如因为 token 的形式特征)，门控信号也会将其实际贡献压到接近零。这相当于模型学会了"屏蔽"这个 token。

**当 Gate 介于 0 和 1 之间时：**
模型对 token 的重要性持有"不确定"或"中等"的判断。这种软门控(Soft Gating)机制比硬掩码更加灵活——它允许模型在训练过程中逐渐学习最优的缩放比例，而不是在架构设计阶段就固化注意力模式。

#### 4.2.3 物理意义：自动学会"忽略"

Gated Attention 的优雅之处在于，它不需要任何人工标注的"重要性标签"来监督训练。门控 MLP 的参数与主模型一起通过标准的语言建模损失进行端到端优化。模型通过反向传播自动学习到：在什么情况下应该"开门"(Gate $\approx 1$)，在什么情况下应该"关门"(Gate $\approx 0$)。

从优化的角度来看，门控信号相当于为每个 token 引入了一个额外的自由度。在标准 Transformer 中，一个 token 对最终预测的贡献只能通过注意力权重来调节; 而在 Gated Attention 中，即使注意力权重由于 softmax 的约束而无法变得很小，门控信号仍然可以独立地将该 token 的贡献压到零。

这种"双重控制"机制赋予了模型更强的表达能力。具体而言，Gated Attention 有助于解决以下几个问题：

**1. 填充 Token 的自动抑制**
在训练数据中，填充 token 的预测目标通常是特殊的 `<pad>` token。由于 `<pad>` 本身不包含语义信息，模型很快学会为填充位置输出 Gate $\approx 0$。这避免了填充 token 在注意力传播中引入噪声。

**2. 长文档的信息筛选**
在处理长文档时，Gated Attention 可以学习将 Gate 值与 token 的信息密度关联起来。例如，在新闻文章中，标题和首段通常信息密度最高，模型可能学到为这些位置的 token 分配更高的 Gate 值; 而过渡性语句和背景介绍的 Gate 值则较低。

**3. 与 MoE 的协同**
Gated Attention 与 MoE 的路由机制形成了有趣的互补。MoE 的 Router 决定的是"哪个专家来处理这个 token"，而 Gated Attention 决定的是"这个 token 是否值得被充分关注"。在 V4 的架构中，Router 和 Gate 的输入都是同一个隐状态 $X$，但它们学习的是不同的映射：Router 关注的是任务的领域特征(如代码、数学、文学)，Gate 关注的是 token 级别的语义重要性。

#### 4.2.4 与 Sparse Attention 的关系：软稀疏 vs 硬稀疏

Gated Attention 与 Sparse Attention 之间的关系值得深入辨析，因为两者都试图解决"注意力资源有限，应该分配给哪些 token"的问题，但采取了截然不同的哲学路径。

**Sparse Attention(硬稀疏)：**
通过预定义的稀疏模式来限制注意力连接。常见的稀疏模式包括：
- **局部窗口(Local Window)** ：只关注距离不超过 $w$ 的邻居 token
- **跨步模式(Strided)** ：只关注固定间隔的 token(如每第 8 个 token)
- **全局-局部混合(Global-Local)** ：某些特定位置(如 `[CLS]`)可以与所有 token 交互，其他 token 只能局部交互
- **随机稀疏(Random)** ：随机采样一部分注意力连接

硬稀疏的优点是计算效率高——如果每个 token 只关注 $k \ll N$ 个其他 token，注意力复杂度就从 $O(N^2)$ 降低到 $O(N \times k)$。但缺点是**稀疏模式是静态的、与输入无关的**。如果重要的信息恰好不在预定义的稀疏连接范围内，模型就无法访问到它。

**Gated Attention(软稀疏)：**
不限制注意力连接的范围(所有 token 仍然可以相互关注)，而是通过学习一个缩放因子来调节每个 token 的"有效参与度"。它的优点是**完全自适应**——根据输入内容动态决定哪些 token 重要。但缺点是计算复杂度仍然是 $O(N^2)$，因为它需要计算完整的注意力矩阵才能决定门控值(实际上 Gate 的计算独立于注意力矩阵，可以在注意力计算之前完成，但注意力本身的 $O(N^2)$ 复杂度不变)。

因此，Gated Attention 和 Sparse Attention 不是竞争关系，而是互补关系。在 V4 的 HCA 中，浅层使用了硬稀疏(局部窗口)，这是由架构决定的静态约束; 而在深层的全局注意力中，Gated Attention 提供了软稀疏的自适应能力。两者结合，使得 V4 在计算效率(硬稀疏)和表达能力(软稀疏)之间取得了平衡。

### 4.3 MTP：Multi-Token Prediction 多词元预测

#### 4.3.1 背景：传统自回归的串行瓶颈

自回归语言模型的核心假设是：文本是一个严格的马尔可夫链——预测 $x_{t+1}$ 只需要知道 $x_{\leq t}$。这一假设在数学上表达为：

$$
P(x_1, x_2, ..., x_T) = \prod_{t=1}^{T} P(x_t | x_{<t}) \tag{24}
$$

这个分解方式使得训练变得极其简单：只需最大化每个位置的对数似然。但它也带来了一个根本性的瓶颈：**推理时必须逐个 token 生成，无法并行。**

假设生成一个包含 512 个 token 的回答，每个 token 的 forward pass 需要 10ms(这是一个在 45B 激活参数的模型上的合理估计)，那么总生成时间为 $512 \times 10\text{ms} = 5.12$ 秒。如果 MTP 能够同时预测未来的 4 个 token，并且这些预测有相当高的准确率，那么可以使用 speculative decoding 策略：先用 MTP 的 draft head 并行生成候选 token，再用主模型的一次验证步骤确认或修正。理想情况下，如果 draft 的接受率达到 80%，那么生成 512 个 token 只需要约 $512 / (4 \times 0.8) \approx 160$ 个串行步骤，时间缩短到约 1.6 秒——**3 倍以上的加速。**

> ** V3 实际部署数据**：DeepSeek-V3 的 MTP 模块在真实生产环境中对第二个后续 token 的预测接受率达到 **80%–90%**，端到端生成 TPS 相比无 MTP 基线提升 **1.8 倍**。虽然 MTP 模块的引入略微降低了系统吞吐量(throughput)，但生成延迟(latency)的改善使其在交互式场景中具有显著优势。此外，MTP 增加了推理 batch size，这对提升 EP(Expert Parallelism)计算强度和硬件利用率至关重要。

但 MTP 的价值不仅限于推理加速。在训练阶段，MTP 提供了更密集的监督信号：每个训练位置不仅产生 1 个损失，而是产生 $k$ 个损失。这使得模型能够从同一批数据中学习到更多的信息，提升了**样本效率(Sample Efficiency)** 。

#### 4.3.2 MTP 的数学形式

在 V4 中，MTP 被深度集成到模型的输出层。模型同时维护 $k$ 个独立的预测头(在 V4 中 $k=4$)，第 $i$ 个预测头负责预测未来第 $i$ 个 token。这种设计的动机是：在训练的每一步，不仅要求模型学会预测下一个 token，还要学会"向前看"若干步，从而从同一批数据中提取更密集的监督信号。

设主模型的最后一层隐状态为 $H \in \mathbb{R}^{N \times d_{model}}$，它编码了当前前缀的全部上下文信息。V4 的 MTP 模块包含 $k$ 个轻量级的投影头，每个头将 $H$ 映射到词表空间上的 logits：

$$
\text{logits}^{(i)} = W_{\text{mtp}}^{(i)} H + b_{\text{mtp}}^{(i)}, \quad i = 1, 2, ..., k \tag{25}
$$

如式 (39) 所示，每个预测头都是一个简单的线性变换(或一层小 MLP)，$W_{\text{mtp}}^{(i)} \in \mathbb{R}^{d_{model} \times |V|}$ 将 $d_{model}$ 维的隐状态映射到 $|V|$ 维的词表空间，$|V|$ 是词表大小。由于参数量极少，这些投影头不会显著增加推理延迟。

对于训练序列中的位置 $t$，第 $i$ 个预测头的目标是预测 token $x_{t+i}$。为了将 logits 转化为可计算的概率分布，需要对每个位置应用 softmax 归一化：

$$
P(x_{t+i} | x_{\leq t}) = \text{softmax}(\text{logits}^{(i)}_t) \tag{26}
$$

如式 (40) 所示，第 $i$ 个预测头在位置 $t$ 的条件概率仅依赖于前缀 $x_{\leq t}$，而不依赖于 $x_{t+1}, ..., x_{t+i-1}$。这与使用链式自回归依赖的 MTP 变体不同——V4 采用**并行的独立预测头**设计，主要出于两个工程考量：一是每个预测头的损失可以独立计算，无需进行链式采样，训练实现更简单; 二是在 speculative decoding 中，每个 draft head 可以独立并行运行，一次性生成 $k$ 个候选 token。

为了联合优化 $k$ 个预测头的学习能力，需要将它们的预测误差整合为统一的训练目标。MTP 的训练损失是 $k$ 个交叉熵损失的加权和，如式 (41) 所示：

$$
\mathcal{L}_{\text{MTP}} = \sum_{i=1}^{k} \lambda_i \cdot \text{CE}(x_{t+i}, \text{logits}^{(i)}_t) \tag{27}
$$

如式 (41) 所示，$\lambda_i$ 是第 $i$ 个预测头的权重系数，反映了不同预测距离的可信度差异：在 V4 的训练中，通常设置 $\lambda_1 > \lambda_2 > ... > \lambda_k$，因为预测越远未来的 token 不确定性越高，不应给予与下一个 token 同等的重要性。典型的配置是 $\lambda_i = \alpha^{i-1}$，其中 $\alpha \in (0, 1)$(如 $\alpha = 0.8$)。

在将 MTP 整合到整体训练框架时，需要将辅助损失与主语言建模损失进行加权融合。总训练损失将主语言建模损失与 MTP 辅助损失相加，如式 (42) 所示：

$$
\mathcal{L}_{\text{total}} = \mathcal{L}_{\text{main}} + \beta \cdot \mathcal{L}_{\text{MTP}} \tag{28}
$$

如式 (42) 所示，$\beta$ 是 MTP 损失在总目标中的权重系数。在预训练阶段，$\beta$ 通常设为 0.3 到 0.5，以充分利用 MTP 的样本效率优势; 在 SFT 阶段，由于数据质量更高、序列更短，$\beta$ 可以适当降低，避免 MTP 头干扰微调后的输出分布。

#### 4.3.3 训练稳定性：V4 的解决方案

MTP 的训练不稳定性主要源于两个方面：梯度冲突和暴露偏差的加剧。V4 通过以下机制来缓解这些问题：

**1. 渐进式 MTP 激活(Progressive MTP Activation)** 
在训练的早期阶段(如前 10% 的 steps)，模型只使用主损失进行训练，MTP 头处于冻结状态。这确保了主模型先学习到基本的语言表示能力，然后再引入 MTP 的多目标优化。在训练中期，逐步解冻 MTP 头并增大 $\beta$。这种课程学习(Curriculum Learning)风格的策略避免了 MTP 头在模型尚未稳定时引入过多的噪声梯度。

**2. 预测头的维度隔离(Dimensional Isolation)** 
V4 的 $k$ 个 MTP 头并非共享同一个输出投影矩阵，而是各自拥有独立的参数。这避免了不同预测目标之间的参数冲突。更重要的是，每个 MTP 头的输入不仅仅是主模型的最后一层隐状态 $H$，而是经过一个轻量级的、头专属的投影变换：

$$
H^{(i)} = \text{MLP}_{\text{mtp}}^{(i)}(H) \tag{29}
$$

如式 (28) 所示，$\text{MLP}_{\text{mtp}}^{(i)}$ 是第 $i$ 个预测头专属的轻量级投影网络，它将共享的隐状态 $H$ 转换为该头独有的输入表示 $H^{(i)}$。这个设计的动机是避免不同预测目标之间的参数冲突，让各头能够独立捕捉各自时间尺度上的 token 依赖模式。

在得到专属输入表示后，每个预测头再通过独立的输出投影生成词表级别的 logits：

$$
\text{logits}^{(i)} = W_{\text{out}}^{(i)} H^{(i)} \tag{30}
$$

如式 (14) 所示，$W_{\text{out}}^{(i)}$ 是第 $i$ 个头的输出投影矩阵。这种"输入投影 + 输出投影"的两层结构，为每个预测头提供了足够的表达能力来捕捉不同时间尺度上的 token 依赖模式。

**3. 动态权重调度(Dynamic Weight Scheduling)** 
V4 在训练过程中动态调整 $\lambda_i$ 和 $\beta$。具体的策略是：根据每个 MTP 头在验证集上的准确率来调整其权重。如果第 2 个预测头的准确率持续低于第 1 个头的 70%，则自动降低其 $\lambda_2$。这种自适应权重机制确保了训练资源(梯度幅度)被优先分配给那些学得好的预测任务。

**4. 与主损失的解耦(Decoupling from Main Loss)** 
在 V3 的实现中，MTP 损失与主损失高度耦合——MTP 头的梯度直接反向传播到主模型的所有层。V4 引入了一种"截断梯度"策略：MTP 头的梯度只传播到主模型的最后 4 层，而不影响浅层的参数。这样做的理由是：浅层负责学习通用的语言表示，不应该被 MTP 这种"高级预测任务"过度干扰; 深层负责将表示映射到具体的输出分布，更适合承担 MTP 的梯度信号。

#### 4.3.4 推理加速：Speculative Decoding 的集成

MTP 在推理阶段的价值体现在 speculative decoding(推测解码)的应用上。传统的 speculative decoding 需要一个独立的 draft model(通常是小模型)来快速生成候选 token，然后用大模型进行验证。但 draft model 与主模型之间的分布差异(distribution mismatch)会导致 draft token 的接受率不高。

V4 的 MTP 天然地提供了一种"内置 draft model"的解决方案：

**推理流程：**

1. **Draft 阶段**：给定当前前缀 $x_{\leq t}$，使用 MTP 的 4 个预测头并行生成候选 token：$\hat{x}_{t+1}, \hat{x}_{t+2}, \hat{x}_{t+3}, \hat{x}_{t+4}$。这一步是并行的，且 MTP 头是轻量级的，延迟极低。

2. **验证阶段**：将候选序列 $[x_{\leq t}, \hat{x}_{t+1}, \hat{x}_{t+2}, \hat{x}_{t+3}, \hat{x}_{t+4}]$ 输入主模型，计算主模型在这 4 个位置上的真实分布。

3. **接受/拒绝决策**：
   - 对于位置 $t+1$，比较主模型的分布 $P(x_{t+1} | x_{\leq t})$ 与 draft token $\hat{x}_{t+1}$。如果 $\hat{x}_{t+1}$ 是主模型分布中的高概率 token(具体地，以概率 $\min(1, P_{\text{main}}(\hat{x}_{t+1}) / P_{\text{mtp}}(\hat{x}_{t+1}))$ 接受)，则接受该 token。
   - 如果 $t+1$ 被接受，继续验证 $t+2$：比较 $P(x_{t+2} | x_{\leq t}, \hat{x}_{t+1})$ 与 $\hat{x}_{t+2}$。
   - 依此类推，直到某个位置被拒绝。

4. **重采样**：如果位置 $t+j$ 被拒绝，从修正后的分布中采样新的 token，并丢弃所有后续的 draft token。

**为什么 MTP 作为 draft model 更好？**

因为 MTP 头与主模型共享了前 $L - 4$ 层的参数，它们的表示空间是高度对齐的。相比之下，独立的 draft model(如一个 7B 模型)与主模型(45B 激活)之间存在巨大的容量差距和训练数据差异，导致 draft token 的分布与主模型差异较大。V4 的 MTP draft 接受率通常在 75%-85% 之间，而独立 draft model 的接受率通常只有 50%-60%。

**加速比分析：**

假设 MTP draft 的接受率为 $p$，每个 MTP head 的 forward 延迟为主模型的 $\epsilon$(由于 MTP head 很轻量，$\epsilon \approx 0.05$)。

每个 speculative decoding 周期：
- Draft 延迟：$k \times \epsilon$(并行计算 $k$ 个 head)
- 验证延迟：1(主模型的一次 forward)
- 平均接受 token 数：$\frac{1 - p^{k+1}}{1 - p} - 1$(几何分布的期望)

当 $k=4, p=0.8, \epsilon=0.05$ 时：

- 平均接受 token 数：$\frac{1 - 0.8^5}{1 - 0.8} - 1 = \frac{1 - 0.3277}{0.2} - 1 = 3.36 - 1 = 2.36$
- 每个周期的总延迟：$4 \times 0.05 + 1 = 1.2$
- 等效速度提升：$2.36 / 1.2 \approx 1.97$ 倍

如果 $p$ 提高到 0.85：
- 平均接受 token 数：$\frac{1 - 0.85^5}{0.15} - 1 = 3.18$
- 等效速度提升：$3.18 / 1.2 \approx 2.65$ 倍

这意味着在理想情况下，MTP speculative decoding 可以将推理速度提升 **2-3 倍**，且无需任何额外的模型权重(MTP 头的参数已经包含在 V4 的 checkpoint 中)。

## 5. 后训练与 OPD 关联

### 5.1 DeepSeek-V4 的后训练 Pipeline

DeepSeek-V4 的后训练流程是一个精心设计的四阶段 pipeline，每个阶段解决不同层面的问题，并在数据和计算资源上形成递进关系：

```
预训练 Checkpoint
    │
    ▼
Stage 1: SFT (监督微调)
    - 数据：数百万条高质量指令-回答对
    - 目标：建立基础指令遵循能力和对话格式
    - 时长：约 2-3 天 (数百 GPU 小时)
    │
    ▼
Stage 2: G-OPD (广义在线蒸馏)
    - 数据：高质量领域数据 (数学、代码、科学)
    - 目标：在学生自身轨迹上蒸馏教师的知识
    - 时长：约 1-2 天
    │
    ▼
Stage 3: GRPO (组相对策略优化)
    - 数据：可规则验证的任务 (数学、逻辑、编程)
    - 目标：通过 RL 激发推理能力
    - 时长：约 3-5 天
    │
    ▼
Stage 4: SCOPE 过滤与自蒸馏
    - 数据：模型自身生成的多轮回答
    - 目标：去噪、提纯、自我提升
    - 时长：约 1-2 天
```

这一 pipeline 的总算力消耗约为纯 RL 方案(如 R1 的早期实验)的 1/8 到 1/6，但最终的模型能力却更为全面——不仅在数学推理上保持顶尖水平，在开放式对话、创意写作、多语言理解等"难以用规则评分"的任务上也有出色表现。

### 5.2 G-OPD 在蒸馏阶段的应用

G-OPD(Generalized On-Policy Distillation)是 OPD 家族的泛化版本。回顾 OPD 基础原理中的核心公式：

$$
\mathcal{L}_{OPD} = \mathbb{E}_{s_t \sim \pi_\theta} [ D_{KL}(\pi_\theta(\cdot|s_t) \| \pi_T(\cdot|s_t)) ] \tag{31}
$$

G-OPD 在此基础上进行了两个关键扩展，使其更适合大规模工程实践：

**扩展 1：支持多个教师(Multi-Teacher Distillation)** 
在 V4 的后训练中，并不存在一个统一的"超级教师"模型。相反，训练数据来自多个来源：一部分来自 V3 自身(通过更长的推理时间或更大的计算预算生成的更高质量回答)，一部分来自专门领域的教师模型(如代码领域的 Codex 级模型、数学领域的专用推理模型)。G-OPD 将单教师扩展到多教师：

$$
\mathcal{L}_{G\text{-}OPD} = \mathbb{E}_{s_t \sim \pi_\theta} \left[ \sum_{m} w_m \cdot D_{KL}(\pi_\theta(\cdot|s_t) \| \pi_{T_m}(\cdot|s_t)) \right] \tag{32}
$$
其中 $w_m$ 是教师 $m$ 的权重，可以根据教师在该领域的权威性进行设置。例如，对于数学问题，数学专用教师的 $w_m$ 更高; 对于一般性问题，通用模型 V3 的 $w_m$ 更高。

**扩展 2：动态温度调度(Dynamic Temperature Scheduling)** 
OPD 中的温度参数 $\tau$ 控制着教师分布的平滑度。在训练早期，使用较高的温度(如 $\tau = 2.0$)可以让教师分布更平滑，提供更多信息丰富的梯度信号; 在训练后期，使用较低的温度(如 $\tau = 0.5$)可以让教师分布更尖锐，引导学生模型做出更"自信"的决策。G-OPD 引入了基于训练进度的自动温度衰减：

$$
\tau(t) = \tau_{\max} \cdot \exp(-\lambda t) + \tau_{\min} \tag{33}
$$

其中 $t$ 是训练步数，$\lambda$ 控制衰减速度。

**V4 中 G-OPD 的工程实现：**

G-OPD 在 V4 中的实施充分利用了 V4 的架构特性。具体来说：

1. **On-Policy 采样利用 MTP**：在 G-OPD 的采样阶段，V4 使用 MTP 的 draft head 快速生成候选轨迹。虽然这些 draft token 需要经过主模型的验证，但即使只接受其中的一部分，也可以将采样速度提升约 2 倍。这使得在相同的 GPU 时间内，可以生成更多的 On-Policy 样本，提高了 G-OPD 的数据效率。

2. **CSA/HCA 降低教师模型推理成本**：在多教师设置中，需要同时运行多个教师模型来评估学生轨迹。CSA/HCA 的高效 KV Cache 管理使得在同一批 GPU 上部署多个教师模型成为可能。在 V4 的训练集群中，教师模型的推理 batch size 可以提升到标准 MHA 架构下的 2-3 倍。

3. **Gated Attention 过滤噪声样本**：在 On-Policy 采样过程中，学生模型可能会生成低质量的轨迹(如重复、跑题、语法错误)。Gated Attention 在教师模型评估这些轨迹时，自动为低信息量的 token 分配低 Gate 值，使得教师提供的分布信号更加集中在关键决策位置上。这相当于一种"隐式的样本筛选"机制。

### 5.3 GRPO 在 RL 阶段的深化

GRPO 在 R1 中已经证明了其价值，V4 在此基础上进行了几项针对性的改进：

**1. 分组大小的自适应调整**
GRPO 的核心是使用组内相对优势来替代 PPO 中的价值网络：

$$
A_i = \frac{r_i - \text{mean}(\{r_1, ..., r_G\})}{\text{std}(\{r_1, ..., r_G\})} \tag{34}
$$
R1 使用的是固定的组大小 $G$(通常为 8 或 16)。V4 发现，对于不同难度的问题，最优的组大小是不同的：
- 简单问题：模型回答的方差小，小分组($G=4$)即可得到稳定的优势估计。
- 困难问题：模型回答的方差大，需要更大的分组($G=32$ 或 64)来获得统计上可靠的优势信号。

V4 引入了一种基于问题难度的动态分组策略：先使用一个轻量级的分类器(基于问题长度、关键词、领域标签等特征)估计问题难度，然后据此选择组大小。这使得 GRPO 的样本效率提升了约 15%。

**2. 过程奖励与 MTP 的结合**
R1 主要使用结果奖励(Outcome Reward，即只看最终答案是否正确)。V4 在某些领域(如数学证明、代码调试)引入了过程奖励模型(PRM)。但 PRM 的推理开销巨大。V4 的解决方案是利用 MTP 头来近似 PRM：

MTP 的第 $k$ 个预测头天然地编码了"预测未来"的能力。如果一个中间推理步骤是正确的，那么模型应该能够以较高的置信度预测出后续的几个 token。反之，如果模型在步骤 $t$ 之后"迷失"了方向，其 MTP 头在步骤 $t$ 的预测置信度会显著下降。基于这一观察，V4 使用 MTP 头的置信度作为过程奖励的代理信号：

$$
\hat{r}_{\text{process}}(s_t) = \frac{1}{k} \sum_{i=1}^{k} P_{\text{mtp}}^{(i)}(x_{t+i} | s_t) \tag{35}
$$

这个代理信号虽然不如真正的 PRM 精确，但它的计算几乎零额外开销(MTP 头的输出在推理时本来就计算好了)。在实验中，这种"MTP 代理奖励"与真正的 PRM 奖励的相关系数达到了 0.72，足以作为一种廉价的辅助奖励信号来使用。

### 5.4 SCOPE 用于过滤自蒸馏中的低质量反馈

在 R1 的纯 RL pipeline 中，模型完全依赖规则奖励来指导学习。但规则奖励有一个致命局限：它只能判断最终答案的正确性，无法评估推理过程的合理性。模型可能通过"凑答案"的方式获得高分(如在数学题中通过枚举法碰巧得到正确答案)，而不是真正学会了正确的推理路径。

SCOPE(Self-Consistency based On-Policy Evaluation)是 OPD 家族中专门用于自蒸馏阶段的质量控制算法。其核心思想是：**如果一个推理步骤是可靠的，那么模型在不同采样温度下应该都能从这个步骤出发得到一致的结果。**

SCOPE 的具体实施如下：

1. 对于一条 On-Policy 轨迹(即学生模型自己生成的回答)，在关键决策位置(如推理链的中间步骤)，使用多个不同的采样温度(如 $\tau \in \{0.3, 0.7, 1.0, 1.5\}$)重新采样后续轨迹。

2. 计算这些重采样轨迹之间的语义一致性。如果原始步骤是一个"稳健"的推理步骤，那么无论采样温度如何变化，后续轨迹应该收敛到相似的结论; 如果原始步骤是"脆弱"的(如模型在瞎猜)，重采样轨迹会发散到各种截然不同的方向。

3. 使用一致性分数作为该轨迹的过滤阈值。只有一致性分数超过阈值的轨迹才会被用于 GRPO 的更新。

在 V4 的后训练中，SCOPE 与 GRPO 形成了一个闭环：

```
GRPO 生成轨迹 → SCOPE 评估一致性 → 过滤低质量轨迹 → 
高质量轨迹更新策略 → 策略改进 → 生成更高质量轨迹
```

这一闭环有效地抑制了 RL 训练中常见的"奖励黑客(Reward Hacking)"现象——模型无法再通过取巧的方式获得高分，因为取巧的路径在 SCOPE 的多温度重采样下会暴露出不稳定性。

### 5.5 算力经济学：V4 的 RL 成本降低分析

让我们以具体的数字来说明 V4 后训练 pipeline 的算力优势。

假设我们需要将一个预训练好的 V4 checkpoint 微调到数学推理的 SOTA 水平。

**纯 RL 方案(R1 风格)：**
- 组大小 $G = 16$
- 每 step 需要生成 16 条轨迹，平均每条 1024 tokens

- 总 forward tokens：$16384 \times 1000 = 16.4M$

- 在 H100 集群上，约需 **4000 GPU 小时**

**V4 的四阶段 Pipeline：**
- Stage 1 (SFT)：约 500 GPU 小时
- Stage 2 (G-OPD)：由于使用 On-Policy 蒸馏，无需大量 Rollout。通过 MTP 加速采样，约 **300 GPU 小时**
- Stage 3 (GRPO)：动态分组减少无效采样，SCOPE 过滤减少低质量梯度更新。约 **800 GPU 小时**
- Stage 4 (SCOPE 自蒸馏)：约 **200 GPU 小时**
- **总计：约 1800 GPU 小时**

**成本降低比：$1800 / 4000 = 45\%$，即成本降低了一半以上。**

如果与 Qwen3 报告中披露的传统 RL 方案(17,920 GPU 小时)相比，V4 的 pipeline 效率提升更为惊人：$1800 / 17920 \approx 10\%$，即仅需要传统 RL 约 **1/10** 的算力。

这种算力经济学上的优势不是通过"偷工减料"实现的，而是通过架构创新与算法创新的系统级协同实现的：CSA/HCA 让教师模型推理更快，MTP 让 On-Policy 采样更快，G-OPD 提供了比 RL 更高效的蒸馏路径，SCOPE 确保每一单位算力都用在高质量的梯度更新上。

### 6.1 场景设定

假设我们正在部署 DeepSeek-V4 进行长文档分析的推理任务，具体配置如下：

- 序列长度 $N = 8192$(约 6000 英文单词或 4000 中文字符，一篇长论文的长度)
- 模型层数 $L = 72$
- 注意力头数 $h = 144$
- 头维度 $d_{head} = 64$
- 隐层维度 $d_{model} = 8192$
- CSA 分组大小 $G = 2$(每 2 层共享一组 KV)
- HCA 浅层分界点 $L_{shallow} = 36$(前 50% 层为浅层)
- HCA 局部窗口大小 $w = 1024$
- 批量大小(Batch Size)$B = 8$
- 词表大小 $|V| = 100000$
- 数据类型：FP16(每个参数 2 字节)

### 6.2 CSA 节省的 KV Cache 量

为了量化 CSA 带来的显存节省，我们从标准 MHA 的 KV Cache 开始，逐步推导到 CSA + MLA 的压缩结果。首先，标准 MHA 每层每序列的 KV Cache 占用为：

$$
\text{KV}_{\text{MHA}}^{\text{per layer}} = 2 \times N \times h \times d_{head} \tag{36}
$$

如式 (43) 所示，标准 MHA 每层的 KV Cache 与序列长度 $N$、头数 $h$ 和头维度 $d_{head}$ 成正比，在长序列场景下呈线性增长。将场景设定中的具体数值代入：

$$
= 2 \times 8192 \times 144 \times 64 = 150,994,944 \text{ elements} \tag{37}
$$

式 (61) 给出了单层的元素总数，接下来需要乘以每个元素占用的字节数(FP16 为 2 字节)来得到实际的显存占用：

$$
= 150,994,944 \times 2 \text{ bytes} = 301,989,888 \text{ bytes} \approx 288 \text{ MB} \tag{38}
$$

式 (17) 中系数 2 分别对应 Key 和 Value 两个矩阵。单层约 288 MB 的占用已经相当可观，扩展到全部 $L=72$ 层后总量将进一步放大：

$$
\text{KV}_{\text{MHA}}^{\text{total}} = L \times 288 \text{ MB} = 72 \times 288 \text{ MB} = 20,736 \text{ MB} \approx 20.25 \text{ GB} \tag{39}
$$
此式描述了变量之间的定量关系，其物理意义将在下文详细阐述。
基于上述分析，建立如下数学关系：

$$
\text{KV}_{\text{MHA}}^{\text{batch}} = 8 \times 20.25 \text{ GB} = 162 \text{ GB} \tag{40}
$$

如式 (45) 所示，在 batch size 为 8 的情况下，标准 MHA 的 KV Cache 总量达到 162 GB，这意味着仅 KV Cache 一项就远超单卡 80GB 的显存容量，这正是 CSA 需要解决的核心痛点。

接下来计算 V4 使用 CSA + MLA 后的结果。假设 MLA 的压缩维度 $d_c = 256$(这是 V3 级别的压缩，V4 可能采用更精细的策略，但此处以此为例)。CSA 将层数从 $L$ 降低到 $L/G = 72/2 = 36$ 个 CSA 组。每组 KV Cache 为：

$$
\text{KV}_{\text{CSA+MLA}}^{\text{per group}} = 2 \times N \times d_c \tag{41}
$$

如式 (46) 所示，CSA 与 MLA 的联合作用将每层缓存从 $N \times h \times d_{head}$ 压缩到 $N \times d_c$，压缩率由头数和压缩维度的比值决定。假设 MLA 的压缩维度 $d_c = 256$(这是 V3 级别的压缩，V4 可能采用更精细的策略，但此处以此为例)，CSA 将层数从 $L$ 降低到 $L/G = 72/2 = 36$ 个 CSA 组。代入具体数值计算每组的元素数量：

$$
= 2 \times 8192 \times 256 = 4,194,304 \text{ elements} \tag{42}
$$

式 (62) 给出了每组需要缓存的元素数量。同样以 FP16 计算显存占用：

$$
= 4,194,304 \times 2 \text{ bytes} = 8,388,608 \text{ bytes} \approx 8 \text{ MB} \tag{43}
$$

式 (63) 表明每组 KV Cache 仅需约 8 MB。扩展到全部 36 组并乘以 batch size，得到单 batch 的总 KV Cache：

$$
\text{KV}_{\text{CSA+MLA}}^{\text{total}} = 36 \times 8 \text{ MB} = 288 \text{ MB} \tag{44}
$$
$$
\text{KV}_{\text{CSA+MLA}}^{\text{batch}} = 8 \times 288 \text{ MB} = 2,304 \text{ MB} \approx 2.25 \text{ GB} \tag{45}
$$

对比式 (45) 和式 (6)，CSA + MLA 将 KV Cache 从 162 GB 压缩到了约 2.25 GB，节省比例为：

$$
\text{节省比例} = 1 - \frac{2.25}{162} = 1 - 0.0139 = 98.6\% \tag{46}
$$
这意味着单张 80GB H100 可以处理的 batch size 从不足 1 提升到约 35，推理吞吐量获得数量级提升。

KV Cache 从 **162 GB** 降低到了 **2.25 GB**——一个惊人的 70 倍压缩。这使得单张 80GB H100 可以处理的 batch size 从不足 1 提升到约 35($80 / 2.25 \approx 35.5$)，极大地提升了推理吞吐量。

### 6.3 HCA 节省的计算量

为了量化 HCA 的计算节省，我们先推导标准全局注意力的总计算量，再与 HCA 的分层策略对比。注意力计算的核心操作是 $QK^T$ 矩阵乘法和乘以 $V$。忽略常数因子和 softmax，单层的核心计算量为：

$$
\text{FLOPs}_{\text{global}}^{\text{per layer}} = 2 \times N^2 \times d_{model} \tag{47}
$$
这里 $QK^T$ 需要 $N \times d_{model} \times N$ 次乘法，乘以 $V$ 需要 $N \times N \times d_{model}$ 次乘法，合计 $2 N^2 d_{model}$。扩展到全部 $L=72$ 层：

$$
\text{FLOPs}_{\text{global}}^{\text{total}} = L \times 2 \times N^2 \times d_{model} \tag{48}
$$

如式 (49) 所示，标准全局注意力在完整网络中的总计算量与层数 $L$ 和序列长度的平方 $N^2$ 成正比，其二次增长特性是长序列推理的主要性能瓶颈。将场景设定中的具体数值代入式 (49)：

$$
= 72 \times 2 \times (8192)^2 \times 8192 \tag{49}
$$

式 (64) 将 $N=8192$ 和 $L=72$ 直接代入。计算 $(8192)^2 = 67,108,864$：

$$
= 72 \times 2 \times 67,108,864 \times 8192 \tag{50}
$$

式 (18) 继续简化，将 $67,108,864 \times 8192$ 合并为科学计数法表示：

$$
= 72 \times 2 \times 5.4975 \times 10^{11} \tag{51}
$$

式 (65) 已经将所有数值转化为便于计算的形式。最终得到标准全局注意力的总计算量：

$$
\approx 7.916 \times 10^{13} \text{ FLOPs} \tag{52}
$$

如式 (19) 所示，标准全局注意力在 $N=8192$、$L=72$ 的配置下需要约 $7.9 \times 10^{13}$ FLOPs。这个数值将作为后续对比 HCA 节省效果的基准。

HCA 将前 36 层(浅层)切换为局部窗口注意力，后 36 层(深层)保持全局注意力。浅层的计算量为：

$$
\text{FLOPs}_{\text{HCA}}^{\text{shallow}} = L_{shallow} \times 2 \times N \times w \times d_{model} \tag{53}
$$

如式 (50) 所示，浅层的局部窗口注意力将复杂度从 $O(N^2)$ 降到 $O(N \times w)$，在 $w \ll N$ 时可节省两个数量级的计算量。代入场景设定中的具体数值展开计算：

$$
= 36 \times 2 \times 8192 \times 1024 \times 8192 \tag{54}
$$

式 (66) 将各维度数值直接代入式 (50)。继续简化计算：

$$
= 36 \times 2 \times 6.8719 \times 10^{10} \tag{55}
$$

式 (67) 计算了中间乘积。最终得到浅层的总计算量：

$$
\approx 4.948 \times 10^{12} \text{ FLOPs} \tag{56}
$$

如式 (20) 所示，式 (50) 中窗口 $w=1024$ 将每层复杂度从 $O(N^2)$ 降到 $O(N \times w)$，浅层计算量从全局注意力的 $O(10^{13})$ 下降到约 $4.9 \times 10^{12}$，节省了约两个数量级。深层的全局注意力计算量保持不变：

$$
\text{FLOPs}_{\text{HCA}}^{\text{deep}} = L_{deep} \times 2 \times N^2 \times d_{model} \tag{57}
$$

如式 (51) 所示，深层保持全局注意力以确保长距离依赖的建模能力，其计算量与标准全局注意力相同，但仅占总层数的一半。将 $L_{deep}=36$ 和 $N=8192$ 代入：

$$
= 36 \times 2 \times (8192)^2 \times 8192 \tag{58}
$$

式 (68) 是深层计算量的展开形式，其数值恰好是式 (64) 的一半(因为深层仅 36 层)。直接给出最终结果：

$$
\approx 3.958 \times 10^{13} \text{ FLOPs} \tag{59}
$$

HCA 的总计算量为浅层与深层之和：

$$
\text{FLOPs}_{\text{HCA}}^{\text{total}} = 4.948 \times 10^{12} + 3.958 \times 10^{13} = 4.453 \times 10^{13} \text{ FLOPs} \tag{60}
$$

如式 (52) 所示，HCA 的总计算量是浅层与深层之和。对比式 (19) 和式 (52)，HCA 节省的计算量比例为：

$$
\text{节省比例} = 1 - \frac{4.453 \times 10^{13}}{7.916 \times 10^{13}} = 1 - 0.5625 = 43.75\% \tag{61}
$$
HCA 将注意力计算量降低了约 **44%**。需要注意的是，这仅是注意力部分的节省。在完整的前向传播中，MoE 层的计算($O(N \times d_{model} \times d_{FFN})$)占据了更大比重。因此，HCA 对端到端推理延迟的实际影响约为 **8-12%**(注意力通常占完整 forward 时间的 20-30%，44% 的 20-30% 约为 8-13%)。

### 6.4 MTP(k=4)的推理加速比

**场景设定：**
- 目标生成长度 $T = 512$ tokens
- MTP 预测头数 $k = 4$
- 每个 MTP head 的 forward 延迟为主模型的 $\epsilon = 5\%$
- MTP draft 接受率 $p = 80\%$

为了量化 MTP 带来的推理加速，我们对比无 MTP 的基准延迟和有 speculative decoding 的延迟。无 MTP 时，生成 $T=512$ 个 token 的基准延迟为：

$$
\text{Latency}_{\text{baseline}} = T \times t_{\text{main}} = 512 \times t_{\text{main}} \tag{62}
$$
其中 $t_{\text{main}}$ 是主模型单次 forward 的延迟。引入 MTP speculative decoding 后，每个周期包含三个部分：并行运行 $k=4$ 个 MTP head 产生 draft token，主模型一次验证 forward，以及接受/拒绝决策。在 $p=0.8$ 的接受率假设下，每个周期平均接受的 token 数为：

$$
E[\text{accept}] = \sum_{i=1}^{k} p^i = p + p^2 + p^3 + p^4 \tag{63}
$$
此式用几何级数刻画了 speculative decoding 每个周期内平均被接受的 draft token 数量，接受率 $p$ 越高，期望收益越大。

$$
= 0.8 + 0.64 + 0.512 + 0.4096 = 2.3616 \tag{64}
$$

如式 (69) 所示，在 $p=0.8$ 的接受率下，每个 speculative decoding 周期平均可以接受约 2.36 个 draft token。式 (55) 的几何级数来源于：第 1 个 draft token 被接受的概率为 $p$，第 2 个为 $p^2$(要求前两个都被接受)，依此类推。由此可得总周期数：

$$
N_{\text{cycle}} = \frac{T}{E[\text{accept}]} = \frac{512}{2.3616} \approx 216.8 \tag{65}
$$
每个周期的总延迟为 draft 延迟加验证延迟。设 MTP head 的延迟为主模型的 $\epsilon = 5\%$，则：

$$
t_{\text{cycle}} = k \times \epsilon \times t_{\text{main}} + t_{\text{main}} = 4 \times 0.05 \times t_{\text{main}} + t_{\text{main}} = 1.2 \times t_{\text{main}} \tag{66}
$$

如式 (33) 所示，每个周期需要运行 4 个 MTP head(占主模型 5% 的延迟 each)加上一次主模型验证，总延迟相当于 1.2 次主模型 forward。总延迟为：

$$
\text{Latency}_{\text{MTP}} = N_{\text{cycle}} \times t_{\text{cycle}} = 216.8 \times 1.2 \times t_{\text{main}} \approx 260.2 \times t_{\text{main}} \tag{67}
$$
对比式 (54) 和式 (57)，加速比为：

$$
\text{Speedup} = \frac{\text{Latency}_{\text{baseline}}}{\text{Latency}_{\text{MTP}}} = \frac{512}{260.2} \approx 1.97 \tag{68}
$$
如式 (58) 所示，MTP 带来了约 2 倍的推理加速。

让我们进一步分析不同接受率对加速比的影响：

| 接受率 $p$ | 平均接受 token 数 | 加速比 |
|-----------|------------------|--------|
| 60% | 1.30 | 1.08 |
| 70% | 1.72 | 1.39 |
| 80% | 2.36 | 1.97 |
| 85% | 2.87 | 2.40 |
| 90% | 3.44 | 2.88 |

从这张表可以看出，**MTP 的加速效果对接受率极其敏感。** 当接受率从 80% 提升到 90% 时，加速比从 2 倍跃升到接近 3 倍; 而当接受率低于 70% 时，加速效果就变得微乎其微(因为 draft 和验证的开销开始抵消节省的时间)。

这也是为什么 V4 将 MTP 头与主模型深度耦合(共享底层参数)的原因——只有 draft 分布与主模型分布高度对齐，才能实现 80% 以上的接受率。独立的 draft model 很难达到这种对齐度。

## 7. 局限性与边界条件

世界上没有包治百病的算法，DeepSeek-V4 的架构创新虽然在多个维度上取得了突破，但它同样面临一系列深层的局限性和边界条件。理解这些局限，对于正确地评估 V4 的适用场景、以及预判下一代架构的演进方向至关重要。

### 7.1 MoE 负载均衡的持续挑战

DeepSeek-V3 的无辅助损失负载均衡(Auxiliary-Loss-Free Load Balancing)在 V4 中得到了保留和微调，但它本质上是一种**启发式(Heuristic)** 方法，而非从第一性原理出发的最优解。

**问题的数学根源：**

MoE 的路由函数通常是一个 softmax 变换：

$$
g_i(x) = \frac{\exp(W_r^{(i)} \cdot x + b_i)}{\sum_{j} \exp(W_r^{(j)} \cdot x + b_j)} \tag{69}
$$

其中 $b_i$ 是可学习的专家偏置项。负载均衡的目标是让每个专家被激活的次数大致相等，即 $\mathbb{E}[g_i(x)] \approx 1/N_{expert}$。无辅助损失方法通过在线调整 $b_i$ 来实现这一目标：如果专家 $i$ 被激活得太多，就降低 $b_i$; 如果太少，就提高 $b_i$。

然而，这种在线调整存在两个根本性问题：

**第一，偏置调整的滞后性。** $b_i$ 的更新通常基于一个滑动窗口内的统计信息(如最近 1000 个 token 的激活频率)。如果数据分布发生突变(如从通用对话突然切换到专业法律文本)，偏置项需要一段时间才能适应新的分布。在这段适应期内，某些专家可能持续过载，而另一些专家持续闲置。

**第二，负载均衡与模型性能的根本张力。** 理论上，最优的负载均衡应该是让每个专家恰好处理它最擅长的数据子集。如果数据分布本身是不均匀的(如 80% 的训练数据是通用文本，20% 是代码)，那么强制每个专家处理等量的数据反而是次优的——代码专家应该被更多地用于代码数据。无辅助损失方法无法感知"数据质量"或"领域匹配度"，它只能机械地统计激活频率。

V4 通过增加专家数量(512 个)和每 token 激活专家数(12 个)来缓解这一问题，但这只是将问题推迟到了更大的规模。当专家数量进一步扩展到 1024 或 2048 时，偏置项的调节能力将最终饱和。

**更深层的物理限制：**

从信息论的角度来看，MoE 的路由本质上是一个聚类问题——将输入空间划分成 $N_{expert}$ 个区域，每个区域由一个专家处理。理想情况下，这个聚类应该由数据本身的结构决定(如代码数据聚到代码专家，数学数据聚到数学专家)。但当前的路由机制是一个简单的线性分类器($W_r \cdot x$)，它的表达能力有限。如果数据的真实分布需要非线性的、高维的决策边界来划分，线性 Router 就无法找到最优的分配方案。

未来的可能解决方向包括：使用更复杂的路由网络(如多层 MLP 或注意力 Router)、引入显式的领域标签作为路由辅助信号、或者完全放弃确定性路由而采用基于 VAE 的随机路由。

### 7.2 长上下文(>128K)的注意力精度衰减

V4 将上下文长度扩展到了 256K，CSA/HCA 和 Gated Attention 都在不同程度上支撑了这一扩展。但 256K 已经接近当前注意力机制的**物理极限**——不是工程实现的极限，而是注意力机制本身的数学极限。

**精度衰减的根本原因：**

注意力的 softmax 操作具有"赢家通吃(Winner-Takes-All)"的特性。当序列长度 $N$ 很大时，即使 Key 向量之间的差异很小，softmax 的指数放大效应也会使得注意力权重分布极度尖锐——一个或少数几个 token 获得接近 1 的权重，其他所有 token 的权重趋近于 0。

数学上，softmax 的输出对输入的敏感度随着输入维度的增加而增加。对于 $N = 256K$，即使 Query 与两个不同 Key 的点积差异仅为 0.1，经过 softmax 后它们的权重比可能达到 $e^{0.1} \approx 1.105$。当这种微小差异在 256K 个候选上累积时，模型可能完全忽略掉那些"稍不重要"的信息。

在长文档检索任务中，这一问题的表现尤为明显。假设用户在 256K 的上下文中询问一个具体细节(如"第三章第二节提到的那个实验用了多少样本？")。模型需要在 256K 个位置中精确定位到"第三章第二节"和"实验样本"这两个关键信息。但由于 softmax 的尖锐化效应，模型可能在 prefill 阶段就将注意力过度集中在某些局部区域，而错过了真正相关的位置。

**CSA/HCA 在这一问题上的局限：**

HCA 的浅层局部窗口策略意味着：在浅层，距离超过 1024 的 token 之间无法直接交互。浅层的信息必须通过逐层传递才能到达远处。对于 256K 的序列，信息需要经过约 256 层(256K / 1024)才能从序列头部传到尾部，但 V4 只有 72 层。这意味着在深层注意力接管之前，浅层无法建立跨越整个序列的依赖关系。

虽然深层注意力(全局注意力)可以理论上建立任意两个位置之间的联系，但深层注意力的输入已经是经过多层变换后的高度抽象表征。如果浅层在局部窗口的限制下丢失了某些关键的细粒度信息(如精确的章节编号、页码引用)，深层注意力也无从恢复这些信息。

**Gated Attention 的双刃剑效应：**

Gated Attention 通过将不重要 token 的 Gate 值压低来过滤噪声。但在超长序列中，"重要性"的判断本身就容易出错。如果模型错误地将一个关键信息 token 的 Gate 值设为接近 0(可能因为该 token 的表面特征与常见噪声相似)，这个信息就被永久地从注意力传播中移除了。在短序列中，这种错误可能是局部性的、可恢复的; 在 256K 的超长序列中，一次 Gate 误判可能导致模型"错过"一整段关键信息。

**当前的行业解决方案及局限：**

业界正在探索多种长上下文方案，但各有局限：
- **Ring Attention / Striped Attention**：将长序列分块处理，降低内存峰值，但没有改变 softmax 尖锐化的问题。

- **Linear Attention / RNN 风格 Attention**：用核技巧或状态空间模型替代 softmax，将复杂度降到 $O(N)$，但表达能力弱于标准注意力。

- **检索增强(RAG)** ：将长文档切分，只检索相关片段输入模型。这绕过了长上下文问题，但引入了检索错误的风险。

V4 在 256K 范围内的表现是业界领先的，但如果未来要扩展到 1M 甚至 10M token，仅靠当前架构的渐进式改良可能不足以维持精度。这需要注意力机制本身的范式革新——例如完全抛弃 softmax 的基于状态的注意力，或者引入显式的记忆读写机制。

### 7.3 MTP 在代码生成中的局限性

MTP 在一般性文本生成中表现出色，但在代码生成这一特定领域面临独特的挑战。

**语法严格的级联错误：**

自然语言具有一定的容错性——即使一个单词预测错误，读者通常仍能理解句子的意思。但代码是"刚性"的：一个括号不匹配、一个缩进错误、一个变量名拼写错误，都可能导致整个程序无法编译或运行。

在 MTP 的 speculative decoding 中，如果第 1 个 draft token(如一个左括号 `(`)被接受，但第 2 个 draft token(如一个错误的变量名)被接受后又导致了后续的语法错误，这种错误具有**级联(Cascade)** 特性。一旦代码的结构被破坏，后续所有 token 的预测都会基于一个错误的语法上下文，导致错误呈指数级扩散。

更为棘手的是，代码中的**长距离依赖**比自然语言更常见。一个函数在开头定义的参数，可能在整个函数体的数十行之后被引用。MTP 的第 3、第 4 个预测头在预测这些远距离引用时，由于只能看到当前前缀(而看不到未来中间可能出现的其他定义)，其预测准确率会显著下降。

**V4 的缓解措施及不足：**

V4 在代码数据上训练时，对 MTP 的权重进行了调整：降低远期预测头的权重(更小的 $\lambda_i$)，并引入语法感知的验证机制(在 speculative decoding 的验证阶段，使用轻量级解析器检查候选代码的语法有效性)。但这些措施只能缓解，无法根除问题。

在 HumanEval 和 MBPP 等代码生成基准上的测试显示，当 MTP speculative decoding 应用于代码生成时，接受率从自然语言的 80% 下降到约 65%。根据 6.4 节的加速比表格，65% 的接受率只能带来约 1.2 倍的加速——收益大打折扣。

**根本矛盾：**

MTP 假设未来的 token 可以从当前上下文以较高的置信度预测。这一假设对于自然语言大致成立(因为自然语言有大量冗余和可预测性)，但对于代码并不总是成立。代码是高度结构化的、低熵的(一旦确定了算法逻辑，token 序列几乎是确定性的)，但这种确定性依赖于对全局结构的把握，而非局部模式的延续。

### 7.4 开源 vs 闭源的差距

尽管 DeepSeek-V4 在架构创新和训练效率上取得了令人瞩目的成就，但客观地说，开源模型与最顶尖的闭源模型之间仍然存在不可忽视的差距。

**差距不在架构，而在数据。**

大模型的能力上限由三个因素共同决定：架构、数据、算力。在架构层面，DeepSeek-V4 的 CSA/HCA、Gated Attention、MTP 等创新已经达到了甚至超越了闭源模型的技术水平。在算力层面，虽然国内团队在高端芯片上受到限制，但通过算法优化和集群工程，V4 的训练算力投入仍然处于第一梯队。

真正的差距在于**数据的质量、多样性和清洗工艺。**

闭源模型(如 GPT-4o、Claude 3.5 Sonnet、Gemini 1.5 Pro)的背后是拥有数十年积累的科技巨头。它们的数据 pipeline 不仅规模庞大，而且经过了高度专业化的处理：
- **多模态数据融合**：闭源模型通常同时训练于文本、图像、音频、视频数据，不同模态之间的知识可以相互迁移。V4 目前仍是纯文本模型(多模态版本可能在后续发布)。

- **高质量对话数据的获取**：顶级闭源模型拥有来自数亿用户的真实交互数据，这些数据包含了极其丰富的人类意图分布。开源模型只能依赖公开数据集和合成数据，在分布覆盖上天然不足。

- **数据清洗的"最后一公里"**：顶级的数据清洗工艺涉及大量的商业机密和人工审核流程。例如，如何识别和去除训练数据中的 PII(个人身份信息)、如何平衡不同文化背景下的价值观、如何处理有争议的内容——这些问题的解决不仅需要算法，更需要庞大的运营团队和法务支持。

**后训练的"黑箱"差距：**

R1 的开源是彻底的——它不仅开放了模型权重，还公开了训练流程和关键超参数。但 R1 的成功并不意味着开源社区可以完全复制闭源模型的后训练 pipeline。

闭源模型的 RLHF / RLAF pipeline 中，奖励模型(Reward Model)的训练数据通常是保密的。这些 RM 训练数据来自于专业的人工标注团队，标注者经过严格的培训和校准。开源社区虽然可以构建类似的 RM，但在标注质量、标注规模、以及标注者与目标用户群体的一致性上，仍然存在差距。

此外，闭源模型可以进行实时的在线 A/B 测试——将模型的不同版本部署给真实用户，根据用户反馈持续迭代。这种"在线进化"的能力是开源模型无法获得的(除非开源模型有广泛的商业部署并收集反馈)。

**V4 的意义在于缩小差距，而非消除差距。**

V4 通过极致的架构效率和算法创新，用更少的资源达到了接近闭源模型的性能。它将"可用的大模型"的门槛大幅降低——中小型企业、学术机构、个人开发者都可以基于 V4 构建高质量的应用。但在最极端的边界条件下(如需要理解整部电影的视频内容、需要进行多轮复杂的多模态推理、需要处理高度敏感的专业领域任务)，闭源模型仍然可能保持微弱但关键的优势。

## 8. 演进与承上启下

### 8.1 DeepSeek 的下一步：原生多模态与 Agent 能力

DeepSeek-V4 在纯文本大模型领域已经接近了当前架构的帕累托前沿。按照技术演进的规律，下一步的突破将来自两个方向：**原生多模态(Native Multimodality)** 和**Agent 能力(Agentic Capabilities)** 。

**原生多模态：**

当前的多模态大模型(如 GPT-4V、Qwen-VL)通常采用"拼接式"架构——一个预训练好的视觉Encoder (如 ViT)将图像转换为 token 序列，然后输入到一个预训练好的语言模型中。这种架构虽然工程上简单，但它将视觉和语言两个模态的信息压缩到了同一个序列空间中，造成了信息损失(如图像的空间关系被线性化为一维序列)和计算浪费(视觉 token 数量巨大，注意力复杂度随视觉 token 数平方增长)。

原生多模态的终极形态应该是一个**统一的、模态无关的表征空间**。在这个空间中，文本、图像、音频、视频的特征被编码为同一种类型的向量，模型通过统一的注意力机制在不同模态之间自由交换信息。CSA/HCA 混合注意力为此提供了基础——HCA 的局部窗口机制非常适合处理图像的局部空间关系(如一个像素只需要关注它的邻居)，而 CSA 的共享 KV 机制可以跨模态共享信息(如图像的某个区域和描述该区域的文本可以共享 KV 投影)。

Gated Attention 在多模态场景下也有自然的延伸：不同模态的信息密度差异巨大。一张高分辨率图像可能包含数百万像素，但只有少数区域与当前的文本查询相关。Gated Attention 可以学会自动抑制图像中的背景区域，只保留与文本 query 相关的视觉 patch。

**Agent 能力：**

Agent 能力的核心在于**规划的递归深度**和**工具调用的可靠性**。当前的模型(包括 V4)在单步推理上已经非常强大，但在需要数十步甚至数百步规划的复杂任务中(如"帮我分析这家公司过去五年的财报，找出潜在风险，并生成一份投资建议书")，模型仍然容易在中间的某一步偏离正确路径，然后一步错步步错。

V4 的 MTP 架构为 Agent 能力提供了有趣的可能性。MTP 的多个预测头可以被重新解释为"多路径规划"——第 1 个头预测"下一步最可能的动作"，第 2 个头预测"如果第 1 步顺利，接下来应该做什么"，以此类推。这种"前瞻性规划"能力如果与外部工具的执行反馈结合，可以让模型在执行动作之前就预判多个步骤后的结果，从而做出更优的决策。

OPD 家族算法在 Agent 训练中的应用前景同样广阔。Agent 的执行轨迹天然是 On-Policy 的——模型自己决定调用哪个工具、传入什么参数。G-OPD 可以用于在 Agent 的轨迹上蒸馏更强大的教师 Agent 的策略，而 SCOPE 可以用于过滤那些"看起来成功了但实际上是碰巧"的 Agent 轨迹。

### 8.2 对国内大模型生态的影响

DeepSeek-V4 的发布，对国内大模型生态的影响是多层次、深远的。

**第一，技术路线的验证与扩散。**

V2 的 MLA、V3 的 MoE、R1 的 GRPO、V4 的 CSA/HCA+MTP，每一步都代表着一条被验证可行的技术路线。国内的其他大模型团队(如智谱、百川、MiniMax、阶跃星辰)不需要再"盲人摸象"式地探索架构，而是可以直接在 DeepSeek 验证过的基础上进行改良和创新。这种"技术外溢"效应极大地提升了整个国内大模型产业的研发效率。

**第二，推理成本的下探。**

V4 的极致效率优化(CSA/HCA 的 KV Cache 压缩、MTP 的推理加速)直接降低了模型部署的门槛。对于国内的中小企业和开发者来说，这意味着他们可以用更少的 GPU 资源、更低的云成本来运行高性能的大模型应用。在当前的算力受限环境下，这种成本优势具有战略意义。

**第三，开源社区的正向循环。**

DeepSeek 的完全开源策略建立了一个正向循环：开源模型 → 社区反馈和贡献 → 模型改进 → 更强的开源模型。V4 的架构细节一旦被社区充分理解和复现，就会催生大量的衍生工作——更高效的实现(如更适合特定硬件的 kernel 优化)、更多的下游适配(如领域专用的微调版本)、以及更丰富的工具生态(如针对 V4 架构的量化、压缩、部署工具)。

**第四，国际竞争格局的重塑。**

DeepSeek 的成功向全球证明：中国团队不仅在应用层(如短视频、电商)有创新能力，在基础层的 AI 架构研究中同样可以达到世界顶尖水平。这在一定程度上打破了"中国 AI 只会跟随、不会创新"的刻板印象，为国内的 AI 研究争取到了更多的国际话语权和合作机会。

## 9. 总结与参考文献

### 9.1 总结

DeepSeek-V4 代表了 2025-2026 年大模型架构演进的重要里程碑。它不是单一技术的突破，而是多个互补创新的系统性融合：

1. **CSA/HCA 混合注意力**通过跨层共享 KV 和层次化窗口注意力，将 256K 长上下文的 KV Cache 降低了约 70 倍，同时将注意力计算量降低了约 44%，从根本上解决了长序列推理的显存和计算瓶颈。

2. **Gated Attention** 引入了可学习的 token 级别重要性门控，使模型能够自适应地抑制噪声和冗余信息。这种"软稀疏"机制与 HCA 的"硬稀疏"形成了互补，在不牺牲表达能力的前提下提升了注意力的信噪比。

3. **MTP 多词元预测** 从辅助训练目标升级为核心推理组件，通过内置的 speculative decoding 将推理速度提升了约 2 倍。其与主模型的深度参数共享确保了高达 80% 以上的 draft 接受率。

4. **OPD 家族算法的深度集成**(G-OPD 蒸馏、GRPO 强化学习、SCOPE 自蒸馏过滤)将后训练的算力消耗降低到了传统 RL 方案的约 1/10，同时保持了甚至超越了纯 RL 的效果。

这些组件围绕一个统一的设计哲学有机整合：**让每一个计算 FLOP、每一字节显存、每一瓦特功耗都产生最大的模型能力收益。** 在算力受限的时代背景下，这种对效率的极致追求不仅是工程上的优化，更是一种战略层面的生存智慧。
