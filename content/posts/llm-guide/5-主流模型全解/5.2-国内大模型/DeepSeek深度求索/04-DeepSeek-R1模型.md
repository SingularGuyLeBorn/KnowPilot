# 04 DeepSeek-R1 完整技术解读：从 R1-Zero 到 GRPO，纯强化学习的推理革命

> **阅读前置要求**：本文默认读者已熟悉 Transformer 基础架构、自回归语言模型训练目标，以及强化学习(Reinforcement Learning)的基本概念(策略、奖励、优势函数)。对于 PPO(Proximal Policy Optimization)的理论背景，具备基础了解将有助于理解 GRPO 的设计动机。

---

## 1. 背景与核心痛点

### 1.1 推理能力的涌现：从"模式匹配"到"深度思考"

大语言模型在 2020-2023 年间取得的能力突破，本质上仍停留在"高级模式匹配"的层面。GPT-3、PaLM、LLaMA 等模型通过在海量文本上进行 next-token prediction 的预训练，学会了语言的结构规律、世界的常识关联、以及各类任务的表面模式。当你问它们"法国的首都是哪里"，它们能准确回答"巴黎"; 当你让它们"写一首关于秋天的诗"，它们能输出韵脚工整的段落。但这些能力的核心机制是**统计关联的提取与复现**——模型在训练数据中见过足够多的"法国-巴黎"配对，足够多的秋天主题诗歌，因此能够"回忆"出合理的答案。

然而，有一类任务无法通过简单的模式匹配来解决：**需要多步逻辑推导的复杂推理**。考虑以下数学问题：

> 求所有满足 a^2 + b^2 + c^2 = a^2 * b^2 的整数三元组 (a, b, c)。

这个问题没有简单的"记忆-提取"路径。解决它需要：分析方程的对称性、讨论 a, b 的奇偶性、利用模运算排除不可能的情况、最终构造出唯一解。人类数学家解决这类问题需要纸笔推演、试错、回溯——这是一个动态的、非线性的思考过程。

2024 年，OpenAI 发布的 o1 系列首次证明了：大模型在推理阶段生成足够长的思维链(Chain-of-Thought, CoT)，可以显著提升这类复杂推理任务的准确率。o1 的核心洞察是**测试时计算扩展(Test-time Compute Scaling)** ——与其在训练时堆砌更多参数，不如在推理时给模型更多"思考时间"，让它通过自我对话逐步逼近正确答案。

但 o1 是一个完全闭源的系统。外界无从得知它的训练细节：它使用了多少人类标注的 CoT 数据？它的奖励模型是如何构建的？它的推理链长度是如何控制的？这种黑箱状态使得学术界和开源社区无法验证、复现或改进这一范式。

### 1.2 后训练中的核心矛盾：人类标注的瓶颈

在 o1 之前，提升模型推理能力的主流路径是**监督微调(Supervised Fine-Tuning, SFT)** 。具体做法是：雇佣大量人类标注者，让他们手动写出复杂问题的详细解题过程(即 CoT 数据)，然后用这些数据微调预训练好的基座模型。

这条路径存在三个根本性瓶颈：

**第一，标注成本极高。** 一条高质量的数学推理数据可能需要专业数学家花费数十分钟甚至数小时来撰写。对于 AIME(美国数学邀请赛)级别的竞赛题，能够写出正确且优雅解答的标注者本身就是稀缺资源。当模型需要处理的领域从数学扩展到物理、化学、编程、法律推理时，找到同时具备领域专业知识和教学表达能力的标注者变得更加困难。

**第二，标注质量的天花板。** 人类标注者的推理过程受限于自身的认知能力。对于那些连人类专家都感到困难的问题(如国际数学奥林匹克竞赛的最后几题)，我们无法获得"完美"的标注数据。SFT 的天花板因此被人类能力的上限所束缚——模型最多只能学会"模仿人类的最好水平"，而无法超越。

**第三，分布覆盖不足。** 人类标注数据只能覆盖有限的场景。模型在 SFT 阶段见过的推理模式是有限的，一旦遇到训练分布之外的变体问题，就容易失效。更深层的问题是：我们甚至不知道"理想的推理过程"应该长什么样——也许最高效的推理路径是人类从未想到过的。

**DeepSeek-R1 的核心命题是：能否完全摆脱对人类标注 CoT 数据的依赖，仅通过强化学习(RL)让模型自主发现推理策略？**

这一命题如果成立，将彻底改变后训练的经济学：不再需要昂贵的人类标注团队，不再受限于人类认知的天花板，模型的推理能力将通过自我博弈(Self-Play)持续进化。

### 1.3 R1 之前的尝试：为什么纯 RL 长期失败？

纯 RL 训练语言模型的想法并非 DeepSeek 首创。早在 2019-2020 年，就有研究者尝试使用 PPO 等 RL 算法来微调 GPT 模型，以改善生成质量。但这些早期尝试普遍面临几个致命问题：

**奖励稀疏(Sparse Reward)** 。对于开放式生成任务(如创意写作)，很难设计一个自动化的奖励函数来判断"这篇故事好不好"。即使是数学问题，如果奖励只在最终答案正确时才给出 +1、错误时给 0，那么模型在探索的绝大多数轨迹上获得的反馈都是零信号，学习效率极低。

**策略梯度的高方差**。语言模型的动作空间是离散的、巨大的(词表大小通常在 10 万级别)。在这么大的动作空间上进行蒙特卡洛采样，策略梯度的估计方差极高。如果没有一个准确的价值函数(Value Function)来引导，模型很容易在随机噪声中迷失方向。

**灾难性遗忘(Catastrophic Forgetting)** 。RL 微调往往导致模型丧失预训练阶段学到的通用能力。一个经过纯 RL 微调的模型可能在数学推理上有所提升，但在日常对话、常识问答上的表现却严重退化，因为它为了迎合奖励函数而"过度特化"。

**DeepSeek-R1 的成功，在于它系统性地解决了上述所有问题。** 它通过 GRPO 算法消除了对价值网络的依赖，通过规则-based 的奖励函数实现了自动化评估，通过冷启动数据和蒸馏技术防止了能力退化。更重要的是，R1 证明了：**纯 RL 不仅能提升能力，还能让模型自发涌现出人类从未教过的推理行为。**

## 2. 为什么重要

### 2.1 开源社区的"o1 时刻"

DeepSeek-R1 的发布日期是 2025 年 1 月，但其影响迅速波及整个 AI 社区。在此之前，o1 的推理能力被视为闭源巨头独有的"黑科技"——社区知道它很强，但不知道如何实现。R1 以完全开源的姿态(模型权重、训练流程、技术报告全部公开)向世人展示了：**开源模型同样可以达到甚至超越 o1-preview 的推理水平。**

在 AIME 2024 数学竞赛基准上，R1 的准确率达到了约 79.8%，超过了 o1-preview 的约 44.6% 和 o1 的约 74.4%。在 MATH-500 基准上，R1 达到了 97.3% 的准确率。这些数字意味着：R1 不仅在开源社区内树立了新标杆，甚至在某些指标上超越了当时最先进的闭源模型。

更深远的影响在于方法论层面。R1 证明了纯 RL 路径的可行性，这为后续所有推理模型的训练提供了一个全新的基础范式。在 R1 之后，Kimi k1.5、Qwen3、GLM-5 等模型纷纷引入或改良了 RL-based 的推理训练流程。R1 如同一块投入湖面的巨石，激起的涟漪正在重塑整个后训练领域的技术版图。

### 2.2 算力经济学的范式转移

R1 的训练成本 reportedly 仅为 557.6 万美元(基于 2048 块 H800 GPU，约 2 个月的训练周期)。这一数字在当时的旗舰模型训练中堪称"廉价"——同期闭源模型的训练成本通常以数千万甚至数亿美元计。

R1 的成本优势并非来自"偷工减料"，而是来自**算法效率的极致优化**。GRPO 算法相比 PPO 节省了约 50% 的显存开销(无需维护价值网络)，规则-based 的奖励函数消除了对昂贵奖励模型的依赖，冷启动数据的小规模 SFT 阶段(仅数千条数据)大幅减少了人工标注的投入。

对于学术界和中小型企业而言，R1 的成功传递了一个重要信号：**在算力受限的环境下，通过聪明的算法设计，依然可以训练出世界一流的推理模型。** 这使得高质量推理能力不再是只有少数科技巨头才能负担的奢侈品，而是成为了可以被广泛复现、改进和定制的基础设施。

## 3. R1-Zero：纯强化学习 without SFT 的大胆实验

### 3.1 实验设计：从零开始的推理涌现

在构建完整的 R1 之前，DeepSeek 团队首先进行了一个极端的实验：**R1-Zero**。这个实验的设计极其简洁——甚至可以说是"粗暴"：

1. 取一个预训练好的 DeepSeek-V3-Base 模型(一个具备基础语言理解和生成能力的基座模型)。

2. **不进行任何 SFT**(即不给模型看任何人类写的推理过程)。
3. 直接使用 GRPO 算法进行强化学习微调，奖励函数仅基于规则：最终答案正确得 +1，错误得 0。

这个实验的核心问题是：**如果模型从未被"教过"如何推理，仅凭"答对就有奖励、答错就无奖励"这一最原始的反馈信号，它能否自己"发明"出有效的推理策略？**

答案是：**不仅能，而且涌现出的行为令人震惊。**

### 3.2 自发涌现的推理行为

R1-Zero 在 RL 训练过程中展现出了几种人类从未显式教授的行为模式：

**自我验证(Self-Verification)** 。模型学会了在生成答案后，自动插入验证步骤来检查自己的结论。例如，在解决数学问题时，模型可能会写出：

```
让我验证一下这个结果... 如果 x = 3，代入原方程左边 = 9，右边 = 9，等式成立。因此答案是正确的。
```

这种自我验证行为不是通过 SFT 模仿人类数据学来的——训练数据中根本没有这样的示例。它是模型在 RL 过程中自发发现的：通过验证，模型能够提前发现自己可能犯的错误，从而提高获得正奖励的概率。

**反思与回溯(Reflection and Backtracking)** 。当模型发现当前的推理路径走入死胡同时，它会生成类似"等等，这个思路似乎有问题，让我重新考虑"的文本，然后尝试另一种方法。这种"意识到自己错了并主动纠正"的能力，在传统的 SFT 模型中极为罕见——SFT 模型通常只会沿着训练数据中的单一路径生成，缺乏动态调整的能力。

**长推理链的自发延长**。在训练初期，模型的回答通常较短(数百个 token)。随着 RL 训练的深入，模型逐渐学会生成越来越长的推理链，有时达到数千甚至上万 token。更重要的是，这种延长不是无意义的重复，而是每一步都在进行实质性的逻辑推进。

### 3.3 R1-Zero 的局限性：可读性与语言混杂

尽管 R1-Zero 展示了纯 RL 的强大潜力，但它也存在明显的问题：

**可读性差**。由于没有经过人类数据的对齐，R1-Zero 生成的推理链往往结构混乱、缺乏清晰的分步标注、符号使用不统一。对于人类读者来说，很难跟随模型的思路。

**语言混杂(Language Mixing)** 。模型在推理过程中会在不同语言之间无规律地切换。例如，一个以中文提出的问题，模型可能在推理的前半段用中文，中间突然切换成英文，最后又切回中文。这种现象的原因是：RL 的奖励只关心最终答案的正确性，不关心推理过程的"语言纯度"。模型在探索过程中发现，某些概念用英文表达更容易得到正确答案(可能是因为预训练数据中英文的数学表达更丰富)，因此形成了混杂的策略。

**这些局限性直接催生了 R1(而非 R1-Zero)的设计动机**：我们需要在保留纯 RL 的自我发现能力的同时，引入少量人类数据来"驯化"模型的输出格式，使其既具备强大的推理能力，又保持对人类友好的可读性。

## 4. R1 的训练 Pipeline：冷启动、SFT 与 GRPO 的协同

### 4.1 四阶段 Pipeline 概览

DeepSeek-R1 的完整训练流程可以概括为四个阶段：

```
DeepSeek-V3-Base (预训练基座)
    │
    ▼
Stage 1: 冷启动 SFT (Cold Start SFT)
    - 数据：数千条高质量人类标注 CoT 数据
    - 目标：教会模型基本的推理格式和可读性
    │
    ▼
Stage 2: 面向推理的 RL (Reasoning-Oriented RL)
    - 数据：数学、代码、逻辑等可规则验证的任务
    - 目标：通过 GRPO 激发深度推理能力
    │
    ▼
Stage 3: 拒绝采样与 SFT (Rejection Sampling + SFT)
    - 数据：RL 阶段模型生成的高质量回答 + 通用任务数据
    - 目标：扩展能力到非推理领域，防止灾难性遗忘
    │
    ▼
Stage 4: 全场景 RL (General RL)
    - 数据：所有任务的混合
    - 目标：对齐人类偏好，提升通用性和有用性
```

这个 pipeline 的设计哲学是：**先用少量人类数据做"播种"(冷启动)，然后用 RL 让模型自主"生长"(推理能力涌现)，最后用蒸馏和拒绝采样来"修剪"(扩展通用能力、防止退化)。**

### 4.2 冷启动数据：小规模但高质量的"种子"

冷启动阶段是 R1 与 R1-Zero 的关键差异。DeepSeek 团队收集了数千条(而非数百万条)高质量的人工标注 CoT 数据，用于对 V3-Base 进行初步的 SFT。这些数据经过精心设计，具备以下特征：

**格式统一**。每条数据都遵循清晰的结构：问题陈述 → 分步推理(每步有明确的步骤编号和说明)→ 最终答案。这种统一格式为后续的 RL 阶段提供了一个"干净"的起点——模型已经知道"推理应该是什么样子的"，RL 只需要优化推理的"内容质量"。

**领域聚焦**。冷启动数据主要集中在数学和代码领域，因为这些领域有明确的规则可以判断答案正确性，适合后续的 RL 训练。数据难度经过筛选，覆盖了从基础到竞赛级别的梯度。

**语言纯净**。与 R1-Zero 不同，冷启动数据严格控制语言一致性——中文问题用中文推理，英文问题用英文推理。这为模型建立了"语言一致性"的初始偏好。

冷启动阶段的 SFT 通常只进行 1-2 个 epoch，学习率较低(约 1e-5 量级)。其目的不是让模型"学会推理"(数据量太少，不足以覆盖所有推理模式)，而是让模型"学会如何表达推理"——获得基本的格式规范、可读性标准和语言一致性。

### 4.3 面向推理的 RL：GRPO 的核心战场

在冷启动 SFT 之后，模型进入 RL 阶段。这是 R1 推理能力真正涌现的核心环节。RL 的数据来自可规则验证的领域：

- **数学**：AIME、AMC、MATH 等竞赛数据，以及自建的更大规模数学题库。奖励基于最终答案的数值正确性(通过符号计算或数值比对验证)。

- **代码**：LeetCode、Codeforces 等编程竞赛题目。奖励基于代码是否能通过所有隐藏的单元测试用例。

- **逻辑谜题**：如 24 点游戏、数独变体、逻辑推理题。奖励基于是否得出唯一正确答案。

RL 阶段持续数千到数万步，模型在每一步都会生成多条回答(GRPO 的组采样)，通过组内相对排名来计算优势并更新策略。

## 5. GRPO：Group Relative Policy Optimization 的数学推导

### 5.1 从 PPO 到 GRPO：为什么要抛弃价值网络？

在深入 GRPO 的数学细节之前，我们需要先理解它试图解决什么问题。

传统的 RL 微调语言模型通常使用 **PPO(Proximal Policy Optimization)** 算法。PPO 的核心机制包括：

1. **策略网络(Policy Network)** pi_theta：即语言模型本身，负责生成 token。

2. **价值网络(Value Network)** V_phi：一个独立的神经网络，负责估计给定状态下未来能获得的期望累积奖励。

3. **优势函数(Advantage Function)** ：A(s_t, a_t) = Q(s_t, a_t) - V(s_t)，表示"在当前状态下采取动作 a_t 比平均水平好多少"。

PPO 的问题在于：**价值网络本身也是一个大模型。** 在语言模型的 RL 场景中，状态 s_t 是到目前为止生成的所有 token 的上下文，其价值估计需要理解整个序列的语义和任务目标。因此，V_phi 通常是一个与策略模型同等规模(或接近规模)的模型。

这意味着：
- **显存开销翻倍**。如果策略模型是 671B 总参数的 MoE(如 V3-Base)，价值网络可能也需要数百亿参数。在 RL 训练中，两个大模型同时驻留显存，极大地限制了 batch size 和序列长度。

- **训练不稳定**。价值网络的估计误差会直接传递到优势函数的计算中。如果 V_phi 对某个状态的价值估计偏低，那么即使是一个好的动作也会被赋予负的优势，导致策略更新方向错误。

- **实现复杂**。价值网络需要与策略网络同步更新，超参数调优(如价值损失的权重系数)非常敏感。

**GRPO 的核心洞察是：对于语言模型的 RL 场景，我们可以用"组内相对表现"来替代"绝对价值估计"。** 具体而言，对于同一个问题，让模型生成一组(Group)回答，然后用这组回答的奖励分布来计算每个回答的相对优势——不需要价值网络，不需要估计绝对价值，只需要比较"这条回答在这组里算好还是算差"。

### 5.2 GRPO 的数学形式

设输入问题为 x，策略模型为 pi_theta。GRPO 的执行流程如下：

**步骤 1：组采样(Group Sampling)** 

对于每个问题 x，从当前策略中采样 G 条独立的回答(通常 G = 8 或 16)：

> {y_1, y_2, ..., y_G} ~ pi_theta.

| x)

这里的 y_i 是完整的 token 序列(包含推理链和最终答案)。采样过程使用温度参数 tau 来控制探索程度——较高的温度鼓励多样性，较低的温度倾向于高概率输出。

**步骤 2：奖励评估(Reward Evaluation)** 

对于每条回答 y_i，使用奖励函数 r(x, y_i) 计算一个标量奖励。R1 使用的是**规则-based 的奖励**：

> r(x, y_i) = 1  if  Answer(y_i) = Answer*(x)
> r(x, y_i) = 0  otherwise

其中 Answer(y_i) 是从回答文本中提取的最终答案(通过正则表达式或解析器)，Answer*(x) 是标准答案。这种二元奖励虽然简单，但完全自动化、无需人工评判、无偏见。

**步骤 3：优势计算(Advantage Computation)——GRPO 的核心**

对于第 i 条回答，其优势函数定义为：

> A_i = (r_i - mean({r_1, r_2, ..., r_G})) / (std({r_1, r_2, ..., r_G}) + epsilon)

其中：
- r_i 是第 i 条回答的奖励。
- mean({r_j}) 是这 G 条回答奖励的算术平均值。
- std({r_j}) 是这 G 条回答奖励的标准差。
- epsilon 是一个小常数(如 1e-8)，防止除以零。

让我们仔细分析这个公式的物理含义：

**分子 r_i - mean** 表示"这条回答相对于同组平均水平的优劣"。如果一条回答的奖励高于组内平均，优势为正，策略梯度将鼓励模型在未来增加这条回答的生成概率; 反之，如果低于平均，优势为负，模型将抑制这条回答的生成概率。

**分母 std + epsilon** 是一个归一化因子。当组内回答的质量差异很大(标准差高)时，归一化因子也大，优势值的绝对幅度被压缩; 当组内回答质量趋同(标准差低)时，即使是微小的奖励差异也会被放大。这种自适应缩放使得 GRPO 的样本效率在不同难度的任务上都能保持稳定。

**为什么这个公式有效？** 关键在于它完全消除了对"绝对价值"的依赖。在 PPO 中，我们需要 V(s) 来估计"从状态 s 出发期望获得多少奖励"; 而在 GRPO 中，我们只需要知道"在这 G 条回答里，这条算好还是算差"。这种相对评估天然适合语言模型场景——我们往往很难判断"一条数学推导值多少分"，但很容易判断"这条推导比那条推导好在哪里"。

**步骤 4：策略更新(Policy Update)** 

GRPO 的策略更新目标与 PPO 类似，使用裁剪后的策略梯度：

> L_GRPO(theta) = - (1/G) * sum_{i=1}^{G} (1/|y_i|) * sum_{t=1}^{|y_i|} min(rho_{i,t} * A_i, clip(rho_{i,t}, 1-epsilon_clip, 1+epsilon_clip) * A_i)

其中 rho_{i,t} = pi_theta(y_{i,t} | x, y_{i,<t}) / pi_theta_old(y_{i,t} | x, y_{i,<t}) 是当前策略与旧策略在位置 t 的概率比。外层的 1/|y_i| 对每条回答内部的所有 token 位置取平均，确保不同长度的回答对总损失的贡献是均衡的。

**裁剪操作** clip(rho, 1-epsilon, 1+epsilon) 是 PPO 的核心创新之一，其目的是防止策略在一次更新中发生剧烈变化。如果某个 token 的概率比 rho 超过了 [1-epsilon, 1+epsilon] 的区间(通常 epsilon = 0.2)，裁剪操作会将其限制在边界值上。这保证了策略更新的"谨慎性"——模型不会突然从一个极端跳到另一个极端。

### 5.3 GRPO 与 PPO 的对比分析

| 维度 | PPO | GRPO |
|------|-----|------|
| 价值网络 | 需要独立的 V_phi | **不需要** |
| 显存开销 | 约 2x 策略模型 | 约 1x 策略模型 |
| 优势估计 | A = Q - V | A = (r - r_bar) / sigma_r |
| 实现复杂度 | 高(需同步训练两个模型) | 低(仅策略模型) |
| 对奖励函数要求 | 可接受连续奖励 | 更适合离散/稀疏奖励 |
| 超参数敏感度 | 高(价值损失权重、GAE lambda 等) | 低(主要调组大小 G) |

### 5.4 数值走查：GRPO 的优势计算实例

让我们用一个极其简单的场景来走查 GRPO 的计算流程。

**场景设定**：
- 问题 x："2 + 3 x 4 = ?"
- 标准答案：14
- 组大小 G = 4
- 4 条采样回答及其提取的最终答案：

| 回答编号 | 最终答案 | 奖励 r_i |
|---------|---------|---------|
| y_1 | 14 | 1(正确) |
| y_2 | 20 | 0(错误，误算为 (2+3)*4) |
| y_3 | 14 | 1(正确) |
| y_4 | 12 | 0(错误) |

**计算组内统计量**：

mean(r) = (1 + 0 + 1 + 0) / 4 = 0.5

std(r) = sqrt((0.25 + 0.25 + 0.25 + 0.25) / 4) = sqrt(0.25) = 0.5

**计算每条回答的优势**：

A_1 = (1 - 0.5) / 0.5 = 1.0
A_2 = (0 - 0.5) / 0.5 = -1.0
A_3 = 1.0
A_4 = -1.0

在这个例子中，正确答案获得了正的优势(+1.0)，错误答案获得了负的优势(-1.0)。策略梯度将推高正确回答中每个 token 的生成概率，同时压低错误回答中每个 token 的生成概率。

**考虑一个更微妙的场景**：如果 4 条回答中有 3 条正确、1 条错误：

| 回答编号 | 奖励 |
|---------|------|
| y_1 | 1 |
| y_2 | 1 |
| y_3 | 1 |
| y_4 | 0 |

mean = 0.75, std = sqrt(0.1875) ≈ 0.433

A_1 = A_2 = A_3 = (1 - 0.75) / 0.433 ≈ 0.577
A_4 = (0 - 0.75) / 0.433 ≈ -1.732

注意这里的不对称性：即使 y_1, y_2, y_3 都答对了，它们的优势仅为 +0.577(而非 +1.0)，因为它们只是"比平均水平稍好"。而 y_4 的优势为 -1.732，惩罚力度更大。这种不对称性反映了 GRPO 的"保守奖励"倾向——当组内大多数回答都正确时，正确回答的边际收益被压缩，而错误回答的惩罚被放大。这鼓励模型追求"稳定正确"而非"碰巧正确"。

### 5.5 GRPO 的简化实现

```python
import torch
import torch.nn.functional as F

def compute_grpo_loss(policy_logits, old_logits, rewards, epsilon=0.2):
    """
    简化版 GRPO 损失计算
    policy_logits: [G, T, V] 当前策略的 logits
    old_logits: [G, T, V] 旧策略的 logits
    rewards: [G] 每条回答的奖励
    """
    G = rewards.shape[0]
    
    # 步骤 1: 计算组内均值和标准差
    mean_reward = rewards.mean()
    std_reward = rewards.std() + 1e-8
    
    # 步骤 2: 计算优势 (对应公式: A_i = (r_i - mean) / std)
    advantages = (rewards - mean_reward) / std_reward
    
    # 步骤 3: 计算概率比
    log_probs = F.log_softmax(policy_logits, dim=-1)
    old_log_probs = F.log_softmax(old_logits, dim=-1).detach()
    ratio = torch.exp(log_probs - old_log_probs)
    
    # 步骤 4: 计算裁剪目标
    adv = advantages.unsqueeze(-1)
    surr1 = ratio * adv
    surr2 = torch.clamp(ratio, 1 - epsilon, 1 + epsilon) * adv
    
    per_sample_loss = torch.min(surr1, surr2).mean(dim=-1)
    loss = -per_sample_loss.mean()
    
    return loss
```

## 6. 冷启动数据的作用：从 R1-Zero 到 R1 的关键跃迁

### 6.1 为什么冷启动不可或缺？

R1-Zero 已经证明了纯 RL 可以让模型自发涌现推理能力，但 R1-Zero 的输出存在两个致命缺陷：可读性差和语言混杂。冷启动 SFT 的作用，就是在 RL 开始之前，为模型植入一些"基础的文明规范"。

类比一下：想象你要训练一个天才儿童成为数学家。R1-Zero 的方法是把儿童关进一间满是数学题的房间，答对就给糖果，答错就不给。这个儿童可能会自己发明出一套独特的解题方法，甚至发现一些数学定理，但他的草稿纸可能写得一团糟，符号使用混乱，而且可能在推导过程中突然从中文切换到拉丁文——因为他发现某个数学概念在拉丁文表达中更容易想到。

冷启动 SFT 相当于在这个过程开始前，先给儿童看几本"优秀数学家的规范笔记"——不是教他具体怎么解题，而是教他：笔记应该有清晰的步骤编号、每个变量要定义清楚、推导要从上到下连贯书写、保持语言统一。这些"元规则"不会限制他的创造力，但会让他的产出对人类(和后续的自动化解析工具)更加友好。

### 6.2 冷启动数据的构造策略

DeepSeek 团队对冷启动数据的构造投入了相当的心思。数据总量控制在数千条级别(远小于传统 SFT 的数百万条)，但每条都经过精心筛选和标注：

**多样性覆盖**。冷启动数据不局限于单一类型的推理。它包括：
- 代数推导(方程求解、不等式证明)
- 几何证明(辅助线构造、全等/相似三角形)
- 组合计数(排列组合、容斥原理)
- 数论问题(整除性、同余方程)
- 基础代码(算法实现、数据结构操作)

这种多样性确保了模型在 RL 阶段不会过度特化到某一类问题上。

**难度梯度**。冷启动数据包含从初中数学到竞赛级别的完整梯度。简单问题教会模型"基本格式"，复杂问题展示"深度推理的组织方式"。

**格式模板化**。每条数据都使用统一的 Markdown 风格模板：

```markdown
## 问题
[问题陈述]

## 思考
步骤 1: [分析]
步骤 2: [推导]
...
步骤 N: [验证]

## 答案
[最终答案]
```

这种模板在后续的 RL 阶段成为模型的"默认输出习惯"——即使 RL 的奖励只关心答案正确性，模型也倾向于保持这种清晰的结构，因为它在冷启动阶段已经被深度编码。

### 6.3 冷启动 vs 大规模 SFT 的本质区别

传统的大模型后训练通常采用"大规模 SFT + 少量 RL"的配方(如 InstructGPT、Llama 2 Chat)。而 R1 采用的是"微量冷启动 SFT + 大量 RL"的配方。这两种路径的差异反映了根本不同的哲学：

**大规模 SFT 的假设**：人类标注数据包含了"正确答案"，模型通过模仿这些数据就能学会推理。RL 只是用来"微调语气"和"对齐偏好"。

**冷启动 + RL 的假设**：人类标注数据只能教会模型"表面的格式"，真正的推理能力必须通过探索和自我反馈来发现。RL 不是微调，而是能力涌现的主要驱动力。

实验证据支持后者的观点：R1-Zero(零 SFT)已经展现了强大的推理能力，只是格式混乱; 经过冷启动 SFT 的 R1 在保持推理能力的同时解决了格式问题。这说明推理能力的"内核"来自 RL，而"外壳"来自冷启动。

## 7. 蒸馏版本：R1-Distill-Qwen 与 R1-Distill-Llama

### 7.1 蒸馏的动机：让小模型拥有大智慧

R1 的基座模型(基于 V3-Base)拥有 671B 总参数(37B 激活参数)，虽然通过 MoE 架构控制了推理成本，但对于端侧部署(智能手机、嵌入式设备)仍然过于庞大。蒸馏(Distillation)的目标是将 R1 的推理能力迁移到更小的模型上(7B、14B、32B 参数级别)，使其在资源受限的环境下也能运行。

但 R1 的蒸馏不是传统的"知识蒸馏"那么简单。传统蒸馏通常是用一个大模型(教师)的 softmax 输出概率来指导小模型(学生)的训练。R1 的蒸馏更进一步：它让学生模型直接在 R1 生成的推理数据上进行 SFT——本质上是用 R1 作为"自动标注机"，为小模型生成海量的高质量 CoT 训练数据。

### 7.2 蒸馏 Pipeline

```
R1 (671B MoE, 教师模型)
    │
    ▼ 生成推理数据
数百万条带推理链的 (问题, 回答) 对
    │
    ▼ SFT 训练
学生模型 (Qwen-7B/14B/32B 或 Llama-8B/70B)
```

蒸馏数据覆盖了数学、代码、科学推理等多个领域。对于每一条问题，R1 被用来生成多个推理版本的回答(使用不同的采样温度)，然后从中筛选出答案正确的那些作为训练数据。

### 7.3 蒸馏效果：小模型的惊人表现

R1 蒸馏版本的表现超出了大多数研究者的预期：

| 模型 | 参数量 | AIME 2024 | MATH-500 | GPQA Diamond |
|------|--------|-----------|----------|--------------|
| R1-Distill-Qwen-32B | 32B | 72.0% | 94.5% | 65.2% |
| R1-Distill-Qwen-14B | 14B | 69.0% | 92.4% | 59.1% |
| R1-Distill-Qwen-7B | 7B | 55.5% | 87.2% | 49.3% |
| R1-Distill-Llama-70B | 70B | 70.1% | 94.0% | 64.2% |
| R1-Distill-Llama-8B | 8B | 50.4% | 85.2% | 46.5% |

这些数据揭示了几个重要规律：

**第一，蒸馏可以跨架构迁移。** R1 基于 DeepSeek 自家的 V3-Base(MoE 架构)，但蒸馏到 Qwen(Dense 架构)和 Llama(Dense 架构)上都能取得优异效果。这说明 R1 学到的"推理知识"不是与特定架构绑定的，而是可以在不同模型家族间迁移的通用能力。

**第二，小模型的性价比极高。** R1-Distill-Qwen-7B(仅 70 亿参数)在 MATH-500 上达到了 87.2% 的准确率，这已经接近了 GPT-4o 的水平。对于端侧部署而言，7B 模型是完全可行的——它可以在现代智能手机上运行(经过 INT4 量化后仅需约 3.5GB 内存)。

**第三，蒸馏数据的质量比数量更重要。** R1 蒸馏使用的是"教师生成的高质量 CoT 数据"，而非人类标注数据。关键发现是：用 R1 生成的 10 万条数据训练的小模型，效果远超用人类标注的 100 万条数据训练的同尺寸模型。这验证了"教师模型的推理质量决定了蒸馏的天花板"这一假设。

### 7.4 蒸馏的局限性：无法蒸馏"探索能力"

然而，蒸馏有一个根本性的局限：**它只能蒸馏"已发现的推理模式"，无法蒸馏"发现新推理模式的能力"。**

R1 本身通过 RL 学会了自我验证、反思、回溯等元推理技能。这些技能不是静态的知识，而是动态的策略——模型在推理过程中根据当前状态自主决定"是否需要验证""是否应该回溯"。蒸馏将 R1 的推理轨迹作为静态数据教给学生模型，学生学会的是"在什么情况下做什么动作"的关联模式，但未必真正理解了"为什么要这么做"。

在实际表现上，这种差异体现在**分布外泛化能力**上。R1-Distill 模型在训练数据分布内的测试题上表现出色，但当遇到全新的题型(如从未见过的数学竞赛题型)时，其表现下降幅度通常大于原始的 R1。这是因为 R1 的 RL 训练赋予了它一种"探索新策略"的元能力，而蒸馏学生更多依赖的是"复现已知策略"。

## 8. 与 OpenAI o1 的对比：两条通往推理之巅的路径

### 8.1 训练范式的差异

| 维度 | OpenAI o1 | DeepSeek-R1 |
|------|-----------|-------------|
| 训练数据透明度 | 完全闭源 | 完全开源 |
| SFT 数据规模 | 未知(推测数百万条人类标注) | 仅数千条冷启动数据 |
| 推理能力来源 | 未知(推测 RL + 大规模 SFT) | 纯 RL(GRPO)为主 |
| 奖励模型 | 未知(推测使用 PRM/ORM) | 纯规则-based，无需奖励模型 |
| 蒸馏开放度 | 仅通过 API 提供 o1-mini | 完整开源蒸馏模型和训练流程 |
| 推理链可见性 | 对用户隐藏 | 完全可见(思考过程透明) |

### 8.2 能力对比

在公开基准上，R1 与 o1 的表现互有胜负：

- **数学推理**：R1 在 AIME 2024 上(~79.8%)优于 o1-preview(~44.6%)，与 o1 正式版(~74.4%)接近或略优。

- **代码生成**：两者在 HumanEval 和 LiveCodeBench 上表现相当，均达到 90%+ 的通过率。

- **科学推理**：在 GPQA Diamond(研究生级别科学问答)上，R1 与 o1 都在 70% 左右，处于同一梯队。

- **通用能力**：R1 在 MMLU(通用知识)和 IFEval(指令遵循)上略逊于 o1，这可能是因为 R1 的 RL 训练更聚焦于推理领域，而 o1 可能使用了更广泛的通用数据。

### 8.3 哲学差异：黑箱 vs 透明

o1 和 R1 的最大差异不在于能力，而在于**可解释性和可控性**。o1 的推理链对用户完全隐藏，你只能看到最终的 polished 答案，看不到模型"思考"的过程。这种设计可能出于商业考量(防止 prompt injection 攻击推理链)，但也使得用户无法判断模型是"真的推理出了答案"还是"碰巧蒙对了"。

R1 则将完整的推理链暴露给用户。你可以看到模型如何从错误中纠正、如何验证中间步骤、如何在多个思路之间切换。这种透明性对于教育场景("让我看看模型是怎么解这道题的")和科研场景("分析模型的推理模式来改进算法")具有巨大价值。

## 9. 局限性与边界条件

### 9.1 规则奖励的覆盖局限

R1 的 RL 训练极度依赖规则-based 的奖励函数。这种设计在数学和代码领域表现优异，因为答案的正确性可以被自动验证。但在以下领域，规则奖励难以构造：

- **开放式创意写作**：如何判断一篇小说是否"好"？

- **伦理决策**：如何自动评判一个道德困境中的选择？

- **多轮对话质量**：如何量化"对话是否自然、有帮助"？

在这些领域，R1 的训练范式难以直接应用。虽然 R1 在通用能力上通过后续的拒绝采样和全场景 RL 做了弥补，但其核心优势仍然集中在可规则验证的领域。

### 9.2 推理链长度的边际效益递减

R1 的推理链在复杂问题上可能长达数千甚至上万 token。虽然更长的推理链通常带来更高的准确率，但这种关系并非线性：

- 当推理链从 0 扩展到 1000 token 时，准确率提升显著(如从 30% 提升到 70%)。
- 当推理链从 1000 扩展到 4000 token 时，准确率提升趋缓(如从 70% 提升到 80%)。
- 当推理链超过 8000 token 时，准确率可能开始下降——模型在过长的序列中容易迷失方向，产生无关的重复或偏离主题的推理。

这意味着"更多思考时间"并不总是等于"更好答案"。如何在推理链长度和准确率之间找到最优平衡点，是一个尚未被完全解决的问题。

### 9.3 对预训练基座的能力依赖

R1 的推理能力建立在 DeepSeek-V3-Base 的基础上。V3-Base 本身已经在预训练阶段接触了大量数学和代码语料，具备了基础的符号操作和逻辑表达能力。如果换用一个预训练数据中缺乏数学内容的基座模型(如某些专注于文学创作的模型)，纯 RL 是否还能涌现出同样的推理能力？这是一个开放性问题。

实验证据表明，预训练阶段的数据分布对 RL 的效果有显著影响。R1-Distill 实验显示，同样的蒸馏数据在不同基座模型上的效果差异很大——Qwen 和 Llama 的基座表现优秀，而某些其他架构的基座则效果平平。这说明 R1 的 RL 范式并非"万能钥匙"，它需要与一个具备良好"推理先验"的基座模型配合才能发挥最大效力。

## 10. 演进与承上启下

### 10.1 R1 对后训练领域的深远影响

R1 的成功标志着大模型后训练进入了一个新阶段：**从"模仿人类"到"自我进化"的范式转移。**

在 R1 之前，后训练的核心任务是"对齐"——让模型模仿人类标注者认为"好"的回答。SFT 和 RLHF 都是对齐工具，它们的天花板由人类标注的质量和规模决定。

R1 之后，后训练的核心任务变成了"激发"——设计合适的奖励信号和环境，让模型自主发现超越人类的策略。RL 不再是对齐的辅助手段，而是能力涌现的主要引擎。

这一范式转移正在催生一系列后续创新：
- **过程奖励模型(PRM)** ：不仅奖励最终答案，还奖励中间推理步骤的质量。OpenAI 的 o1 reportedly 使用了 PRM，但具体实现仍是秘密。

- **多智能体 RL**：让多个模型实例相互协作或竞争，通过社会性交互提升推理能力。

- **元学习 + RL**：训练模型"学习如何学习"，使其在面对全新类型的问题时能快速调整推理策略。

### 10.2 从 R1 到 V4：推理与通用的融合

DeepSeek-R1 专注于推理能力，但它并不是一个通用对话模型。R1 在开放式闲聊、创意写作、情感支持等任务上的表现虽然不差，但并未经过针对性的优化。DeepSeek 的下一步可以预见是**将 R1 的推理能力与 V3/V4 的通用能力进行深度融合**。

这种融合的方向在 DeepSeek-V4 的设计中已经初见端倪：V4 的后训练 pipeline 将 G-OPD(广义在线蒸馏)与 GRPO 结合，先用蒸馏快速预热，再用 RL 精细调优，最后用 SCOPE 过滤噪声。这种"先蒸馏、后 RL"的 pipeline 正是 R1 经验向通用模型迁移的体现。

### 10.3 对行业的启示：开源推理模型的未来

R1 的开源发布建立了一个新的行业基准：**顶级的推理能力不再必须依赖闭源系统。** 在 R1 之后，开源社区出现了大量基于 R1 的衍生工作——更高效的 GRPO 实现、适用于不同领域的奖励函数设计、与 RAG(检索增强生成)结合的推理框架等。

未来的开源推理模型可能会沿着以下方向演进：
1. **更轻量的 RL 算法**：GRPO 已经比 PPO 高效很多，但仍有优化空间。如何在更少的 GPU 小时数内达到相同的 RL 效果，是社区的热点研究方向。

2. **多模态推理**：将 R1 的文本推理能力扩展到图像、视频、音频领域。例如，让模型在解决几何问题时能够"看到"图形并进行视觉推理。

3. **工具使用与 Agent 能力**：将推理能力与外部工具(计算器、搜索引擎、代码执行器)结合，让模型在推理过程中主动调用工具来验证和辅助推导。

## 11. 总结与参考文献

### 11.1 核心要点总结

1. **R1-Zero 的历史性证明**：纯强化学习(无需任何人类标注的 CoT 数据)可以让大模型自发涌现自我验证、反思、回溯等高级推理行为。

2. **GRPO 算法的工程突破**：通过组内相对优势替代价值网络，将 RL 训练的显存开销降低约 50%，同时消除了价值网络训练不稳定的问题。

3. **冷启动 SFT 的"播种"作用**：仅需数千条高质量人类数据即可教会模型输出格式和可读性，为后续的 RL 阶段提供干净的起点。

4. **蒸馏的跨架构迁移能力**：R1 的推理能力可以高效蒸馏到 Qwen 和 Llama 架构的小模型上，7B 参数的蒸馏模型即可达到接近 GPT-4o 的数学推理水平。

5. **开源推理的新纪元**：R1 以完全开源的姿态证明了开源模型在推理能力上可以达到甚至超越闭源对手，为学术界和产业界提供了可复现、可改进的推理训练基线。
