## 怎么写一条

每条建议包含：

| 字段 | 说明 |
|------|------|
| **日期** | `YYYY-MM-DD` |
| **模块** | 如 Chat / Memory / Auto-Compact / Swarm / 前端 / Prisma |
| **现象** | 用户或开发者看到了什么 |
| **根因** | 为什么发生（尽量说到职责越界或缺失的不变量） |
| **修法** | 已落地 / 计划中 / 仅规避 |
| **教训** | 一句话，下次怎么避免 |

**反模式**：只写「修了 bug」不写根因；用 `setTimeout` / `await hydrate` 当修法却不说明是补丁栈（见 `AGENTS.md` 架构纪律）。

## Chat / 状态机

### 2026-07-10 — 编排层补丁栈 vs reducer 不变量

- **现象**：刷新 / 切 session / 双路 SSE 竞态下，消息丢失、时间线重复、done 后 UI 被清空。
- **根因**：`onDone` 里 `await hydrate`、`useEffect` 猜 phase、`queueMicrotask` 守卫——不变量没在 store reducer 里强制。
- **修法**：Stream commit 不变量收进 reducer；`onStreamCommitted` 作唯一 drain 钩子。见 `chat-state-architecture.md` §4.2。
- **教训**：**删掉编排层补丁，bug 仍复现 = 还在打补丁。**先画状态机，再写代码。

### 2026-07-11 — 子会话流式恢复与父会话串台

- **现象**：父 Agent 流式中切走再回来，thinking 重复或子 Agent 事件跑进父时间线。
- **根因**：`originSid` / `sessionId` 在 SSE 回调里未始终绑定到「发起请求的会话」。
- **修法**：`streamLifecycleStore` 按 session 隔离；E2E `chat-subagent-resume-mock` 锁住场景。
- **教训**：多 session 并存时，**任何 emit/handler 必须带明确 session 键**，不能用全局单例隐式当前会话。

## Auto-Compact / 上下文

### 2026-07-12 — 压缩完全静默，用户以为「模型卡了」

- **现象**：长对话发送后助手区空白 5–30s，无工具条、无进度，然后突然开始 thinking。
- **根因**：`maybeCompactMessages` 在 `runAgentLoopStream` 里、**第一轮 `round_start` SSE 之前**同步 `await`（含 memoryFlush + 摘要 LLM）；前端无 `compact_start/end` 事件。
- **修法**：**计划中**——SSE 显式 compact 阶段 + 时间线 `__context_compact__` 伪工具转圈；边界消息落库。见本轮讨论。
- **教训**：**阻塞式后台阶段必须占用与用户可见的「工具时间线」**，不能藏在 LLM 轮次之前。

### 2026-07-12 — UI 两个「上下文 %」不是一回事

- **现象**：顶栏显示上下文 30%，设置面板却显示压缩进度 90%+，不知道会不会压。
- **根因**：`ratio` = 估算 token / 128k；`compactRatio` = 估算字符 / 压缩阈值——两个分母。
- **修法**：后端已改为 `triggerRatio × model.contextWindow` 动态阈值；前端 `contextUsage` / `tokenBudget` 对齐同一 `compactCharThreshold`。**仍缺** UI 只展示一条「距自动压缩」进度。
- **教训**：**一个用户概念只对应一个分母**；配置项改名要比加第三条进度条清晰。

### 2026-07-12 — 送 LLM 先拉全量 200 条再内存里压

- **现象**：session 消息很多时，压缩前后 CPU/延迟高；逻辑上「只需上次压缩点之后」却仍每次 `list(200)` + 全量 `buildLlmMessagesFromHistory`。
- **根因**：`contextSummary` 只在运行时合成进 messages，没有 `lastCompactMessageId` 裁剪历史。
- **修法**：**计划中**——`buildLlmContextSinceCompact` + 聊天列表「已压缩 N 条」折叠区。
- **教训**：**展示用全历史、推理用窗口**要显式分叉，不能靠每次 macro 摘要赌运气。

### 2026-07-12 — 压缩摘要只写 session 字段，时间线无分界

- **现象**：用户往上滚能看到旧消息，但不知道「从哪条起模型已经看不见了」；刷新后摘要只在顶栏胶囊里。
- **根因**：摘要不写入 `ChatMessage`，仅 `ChatSession.contextSummary`。
- **修法**：压缩完成后写一条 `assistant` 边界消息（`source: "system"`，`toolCalls` 含 `__context_compact__`），`MessageService.afterCreate` 自动推 `message_upserted`，前端时间线渲染为 ToolStep。
- **教训**：**影响用户心智的状态变化，必须在主时间线留痕。**

### 2026-07-12 — Auto-Compact 显式化（SSE + 时间线 + 边界消息）

- **现象**：长对话发送后助手区空白 5–30s，无工具条、无进度，然后突然开始 thinking。
- **根因**：`maybeCompactMessages` 在 `runAgentLoopStream` 里、第一轮 `round_start` SSE 之前同步 `await`（含 memoryFlush + 摘要 LLM）；前端无 `compact_start/end` 事件。
- **修法**：
  - `AgentStreamEvent` 新增 `compact_start` / `compact_end` / `compact_error`。
  - `maybeCompactMessages` 接收 `emit`，仅在真正 macro 压缩时 emit `compact_start`（reused / 未触发不 emit）；成功不 emit `compact_end`（留给调用方），失败 emit `compact_error`。
  - `runAgentLoopStream` 调 maybeCompactMessages 时传 `options.emit`；写入 `ChatSession.contextSummary` 后再写一条 `assistant` 边界消息（`source: "system"`，`toolCalls: [{ name: "__context_compact__", kind: "compact", ... }]`），然后 emit `compact_end` 带 `boundaryMessageId`。
  - 前端 `agentStream.ts` 把 `compact_start/end/error` 映射成 `onToolStart/onToolEnd("__context_compact__", ...)`，复用现有工具时间线转圈 UI。
  - `ToolStep` 显示名映射：`__context_compact__` → 「上下文压缩」。
- **教训**：**阻塞式后台阶段必须占用与用户可见的「工具时间线」**，不能藏在 LLM 轮次之前。边界消息落库让刷新 / 切 session 回来仍能看到「第 N 次压缩发生在哪」。

## Memory / 长期记忆

### 2026-07-12 — Memory 与 Auto-Compact 被混为一谈

- **现象**：以为「压缩会自动记住关键事实」，或以为 `/memories` 会自动从对话提取偏好。
- **根因**：产品文案 / 心智模型未区分：Memory = 跨 session 的 system 注入；Compact = 单 session 的 messages 替换。
- **修法**：文档厘清；`memoryFlush` 在 macro 摘要前静默写 Memory（已落地 `memoryFlush.ts`）；默认 assistant 开启 `memory_create` / `memory_search`（2026-07-12）。
- **教训**：**两种「记忆」必须在文档和 UI 用不同名词**（长期记忆 vs 会话摘要）。

### 2026-07-12 — `experience` 污染 Chat 召回

- **现象**：助手回复里突然出现大段 JSON 任务日志。
- **根因**：`accumulateExperience` 每轮 Chat 写 `type: experience`；`buildMemoryContext` 不过滤 type，FTS 命中即注入 system。
- **修法**：`isMemoryInjectable` / 常量层已区分；**待彻底**——`buildMemoryContext` 默认排除 `experience`。见 `memory-research-plan.md` S1。
- **教训**：**内部遥测型 Memory 默认不得进用户对话 prompt。**

### 2026-07-12 — `/memories` 写「向量化」实际只有 FTS

- **现象**：开发者按向量检索去对接，发现只有 FTS5 + LIKE。
- **根因**：产品页文案超前于实现。
- **修法**：**待改** UI 文案或落地 embedding；研究计划明确「先不上向量库」。
- **教训**：**管理页描述必须和 `agentRuntime.buildMemoryContext` 实现一致。**

### 2026-07-12 — 默认 assistant 长期没有 memory 工具

- **现象**：Agent 说「记住了」但未落库；只有被动 FTS 召回。
- **根因**：`content/agents/assistant.md` 未声明 `memory_create`；`resolveAgent` 自动补齐列表也未包含。
- **修法**：assistant 默认工具 + `swarmPermissionGuard` 限制 sub 不可用；`assistant.md` 与 `DEFAULT_ASSISTANT_TOOLS` 同步。
- **教训**：**能力要同时在 Markdown 源、DEFAULT_* 常量、tier 权限三层对齐。**

## Swarm / Agent 工具

### 2026-07-15 — 子 Agent 报告双通道重复投递：父 Agent 对同一份结果说了两遍话

- **现象**：用户让父 Agent 启动子 Agent 执行「sleep 10s 并汇报」。观察到的怪异链路：父 Agent 正式回复还没结束就能在异步任务区看到结果 → 正式回复说「任务完成」→ 回复结束后同一份 report 又作为气泡注入、父 Agent 又回复一遍 → 期间父 Agent 还多发了条指导消息把子 Agent 唤醒跑了第二轮，于是收到第三份回复。一份结果，父 Agent 对用户说了三遍。
- **根因**（逐层拆解，证据：父 session `cmrkvwhi` / 子 session `cmrkvws7`，2026-07-15 00:47~00:48）：
  1. **结果双通道无幂等**：子 Agent 结果可经「A 父轮询 `async_task_status` 读 task.output 全文」和「B report_back → 跟踪 Task success → 异步结果队列 `autoConsumeAsyncDelivery` 原子认领 → 气泡注入 + drain」两条路到达父会话。A 读了内容不改任何状态，B 不知道 A 读过，照插气泡 → 父 Agent 把同一份报告当「新消息」又回复一遍。（注：B 管道内部已有原子 claim 幂等，问题在 A/B 之间无对账。）
  2. **AgentMessage 记账不回写（真 bug）**：report_back 会经 swarmBus 顺手写一条 AgentMessage（旁路邮箱/审计记录，不是投递载具——投递走的是 B 的 Task 管道），两条 report（`cmrkvxdu`、`cmrkvxyd`）对应结果已注入父 session，但 AgentMessage 至今 `status=pending, deliveredAt=null`——pending 残留 = 未来重复投递的定时炸弹。
  3. **异步 spawn + 同步轮询反模式**（LLM 行为）：父 Agent spawn 用异步模式，却在 ReAct 循环里连调 13 次 `async_task_status` + 3 次 `agent_inspect` 把自己变成同步等待（36 秒、十几轮 LLM 调用）。
  4. **父 Agent 多余干预**（LLM 行为）：轮询没等到 report（其实 task 已完成、report 已创建只是未注入），判断子 Agent 迷路，主动发 `agent_send_message` 指导 → 把已完成任务的子 Agent 唤醒跑了第二轮。
  5. **指令保真丢失**（LLM 行为）：用户说「调用 sleep 工具」，父 Agent spawn 时改写成「用 web_search 消耗时间」（它以为子 Agent 没有 sleep；实际有，第二轮子 Agent 就用了）。
- **修法**：**部分计划中**——
  - 记账 bug（②）：W14 工单已立项（`architecture-fix-prompt-2026-07-v3.md`），按 taskRef 对账，Task 管道消费时同步回写 AgentMessage `delivered/deliveredAt → consumed`。
  - 双通道去重（①）：消费状态语义区分「A 查询消费」vs「B 气泡呈现」——设计问题已挂 `design-decisions.md` 问题 G 待拍板（推荐方案一：`async_task_status` 不回 output 全文，让结果全文只有 B 一条到达路径）。
  - ③④⑤ 属 prompt/工具描述引导：spawn 语义（等结果用 `waitForResult=true`，异步就先回复用户别轮询）+ 指令原样传递，暂不动系统。
- **教训**：**同一结果的到达路径有两条时，幂等去重必须做在投递层，不能指望 LLM 自己认出「这个消息我见过」。** 以及：查代码再下结论——最初把 B 管道误记成「report_back 直接注入」的第三条独立通道，实际是 report_back → Task → 队列消费的同一条管道，AgentMessage 只是旁路账本；通道数错，去重设计就会画错靶子。

### 2026-07-10 — 重复超级 Agent 副本

- **现象**：sync 后出现两个超级 Agent；心跳、会话挂在不同 id 上。
- **根因**：`content/agents/` 与 `swarmInitializer` 各创建一条；`sync-agents` 未跳过 `tier === super"`。
- **修法**：删重复文件 + 软删 DB 副本；`sync-agents` upsert 跳过 super。
- **教训**：**系统级 singleton Agent 不要走内容同步双向写回。**

### 2026-07-12 — 子 Agent 工具不能靠「默认列表」 alone

- **现象**：若在子 Agent 的 `tools` 里手写 `native:memory_create`，仍可能进 LLM schema。
- **根因**：仅 `DEFAULT_SUBAGENT_TOOLS` 不含 memory，但无 tier 硬拦时创建参数可传入。
- **修法**：`memory_*` 列入 `TIER_RESTRICTED_TOOLS` manager 档；`getAllowedToolsForTier` + `checkToolPermission` 双保险。
- **教训**：**「默认不开」不等于「禁止开」；sub 的限制要进 permission guard。**

### 2026-07-11 — `spawn_subagent` 同步等待 vs `report_back`

- **现象**：父 Agent 挂起等子会话，子 Agent 不知道要不要 `report_back`。
- **根因**：`waitForResult` 两种语义共用一个工具，systemPrompt 默认文案不一致。
- **修法**：`waitForResult=true` 时子 prompt 写明「无需 report_back」；系统抓取最后 assistant。
- **教训**：**工具参数改变交付语义时，子 Agent 的 prompt 必须跟着变。**

## 前端 / UI

### 2026-07-12 — Chat 右侧设置面板横向溢出

- **现象**：360px 侧栏出现横向滚动条；搜索引擎优先级一长串 URL 撑破布局。
- **根因**：Token 预算与搜索引擎横向 pill + `break-all` 优先级字符串重复展示。
- **修法**：`SettingsSection` 重组；搜索引擎改纵向编号列表；`min-w-0 overflow-x-hidden`。
- **教训**：**窄侧栏一律 `min-w-0` + 纵向列表**，不要假设用户拉宽面板。

### 2026-07-11 — 手动压缩有 `compactPending`，自动压缩没有

- **现象**：点「立即压缩」有「压缩中…」，自动触发却无任何反馈。
- **根因**：`compactPending` 只绑 `trpc.session.compact` mutation，自动路径无 SSE。
- **修法**：同「压缩完全静默」条，需 SSE + 时间线。
- **教训**：**同一能力的手动/自动路径应共享 UI 状态机。**

## 数据库 / 同步 / 工程

### 2026-07-11 — `ChatSession` 列表按 `createdAt` 排序

- **现象**：活跃会话沉到下面，用户以为会话丢了。
- **根因**：`SessionService.list` 用 `createdAt` 而非 `updatedAt`。
- **修法**：改 `updatedAt` 排序 + `@@index([updatedAt])`。
- **教训**：**会话列表默认按「最后活动时间」排。**

### 2026-07-11 — `asyncJobOrchestrator` 排队超时回调残留

- **现象**：长时间运行后 `queuedTimeouts` Map 增长；偶发重复超时回调。
- **根因**：超时触发后未从 map 删除。
- **修法**：回调内 `queuedTimeouts.delete(jobId)`。
- **教训**：**所有 `setTimeout` 注册都要在 fire 或 cancel 时成对清理。**

### 2026-07-11 — `heartbeatEngine.refresh()` 竞态孤儿 cron

- **现象**：`stop()` 后仍有个别 cron 触发。
- **根因**：`refresh` 内 `await` 之后未再检查 `this.started`。
- **修法**：await 前后双检查 `this.started`。
- **教训**：**异步 refresh/stop 必须防重入，用代数或 started 标志。**

### 2026-07-10 — Windows 上 `prisma generate` EPERM

- **现象**：dev 中 generate 报文件被占用。
- **根因**：server 进程锁着 query engine DLL。
- **修法**：先停 server 再 generate；或仅改 schema 索引时用 SQL 直改。
- **教训**：**Windows 开发习惯：改 Prisma 前先停后端。**

### 2026-07-10 — FTS 表 `db push` 失败

- **现象**：migrate/push 在 FTS 虚拟表上报错。
- **根因**：Prisma 对 SQLite FTS 支持有限。
- **修法**：索引变更用原生 SQL migration 或手工执行。
- **教训**：**FTS 是旁路索引，不要指望 Prisma 全自动迁移。**
